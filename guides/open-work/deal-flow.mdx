## Orbiter Deal Flow — Definitive Backend Design

Deal Flow is a single, organization-scoped module that unifies three surfaces on one data spine: **Projects** (hierarchical, section-organized workspaces of work), **Tasks** (the atomic work unit, which doubles as a personal to-do when unassigned), and **Deal Flow** (fully-customizable CRM pipelines whose cards are opportunities anchored to companies/people in Orbiter's FalkorDB relationship graph). The design philosophy is **"one entity, many lenses"**: the same `df_task` is a subtask, a personal todo, or a project obligation; the same `df_status` engine drives task workflows and deal stages; the same polymorphic grant / comment / event / presence / follower / attachment layer makes *any* resource shareable, multiplayer, auditable, and notifiable. This collapses what most CRMs ship as 5–6 disjoint subsystems into a small, orthogonal core, and — critically — adds the three things that separate a "passive datastore" from best-in-class deal flow: an **automation rules engine**, a **forecasting/target model**, and a **universal event timeline**.

The customization story is the differentiator. Statuses and stages are **first-class rows, never enum columns** — users rename, recolor, reorder, and add them with zero migrations — while an *immutable* `category` enum on each row keeps analytics, forecasting, and won/lost rollups stable across renamed pipelines. Custom fields are **schema-as-data**: a typed field-definition table, first-class select options, and a **hybrid typed-EAV value table** (type-specific columns), chosen over a JSON blob so Xano's column/join query builder and the FalkorDB enrichment jobs can filter and sort custom fields. Because this workspace exposes **no GIN/partial index primitive** (verified: every live table carries only a PK plus single-field btrees; `createBtreeIndex`/`createUniqueIndex` accept multi-field arrays, `createSearchIndex` is full-text-only), multi-value fields use a dedicated `df_custom_field_value_option` child join table rather than a JSON array + GIN, and every index in this design is a real, buildable btree/unique/search index.

Sharing and multiplayer run through **one polymorphic `df_grant` + a single `resolve_access()` resolver**, with a denormalized `df_grant_effective` cache so the resolver is not an N+1 bottleneck on every endpoint and realtime broadcast. Grants target users, the existing `team` (613) primitive, or hashed link tokens (`df_share_link`); modes **extend** `note_collaborator` (655)'s exact `[editor,viewer,commentor]` set with a `manager` tier, ordered by an explicit `role_rank()` map (Xano enums have no intrinsic order). Realtime reuses the TipTap/Hocuspocus infra already on `user.tiptap_collab_jwt`: ephemeral presence for cursors, last-write-wins per-field board sync, CRDT reserved for rich-text notes.

Everything is scoped to `organization_id` (-> organization 695, the WorkOS tenant). Because the existing core tables (`user`/`team`/`note_collaborator`) are tenant-agnostic and carry **no** organization_id, the user↔org mapping is resolved exclusively through `organization_membership` (696) — which requires a new composite `(user_id, organization_id, status)` btree to keep `resolve_access()` cheap. The PM "project" entity is named **`df_project`** to avoid the hard collision with the existing graph-artifact `project` (575), and the whole module uses the `df_` prefix for one unmistakable namespace.

## Requirements coverage

The eight requirements that scoped this design, and where each is satisfied:

| # | Requirement | How it is met |
| - | ----------- | ------------- |
| 1 | Users can create projects | `df_project` (org-scoped, owner `user_id`, container shape mirrors `team` 613) via `POST /df/project` |
| 2 | Projects can be siblings / nested | `df_project.parent_df_project_id` self-FK + materialized `path`/`depth`; siblings share a parent, arbitrary depth |
| 3 | A task with no project is a to-do | `df_task.df_project_id` is **nullable** — null = personal to-do; "My Tasks"/Inbox is a saved `df_view`, not a separate table |
| 4 | Assign tasks to teammates | `df_task_assignee` join (multiple assignees) + `df_follower` watchers + `df_event`/`activity` notifications |
| 5 | Assign a project to a team | `df_grant` with `subject_type=team` → `team` 613 / `team_member` 614 (`status=accepted`) |
| 6 | 100% customizable pipelines & statuses | `df_pipeline` (many per org) + `df_status` rows (rename/recolor/reorder, **immutable `category`** drives analytics) + custom fields (`df_custom_field_*`) |
| 7 | Share + make it multiplayer | Polymorphic `df_grant` + `resolve_access()` + `df_grant_effective` cache; realtime via existing TipTap/Hocuspocus + `df_presence`; `df_comment`/`df_follower` |
| 8 | Leverage Orbiter's relationship graph | `df_deal.master_company_id` → graph node, `df_deal_association` stakeholders, `df_deal_intro_path` cached warm-intro paths |

## Data model

The module adds **29 new tables, all prefixed `df_`** (one unmistakable namespace for the **D**eal **F**low module, and a hard separation from the unrelated graph-artifact `project` table 575). They reuse — via foreign key, never by duplication — a small set of **already-built** core tables for identity, tenancy, sharing, and graph anchoring. The two are kept strictly apart below.

### Reused existing tables (already built)

These exist in Xano workspace 3 today. The new module references them by FK and does **not** modify them, with one exception (a new composite index on `organization_membership`).

| Table | ID | Status | How the module uses it |
| ----- | -- | ------ | ---------------------- |
| `organization` | 695 | built | WorkOS tenant. Every new root table FKs `organization_id` here for multi-tenant isolation. |
| `organization_membership` | 696 | built + new index | user↔org mapping & role (admin/member). The `resolve_access()` resolver reads it; this design adds a composite `(user_id, organization_id, status)` btree. |
| `user` | 125 | built | Every actor field (owner/creator/assignee/lead). Already carries `tiptap_collab_jwt` → realtime reuses existing Hocuspocus infra. |
| `team` | 613 | built | The "assign a project to a team" subject (`df_grant.subject_team_id`). Its container shape (icon/color/title) is mirrored by `df_project`/`df_pipeline`. |
| `team_member` | 614 | built | Team membership for team-grant resolution; filtered on `status=accepted`. |
| `note_collaborator` | 655 | built | Per-resource share precedent. `df_grant.mode` extends its `[editor,viewer,commentor]` enum with a `manager` tier. |
| `activity + activity_type` | 218 / 217 | built | User-facing notification feed (`unread` bool). `df_event` fans surfaced verbs into it via a verb→activity_type map. |
| `master_company / master_person` | 142 / 139 | built | Canonical FalkorDB graph nodes that deals and `df_deal_association` stakeholders anchor to (no shadow contact records). |

## Customization at the data layer

Three orthogonal customization axes, all schema-as-data (zero migrations to customize), all re-specced against this workspace's real index primitives (verified: only PK + single-field btrees exist live; `createBtreeIndex`/`createUniqueIndex` accept multi-field arrays so composite indexes ARE buildable; there is NO GIN/partial-index primitive; `createSearchIndex` is full-text only).

### 1. Statuses & stages — first-class rows with an immutable category
`df_status` is the single status engine for tasks, projects, and deals (disambiguated by `scope`). Users freely rename (`name`), recolor (`color`), reorder (LexoRank `position`), add/remove stages. The decisive trick (unanimous across Linear/Attio/Pipedrive/Notion): every row carries an **immutable `category` enum** `[backlog,unstarted,started,done,canceled]`. All UX flexibility lives in name/color/position; all logic — forecasting, won/lost rollups, cycle-time, cross-pipeline reporting — keys off `category`, never the name. Deal stages reuse the same enum with a fixed mapping (`started=open, done=won, canceled=lost`). Deal-only forecasting attributes (`probability`, `rotting_days`) ride as nullable columns on the same row.

**Cross-scope safety (critique fix):** because task and deal statuses share one table, a bug could attach a deal stage to a task. Two table triggers close this: an `df_task` insert/update trigger asserts `status.scope='task'`; an `df_deal` trigger asserts `status.scope='deal' AND status.df_pipeline_id == deal.df_pipeline_id`. `df_deal.status` is a denormalized open/won/lost cache derived from the stage's category (trigger-maintained, never user-edited).

### 2. Custom fields — hybrid typed-EAV with a child join for multi-value (the deliberate, workspace-correct choice)
Five tables: `df_custom_field_def` (typed field definitions + `config` JSON), `df_custom_field_option` (first-class select options, stable ids, mutable label/color), `df_custom_field_value` (ONE row per field-per-record, typed columns), `df_custom_field_value_option` (child join for multi-select and multi-ref), and the option/def pair scoping per-pipeline (deals) or per-project/org (tasks/projects via nullable `df_project_id`).

**The value store is typed-EAV with typed columns** (`value_text`/`value_number`/`value_timestamp`/`value_bool`/`value_ref_id`+`value_ref_table`), NOT a JSON `properties` blob.

- **Why typed-EAV over JSON blob:** the dominant query is cross-record filter/sort/rollup ("deals where ARR > 1M and stage = started", forecast sums, board grouping). These need a real btree on `value_number`, which a typed column provides and JSON paths do not.
- **Multi-select WITHOUT GIN (critique fix):** the v1 plan stored `value_option_ids` as a JSON array filtered by a `GIN` index — **not buildable in this workspace**. Replaced with `df_custom_field_value_option`, a child join carrying one row per selected option, btree-indexed on `(organization_id, df_custom_field_option_id)`. "Records where multi-select contains X" becomes an indexed btree join, not a JSON scan. This also resolves the unique-index contradiction: `df_custom_field_value` keeps its clean `unique(def_id, entity_type, entity_id)` (one value row per field), and multi-cardinality lives entirely in the child table.
- **Clearing a field (critique fix):** the unique index means "clear a field" sets the value columns to NULL on the existing row (it does not delete the row) and deletes any child option rows — specified, not ambiguous.
- **Single vs multi ref:** v1 stores a single `value_ref_id`+`value_ref_table` on the value row; `is_multi_ref=true` defs use `df_custom_field_value_option` ref columns for multiple. v1 default is single-ref; auto-inverse is deferred.
- **Why not dynamic DDL:** migration storms, table locks, per-tenant schema sprawl.

### 3. Formula & rollup fields — a real compute layer (critique fix; not a dead enum)
`field_type` includes `formula` and `rollup`, and each is actually evaluated rather than left as a dead enum value:
- `config.expr` holds the expression and `config.refs` lists the field keys it depends on.
- A change to any referenced field sets `is_dirty=true` on dependent value rows (dependency tracking at write time).
- A recompute fn (and a cron sweep over `btree(is_dirty)`) evaluates dirty rows, materializing the result into `value_number` (numeric formulas/rollups) or `value_text` (string), so results are filterable/sortable like any other typed value.
- `rollup` config `{source, agg, filter}` aggregates related records (e.g. sum of subtask estimates, count of won deals on a company).

### 4. Bool-flag defaults (verified gotcha)
Per the workspace memo, defaulted bools (`is_active` default true; `is_default`/`is_required`/`is_archived`/`is_shared` default false) are read via `first_notnull` so a stored `false` is not coerced back to a `true` default — and the true-default vs false-default flags are kept on separate fields so they never collide.

### 5. Views — saved query objects
`df_view` stores `view_type` + `filters`/`sorts`/`group_by_field`/`visible_field_keys` as JSON, decoupled from storage. Kanban = `view_type=board` grouped by status or a single_select field; the same data renders as list/calendar/table/timeline.

## Hierarchy: projects, sections, tasks, subtasks

### Project nesting & siblings
`df_project.parent_df_project_id` is a nullable self-FK. A root project has `parent=null`; children of the same parent are **siblings**, satisfying the requirement directly. Nesting is arbitrary-depth. To make subtree reads cheap without recursive CTEs (Xano has none), each row carries a **materialized `path`** (`/12/40/91/` ancestor chain) and a denormalized `depth`. Subtree query = `WHERE path LIKE '/12/40/%'`.

**Index fix (critique):** the v1 composite `btree(organization_id, path)` cannot serve a left-anchored `path LIKE` prefix (the leading `organization_id` kills prefix matching). `path` is therefore indexed **standalone** (text prefix), with `organization_id` filtered in the WHERE clause. Moving a project rewrites `path`/`depth` for the moved node and all descendants in one batched fn (the descendant set is itself a `path LIKE` query); cycle-guarded by checking the new parent is not in the moved subtree.

### Sections / milestones within a project (critique fix)
`df_project_section` adds Asana-style sections and Linear-style milestones inside a project. `df_task.section_id` (nullable) slots a task into a section; multi-homed tasks carry a per-project `section_id` on `df_task_membership`. Sections order by LexoRank; a board can `group_by_field='section'` instead of status.

### Task = subtask = personal to-do (one entity)
`df_task` is the single entity for all three. The nullable self-references do the work:
- **`df_project_id` nullable** -> a project-less task IS a personal to-do. "My Tasks"/Inbox is a saved `df_view` filter (`assignee=me`), not a separate table.
- **`parent_task_id` nullable self-FK** -> subtasks, first-class (assignable, statusable, dependency-capable, searchable) at arbitrary depth.

**Why one entity:** separate todo/subtask tables would fracture assignment, status, dependencies, search, rollups, and realtime into 3x parallel code paths. Single-entity self-reference is the Linear/Asana-proven path.

### Ordering, recurrence, dependencies
- **Ordering uses LexoRank `text` rank, NOT decimal (critique fix):** decimal midpoints exhaust float precision after ~50 inserts and silently collide. `position` is `text` carrying a base-36 lexorank string; reorder = single-row write, with a periodic rebalance fn as a safety net. (team_member.position is `int` in the legacy schema; new tables standardize on text lexorank to avoid reorder cascades.)
- **Recurrence:** `df_task.recurrence_rule` (RFC-5545 RRULE) + `recurrence_parent_task_id`; on completion a spawn fn (cron-backed) creates the next instance.
- **The subtask tree (parent_task_id) is distinct from the dependency graph (df_task_dependency)** — tree = containment, graph = directed blocks/relates. Acyclicity for both parent chains and dependency edges is enforced on write (ancestor walk / DFS) since Xano can't guard cycles natively.

## Sharing & multiplayer

### Sharing scopes & permission modes
Sharing is one polymorphic `df_grant` table — never per-resource collaborator tables. A grant targets any resource (`resource_type` + `resource_id`) for any subject: an **individual user** (mirrors note_collaborator/655), a **team** (reuses team/613 + team_member/614 — this is precisely how a project is "assigned to a team"), or a **link token** (`df_share_link`, hashed, expiring, referenced by `subject_link_id`). Org-wide visibility comes from `organization_membership` (696, role admin/member, status=active); team visibility from `team_member` (614, status=**accepted** — verified value, filtered everywhere).

**Modes (critique fix — honest framing):** `df_grant.mode` is `[manager,editor,commentor,viewer]`. note_collaborator's verified set is exactly `[editor,viewer,commentor]` (lowercase, default "Viewer"), so this is an **extension** with a `manager` tier, not a literal mirror. Because Xano enums have **no intrinsic order**, the resolver uses an explicit `role_rank()` lookup (`viewer=1, commentor=2, editor=3, manager=4`) for all MAX comparisons.

**df_grant unique-index fix:** the v1 single composite `unique(resource_type,resource_id,subject_type,subject_user_id,subject_team_id)` was broken — Postgres treats NULLs as distinct, so duplicate team/link grants slipped through, and there was no `subject_link_id`. Replaced with `subject_link_id` plus **three separate unique indexes** (one per subject column), each covering only the rows where that subject is set (subject_type guard in the fn layer), so duplicates are actually prevented.

### One resolver, defense by gateway, plus a cache (critique fix — N+1)
Effective permission is computed by `resolve_access(user, resource_type, resource_id)`: `effective_mode = MAX` (via `role_rank()`) over (org membership role -> team grants the user belongs to -> individual grants -> link grant if a token is presented), walking the parent chain with inheritance (deal -> df_pipeline; task -> df_project -> ancestor df_projects via `path` -> org) and honoring child overrides via `inherited_from_grant_id`. Xano has no DB-level RLS, so this resolver is the **only data gateway** — every endpoint and realtime broadcast goes through it.

To avoid the resolver being an N+1 hot path, results are materialized into **`df_grant_effective`** (`user x resource -> effective_mode + source`), refreshed when a grant changes, when org/team membership changes, or when project hierarchy moves. Endpoints read the cache (single indexed lookup) and only fall back to a live recompute on a cache miss; a per-request memo layer dedupes repeated checks within one call. Board listing filters directly on `df_grant_effective(user_id, resource_type)`.

**org-scope reality (verified):** existing core tables carry NO organization_id, so user->org cannot be joined cheaply on the resource tables; the resolver resolves membership through `organization_membership` (696). 696 currently has only a `workos_user_id` btree, so this design **adds a composite btree `(user_id, organization_id, status)` on 696** to keep the resolver cheap.

### Realtime / presence (reuse existing infra)
Users already carry `tiptap_collab_jwt`/`tiptap_ai_jwt` — TipTap/Hocuspocus is in the stack. Realtime is split by data shape: (a) **ephemeral presence/cursors** via `df_presence` on short-TTL channels (cron sweeps rows older than ~60s); (b) **last-write-wins per-field sync** for structured board edits (stage moves, value changes) over per-resource channels with optimistic local writes; (c) **CRDT (Yjs/TipTap)** reserved for free-form rich-text deal/task notes — boards do not need full CRDT. New realtime channels follow the existing `*`/`*-members`/`*-collaborators` naming: `deal-flow`, `deal-flow-presence`, `deal-flow-collaborators`.

### Comments, followers, attachments, notifications
Threaded `df_comment` writes a `df_event` (verb=commented) and fans @mentions into the existing `activity` (218, `unread` bool) feed. `df_follower` (polymorphic) defines who is notified beyond assignee/owner; `df_attachment` (polymorphic) carries files. The notification producer is `df_event`: a defined `df_event.verb -> activity_type` map writes activity rows for assignment, owner change, stage move, mention, grant-added, and rotting; `df_reminder` handles scheduled due-date/follow-up nudges separately. Backing tables: df_grant, df_grant_effective, df_share_link, df_comment, df_follower, df_attachment, df_reminder, df_event, df_presence — all org-scoped, all reusing user/team/team_member/activity.

## Relationship-graph (FalkorDB) integration

Orbiter's edge is the relationship graph, so deals are wired into it at the data layer rather than treated as standalone CRM rows.

**Anchoring to canonical graph entities.** `df_deal.master_company_id` -> `master_company` (142) is the deal's target node (nullable to support blind/early fundraising, where the target is attached via `df_deal_association.role=target` instead). `df_deal_association` attaches `master_person` (139) and additional `master_company` stakeholders with labeled roles (champion/decision_maker/blocker/influencer/intro_path/target). Because these are the same canonical tables backing the FalkorDB nodes (org + user carry `falkor_graph_id`), a deal's company/people resolve directly to graph nodes — no shadow contact records.

**Warm-intro paths (headline feature).** `df_deal_intro_path` caches FalkorDB path-finding per deal: `target_master_person_id` (who to reach), `connector_user_id` (internal teammate who can warm-intro), `path_nodes` (ordered node_uuids), `strength_score`, `hops`, `computed_at`. This is a cache with TTL: a cron recomputes and invalidates on graph-edge change, tying into the existing edge-audit workflow. The resolver cross-checks a stale `connector_user_id` against `organization_membership` (696, status=active) so a departed teammate is not surfaced.

**strength_score has a defined source (critique fix).** It is computed from an interaction/recency feed — FalkorDB relationship-edge weights combined with the existing email/event activity already in the workspace (the same signals that drive `activity` 218: thread/message/event recency and frequency). This makes "last contacted" and interaction-recency surfacing concrete rather than a magic number, and feeds intro-path ranking.

**Stakeholder org-charts & signals.** The same graph (queried live by node_uuid) supplies the org-chart around a deal's company and any Signal nodes (funding, hiring, news) as deal context — read-only in the UI, hot results cached. No new signal table; Signals are read through FalkorDB keyed on the deal's `master_company_id`.

**Graph-aware custom fields.** A `df_custom_field_def` of type `record_ref` stores `value_ref_id`+`value_ref_table` pointing at `master_company`/`master_person`/`user`, so user-defined relationship fields are graph-aware by construction (Attio-style; multi-ref via `df_custom_field_value_option`, auto-inverse deferred).

**Naming note.** The graph already has an unrelated `project` (575) career-history node; `df_project` never touches it — deal->graph linkage is exclusively through master_company/master_person/intro_path.

## Integration with the existing schema

**Tenant scoping (verified).** `organization` (695) is the WorkOS tenant. Every new ROOT table carries `organization_id : int -> organization(695)` NOT NULL; `resolve_access()` and every endpoint filter on it. **Verified critical fact:** existing core tables (`user` 125, `team` 613, `note_collaborator` 655, `master_company` 142) carry NO organization_id — they are tenant-agnostic. The user↔org mapping therefore comes ONLY from `organization_membership` (696, role admin/member, status active/inactive/pending). 696's live indexes are PK + btree(created_at) + btree(workos_user_id); this design **adds a composite btree `(user_id, organization_id, status)` on 696** so the resolver does not table-scan membership on every call.

**Index reality (verified, drives the whole design).** Every live table in workspace 3 carries only a PK plus single-field btrees (confirmed on 655/614/696/218/125). `createBtreeIndex`/`createUniqueIndex` DO accept multi-field arrays (each field needs an explicit `op` asc/desc), so composite indexes in this design are buildable. There is **NO GIN/partial-index primitive**, and `createSearchIndex` is full-text only. Consequences folded in: (1) multi-select uses `df_custom_field_value_option` (btree join), not JSON-array+GIN; (2) `df_grant` uses three separate unique indexes, not one NULL-bearing composite unique; (3) `path` is indexed standalone for prefix scans, not as a composite with organization_id; (4) full-text `df_search` uses `createSearchIndex` (english) on `df_task.title`/`df_deal.title`.

**Reused primitives (no parallel tables):**
- **user (125)** for every actor field. Confirmed it carries `tiptap_collab_jwt`/`tiptap_ai_jwt` -> realtime reuses existing Hocuspocus infra.
- **team (613) + team_member (614)** are the "assign a project to a team" primitive: a `df_grant` row with `subject_type=team, subject_team_id->613`. Verified team_member.status enum is `[pending,accepted,rejected]` (NOT active) and team_member.position is `int`; the resolver filters `status=accepted` everywhere. Confirmed team's container shape is `user_id + title + description + icon_name(non-null '') + custom_icon_url + team_color` — `df_project`/`df_pipeline` mirror it exactly.
- **note_collaborator (655)** is the per-resource share precedent. Verified mode enum is exactly `[editor,viewer,commentor]` (default "Viewer"); `df_grant.mode` is documented as an EXTENSION (+manager), not a mirror, with an explicit `role_rank()` ordinal because Xano enums are unordered. (655.my_note_id -> table 156; our grants are polymorphic.)
- **activity (218) + activity_type (217)** are the NOTIFICATION feed (verified `unread` bool default 1). But 218 is email/thread-specific (my_person_id, interlocutor_email, message_id, thread_id, data_source_id) and is NOT a general product-event log — so field-level record history lives in the new `df_event` table, and only user-facing notifications fan into 218 via a defined `df_event.verb -> activity_type` map.

**project-575 collision resolution (explicit, verified).** `project` (575) is an UNRELATED graph artifact (career-history node: node_uuid, master_person_id, start/end year-month objects, contributors_raw). We do NOT touch it. **Convention conformance (verified):** int PK `id`; `created_at` timestamp default now; `updated_at` timestamp nullable on EVERY table including joins/logs (critique fix: df_task_assignee etc. now carry it); FKs are int with a tableref; enums for status/mode/type; snake_case; `is_*` bool flags read via `first_notnull`; LexoRank `text` position for ordering. **Cascade / orphan policy (critique fix).** Polymorphic tables (df_event, df_comment, df_attachment, df_follower, df_grant, df_grant_effective, df_presence, df_custom_field_value, df_reminder) have no DB FK to their resource. Each root resource gets a `cascade_delete(resource_type, resource_id)` fn that runs on `deleted_at` set, removing dependents; a nightly orphan-audit cron sweeps each polymorphic table for rows whose (resource_type, resource_id) no longer resolves. Soft-delete via `deleted_at` precedes hard purge by a restore window.

## Build phasing

### Phase 1 — Core projects & tasks

df_project (nesting via parent + standalone path/depth + lexorank text), df_project_section, df_task (nullable project_id = personal todo, parent_task_id subtasks, section_id, remind_at, recurrence_rule), df_task_assignee, df_status (scope=task/project with immutable category + cross-scope trigger), df_reminder, basic My-Tasks saved filter. Covers requirements 1-4. Reuses organization/user; soft-delete via deleted_at + is_archived.

### Phase 2 — Sharing, teams & multiplayer

df_grant (+ three unique indexes, subject_link_id) + resolve_access() + df_grant_effective cache, composite (user_id,organization_id,status) index on organization_membership(696), df_share_link, df_comment, df_follower, df_attachment, df_event (universal timeline), df_presence + realtime channels. Assign-project-to-team via subject_type=team (team/team_member 613/614, status=accepted). Notifications fan df_event into activity(218). Covers requirements 5 and 7. Reuses tiptap_collab_jwt.

### Phase 3 — Customizable deal pipelines (+ seed templates)

df_pipeline (multiple per org), df_status extended to scope=deal (stages with category/probability/rotting_days + invariant trigger), df_deal (derived status, value_base, lifecycle stamps), df_deal_association, df_template + system seed pipelines/status sets shipped at first-run. Covers requirement 6 and the CRM core. (Templates pulled forward from v1 Phase 6 per critique.)

### Phase 4 — Custom fields & views

df_custom_field_def (per-pipeline + per-project scoping), df_custom_field_option, df_custom_field_value (typed-EAV), df_custom_field_value_option (multi-select/multi-ref child join), formula/rollup compute layer (config.expr + refs + is_dirty + recompute fn), df_view (board/list/calendar/table/timeline). df_search full-text. Completes the 100%-customizable requirement.

### Phase 5 — Automation & forecasting

df_automation_rule + df_automation_run (stage/rotting/field/date triggers -> move_stage/assign/create_task/notify/set_field/webhook actions), df_target + weighted-pipeline forecast rollups, rotting-deal + reminder + recurring-task cron sweeps. Turns the datastore into active deal flow.

### Phase 6 — Relationship-graph context

df_deal_intro_path (cached FalkorDB warm-intro paths + strength_score from interaction/recency feed), live stakeholder org-chart + signal reads via master_company_id, graph-aware record_ref custom fields, TTL refresh cron tied to edge-audit workflow. Covers requirement 8 — Orbiter's unique edge.

### Phase 7 — Advanced / optional

df_task_membership multi-home + Gantt over df_task_dependency, multi-pipeline deal membership (df_deal_membership v2 toggle), record_ref auto-inverse, denormalized board projection cache, nested-formula topological recompute, FX provider integration for value_base.

## Open questions

- Multi-pipeline deal membership: v1 keeps a deal in exactly one pipeline (stage on df_deal). If Attio-style 'one company in many pipelines at different stages' is needed at launch, promote to df_deal_membership (deal_id, pipeline_id, status_id, board_position) and move stage off the deal — flagged as a v2 toggle.
- FX source & cadence for value_base: which provider feeds the FX table and how often is value_base recomputed (daily cron vs on-write)? Affects forecast accuracy at period boundaries.
- Formula/rollup compute budget: dirty-flag + cron recompute is specified, but deep dependency chains (rollup of a formula of a rollup) need a topological recompute order and a max-depth guard — confirm acceptable recompute latency before exposing nested formulas.
- Automation safety rails: loop prevention (rule A sets a field that triggers rule B that re-triggers A), per-org rate limits, and dry-run mode — confirm the guardrail policy before enabling webhook/set_field actions.
- df_grant_effective refresh strategy: full recompute-on-change vs incremental. A team grant on a deep project subtree fans to many (user x resource) rows; confirm whether to materialize the full subtree eagerly or compute lazily on first access per resource.
- record_ref auto-inverse: Attio auto-creates the inverse link on the referenced object. Defer to a paired-def concept in v2, or ship one-directional refs in v1 (current plan)?
- Guest/external access depth: link tokens cap at editor; do authenticated non-org guests need a first-class role distinct from link tokens, and how do they map onto organization_membership (696)?
- Intro-path compute trigger: on-demand (lazy on deal open) vs scheduled cron vs graph-edge-change event — what staleness window and FalkorDB compute budget are acceptable for strength_score recompute?
