New work enters the IMDB (Film & TV) waterfall through the two **get-add gates** — one per entity type. Each gate normalizes the `imdb_url`, dedups, ensures a stub row, then branches: `cascade_depth == 0` \+ `queue == false` invokes the orchestrator synchronously and returns `action: "enriched_direct"`; anything else upserts a queue row for the [queue crons](/guides/enrichment/waterfall/imdb-crons) to drain. Existing `master_person` rows can also enter through the central link writer or the person pipeline's Phase 8 attach/verify path. Architecture spec: [IMDB Waterfall overview](/guides/enrichment/waterfall/imdb-waterfall) · function roster: [IMDB Waterfall Functions](/guides/enrichment/waterfall/imdb-functions) · queue schemas: [IMDB Waterfall Tables](/guides/enrichment/waterfall/imdb-tables).

      <li><code>source_function</code>, <code>source_entity_id</code>, <code>source_entity_type</code>: Trace the upstream function/entity that discovered the IMDB person URL.</li>
      <li><code>seed_imdb_person_id</code>: Provenance seed for person-originated cascades; defaults to the created/found person stub when omitted.</li>
      <li><code>seed_film_tv_id</code>: Provenance seed for title-originated cascades so person queue rows can point back to the film/TV source.</li>
    </ul>
  </div>

  Gate for any IMDB person URL. **v1.3 (2026-04-22):** normalizes URL, dedups on a 30-day window, ensures stub, then either sync-enriches (depth 0 \+ `queue: false` → invokes `run-base-imdb-person-enrich` directly) or upserts the queue row. Seed-id propagation: when no caller seed is passed, the just-created stub IS the seed. **Primary entry point for new work.**

## Person-pipeline entry paths

These paths start IMDB enrichment for an existing `master_person` rather than creating a standalone IMDB-seeded person first.

<Card icon="link">
  ```text
  mvp/link/create-master-link — #2541
  ```

  <div style={{border: "2px solid #3b82f6", borderRadius: "8px", padding: "14px 16px", margin: "12px 0 16px", background: "rgba(59, 130, 246, 0.06)"}}>
    <div style={{fontSize: "0.78rem", fontWeight: 800, letterSpacing: "0.08em", textTransform: "uppercase", color: "#1d4ed8", marginBottom: "8px"}}>Live inputs</div>
    <ul style={{margin: 0, paddingLeft: "1.2rem"}}>
      <li><code>url</code>: Any URL to normalize and store in <code>master_link</code>; IMDB person URLs are detected by <code>imdb.com/name/</code>.</li>
      <li><code>master_person_id</code>: Existing person id. Required for the IMDB attach dispatch.</li>
      <li><code>source</code>: Business provenance copied to <code>master_link.source</code>, such as Enrich Layer, People Data Labs, or user input.</li>
      <li><code>suppress_downstream</code>: Still used by social/link/company fanout controls; IMDB person links intentionally dispatch the controlled IMDB attach path when a <code>master_person_id</code> exists.</li>
    </ul>
  </div>

  Central link writer. **v1.5 (2026-07-06):** when a person link is an IMDB name URL, dispatches `mvp/imdb/add-imdb-to-master-person` synchronously with the normalized URL and existing `master_person_id` so sandbox/live data-source context is preserved. The dispatch runs on new links, duplicate person links, and existing global-link returns, so partial earlier runs can be nudged by re-submitting the link. Title URLs do not use this path. **2026-07-07:** the downstream attach path now always creates/links an `imdb_person` stub before scrape; if optional Firecrawl returns no markdown, it upserts `queue_imdb_person` for retry instead of ending at a log-only skip.
</Card>
