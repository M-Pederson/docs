One function per evidence source. Each is called by the relevant Nylas / LinkedIn / CRM / phone / manual import path. Every ingest function follows the same pattern.

## Core ingest primitives (built 2026-04-10)

The three functions that touch the `:KNOWS` edge type in FalkorDB have all been built in Xano:

| Function | Xano ID | Role | Status |
|---|---|---|---|
| `known/ingest/lookup_knows` | **#12777** | MATCH-only read of an existing KNOWS edge by canonical pair; returns the edge's uuid, evidence array, confidence, and temporal fields | ✅ Built |
| `known/ingest/create_knows_edge` | **#12778** | MERGE with `ON CREATE` / `ON MATCH`; writes the initial KNOWS envelope with `timestamp()` for `valid_from` and `created_at`. Uses the canonical KNOWS writer pattern from [Schema Design → KNOWS](/guides/known/schema-design/edges#knows-the-envelope). | ✅ Built — FalkorDB-clean |
| `known/ingest/append_evidence` | **#12779** | MATCH + SET by uuid; blindly overwrites `evidence`, `confidence`, `known_since`, `last_evidence_at`, `last_direct_interaction_at` with values computed by the caller. Uses plain ISO string quoting for the temporal fields, matching the FalkorDB-compat convention. | ✅ Built — FalkorDB-clean (fixed earlier today) |

**FalkorDB compatibility:** All three use `timestamp()` for metadata fields and plain ISO strings for `valid_from`/`valid_to`/`last_evidence_at`. Neither `datetime()` nor `duration(...)` appears anywhere in their cypher — see the [FalkorDB compat note](/guides/cyphers/known#falkordb-compatibility-read-this-before-writing-cypher) on the Known cyphers page.

`known/ingest/_fanout_email_pairs` (**#12781**) and `known/ingest/email_direct` (**#12780**) are also built and wire email evidence through the four primitives above. Also built, also FalkorDB-clean.

## KNOWS uniqueness enforcement

`KNOWS` is declared as "at most one edge per canonicalized `(person_a_id, person_b_id)` pair at any time" in [schema-design → Constraints](/guides/known/schema-design#constraints). FalkorDB does **not** support unique constraints on relationships, so this invariant is enforced at the application layer in three layers of defence:

<Steps>
  <Step title="Canonicalization at the application layer">
    Every ingest path calls `canonicalize_pair` (**#12767**) before any graph write. The lower-id person becomes `person_a_id`, the higher-id person becomes `person_b_id`. This eliminates the `(a, b)` vs `(b, a)` race — two concurrent writes for the same pair of humans will land on the same canonical direction every time.
  </Step>
  <Step title="Directed MERGE on the canonical pair">
    `create_knows_edge` (#12778) uses `MERGE (a)-[k:KNOWS]->(b)` — **directed**, from the canonical low-id to the canonical high-id. Directed MERGE is used because FalkorDB does not support undirected MERGE; at read time, `lookup_knows` and every query pattern treat `KNOWS` as undirected via `-[k:KNOWS]-` (no arrow). On repeat writes for an existing pair, the `ON MATCH` branch runs and updates `evidence`, `confidence`, `known_since`, and `last_evidence_at` — it never creates a duplicate.
  </Step>
  <Step title="Dedupe cron as safety net">
    `MERGE` is atomic within a single cypher query but does **not** guarantee serializable uniqueness across concurrent queries in FalkorDB. Two racing `create_knows_edge` calls can both evaluate the pattern, both see nothing, and both proceed to create. The race window is the cypher execution time (single-digit milliseconds) — narrow but real. As a safety net, [`known/lifecycle/dedupe_knows_edges`](/guides/known/scope/lifecycle#known-lifecycle-dedupe-knows-edges-cron) runs nightly and collapses any pair with more than one live `KNOWS` edge into the oldest surviving row, absorbing the duplicates' evidence arrays via `append_evidence`.
  </Step>
</Steps>

**Why not a Xano mutex table?** A per-pair lock table would close the race window completely at the cost of slowing every write — a table INSERT + DELETE around every ingest call. Given Known's current concurrency profile (sequential ingest crons on a fresh graph, not high-volume live webhooks), the MERGE + dedupe-cron combination is the right tradeoff. If we start seeing duplicates in the cron's daily scan, the mutex table is a clean upgrade path.

**What callers MUST do:**
- Call `canonicalize_pair` before `create_knows_edge`. Do not pass raw uuids.
- Treat `{knows_uuid: null}` from `create_knows_edge` as a soft-skip — it means one or both Person nodes were missing from the graph when the MATCH ran. Log and move on.
- Never emit raw `CREATE (a)-[:KNOWS]->(b)` cypher. The three primitives in the table above are the only write path.

## Shared ingest pattern

Every ingest function does these six steps, in order:

<Steps>
  <Step title="Normalize and canonicalize the pair">
    Resolve identities via the person lookup pipeline, then put the two UUIDs in canonical order (lower id first) using `canonicalize_pair`.
  </Step>
  <Step title="Build the evidence row">
    Construct the evidence row with correct `source`, `weight` (strong = 0.5, weak = 0.25), `first_seen_at` (now), `evidence_occurred_at` (the actual domain time the event happened), and any source-specific `raw_metadata`.
  </Step>
  <Step title="Look up existing KNOWS">
    If a `KNOWS` edge already exists between the pair, call `append_evidence` — **then also check staging** (see next step).
  </Step>
  <Step title="Look up staged evidence">
    Call `lookup_pending_for_pair` regardless of whether a `KNOWS` already exists. If staging has rows, combine them with the new evidence.
  </Step>
  <Step title="Evaluate floor and create or append">
    If no existing KNOWS: evaluate the combined (new + staged) evidence against the floor. If it passes, call `create_knows_edge` with the full combined evidence and then `consume_pending_for_pair` to delete the staged rows. If it doesn't pass, call `insert_pending_evidence` to stage the new row.
  </Step>
  <Step title="Update introducer metadata (if applicable)">
    If this ingest path detected an introducer (e.g. email cc pattern), also call the Introduction node creation function. See [Introductions](/guides/known/scope/introductions).
  </Step>
</Steps>

## Strong-source ingests

Strong sources have weight **0.5** and alone clear the evidence floor.

| Function | Called by |
|---|---|
| `known/ingest/email_direct` | Nylas `message-created` parsing pipeline |
| `known/ingest/calendar_meeting` | Nylas `event-created` pipeline |
| `known/ingest/phone_contact` | Phone contacts import |
| `known/ingest/crm_contact` | CRM connector imports |
| `known/ingest/manual_user_add` | User frontend action |
| `known/ingest/sms_chat` | iMessage / SMS import |

## Weak-source ingests

Weak sources have weight **0.25** and require at least one additional source (strong or weak) to clear the floor. On arrival alone, they go to staging.

| Function | Called by |
|---|---|
| `known/ingest/linkedin_connection` | LinkedIn sync |
| `known/ingest/twitter_follow` | Twitter/X sync |
| `known/ingest/instagram_follow` | Instagram sync |
| `known/ingest/shared_event_attendance` | Event participant overlap detector |
| `known/ingest/shared_employment_inferred` | `WORKS_AT` overlap detector cron |
| `known/ingest/mutual_friend_mention` | Content parsing pipelines |
| `known/ingest/press_co_mention` | Press/news enrichment pipelines |

## Key rules for ingest authors

- **Always canonicalize before any graph write.** Use `canonicalize_pair`.
- **Always set `evidence_occurred_at`** from the source's actual event time (LinkedIn connection date, email timestamp, calendar event time, etc.) — not from ingest time. This is what enables retroactive `known_since` backdating.
- **Always check staging** before creating or updating `KNOWS`, even if the new evidence alone clears the floor.
- **Do not skip the floor check** for "obvious" strong sources. The floor is a uniform invariant.
- **Do not create facets from ingest functions.** Facets (BONDED, PROFESSIONAL_TIE, etc.) are created by the [lifecycle crons](/guides/known/scope/lifecycle), not by ingest paths.
