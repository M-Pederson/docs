## Goal

When Orbiter Universe finishes enriching a person or company, Orbiter Backend needs enough data to render:

- fast cards
- contact side panels
- company side panels
- search, filters, and drill-down sections

The expensive part is not the network call. The expensive part is coordinating many downstream tables without turning Universe into a second owner of the App backend schema.

The clean pattern is:

```text
Backend seed created
  -> fast UI response with App person/company id
  -> Universe enrichment runs asynchronously
  -> Universe enrichment complete
  -> one versioned projection payload
  -> Backend projection applier
  -> card display rows + normalized side-panel display facts
```

Universe sends one package. Backend owns the process that applies it.

## Seeded Fast Response

When the product needs a new person or company, Orbiter Backend should not wait for full enrichment before responding to the UI.

Instead, Backend creates a minimal App-owned seed record immediately:

- `master_person` or `master_company`
- `person_display` or `company_display`
- a projection/enrichment status row, if needed for tracking

That seed row only needs enough information to make the product usable right away. For a person, that can be name, LinkedIn URL, email, company hint, or avatar if already known. For a company, that can be name, domain, website, logo, or LinkedIn URL.

The first response back to the UI should include the Backend identity and a pending display row:

```json
{
  "entity_type": "person",
  "master_person_id": 12345,
  "display": {
    "display_name": "Ava Chen",
    "headline": "Enrichment in progress",
    "avatar_url": null,
    "primary_company_name": "Northstar AI",
    "enrichment_status": "enriching"
  }
}
```

For a company, the same fast response uses the company identity and the best known domain or website:

```json
{
  "entity_type": "company",
  "master_company_id": 67890,
  "display": {
    "display_name": "Northstar AI",
    "domain": "northstar.ai",
    "logo_url": null,
    "headline": "Enrichment in progress",
    "enrichment_status": "enriching"
  }
}
```

The card can render immediately from that seeded display row. The side panel can open against the same `master_person_id` or `master_company_id`, with sections showing seeded values, loading states, or empty states while Universe works.

After the response, Backend starts the Universe enrichment job asynchronously and passes the Backend identity with the seed input:

```text
start_universe_person_enrichment({
  master_person_id,
  seed_name,
  seed_linkedin_url,
  seed_company_hint
})
```

For companies, the same pattern uses `master_company_id`, domain, website, LinkedIn URL, and any known company label.

When Universe finishes, it sends one projection package back to Backend. The projection package includes the same Backend identity, Universe identity, run id, card display data, and side-panel facts. Backend applies it idempotently to the existing rows:

```text
seeded card
  -> enriching card
  -> completed card + populated side panel
```

The UI keeps reading from Backend-owned IDs the whole time. It does not need to switch records after enrichment completes.

## Display And Side-Panel Models

Cards and side panels should not be forced into the same storage shape.

Cards need compact, display-ready data. They should read one row whenever possible. That same row can also feed the side-panel header.

Side panels are composed views. They need normalized rows where the UI supports sections, filtering, history, drill-down, and source-specific display. Some of those rows are Universe-projected public facts. Some are App-owned dynamic rows scoped to a user, team, or organization: files, notes, collections, private relationship context, reminders, permissions, and workflow state.

So Backend has three related read surfaces:

| Target | Purpose | Shape |
| --- | --- | --- |
| Entity display table | Fast card rendering and side-panel headers | One compact display row per display type |
| Projected side-panel facts | Public/enrichment-backed structured sections | Multiple normalized rows, upserted or replaced by Universe-owned section |
| User/org dynamic side-panel data | Private or workflow-specific sections | App-owned rows joined at read time, never overwritten by Universe projection |
| Projection log/status | Replay and debugging | One row per projection run |

## Card Types

A card type is the Backend read contract the UI can render from one compact row.

### Combined Display Tables

The physical table naming should make the read-model role obvious. Prefer `*_display` names for greenfield App tables; shorter `app.*_card` names are also acceptable when a table is truly card-only. The important rule is that these are product display models, not Universe source tables.

`person_display` and `company_display` are broader display tables: they should hold the compact fields needed for cards, search result rows, side-panel headers, and side-panel summary sections. The other display tables below are narrower card display tables by default: they should hold only the fields needed to render cards unless the product later adds first-class detail panels for that type.

| Physical App display table | Label(s) | Ideal card types it can cover | Discriminator fields | Notes |
| --- | --- | --- | --- | --- |
| `person_display` | `Person`, `Person:Angel` | Person | `person_labels`, `is_angel` | Broader display table for cards, search rows, side-panel headers, and compact side-panel summary data. |
| `company_display` | `Company`, `School`, `Company:VC_Firm`, `Organization:Company`, `Company:Music_Label` | Company | `company_type`, `company_labels`, `is_school`, `is_vc_firm`, `is_music_label` | Broader display table for cards, search rows, side-panel headers, and compact side-panel summary data. School / VC / Organization / Music label stay company-shaped unless the UI later needs a different layout. |
| `work_product_display` | `Project`, `Work:Publication`, `Work`, `Work:Song`, `Work:Symphony` | Work | `work_product_type`, `work_type`, `music_genres` | Card-only by default. Covers projects, publications, and music/composition works under one Work card contract. |
| `funding_round_display` | `Funding_Round` | Funding round | `funding_type` | Card-only by default: round amount, announced date, funded company, investors, and source freshness. |
| `live_event_display` | `Event:Live_Summit`, `Event:Live_Conference`, `Event:Live_Festival`, `Event:Live_Trade_Show`, `Event:Live_Workshop`, `Event:Live_Webinar`, `Event:Live_Hackathon`, `Event:Live_Meetup`, `Event:Live_Networking_Event`, `Event:Music_Festival`, `Event:Music_Concert`, `Event:Music_Convention`, `Event:Music_Award_Ceremony` | Event | `event_type`, `series_slug`, `edition_year` | Card-only by default for `Event:Live_*` and `Event:Music_*` labels. |
| `film_tv_display` | `Film_TV:Movie`, `Film_TV:TV_Series`, `Film_TV:TV_Movie`, `Film_TV:TV_Mini_Series`, `Film_TV:Short_Film`, `Film_TV:Documentary`, `Film_TV:TV_Special`, `Film_TV:Video_Game`, `Film_TV:Music_Video` | Film/TV title | `title_type` | Card-only by default. | `film_tv_award_display` | `Film_TV_Award:Winner`, `Film_TV_Award:Nominee` | Film/TV award | `award_status` | Card-only award/badge display contract for wins, nominations, title-linked awards, and honorary person awards. |
| `music_artist_display` | `Person:Music_Artist` | Music artist | `artist_type`, `music_genres` | Card-only by default. Distinct from the generic Person card because the music UI emphasizes artist facts, roles, genres, media, and MusicBrainz provenance. |
| `music_group_display` | `Music_Group` | Music group | `group_type`, `music_genres` | Card-only by default for bands, orchestras, choirs, and other group artists. |
| `recording_display` | `Recording` | Recording | `recording_type`, `music_genres` | Card-only by default for track/recording cards. |

Split a combined table only when one of these becomes true: the card needs materially different required fields, permissions, moderation rules, freshness rules, indexing, or side-panel section payloads.

## Person Flow

For a person, Universe publishes one package after the enrichment run is complete:

```json
{
  "schema_version": "person_projection.v1",
  "run_id": "enrich_run_01H...",
  "universe_person_uuid": "up_01H...",
  "source_updated_at": "2026-06-30T19:00:00Z",
  "display": {
    "display_name": "Ava Chen",
    "headline": "Founder and CTO at Northstar AI",
    "avatar_url": "https://...",
    "primary_company_name": "Northstar AI",
    "primary_company_domain": "northstar.ai",
    "location_label": "San Francisco, CA",
    "summary": "Founder building AI workflow infrastructure for enterprise teams.",
    "top_skills": ["AI strategy", "workflow automation", "enterprise SaaS"],
    "top_links": [
      {"type": "linkedin", "url": "https://www.linkedin.com/in/..."},
      {"type": "website", "url": "https://..."}
    ],
    "enrichment_status": "complete"
  },
  "side_panel_facts": {
    "roles": [
      {
        "company_name": "Northstar AI",
        "company_domain": "northstar.ai",
        "title": "Founder and CTO",
        "is_current": true,
        "start_date": "2022-01-01"
      }
    ],
    "skills": [
      {"name": "AI strategy", "confidence": 0.94, "source": "Universe"},
      {"name": "enterprise SaaS", "confidence": 0.88, "source": "Universe"}
    ],
    "work_experience": [],
    "links": [],
    "emails": [],
    "phones": [],
    "primary_location": {
      "city": "San Francisco",
      "region": "California",
      "country": "United States",
      "formatted_location": "San Francisco, CA"
    }
  }
}
```

Orbiter Backend receives that package and runs one internal applier:

```text
apply_person_projection(payload)
  1. validate schema_version
  2. resolve or create App person identity
  3. upsert person_display
  4. replace current Universe-owned role/skill/location sections
  5. upsert contact methods and links where allowed
  6. record projection run status
```

## Side-Panel Data

Side-panel data should stay normalized when structure matters, but not every side-panel section comes from Universe.

Universe projection can update public/enrichment-backed person sections:

- `role` / `role_join`
- `skills`
- `work_experience`
- `primary_location`
- `master_link`
- `master_email`
- `master_phone`
- expertise/display sections that need drill-down

Universe projection can update public/enrichment-backed company sections:

- `company_financial`
- `primary_location`
- `master_link`
- company social/profile links
- founders and leadership sections
- funding/acquisition sections
- display facts needed by the company detail endpoint

The App backend also owns dynamic side-panel sections that are user, team, organization, or workflow scoped:

- files and file-derived artifacts attached to the entity
- notes, comments, and private annotations
- collections, lists, favorites, and saved views
- user-specific relationship context and contact state
- reminders, tasks, follow-ups, and activity history
- permissions, visibility, and sharing state
- product workflow state such as open suggestions, drafts, or pending actions

Those dynamic sections are composed into the side panel at read time. Universe projection should never replace them. The key is ownership: Universe owns public enrichment results; Backend owns product display tables, projected public facts, private user/org data, and the final side-panel composition.

## Replace vs Merge

The applier should use section-level rules instead of treating every table the same.

| Section | Suggested write mode | Reason |
| --- | --- | --- |
| Entity display table | Upsert one row | Cards and side-panel headers need one current display state |
| Current roles | Replace Universe-owned current set | Avoid stale titles |
| Historical work experience | Merge by source key/date/company/title | Preserve history |
| Skills | Replace or score-merge Universe-owned set | Avoid duplicate skills |
| Links | Upsert by normalized URL/type | Avoid duplicate URLs |
| Emails/phones | Conservative upsert | Contact data has product/privacy implications |
| Financial/company facts | Upsert by company and source | Preserve freshness |
| User/org dynamic side-panel data | Do not touch from Universe projection | Files, notes, permissions, collections, and workflow state are App-owned |
| Projection status | Append per run | Needed for replay/debug |

## Idempotency

Every projection package should carry:

- `schema_version`
- `run_id`
- `universe_entity_uuid`
- `source_updated_at`
- `projection_generated_at`

The Backend applier should be safe to run more than once for the same `run_id`. That makes retries, replays, and backfills much safer.

## Resulting Architecture

```text
Orbiter Universe
  owns enrichment, source payloads, queues, and graph writes

Projection package
  one versioned request per completed entity enrichment

Orbiter Backend projection applier
  validates, resolves identity, applies table writes, records status

Orbiter Backend read models
  display tables for fast cards and side-panel headers
  projected public facts for contact and company side panels
  user/org dynamic rows composed into side panels at read time
```

This keeps the Universe backend independent while giving the product backend a stable, efficient way to hydrate display rows and public side-panel facts from enrichment, without overwriting private or workflow-specific side-panel data.
