The [Person Enrichment Waterfall](/guides/enrichment/waterfall/person-waterfall) uses the canonical graph node definitions in [Ontology → Node Types](/guides/ontology/nodes). People land on the shared `Person` node with dynamic `:Angel` / `:C_Suite` modifier labels; the phase-7 edge resolvers mint `Certification`, `Honor`, `Project`, and `Work:Publication` nodes plus `Organization`-shaped companies; phase 2 fans out to the shared `City` / `Region` / `Country` chain; phase 8 mints `SubDomainExpertise` taxonomy nodes and bridges film people into the IMDB waterfall; the opt-in phase-9 deal cascade materializes `Funding_Round`. Company nodes belong to the company pipeline — since the 2026-06-23 containment sweep, person runs resolve them lookup-/local-only and never enqueue company enrichment.

This page is the **pipeline-scoped catalog**: which node types this pipeline writes, keyed how, by which function. Node labels, properties, vector fields, and examples should be edited only in the ontology source of truth.

<Note>
  Ground truth: [Node Types](/guides/ontology/nodes). Write-side Cypher for the person nodes lives on [Cyphers → Person Pipeline](/guides/cyphers/person-pipeline). Function versions cited below are the live workspace-3 bodies as of the 2026-07-02 audit — writer internals are on [Person Waterfall Functions](/guides/enrichment/waterfall/person-functions).
</Note>

## People

### Person

Canonical section: [Person](/guides/ontology/nodes#person).

**Create** — `add-person-node` #2601 (v2.1, 2026-06-25), invoked from the entry gate `master-person-new` #13039 right after the `master_person` insert (also by the Exa C-suite helper #12997 and the IMDB person path). `MERGE` keyed on `master_person_id` — **idempotent**, unlike the company pipeline's raw-CREATE #2602. `ON CREATE` sets a random `uuid`, `display_label: "Person"`, and `visibility: false`; the `:Entity` base label is added on every run. 3-retry Falkor loop; v2.1 returns a safe `$nodeUuid` so a Falkor timeout can't throw; guarantees an idempotent `person_enrich_data` shell. Avatar fallback: Serper image search (`find-avatar` #12846) → `create-placeholder-avatar` #1832 (v1.3 — UI Avatars initials, dedup by `original_source_url`), with the Serper `replace-avatar` step **best-effort** since v2.0 (2026-06-22, try/catch non-blocking). Visibility flips to `true` at phase 11 via `complete-person-enrich` #12590 — fresh Exa C-suite leaves deliberately start `visibility: false` (#12997 v1.7).

**Sync** — `update-person-node` #2758 (v1.5, 2026-06-15): `MATCH (n:Entity {uuid: node_uuid}) SET …` property \+ label sync including bios and investor-profile fields; forces `master_person.visibility = true` and mirrors it onto the node. Three callers: `complete-person-enrich` #12590 v1.3 (phase 11), the [deep-research webhook fan-out](/guides/enrichment/waterfall/deep-research-bio) (#8303), and the #13040 v3.23 **post-thesis best-effort retry** (2026-06-30). Since 2026-06-09 embedding text falls back `bio_500` → `bio` → `name`, so Exa C-suite people get searchable nodes before a long bio exists. **One-line Cypher-safe bio projection:** node string properties (including `short_biography` / `long_biography`) get control characters/newlines stripped and whitespace collapsed; the Xano `bio` / `bio_500` keep their line breaks.

**Dual vectors** — both writers dual-write `name_embedding` (OpenAI, 1536) **and** `embeddings` (Gemini, 3072, via helper #13125) since 2026-06-15 (#2601 gates the Gemini line in the builder lambda, #2758 v1.5 in the Twig template — a Gemini miss still creates/syncs the node with `name_embedding` only).

### Angel \+ C\_Suite (dynamic labels)

Canonical section: [Angel](/guides/ontology/nodes#angel). Label-only modifiers — no extra properties. #2758 builds the label list in a lambda from the `is_angel` / `is_c_suite` bools on `master_person`: always `:Person`, plus `:Angel` and/or `:C_Suite`. `is_angel` is set by phase 8 (#12829 v1.1) from the **legacy `expertise_join` table, domain 22** — not yet migrated to `person_has_expertise` #711; `is_c_suite` is set by `master-person-from-exa` #12997 on Exa C-suite/founder discovery.

**As-built (flagged):** the `:C_Suite` label is live in #2758's label lambda and documented on [Cyphers → Person Pipeline](/guides/cyphers/person-pipeline), but **missing from the ontology [master label list](/guides/ontology/nodes#master-label-list)** — known ontology drift; fix it in the ontology source of truth rather than propagating the gap.

## Organizations

Person enrichment **links** organization-shaped nodes but — since the 2026-06-23 containment sweep — never enqueues company enrichment: phase 3 (#12824 v3.4) resolves the primary employer local-only (`hold_downstream=true`); work-history companies stopped queueing (#12562 v1.7, #13031 v1.5); schools (#12560 v1.2), publishers (#12579 v5), and honor/cert issuers (#4574 v4 / #12578 v2.5) are local-only depth-1 leaves. Since `master-company-new`'s 2026-06-25 change, `resolve-edges-work` leaves are **lookup-only** — work edges link existing companies but never mint new ones. Full Company node lifecycle: [Company Nodes](/guides/enrichment/waterfall/company-nodes).

### Company (stub)

Canonical section: [Company](/guides/ontology/nodes#company). Resolution goes through `get-add/master-company-new`, which delegates real node creation to the company writer #2602 — the person pipeline never writes a full Company node. The work-edge writer `create-work-edges` #2800 (v1.8, 2026-06-25) additionally stub-`MERGE`s both endpoints — `MERGE (c:Company {uuid})` with a name-backfill `CASE` and coalesce-stable timestamps (patched 2026-06-01) — so edges never dangle. The stub carries **no vectors and no profile fields**.

### School

Canonical section: [School](/guides/ontology/nodes#school). Schools from `education_experience` resolve via `resolve-edges-education` #12560 (v1.2, 2026-06-23; LinkedIn URL → domain → name) through `master-company-new`; the `:School` create-time label is applied by #2602 (`.edu` domain or `is_school`). Local-only since v1.2; depth-1 leaves since v1.1.

### VC\_Firm

Canonical section: [VC\_Firm](/guides/ontology/nodes#vc_firm). Additive label only: `resolve-investors-edges` #12702 (v1.13, 2026-07-02) runs `MATCH (vc:Company) SET vc:VC_Firm, vc.updated_at = timestamp()` during the opt-in phase-9 deal cascade — **live again since the 2026-07-02 v1.12 depth-guard repair** (v1.11 had self-skipped every invocation since 2026-06-23).

### Organization

Canonical section: [Organization](/guides/ontology/nodes#organization). Two shapes: `resolve-edges-volunteering` #4575 (v4, 2026-06-04) standalone-`MERGE`s `(org:Organization:Company {name})` — keyed on name, created fresh when no `master_company` matched (`createdAt` → `created_at` rename \+ `ON MATCH SET`, 2026-06-01) — while `resolve-edges-certifications` #12578 adds the label to an **existing** Company node (`SET organization:Organization` \+ `updated_at`).

## Locations

Phase 2 (#12823, unversioned) writes the primary location only when `master_person.primary_location_id` is null, cascading Enrich Layer `location_str` → LinkedIn `location` → PDL `location_name`, then geocoding via Radar in `add-person-primary-location` #2766 (headerless since 2026-03-19; **fail-open** — failures land in `log_crash`) before handing the structured result to the person-side writer `primary-location-nodes-edges_v2` #4690 (four Cypher branches: city\+region\+country / city\+country / country-only / region\+country). The `LOCATED_IN` / `IN_REGION` / `IN_COUNTRY` edges are on [Person Edges](/guides/enrichment/waterfall/person-edges).

### Country

Canonical section: [Country](/guides/ontology/nodes#country). MERGE-keyed on `{name: country_name}` from the relational `country` row; profile fields written `ON CREATE` only. The Country branch's timestamps \+ `ON MATCH SET` were fixed 2026-06-01. Canonical complete Country creator: `add-all-country-nodes` #12583; company sibling writer: #4528.

### Region

Canonical section: [Region](/guides/ontology/nodes#region). MERGE-keyed on `region_key = "<region>::<country>"` (lowercased). The person writer always labels `:Entity:Region` with `display_label` `"State"` or `"Region"` — contrast the company writer #4528, which mints a dynamic `:State` label for US rows. Carries an LLM-generated `summary`; `name_embedding` refreshed on match.

### City

Canonical section: [City](/guides/ontology/nodes#city). MERGE-keyed on `city_key = "<normalized city>::<region>::<country>"` (lowercased); the raw input city stays the node `name`, and `ON MATCH` refreshes name, summary, and `name_embedding`.

## Expertise

### SubDomainExpertise

Canonical section: [SubDomainExpertise](/guides/ontology/nodes#subdomainexpertise). Phase 8 (#12829 v1.1) calls `llm-identify-person-expertise` #12666 (`qwen/qwen3.7-plus`, reasoning disabled), which clears the person's existing `HAS_EXPERTISE` edges and hands the signals to `resolve-person-expertise-v2` #12926 (**v2.4, 2026-07-01; chat model moved to Qwen on 2026-07-03**): Gemini `expert_embeddings` KNN against the expertise taxonomy with a **3-way guard band** — distance \< 0.15 auto-match, 0.15–0.25 Qwen LLM adjudicate, \> 0.25 create. Only a guard-band miss `CREATE`s a new `Entity:SubDomainExpertise` (`source: "llm_identified"`, dual vectors `name_embedding` \+ `expert_embeddings`). No MERGE key — **dedup is the matcher**, with the v2.4 exact-name guard as backstop; v2.4 also verifies the CREATE (`create_failed`, no dangling rows), stamps `created_at`, and dual-writes the relational `domain_expertise_id` FK. Every verified create runs `attach-subdomain-parent` #13142 → `resolve-parent-domain` #13141 (v1.4, 2026-07-02 — LLM-picks a curated parent, **never auto-creates a domain**; a NONE verdict flags the node `parent_unresolved`) and MERGEs `HAS_SUBDOMAIN {weight: 100}` — deliberate hub damping. Matches and creates alike dual-write `person_has_expertise` #711 via #12925 v1.2; the deep-research webhook (#8303 v1.1) safely re-runs the whole chain. `HAS_EXPERTISE` weights and properties are on [Person Edges](/guides/enrichment/waterfall/person-edges) — linked, not duplicated.

### DomainExpertise

Canonical section: [DomainExpertise](/guides/ontology/nodes#domainexpertise). **Never created by the person pipeline** — the curated domain roster is synced from the reference tables by `sync-domain-expertise-nodes` #12663; person enrichment touches domains only through the #13142 parent attach.

## Credentials & achievements

All four types are minted by the phase-7 edge resolvers (#12828 v1.2, 2026-06-25 — phase 7 is **skipped entirely** for `current_company_only` people); their edges (`HAS_CERTIFICATION` / `GRANTED_BY` / `CERTIFIED_BY`, `RECEIVED_HONOR` / `ISSUED_BY`, `CONTRIBUTED_TO`, `AUTHORED` / `PUBLISHED_BY`) are on [Person Edges](/guides/enrichment/waterfall/person-edges).

### Certification

Canonical section: [Certification](/guides/ontology/nodes#certification). `MERGE (certification:Entity:Certification {name})` — keyed on the certification name, `issue_date` / `description` / `issued_by` written when present, carries `name_embedding`. Written by `resolve-edges-certifications` #12578 (v2.5, 2026-06-23): LLM cert descriptions run on `openai/gpt-oss-120b` (v2.2) with OpenRouter web search for issuer domains (v2.3); v2.5 scopes every query to the input `master_person_id`, and issuers are always `cascade_depth=1` leaves (v2.4). Timestamp coalesce fixed 2026-06-01 (legacy mirror #4572 patched too). _(A headerless edit landed 2026-06-30 — the Cypher on the cyphers page may lag it.)_

### Honor

Canonical section: [Honor](/guides/ontology/nodes#honor). Raw `CREATE (h:Honor {…})` — a **legacy non-`Entity` label, not MERGE-keyed** (uniqueness is caller-side), no vector. Written by `resolve-edges-honor` #4574 (v4, 2026-06-23): v4 scoped issuer-resolution queries to the input `master_person_id`; issuers are depth-1 leaves (v3). Timestamps added 2026-06-01.

### Project

Canonical section: [Project](/guides/ontology/nodes#project). Raw `CREATE (proj:Project {…})` — legacy non-`Entity`, no vector. Re-run protection lives in the relational row, not the graph: `resolve-edges-projects-publications` #12579 (v5, 2026-06-23) processes only rows whose `node_uuid` is still null. Contributor resolution is seed-person-only since v4. Sibling creators: `add-projects-publications-clado` #12576 / `-pdl` #4576. _(A headerless edit landed 2026-07-01 — same caveat as #12578.)_

### Work:Publication

Canonical section: [Work:Publication](/guides/ontology/nodes#publication). `CREATE (pub:Entity:Work:Publication {…})` — **relabeled from bare `Publication` to the unified `Work` parent on 2026-07-01**. CREATE-only with a Project-style row guard, but keyed on `edge_uuid` — #12579 processes only publication rows whose `edge_uuid` is still null; no vector. Publisher companies are leaf/local-only since v5 (`queue=false` \+ `hold_downstream=true`).

## Funding

### Funding\_Round

Canonical section: [Funding\_Round](/guides/ontology/nodes#funding_round). MERGE-keyed on `uuid = fundable_deals.id` — the Fundable deal id is the node uuid, a natural key. Written by `add-fundable-deal-node` #12701 per deal from the phase-9 deal cascade (`cascade-deal-participants` #12856 v1.6, 2026-07-02). On the person side the fanout is **explicit opt-in**: `allow_investor_network_cascade=true` **and** `cascade_depth=0` (#12830 v3.3) — depth 0 alone no longer cascades. #13040 v3.23 invokes phase 9 twice (early `build_investment_thesis=false`, late `=true`); the MERGE keeps the node write idempotent, and #12701's 2026-07-02 date hardening floors the ms→s division so deal dates with stray milliseconds no longer crash the Twig `date()` render. Carries `name_embedding` only — **no** Gemini `embeddings` on this type. The same writer serves the company waterfall ([Company Nodes](/guides/enrichment/waterfall/company-nodes)); the `RAISED` and investor edges are on [Person Edges](/guides/enrichment/waterfall/person-edges).

## Film & TV

### Film\_TV \+ Film\_TV\_Award

Canonical sections: [Film\_TV](/guides/ontology/nodes#film_tv) and [Film\_TV\_Award](/guides/ontology/nodes#film_tv_award). The person pipeline never writes either type directly. Phase 8, on `film_industry=true`, verifies the person's IMDb URL via `verify-imdb-url` #12677 (`google/gemini-3-flash-preview`, temp 0.1, Tavily candidates), stores it through `create-master-link` #2541, and links the row via `add-imdb-to-master-person` — from there the IMDB waterfall's writers own the nodes: `add-film-tv-node` #2320 (MERGE on `imdb_url`) and `add-node-edge-imdb-award` #2334 (`:Winner` / `:Nominee` subtype labels). Linked, not duplicated: [IMDB Nodes](/guides/enrichment/waterfall/imdb-nodes).

## Timestamp coverage

**Graph node timestamp coverage (audited 2026-06-01).** Every node create/update reachable from the person waterfall stamps `created_at` on create and `updated_at` on update — verified across `add-person-node` #2601, the phase-7 `resolve-edges-*` writers, phase-8 expertise \+ IMDB, phase-9 funding, the phase-2 location chain, and the `create-work-edges` #2800 stub-`MERGE` (all patched 2026-06-01). The orchestration functions — `master-person-new` #13039, `enrich-master-person` #13040, `cascade-person-data` #13037, `match-master-person` #13038, `process-person-exa-profile` #13031 — perform **no** graph-node Cypher; they delegate every node write to the leaf writers above. Full per-function record: [Node Types → Timestamp coverage audit](/guides/ontology/nodes#timestamp-coverage-audit).
