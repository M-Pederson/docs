This page tracks the current Mem0-related Xano functions. It is a live implementation reference, not a future-state scope doc.

<Info>
  **Current split documented:** `mem0/create-master-person-memories` (`#13148`), `mem0/create-memories-from-contact-id-v2` (`#13149`), `mem0/get-contacts-memories` (`#13147`), QA helper `mem0/delete-master-person-memories` (`#13150`), QA helper `mem0/delete-contact-memories` (`#13151`), and QA helper `mem0/delete-all-memories` (`#13152`), created in Xano workspace `3`.
</Info>

## Current Scope Split

Orbiter now separates Mem0 memory work into three core functions plus QA cleanup helpers:

| Role | Function | Xano ID | Primary input | Mem0 scope |
| --- | --- | --- | --- | --- |
| Shared master-person writer | `mem0/create-master-person-memories` | `13148` | `master_person_id` | `user_id = "master_person:{master_person_id}"` |
| Private contact writer | `mem0/create-memories-from-contact-id-v2` | `13149` | `contacts_id` | mailbox owner's private `user.workos_id` |
| Combined contact read path | `mem0/get-contacts-memories` | `13147` | `contacts_id` | private contact memories plus shared master-person memories |
| QA master-person cleanup | `mem0/delete-master-person-memories` | `13150` | `master_person_id` | deletes only `user_id = "master_person:{master_person_id}"` |
| QA private-contact cleanup | `mem0/delete-contact-memories` | `13151` | `contacts_id` | deletes mailbox-owner private memories under `user.workos_id` |
| QA global Mem0 cleanup | `mem0/delete-all-memories` | `13152` | none | deletes every Mem0 memory visible to the API key |

The intended shape is:

1. One function creates shared memories for `master_person`.
2. One function creates private email and calendar memories for `contacts_id`; those memories stay in the mailbox owner's Mem0 scope but include `master_person_id` metadata.
3. One function takes `contacts_id`, resolves the mailbox owner and linked `master_person`, and returns all memories for the person as one ordered array.
4. One QA helper clears shared `master_person` memories so tests can be rerun from a clean Mem0 scope.
5. One QA helper clears private contact memories, optionally filtered to stale source types such as `graph_edge` / `falkordb`.
6. One QA-only global helper can clear every Mem0 memory visible to the project API key.

<Note>
  `mem0/create-memories-from-contact-id` (`#13146`) remains the source function used for the split. It still contains the old inline FalkorDB graph-edge stage. `#13149` is the new contact-scoped version with that master-person Cypher stage removed.
</Note>

## `mem0/create-master-person-memories` (`#13148`)

| Field | Value |
| --- | --- |
| Xano function name | `mem0/create-master-person-memories` |
| Xano ID | `13148` |
| Workspace | `3` |
| Primary input | `master_person_id` |
| Default mode | `dry_run=true` |
| Mem0 write mode | `dry_run=false`, requires `$env.MEM0_API_KEY` |
| Memory target | Shared synthetic Mem0 user: `master_person:{master_person_id}` |
| Memory scope | `global_master_person` |
| Source | One-hop FalkorDB graph edges from `master_person.node_uuid` |

### What It Does

This function creates shared objective memories for one `master_person`. It extracts the FalkorDB graph-edge portion from `#13146` and writes those graph facts once to the synthetic master-person Mem0 scope.

It does not use `contacts_id`, `user.workos_id`, Nylas email, calendar, or email tone. The resulting memories are intended to be visible to any user viewing a contact linked to the same `master_person`.

### Inputs

| Input | Default | Purpose |
| --- | --- | --- |
| `master_person_id` | required | The shared `master_person` row to sync. |
| `dry_run` | `true` | Returns candidate memories without writing to Mem0. |
| `max_graph_edge_memories` | `500` | Max one-hop FalkorDB edge descriptions to read. |

### Step-by-step

1. Load `master_person` by `master_person_id`.
2. Read `master_person.node_uuid`; if missing, return a skipped `graph_edge_stage`.
3. Run the same one-hop Cypher pattern used in `#13146` through `mvp/falkor/send-cypher`.
4. Keep only relationships with `rel.description` or `rel.fact`.
5. Convert each edge description into one memory candidate.
6. Scrub large arrays, embeddings, and vectors from edge metadata.
7. Build deterministic hashes:
   - `canonical_memory_hash` from graph root, edge identity, from/to node identity, edge type, and description.
   - `memory_hash` from the canonical hash plus `master_person_id`.
8. If the edge has `valid_from`, normalize it and write it as `valid_from`, `occurred_at`, `memory_date`, and `memory_date_source: "edge.valid_from"`.
9. Tag metadata with `memory_scope: "global_master_person"`, `visibility: "global_objective"`, `master_person_id`, graph node/edge provenance, `source_type: "graph_edge"`, and `source_platform: "falkordb"`.
10. When `dry_run=false`, duplicate-check Mem0 by `user_id = "master_person:{master_person_id}"` plus `run_id = memory_hash`.
11. Write non-duplicates to `POST https://api.mem0.ai/v3/memories/add/` with `infer=false`.
12. Return `graph_edge_stage`, `candidate_memory_count`, and full `candidate_memories`.

## `mem0/delete-master-person-memories` (`#13150`)

| Field | Value |
| --- | --- |
| Xano function name | `mem0/delete-master-person-memories` |
| Xano ID | `13150` |
| Workspace | `3` |
| Primary input | `master_person_id` |
| Default mode | `dry_run=true` |
| Mem0 delete mode | `dry_run=false`, requires `$env.MEM0_API_KEY` |
| Memory target | Shared synthetic Mem0 user: `master_person:{master_person_id}` |
| QA purpose | Clear master-person memories before rerunning master-person memory tests |

### What It Does

This function is a QA helper for clearing the shared Mem0 scope used by `mem0/create-master-person-memories`.

It lists or deletes memories whose Mem0 `user_id` is exactly `master_person:{master_person_id}`. It does not delete mailbox-owner private contact memories stored under `user.workos_id`, including private email or calendar memories created by `#13149`.

<Note>
  The input must be the `master_person.id`, not a `contacts.id` and not the Xano function ID. For the Richard Kerris QA path documented here, the `master_person_id` is `4464`.
</Note>

<Warning>
  Use `dry_run=true` first. Setting `dry_run=false` deletes every memory found in the synthetic master-person Mem0 scope for the given `master_person_id`.
</Warning>

### Inputs

| Input | Default | Purpose |
| --- | --- | --- |
| `master_person_id` | required | The shared `master_person` whose synthetic Mem0 scope should be inspected or cleared. |
| `dry_run` | `true` | When true, returns matching memories without deleting them. |
| `max_pages` | `20` | Maximum Mem0 list pages to inspect. |
| `page_size` | `1000` | Page size used when listing Mem0 memories. |

### Step-by-step

1. Load `master_person` by `master_person_id`.
2. Build the synthetic Mem0 user id: `master_person:{master_person_id}`.
3. List Mem0 memories from `POST https://api.mem0.ai/v3/memories/` filtered to that exact `user_id`.
4. If the list call fails, fall back to `POST https://api.mem0.ai/v3/memories/search/` with the same `user_id` filter.
5. De-duplicate returned memories by Mem0 memory id.
6. If `dry_run=true`, return the matching memories and `would_delete_count` without making delete calls.
7. If `dry_run=false`, delete each memory id, trying the v1 delete endpoint first and the v3 delete endpoint second.
8. Treat HTTP `404` as already deleted so the function is idempotent for QA reruns.
9. Return counts, memory previews, and per-memory delete attempt details.

### Response Shape

| Field | Purpose |
| --- | --- |
| `status` | `dry_run_complete`, `delete_complete`, or `delete_partial_failure`. |
| `mem0_user_id` | The synthetic Mem0 user scope that was inspected or cleared. |
| `read_method` | `list` or `search_fallback`. |
| `found_count` | Number of unique memories found with a Mem0 memory id. |
| `missing_id_count` | Number of returned memory objects that did not include a deleteable id. |
| `would_delete_count` | Number of memories that would be deleted in dry-run mode. |
| `deleted_count` | Number of memories deleted, or treated as already deleted. |
| `failed_count` | Number of delete attempts that failed. |
| `memories` | A preview array of matching memories. |
| `delete_results` | Per-memory delete details when `dry_run=false`. |

## `mem0/delete-contact-memories` (`#13151`)

| Field | Value |
| --- | --- |
| Xano function name | `mem0/delete-contact-memories` |
| Xano ID | `13151` |
| Workspace | `3` |
| Primary input | `contacts_id` |
| Default mode | `dry_run=true` |
| Mem0 delete mode | `dry_run=false`, requires `$env.MEM0_API_KEY` |
| Memory target | Mailbox owner's private Mem0 user, keyed by `user.workos_id` |
| QA purpose | Clear private contact memories before rerunning contact memory tests |

### What It Does

This function is a QA helper for clearing private Mem0 memories tied to one `contacts` row.

It resolves `contacts_id` to `contacts.user_id`, then resolves that mailbox owner to `user.workos_id`, which is the private Mem0 `user_id`. It reads private memories matching `metadata.contacts_id`; by default it also reads private memories matching the linked `metadata.master_person_id`, because `mem0/get-contacts-memories` uses both match paths.

It does not call FalkorDB or run Cypher. It only lists matching Mem0 memories and optionally deletes them by Mem0 memory id.

<Warning>
  Use `dry_run=true` first. Setting `dry_run=false` deletes matching private Mem0 memories under the mailbox owner's `user.workos_id`.
</Warning>

<Note>
  For safety, `dry_run=false` refuses broad deletion unless `source_type` or `source_platform` is provided, or `delete_all_sources=true` is explicitly set.
</Note>

### Inputs

| Input | Default | Purpose |
| --- | --- | --- |
| `contacts_id` | required | The `contacts` row whose private memories should be inspected or cleared. |
| `user_id` | `contacts.user_id` | Optional mailbox owner override for manual testing; must own the contact. |
| `dry_run` | `true` | When true, returns matching memories without deleting them. |
| `source_type` | optional | Optional metadata filter, for example `graph_edge` or `email_thread`. |
| `source_platform` | optional | Optional metadata filter, for example `falkordb` or `nylas`. |
| `memory_scope` | optional | Optional metadata filter, normally `mailbox_owner_private`. |
| `include_master_person_matches` | `true` | Also inspect private memories matched by `metadata.master_person_id`. |
| `delete_all_sources` | `false` | Required for `dry_run=false` when no source filter is provided. |
| `max_pages` | `20` | Maximum Mem0 list pages to inspect per filter. |
| `page_size` | `1000` | Page size used when listing Mem0 memories. |

### Step-by-step

1. Load `contacts` by `contacts_id`.
2. Resolve the mailbox owner from `contacts.user_id`, or validate the optional `user_id` override owns the contact.
3. Load `user.workos_id`; this is the private Mem0 `user_id`.
4. Resolve the linked `master_person_id` and display name for response context.
5. Read Mem0 memories from `POST https://api.mem0.ai/v3/memories/` where `user_id = user.workos_id` and `metadata.contacts_id = contacts_id`.
6. When `include_master_person_matches=true`, also read private memories where `user_id = user.workos_id` and `metadata.master_person_id = contacts.master_person_id`.
7. De-duplicate returned memories by Mem0 id, memory hash, canonical hash, or run id.
8. Client-side confirm each returned memory still has `metadata.contacts_id = contacts_id` or, when enabled, `metadata.master_person_id = contacts.master_person_id`; then apply optional source filters such as `source_type = "graph_edge"` and `source_platform = "falkordb"`.
9. If `dry_run=true`, return the matching memories and `would_delete_count` without making delete calls.
10. If `dry_run=false`, refuse broad deletion unless a source filter is present or `delete_all_sources=true`.
11. Delete each matching memory id, trying the v1 delete endpoint first and the v3 delete endpoint second.
12. Treat HTTP `404` as already deleted so the function is idempotent for QA reruns.

### Response Shape

| Field | Purpose |
| --- | --- |
| `status` | `dry_run_complete`, `delete_complete`, `delete_partial_failure`, or `refused_missing_source_filter`. |
| `contacts_id` | The contact row being inspected or cleared. |
| `mem0_user_id` | The mailbox owner's private Mem0 user scope. |
| `master_person_id` | The linked shared person id used for optional private master-person matching. |
| `filters_applied` | Source and scope filters used after the Mem0 read. |
| `read_scopes` | Per-filter read summary for `metadata.contacts_id` and optionally `metadata.master_person_id`. |
| `read_count` | Unique private memories read before source filtering. |
| `found_count` | Unique private memories matching the source filters. |
| `filtered_out_count` | Unique private memories read but not targeted for deletion. |
| `would_delete_count` | Number of memories that would be deleted in dry-run mode. |
| `deleted_count` | Number of memories deleted, or treated as already deleted. |
| `failed_count` | Number of delete attempts that failed. |
| `memories` | A preview array of targeted memories. |
| `delete_results` | Per-memory delete details when `dry_run=false`. |

## `mem0/delete-all-memories` (`#13152`)

| Field | Value |
| --- | --- |
| Xano function name | `mem0/delete-all-memories` |
| Xano ID | `13152` |
| Workspace | `3` |
| Primary input | none |
| Default mode | `dry_run=true` |
| Mem0 delete mode | `dry_run=false` plus `confirm_phrase = "DELETE ALL MEM0 MEMORIES"` |
| Memory target | Every Mem0 memory visible to `$env.MEM0_API_KEY` |
| QA purpose | Full Mem0 reset before high-risk QA reruns |

### What It Does

This function is the global QA reset helper for Mem0. It lists every memory visible to the configured Mem0 API key, de-dupes by Mem0 id, memory hash, canonical hash, or run id, and optionally deletes each memory by id.

<Warning>
  This is intentionally broad. Use `dry_run=true` first. Actual deletion requires `dry_run=false` and the exact `confirm_phrase`: `DELETE ALL MEM0 MEMORIES`.
</Warning>

<Note>

## `mem0/get-contacts-memories` (`#13147`)

| Field | Value |
| --- | --- |
| Xano function name | `mem0/get-contacts-memories` |
| Xano ID | `13147` |
| Workspace | `3` |
| Primary input | `contacts_id` |
| Mem0 read key | `contacts.user_id -> user.workos_id` |
| Master-person read key | `contacts.master_person_id -> user_id = "master_person:{master_person_id}"` |
| Response | Array of merged, de-duped memories, sorted by newest `memory_date` first |

### What It Does

This function is the combined Contact Side Panel read path. It takes a `contacts_id`, resolves the mailbox owner's Mem0 user and the linked `master_person`, fetches both private and shared memories, then returns one array ordered by newest memory date first.

### Step-by-step

1. Load `contacts` by `contacts_id`.
2. Resolve `contacts.user_id` to the mailbox owner row.
3. Require `user.workos_id`; this is the private Mem0 `user_id`.
4. Resolve `contacts.master_person_id` and the best display name from `master_person.name` or `contacts.full_name`.
5. Read the mailbox owner's private Mem0 scope using `user_id = user.workos_id`.
6. Client-side keep only private memories whose metadata explicitly has `contacts_id = contacts_id` or `master_person_id = contacts.master_person_id`; missing metadata is not defaulted into a match.
7. Read shared master-person memories from Mem0 using `user_id = "master_person:{master_person_id}"`.
8. Normalize each memory into a common response object with `scope`, `memory`, `memory_date`, `memory_date_source`, `created_at`, `updated_at`, `contacts_id`, `master_person_id`, `source_type`, hashes, and raw `metadata`.
9. De-dupe by `memory_hash`, `canonical_memory_hash`, Mem0 id, or normalized memory text.
10. Sort by `memory_date` descending, so the most recent memory date appears first. Memories with no parseable date are last.

### Memory Date Ordering

`memory_date` means the date of the event described by the memory when that date exists. The reader checks metadata in this order:

1. `memory_date`
2. `occurred_at`
3. `source_timestamp`
4. `source_date`
5. `event_at`
6. `event_date`
7. `event_start`
8. `start_date`
9. `valid_from`
10. `edge_properties.valid_from`
11. `date`
12. `evidence_dates`

If no event date is defined, the reader falls back to the Mem0 memory creation/update date: `created_at`, `createdAt`, `created`, `metadata.created_at`, `updated_at`, or `updatedAt`.

### Step-by-step

### 1. Resolve The Effective User

The function sets `effective_user_id` from `input.user_id`, falling back to `auth.id`. It stops immediately if neither is present.

It also loads the runtime keys:

- `$env.openRouter` is required because the email-thread prefilter and extraction stages call OpenRouter.
- `$env.MEM0_API_KEY` is required only when `dry_run=false`.

The function writes temporary checkpoints to `log_crash` throughout the run.

### 2. Load And Validate The Contact

The function loads `contacts` by `my_contact_id`, verifies that the contact belongs to `effective_user_id`, and then loads the linked `master_person`.

It chooses `subject_person_name` from `master_person.name`, falling back to `contacts.full_name`. It also loads the mailbox owner from `user` and requires `user.workos_id`; that WorkOS ID becomes the Mem0 `user_id`.

### 3. Build The Contact And Mailbox Context

The function prepares the lookup context used by every later stage:

- Reads `my_person_email_join` plus `master_email` to collect the contact's active email addresses.
- Reads active `nylas_grant` rows for the mailbox owner.
- Collects mailbox owner email accounts from those grants.
- Loads up to 1,000 of the user's `contacts` rows, each with active emails, so extracted related contacts can be resolved back to known `contacts_id` values.

### 4. Import One-hop Graph Edge Memories

If the linked `master_person` has a `node_uuid`, the function queries FalkorDB through `mvp/falkor/send-cypher`.

The Cypher query finds one-hop edges around the master-person node where the relationship has a `description` or `fact`, returning edge type, edge properties, from/to node metadata, and labels. The result is capped by `max_graph_edge_memories`.

For each graph edge, a lambda:

- Uses the edge description as the memory text.
- Scrubs large arrays, embeddings, and vectors from metadata.
- Builds deterministic `canonical_memory_hash`, `memory_hash`, `source_id`, and `run_id` values.
- Tags metadata with `source_type: "graph_edge"`, `source_platform: "falkordb"`, `contacts_id`, `master_person_id`, edge identifiers, node labels, and `schema_version: "2026-06-22"`.
- Adds the candidate to `graph_edge_stage.candidate_memories` and the global `candidate_memories` array.

When `dry_run=false`, each graph memory is duplicate-checked in Mem0 with `user_id + run_id`. If no duplicate is found, it is written to `POST https://api.mem0.ai/v3/memories/add/` with `infer=false`.

<Note>
  This current implementation writes graph-derived facts as contact-scoped memories. The target architecture in [mem0 Full Scope](/guides/open-work/mem0/full-scope#1-graph-to-global-master-person-memories) moves objective graph facts into a shared master-person scope.
</Note>

### 5. Decide Whether Email Thread Processing Can Run

Email processing is skipped when:

- The contact has no active email addresses.
- The mailbox owner has no active Nylas grants.

Otherwise the function loops through each active Nylas grant and each active contact email.

### 6. Find Candidate Nylas Threads

For each grant/contact-email pair, the function calls `mvp/nylas/fetch-threads` with:

- `any_email` set to the contact email.
- `limit` set to `max_threads_per_batch`.
- `page_token` passed through from input when present.

It records any `next_cursor` values in `continuation_cursors` so later runs can resume. It also de-dupes threads by a key made from the Xano grant id, the Nylas grant id, and the Nylas thread id.

### 7. Fetch Messages And Clean The Transcript

For each unprocessed thread, the function calls `mvp/nylas/fetch-messages` with the thread id and `max_messages_per_thread`.

The thread-processing lambda then:

- Sorts messages by date.
- Strips HTML, scripts, styles, quoted reply chains, unsubscribe/tracking links, common email footers, and repeated whitespace.
- Rejects threads where the contact is not a direct `from` or `to` participant.
- Rejects too-short transcripts and fully automated or transactional threads.
- Builds a cleaned transcript capped by `max_cleaned_transcript_chars`.
- Collects sent-email samples where the mailbox owner wrote to the contact, for the optional email tone stage.

### 8. Prefilter Threads With OpenRouter

The first model call is a cheap JSON prefilter:

| Purpose | Model |
| --- | --- |
| `cheap_prefilter` | `deepseek/deepseek-v4-flash` |

It returns `should_extract` and a reason, but the result is not authoritative. The function overrides soft `should_extract=false` decisions unless the transcript or model rejection reason has hard noise signals such as newsletters, receipts, invoices, unsubscribe language, automated notifications, no-reply senders, calendar-forwarding noise, meeting-forwarding noise, or purely transactional content. This keeps recall high and lets the extraction model make the final memory-quality decision.

### 9. Extract Durable Memories

Threads that pass the prefilter go to the extraction model:

| Purpose | Model |
| --- | --- |
| `final_memory_extraction` | `deepseek/deepseek-v4-flash` |

The model must return strict JSON with a thread summary and candidate memories. The extraction instructions prefer short, neutral, standalone facts about preferences, priorities, goals, objections, decisions, commitments, relationship history, and business context.

The function stores only candidates that:

- Have `should_store=true`.
- Have confidence at or above `0.75`.
- Are not marked `sensitive` or `secret`.
- Have non-empty memory text.
- Do not contain CJK-script text.

The function normalizes email event and evidence dates before storing them. Numeric Unix timestamps from Nylas or the model are converted to ISO timestamps, date-only values remain valid, and `occurred_at` / `memory_date` use the supported `event_date` first, then fall back to the latest normalized evidence date.

### 10. Resolve Related Contacts Without Fan-out

Each accepted email memory remains attached to the input contact. Related people are annotations, not extra Mem0 writes.

The function resolves related contacts against the mailbox owner's known contacts by email or by visible participant name plus email. It excludes the mailbox owner and excludes the primary subject contact from the surfaced-contact list.

Metadata includes:

- `surfaced_contacts`
- `surfaced_contact_ids`
- `retrieval_contact_ids`
- `retrieval_person_names`
- `related_contacts`
- `unresolved_related_contacts`
- `fanout_target_count: 1`

This lets a memory remain primary-subject-scoped while still being discoverable when another resolved contact was involved.

### 11. Write Email Memories To Mem0

For each accepted email memory, the function builds deterministic hashes from the thread id, source hash, memory text, and primary contact id.

The Mem0 metadata includes:

- `source_type: "email_thread"`
- `source_platform: "nylas"`
- `source_table: "my_person"`
- `contacts_id`
- `master_person_id`
- `memory_scope: "mailbox_owner_private"`
- `mailbox_owner_user_id`
- `xano_nylas_grant_id`
- `nylas_grant_id`
- `source_email_account`
- `nylas_thread_id`
- `nylas_message_ids`
- `source_hash`
- `canonical_memory_hash`
- `memory_hash`
- `run_id`
- `extraction_model`
- `schema_version: "2026-06-23"`

When `dry_run=false`, the function duplicate-checks Mem0 with `user_id + run_id`. If no duplicate is found, it calls `POST https://api.mem0.ai/v3/memories/add/` with `infer=false`.

When Mem0 returns an `event_id`, the function polls `GET https://api.mem0.ai/v1/event/{event_id}/` up to three times and records event status, event errors, and response previews in each candidate's `mem0` object.

### 12. Process Calendar Events After Email

After email-thread processing, the function scans Nylas calendar events for active contact emails.

For each matched event, it:

- Builds a `calendar_event` memory under the mailbox owner's private Mem0 user.
- Resolves participant names from known contacts and Nylas participant data.
- Avoids copying generic titles such as `+ 2 others` into the memory text when resolved participant names are available.
- Stores participant metadata, including `participant_emails`, `participant_names`, `resolved_participants`, and `resolved_participant_count`.
- Tags the memory with `contacts_id`, `master_person_id`, `occurred_at`, `event_title`, `calendar_id`, and Nylas event provenance.

When `dry_run=false`, the calendar stage duplicate-checks Mem0 with `user_id + run_id` and writes non-duplicates with `infer=false`.

### 13. Optionally Build The Sent-email Voice Profile

When `run_email_tone=true`, the function also builds a per-contact writing-style profile from messages the mailbox owner sent to the contact.

That stage:

1. Calls `mvp/nylas/get-nylas-keys`.
2. Uses the Nylas folders API to find the real Sent folder, falling back to `SENT`.
3. Searches Nylas messages for each active contact email using `any_email` and the sent folder.
4. Falls back to a folder-less message search if the folder-scoped search fails.
5. Keeps only messages from the mailbox owner to the contact.
6. Tracks pages queried, messages seen, qualifying sent emails, next cursors, folder resolution, and errors in `sent_email_collection`.
7. De-dupes samples and caps them by `max_sent_email_samples` and `max_sent_email_chars`.
8. Computes a corpus hash from the selected sample set.
9. Skips the model call when the existing `email_draft_tone.email_tone_json.corpus_hash` and prompt version already match, unless `force_email_tone_refresh=true`.
10. Calls OpenRouter with `anthropic/claude-opus-4.8` and a v3 forensic writing-style prompt.
11. If `dry_run=false` and the model returns a profile, upserts `email_draft_tone` by `contacts_id`.

This stage does not write to Mem0. It creates the reusable voice profile used by email drafting.

### 14. Return The Run Summary

The response includes:

- Function identity and status.
- Contact, mailbox owner, and Mem0 user ids.
- Active emails, known-contact count, and Nylas grant count.
- `threads_seen`, `threads_processed`, and `threads_skipped`.
- `sent_email_collection` and `sent_email_sample_count`.
- `calendar_stage`.
- `email_tone_stage`.
- `candidate_memory_count` and full `candidate_memories`.
- `model_usage` for prefilter, extraction, and email tone stages.
- `continuation_cursors`.
- Per-thread diagnostics in `thread_results`.

Status is `dry_run_complete` for dry runs and `write_complete` when `dry_run=false`, unless the run is skipped or blocked before thread processing.
