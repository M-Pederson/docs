### v0.17 (2026-05-06) тАФ Gaps 1/2/3 + suspect-lifecycle bug fix

Three concrete gaps closed plus one bug found in real-world testing:

**Gap 1 тАФ `conflicts_resolved` logging in orchestrators (was: schema column existed but
orchestrator never wrote to it).** New helpers `merge/helpers/detect-conflicts-company` (#12969,
covers `company_name`, `company_domain`, `linkedin_url`, `tagline`) and
`merge/helpers/detect-conflicts-person` (#12970, covers `name`, `linkedin_url`, `bio`, `bio_500`).
Each scans survivor + duplicates, finds the master row whose value matches the winner picked
by `aggregate-scalars-*`, and collects disagreeing non-null values from other rows as
`{field, winner_value, winner_source_master_id, loser_values[], loser_source_master_ids[],
rule_applied}`. Orchestrators call the helper after aggregate-scalars and write the result to
`merge_audit.conflicts_resolved`. Skipped entirely when match_kind is in the deterministic class
(see [match_kind taxonomy](#confidence-escape-hatch--deterministic-matches-skip-even-the-logging)).
Verified on Adjacent fixture (audit 73) тАФ 3 conflicts logged (domain, linkedin, tagline);
verified end-to-end on real Atlantic Pictures merge (audit 90) тАФ 1 tagline conflict persisted
to disk on a `status: success` row.

**Gap 2 тАФ `work_experience` + `education_experience` dedup after long-tail repoint (was:
long-tail helper was repoint-only; James Ruse merge surfaced the gap when master_person 187
ended up with two education rows for the same school/years).** New helpers
`merge/helpers/dedup-experience-after-company-merge` (#12971) and
`-after-person-merge` (#12972). Each takes the survivor id, queries
`master_*_id == survivor_id`, groups by natural key (work: `(person, start_year, end_year, lower-trim title)`;
education: `(person/company, start_year, end_year)`), picks oldest by `created_at` as keeper,
deletes losers. dry_run=true returns the plan. Wired into both orchestrators between long-tail
and fold. Verified on synthetic fixture (audit 92) тАФ row 22750 deleted, 22749 kept.

**Gap 3 тАФ `match_kind` enum normalized to broader taxonomy (was: spec listed 3 values but real
merges used 5+ subtypes like `domain_exact`, `linkedin_exact`, `manual_inspection`).** Both
orchestrators now compute a `$is_deterministic_match` predicate that accepts
`'deterministic' | 'email_exact' | 'phone_exact' | 'domain_exact' | 'linkedin_exact'`. The
`merge_audit.match_kind` column description and the conflict-logging skip-rule documentation
have been rewritten to reflect the actual taxonomy. No backfill of historic audit rows тАФ
existing values like `linkedin_exact` carry useful audit signal vs. flattening to generic
`deterministic`. Verified on Adjacent fixture: `linkedin_exact` (audit 86) and `domain_exact`
(audit 87) both correctly skip; `manual_inspection` (audit 88) still logs 3 conflicts.

**Bug found + fixed тАФ suspect-lifecycle else-branch was over-eager.** In both orchestrators,
the `else` branch of the `merge_suspect` collapse/repoint logic was unconditionally setting
`merged=true` on every survivor-touching suspect, even when neither side of the suspect's pair
was an actual duplicate in the merge. Cascade: the subsequent `needs_rescore` step queries
`WHERE merged == false AND survivor side matches`, which always returned zero rows. The
planned re-score cron loop would have had nothing to process. Fixed by adding an inner
conditional that only marks `merged=true` when at least one side was actually a duplicate
(`$a_match_count > 0 || $b_match_count > 0`); otherwise the suspect stays `merged=false`
so the rescore step can flag it. Verified on synthetic fixture (audit 96) тАФ collapsed-pair
deleted, survivor-only-touched pair correctly flagged `needs_rescore=true`.

**XanoScript gotcha learned the hard way.** The `|` character inside string literals breaks
the runtime parser with `ERROR_FATAL: Invalid syntax. Try using the "transform" filter or an
expression.` тАФ even though the function deploys cleanly. Discovered while building
`dedup-experience-after-company-merge` (#12971) тАФ the original key `"p" ~ id ~ "|y" ~ year`
errored at runtime; switching to `"p" ~ id ~ "_y" ~ year` fixed it. Worth adding to the
xanoscript-builder skill's gotchas section. Note also: every failed test left 2 dangling
`merge_lock` rows that needed manual cleanup before the next try (the orchestrator dies mid-
stack on these errors and never reaches the lock-release block).

**v0.14 (2026-05-06) тАФ closed two race-condition gaps in merge_lock handling.**
(1) **Lock all_ids, not just the survivor.** Previously a merge with `survivor=A,
duplicate_ids=[B]` only locked A. Concurrent merge2 picking `survivor=B` (a duplicate of merge1)
would race because B existed up until merge1's master_person delete near the end. v0.14
sequentially acquires `merge_lock` rows for every id in `[survivor, ...duplicate_ids]`. On any
unique-constraint failure, rolls back the locks already acquired and aborts with

marks the corresponding merge_audit row `status=aborted_stale` if still `pending`. Defends against
orchestrator crashes that leave the lock dangling.

**v0.13 (2026-05-06) тАФ `merge/helpers/graph-merge-person` (#12949) extended with smart
property merge on edge collisions.** Previous behavior: `MERGE (s)-[r_new:T]->(t) ON CREATE SET
r_new = properties(r_old)` тАФ orphan's edge properties were silently dropped on collision. New
behavior: split into two cyphers per edge-type-direction. (1) **NO-COLLISION MOVE** тАФ when
survivor has no edge to t, `CREATE (s)-[r_new:T]->(t) SET r_new = properties(r_old) DELETE r_old`.
(2) **COLLISION MERGE** тАФ when survivor already has an edge: capture both property maps, set
r_new = oldprops, then overlay newprops. Net result: survivor's existing values win on key
conflicts, but any keys present only on the orphan's edge get inherited (fill-missing semantics
тАФ "merge all parameters" without information loss). Total cyphers per orphan: 29 move + 29
merge = 58 (was 29 in v2.3). Verified clean execution: 58/58 cyphers succeed against FalkorDB.

### Critical finding: Xano `tableref_id` is documentation, not an enforced FK constraint

When v0.5 ran for-real on Sam Seely (4458 тЖР [4457, 4459]), the duplicate `master_person`
deletes succeeded **silently leaving orphans** in 8 dependent tables (61 rows total: log_crash 26,
work_experience 12, enrich_history_person 8, education_experience 6, person_view 3, plus
expertise_identification_log/log_enrichment_person/vector_logs).

The orphans were rows in tables we hadn't covered in v0.5's main path. Their `master_person_id`
columns now pointed at deleted IDs (4457/4459). His Frame.io / Knock work history was effectively
lost from the survivor's profile until a cleanup pass repointed everything to 4458.

Lesson: every uncovered table needs a repoint pass **before** the master_person delete, or the
delete strands data. The try_catch we put around the delete in v0.5 was the wrong instinct тАФ it
masked the correctness gap rather than preventing it.

### FalkorDB graph-merge-person v0.7 тАФ incident, recovery, fix (2026-05-05)

**Incident:** First attempt at `merge/helpers/graph-merge-person` (#12949) used a single complex
cypher per edge type with `OPTIONAL MATCH` chains and a `FOREACH (CASE WHEN ... | CREATE ...)`
trick. In FalkorDB, when the inner `OPTIONAL MATCH` returned NULL for the target node, the FOREACH
CREATE failed with `"Failed to create relationship; endpoint was not found"` тАФ but the outer
DETACH DELETE on the orphan still ran. Result: Traver's two orphan Person nodes (1cadc19b...,
12329e15...) were deleted with all 28 of their edges still attached. **Edges were lost.**

**Recovery:**
- The only truly unique lost edge (WORKS_AT to Unknown Company `34951bf7-5d54-46e9-83cf-2bfb0e2c05fe`,
  master_company_id 10170) was restored via direct cypher MERGE with the original properties
  recovered from inspection logs
- The other ~26 lost edges were almost certainly duplicates of survivor's existing edges (HAS_EXPERTISE,
  WORKS_AT, FOUNDED, LOCATED_IN to common targets), so the loss was effectively absorbed by
  survivor's intact edges
- **Acknowledged real loss:** ~2 KNOWS edges from orphans to other Persons. KNOWS isn't
  re-derivable without re-running the original ingest. Documented as accepted loss.

**Fix тАФ v2.1 graph-merge-person:** completely restructured cypher pattern:

1. **Per-orphan, per-edge-type cypher** instead of one big chained query. 5 edge types ├Ч N orphans.
2. **MATCH (not OPTIONAL MATCH)** тАФ if no edges exist, returns 0 rows, no-op.
3. **MERGE on survivor** with `ON CREATE SET r_new = properties(r_old)` тАФ copies orphan props
   to new edge if survivor didn't have one. On collision (survivor already has edge to same target):
   survivor's edge wins, orphan's props discarded. **v0.8 will add per-type smart merge.**
4. **DELETE orphan edge** after MERGE.
5. **Safety gate:** after all 5 cyphers run, query `MATCH (o)-[r]-() RETURN count(r) AS remaining`.
   `DETACH DELETE` only fires if `remaining == 0`. Otherwise orphan stays alive and the function
   reports it in `orphans_skipped`.
6. **try_catch around every cypher** тАФ failure on one edge type doesn't kill the whole orphan.

**Verification on Sam Seely orphans (still alive at this point):**

- Isolated test: moved 1 FOUNDED edge from orphan 6cfa7efd to survivor 2f883678 cleanly. Other
  edges intact. Cypher pattern verified.
- Full v2.1 run on both Seely orphans: **24 edges moved (HAS_EXPERTISE 13, WORKS_AT 8, FOUNDED 1,
  LOCATED_IN 2, KNOWS 0), 2 orphan nodes DETACH DELETEd, zero errors.** Survivor 2f883678 ended
  with 10 HAS_EXPERTISE, 5 WORKS_AT, 3 KNOWS, 1 FOUNDED, 1 LOCATED_IN.

**Lessons:**
- Test cypher in isolation before wrapping it in a delete. The first attempt would have failed
  silently in dry_run if I'd separated the cypher generation from the execution тАФ but I'd jumped
  straight to running it.
- "Defense in depth" via try_catch around the master_person delete (v0.5) was the wrong instinct.
  It masked the orphan-leaving bug instead of preventing it. The fix is to gate destructive ops
  on positive confirmation that the migrate succeeded тАФ not to swallow errors.
- FalkorDB's `OPTIONAL MATCH` chains with `FOREACH (CASE | CREATE...)` patterns are dangerous.
  Prefer simple `MATCH ... MERGE ... DELETE` per-edge-type sequences.

### v0.14 тАФ lock all_ids + stale-lock cleanup cron (2026-05-06)

```text
function/12945  merge/person-v2
function/12952  merge/helpers/release-stale-locks
task/163        merge/task/release-stale-locks
```

Two race-condition fixes for the lock subsystem.

**Lock all_ids.** v0.12 only acquired a lock for the survivor. Concurrent attack: merge1 starts
with `survivor=A, duplicate_ids=[B,C]`. Before merge1's master_person delete runs, merge2 calls
the orchestrator with `survivor=B` (which still exists). merge2's lock for B succeeds (different
master_id from merge1's lock on A), and now both merges run on overlapping data. v0.14 fixes by
acquiring locks for every id in `$all_ids` sequentially:

```text
for $id in $all_ids:
  try: db.add merge_lock {entity_kind: "person", master_id: $id, ...}
  catch: $acquire_failed = true; $failed_lock_target = $id

if $acquire_failed:
  for $lid in $acquired_lock_ids: db.del merge_lock {id: $lid}
  $lock_acquired = false   тЖТ status = "blocked_locked"
else:
  $lock_acquired = true    тЖТ proceed
```

**Stale-lock cleanup.** If the orchestrator crashes between acquire and release, the lock rows
remain тАФ blocking all future merges of those master_ids forever. The new
`merge/helpers/release-stale-locks` (#12952) takes `stale_threshold_seconds` (default 600) and
deletes any merge_lock row where `acquired_at < (now - threshold_ms)`. Each released lock's
corresponding merge_audit row is marked `status=aborted_stale` if it was still `pending`, so
post-mortem inspection knows which merges were aborted by the cleanup vs explicitly blocked.

Scheduled as task **#163** (`merge/task/release-stale-locks`) тАФ every 5 min, threshold 10 min.
A normal merge takes 1тАУ10s; 10 min is comfortably above any legitimate run time.

**Verification matrix:**

| Scenario | Expected | Verified |
| --- | --- | --- |
| Single-id merge, no conflict | `lock_acquired=true`, 1 lock acquired+released | тЬЕ |
| Multi-id merge, no conflict | `lock_acquired=true`, N locks acquired+released | тЬЕ (3 of 3) |
| Multi-id merge, pre-existing lock on duplicate id | `lock_acquired=false`, `failed_lock_target=conflicting_id`, 0 acquired (rollback) | тЬЕ |
| Stale lock present | Cleanup releases + marks audit `aborted_stale` | тЬЕ |

Result now includes `lock_plan: {locked_master_ids, acquired_lock_ids, failed_lock_target,
released_lock_ids}` for full auditability.

### v0.13 тАФ smart per-edge-type property merge in graph-merge (2026-05-06)

```text
function/12949
merge/helpers/graph-merge-person
```

Splits each edge-type-direction into **two cyphers** per orphan instead of one:

**1. NO-COLLISION MOVE** тАФ when survivor has no edge of this type to the target node:

```cypher
MATCH (o:Person {uuid: $orphan})-[r_old:EDGE_TYPE]->(t)
MATCH (s:Person {uuid: $survivor})
WHERE id(t) <> id(s) AND NOT (s)-[:EDGE_TYPE]->(t)
CREATE (s)-[r_new:EDGE_TYPE]->(t)
SET r_new = properties(r_old)
DELETE r_old
RETURN count(*) AS edges_moved
```

Identical to v2.3 behavior тАФ moves the edge with all its properties intact.

**2. COLLISION MERGE** тАФ when survivor already has an edge of this type to the target:

```cypher
MATCH (o:Person {uuid: $orphan})-[r_old:EDGE_TYPE]->(t)
MATCH (s:Person {uuid: $survivor})-[r_new:EDGE_TYPE]->(t)
WHERE id(t) <> id(s)
WITH r_new, r_old, properties(r_old) AS oldprops, properties(r_new) AS newprops
SET r_new = oldprops              -- replace with orphan's props
WITH r_new, r_old, newprops
SET r_new += newprops              -- overlay survivor's original props (newprops wins on conflict)
DELETE r_old
RETURN count(*) AS edges_merged
```

The two-phase SET is the trick: capture both prop maps before any writes, replace with orphan's,
then overlay survivor's. End state: every key the survivor had keeps the survivor's value;
every key only the orphan had gets inherited. **Survivor wins on conflict, orphan fills gaps.**

This is the "merge all parameters" semantic тАФ no orphan-edge data is lost on collision unless
the same key already exists on the survivor's edge with a different value (in which case
survivor's wins, which is the safe default).

**Counts:** 25 outgoing-only ├Ч 2 phases + 2 bidirectional ├Ч 2 directions ├Ч 2 phases = 50 + 8 =
58 cyphers per orphan (was 29 in v2.3). Each cypher is logged with `phase: "move"` or
`"merge"` and a `count` of edges affected.

The DETACH DELETE safety gate is unchanged тАФ orphan node deletion is still gated on a final
`MATCH (o)-[r]-() RETURN count(r) AS remaining` query confirming `remaining == 0`.

Verified: 58/58 cyphers ran cleanly against FalkorDB on a non-existent test orphan
(00000000-...) тАФ no errors, all returned `count: 0` as expected.

### v0.8 тАФ merge_suspect lifecycle

After the master_person delete (so suspects reflect the final state), the orchestrator now

## Xanoscript gotchas (learned the hard way)

These are runtime-vs-docs mismatches discovered during the v0.1 build. Apply them when extending
the orchestrator or building per-table dedup helpers.

| Don't write | Write instead | Why |
| --- | --- | --- |
| `\|filter:$$.field == X` | `\|filter:"return $this.field == X;"` | The bare `$$.field` syntax compiles but matches zero rows at runtime. Legacy quoted JS form is the only one that actually works. |
| `\|some:$$.field == true` | `\|some:"return $this.field === true;"` | Same reason. |
| `$result.field` (when `$result` came from `function.run`) | `$result\|get:"field"` | `.field` access fails on function.run results with "Unable to locate var: result.field". `\|get:"field"` works. (`.field` does work on `db.add`/`db.get` results.) |
| `$input.X\|default:false` | `($input.X == true)` or ternary | `\|default` is not a real filter. Using it produces a misleading `"Unable to locate func entry: default"` runtime error. |
| `db.direct_query { sql = $var }` or `sql = "..." ~ $var ~ "..."` | Use static `db.query <table_name>` blocks per table | `db.direct_query.sql` only accepts a string literal. Means generic FK repointing across N tables can't be one helper тАФ has to be one block per table (in the orchestrator) or per-table helper functions. |
| `db.edit table { data = $merged_object }` | `data = { name: $merged\|get:"name", ... }` (one key per field) | `data` field rejects whole-object var refs. Have to spread the object into individual keys. |
| Multi-line `description = "..."` | One-line description | Newlines inside `description` strings break the parser with `unexpected newline`. |

## Why merge is hard

A duplicate is not just two rows in `master_person`. It's two rows that are each pointed at by
**86 other tables for person** and **65 other tables for company**, plus a node in FalkorDB with
N edges, plus join rows under `my_person` for every user who has either copy in their CRM.

A correct merge has to:

1. Pick a survivor and aggregate scalar fields from the duplicates onto it
2. Repoint every FK in every dependent table from each duplicate тЖТ survivor
3. Collapse rows that become duplicates after repoint (e.g. two `master_email` rows with the same address)
4. Collapse 1-row-per-entity caches by merging columns first-not-empty (`person_enrich_data`, `company_enrich_data`)
5. Re-route every graph edge from the duplicate Person/Company node тЖТ survivor node, deduping on type
6. Delete the duplicate rows + duplicate nodes
7. Re-derive computed state (employment, embeddings, person/company nodes)
8. Be idempotent тАФ re-running with the same input is a no-op
9. Be audited тАФ every merge writes a snapshot row that can roll back
10. Be dry-runnable тАФ every `db.edit` / `db.del` / Cypher write gates on a `dry_run` flag so the function can be tested against live data without mutating it

Inputs follow the same shape for both:

```ts
type MergeInput = {
  survivor_id: number;          // master_person_id or master_company_id we keep
  duplicate_ids: number[];      // array of master IDs that collapse into the survivor
  initiated_by: 'user' | 'cron' | 'admin';
  merge_suspect_id?: number;    // when triggered from merge_suspect (#284)
  dry_run?: boolean;            // when true, every db.edit/db.del/Cypher write is logged but skipped
};
```

When `dry_run=true`, the function still writes a `merge_audit` row (with `status='dry_run'`) and
populates `pre_merge_snapshot` and `conflicts_resolved` exactly as it would for a real merge,
so the operator can inspect what *would* have changed. No FK repoint, dedup, graph mutation, or
row deletion happens. The full read path runs тАФ including FK count probes тАФ so the audit row's
counts reflect what's actually in the database.

## Graph merge cypher

The graph merge runs **after** every Xano FK is repointed but **before** the duplicate
`master_*` rows are deleted (so we still know which `node_uuid` to delete from the graph).

FalkorDB does not support APOC, so edge re-routing must enumerate relationship types. We have
~25 edge types in production тАФ see [Edges](/guides/ontology/edges) тАФ so the cypher template
below loops over them in the orchestrator.

### Step 1 тАФ Aggregate properties onto the survivor

<CodeGroup>
```cypher Person
MATCH (s:Person {uuid: $survivor_uuid})
UNWIND $duplicate_uuids AS dup_uuid
MATCH (d:Person {uuid: dup_uuid})

SET s.full_name      = coalesce(s.full_name, d.full_name),
    s.first_name     = coalesce(s.first_name, d.first_name),
    s.last_name      = coalesce(s.last_name, d.last_name),
    s.headline       = coalesce(s.headline, d.headline),
    s.bio_500        = coalesce(s.bio_500, d.bio_500),
    s.bio_2000       = coalesce(s.bio_2000, d.bio_2000),
    s.is_angel       = coalesce(s.is_angel, false) OR coalesce(d.is_angel, false),
    s.is_c_suite     = coalesce(s.is_c_suite, false) OR coalesce(d.is_c_suite, false),
    s.is_investor    = coalesce(s.is_investor, false) OR coalesce(d.is_investor, false),
    s.is_founder     = coalesce(s.is_founder, false) OR coalesce(d.is_founder, false),
    s.profiles       = CASE
      WHEN d.profiles IS NULL THEN s.profiles
      WHEN s.profiles IS NULL THEN d.profiles
      ELSE s.profiles + ',' + d.profiles
    END

RETURN s.uuid AS survivor_uuid, count(d) AS duplicates_seen
```

```cypher Company
MATCH (s:Company {uuid: $survivor_uuid})
UNWIND $duplicate_uuids AS dup_uuid
MATCH (d:Company {uuid: dup_uuid})

SET s.company_name   = coalesce(s.company_name, d.company_name),
    s.company_domain = coalesce(s.company_domain, d.company_domain),
    s.tagline        = coalesce(s.tagline, d.tagline),
    s.summary_500    = coalesce(s.summary_500, d.summary_500),
    s.summary_2000   = coalesce(s.summary_2000, d.summary_2000),
    s.is_public      = coalesce(s.is_public, false) OR coalesce(d.is_public, false),
    s.is_acquired    = coalesce(s.is_acquired, false) OR coalesce(d.is_acquired, false),
    s.is_school      = coalesce(s.is_school, false) OR coalesce(d.is_school, false),
    s.profiles       = CASE
      WHEN d.profiles IS NULL THEN s.profiles
      WHEN s.profiles IS NULL THEN d.profiles
      ELSE s.profiles + ',' + d.profiles

```
</CodeGroup>

After this query returns, normalize comma-separated `profiles` (split тЖТ trim тЖТ unique тЖТ join) in
a follow-up `SET` to keep the column compact. The pattern is borrowed from
`mvp/dupe/merge-company-nodes` (#10505) but generalized to all string-list properties.

### Step 2 тАФ Re-route edges per relationship type

For each edge type, run the template below with `$rel_type` substituted in. The orchestrator
loops over the edge type list from [Edges](/guides/ontology/edges).

```cypher
// $rel_type is substituted in by the template engine тАФ Cypher does not allow parameterized rel types
MATCH (d:Person)-[r:KNOWS]-(other)
WHERE d.uuid IN $duplicate_uuids AND other.uuid <> $survivor_uuid

WITH d, other, r, properties(r) AS props,
     CASE WHEN startNode(r) = d THEN 'out' ELSE 'in' END AS direction

MATCH (s:Person {uuid: $survivor_uuid})

// Branch on direction so we preserve edge orientation
FOREACH (_ IN CASE WHEN direction = 'out' THEN [1] ELSE [] END |
  MERGE (s)-[new_r:KNOWS]->(other)
  SET new_r += props,
      new_r.merged_from = coalesce(new_r.merged_from, []) + d.uuid
)
FOREACH (_ IN CASE WHEN direction = 'in' THEN [1] ELSE [] END |
  MERGE (other)-[new_r:KNOWS]->(s)
  SET new_r += props,
      new_r.merged_from = coalesce(new_r.merged_from, []) + d.uuid
)

DELETE r
RETURN count(*) AS edges_rerouted
```

**Per-edge-type aggregation rules** when the `MERGE` lands on an existing edge:

| Edge type | Aggregation |
| --- | --- |
| `KNOWS` | Apply `known/ingest/append_evidence` (#12779) semantics: union evidence array, max confidence, min `known_since`, max `last_evidence_at` |
| `HAS_EXPERTISE` | Apply `mvp/expertise/upsert-person-has-expertise` (#12925) semantics: max confidence, strongest depth, most current recency, pipe-concat evidence |
| `WORKED_AT`, `EDUCATED_AT` | Keep both edges if `start_date` differs; merge if same `start_date` (sum tenure-derived weight) |
| `INVESTED_IN`, `RAISED` | Keep both тАФ investments are time-stamped events, not duplicates |
| `REGISTERED_FOR`, `ATTENDED` | Dedup on target uuid (one event per person) |
| `WON_AWARD`, `NOMINATED_FOR` | Dedup on target Award uuid |
| Any other relationship | Default: `SET new_r += props` (last write wins on scalar props) |

### Step 3 тАФ Detach delete the duplicate nodes

```cypher
MATCH (d:Person)
WHERE d.uuid IN $duplicate_uuids
DETACH DELETE d
RETURN count(d) AS nodes_deleted
```

For company side, swap `:Person` тЖТ `:Company`.

### Step 4 тАФ Re-sync the survivor node

After the merge, call `mvp/node/update-person-node` (#2758) for person or the equivalent
`mvp/node/add-update-company-node` for company. This recomputes the embedding from the now-merged
`master_person` / `master_company` row and re-applies any dynamic labels (e.g. `:Angel`, `:C_Suite`).

## Idempotency & safety guarantees

1. **Idempotent**: re-running `merge/person-v2` with the same `(survivor_id, duplicate_ids)` after a
   successful merge is a no-op тАФ the duplicate IDs no longer exist, so all `WHERE id IN duplicate_ids`
   queries match zero rows. The orchestrator should detect this and return `{status: 'already_merged'}`
   instead of failing.
2. **Atomic per-table**: each table's repoint+dedup runs inside a single transaction. The whole merge
   is *not* one mega-transaction (Xano can't hold one open across 80+ tables) тАФ instead, the
   `merge_audit` snapshot is the rollback point. A `merge/rollback` function uses the snapshot to
   un-merge.
3. **Fail-loud**: any FK constraint violation or unexpected row count writes a `log_crash` row and
   sets `merge_audit.status = 'failed'`. No silent partial merges.
4. **Survivor lock**: while a merge is in flight, the survivor + duplicate IDs are written to a
   `merge_lock` table. Other writers (enrichment crons, user actions) check this table and either
   wait or skip.

## Conflict resolution (decided)

Scalar field conflicts (survivor and a duplicate both have non-null values that disagree) split
into two tiers:

### High-stakes fields тАФ surface to the user when interactive

- `headline`, `bio_500`, `bio_2000`, `tagline`, `summary_500`, `summary_2000`
- `name`, `first_name`, `middle_name`, `last_name`, `suffix`, `nickname`
- `legal_name` (company)
- `domain`, `linkedin_url`, `twitter_url`, `website_url` (company)

When a **user-initiated** merge hits a conflict on any of these, the merge orchestrator pauses
and surfaces a side-by-side conflict-resolution UI. The user picks per field, or accepts the
"longest non-null" auto-pick default. Then the merge proceeds with the user's choices.

### Everything else тАФ auto-pick silently

All other scalar fields auto-pick by the rules in the [scalar aggregation tables](#scalar-field-aggregation-rules-master-person-table-139)
without surfacing anything.

### Cron-initiated merges тАФ never block on a human

Cron paths (deterministic exact-match via `mvp/merge/master-merge` #12736, future scheduled
re-merges of high-confidence suspects) auto-pick high-stakes fields too, AND write each
overridden field to `merge_audit.conflicts_resolved[]`:

```json
{
  "conflicts_resolved": [
    {
      "field": "headline",
      "winner_value": "Founder & CEO at Acme",
      "winner_source_master_id": 12345,
      "loser_values": ["Co-founder, Acme Inc."],
      "loser_source_master_ids": [67890],
      "rule_applied": "longest_non_null"
    }
  ]
}
```

A "review recent auto-merges" feed reads from `merge_audit` where `conflicts_resolved` is
non-empty, letting the user override after the fact (the override writes a follow-up
`merge_field_override` row but does NOT re-run the full merge).

### Confidence escape hatch тАФ deterministic matches skip even the logging

When the merge was triggered by a deterministic exact-match (same email, normalized phone,
canonical domain, or LinkedIn URL), the records are by definition the same entity. Conflicts
on scalar fields are just stale data from one source тАФ auto-pick "longest non-null" without
writing to `conflicts_resolved`. This keeps the after-the-fact review feed focused on actually
ambiguous merges.

The trigger source is recorded in `merge_audit.initiated_by` plus the
`merge_audit.match_kind` column. Values fall into three classes (orchestrator skips
`conflicts_resolved` logging only for the deterministic class):

| Class | Values | Conflict logging |
| --- | --- | --- |
| Deterministic | `'deterministic'`, `'email_exact'`, `'phone_exact'`, `'domain_exact'`, `'linkedin_exact'` | Skipped тАФ auto-pick is by definition correct |
| LLM | `'llm_confirmed'`, `'llm_flash_confirmed'`, `'llm_pro_confirmed'` (Tier 1/2 from `mvp/merge/detect-suspects` #12709) | Logged to review feed |
| User | `'user_picked'`, `'manual_inspection'` (human picked from suspect UI or admin tool) | Logged to review feed |

The skip rule is implemented in both orchestrators as a `$is_deterministic_match` predicate;
extending the deterministic class is a 1-line change in each orchestrator.

## Suspect lifecycle after merge (decided)

When a merge completes, the orchestrator finalizes `merge_suspect` (284) in three steps:

1. **Repoint** тАФ `UPDATE merge_suspect SET master_person_id = survivor_id WHERE master_person_id IN duplicate_ids` and same for `merge_with_master_person_id` (or the company-side equivalent FKs once `kind` is added).
2. **Delete collapsed pairs** тАФ after repoint, delete any row where both sides now equal `survivor_id`. The pair is meaningless (the suspect was for two records that are now one).
3. **Flag for re-score** тАФ on every remaining row touching the survivor, set `needs_rescore=true`. A cron (new or extension to `mvp/merge/detect-suspects` #12709) picks up flagged rows and re-runs Tier 1 / Tier 2 scoring just for the affected pairs. The new score writes back to the same row and clears the flag.

UI behavior while `needs_rescore=true`: hide the row from the active suspect feed, or show with a "re-evaluating" tag. Don't let users action a stale-score suspect.

**Why deferred re-score, not synchronous**: scoring at merge time would burn Gemini Flash + Pro tokens on every merge тАФ most of which are routine deterministic matches that don't need their downstream suspects re-evaluated. Deferring lets the cron batch them.
