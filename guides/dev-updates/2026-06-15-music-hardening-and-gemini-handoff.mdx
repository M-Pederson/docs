## 🎯 NEXT SESSION — Add Gemini `embeddings` (3072) to Person / Company

**Why:** business-side `Person` / `C_Suite` / `Company` nodes (created by the people/company pipeline) carry **`name_embedding` (OpenAI `text-embedding-3-small`, 1536)** but **not** the Gemini **`embeddings` (`google/gemini-embedding-001`, 3072)**. The [dual-embedding migration](/guides/enrichment/waterfall/music-nodes#embeddings-per-node-type) (2026-06-14) was **scoped to the 10 music node types only** — it never extended to the business graph.

**Verified evidence (sandbox):** 12/12 business `Person` nodes have `name_embedding`, **0/12** have the Gemini `embeddings`. The Cinq C-suite (Diana Schweinbeck, Karin Harvey, Jason Peterson, Steven Scott, Barry Daffurn) are all `name_embedding ✓` / `embeddings ✗`. *(Note: `vecf32` vectors don't render in graph browsers — verify via `RETURN n.name_embedding` through `tool/run-cypher` #8665, not the node-inspector panel.)*

**The task — two halves:**

1. **Forward path (new nodes).** Find where the people/company pipeline writes `name_embedding` (it calls `vectors/create-vectors-string` #4676 today) and wire the Gemini equivalent **`vectors/create-gemini-vectors` #13114** alongside it, writing `n.embeddings = vecf32(...)`. Likely homes to check: `mvp/node/update-person-node`, the company node writer (e.g. `update-company-node`), or whatever helper sets the person/company graph vector. **Use the SAME embed text** that `name_embedding` was built from (find it first — probably the person/company name, possibly + title) so the two vectors are comparable.
2. **Backfill (existing nodes).** Either **widen `mvp/music/backfill-node-embeddings` #13115's label set** to include `Person` + `Company` (it currently selects only the 10 music labels) — being careful with the embed-text branch (music uses `description` for Recording/Music_Group else name+disambiguation; business should use the people-pipeline text) — **or** write a parallel people-side backfill. Gate on `embeddings IS NULL`, same as music.

**Gotchas / decisions:**
- The **`:Entity(embeddings)` 3072 vector index is DEFERRED** (written unindexed until the OpenAI `name_embedding` is deprecated/removed) — so no index work needed; just write the property.
- Pin the **embed text per type** (Person: name? name + current_title? ; Company: name? + domain?) — match the `name_embedding` source.
- **Scale check** before a live backfill: live has ~6,291 `master_person` rows + companies; `tool/run-cypher` can count `Person`/`Company` nodes with `embeddings IS NULL`.
- The XanoScript embedding traps still apply ([#8](/guides/enrichment/waterfall/music-functions#node-embeddings)): both fns return a JS **array** in lambda context → `JSON.stringify` into `vecf32(...)`.

## Operational notes (carried)

- **Sandbox is header-only:** every call needs `X-Data-Source: sandbox` (+ `X-Branch: v1`); omitting it writes to **live**. The FalkorDB graph is a **separate system** from the Xano data source — a data-source table wipe does **not** clear the graph (needs a Cypher `DETACH DELETE`).
- **Curl response-timeouts during long batches are client-side** — Xano completes the enrich server-side, so verify with a graph query, not the curl exit code.
- **A `Recording` is fully enriched** (length/ISRCs/cover-art + credit edges) only via a direct `seed/recording` or a queue drain; a `seed/release-group` leaves title-only partials.
- Crons **#180** (MB queue), **#182** (person-resolution), **#181** (edge descriptions) all ship **inactive**.
