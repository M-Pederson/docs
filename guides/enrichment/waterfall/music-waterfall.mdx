Queue-driven MusicBrainz enrichment with a tiered cascade, depth cap, loop prevention, and a **global 1 req/s rate limiter** — pulling artists, groups, labels, recordings, release-groups, releases, works, events, and places into the Orbiter graph, with identity bridges to `master_person` and `Company`. This page is the architecture spec: source contract, rate-limit design, depth model, tiers, seed flows, loop prevention, and the resolved design decisions. The live function inventory, table schemas, and graph node \+ edge references live on the companion pages below.

It mirrors the [IMDB Waterfall](/guides/enrichment/waterfall/imdb-waterfall) — queue tables, depth-capped cascade, priority tiers, upsert dedup — with two structural differences: MusicBrainz is a **structured JSON API** (no HTML scrape, no LLM extraction in the structural path), and it enforces a **hard 1 request/second global limit**, which becomes the central design constraint.

## Source contract — the MusicBrainz API

| Property | Value |
| --- | --- |
| Base | `https://musicbrainz.org/ws/2/` |
| Format | append `?fmt=json` (or `Accept: application/json`) |
| Auth | none required |
| **Rate limit** | **1 request/second, global**, with a meaningful `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`). Exceeding it returns `503`. |
| Lookup | `/{entity}/{MBID}?inc={subqueries}` |
| Browse | `/{result}?{linked}={MBID}&limit=100&offset=N&inc=...` |
| Search | `/{entity}?query={lucene}&limit=N` |
| Pagination | `limit` (default 25, **max 100**) \+ `offset`; increment by results returned |

**`inc=` is the efficiency lever.** A single lookup can pull aliases, tags, genres, ratings, and every relationship type (`artist-rels`, `recording-rels`, `work-rels`, `release-rels`, `label-rels`, `event-rels`, `place-rels`, `area-rels`, `url-rels`) in **one** request. One artist \+ two browse pages ≈ 3 requests ≈ 3 seconds. This is why the cascade must be tightly depth- and tier-capped.

The per-orchestrator `inc=` sets (the exact request each entity type makes) are on [Music Waterfall Functions → Orchestrators](/guides/enrichment/waterfall/music-functions#orchestrators).

## Rate-limit architecture

The defining constraint — **the 1 req/s ceiling is global, not per-cron**. Independent queue crons each firing lookups would collectively blow the limit, so every MusicBrainz call passes through one token bucket, and **one** cron drains all queues so total throughput never exceeds 1 req/s.

- **`mb/rate-limit/acquire` (13079)** — the fail-open token gate backed by the singleton `mb_rate_limit` row (#736).
- **`mb/api/request` (13080)** — the single choke-point for every MusicBrainz call: pacing via `acquire`, canonical `User-Agent`, `fmt=json`, the [retry policy](#retry-policy) below. Orchestrators never call `api.request` directly.
- **`mb/queue/process-next` (13102) \+ Task 180** — **one unified cron**, not six. Each `process-next` call selects the single best pending **tier-0** row across all six queues (order `cascade_depth ASC, entity_rank ASC, id ASC`; nonzero tiers wait for [duplicate-discovery promotion](#priority-tiers)), locks it, dispatches the matching orchestrator, deletes on success. Task 180 runs every **60 s and makes up to 10 such calls per tick** (Xano cron granularity is 60 s; the design's ~2 s tick — see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals)). A unified cron makes the global 1 req/s trivially enforceable; parallel crons would not.

Behavioral detail (sleep timings, retry internals, drain backoff): [Foundation](/guides/enrichment/waterfall/music-functions#foundation) and [Unified queue drain](/guides/enrichment/waterfall/music-functions#unified-queue-drain).

<Warning>
  **Throughput is intentionally low.** At ~1 entity / 3–6 s, a depth-0 artist with its top recordings \+ albums \+ members takes minutes, and a deep cascade takes hours — by design. The depth cap (2), tier caps, and per-entity caps (20 albums, 50 recordings, 3 browse pages) keep total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.
</Warning>

## API errors, retries & rate limits

<Warning>
  **A meaningful `User-Agent` is mandatory.** Requests without one (or with a generic library default) are throttled hard or blocked. Use `Orbiter/1.0 ( mark@orbiter.io )`. There is no API key.
</Warning>

### Status codes

| Status | Meaning | Retry? | Handling |
| --- | --- | :-: | --- |
| `200` | OK | — | parse JSON |
| `400` | Bad request — malformed MBID / Lucene query | ❌ terminal | `log_crashphase: api_bad_request`; drop the queue row |
| `404` | Entity not found / merged away | ❌ terminal | stamp `enrich_success = false`, mark stub `gone`, drop row |
| `429` / `503` | **Rate limited / temporarily unavailable** | ✅ backoff | honor `Retry-After`; exponential backoff — the common case |
| `500` / `502` / `504` | Server / gateway error | ✅ backoff | transient — retry with backoff |
| timeout / network | no response | ✅ backoff | retry with backoff |

<Note>
  MusicBrainz returns **`503` for both** genuine overload **and** rate-limit violations (it does not consistently use `429`). Treat `503` as "back off and retry," never as a hard failure.
</Note>

### Retry policy

```text
max_attempts   = 5
base_delay     = 1000 ms                       # ≥ the 1 req/s floor
backoff        = base_delay * 2^(attempt-1)    # 1s → 2s → 4s → 8s → 16s
jitter         = ± 250 ms                       # avoid herd on the shared cron
respect        = Retry-After header when present (overrides computed delay)
terminal       = {400, 404}                     # never retried
on_exhaustion  = release lock · log_crash phase="api_exhausted" · re-queue one tier lower
```

- **Live implementation uses bounded sleeps inside the API helper.** `mb/rate-limit/acquire` sleeps when the global 1 req/s floor has not elapsed, and `mb/api/request` sleeps between its internal retry attempts. Queue-level failures use `next_attempt_at`, so the unified cron skips rows whose enrich retry window is in the future (5 min × retries; dropped after 5 failures).
- **Backoff is per-entity; pacing is global.** The 1 req/s token bucket governs _spacing_ between any two calls; the API retry backoff governs _re-attempts_ of one failed HTTP call; the queue backoff governs failed enrich rows. All three invariants hold simultaneously.
- **Retries are idempotent.** Every write is `MERGE`-based with `coalesce`d `created_at`, so re-running a partially-applied enrich never duplicates nodes or edges.

## Core entities → graph nodes

MusicBrainz's 13 core entities map to Orbiter graph nodes as follows. **Seed entities** (minimum bootstrap set) are **artist, recording, label, event**; "cascade" entities get their own queued lookup; inline entities/values are handled from data embedded in other lookups — no queue, no extra API call.

| MusicBrainz entity | Graph node label | Seed? | Bridges to existing Orbiter node |
| --- | --- | :-: | --- |
| `artist` (type = Person) | `Entity:Person:Music_Artist` | ✅ | Person-shaped music node; depth-0 person-resolution bridges to canonical `master_person` via `SAME_AS` |
| `artist` (Group / Orchestra / Choir) | `Entity:Music_Group` | ✅ | — (new node type — a band is never a `Person`) |
| `recording` | `Entity:Recording` | ✅ | — |
| `label` | `Entity:Company:Music_Label` | ✅ | Company-shaped; depth-0 `SAME_AS` to canonical `Company` \+ `OWNED_BY` owner resolution |
| `event` | `Entity:Event` (\+ `Music_*` type sub-label) | ✅ | Shared Event model |
| `work` | `Entity:Work` (\+`:Song` etc.) | cascade | — |
| `release-group` | `Entity:Release_Group` (\+`:Album`/`:Single`/`:EP`) | cascade | — |
| `release` | `Entity:Release` | cascade | — |
| `place` | `Entity:Place` (\+`:Studio`/`:Venue`) | inline | — |
| `area` | reuse `Country` / `Region` / `City` (MERGE by name; stamp `mbid`) | inline | Orbiter geography |
| `genre` | property on source music node | inline | — |
| `series` / `instrument` / `url` | properties / deferred | — | — |

Every node follows Orbiter conventions: base `Entity` label, `uuid` (random) **plus `mbid`** as the natural MERGE key, `display_label`, `visibility`, dual vectors, and `created_at` / `updated_at` stamped via Cypher `timestamp()`. The pipeline-scoped catalog (keys, writers, vector/description coverage) is on [Music Nodes](/guides/enrichment/waterfall/music-nodes); canonical schemas are in [Ontology → MusicBrainz Nodes](/guides/ontology/nodes#musicbrainz-nodes).

## Depth model

Identical to IMDB — hard cap at depth 2.

| Depth | Meaning | Behavior |
| :-: | --- | --- |
| **0** | Seed (user / bootstrap) | Full lookup \+ all edges \+ cascade fanout |
| **1** | Direct discovery from a seed | Stub via `get-add`. Queued. Cron enriches \+ expands, but **only T1–T2 children queue at depth 2** |
| **2** | Transitive | Stub queued. Cron enriches \+ writes edges, **no further cascade** |
| **3\+** | Never queued | Hard cap |

<Check>
  **Why cap at 2?** MusicBrainz is ~2.5 M artists / ~30 M recordings. One prolific artist → ~20 albums → ~250 recordings → each recording → 1 work \+ ~5 credited people → thousands of entities at depth 2. Combined with the 1 req/s limit, an uncapped cascade would never drain. Depth 2 \+ tier caps \+ per-entity caps (20 albums, 50 recordings) keep a seed's full expansion in the low thousands of API calls.
</Check>

## Priority tiers

<Note>
  **Tier 0 is the reserved bootstrap lane** ([decision 5](#resolved-design-decisions-2026-06-11)) — manual seed batches queue at `priority_tier = 0` and drain ahead of all cascade traffic. The tiers below (1–4) apply to cascade-discovered entities.

  **Drain eligibility is tier-0-only (v1.2, 2026-07-01).** The unified drain (13102) picks only `priority_tier = 0` rows. Cascade rows enter at tiers 1–4 and are **promoted one step toward 0 on each duplicate discovery** (`priority_tier = max(0, min(existing_tier, incoming_tier) - 1)`), so repeatedly-referenced entities earn their drain slot. Tier assignment below still determines how fast that happens.
</Note>

### `queue_mb_artist` (1 = highest)

| Tier | Rule |
| :-: | --- |
| **1** | Band member of a seed group; main performer on a seed recording/event; composer of a seed work |
| **2** | Collaborator; producer/engineer on a seed recording; supporting musician |
| **3** | Co-performer at a seed event; artist credited on a label-discovered release _(built: #13091 queues label-credited artists at **T2** — see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals))_ |
| **4** | Everyone else mentioned |

### `queue_mb_recording` (1 = highest)

| Tier | Rule |
| :-: | --- |
| **1** | Recording where the seed artist is the **main** performer; title track of a seed release-group _(built: #13099 queues **all** tracklist recordings at **T3** — see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals))_ |
| **2** | Other recordings credited to the seed artist |
| **3** | Recording on a release discovered via a seed label |
| **4** | Low-signal (alternate takes, untitled, video) |

### `queue_mb_release_group` / `queue_mb_event`

| Tier | Release-group rule | Event rule |
| :-: | --- | --- |
| **1** | Album by seed artist; award-winning | Event the seed artist **headlined** |
| **2** | EP / single by seed artist | Event the seed artist performed at (support) |
| **3** | Compilation / live / soundtrack | — |
| **4** | Other appearance | — |

### Cascade filter — artist seed

```text
from artist lookup + browse:
    member of band (each)                          → T1 (artist)
    release-group type=Album (cap 20 by date)      → T2 (release-group)
    recording main-credit (cap 50, as returned)    → T2 (recording)
    collaborator / supporting musician             → deferred to v2
    event main performer (depth-0 seeds only)      → T3 (event)
    # depth ≥ 1: keep only T1–T2
```

### Cascade filter — recording seed

```text
from recording lookup:
    work (performance)                 → T1 (work)
    main performer artist-credit       → T1 (artist)
    producer / engineer / instrument   → T2 (artist)
    recorded-at place                  → Place node inline (no queue)
```

The built per-orchestrator fanout (which children each entity type queues, at which tier, with which caps) is on [Music Waterfall Functions → Orchestrators](/guides/enrichment/waterfall/music-functions#orchestrators).

## Queue & upsert semantics

- **Natural key is the `mbid`** — a stable UUID straight from the API. No URL normalization needed (unlike IMDB's `imdb_url` keying).
- **Queue upsert is the single source of truth** (`mb/queue/upsert-*`, 13081–13086): duplicate discovery → `count++` \+ tier promotion \+ FK fill once the stub exists. Source metadata is set on INSERT only. One row per MBID, enforced by a unique index.
- **`first_notnull` defaults are load-bearing.** Upsert inputs default via `first_notnull` (tier 4 / depth 1), so an explicitly passed `0` survives — this is what lets bootstrap-lane rows (`priority_tier = 0`, `cascade_depth = 0`) queue without the defaults coercing them.
- **30-day dedup gate:** `mbid` already enriched within 30 days → the gate returns the existing row, no lookup. `gone`-tombstoned rows (404) are never re-queued.
- **Sync-enrich branch:** `cascade_depth == 0 && queue == false` invokes the orchestrator directly; otherwise the row drains through the cron.
- **`entity_rank`** biases identity-rich entities ahead of bulk catalog in the unified drain (artist 1 · label 2 · event 3 · work 4 · release\_group 5 · recording 6) — see [Music Waterfall Tables](/guides/enrichment/waterfall/music-tables#unified-cron-ordering--entity_rank).

Full schemas, indexes, and conventions: [Music Waterfall Tables](/guides/enrichment/waterfall/music-tables).

## Seed flows

### Seed artist (depth 0)

<Steps>
  <Step title="Dedup gate">
    Lookup `mb_artist` by `mbid`, return if enriched \< 30 days. (No kill-switch check — skipped by design; pause the unified cron to stop the pipeline.)
  </Step>
  <Step title="Throttled lookup">
    `acquire` token → `GET artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels+event-rels`. Save raw JSON.
  </Step>
  <Step title="Node + base edges">
    `MERGE` the artist node on `mbid` — `Entity:Person:Music_Artist` (person) or `Entity:Music_Group` (group) by `type` — stamp `created_at`/`updated_at`, dual vectors, and media/description where the finalizer applies. No inline reconciliation here — depth-0 person artists enter the person-resolution bridge.
  </Step>
  <Step title="Membership edges">
    For each `member of band` rel → `MEMBER_OF` edge (instruments/dates/original) \+ queue the member artist at **T1**.
  </Step>
  <Step title="Cascade (browse, throttled, paginated)">
    Queue children per the [artist cascade filter](#cascade-filter--artist-seed) above, each via `get-add` at `depth+1`. Depth-0 person artists also enqueue [person-resolution](/guides/enrichment/waterfall/music-functions#person-resolution) and fetch a serper avatar inline.
  </Step>
  <Step title="Mark success">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

### Seed recording / label / event

Same shape: 1 throttled lookup with the entity's `inc=` set → `MERGE` node → write its edges → queue discovered neighbors at the [tiers above](#priority-tiers) → stamp success. A label additionally browses `release?label=` (capped 3 pages / 300 releases) and, at depth 0, runs the **Company bridge** (label→`Company` `SAME_AS` \+ [owner resolution](/guides/enrichment/waterfall/music-functions#label--company-bridge--owner-resolution) → `OWNED_BY`); an event reads `artist-rels` \+ `place-rels`. Per-orchestrator phase detail: [Music Waterfall Functions → Orchestrators](/guides/enrichment/waterfall/music-functions#orchestrators).

### Depth-1 / depth-2 queued processing

When the cron picks up a queue row it locks it (`processing: true`) and dispatches the matching orchestrator at the row's `cascade_depth`. Depth 1 runs all phases, but **only T1–T2 discoveries queue at depth 2**. Depth 2 enriches and writes edges but **cascades nothing**. Depth guarding is inlined in each orchestrator's `mb_cascade` phase: `cascade_depth + 1 >= 3` → no queue.

## Cascade example — seed artist end-to-end

Here's what happens end-to-end when an artist enters as a depth-0 seed — the seed flow, the [artist cascade filter](#cascade-filter--artist-seed), the unified drain, and the depth model above, restated as one tree:

```text
Depth 0: seed artist (mbid)
├── Dedup gate (get-add)
│   ├── mbid enriched < 30 days → return existing, no lookup
│   └── no kill-switch check (by design — pausing the unified cron is the manual kill)
├── Throttled lookup — acquire token (global 1 req/s), then
│   GET artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels+event-rels
│   → save raw JSON
├── Node + base edges
│   └── MERGE on mbid → Entity:Person:Music_Artist (person) or Entity:Music_Group
│       (group) by type + created_at/updated_at, dual vectors, media/description
│       where the finalizer applies
├── Membership: each "member of band" rel
│   → MEMBER_OF edge (instruments/dates/original) + member artist queued at T1
├── Cascade fanout (browse, throttled, paginated; each child via get-add at depth+1)
│   ├── release-group type=Album (cap 20 by date)    → T2 (queue_mb_release_group)
│   ├── recording main-credit (cap 50, as returned)  → T2 (queue_mb_recording)
│   ├── event main performer (depth-0 seeds only)    → T3 (queue_mb_event)
│   └── collaborator / supporting musician           → deferred to v2, not queued
├── Depth-0 person artists only
│   ├── enqueue person-resolution → SAME_AS (weight 1) bridge to canonical
│   │   master_person + verified social links
│   └── fetch a serper avatar inline
└── Mark success: enriched_at = now · enrich_success = true · cascade_depth = 0

Unified drain — Task 180 (every 60 s, ≤ 10 process-next calls/tick; ships INACTIVE)
├── picks the single best pending priority_tier = 0 row across all six queues
│   (order: cascade_depth ASC, entity_rank ASC, id ASC —
│    identity-rich entities ahead of bulk catalog)
├── cascade rows enter at T1–T4 and wait for duplicate-discovery promotion:
│   each re-discovery → count++ + priority_tier = max(0, min(existing, incoming) - 1)
│   (one row per MBID, enforced by a unique index)
└── lock (processing: true) → dispatch the matching orchestrator at the row's
    cascade_depth → delete row on success

Depth 1: queued discovery (e.g. a T1 band member)
├── full lookup + node + edges (all phases run)
└── cascade: only T1–T2 discoveries queue, at depth 2
    (events are depth-0-only and are not queued from here)

Depth 2: transitive stub
├── enrich + write edges
└── cascades NOTHING — mb_cascade guard: cascade_depth + 1 >= 3 → no queue;
    depth 3+ is never queued
```

**API-call pacing:**

- Every MusicBrainz call in this tree passes through the global 1 req/s token bucket (`mb/rate-limit/acquire` \+ the single unified cron), so total throughput never exceeds 1 req/s no matter how many queue rows are pending.
- One artist \+ two browse pages ≈ 3 requests ≈ 3 seconds — effective throughput is ~1 entity / 3–6 s.
- That is why a depth-0 artist with its top recordings \+ albums \+ members takes **minutes**, and a deep cascade takes **hours** — by design.
- The depth cap (2), tier caps, and per-entity caps (20 albums, 50 recordings, 3 browse pages) keep a seed's total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.

## Loop prevention (six layers)

1. **Dedup gate in `get-add`:** `mbid` already enriched within 30 days → return existing, no lookup.
2. **Queue upsert:** duplicate `mbid` → `count++`, tier promotion. One row per MBID.
3. **Depth cap:** depth 2 enriches but does not cascade; depth 3 never queued.
4. **Tier cap at depth ≥ 1:** only T1–T2 discoveries queue at depth 2.
5. **Manual kill (no dedicated switch — decided 2026-06-11):** layers 1–4 make the cascade structurally non-recursive (depth is monotonic with a hard cap; cycles collapse into `count++` on existing rows), so the planned `check-kill-switch-mb` env-var gate was skipped. Pausing the single unified cron task in Xano stops the entire pipeline.
6. **Global rate limiter:** `mb/rate-limit/acquire` \+ the unified cron enforce 1 req/s across the whole pipeline (the layer IMDB doesn't need).

## Resolved design decisions (2026-06-11)

The open questions from the original design, each resolved before build:

<CardGroup cols={1}>
  <Card title="Artist ↔ Person reconciliation — separate post-ingest pass" icon="link">
    **Decided; built.** Ingest creates `Person:Music_Artist` and `Company:Music_Label` nodes keyed on `mbid` — no reconciliation inline. The built person-resolution path (#13118–#13123) bridges depth-0 person artists to a canonical `master_person` with `SAME_AS` and verified social links; label-owner resolution (#13132–#13134) writes `OWNED_BY`. Label→Company `SAME_AS` runs in the depth-0 label orchestrator. Fixed-forward 2026-07-01: both live `SAME_AS` writers use identity weight `1`.
  </Card>

  <Card title="Area mapping — reuse Orbiter geography" icon="earth-americas">
    **Decided.** MusicBrainz areas MERGE into the existing `Country` / `Region` / `City` nodes by name, stamping `mbid` on match. No dedicated `Music_Area` node type and no mapping table. Unmappable or historical areas fall back to the nearest modern parent (e.g. a historical region maps to its present-day country). _(Forward-path gap: `FROM_AREA` is not yet wired into the audited ingest path.)_
  </Card>

  <Card title="Genre vs tag — genres as properties" icon="tags">
    **Decided.** Curated `genres` and free-form `tags` stay as node properties for search, filtering, facets, and badges. Do not create `Genre` hub nodes or `HAS_GENRE` edges.
  </Card>

  <Card title="Canonical release — one Official release per release-group" icon="compact-disc">
    **Decided.** Each `Release_Group` materializes exactly **one** `Release` node: normally `status = Official`, earliest release date (any country). If the release-group came from a seed label, an Official release on that seed label wins before the generic earliest-Official fallback. Tracklists and `APPEARS_ON` edges hang off that canonical edition. All other pressings remain in the raw JSON on the `mb_*` source tables, so the decision is reversible without re-fetching.
  </Card>

  <Card title="Bootstrap — live API only, priority lane" icon="gauge-high">
    **Decided.** No offline import from the [MusicBrainz database dumps](https://musicbrainz.org/doc/MusicBrainz_Database/Download). Large bootstraps (e.g. a full label roster) ride the same 1 req/s live API, queued at **`priority_tier = 0`** — a reserved bootstrap lane that drains ahead of all cascade traffic. One code path, no dump tooling; multi-day seed loads are accepted.
  </Card>

  <Card title="Places — inline materialization, no queue" icon="location-dot">
