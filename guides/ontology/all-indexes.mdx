Every index in the Orbiter FalkorDB graph — scalar (btree) and vector (HNSW) — is created by a single Xano function: **`tool/create-indexes`** — **#4669**. This page is the canonical, up-to-date list of what that function creates and why.

## `tool/create-indexes` — #4669

**Location:** Xano workspace 3, function `tool/create-indexes`, id `#4669`.

**Shape:** Top-level groups, each with a `util.template_engine` block holding one `CREATE INDEX` / `CREATE VECTOR INDEX` statement, piped through `mvp/falkor/send-cypher`. The final group calls `CALL db.indexes()` and returns the full index list for confirmation.

**Idempotency:** FalkorDB's `CREATE INDEX` is idempotent — re-running is safe. Run #4669 after provisioning a new FalkorDB instance or after a graph reset.

**How to run:**

```
Xano workspace 3 → functions → tool/create-indexes (#4669) → Run
```

No inputs. Returns the full `CALL db.indexes()` result as the response.

As of 2026-04-11, #4669 creates **14 indexes**: 12 scalar (btree) + 2 vector (HNSW).

### Node scalar indexes (6)

All on the base `Entity` label. Because every node in the graph carries `Entity` plus a type-specific label, these indexes accelerate lookups across every node type.

| Label | Property | Purpose |
|---|---|---|

| `Entity` | `imdb_url` | Film/TV dedupe by external IMDB URL |
| `Entity` | `region_key` | Region node lookup during location hydration |
| `Entity` | `city_key` | City node lookup during location hydration |
| `Entity` | `announced_date` | Funding round time-range filters (`WHERE earlierRound.announced_date < ...`) |

### Edge scalar indexes (6)

| Edge | Property | Purpose |
|---|---|---|
| `ATTENDED` | `valid_from` | Time-range queries on education history |
| `ATTENDED` | `valid_to` | Same — currently-attending filters |
| `ATTENDED` | `degree` | Filter by degree (e.g. "PhDs only") |
| `ATTENDED` | `major` | Filter by major |
| `ATTENDED` | `field` | Filter by field of study |
| `SPECIALIZES_IN` | `match_score` | Expertise quality-threshold filters on `Company → SubDomainExpertise` |

### Vector HNSW indexes (2)

Both use identical HNSW options: **`dimension: 1536, similarityFunction: 'cosine', M: 16, efConstruction: 200, efRuntime: 100`**. See [Vector Indexes](/guides/ontology/vector-indexes) for the full deep-dive on how these are queried.

| Label | Property | Purpose |
|---|---|---|
| `Entity` | `name_embedding` | Semantic lookup across all node types that carry a name embedding (Person, Company, Startup, School, City, Region, Country, Film_TV, etc.) |
| `Entity` | `expertise_embedding` | Expertise-specific semantic lookup on `DomainExpertise` / `SubDomainExpertise` |

The committed index list lives in [Known → Schema design → Indexes](/guides/known/schema-design#indexes).

Status: **materialized on 2026-04-11 via `tool/create-indexes` #4669.** 22 index-creation statements were added; FalkorDB materialized them as 23 property-level indexes (see the KNOWS composite note below), bringing the live total to **37**. Every Known label was accepted on a zero-cardinality graph — `status: "OPERATIONAL"`, `numDocuments: 0` — meaning Phase 1 ingest will be index-accelerated from the very first row written.

### KNOWS envelope edge (5 materialized — 4 requested, composite split)

| Edge | Property | Purpose |
|---|---|---|
| `KNOWS` | `person_a_id` | Half of the canonical pair lookup — see composite note below |
| `KNOWS` | `person_b_id` | Other half of the canonical pair lookup |
| `KNOWS` | `last_score_computed_at` | Differential scoring cron — only rescore edges stale beyond the watermark |
| `KNOWS` | `last_counts_updated_at` | Differential counts cron — same pattern, different cadence |
| `KNOWS` | `strength_score` | "High-strength edges" queries + BONDED inference threshold checks (`strength_score > 70`) + ranked "closest contacts" lists |

<Note>
  **Composite → split-property behavior.** #4669 ships a single composite statement: `CREATE INDEX FOR ()-[r:KNOWS]-() ON (r.person_a_id, r.person_b_id)`. The first run on 2026-04-11 confirmed that FalkorDB on Orbiter's live version does **not** materialize edge composites as a single index — it splits the request into two independent RANGE indexes (`range:person_a_id` and `range:person_b_id` on the KNOWS label). For our canonical pair lookup (`MATCH (a)-[r:KNOWS {person_a_id: $a, person_b_id: $b}]-(b)`) this is operationally equivalent: the planner intersects the two indexes. The composite is kept in #4669 source as the "declared intent" — if a future FalkorDB version supports true composite edge indexes it will upgrade automatically.

  **Note:** this index **accelerates** canonical pair lookup but **does not enforce** uniqueness. FalkorDB does not support unique constraints on relationships at all. `KNOWS` uniqueness — "at most one edge per canonicalized pair" — is enforced at the application layer by `canonicalize_pair` + directed `MERGE` in `create_knows_edge` + a nightly `dedupe_knows_edges` cron. See [Known → ingest → KNOWS uniqueness enforcement](/guides/known/scope/ingest#knows-uniqueness-enforcement) for the full three-layer defence.
</Note>

### Facet overlay edges (8 new)

| Edge | Property | Purpose |
|---|---|---|
| `BONDED` | `source` | Separate cron-touched (`inferred`) rows from user-touched (`declared`) rows. The scoring cron filters by `source='inferred'` |
| `ESTRANGED` | `source` | Same split — cron-vs-declared separation |
| `ESTRANGED` | `valid_to` | "Currently-valid estrangements" is the near-universal filter on this edge type; the penalty-to-KNOWS computation needs it on every scoring pass |
| `PROFESSIONAL_TIE` | `company_id` | Company-scoped tie queries ("Alice's coworkers at Acme") and the `mvp/edges/create-work-edges` trigger |
| `KINSHIP` | `relation` | Filter by `parent` / `child` / `sibling` / `in_law` / `step_*` — every co-parent, in-law, and `kinship_divorce_cascader` trigger needs this |
| `ROMANTIC` | `is_current` | Fast "Alice's current partner" lookup. `is_current` is maintained as an invariant alongside `valid_to` (see [schema-design constraints](/guides/known/schema-design#constraints)) |
| `RESIDED_AT` | `move_out` | Find currently-active residencies (`move_out IS NULL`). Used by household overlap queries and the deceased-freezer cascade that auto-closes residencies |
| `MENTOR` | `valid_to` | `mentor_invalidator` cron (nightly) scans for `valid_to IS NULL AND valid_from < now - 180d` to invalidate stale inferred mentorships. The cron explicitly excludes `domain='accelerator'` rows but still needs the `valid_to` scan to be index-accelerated |

### Not proposed (deliberately)

These were considered but held back from the initial addition:

- **`valid_to` on every facet** (BONDED, KINSHIP, ROMANTIC, PROFESSIONAL_TIE). The workshop committed only to `ESTRANGED.valid_to` and `MENTOR.valid_to`. Other facets currently get accelerated through `source` / `relation` / `is_current` / `company_id` + an implicit scan of the already-narrowed candidate set. If production scans start regressing we'll add the full `valid_to` set as a Tier 2 pass.
- **`Household.household_type`** — cardinality is too low (4–5 values) to justify an index.
- **`Person.name_tokens`** for the identity resolver. The resolver uses the existing `Entity.name_embedding` vector index for fuzzy name matching; a scalar `name_tokens` index would be a denormalization that duplicates the work.

## FalkorDB index gotchas

Rules that have bitten us in this codebase and should shape every new index proposal:

- **Not-equal filters (`<>`) are not index-accelerated.** Prefer positive predicates. Don't write `WHERE f.source <> 'inferred'`; write `WHERE f.source = 'declared'` instead.
- **`IS NULL` and `IS NOT NULL` ARE accelerated** by btree indexes. This is why indexing `valid_to` on ESTRANGED / MENTOR directly speeds up "currently valid" filters.
- **Composite indexes on edges are NOT materialized as single indexes** on the FalkorDB version Orbiter runs. A composite `CREATE INDEX FOR ()-[r:E]-() ON (r.a, r.b)` statement is accepted and silently split into two independent single-property indexes. This is why the KNOWS canonical-pair pseudocomposite shows up as `person_a_id` + `person_b_id` in `CALL db.indexes()`. Queries that filter on both properties still plan through the two indexes at intersection cost — acceptable for now, but re-verify when FalkorDB upgrades land.
- **Vector indexes require populated embeddings.** `CREATE VECTOR INDEX` against a label with no rows that have the embedding property is fine (the index is empty), but querying it with `db.idx.vector.queryNodes()` will return zero results until nodes are written.
- **Label-specific vs. `Entity`-base indexes.** Nodes carry `Entity` as the base label plus a type label (`Person`, `Household`, etc.). Properties that exist across all node types (`uuid`, `name`) index on `Entity`. Properties that only exist on one type (`Person.deceased_at`, `Household.address_hash`) index on the specific type label. Do not index `Entity.deceased_at` — it would force a full-table scan for a property 95% of rows don't carry.
- **Edge direction in index syntax.** Use undirected syntax `()-[r:EDGE]-()` for index creation regardless of whether the edge is semantically directed. FalkorDB treats edge indexes the same in both directions.
