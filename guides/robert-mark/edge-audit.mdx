## Scope

Audited ~50 exported edges across two FalkorDB pulls — HAS_SUBDOMAIN ontology edges plus WORKS_AT / FOUNDED / LOCATED_IN / IN_REGION / IN_COUNTRY / RAISED / INVESTED_IN entity edges. Inspection focuses on the `r.properties` object on the relationship itself (not endpoint node properties). Eleven distinct issues, three of them HIGH priority. Full findings appear in the [Audit findings](#audit-findings) section at the bottom.

Every issue below has a **fix-forward** action (change the edge-creation / update function so future writes are clean) and, where existing data is already polluted, a **backfill** action (sweep + repair sweep). Some issues are write-side only (e.g. a new field on creation); some are read-side only (e.g. cast `visibility` on already-written nodes).

## Fix-forward — edge creators / updaters

These are the Xano functions that write edges (or write node properties consumed by edges). Each card lists the issue, the function(s) to patch, and the change.

```
HIGH §3.1 — Domain-parse exception strings persisted as live data
```

**Problem.** Stealth Startup (master_company_id 5) has `company_domain: "[Line 16:Char 9] Error parsing domain: …"`. The exception text was interpolated into `logo` (logo.dev call with `pk_` token against a garbage path) and `website` (dead link).

**Function to patch.** The company-domain extractor — `mvp/get-add/master-company` (#12558) → its domain-parse helper `mvp/format/domain-extractor` (#4587). Wrap the parse in try / catch; on exception, write `null` (or empty string per existing field-empty convention), never the exception message.

**✅ Fixed-forward (2026-05-31).** Root cause was `mvp/format/domain-extractor` (#4587): it ran `psl.parse()` and **threw** `Error parsing domain: …` on bad input — Xano surfaced that thrown string back as the result (hence the `[Line 16:Char 9]` prefix = the `throw` line), and it was persisted into `company_domain` / `logo` / `website`. The extractor now returns `null` on any parse error or exception — never the message. Defensively, `mvp/get-add/master-company` (#12558) now nulls `logo` when there is no valid domain, so the logo.dev `pk_` token is never fired against a garbage path.

**Backfill required.** See §3.1 below.

```
HIGH §3.3 — Duplicate edges between same node pair across edge types
```

**Problem.** Carl Bressler → Unknown Company has two edges with identical title "Limited Partner": INVESTED_IN (id 8348, weight 15) and WORKS_AT (id 8642, weight 20). Target node is also an unresolved stub.

**Function to patch.** Two changes:

1. The edge-creator that decides `WORKS_AT` vs `INVESTED_IN` from a `work_experience` / employment row needs a precedence rule. Limited Partner / Angel Investor / Board Observer roles should go to INVESTED_IN only — never both.
2. The edge writer should also refuse to attach to a node whose `name == "Unknown Company"` and `company_domain == ""` — those are unresolved stubs and need company resolution to complete first.

**✅ Fixed-forward (2026-05-31).** In `mvp/edges/create-work-edges` (#2800): a new `isOperatingPartner()` helper distinguishes general/managing partners (employment → WORKS_AT eligible) from Limited Partners. `isWorksAtEligible()` now routes Limited Partner / Angel / Board Observer / Advisor titles to their specialized edge **only**, never WORKS_AT (the old `indexOf('partner') === -1` guard let "Limited Partner" slip through to both). The grouping loop also skips any company whose node is an unresolved stub (`company_name == "Unknown Company"` AND empty `company_domain`) — the `master_company` addon now returns `company_domain` to support this check.

**v1.2 refinement (2026-05-31).** Caught during the sandbox Carl test: "Advisor to CEO" / "Advisor to COO" were still producing a WORKS_AT because the C-suite keyword check (`['chief','ceo','cto','cfo','coo']`) matched the `ceo`/`coo` *substring* and ran *before* the advisor exclusion. Fixed by moving the advisory / investor / board / membership exclusions **ahead of** the loose title-keyword matching in `isWorksAtEligible()`. Founders and operating partners are still short-circuited to WORKS_AT first.

**v1.4 refinement (2026-05-31).** Caught during the graph-wide §3.3 dedupe dry run: the board-member exclusion's executive carve-out (`indexOf('executive') === -1`, meant to keep "Executive Chairman") also fired for **"non-executive director"** (substring "executive"), wrongly keeping non-exec directors as WORKS_AT. Tightened to `(executive present AND non-executive absent) OR managing` so non-exec directors route to BOARD_MEMBER_OF only. The dedupe (#13052) ports the same v2.1 classifier.

**Backfill required.** See §3.3 below.

```
MEDIUM §3.5 — `valid_from` conflates valid time with ingestion time + missing edge `created_at`
```

**Problem.** Edges have no `created_at`. Geography edges (LOCATED_IN, IN_REGION, IN_COUNTRY) stamp `valid_from: "2026-05"` (current ingestion month) instead of the true validity start. Bitemporal gap.

**Function to patch.** Every edge creator. Two additive changes:

1. Always write `r.created_at = now` on edge insert (epoch millis or ISO — pick one per §3.9). This is ingestion time.
2. Only write `r.valid_from` when an actual real-world validity date is known. For geography edges where validity = "since founding" or "since incorporation", either omit `valid_from` or source it from the node's `founded_year`.

**✅ Fixed-forward (2026-05-31) — created_at + updated_at.** Every edge creator now stamps both timestamps on write. The pattern, idiomatic to the existing `coalesce(r.uuid, randomUUID())` style:

```cypher
MERGE (a)-[r:REL]->(b)
SET r.created_at = coalesce(r.created_at, timestamp()),   // set once, at creation
    r.updated_at = timestamp(),                           // refreshed on every write
    ...existing props...
```

Canonical format is **epoch milliseconds** — FalkorDB's `timestamp()` — matching §3.9 and the FalkorDB skill. `CREATE`-style edges use `created_at: timestamp(), updated_at: timestamp()` in the inline map. Applied across: `create-work-edges` (#2800), `resolve-investors-edges` (#12702), `create-education-edges` (#1898), `location-nodes-edges-company` (#4528), `primary-location-nodes-edges_v2` (#4690), `resolve-edges-honor` (#4574), `resolve-edges-volunteering` (#4575 — also renamed camelCase `createdAt` → `created_at`), `resolve-edges-certifications` (#12578), `resolve-edges-projects-publications` (#12579), and `create_knows_edge` (#12778 — added the missing `updated_at`). `add-credit-edges-from-person/title` (#13013 / #13015) and `add-node-edge-imdb-award` (#2334) already stamped both. **Note:** the `valid_from` bitemporal cleanup (point 2 above) is *not* part of this pass — tracked as a follow-up.

**Backfill required.** Add `created_at` to every existing edge based on the earliest evidence we have (could fall back to the connected node's `created_at`).

</Card>

```
MEDIUM §3.7 — Union membership modeled as WORKS_AT
```

**Problem.** Carl Bressler → SAG-AFTRA: `WORKS_AT` with `title: "Actor / Artist"`, `seniority_level: "C Suite"`. Membership ≠ employment, and an actor isn't C-Suite.

**Function to patch.** Two changes:

1. The WORKS_AT edge creator needs a guard: if the company is tagged `is_union` (or matches a union allowlist — SAG-AFTRA, WGA, DGA, IATSE, etc.), emit `MEMBER_OF` instead.
2. The `seniority_level` classifier needs tighter rules — `C Suite` should only fire for `CEO / CTO / CFO / COO / Chief …` titles, not loosely across non-exec roles.

**✅ Fixed-forward (2026-05-31, v1.3).** In `mvp/edges/create-work-edges` (#2800): a new `isUnionCompany()` name-allowlist helper (SAG-AFTRA, Screen Actors Guild, Writers/Directors Guild, IATSE, AFM, Actors' Equity, Animation/Editors/Producers Guild, …) — when the company matches, the group emits `MEMBER_OF` and suppresses `WORKS_AT`. And `getSeniority()` now only tags `C Suite` for genuine exec titles (CXO as a whole word via `\b…\b`, or "Chief … Officer"/"Chief Executive/Technology/…"), explicitly excluding advisor / assistant / board / "to CEO" / deputy titles. (No `is_union` schema field is required — detection is name-based; add the allowlist entry if a new guild shows up.)

**Backfill required.** See §3.7 below.

```
LOW-MED §3.10 — Junk enrichment profiles on company nodes
```

**Problem.** Harmonic AI's `profiles` array includes `facebook.com/forbes`, `instagram.com/popular`, `crunchbase/akhenaten` — unrelated identities attached by profile-matching.

**Function to patch.** The profile-matcher (likely inside `mvp/get-add/master-company` or `mvp/link/create-master-link`). Add a domain-affinity check: a profile only attaches if (a) the URL's handle / slug matches the company name above some similarity threshold, OR (b) the profile URL's domain is the company domain. Discard otherwise.

**✅ Fixed-forward (2026-05-31) — root cause was the projection addon.** The Company node's `profiles` array is populated by `update-company-node` (#4659) via addon `master_link:profiles:master_company_id` (#690), which selected links by **`service`-domain only** — missing the `profile == true` filter that the person addon (#721) has. So any link on one of the 5 social domains surfaced, including `facebook.com/forbes`, `instagram.com/popular`, `crunchbase/<junk>`, and personal `linkedin.com/in/…` (all written with `profile = false`). Addon #690 now requires `profile == true` **and `master_person_id == null`** — so it returns only the company's *own* links, never an employee's personal profile (the sandbox detector found 41 person `linkedin.com/in/` links on companies, all `master_person_id`-set legitimate person→employer associations that were wrongly surfacing as company `profiles`; the `profile == true` guard alone didn't catch them because they're `profile: true`). `create-master-link` (#2541) only sets `profile = true` for company/school LinkedIn, x.com, and crunchbase URLs.

**Deeper follow-up (not done):** the *write* path (`apply-company-llm-payload` #12989 → `add-bulk-company-links` #2446) still stores the raw LLM `profiles` array (and Fundable links) into `master_link` with no affinity check — they're just now correctly flagged `profile = false` and filtered out of the node. A true domain/name-affinity gate on the write path remains a future enhancement.

**✅ Placeholder-company guard (2026-05-31).** Also in `mvp/get-add/master-company` (#12558), added as the **first** thing the function does: if the incoming `linkedin.com/company/<slug>` slug **or** the `company_name` normalizes to a known employment-status / placeholder value, it logs to `log_crash` and returns `null` — **no `master_company` row and no graph node are created** (callers skip via their `is_empty` gate). This is upstream of the §3.3 stub guard (which only blocks *edges* to an already-created stub). Blocklist (exact normalized match): `independent contractor`, `self employed`, `freelance`/`freelancer`/`freelancing`, `unemployed`, `homemaker`, `retired`, `student`, `various`/`various companies`, `unknown company`/`unknown`, `stealth startup community`. Extend the array as new junk slugs surface. **Also blocks incomplete `linkedin.com/company` URLs** — a bare `.../company` or `.../company/` with no company slug (reason `incomplete-linkedin-company-url`).

**Backfill required.** See §3.10 below.

<Note>
**Backfill workflow — sandbox-only until everything is resolved, then one consolidated live pass.**

1. Each backfill ships as a data-source-aware Xano **function** (Run & Debug with the target `X-Data-Source`), promoted to a scheduled Task only if a recurring live drain is needed.
2. **Run every backfill against `X-Data-Source: sandbox` only.** Sandbox has a known-good Carl Bressler (`master_person.id = 1`) end-to-end fixture from the session-3 fresh seed — verify the sweep produces the expected Carl outcome and nothing else regresses.
3. **🚫 LIVE backfills are HELD.** We do *not* run any backfill against `X-Data-Source: live` until **every** audit issue below is resolved and validated on sandbox. Rationale: several sweeps overlap (node-property normalization for §3.8 / §3.9 / §3.12; profile cleanup for §3.2 / §3.10; edge timestamps + renames) and should be **consolidated into the fewest possible live passes** rather than run piecemeal — fewer full-graph scans, one snapshot, one verification. The live-promotion plan is assembled at the end.
4. **Fix-forward is separate from this hold.** Fix-forward code lands on branch `v1` and is shared across data sources, so it takes effect for live enrichment immediately — only the *data backfills* are held.
5. **Document every backfill as you build it** in the standard artifact-card pattern (e.g. ``tool/backfill-edge-timestamps — 13051``).
6. Before the eventual LIVE run: snapshot the relevant tables / export the affected edges via Cypher. All sweeps are reversible in principle but the snapshot is the safety net.
7. **Verify with `tool/edge-qa-gate` (#13055, v1.3)** — one **read-only** call returns every should-be-zero check (edge/node `created_at`, string `visibility`, geo `fact`, WORKS_AT sibling-dups, union WORKS_AT, description coverage, poisoned node domains, **§3.6 `offscale_weights`, §3.6 `was_founded_edges`**) **plus the target FalkorDB `graph_name`**. Always glance at `target_graph` to confirm you're on sandbox vs live before trusting the numbers — the data source silently defaults to live, and a whole-graph dedupe *dry* was accidentally run against live this way (read-only, so no harm, but a destructive run would not be).

It deliberately excludes §3.7 Pass 2 (C-Suite regex), the §3.3 dedupe (#13052, destructive), and descriptions (LAST) — run those individually.

### 3. Issues found

#### §3.1 — Domain-parse error strings persisted as live data (HIGH)

The "Stealth Startup" company node (`master_company_id: 5`, shared by Gennady S. and Mark Phillips) has a parser exception baked into four fields:

```
company_domain: "[Line 16:Char 9] Error parsing domain: Domain name label can only contain alphanumeric characters or dashes."
```

The same error string is interpolated into the `logo` and `website` URLs:

```
logo:    https://img.logo.dev/[Line 16:Char 9] Error parsing domain: ...?token=pk_...
website: https://[Line 16:Char 9] Error parsing domain: ...
```

The enrichment pipeline caught a parse exception and persisted the exception text instead of null-ing the field. Downstream effect: broken images, dead links, and logo.dev calls firing the `pk_` token against a garbage path.

#### §3.3 — Duplicate edges between same node pair + unresolved stub (HIGH)

Carl Bressler → "Unknown Company" (VC_Firm) has **two** edges with the same role title:

- INVESTED_IN — `id 8348`, "Limited Partner", weight 15
- WORKS_AT — `id 8642`, "Limited Partner", weight 20

#### §3.4 — `description` vs `fact` field-name inconsistency (MEDIUM)

The same semantic slot (human-readable edge label) uses different field names by edge type:

- LOCATED_IN → `description` ("X is located in Y")
- IN_REGION / IN_COUNTRY → `fact` ("X is a city located in Y")

Any consumer reading edge prose must branch on edge type. Pick one field name.

#### §3.5 — `valid_from` conflates valid time with ingestion time (MEDIUM)

- WORKS_AT / FOUNDED use real role dates (e.g. `2023-01`).
- Every LOCATED_IN, plus IN_REGION / IN_COUNTRY, either stamp `valid_from: "2026-05"` (current month = ingestion date masquerading as validity) or omit it entirely.

A company located in a city since founding should not have `valid_from = this month`. Combined with the missing edge `created_at` (§1), there is no way to recover true validity.

#### §3.6 — `weight` has no consistent scale (MEDIUM) — ✅ RESOLVED 2026-05-31

Observed values: 5, 8, 10, 15, 20, 25, 28, 38, 50 (plus sub-1 award weights).

- Geo-hierarchy edges look like schema constants (IN_REGION = 25, IN_COUNTRY = 50).
- Some look computed (28, 38).
- The same person's FOUNDED edge (always 5) disagrees with their WORKS_AT edge (5-28) for what is effectively the same relationship.

Looks like residue from two different weighting passes. Confirm intended scale.

**Resolution.** Scale locked in [ontology/edge-weights](/guides/ontology/edge-weights); fix-forward applied to all flagged creators and backfill built (sandbox-first, live-held). The "FOUNDED 5 vs WORKS_AT 5–28" observation is **by design** — `FOUNDED` is a flat 5 and `WORKS_AT` is computed (`seniority_rank + past-penalty`), now documented as [WORKS_AT buckets](/guides/ontology/edge-weights#works_at-weight-buckets). The "28 / 38" computed values are exactly the past-role buckets (Founder/Partner/President past = 28; Manager/Lead past = 38). See the [§3.6 — Weight scale](#36--weight-scale-unblocked--fix-forward-applied-2026-05-31) fix-forward card, and the **backfill §3.6** card (`tool/backfill-edge-weights` #13063 + `tool/backfill-purge-was-founded` #13064) in the [Backfill](#backfill--clean-existing-data) section.

#### §3.7 — Semantically wrong WORKS_AT for union membership (MEDIUM)

Carl Bressler → SAG-AFTRA: `title: "Actor / Artist"`, `seniority_level: "C Suite"` (`id 5919`). Union membership is being modeled as a WORKS_AT employment edge, and an actor's membership is tagged C-Suite. `seniority_level: "C Suite"` is applied loosely across non-exec titles generally.

#### §3.8 — `visibility` typed inconsistently (MEDIUM)

Same field, mixed types across labels:

- Person nodes: boolean `false`
- Company / Organization nodes: string `"true"`
- Funding_Round / VC_Firm: boolean `true` / `false`

Any `WHERE visibility = true` Cypher filter will misbehave.

#### §3.9 — `created_at` stored in three formats across node labels (MEDIUM)

- Person: ISO-Z — `2026-05-30T22:15:44.246Z`
- Company: epoch millis — `1780179235150`
- Region / State: ISO + microseconds + offset — `2026-05-30T22:15:06.425258+00:00`

Cross-label sorting / range queries on creation time require normalization first.

#### §3.10 — Junk enrichment profiles on company nodes (LOW-MEDIUM)

Beyond Stealth Startup (§3.2), Harmonic AI's `profiles` include obvious mismatches: `linkedin.com/company/stealth-startup-community`, `facebook.com/forbes`, `instagram.com/popular`, `crunchbase/akhenaten`. Profile-matching is attaching unrelated web identities to company nodes.
