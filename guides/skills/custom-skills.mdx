## Overview

Custom Skills are [Claude agent skills](https://code.claude.com/docs/en/skills) authored in-house for Orbiter.io workflows and tools that don't have a high-quality third-party skill available. They encode our conventions, API surface, and guardrails so agents stay aligned with how we actually build and ship.

Custom skills live in `~/.claude/skills/<skill-name>/SKILL.md` on each developer's machine. This page is the source of truth for what each skill does — update it when you add, rename, or retire a custom skill.

## check-models

**Purpose:** Audit every LLM model used in the Orbiter enrichment pipelines against live Xano functions in workspace 3, then reconcile three different things on the Mintlify model documentation: configuration, pricing, and derived cost.

**Location:** `~/.claude/skills/check-models/SKILL.md`

**What it covers:**

<Steps>
  <Step title="Xano → per-pipeline model summaries (config)">
    For every function in the skill's `model-registry.md`, fetches the live XanoScript from workspace 3 (`mcp__xano__getFunction`, batched in parallel groups of 5–8) and extracts `model`, `temperature`, `max_tokens`, and `timeout`. Resolves `$env.LLM_MODEL_*` variables (asking the user if the Xano tenant env API returns empty and caching for the run). Then diffs each value against the per-pipeline `model-summary.mdx` files under `guides/enrichment/**/`, including moving function rows between `## model/id` sections if a model was swapped, creating new model sections when needed, removing empty ones, and updating Mermaid diagram labels in lockstep.
  </Step>
  <Step title="Canonical model list from all-models-summary.mdx">
    Reads `guides/enrichment/all-models-summary.mdx` and extracts every `## \`model/id\`` heading. This is the authoritative list of models the audit cares about — any model found in Xano that isn't listed here is reported as *new* and surfaced for the user to add before pricing runs.
  </Step>
  <Step title="OpenRouter → every Model Summary page (pricing)">
    For each model in the canonical list, fetches `https://openrouter.ai/{model_id}` in parallel and extracts context window, input cost per 1M tokens, and output cost per 1M tokens. Updates the pricing table beneath that model's heading on both `all-models-summary.mdx` (keeping the cost comparison table sorted by input cost ascending) and every per-pipeline `model-summary.mdx` that has a section for the model.
  </Step>
  <Step title="Derived Cost/Call on per-pipeline pages only">
    On every per-pipeline `model-summary.mdx` (never `all-models-summary.mdx`), walks every row of every function table. If both `Avg Input Tokens` and `Avg Output Tokens` are populated (non-TBD), recomputes `Cost/Call` using the formula `(avg_in / 1,000,000) × input_cost + (avg_out / 1,000,000) × output_cost`. Uses the **new** prices from the previous step, not the old ones. Rows where either average is still `TBD` are left untouched. Formatting: six significant digits below `$0.01`, four decimal places above.
  </Step>
  <Step title="Report, confirm, apply, verify">
    Presents a single consolidated diff — config sync, pricing changes, cost recalcs, new models, stale models — and asks for a single confirmation before touching anything. After writes, re-reads every touched file to verify.
  </Step>
</Steps>

**Triggers on:** `check models`, `update models`, `model audit`, `sync models`, `are my docs up to date`, `model check`, or after any Xano function model swap.

**Guardrails:**
- Never prices a model that isn't already in `all-models-summary.mdx` — surfaces it as *new* and waits for user decision.
- Never recalculates `Cost/Call` on `all-models-summary.mdx`.
- Never invents an OpenRouter price — if the page is inaccessible, warns and leaves the existing price in place.
- Never auto-removes an orphaned model section — asks first.
- Never guesses env var values.

**Resource files:** `model-registry.md` — authoritative list of function IDs the audit covers, with their model sources (hardcoded vs env var) and the doc files that reference them. Updated whenever enrichment pipeline functions are added or removed.

## sync-system-prompts

**Purpose:** Keep every LLM system prompt in Xano workspace 3 in lockstep with the Mintlify `llm-system-prompts.mdx` documentation. Xano is always the source of truth. **The user owns version numbers in Xano** — when editing a prompt, the developer bumps the `// vN` header themselves. The skill copies Xano → Mintlify verbatim (version header included) and only rewrites the Xano version as a **safety net** when drift is detected but the user forgot to bump.

**Location:** `~/.claude/skills/sync-system-prompts/SKILL.md`

**What it covers:**

<Steps>
  <Step title="Discover every $systemPrompt block in Xano">
    Lists all functions in workspace 3 (paginated), filters to the enrichment-pipeline name prefixes (`mvp/imdb/`, `mvp/enrich/`, `mvp/about/`, `mvp/bios/`, `mvp/twitter/`, `mvp/school/`, `mvp/company/`, `mvp/link/`), and fetches XanoScript in parallel batches. For each function, finds every `util.template_engine { value = """..."""} as $systemPrompt` block via regex. Multiple blocks per function are supported and matched by order of appearance — function `#2537` has two (short bio and long bio) and they are handled independently.
  </Step>
  <Step title="Parse every Mintlify llm-system-prompts.mdx">
    Reads the five per-pipeline files under `guides/enrichment/**/llm-system-prompts.mdx` (tolerant of missing files). For each `## N. Title` section, extracts the `**#NNNN**` function reference and the prompt body from the `\`\`\`text\` fenced block under `### System Prompt` (typically inside an `<Accordion>`).
  </Step>
  <Step title="Match and classify with version-line-stripped diff">
    Groups Mintlify sections by function ID and pairs them with Xano blocks by `(function_id, order_within_function)`. **Strips the `// vN` version line from both sides before diffing** — the version is metadata, not content, which prevents a phantom-drift loop where a stale version header keeps re-triggering a bump. Classifies each pair as: **in-sync**, **drift**, **orphan** (Xano has it, Mintlify doesn't), or **stale doc section** (Mintlify has it, Xano doesn't).
  </Step>
  <Step title="Report before touching anything">
    Shows a full summary: counts for each category, a drift table with current version → new version, an orphans table with the first 80 characters of each unhomed prompt, and a stale-sections table. Nothing is written until the user confirms.
  </Step>
  <Step title="Resolve orphans one at a time">
    For each orphan, asks the user (via `AskUserQuestion`) which Mintlify file it belongs on, with the option to skip. Asks for a section title, deriving one from the function name if none is given. If the chosen file doesn't exist yet, offers to create it from the `person-pipeline/llm-system-prompts.mdx` template.
  </Step>
  <Step title="Apply: copy Xano → Mintlify verbatim, safety-net bump only if needed">
    Default behavior is to copy Xano's body (version header included) into Mintlify unchanged. The skill **does not auto-bump versions** — the user is responsible for bumping `// vN` in Xano when they edit a prompt.

    The **safety-net rule** fires only when drift is detected AND Xano's version is not ahead of Mintlify's (i.e. user edited the body but forgot to bump):

    | Xano ver | Mintlify ver | Action on Xano |
    |---|---|---|
    | `vN` | `vM` where `M < N` | No bump — user already bumped; copy body to Mintlify. |
    | `vN` | `vN` (equal) | **Safety-net bump** → rewrite Xano with `v{N+1}`. |
    | `vN` | `vM` where `M > N` | **Safety-net bump** → Mintlify is ahead (shouldn't happen); rewrite Xano with `v{M+1}`; warn in report. |
    | no version | `vM` or none | **Safety-net bump** → treat missing Xano version as `v1`, rewrite as `v{M+1}` (or `v2`). |
    | `vN` | no version | No bump — Mintlify is just missing it; copy Xano body verbatim. |

    **Critical correctness detail:** multiple drifted blocks inside the same Xano function are batched into a single `mcp__xano__updateFunction` call — writing them sequentially would invalidate byte offsets within the XanoScript. After writes, the final Xano body (either unchanged or freshly bumped) is the body written to Mintlify.

    Every safety-net bump is surfaced loudly in the final report so the developer knows which prompts they edited without bumping.
  </Step>
  <Step title="Orphans get v1 + a new Mintlify section">
    Prepends `// v1` to the Xano body (batched with other in-function edits) and appends a new numbered section to the chosen Mintlify file using a standard template with TODO placeholders for description, model, and curl example (model/temperature/etc. are owned by the `check-models` skill, not this one).
  </Step>
  <Step title="Stale sections, on confirmation">
    For each stale Mintlify section (prompt removed from Xano), asks the user per-entry whether to delete. Offers a bulk "delete all" option when there are more than three. Never deletes without confirmation.
  </Step>
  <Step title="Verify">
    Re-fetches each touched Xano function and re-reads each touched Mintlify file to confirm writes landed correctly. Reports final counts.
  </Step>
</Steps>

**Triggers on:** `sync system prompts`, `update system prompts`, `system prompt audit`, `check system prompts`, `system prompt drift`, or after editing any `$systemPrompt` template in Xano.

**Guardrails:**
- Xano is always the source of truth — Mintlify never writes back to Xano body content.
- **The user owns version numbers.** The skill does not auto-bump. It only rewrites the Xano version as a safety net when drift is detected and the user clearly forgot to bump (see table above).
- Diff with the version line stripped so version metadata never creates phantom drift — version numbers are only compared separately to decide whether the safety-net bump applies.
- `**#NNNN**` function reference is mandatory on every new Mintlify section (it's the cross-system join key).
- Never touches models, temperature, max_tokens, or curl examples — those belong to `check-models`.
- All in-function edits are batched into a single `updateFunction` call.
- Preserves original XanoScript indentation when rewriting `value = """..."""` blocks.
- Missing Mintlify files are tolerated unless explicitly chosen as a home.

## cypher-sync-audit

**Purpose:** Audit every FalkorDB Cypher snippet in the Mintlify cypher pages against the actual Cypher embedded inside the Xano functions they reference, then produce a markdown drift report. Read-only by design — the skill never edits docs, it only describes what's out of sync so the user can decide what to fix manually.

**Location:** `~/.claude/skills/cypher-sync-audit/SKILL.md`

**What it covers:**

<Steps>
  <Step title="Extract documented cyphers from Mintlify">
    Runs `scripts/extract_cyphers.py` over `guides/cyphers/*.mdx`, walking every `\`\`\`cypher` fenced block. Uses a heading-level stack so sub-headings inherit the function ID and status emoji from their parent — `### Single founder branch` under `## MENTOR ingest — mvp/yc/process-yc-people (#12700) 🟢` correctly inherits `function_id: 12700` and `status: live`. Falls back to a 30-line scan above the fence if heading inheritance fails. Skips 🔵 planned and ⚪ illustrative entries automatically since those aren't backed by a function. Outputs a JSON inventory keyed by `(source_file, source_line, function_id, status, cypher)`.
  </Step>
  <Step title="Fetch live cyphers from Xano workspace 3">
    For each unique `function_id` in the doc inventory, fetches the live XanoScript via `mcp__xano__getFunction` (with `getAPI` as a fallback for endpoints), in parallel batches of 5–8. Extracts every `util.template_engine { value = """...""" } as $var` block from the script — and also the `var $cypher { value = """...""" }` shape used by callers of `mvp/falkor/send-cypher` — capturing each block's `as $var_name` so multi-block functions (like MENTOR's multi-founder vs single-founder branches) can be matched to the right doc snippet. Functions that 404 are flagged `xano_missing`; functions that exist but have zero cypher blocks are flagged `stale_no_cypher`.
  </Step>
  <Step title="Diff doc vs Xano">
    Runs `scripts/compare.py` which normalizes both sides (collapses whitespace, strips comments) while preserving Twig substitutions like `{{$input.foo|json_encode}}` literally — both sides are templates so they should match character-for-character after normalization. If they don't, that's a real drift. Multi-block functions match by `as $var_name` first, then by order of appearance within the doc section as a fallback.
  </Step>
  <Step title="Report categories">
    The output is a markdown report with five sections:

    - **✅ In sync** — count only.
    - **⚠️ Drifted** — per-snippet `diff` blocks showing the doc line vs the Xano line, with file:line pointers.
    - **🆕 Undocumented Xano cyphers** — cypher blocks that exist in Xano but no doc snippet references them. Intentional firehose so engineers know when graph writes are happening that aren't documented.
    - **💀 Stale doc references** — doc snippets referencing a function that no longer exists in Xano workspace 3 or that no longer contains any `util.template_engine` block.
    - **❓ Unlinked doc cyphers** — doc snippets where no function ID could be found nearby. The fix is either to add `(#NNNNN)` to the section header or to mark the snippet ⚪ illustrative.
  </Step>
  <Step title="Present, never auto-fix">
    Reads `/tmp/cypher-audit-report.md` and shows it verbatim, then asks *"Want me to update any of the docs to match Xano?"* — but doc edits themselves are out of scope. The skill hands off back to the main conversation if the user wants to apply fixes. This is deliberate: cypher edits are sensitive (they touch live graph write paths) and the user must approve each one.
  </Step>
</Steps>

**Triggers on:** `cypher audit`, `cypher sync`, `cypher drift`, `audit cyphers`, `sync cyphers`, `check cyphers`, `are my cypher docs in sync`, or after deploying any Xano function that contains a `util.template_engine` block.

**Why this skill exists:** Mintlify cypher docs drift fast. Every time a Xano function's `util.template_engine` block changes — switching `datetime()` to `timestamp()`, swapping undirected `MERGE` for directed, renaming an input variable — the corresponding doc snippet silently goes stale unless someone manually re-syncs it. The cost of letting it drift is high: future engineers (and Claude) read the docs as the source of truth and reproduce broken cyphers from them. This skill is the periodic reconciliation pass.

**Status legend in the docs:** 🟢 live (in production), 🟡 built\_not\_live (function exists but hasn't been triggered end-to-end), 🔵 planned (spec only), ⚪ illustrative (pedagogical only). **Only 🟢 and 🟡 are audited.**

**Guardrails:**
- Read-only — never edits cypher docs, only reports drift. Doc edits are explicitly out of scope and require a separate user approval.
- Never tries to "render" Twig templates — both sides are templates and should match literally after whitespace normalization.
- Skips 🔵 planned and ⚪ illustrative snippets — those are intentionally untethered to a Xano function.
- Surfaces undocumented Xano cyphers by default (firehose) so engineers see graph writes that aren't documented.
- Intermediate JSON and the final report live in `/tmp/cypher-audit-*.{json,md}` and are overwritten on every run — no history is kept.

**Resource files:**
- `scripts/extract_cyphers.py` — walks the docs dir, builds the heading-level stack, and emits the doc cypher inventory as JSON.
- `scripts/compare.py` — normalizes and diffs the doc inventory against the Xano inventory, emits the markdown report.

# Linear Design Refresh Skill

Apply Linear's 2026 design refresh philosophy to create clearer, calmer product interfaces.

## Core Principles

Linear's design refresh is built on two foundational ideas:

1. **"Don't compete for attention you haven't earned"** — Visual hierarchy, restraint, and intentionality
2. **"Structure should be felt, not seen"** — Soften visual elements, use whitespace strategically

The goal: interfaces that reduce cognitive load while increasing clarity.

## When to Use This Skill

Use this skill when:
- A dashboard, admin tool, or app feels cluttered, overwhelming, or fatiguing to look at
- Visual hierarchy is unclear (important actions blend in, secondary elements compete for attention)
- Too many icons, borders, or visual separators are creating visual noise
- You're planning a design system to prevent future clutter
- Building a component library with intentional spacing and visual weight
- Redesigning a UI that's suffering from feature creep or incremental additions
- Shifting a color palette from clinical cool tones to warmer, more intentional neutrals
- Planning an incremental UI refresh with feature flag rollout

Do NOT use this skill for: landing page design, logo work, brand identity, marketing sites, or isolated CSS tweaks.

---

## The Design Review Workflow

When analyzing a cluttered interface, follow this structure:

### 1. Identify Visual Noise
- **Icon audit**: Count decorative icons. Which ones actually communicate? Which are visual clutter? Linear often reduces 50+ scattered icons to a intentional, minimal set.
- **Border/separator analysis**: Hard 1px borders between every card, divider lines everywhere? These create visual weight. Look for opportunities to use whitespace instead.
- **Color overload**: Are there many colors competing for attention? Reducing the palette and being intentional about accent colors helps.
- **Spacing inconsistency**: Irregular gaps, misaligned padding, and dense packing all contribute to the feeling of clutter.

### 2. Establish Visual Hierarchy
The goal: make important actions obvious, secondary elements subtle.

- **Primary actions**: Bold, prominent, high contrast (clear focal point)
- **Secondary actions**: Lighter weight, lower contrast, smaller text
- **Tertiary elements** (help text, optional fields, metadata): Even lighter, receded with color/opacity/size reduction
- **Sidebar weight**: If a sidebar is as visually prominent as the main content, it competes for attention. Reduce its visual weight.

### 3. Measure Cognitive Load
Ask: "How hard is it to scan this interface and find what I need?"

- Dense dashboards with many sections, no clear hierarchy → high cognitive load
- Panels with inconsistent spacing and unclear grouping → high cognitive load
- Interfaces where secondary UI elements are as prominent as primary controls → high cognitive load

Solutions: group related elements, increase breathing room, reduce decorative elements, establish clear visual weight differences.

## Redesign Patterns

### Pattern 1: The Dashboard Clutter Audit

When a dashboard feels overwhelming:

1. **Screenshot the current state** — helps identify visual patterns
2. **List every element** — headers, cards, icons, separators, colors, typography levels
3. **Categorize by importance** — What does the user need first? What's supporting information?
4. **Sketch a reduced version** — remove half the visual elements, apply whitespace
5. **Apply Linear principles**:
   - Reduce icon count (keep only essential ones)
   - Replace hard borders with subtle shadows or whitespace
   - Increase vertical rhythm and breathing room
   - Establish typography hierarchy (headline, subheading, body, caption)

**Expected output**: A cleaner layout with fewer visual elements but clearer hierarchy.

### Pattern 2: Color Palette & Token System Design

Linear's refresh shifted from a cool, clinical blue palette to warmer, intentional neutrals.

**Define your palette:**
- **Base neutrals**: 2-3 neutral colors for backgrounds, text, borders (grays with slight warm undertone)
- **Accent**: 1-2 intentional accent colors for actions and highlights (avoid neon, favor warm tones)
- **Semantic colors**: Green for success, red for destructive, yellow for warning (use sparingly)

**Token system structure:**
- **Spacing scale**: 4px, 8px, 12px, 16px, 24px, 32px, 48px (multiples of 4 or 8)
- **Typography**: Define 4-5 levels (headline, subheading, body, small, caption) with specific sizes, weights, line-heights
- **Shadows**: Subtle, single shadow for elevation (e.g., `0 1px 3px rgba(0, 0, 0, 0.12)`) instead of heavy shadows
- **Border radius**: Minimal (2px-4px) for a clean, modern feel
- **Borders**: Light, subtle (#e5e7eb or similar), used only when necessary for containment

**Why this matters**: Consistent tokens prevent the "accumulated clutter" problem where designers add new spacing/colors over time without system thinking.

### Pattern 3: Incremental UI Refresh with Feature Flags

Rolling out a redesign all at once is risky. Linear's approach uses feature flags for staged rollout:

1. **Phase 1 (20% of users)**: Deploy new design to a small cohort, monitor for issues
2. **Phase 2 (50% of users)**: Expand to half, gather feedback
3. **Phase 3 (100%)**: Full rollout

**Implementation strategy**:
- Wrap new UI components in feature flag conditionals
- A/B test if possible: measure engagement, error rates, user feedback
- Keep old code alongside new code until 100% rollout is stable
- Plan rollback: if issues arise, quickly revert via flag

**CSS/code considerations**:
- New component CSS should not leak into old UI (scope carefully)
- Keep database schema changes backward-compatible during rollout
- Log feature flag state in analytics to correlate issues with rollout phases

## Component Design: Spacing, Typography, Hierarchy

When designing individual components (forms, panels, cards), apply Linear's philosophy:

### Spacing Strategy
- Use an 8px-based system: gaps of 8px, 16px, 24px between sections
- Within components: 4px-8px for internal spacing (between fields, label and input)
- Breathing room: larger gaps (24px+) between major sections create calm
- Rule: "more whitespace than you think you need"

### Typography Hierarchy
- **Headline** (18-20px, weight 600): Section titles, panel headers
- **Subheading** (14-16px, weight 500): Subsection titles, field groups
- **Body** (14px, weight 400): Default text, descriptions, form labels
- **Small** (12px, weight 400): Helper text, metadata, timestamps
- **Caption** (12px, weight 400, lighter color): Optional fields, secondary info

Why 14px body? Linear found it strikes the right balance for readability and calm—larger than 12px (less fatiguing), smaller than 16px (more dense).

### Making Secondary Elements Recede
Techniques to use in combination:
- **Size reduction**: Help text smaller than labels
- **Color reduction**: Secondary text at 60-70% opacity instead of full black
- **Weight reduction**: 400 weight for secondary, 500-600 for primary
- **Spacing**: More gap around important elements, less around secondary ones

**Example**: In a settings panel, optional fields are grouped together, use smaller text, lighter color, and more padding around required fields to establish hierarchy.

## Verification Checklist

Before shipping a redesign, verify:

✓ **Visual hierarchy is clear**: Can a user identify primary actions, secondary controls, and tertiary info at a glance?
✓ **Whitespace is intentional**: Not too dense, not too sparse. Breathing room without feeling empty.
✓ **Icon usage is minimal**: Removed decorative icons, kept only the essential ones that communicate clearly.
✓ **Borders and separators are reduced**: Replaced hard lines with whitespace, subtle shadows, or color gradations.
✓ **Color palette is limited**: Accent colors are rare and intentional, neutrals dominate.
✓ **Typography hierarchy is consistent**: Headings, subheadings, body, and captions are visually distinct.
✓ **Token system is documented**: Spacing, typography, colors, shadows, border-radius all defined and reusable.
✓ **Cognitive load is lower**: The interface is easier to scan and understand than before.
✓ **Feature flags are in place**: Rollout strategy tested, rollback plan documented.
