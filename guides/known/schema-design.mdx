## Conventions

- **Temporal fields:** Every facet edge carries `valid_from` and `valid_to`. `valid_to IS NULL` means currently valid. This matches the existing graph ontology.
- **Canonicalization:** Undirected facet edges are stored with the lower person `uuid` (or `id`) as the source. The application layer MUST enforce this before any `CREATE` or `MERGE`. A helper like `createKnowsEdge(a_id, b_id, props)` should wrap every write.
- **Directionality:** All facets except `MENTOR` are undirected-modeled-as-canonicalized-directed. `MENTOR` is the only directed facet (mentor â†’ mentee). Introductions are not facets at all â€” they are first-class [`Introduction` nodes](/guides/known/schema-design/nodes#introduction).
- **Mixed-mode graph model:** Known is primarily edge-based (facets between two persons), with two surgical node-based exceptions: `Introduction` (genuinely N-ary, carries its own metadata, can't work as an edge) and `Household` (genuinely N-ary, shared living space with multiple residents). Everything else stays as direct person-to-person edges for query performance.
- **Asymmetric roles in symmetric facets:** Facets that are mutual but carry asymmetric roles (KINSHIP parent/child, PROFESSIONAL_TIE manager/report) use a pair of role fields: `role_a_to_b` and `role_b_to_a`. This is a required pattern â€” apply it to any symmetric facet with role asymmetry.
- **Invalidation preserves history:** Facets are never deleted on state change. `valid_to` is set, and the row stays. New facets are created for new states.
- **Source field on lifecycle facets:** `BONDED`, `ESTRANGED`, `PROFESSIONAL_TIE`, and other lifecycle overlays carry `source: 'inferred' | 'declared'`. Cron logic only touches `inferred` rows; declared rows are immutable except by user/LLM action.
- **Weight convention:** Every weighted edge carries a `weight` field (float). **Lower weight = higher importance** â€” matches the existing Orbiter ontology convention. Weight is computed dynamically within a per-type range from the instance's `strength_score`. See [Weight model](#weight-model) below.

## Weight model

Every weighted edge in the Known graph has a `weight` field that determines its priority in ranked queries. The convention matches the existing Orbiter ontology: **lower weight means higher importance**.

### Hybrid strategy: per-type ranges, dynamic within

Each edge type has a declared `weight_floor` and `weight_ceiling`. The per-instance value is computed by the scoring cron from `strength_score` using:

```
weight = floor + (ceiling - floor) * (1 - strength_score / 100)
```

Higher `strength_score` â†’ lower weight â†’ higher priority. For edges without a live `strength_score` (e.g. `Introduction` â†’ `CONNECTS` structural edges), the floor is used.

The cron writes `weight` to the edge on every pass. Queries like `MATCH (a)-[e]-(b) WHERE e.weight < 10` surface the most important edges first, and can predictably hit certain edge types because each type has a known range.

### Dynamic modifiers

**ESTRANGED penalty.** `ESTRANGED` does not have its own weight. Instead, when an `ESTRANGED` facet is currently valid on a pair, a penalty of `severity Ă— 5` is added to the `KNOWS` edge weight for that pair. Severity ranges 1â€“5, so the penalty ranges +5 to +25. A mild drift (severity 2) barely affects ranking; active hostility (severity 5) drops the relationship well below baseline KNOWS.

**Invalidated facets.** Facets with `valid_to IS NOT NULL` are **excluded from default weight queries**. The default query pattern filters `WHERE e.valid_to IS NULL`. Historical relationships can be queried explicitly by removing this filter, but they do not participate in ranked "most important relationships" queries.

### How rankings interleave

A concrete example â€” Sarah Okafor (from the [divorce scenario](/guides/known/documentation#example-2-divorce-with-co-parenting)) post-divorce with co-parenting:

| Relationship | Edge | `strength_score` | Weight | Notes |
|---|---|---|---|---|
| Sarah â†’ Emma (daughter) | `KINSHIP` tier 1 | â€” | **3** | Biological parent, highest priority |
| Sarah â†’ Lily (daughter) | `KINSHIP` tier 1 | â€” | **3** | |
| Sarah â†’ Michael (new partner, domestic partner) | `ROMANTIC` tier 1 | 72 | **5.68** | |
| Sarah â†’ Michael | `BONDED` (inferred) | 78 | **13.3** | |
| Sarah â†’ David (ex, still co-parent) | `KNOWS` (+ESTRANGED penalty sev 2) | 38 | **25 + 10 = 35** | Mild estrangement, penalty moves him down |
| Sarah â†’ David | `ROMANTIC(married)` | â€” | **â€”** | Invalidated (divorce), excluded from default queries |

Query `MATCH (sarah)-[e]-(other) WHERE e.valid_to IS NULL AND e.weight < 15 RETURN other ORDER BY e.weight` surfaces her children and current partner. David is excluded because his weight is above the filter threshold â€” correct for "who matters most right now" queries.

## Supporting structures

### `staging_evidence` table (Xano)

Not an edge. A Xano table for holding weak evidence that hasn't yet cleared the KNOWS creation floor.

| Column | Type | Description |
|---|---|---|
| `id` | integer | PK |
| `person_a_id` | uuid | Canonicalized (lower id) |
| `person_b_id` | uuid | |
| `source` | enum | Evidence source name |
| `weight` | float | Evidence weight |
| `first_seen_at` | timestamp | When ingested |
| `evidence_occurred_at` | timestamp | When the real-world event occurred |
| `raw_metadata` | json | Source-specific payload |
| `created_at` | timestamp | |

**Lifecycle:**
- **Insert** when evidence arrives but does not clear the floor (e.g. a lone LinkedIn connection).
- **Consume + delete** when a subsequent evidence event for the same pair brings them over the floor. The staged rows are merged into the resulting `KNOWS.evidence` array and then removed from staging.
- **Expire** after 730 days with no matching second signal (tunable cron parameter).

**Critical rule:** Every evidence ingest pipeline MUST check `staging_evidence` for the same `(a, b)` pair before creating or updating a `KNOWS` edge, regardless of whether the new evidence alone clears the floor. Skipping the staging check on strong-source ingests would orphan previously-staged weak evidence.

## Invalidation reasons

Every facet carries `invalidation_reason` when `valid_to` is set. Valid values:

| Reason | Meaning |
|---|---|
| `ended` | Relationship naturally ended (e.g. score dropped, context expired) |
| `superseded` | Invalidated because another facet or upgraded version took precedence. Examples: BONDED invalidated when ESTRANGED was created; ROMANTIC(dating) invalidated when ROMANTIC(domestic_partner) was created as an upgrade; KINSHIP(step_parent) invalidated when a formal adoption created KINSHIP(adoptive_parent). |
| `data_correction` | The facet was wrong â€” it should never have existed. Used for cleanup. |
| `deceased` | One party died |
| `reconciliation` | Only for ESTRANGED â€” the relationship recovered |
| `divorce` | Only for ROMANTIC(type='married') |
| `breakup` | Only for ROMANTIC(type='dating' \| 'engaged' \| 'domestic_partner') |
| `death` | Only for ROMANTIC â€” one party died (set automatically by `deceased_freezer`) |
| `divorce_severed_in_law` | Only for KINSHIP with relation in `{parent_in_law, child_in_law, sibling_in_law}` â€” the connecting marriage ended in divorce |
| `divorce_severed_step` | Only for KINSHIP with relation in `{step_parent, step_child, step_sibling}` â€” the connecting marriage ended in divorce |

Implementations should use the most specific reason available.

## Constraints

- **KNOWS uniqueness:** at most one `KNOWS` edge per canonicalized `(person_a_id, person_b_id)` pair at any time. **This is enforced at the application layer, not by a FalkorDB unique constraint** â€” FalkorDB does not support unique constraints on relationships. The enforcement is three-layered: (1) `canonicalize_pair` (#12767) puts the pair in canonical order; (2) `create_knows_edge` (#12778) uses a directed `MERGE` on the canonical pair with `ON CREATE` / `ON MATCH` branching; (3) `known/lifecycle/dedupe_knows_edges` runs nightly as a safety net against the narrow concurrent-write race window that MERGE cannot close. The composite index on `(person_a_id, person_b_id)` accelerates the MERGE's pattern match but does not enforce uniqueness. See [ingest â†’ KNOWS uniqueness enforcement](/guides/known/scope/ingest#knows-uniqueness-enforcement) for the full explanation.
- **Facet uniqueness:** the uniqueness key for lifecycle facets is `(a_id, b_id, facet_type, valid_at)`, NOT `(a_id, facet_type, valid_at)`. A person can have multiple valid `ROMANTIC` facets to different partners simultaneously; two people cannot have two valid `ROMANTIC` facets between them simultaneously. Same application-layer enforcement pattern as KNOWS â€” no DB constraint, canonicalized writes, and (for inferred facets) lifecycle crons that detect and collapse duplicates as part of their normal invariant checks.
- **Canonicalization:** application layer MUST enforce `a_id < b_id` for undirected facets. No exceptions. Wrap all writes in helper functions.
- **Evidence append-only:** the `KNOWS.evidence` array is never modified retroactively except to append new rows. The one exception is `dedupe_knows_edges`, which merges evidence from absorbed duplicates into the survivor â€” still additive, never retroactively removing rows.
- **`is_current` invariant:** `ROMANTIC.is_current = (valid_to IS NULL)`. Must be updated atomically with `valid_to`.
