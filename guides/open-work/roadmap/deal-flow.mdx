## Scope Frame

Deal Flow should be scoped as a flexible opportunity-operating framework, not as one locked fundraising or sales workflow. The product needs to let teams define the shape of their own pipeline, while giving agents enough structured state to suggest the next step toward momentum.

The durable idea is:

> A deal is an opportunity with state, relationships, evidence, and a machine-readable path to advancement.

That means the first build should prioritize primitives that agents and humans can both use:

- Custom pipelines and stages, owned by the organization.
- Stage contracts that define what must be true to enter, advance, close, pause, or revive an opportunity.
- A shared action catalog that powers UI buttons, command-palette actions, and agent tools.
- Suggestion objects that explain the gap, the proposed action, the evidence, and the expected outcome.
- Human-in-the-loop controls so agents can suggest, draft, run with approval, or run with undo based on policy.

Do not scope this as "a kanban board with AI bolted on." The board is only one view over the real framework: stage contracts, graph context, actions, and suggestions.

## Product Scope

Deal Flow is the planned workspace for managing opportunities through customizable pipelines while using Orbiter's relationship graph, enrichment data, calendar/email activity, notes, and team context to suggest what should happen next. The first version should feel like a focused operating surface that helps the user answer four questions:

| Question | Product answer |
|---|---|
| What is true right now? | Stage, owner, stakeholders, timeline, notes, signals, last touch, and current health. |
| What is missing? | Unmet stage criteria, missing decision maker, stale touch, missing company/person data, or no next task. |
| What can move this forward? | Ranked suggestions with grounded actions: intro, follow-up, enrich, schedule, assign, advance, pause, or disqualify. |
| What should the user or agent do? | One-click actions, editable drafts, delegation, snooze, undo, and transparent "why" reasoning. |

The module should support:

| Surface | Purpose |
|---|---|
| Pipeline board | Kanban-style stage view for open opportunities, with agent-suggested next actions visible on cards. |
| List / forecast view | Table view for value, probability, close date, owner, stale deals, health, and reporting. |
| Deal detail | Timeline, stakeholders, notes, tasks, next moves, stage criteria, and graph context for one opportunity. |
| Focus lane | AI-ranked queue of opportunities needing user attention today. |
| Stage contract inspector | Developer/admin surface showing what an agent reads: criteria, allowed actions, source maps, and autonomy policy. |
| Network context | Optional relationship view showing stakeholders, warm paths, and graph signals. |

## Core Framework

These are the flexible primitives developers should protect. If these stay generic, the same Deal Flow system can support fundraising, sales, partnerships, hiring, customer expansion, introductions, or any future opportunity workflow.

| Primitive | What it means | Agent leverage |
|---|---|---|
| Pipeline | A user/org-defined workflow, not a hard-coded business type. | Agent scopes suggestions to the active pipeline and its rules. |
| Stage / status contract | A row with entry criteria, exit criteria, category, order, color, source map, and autonomy policy. | Agent evaluates what is missing and which actions can close the gap. |
| Opportunity / deal | The thing moving through a pipeline. It can point at a company, person, funding round, project, or custom target. | Agent gathers evidence, ranks next steps, and watches for stale state. |
| Stakeholder | A person/company connected to the opportunity with a role. | Agent can detect missing buyer, champion, investor partner, sponsor, or approver coverage. |
| Evidence | Signals, emails, meetings, notes, enrichment, graph paths, field values, and activity events. | Agent grounds each suggestion in concrete data instead of generic advice. |
| Action definition | A versioned verb with input schema, permission policy, reversibility, and agent-callable flag. | Agent can only propose or run approved verbs through the same path as the UI. |
| Suggestion | A proposed move, action, or hint with rank, grounding, lifecycle state, and undo metadata. | Agent output becomes inspectable product state, not an invisible chat response. |
| Focus entry | A user-specific surfaced item from many possible suggestions. | Agent respects attention budget and avoids flooding the user. |
| Agent grant | A revocable per-agent/per-stage/per-action autonomy ceiling. | Agent can be safely delegated work without receiving blanket write access. |
| Event timeline | Append-only record of user, agent, automation, and system activity. | Agent actions are attributable, reversible, and learnable. |

## Agent Next-Step Loop

Agents should not invent work from prose. They should follow a predictable loop over structured state:

<Steps>
  <Step title="1. Observe the opportunity">
    Read the deal, pipeline, current stage, stakeholders, custom fields, timeline, health factors, graph context, and recent email/calendar/activity evidence.
  </Step>

  <Step title="2. Evaluate the stage contract">
    Compare the opportunity against `entry_criteria_json` and `exit_criteria_json`. Store the result as criterion state so the UI and agent see the same gaps.
  </Step>

  <Step title="3. Map each gap to an action">
    Use the stage `source_map_json` and `df_action_def` catalog to determine which approved actions can close each missing criterion.
  </Step>

  <Step title="4. Generate ranked suggestions">
    Create `df_suggestion` rows with one or more ordered `df_suggestion_action` steps. Each suggestion must include grounding, confidence, reversibility, and the expected stage impact.
  </Step>

  <Step title="5. Surface the right amount">
    Push only the best items into `df_focus_entry` based on urgency, rank, user attention budget, and whether the action needs approval.
  </Step>

  <Step title="6. Execute through the shared action path">
    The user, UI, automation, or agent runs the same action definition. Suggestions can be accepted, edited, delegated, snoozed, dismissed, executed, or undone.
  </Step>

  <Step title="7. Learn from outcome">
    Record lifecycle events and outcomes so the ranker can reduce low-value suggestions and promote actions that actually move opportunities forward.
  </Step>
</Steps>

## Suggestion Examples

| Scenario | Agent should suggest | Helper/action needed |
|---|---|---|
| Existing deal has no recent touch | Draft a follow-up and optionally create a next task. | `draft_follow_up`, `create_next_task`, `log_touch` |
| Deal is stuck before next stage | Show unmet criteria and propose the smallest action that closes one gap. | `advance_stage`, `set_field`, `request_update` |
| No warm stakeholder is mapped | Find a relationship path and draft an intro request. | `find_warm_path`, `draft_intro` |
| Company/person data is incomplete | Enrich the missing company or person before suggesting outreach. | `enrich_company`, `enrich_person` |
| Meeting is upcoming | Generate a prep brief and list the decision criteria still missing. | `summarize_deal`, `generate_stakeholder_map` |
| Strong signal appears | Recommend a timely outreach or stage review. | `re_engage`, `draft_follow_up`, `advance_stage` |
| Deal is clearly low-fit | Suggest pause/disqualify with evidence, but require human confirmation. | `set_field`, `advance_stage` |
| User delegates a batch | Agent runs approved reversible actions and writes an Agent Run Ribbon to the timeline. | `df_agent_grant`, `df_agent_run`, action-specific undo |

## Flexibility Rules

- Do not hard-code stage names like Prospect, Warm, Diligence, or Closed. They can be seed templates only.
- Do not hard-code one business type. A pipeline can be fundraising, sales, partnerships, hiring, investor relations, customer expansion, or something custom.
- Do not make AI suggestions chat-only. Every suggestion must be saved as product state with lifecycle, grounding, and audit trail.
- Do not let agents bypass the action catalog. If a thing can be done, it needs a `df_action_def` contract.
- Do not require full autonomy to get value. Suggest-only mode should be useful from day one.
- Do not hide reasoning. Every next step needs a visible "why", source evidence, and expected impact.
- Do not make the graph view mandatory for v1. The graph should power suggestions even before a visual network view ships.

## UX Principles

- Keep the board quiet by default; surface urgency through stage age, health, and agent-ranked next moves.
- Treat every stage as user-configurable data, not a hard-coded enum.
- Make the next action obvious: follow up, ask for intro, enrich, schedule, advance, pause, revive, or disqualify.
- Use the same action contracts for UI buttons, command palette actions, automations, and agent tools.
- Prefer progressive disclosure: card -> peek -> deal detail -> stage contract -> reasoning.
- Show agent work on the timeline beside human work, with actor, source, undo, and outcome.

## Example Stage Contract

Developers can use this shape as a mental model. The final field names should follow the backend-design page, but the key point is that stages are readable by both humans and agents.

```json
{
  "stage_key": "active_conversation",
  "category": "open",
  "exit_criteria": [
    {
      "key": "decision_maker_identified",
      "label": "Decision maker is identified",
      "predicate": {
        "type": "stakeholder_role_exists",
        "role": "decision_maker"
      }
    },
    {
      "key": "recent_touch",
      "label": "There has been a touch in the last 14 days",
      "predicate": {
        "type": "activity_within_days",
        "days": 14
      }
    }
  ],
  "source_map": {
    "decision_maker_identified": ["generate_stakeholder_map", "find_warm_path"],
    "recent_touch": ["draft_follow_up", "schedule_meeting"]
  },
  "autonomy_policy": {
    "draft_follow_up": "one_click",
    "find_warm_path": "suggest_only",
    "advance_stage": "suggest_only"
  }
}
```

## First-Viewport Experience

The first shipped screen should be the actual pipeline, not a marketing or explanation page.

Expected layout:

| Area | Content |
|---|---|
| Left rail / view switcher | Board, list, focus lane, saved views. |
| Header | Pipeline selector, stage filters, owner filter, value summary, create deal. |
| Main surface | Board columns in v1, with cards grouped by current stage. |
| Card | Company/deal name, stage age, value, owner, stakeholder chips, graph/warm-path signal, health beads, and next-move badge. |
| Peek / next move | Shows the top suggestion, why it matters, evidence, and the action button. |
| Detail drawer | Opens on card click; shows timeline, contacts, notes, tasks, stage criteria, agent suggestions, and graph context. |

## Frontend Decisions

| Concern | Decision |
|---|---|
| Kanban drag/drop | Use `@hello-pangea/dnd` for the first MVP unless the frontend team already has a stronger local pattern. It is fast to ship and familiar. |
| Table view | Use TanStack Table with existing app table primitives. |
| Detail surface | Use a right-side drawer or route-backed detail panel, matching existing app navigation patterns. |
| Network view | Defer React Flow until board + list + detail are stable. |
| Agent suggestions | Render as action rows/cards backed by the backend `df_suggestion` contract. |
| Stage contract inspector | Build a simple read-only inspector for criteria, source maps, and autonomy policy before giving admins editing tools. |

## Build Sequence

<Steps>
  <Step title="1. Ship read-only board from flexible fixtures">
    Render multiple fixture pipelines with different stage names and custom fields. This proves the UI is not hard-coded to one sales/fundraising workflow.
  </Step>

  <Step title="2. Add real pipeline and opportunity reads">
    Read pipelines, statuses, deals, owners, and stakeholders from the backend contracts in [Backend design](/guides/open-work/deal-flow). Keep all actions disabled until permissions are wired.
  </Step>

  <Step title="3. Add stage contract reads">
    Render stage criteria, missing gaps, source maps, and allowed actions in a simple inspector. This is the first agent-readiness milestone.
  </Step>

  <Step title="4. Add stage move writes">
    Dragging a card updates the deal's status through the backend move endpoint and writes an event row. Reverting on failed writes is mandatory.
  </Step>

  <Step title="5. Add deal detail drawer">
    Show timeline, stakeholders, custom fields, tasks, notes, graph context, stage criteria, and existing suggestions. Keep editing narrow at first: owner, stage, value, close date, notes.
  </Step>

  <Step title="6. Add list / forecast view">
    Use TanStack Table for filtering, sorting, owner/stage/date filters, and weighted pipeline summary.
  </Step>

  <Step title="7. Add suggestion generation">
    Surface existing `df_suggestion` rows as human-approved actions with grounding, confidence, expected impact, and lifecycle controls. Do not auto-run actions in v1.
  </Step>

  <Step title="8. Add focus lane">
    Rank deals needing attention by stale stage age, upcoming close date, missing stakeholder coverage, and suggestion readiness.
  </Step>

  <Step title="9. Add agent delegation controls">
    Add suggest-only, one-click, and auto-with-undo policies behind `df_agent_grant`. Start with suggest-only defaults.
  </Step>

  <Step title="10. Add network context">
    Add relationship/warm-path visualization after the board, list, detail, and suggestion surfaces pass smoke tests.
  </Step>
</Steps>

## Smoke Tests

Use fixture pipelines before broad production testing.

| Smoke case | Expected result |
|---|---|
| Load empty pipeline | Board renders empty stages and no errors. |
| Load seeded pipeline | Cards appear in correct stages with stable order. |
| Load two very different pipelines | UI works with different stage names, custom fields, and action sets. |
| Inspect stage contract | Criteria, missing gaps, source map, and autonomy policy are visible for the selected stage. |
| Move card between stages | Backend status updates, event is written, board updates without duplicate cards. |
| Failed stage move | Card returns to original stage and error is visible. |
| Open detail drawer | Deal timeline, stakeholders, notes, and tasks load for the selected card. |
| URL refresh | Reloading a deal-detail URL opens the same deal. |
| List view parity | The same deals visible on board are visible in table filters. |
| Agent suggestion row | Suggestion renders with action, grounding, confidence, expected impact, lifecycle state, and no silent execution. |
| Dismiss/snooze suggestion | Lifecycle event is written and the suggestion no longer floods the focus lane. |
| Delegated action dry run | Agent can propose the same action a UI button can run, but cannot exceed the stage/user autonomy policy. |

## Acceptance Criteria

- Deal Flow appears under Planned Work as one group with this overview page and the backend-design subpage.
- Open Work no longer lists Deal Flow as active/open implementation work.
- Board, list, detail, focus lane, and suggestions can be developed independently against fixture data.
- The first implementation works with at least two different pipeline templates without code changes.
- Stages are machine-readable contracts, not enums hidden in frontend code.
- Agent suggestions are saved, inspectable, auditable product state.
- UI actions and agent actions share the same action-definition path.
- Suggest-only mode is valuable before any autonomy is enabled.
- Backend schema and access details live only in the backend-design subpage.
- The UI page does not duplicate the backend table inventory, FalkorDB schema contracts, or long sales/fundraising examples.
