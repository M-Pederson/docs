## Current Implementation Snapshot

The existing OpenUI AI chat path is already server-side. The frontend calls one Xano endpoint and does not call an LLM, Zep, or Mem0 directly.

| Item | Current state |
| --- | --- |
| Frontend entrypoint | `POST /openui05/dispatch` |
| Xano API | Endpoint `#8497` in `Robert OpenUI 0.5` group `#1276` (`api:C5i2nPpF`) |
| Auth | `user` table auth (`#125`) |
| Memory provider today | Zep |
| Frontend inputs | `query`, stable `thread_id`, `suggestion_request_id`, and YAML context blocks |
| Server output | OpenUI response payload plus `mem_used` and `mem_ingested` flags |

`openui05/dispatch` does the memory work inside Xano:

1. Builds a Zep user id as `user-{auth.id}`.
2. If `thread_id` and `$env.zep` exist, ensures the Zep user and thread exist.
3. Reads Zep thread context from `GET /api/v2/threads/{thread_id}/context`.
4. Builds `$mem_context` from frontend-provided YAML blocks plus Zep's `[NETWORK MEMORY]`.
5. Routes the turn through classification and the appropriate OpenUI tool endpoint.
6. Writes back to Zep with `POST /api/v2/threads/{thread_id}/messages`.

The current Zep write stores the user's raw query and an assistant route summary like `Routed to find_investors. ...`; it does not appear to store the full rendered OpenUI response as a Xano transcript inside endpoint `#8497`.

### Context Blocks

The frontend-provided YAML blocks are injected into `$mem_context` in this order when present:

1. `[YOU]` from `self_yaml_context`.
2. `[NETWORK MEMORY]` from Zep.
3. `[HELPING TARGET -- master_person #{for_master_person_id}]` from `target_yaml_context`.
4. `[YOUR RELATIONSHIP WITH TARGET]` from `target_relationship_summary`.
5. `[YOUR COMPANY]` from `self_company_yaml_context`.
6. `[TARGET COMPANY -- master_person #{for_master_person_id}'s current company]` from `target_company_yaml_context`.

### User Context Snapshot (#13069)

Use `mvp/context/get-all-user-context` (`function #13069`) to create the user's frozen self-context at the start of a suggestion request draft.

Function contract:

```txt
function: mvp/context/get-all-user-context
id: 13069
input: { user_id: auth.id }
response: markdown string
```

This function returns compact Markdown for the authenticated user. It includes person signals, experience, education, and current work context. It intentionally excludes noisy/internal context such as user IDs, master person IDs, raw YAML IDs, `bio_500`, and current company rows where `master_company_id` is null.

#### When to call it

Call `#13069` once immediately after the `suggestion_request` draft row is created and before the first OpenUI chat turn.

Current table:

```txt
suggestion_request (#529)
- id
- user_id
- copilot_mode
- request_context
- master_person_id
- status
- llm_summary_context
- number_of_results
- deliver_format
```

Do not store the user context in `request_context`. Keep `request_context` for the user's actual ask. Store the generated Markdown snapshot in the JSON field `suggestion_request.llm_summary_context` under a namespaced key:

```json
{
  "openui_context": {
    "self_context_function_id": 13069,
    "self_context_function": "mvp/context/get-all-user-context",
    "self_context_format": "markdown",
    "self_context_generated_at": "2026-06-08T00:00:00Z",
    "self_context_user_id": 15,
    "self_markdown": "# User Context\n\n## Robert Boulos\n\n..."
  }
}
```

Implementation sketch:

```txt
1. POST /suggestion-requests creates suggestion_request #529 with status = draft.
2. Run function #13069 with { user_id: auth.id }.
3. Edit the new suggestion_request row.
4. Merge the function result into llm_summary_context.openui_context.self_markdown.
5. Use the same stored snapshot for every chat turn on that suggestion_request.
```

If we later add `ai_chat_context_snapshot`, store the full Markdown there and keep only `context_snapshot_id` plus metadata in `llm_summary_context`. Until then, `llm_summary_context.openui_context.self_markdown` is the least-invasive storage location because the field already exists on `suggestion_request`.

#### How it feeds back into chat

Every call to `POST /openui05/dispatch` for the same `suggestion_request_id` should include the stored Markdown snapshot as `self_yaml_context`.

The input name is still `self_yaml_context` for backwards compatibility, but the value can be Markdown. Endpoint `#8497` treats it as text and injects it under `[YOU]`.

Current flow:

```txt
suggestion_request.llm_summary_context.openui_context.self_markdown
  -> dispatch input self_yaml_context
  -> [YOU] block in $mem_context
  -> downstream OpenUI tool endpoint context
```

Preferred server-side cleanup: update `openui05/dispatch` so that when `suggestion_request_id` is present and `self_yaml_context` is empty, it loads `suggestion_request.llm_summary_context.openui_context.self_markdown` itself. That keeps the frontend contract smaller and ensures Xano owns context assembly.

Do not rerun `#13069` on every chat turn. Refresh it only when:

- creating the draft `suggestion_request`
- the user explicitly asks to refresh their context
- the product intentionally restarts the draft with a new context snapshot

Do not write this full context snapshot to Mem0. Mem0 should receive only extracted durable memories, not the whole profile/work context.

### Meeting Prep Memo Migration

Use `mvp/meeting-prep/classify-participants_v2` (`function #13070`) as the canonical meeting-prep participant context builder.

Function contract:

```txt
function: mvp/meeting-prep/classify-participants_v2
id: 13070
response: participant JSON with embedded Markdown context
```

The function expands the older participant classification response with Markdown context fields:

| Location | Context field | Source helper |
| --- | --- | --- |
| `my_team` | `my_team_company_context` | `mvp/context/get-master-company-markdown-context` (`#13072`) |
| `my_team.members[]` | `person_context` | `mvp/context/get-master-person-markdown-context` (`#13071`) |
| `participating_companies[]` | `company_context` | `mvp/context/get-master-company-markdown-context` (`#13072`) |
| `participating_companies[].members[]` | `person_context` | `mvp/context/get-master-person-markdown-context` (`#13071`) |
| `personal_participants[]` | `person_context` | `mvp/context/get-master-person-markdown-context` (`#13071`) |

#### When to call it

Call `mvp/meeting-prep/classify-participants_v2` (`#13070`) only when the user chooses meeting prep for a calendar event.

Preferred lifecycle: keep Nylas webhook ingestion and calendar sync focused on calendar/event freshness. Do not run `#13070` for every incoming meeting by default. When the user actively opens or starts meeting prep, create or load the `meeting_prep_context`, run `#13070`, and store the packaged participant context there.

Use the stable meeting key that already exists in Xano:

```txt
meeting_prep_context.id        -> internal meeting prep context id
meeting_prep_context.nylas_event_id -> external calendar event id
```

If the product introduces a separate `meeting_id`, store that alongside `nylas_event_id`, but do not lose the Nylas event id because it is needed for webhook reconciliation.

Implementation sketch:

```txt
1. User selects "meeting prep" for a calendar event.
2. Create or load meeting_prep_context for user_id + nylas_event_id.
3. If participant_context exists and is still fresh for the current event version, reuse it.
4. Otherwise, skip cancelled events and events with no useful participants.
5. Extract participant emails from the calendar event.
6. Run mvp/meeting-prep/classify-participants_v2 (#13070).
7. Store the full JSON result in meeting_prep_context.participant_context.
8. Derive a single meeting prep memo Markdown string from the embedded context fields.
9. Store the derived memo under meeting_prep_context.participant_context.meeting_prep_memo_markdown.
10. Store lightweight metadata such as participant_context_generated_at and source_event_updated_at.
11. Use that stored memo for every meeting-prep chat turn for the same event.
```

Rerun `#13070` idempotently only when the user chooses meeting prep and the stored participant context is missing or stale because the event participant list, attendee emails, organizer, or linked company/person identities changed. Do not rerun it on every chat turn.

<Note>
  **Candidate `meeting_prep_context` cleanup.** Once `#13070` owns the packaged participant context and memo Markdown, review older fields before continuing to feed or duplicate them in chat:

  | Field | Recommendation | Why |

Do not ask the frontend to re-send the full nested participant payload on every chat turn. The frontend should send a stable meeting prep identifier, such as `meeting_prep_context_id` or `nylas_event_id`, and Xano should load the stored `participant_context` server-side.

For meeting prep, the LLM should receive all participant Markdown, but as deterministic Xano context rather than Mem0 memory.

Build a single memo from:

```txt
# Meeting Prep Context

## My Team
{my_team.my_team_company_context}

### My Team Members
{my_team.members[].person_context}

## Participating Companies
{participating_companies[].company_context}

### External Participants
{participating_companies[].members[].person_context}

{personal_participants[].person_context}
```

Then pass the stored memo into the meeting-prep chat input before retrieved memories:

```txt
System instructions

User context snapshot from #13069, when available

Meeting prep memo Markdown from #13070 participant_context

Relevant durable user memories from Mem0

Relevant run memories from Mem0

Recent meeting-prep chat transcript

```

This gives the model the full participant context every turn without polluting Mem0 with large profile/company snapshots.

#### Mem0 processing

Do not POST the full `#13070` participant result, full meeting memo, or raw participant Markdown to Mem0.

Use Mem0 around meeting prep this way:

1. **Before the LLM turn:** search Mem0 with a query built from the latest user message, meeting title, desired outcome, and participant names.
2. **During the LLM turn:** include the `#13070` meeting memo as deterministic Xano context and the Mem0 search results as retrieved memory.
3. **After the assistant response:** run memory extraction asynchronously against the latest user message, assistant response, meeting flow state, and retrieved memories.
4. **Write to Mem0 only when extracted facts are durable:** relationship facts confirmed by the user, reusable meeting preferences, strong strategy decisions, or meeting-specific commitments.
5. **Use run scope for event-specific facts:** `run_id = "meeting_prep:{meeting_prep_context_id}"`.
6. **Use user scope for durable facts that should follow the user across meetings:** `user_id = "{auth.id}"`.

### Conversation Tables Found

The live Xano tables that look like first-party conversation storage are:

| Table | ID | Live schema | Current relationship to `openui05/dispatch` |
| --- | --- | --- | --- |
| `copilot_conversations` | `636` | `user_id`, `mode`, `title`, `master_person_id` | Conversation threads for copilot/meeting-style chat; no direct write found in endpoint `#8497`. |
| `copilot_chat_messages` | `637` | `conversation_id`, `role`, `content`, `template_name`, `template_props` | Individual messages for `copilot_conversations`; no direct write found in endpoint `#8497`. |

The meeting prep docs already list these as the copilot conversation tables. They are useful reference tables if OpenUI05 should gain first-party transcript persistence, but the current OpenUI05 dispatch memory path is Zep-first.

## Decision

Use Mem0 as a **retrieval layer for distilled memories**, not as the transcript store and not as the canonical profile store.

For the current OpenUI05 implementation, Mem0 should replace or augment the Zep read/write steps inside `openui05/dispatch`. Keep the frontend contract the same: the frontend sends `query`, stable `thread_id`, and YAML context blocks to Xano; Xano owns memory retrieval, LLM/tool routing, and memory writes.

For the target Mem0 architecture, Xano should remain the source of truth for:

- raw user and assistant messages, if first-party transcript persistence is added
- flow state for outcomes, leverage loops, meeting prep, and how-I-know chats
- deterministic context snapshots generated from Orbiter tables/functions

Mem0 stores:

- durable user preferences and constraints
- important decisions made during a flow
- run-specific working assumptions
- distilled memories from chat, email, calendar, meeting prep, and how-I-know context

Do not send every raw turn or full Xano context dump to Mem0.

## Supported Chat Surfaces

Use the same memory architecture for these surfaces:

- `outcome`: defining or refining an outcome.
- `leverage_loop`: defining or refining a leverage loop.
- `meeting_prep`: preparing for a meeting, including participants, context, questions, and follow-ups.
- `how_i_know`: capturing how the user knows a person and what relationship context should matter later.

Each surface has its own Xano thread/flow state, but they all use the same Mem0 read/write pattern.

## Entity Scope Rules

Use **one primary Mem0 entity per write**.

| Scope | Mem0 entity | Use for |
| --- | --- | --- |
| User memory | `user_id` | Durable facts that should follow the user across flows |
| Run memory | `run_id` | Facts that should only shape one outcome, leverage loop, meeting prep, or how-I-know chat |
| Agent memory | `agent_id` | Rare global behavior notes about an agent, not user-specific facts |

Do not write a single memory with both `user_id` and `run_id` unless you intentionally want Mem0's entity scoping behavior. For Orbiter, if a fact matters at both levels, write two memories: one user-scoped and one run-scoped.

Use metadata for everything else:

```json
{
  "flow_type": "leverage_loop",
  "thread_id": "suggestion_request:123",
  "source_type": "chat",
  "source_id": "copilot_message:456",
  "user_id": "15"
}
```

## Turn Lifecycle

This is the target Mem0 lifecycle. For the current implementation, map "Mem0 search" to the place where `openui05/dispatch` now reads Zep thread context, and map "Mem0 write" to the place where it now posts messages back to Zep.

### 1. Start or resume a chat

1. Create or load the Xano thread state.
2. At draft creation, run `mvp/context/get-all-user-context` (`#13069`) with `{ user_id: auth.id }`.
3. Store the Markdown snapshot in `suggestion_request.llm_summary_context.openui_context.self_markdown`.
4. On every chat turn, feed that stored snapshot into `openui05/dispatch` as `self_yaml_context`, or have dispatch load it server-side from `suggestion_request_id`.
5. Do not write the full context snapshot to Mem0.

For `outcome` and `leverage_loop`, the thread ID must be stable for the lifetime of the `suggestion_request`. The current OpenUI05 comment references:

```txt
thread-outcome-{suggestion_request_id}
```

A cleaner future convention is:

```txt
suggestion_request:{suggestion_request_id}
```

Do not change the convention without migrating existing Zep/Mem0 thread keys.

For `meeting_prep`:

```txt
meeting_prep:{meeting_prep_context_id}
```

For `how_i_know`:

```txt
how_i_know:{user_master_person_id}:{target_master_person_id}
```

### 2. Before each LLM response

Load:

1. System prompt for the surface.
2. Latest context snapshot from Xano.
3. Structured flow state from Xano.
4. Recent transcript from Xano, usually last `8-16` turns.
5. Relevant Mem0 memories from user scope.
6. Relevant Mem0 memories from run scope.

Build the model input in this order. In the existing OpenUI05 path, `$mem_context` already plays this role for the downstream tool endpoints.

```txt
System instructions

Deterministic Orbiter context snapshot from Xano

Current flow state from Xano

Relevant durable user memories from Mem0

Relevant run memories from Mem0

Recent chat transcript from Xano

Latest user message
```

### 3. After each LLM response

1. Save user and assistant messages to Xano if first-party transcript persistence is enabled.
2. Run memory extraction asynchronously.
3. Save extractor output to `ai_memory_event_log`.
4. POST only approved high-confidence memories to Mem0.
5. Poll or receive the Mem0 event result and update `ai_memory_event_log`.

If memory extraction fails, the chat still succeeds.

## When Orbiter Calls Mem0

Use this rule across Orbiter:

```txt
Read from Mem0 right before an LLM response.
Write to Mem0 after the LLM turn, only if memory extraction finds something durable.
```

Mem0 is recall. It is not the place to store full Xano context snapshots, participant JSON, raw emails, raw meeting notes, or raw transcripts.

| Moment | Call Mem0? | Reason |
| --- | --- | --- |
| User opens calendar | No | Calendar browsing does not need memory recall. |
| Nylas webhook receives or updates a meeting | No | Webhooks should keep event data fresh; they should not trigger memory retrieval by themselves. |
| User chooses meeting prep | Usually no | Create/load `meeting_prep_context`, run `#13070` if needed, and store the meeting memo. If this same action immediately asks an LLM to generate prep, then search Mem0 right before that LLM call. |
| User sends the first meeting-prep chat message | Yes, search | Retrieve relevant user and run memories before building the LLM messages. |
| Every later chat turn | Yes, search | Each turn may need different memories depending on the user's latest question. |
| User opens a contact side panel | No | Load the stored relationship summary from Xano; memory recall is only needed when an AI turn starts. |
| User starts or continues a How I Know X chat | Yes, search | Retrieve prior relationship memories before the interviewer asks the next question or summarizes. |
| After the assistant responds | Maybe write | Run extraction and write only durable, high-confidence memories. |
| Running `#13069` user context | No | This is deterministic Xano context, not memory recall. |
| Running `#13070` meeting participant context | No | This is deterministic meeting context, not memory recall. |

##### POST `memory` (#8223)

Step-by-step behavior:

1. Accepts `key`, `value`, optional `category`, `scope`, `team_id`, and `source`.
2. Queries `copilot_memory` for an active row for the authenticated user with the same `key` and `scope`.
3. If an active memory exists, updates that row's `value`, `category`, and `source`.
4. If no active memory exists, creates a new `copilot_memory` row.
5. Writes a create/update entry into `copilot_memory_log`.
6. Returns `{ memory, action }`.

Mem0 status: **no external Mem0 call**; this is local Xano memory storage.

##### GET `memory` (#8222)

Step-by-step behavior:

1. Accepts optional `scope`, `category`, and `team_id` inputs.
2. Queries active `copilot_memory` rows for the authenticated user.
3. Returns `{ memories, total }`.

Mem0 status: **no external Mem0 call**; this reads local Xano memory rows.

##### PATCH `memory/{memory_id}` (#8229)

Step-by-step behavior:

1. Loads the requested `copilot_memory` row by `memory_id`.
2. Confirms the row exists.
3. Confirms the row belongs to the authenticated user.
4. Updates `value`, `active`, `category`, and `confidence`.
5. Writes an update entry to `copilot_memory_log`.
6. Returns `{ memory }`.

Mem0 status: **no external Mem0 call**; this edits a local Xano memory row.

##### DELETE `memory/{memory_id}` (#8230)

Step-by-step behavior:

1. Loads the requested `copilot_memory` row by `memory_id`.
2. Confirms the row exists.
3. Confirms the row belongs to the authenticated user.
4. Soft-deletes the row by setting `active: false`.
5. Writes a delete entry to `copilot_memory_log`.
6. Returns `{ success: true }`.

Mem0 status: **no external Mem0 call**; this soft-deletes local Xano memory.

##### POST `chat` (#8064)

Step-by-step behavior:

1. Loads the selected copilot mode config by `input.mode`.
2. Builds the base system prompt from `copilot_config.prompt`.
3. Runs `mvp/copilot/build-memory-context` (#12665).
4. Appends the returned `[USER MEMORY]` block to the system prompt when memory exists.
5. Appends active person context and network data when provided.
6. Builds the OpenRouter chat-completions message array.
7. Calls `https://openrouter.ai/api/v1/chat/completions`.
8. Parses the LLM response into the frontend response shape.

Mem0 status: **no external Mem0 call**. The memory function reads `copilot_memory`; it does not call Mem0.
