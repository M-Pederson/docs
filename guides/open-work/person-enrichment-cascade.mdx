<Note>
Two functions cooperate on this cascade. `mvp/get-add/master-person` (#12553) runs the synchronous tiers needed to land a clean `master_person` row (linkedin_url + name). `mvp/enrich/enrich-master-person` (#13040) / `mvp/enrich/run-base-person-enrich` (#12554, retired predecessor) then fires Phase 4b (Exa) which fully processes the stashed Exa payload into bios, avatar, and work_experience. ## Why this exists

The seed inputs to `mvp/get-add/master-person` can be sparse ‚Äî sometimes we have a `linkedin_url`, sometimes just an email, sometimes a phone, sometimes a name + company. People Data Labs is the cheapest first call; it accepts email + name + linkedin and returns a 200 with profile data or a 404 with `{"data":{},"error":{"type":"not_found"},"status":404}`. Even when PDL returns 200, the `data.experience` array can be empty.

Before v3.8, a PDL 404 or PDL-with-no-experience produced a junk placeholder row: `master_person.name = "Someone"` (derived from the email local-part), no `linkedin_url`, no profile. That's the failure mode this cascade fixes.

## Tier-by-tier

### Tier 1 ‚Äî People Data Labs

Always fires (when `cascade_depth == 0`). Returns a profile or a 404.

```
GET https://api.peopledatalabs.com/v5/person/enrich
  ?email=...&first_name=...&last_name=...&profile=...
```

Stored at `person_enrich_data.people_data_labs`.

### Tier 2 ‚Äî EnrichLayer Fallback (v3.21)

**Functions:** `api/enrich_layer/person-email` (#12734), `api/enrich_layer/person-linkedin` (#12612)
**Env var:** `$env.ENRICH_LAYER`
**Gate:** `$api1.status == 404`

Two sub-paths, picked by what input we have:

- **Email path (Tier 2a)** ‚Äî gate: `($api1.status == 404) && ($input.email|first is non-empty)` ‚Üí calls `#12734` (`/v2/profile/resolve/email?lookup_depth=deep&enrich_profile=enrich`). The `lookup_depth=deep` parameter goes beyond EnrichLayer's cache and runs heuristic resolution. The helper extracts `response.result.profile`, which has the same shape as `api/enrich_layer/person-linkedin` ‚Äî so downstream code that already knows how to consume `enrich_layer_data` can consume this too. We rebuild the URL from `public_identifier` (`https://www.linkedin.com/in/{slug}`) and assign it to `$linkedinUrl`.

- **LinkedIn path (Tier 2b)** ‚Äî gate: `($api1.status == 404) && ($input.email is empty) && linkedin present` ‚Üí calls `#12612` (`/v2/profile/{linkedin_url}`). Returns the same profile shape; we extract `linkedin_url` (rebuilt from public_identifier) and `profile_pic_url`.

Stored at `person_enrich_data.enrich_layer_email` (email-path response) or `person_enrich_data.enrich_layer_data` (linkedin-path response). `master_link` writes from EL use `source = "Enrich Layer"`. `$inputEmailSource` resolves to `"People Data Labs" | "Enrich Layer" | null`.

### Tier 3 ‚Äî Exa Profile Search

**Function:** `mvp/enrich/get-exa-profile` (#12936)
**Gate:** `$api1.status == 404 && $linkedinUrl is non-empty`

```
POST https://api.exa.ai/search
  { query: <name>, category: "people", numResults: 5, contents.summary, contents.highlights }
```

Exa runs a "people"-category search keyed off the candidate's name, then filters results to the one whose URL contains the LinkedIn slug. It returns a rich `matched` object with `summary`, `highlights[]`, `image`, and `entities[0].properties.workHistory[]`.

The Exa response is **stashed only here** ‚Äî actual processing of `workHistory` into `work_experience` rows happens later in the orchestrator's Phase 4b via the helper `mvp/enrich/process-person-exa-profile` (#13031). That phase reads `person_enrich_data.exa`, iterates `workHistory`, resolves each company via `get-add/master-company`, and adds `master_person` biographies + avatar.

Stored at `person_enrich_data.exa` (json, added in v3.8).

## What lands on `person_enrich_data`

| Column | Source | When populated |
|---|---|---|
| `people_data_labs` | PDL `/v5/person/enrich` envelope | Always (cascade_depth == 0) |
| `enrich_layer_data` | EL `/v2/profile/{linkedin_url}` | When linkedin path taken (Tier 2b) |
| `enrich_layer_email` | EL `/v2/profile/resolve/email` profile | When email path taken (Tier 2a) |
| `exa` | Exa `/search` matched object | Tier 3 ‚Äî PDL 404 + linkedin URL present |

All columns are `json`, `nullable: false`, default `""`. The writes happen at two persist sites:

- **Section 7** of `mvp/get-add/master-person` (existing person path) ‚Äî separate gated `db.edit person_enrich_data` blocks, each conditional on the corresponding stash being non-empty.
- **Section 9** (new person path) ‚Äî a single `db.edit person_enrich_data` with all fields in the data block.

## Resilience

Every tier is wrapped in its own `try_catch`. An EnrichLayer failure doesn't block Exa. Failures write a `log_crash` row with the email/url payload and continue. Phase 4b (Exa processing in the orchestrator) wraps each sub-step independently so one bad workHistory entry doesn't poison the rest.

## What's NOT in the cascade

- **No name/phone-only resolution.** All tiers require either an email or a linkedin URL. A phone-only seed still falls back to the thin-record path.
- **No second-pass on Tier 3.** If EnrichLayer returns a linkedin URL that's wrong, downstream phases happily use it and we end up with a polluted `master_person.linkedin_url`. Watch `log_crash` for issues in the first weeks of production traffic.
- **No location handling yet.** Deferred pending a `master_location` resolver helper.
