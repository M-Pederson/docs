    | Xano ver | Mintlify ver | Action on Xano |
    |---|---|---|
    | `vN` | `vM` where `M < N` | No bump — user already bumped; copy body to Mintlify. |
    | `vN` | `vN` (equal) | **Safety-net bump** → rewrite Xano with `v{N+1}`. |

After writes, the final Xano body (either unchanged or freshly bumped) is the body written to Mintlify.

    Prepends `// v1` to the Xano body (batched with other in-function edits) and appends a new numbered section to the chosen Mintlify file using a standard template with TODO placeholders for description, model, and curl example (model/temperature/etc. are owned by the `check-models` skill, not this one).

  <Step title="Stale sections, on confirmation">
    For each stale Mintlify section (prompt removed from Xano), asks the user per-entry whether to delete. Offers a bulk "delete all" option when there are more than three. Never deletes without confirmation.
  </Step>
  <Step title="Verify">
    Re-fetches each touched Xano function and re-reads each touched Mintlify file to confirm writes landed correctly. Reports final counts.
  </Step>

## Guardrails

- Xano is always the source of truth — Mintlify never writes back to Xano body content.
- **The user owns version numbers.** The skill does not auto-bump. It only rewrites the Xano version as a safety net when drift is detected and the user clearly forgot to bump (see table above).
- Diff with the version line stripped so version metadata never creates phantom drift — version numbers are only compared separately to decide whether the safety-net bump applies.
- `**#NNNN**` function reference is mandatory on every new Mintlify section (it's the cross-system join key).
- Never touches models, temperature, max_tokens, or curl examples — those belong to `check-models`.
- All in-function edits are batched into a single `updateFunction` call.
- Preserves original XanoScript indentation when rewriting `value = """..."""` blocks.
- Missing Mintlify files are tolerated unless explicitly chosen as a home.

# Sync System Prompts — Xano → Mintlify

## Purpose

Keep every LLM system prompt in Xano workspace 3 in lockstep with the Mintlify
\`llm-system-prompts.mdx\` docs. Xano is always the source of truth. The skill
copies Xano → Mintlify verbatim — version header included — and does **not**
auto-bump versions. The user owns version numbers in Xano and will bump them
manually when editing a prompt. The skill only bumps as a **safety net**:
when drift is detected *and* the user forgot to bump in Xano (i.e. Xano's
version is equal to or lower than Mintlify's).

## When to Use This Skill

Any of these trigger activation:
- "sync system prompts" / "update system prompts"
- "system prompt audit" / "check system prompts"
- "are my system prompt docs up to date?"
- "I just changed a system prompt"
- After editing any \`util.template_engine ... as $systemPrompt\` block in Xano

---

## Key Definitions

### What is a "system prompt" in Xano?

A **Template Engine block whose output variable is \`$systemPrompt\`**:

\`\`\`xanoscript
util.template_engine {
  value = """
    // v2
    # Biography Writing System Prompt
    ...
  """
} as $systemPrompt
\`\`\`

- A single function may contain **multiple** \`$systemPrompt\` blocks (e.g.
  function #2537 has one for the short bio and one for the long bio). They
  must be handled independently and matched by order of appearance.
- The \`// vN\` line at the very top of \`value = """..."""\` is the **version
  header**. If absent, treat as unversioned (equivalent to v1).
- **The version line is metadata, not content.** It is stripped before diffing
  Xano vs Mintlify so a stale version header never creates a phantom drift
  loop. It is written back in lockstep on both sides when a bump happens.

### Mintlify system prompt location

Under \`/Users/markl.pederson/Desktop/docs/guides/enrichment/\`:

- \`general/llm-system-prompts.mdx\`
- \`company-pipeline/llm-system-prompts.mdx\`
- \`person-pipeline/llm-system-prompts.mdx\`
- \`film-tv-pipeline/llm-system-prompts.mdx\`
- \`link-url-pipeline/llm-system-prompts.mdx\`

If a file doesn't exist (e.g., a new pipeline), Phase 2 tolerates the
absence — it's only a hard error if the user later chooses that file as a home
for an orphan, at which point the skill offers to create it from the
\`person-pipeline/\` template.

Each prompt appears as a numbered section:

\`\`\`mdx
## N. Title (short description)

\`function/path\` — **#NNNN**

Description…

### Model
…

### System Prompt

<Accordion title="Full system prompt — click to expand">
\`\`\`text
// v2
...prompt body...
\`\`\`
</Accordion>
\`\`\`

Sections are matched to Xano blocks by **function ID (\`#NNNN\`)** plus **order
of appearance within the function** (1st \`$systemPrompt\` block → first
matching doc section for that function, 2nd → second, etc.).

## Execution Protocol

### Phase 1 — Discover every \`$systemPrompt\` in Xano workspace 3

1. \`mcp__xano__listFunctions\` with \`workspace_id: 3\`, \`per_page: 100\`,
   paginate until exhausted.
2. **Scope filter (performance):** only fetch XanoScript for functions whose
   name starts with a pipeline prefix known to contain system prompts —
   \`mvp/imdb/\`, \`mvp/enrich/\`, \`mvp/about/\`, \`mvp/bios/\`, \`mvp/twitter/\`,
   \`mvp/school/\`, \`mvp/company/\`, \`mvp/link/\`. Skip the rest. Re-run without
   the filter if the user reports a missing prompt.
3. For each in-scope function, call \`mcp__xano__getFunction\` with
   \`include_xanoscript: true\`. **Batch in parallel groups of 5–8.**
4. In each \`xanoscript.value\`, find every occurrence of:

   \`\`\`
   util.template_engine {
     value = """
       ...
     """
   } as $systemPrompt
   \`\`\`

   Use regex: \`util\\.template_engine\\s*\\{\\s*value\\s*=\\s*"""([\\s\\S]*?)"""\\s*\\}\\s*as\\s*\\$systemPrompt\`

5. For each match, record:
   - \`function_id\`
   - \`function_name\`
   - \`index\` (1-based order within the function)
   - \`raw_body\` — text between the triple quotes (preserves indentation)
   - \`dedented_body\` — same text with common leading whitespace stripped
     (this is what ships to Mintlify and what gets compared)
   - \`version_line\` — the literal first line if it matches \`^//\\s*v\\d+\\s*$\`,
     else \`null\`
   - \`body_without_version\` — \`dedented_body\` with \`version_line\` removed
     (this is the **canonical diff surface**)
   - \`byte_offset\` — where the block lives in \`xanoscript.value\` (needed to
     rewrite). **Keep all offsets for one function together** so Phase 6 can
     apply every in-function change in a single write.

### Phase 2 — Parse every Mintlify \`llm-system-prompts.mdx\`

Read all five files if present. Missing files are logged but not fatal.

In each file, walk the \`## \` headings and extract for each section:

- \`title\` — the \`## N. ...\` heading text
- \`function_ref\` — the \`**#NNNN**\` reference (required for matching; skip
  with a warning if absent)
- \`prompt_body\` — text inside the first \`\`\`text ... \`\`\`\` fenced block
  under \`### System Prompt\` (typically inside an \`<Accordion>\`)
- \`file_path\` and the byte range of the fenced block (for rewriting)
- \`version_line\` and \`body_without_version\` — same extraction rules as Xano

### Phase 3 — Match Xano blocks to Mintlify sections

Group Mintlify sections by function ID, preserving file order. Match Xano
\`$systemPrompt\` blocks to doc sections by \`(function_id, index)\`.

**Diff surface:** \`body_without_version\` on both sides, compared line-by-line
with trailing whitespace stripped and line endings normalized to \`\n\`.

Classify each pair:

- **Matched + in-sync** — bodies equal
- **Matched + drift** — bodies differ
- **Orphan** — Xano has a block at \`(function_id, index)\` but Mintlify has no
  matching section (new prompt with no home)
- **Stale doc section** — Mintlify has a section referencing a \`function_id\`
  with no matching Xano block at that index (prompt removed from Xano)

### Phase 4 — Report findings

Present the full picture **before touching anything**:

\`\`\`

| ...      | ... | ...      | ...     | ...      | ...     |

❓ Orphans — in Xano but not in Mintlify (Z blocks) — need a home:

| Function | Idx | First 80 chars of prompt |
|----------|-----|--------------------------|
| #NNNN name | 1 | "You are an assistant..." |

⚠️ Stale doc sections (W blocks) — in Mintlify but no matching Xano block:

| Doc File | Section | Referenced Function |
|----------|---------|---------------------|
| ...      | ...     | ...                 |

Show the full planned change set. Ask: **"Apply all changes?"**

On yes, for each function with pending edits:

#### 6a. Drifted blocks — copy Xano → Mintlify verbatim

The user owns version numbers in Xano. Default behavior is to copy Xano's
current body (with whatever version header it has) to Mintlify as-is. The
skill only touches the Xano version as a **safety net** when the user
clearly forgot to bump.

1. **Decide whether to bump Xano's version (safety-net rule).** Compare
   \`xano_version\` and \`mintlify_version\`:

   | Xano version | Mintlify version | Action on Xano |
   |---|---|---|
   | \`vN\` | \`vM\` where \`M < N\` | **No bump.** User already bumped — just copy body to Mintlify. |
   | \`vN\` | \`vN\` (equal) | **Safety-net bump.** User edited the body but forgot to bump → rewrite Xano body with \`v{N+1}\`. |
   | \`vN\` | \`vM\` where \`M > N\` | **Safety-net bump.** Mintlify is ahead, which shouldn't happen — rewrite Xano with \`v{M+1}\`. Warn in the report. |

   In all "no bump" cases, Xano's body is written to Mintlify unchanged.

2. **Apply Xano writes (only if a safety-net bump is required).** Batch all

   \`xanoscript.value\` in-memory: for each bumped block, replace its

   preserving the original indentation of the enclosing block. Write once.
3. **Mintlify:** use \`Edit\` to replace each drifted fenced text block with

**Report every safety-net bump loudly** so the user sees which prompts they

## Rules

- **Xano is ALWAYS the source of truth.** Never edit a Xano prompt body
  based on Mintlify content. Mintlify only tracks Xano.
- **The user owns version numbers.** Do not auto-bump. Copy Xano's version
  header to Mintlify verbatim. The **only** exception is the safety-net
  rule: when drift is detected AND Xano's version is not ahead of Mintlify's
  (user edited the body but forgot to bump), rewrite Xano with \`v{max(xano,
  mintlify) + 1}\` and then copy that to Mintlify. Always surface safety-net
  bumps loudly in the report.
- **Diff with the version line stripped.** The version is metadata. Compare
  \`body_without_version\` on both sides to decide drift. Version numbers are
  compared separately only to decide whether the safety-net bump applies.
- **Always include the function ID** when adding an orphan to Mintlify for
  the first time. The \`**#{function_id}**\` reference is **mandatory** on
  every new section — it is the key used to match Xano blocks to doc sections
  on every future run. Never create a Mintlify section without it.
- **Batch all in-function writes.** Multiple drifted blocks in one function
  must be applied as one \`updateFunction\` call. Writing them sequentially
  invalidates byte offsets.
- **Never touch models, temperature, max_tokens, or curl examples** — those
  are handled by the \`check-models\` skill. This skill only touches the system
  prompt body.
- **Always confirm before applying.** Show the full change set first.
- **Preserve indentation** when rewriting \`value = """..."""\` blocks in
  XanoScript. The dedented comparison is for diff only; the write-back must
  re-indent to match the original block's leading whitespace.
- **Orphan detection must include title/context**, not just a byte offset, so
  the user can make a sensible page assignment.
- **Missing files are tolerated in Phase 2.** The skill only fails if the
  user later picks a nonexistent file as a home, at which point it offers to
  create the file.

## Edge Cases

### Multiple prompts in one function
Function #2537 currently has two \`$systemPrompt\` blocks (short bio, long
bio). Match them to Mintlify sections in the **same order** they appear in
XanoScript. The Mintlify file may have consecutive \`## N.\` and \`## N+1.\`
sections both referencing \`**#2537**\` — use order-within-function to
disambiguate.

### Prompt uses \`{{ $var.context }}\` or similar template tags
These are literal text inside the template. Copy them verbatim to Mintlify —
do not try to resolve or escape them.

### \`// vN\` header inside the prompt body affects the LLM
This is by design. \`//\` lines are benign to most LLMs and the user has chosen
this convention. Do not move the header elsewhere.

### User says "don't bump, even as a safety net"
Default behavior already does not auto-bump — Xano's version is copied
verbatim. If the user explicitly disables the safety-net bump for this run,
skip the Xano version rewrite even when drift is detected with equal
versions. Report the drift in the final summary so they can bump manually
in Xano before the next run.

### Xano API returns empty for \`getFunction\`
Skip and warn. Do not invent content.

### Doc file has a section with no \`**#NNNN**\` ref
Warn the user and skip — cannot match without a function ID.

### New Mintlify file needed
See Phase 5 — offer to create from the \`person-pipeline/\` template.

### Prompt body contains triple quotes
The regex uses non-greedy matching and will stop at the first \`"""\`. If a
prompt literally contains \`"""\` it will be mis-parsed. Flag this case and
ask the user to re-encode (e.g., escape or use a different string delimiter
in XanoScript).
