## The problem

A `Person:Music_Artist` node created during MusicBrainz enrichment never ran the Orbiter **people pipeline** — it's keyed on `linkedin_url`, which MB artists essentially never carry (MB tracks performers, not business identities). So music persons landed in the graph with no avatar, no socials, and no link to the business-side people graph. Person-resolution closes that gap — and, crucially, **the social enrichment (follower scrapers, social_insights, influence) runs off `master_link`, not off LinkedIn**, so it can enrich music persons even without one.

## What shipped this session

### The feature shape (async, Exa-gated, master_person + master_link)
A depth-0 `Person:Music_Artist` is **enqueued** at enrichment time; a standalone cron worker drains the queue and, per artist:

1. **Builds a verification set** from `mb_artist.json` (no extra MB call): known social URLs from `relations[]` (`target-type == "url"`), aliases / legal names from `aliases[]`, plus area / birth-year / genres / disambiguation.
2. **Calls Exa** (`mvp/music/find-person-social`) → classifies results into `linkedin[]`, `socials[]`, `others[]`.
3. **Cross-checks** candidates against the MB set to defeat same-name false positives (the "Winchester 30-30" problem): a non-LinkedIn social is verified by an exact MB url-rel match (host + handle) or an alias-name match in the result title; a LinkedIn hit is verified only with an alias-name match **and** a corroborating signal (verified social, area, or music role / genre in the title).
4. **Creates/links a `master_person`** via `mvp/get-add/master-person-new` (#13039), passing **all verified socials in `links`** (LinkedIn first, if any). #13039 writes **every** link as a `master_link` row — which **auto-triggers the platform scrapers** (`x.com`→twitter #2106, `instagram.com`→IG #2108, `youtube.com`→YT #12670, `linkedin.com`→LinkedIn #12556) that feed `social_insights` (#12669) and `social_media_influence` / follower counts (#2495). Then stamp `mb_artist.master_person_id` and write a **`SAME_AS` bridge** `Music_Artist -w100-> master_person Person`.
   - **`hold_downstream`**: `false` when a LinkedIn is verified (run the full PDL→EnrichLayer people pipeline) — `true` otherwise (skip the LinkedIn-only PDL cascade; the social scrapers still fire from the link writes).
5. **Annotates the node** (`SET n.social_links = [...]`, `n.social_resolved_at`) for graph display, and fetches a **serper.dev avatar** (`fetch-person-avatar`) → `SET n.image_url` (modeled on the group-image flow #13109).

### Decisions locked
- **Every resolved music person becomes a `master_person`** (when it has at least one verified social or LinkedIn), with all socials saved as `master_link` rows — *not just* the LinkedIn ones. This is what unlocks the social scrapers + insights + influence for the ~95% of artists who have no LinkedIn. (Reverses the earlier "annotate-node-only for non-LinkedIn" decision — the socials carry real downstream enrichment value.)
- **Attach ALL verified socials** (not just the high-value twitter/IG/YT/LinkedIn) — the low-value ones (spotify/deezer/bandcamp/apple/…) still feed the platform-agnostic Grok `social_insights`. **IMDb** url-rels are also folded in (v2.1) — `create-master-link` classifies `imdb.com` with a real "IMDb" label + `profile:true`. Database/lyrics/purchase/library-authority rels are excluded. Caps: 25 socials extracted, 20 on the node `social_links`, 25 saved as `master_link`.
- **`SAME_AS` bridge + `master_person_id` stamp for every resolved person.** Node also keeps a `social_links` property (graph display) + serper avatar (immediate node image, independent of the async master_person avatars).
- **Depth-0 seed artists only** (mirrors the label→Company gate) — controls Exa/scraper spend.
- **Standalone cron** (not the MB `process-next` drain) so Exa/serper pacing is decoupled from MB's 1 req/s budget. Batch 5/tick.

## Design evolution
[Music Reconciliation](/guides/enrichment/waterfall/music-reconciliation) originally scoped the person side as a **node merge** (dedup the `Music_Artist` into an existing `Person`). We built a **bridge, not a merge**: MB artists have no `linkedin_url`, so there's rarely an existing business Person to merge into. Instead we **mint/link a `master_person`** (carrying the artist's socials), `SAME_AS`-bridge it to the music node, and let the existing social/people enrichment run on it. The label→Company side remains the `SAME_AS` bridge as scoped.

## XanoScript traps discovered
Captured in `docs/CLAUDE.md` § XanoScript authoring traps:
- **#16** — MCP `updateTableSchema` (full-schema replace) silently **drops column `filters=trim`/validators** even when passed; data preserved, only the validator lost. No incremental column-add via the MCP.
- **#17** — a `var` declared + `var.update`'d **outside** a `try_catch` reads back **empty inside** the try block's `api.lambda`s. Compute/assign the var **inside** the same `try` scope where the lambda consumes it. (Silently zeroed `social_count` until the vset extraction was moved into the try.)
- **#18** — `function.run` targets can store as `""` **even when the callee already exists** (hit on a task create). Re-store the caller after the callee is confirmed; always re-fetch and check the stored target is non-empty.
- **#19 🔴** — the **Xano CLI `xano workspace push` WIPES a data source's records** (even with `--include` for one table — it re-imports the workspace). Restoring `mb_artist` `filters=trim` via the CLI **emptied the entire `sandbox` data source**; `live`/`staging` survived. `--dry-run` didn't show it. NEVER `workspace push` for a schema edit on a live workspace — use the Xano UI.
