## Finding ‚Äî Cinq's `master_company` shows a domain slug, not its name

While checking the label‚Üícompany side, traced why **Cinq Music Group's `master_company` (row 11788) displays `cinqmusicgroup_com` instead of its name**:

- The real name **did** resolve ‚Äî into `legal_name` (`"Cinq Music Group LLC"`), alongside correct `company_domain`, `linkedin_url`, `about`, and `founded`. Only `company_name` (the display field) is wrong: it holds the domain with dots‚Üíunderscores.
- **Root cause is a general company-pipeline gap, not music-specific.** `mvp/get-add/master-company-new` (#12558) falls back to the domain for `company_name` when no clean name is available at create time (line ~1832, `$varCompanyName|is_empty ‚Üí $varDomain`), and nothing ‚Äî `enrich-master-company` (#12992) included ‚Äî ever backfills `company_name` from the later-resolved `legal_name`. The row was created via the general `ScrapeCreators Linkedin` path, then enriched; `company_name` never caught up.
- The `OWNED_BY` linker is **not** at fault ‚Äî it attached to this pre-existing row, and it *names* fresh owner companies correctly (it created "GoDigital Media Group" and "Universal Music Group" cleanly the same run). This is distinct from the existing [placeholder-company guard](/guides/robert-mark/edge-audit) (which blocks `"Unknown Company"`-style names): a domain slug like `cinqmusicgroup_com` isn't in that blocklist and the row has a real domain, so it passes the guard.

**Fix options (not applied):** (a) narrow ‚Äî backfill `company_name` from `legal_name` on the affected rows; (b) root ‚Äî add a `company_name`-from-`legal_name`/LLM-name promotion step to `enrich-master-company` (#12992), which fixes Cinq and every other slug-named row.
