`Household` nodes represent shared living spaces. Multiple residents connect to a household via `RESIDED_AT` edges carrying per-person move-in/move-out windows. Shared cohabitation between two people is derived by finding a common `Household` with overlapping residency windows.

See [Schema Design â€” Household](/guides/known/schema-design#household) for the full node shape.

---

## `known/households/create_household`

Creates a `Household` node with address hash, type, and start date. Does **not** store raw address data â€” only a privacy-preserving hash.

**Inputs:** `address_hash`, `household_type`, `start`, `source`
**Outputs:** `household_uuid`
**`household_type` values:** `family`, `couple`, `roommates`, `care_setting`, `parent_child`, `other`

## `known/households/add_resident`

Creates a `RESIDED_AT` edge from a person to an existing household with move-in timestamp and role.

**Inputs:** `person_uuid`, `household_uuid`, `move_in`, `role`, `source`
**Outputs:** `resided_at_uuid`
**`role` values:** `primary`, `partner`, `child`, `roommate`, `relative`, `other`

## `known/households/end_residency`

Sets `move_out` on an existing `RESIDED_AT` edge. **Does not delete.** If all residents of the household now have `move_out` set, also sets the `Household.end` field to the most recent move-out date.

**Inputs:** `resided_at_uuid`, `move_out_date`
**Outputs:** updated `RESIDED_AT` edge; optionally updated `Household.end`

## `known/households/match_or_create_by_address`

Given an address (or address hash), finds an existing `Household` with the same hash + overlapping time window or creates a new one. Used by ingest pipelines that detect shared addresses from contacts / email headers / calendar locations.

**Inputs:** `address_hash`, `reference_date`, `household_type_hint`
**Outputs:** `household_uuid` (existing or newly created)
**Matching logic:**
1. Look up all `Household` nodes with the same `address_hash`
2. Filter to those whose `[start, end]` window contains or overlaps with `reference_date`
3. If one match, return it
4. If multiple matches, return the one with the tightest temporal fit
5. If no match, create a new `Household` and return the new uuid

## `known/households/compute_cohabitation_overlap`

For a given pair of persons, finds all `Household` nodes they both have `RESIDED_AT` edges to, computes the overlap of their per-person windows, and returns an array suitable for inclusion in `KNOWS.shared_context_cache.households`. Called by [`compute_shared_context`](/guides/known/scope/cache#known-cache-compute-shared-context).

**Inputs:** `person_a_uuid`, `person_b_uuid`
**Outputs:**
```json
[
  {
    "household_uuid": "...",
    "overlap_start": "2008-06-14",
    "overlap_end": "2020-04-02",
    "overlap_days": 4309,
    "household_type": "family"
  }
]
```

Overlap is computed as `[max(a.move_in, b.move_in), min(a.move_out, b.move_out)]`. If either `move_out` is null, current date is used.

## Design notes

- **Address data is never stored raw.** Only `address_hash` (sha256 or similar) lives on the node. The raw address is hashed at ingest time and discarded.
- **A household persists after everyone moves out.** The `end` field records when it dissolved. Historical residents remain linked via their `RESIDED_AT` edges.
- **People can have concurrent residencies.** College students who live at school part of the year and with parents the rest; parents with shared custody; people in transition. Multiple `RESIDED_AT` edges to different `Household` nodes with overlapping time windows are allowed.
- **Household merging** happens when the ingest pipeline realizes two separate Household nodes are the same physical residence (e.g. `address_hash` collision or manual merge). This is deferred to implementation time â€” see [open questions](/guides/known/scope#open-implementation-questions).
