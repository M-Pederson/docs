Cascade leaves discovered at depth greater than 0 land in `queue_enrich_company` and wait there until something drains them into full enrichment. Two paths drain it: the scheduled cron **Task #127** `process-company-queue` (ships `active = false`) and the on-demand operator drainer [`process-enrichment-queue` #12816](/guides/enrichment/waterfall/company-functions#queue--gate-infrastructure). The person waterfall has an identical sibling — **Task #126** `process-person-queue` over `queue_enrich_person` — that shares the exact drain logic below. Queue schema: [Company Waterfall Tables](/guides/enrichment/waterfall/company-tables) · upsert and priority semantics: [Company Waterfall overview](/guides/enrichment/waterfall/company-waterfall#queue--upsert-semantics).

## Drain logic

Every drainer reads the same signals off each queue row:

- **`priority_tier`** — an int, **lower is higher priority**. Meanings (as of 2026-07-03): **0** = earned super-priority (any entity escalated to the top by repeated references), **1** = investors — VC firms, angels, and deal partners all enter here — plus current employer / founders, **2** = past employers, **3** = schools, **4** = certification issuers / volunteering orgs / contributors. Non-investor entities enter at their base tier and drop one tier per **3** re-references (`count`), floor 0.
- **`cascade_depth`** — hops from the seed (0 = seed, 1 = direct, 2+ = secondary).
- **`processing`** — a boolean in-flight lock.
- **`count`** — a reference counter that increments each time the entity is re-queued (dedup is by `master_company_id` / `master_person_id`).

### Pick order

Both crons select one row per tick:

```text
where  = processing == false AND priority_tier <= 1
sort   = priority_tier asc, cascade_depth asc, id asc
return = single
```

Since 2026-07-03 the scheduled crons **filter `priority_tier <= 1`** — only drain-eligible rows (investors at tier 1, plus anything escalated to tier 0) are picked; tiers 2–4 are never drained by the crons. Within the eligible set the drainer takes the **lowest tier first** (0 before 1), then the **shallowest** depth, then FIFO by id. The on-demand `#12816` drainer is separate and still reaches every tier (default `max_tier: 4`).

### Per-row lifecycle

<Steps>
  <Step title="Lock">
    `db.edit … processing = true` on the picked row. The lock is advisory — it is not atomic with the pick, so two ticks racing the same row is possible.
  </Step>
  <Step title="Enrich">
    `function.run` the orchestrator synchronously at the row's `cascade_depth` — `enrich-master-company` #12992 for Task #127, `enrich-master-person` #13040 for Task #126.
  </Step>
  <Step title="Delete on success">
    `db.del` the row. Completed rows leave the table entirely — `processing` is a transient lock, not a persisted "done" marker.
  </Step>
  <Step title="Release on failure">
    The `try_catch` catch resets `processing = false` and writes a `log_crash` row, so the row is re-eligible on the next tick.
  </Step>
</Steps>

<Warning>
  No retry counter and no dead-letter. With stable sort keys, a deterministically-crashing row is re-picked on every tick forever — head-of-line poison-pill blocking that stalls the whole queue behind one bad row.
</Warning>

### The drainers

<Card icon="clock-rotate-left">
  ```text
  process-company-queue — Task #127   ·   process-person-queue — Task #126
  ```

  **Single-row tick every 600 s (10 min), `active = false`.** Each picks one `processing == false AND priority_tier <= 1` row by the sort above, locks it, runs the orchestrator synchronously at the row's `cascade_depth`, then deletes on success or releases + `log_crash` on failure. No kill-switch check of their own — they rely on the orchestrator's internal gates. Max throughput if flipped on: ~6 entities/hour per cron. The 2026-07-03 change added the `priority_tier <= 1` drain filter (investors + escalated rows only); the 2026-07-02 change removed the out-of-enum `phase: "queue"` `log_crash` write (which could make the crash logger itself fail).
</Card>

<Card icon="globe">
  ```text
  process-enrichment-queue — #12816
  ```

  **On-demand batch drainer (a function/endpoint, not a scheduled task).** Per queue it fetches up to 100 `processing == false` rows sorted `priority_tier asc`, then filters in-app by `priority_tier <= max_tier` (default 4), `cascade_depth <= max_depth` (default 99), and `count >= min_count`, and processes the first `max_items` (default 10). Since v1.1 it routes through the gated orchestrators and checks the person / company kill switches before each block. The default `max_tier: 4` means it drains all tiers 0–4 unless you pass a lower `max_tier`.
</Card>

<Warning>
  **Both crons ship `active = false`** (verified 2026-07-02) — nothing drains the queue automatically today. Queued leaves sit in `queue_enrich_company` / `queue_enrich_person` until an operator runs `#12816`. Flip a task active in the Xano UI to enable scheduled draining.
</Warning>

## Priority-tier model (built 2026-07-03)

<Note>
  **Shipped.** The popularity-gated model is live. **All investors** — VC firms, angels, and deal partners (companies and people) — enter at `priority_tier: 1` (set by `resolve-investors-edges` #12702; investor firms moved 2 → 1, partners/angels were already 1). Every non-investor enters at its base tier (default 4) and drops one tier per **3** re-references (`count`), floor 0 — so an entity referenced enough climbs to the top. The re-upsert merge in `#12854` / `#12855` composes this with the existing `priority_tier = min(existing, incoming)` rule and applies the drop **incrementally** (per-tick, against the stored tier) so it never compounds. The `priority_tier | first_notempty:4` coercion is now **`first_notnull:4`**, so an explicit tier `0` / `1` is stored (previously an explicit `0` silently became `4`). The scheduled crons drain **`priority_tier <= 1`** (investors, plus anything escalated to `0`) and remain `active = false` until go-live. **Note:** the pre-existing backlog is all tier ≥ 1, so when the crons are enabled the `<= 1` filter will drain existing tier-1 rows (old partner/angel enqueues) as well as new ones — expected, since those are genuine investors.
</Note>
