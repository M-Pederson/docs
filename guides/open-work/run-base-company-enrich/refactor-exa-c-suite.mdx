## Build instructions

### Context

- **Function:** `mvp/enrich/run-base-company-enrich-v4` (#12992)
- **Group:** `add new exa person` — visual-builder group wrapping the per-match logic inside Section 7's `foreach ($exaMatches) { each as $match }` loop
- **Loop state at this point:** we have already decided we're going to add this Exa person as a `master_person`. The call site is `function.run "mvp/get-add/master-person"`. We still need to derive the right `title` to pass for this person *at the company we're currently enriching*.

### Goal

For the current `$match` (one Exa person from `exa_c_suite.matches[]`), find the workHistory entry representing the person's **current role** (whichever company that's at — not necessarily the master_company we're enriching), and derive both `$currentTitle` and a resolved `master_company_id` for that current company.

### Test case

- **Master company we're enriching:** Frame.io (`$input.master_company_id`)
- **Exa person:** Samantha Gioia — workHistory has multiple Frame.io entries, but the entry with `dates.to == null` is `Technical Product Manager` at Frame.io.
- **Expected:** `$currentTitle = "Technical Product Manager"`. Resolve Frame.io's domain via `company.website` → call `mvp/get-add/master-company` → get master_company_id (likely the same as `$input.master_company_id`). Write that id + the title onto master_person.

### Steps to derive `$currentTitle` and the current `master_company_id`

1. **Read workHistory.** Pull `$match.entities.0.properties.workHistory` (array of `{dates: {from, to}, title, company: {id, name, website, linkedinUrl, ...}, location}`).
2. **Find the current-role entry.** Scan for the entry where `entry.dates.to == null` (no end date = currently held). This is the person's current job, **regardless of which company it's at** — Exa C-suite results sometimes surface people whose current role is at a different company than the one we're enriching.
   - If exactly one entry has `dates.to == null`: that's `$currentEntry`.
   - If multiple have `dates.to == null` (rare — concurrent positions): take the one with the latest `dates.from`.
   - If zero have `dates.to == null` (person has only past roles): `$currentTitle` stays null, the master_person row gets no `current_title` / `master_company_id` fill from this path. Still proceed with steps 2–7 (biography, avatar, work_experience writes, etc.).
3. **Extract domain from the current-role company's website.** `$currentEntry.company.website` → strip protocol + www + trailing slash to get bare domain (`https://katabatic.tv` → `katabatic.tv`). Regex: `^https?:\/\/(?:www\.)?([^\/\s]+)`.
4. **Resolve to a master_company.** Call `function.run "mvp/get-add/master-company"` with the derived domain (and any other fields it expects). Returns a `master_company_id` for the current-role company. Store as `$currentCompanyId`.
5. **Set `$currentTitle`.** From `$currentEntry.title`.
6. **Use these on the master_person row** (in the upsert helper):
   - `master_person.master_company_id` ← `$currentCompanyId` (the resolved id from step 4 — may or may not equal `$input.master_company_id`)
   - `master_person.current_title` ← `$currentTitle`

### Person enrichment strategy — locked in

- **We are NOT calling `mvp/get-add/master-person` from this loop.** That function re-runs the full add+enrich cascade (PDL, Enrich Layer, deep-research, etc.) and we don't want that cost/latency on every Exa C-suite hit when we already have rich data in hand.
- **Use the Exa data directly to enrich the person inline.** Within the `add new exa person` group, for each `$match`:
  1. Upsert a `master_person` row (and linked `master_email` / `master_link` / `master_company_person` rows as needed) using fields derived from the Exa payload — `$match.title` (display name), `$match.entities.0.properties.location`, `$match.url` (LinkedIn), `$currentTitle`, plus the company-link edge tying them to `$input.master_company_id` with the title attached. **Set `visibility: true`** on the new master_person row (confirmed field name on schema #139) — Exa-discovered C-suite are public-facing identities, so they should be surfaceable in profile/search results from the moment they're created.
  2. **Add biography rows from Exa.** Call `function.run "mvp/bios/add-person-biography"` (#2536) **twice** — once per Exa source. Both calls use:
     - `master_person_id`: from step 1
     - `data_source_id`: `101` (Exa)

     With different `biography` inputs per call:

     | Call | `biography` input |
     |---|---|
     | **2a — highlight** | `$match.highlights.0` (first/most-relevant highlight string Exa surfaced for this person — typically a markdown-formatted bio excerpt) |
     | **2b — summary** | `$match.summary` (the longer LLM-summarized professional bio at the top of each match; includes current role, full work history, education, location) |

     #2536 handles its own dedup (md5 hash on cleaned text), emoji-strip, URL-only guard, and empty-input guard — safe to call unconditionally on both. If either field is empty/missing, #2536 will short-circuit on its empty-biography validation; no need to gate the calls from the caller side. The hash-based dedup also prevents a duplicate row if `highlights.0` and `summary` happen to be identical (rare).
  3. **Resolve the avatar.**
     - Call `function.run "mvp/enrich/find-avatar"` (#12846) with the master_person identifiers as input. Capture the response as `$avatarResult`.
     - Call `function.run "mvp/avatar/replace-avatar"` passing `avatar: $avatarResult.avatar` (or whatever the avatar field is named on #12846's response — verify before wiring) along with the `master_person_id`.
  4. **Write `work_experience` rows from `workHistory[]`.** After master_person + avatar are done, loop over `$match.entities.0.properties.workHistory` and insert one `work_experience` row per entry into table `work_experience` (#147).
     - **Order the loop master_company-matches first.** Process every entry where `entry.company.name|icontains:$masterCompany.company_name` first (the matching entries can be multiple — a single person may have held multiple positions at the same company; write a row for each), then process the remaining entries.
     - **Field mapping (verified against schema #147):**

       | `work_experience` field | Source |
       |---|---|
       | `master_person_id` | from step 1 (just-created master_person) |
       | `master_company_id` | `$input.master_company_id` **on matching-company rows only**; leave default on non-matching rows (downstream "link work history to master_companies" pass will resolve those) |
       | `first_name` | first token of `$match.title` (Exa puts the person's display name here, e.g. `"Samantha Gioia"` → `Samantha`) |
       | `last_name` | remainder of `$match.title` after first token (`Gioia`) |
       | `title` | `entry.title` |
       | `company_name` | `entry.company.name` |
       | `company_domain` | **domain extracted from `entry.company.website`.** `https://katabatic.tv` → `katabatic.tv`. Strip `https?://`, strip leading `www.`, strip trailing slash and any path. Regex: `^https?:\/\/(?:www\.)?([^\/\s]+)` — take group 1. Null if `entry.company.website` is null. |
       | `company_linkedin_url` | `entry.company.linkedinUrl` (now populated thanks to the #12988 fix above) |
       | `start_year` / `start_month` / `start_day` | parse `entry.dates.from` (ISO format, can be `YYYY-MM-DD`, `YYYY-MM`, or `YYYY`). Split on `-`: 3 parts → all three populated; 2 parts → year + month, day blank; 1 part → year only. Schema fields are text, so store as zero-padded strings (`"01"`, not `1`). |
       | `end_year` / `end_month` / `end_day` | same parser, against `entry.dates.to`. If `entry.dates.to` is null, leave all three blank and set `current: true` (see below). |
       | `current` | `true` if `entry.dates.to == null`, else `false` |
       | `location` | `entry.location` |
       | `data_source_id` | `101` (Exa — same data_source_id v4 uses for Exa-attributed `enrich_history_company` rows) |

     - **Do NOT write `entry.company.description`.** That field is multi-KB scraped LinkedIn/Crunchbase prose (news headlines, company updates, workforce breakdowns). Skip it entirely — `work_experience.description` stays default-empty.
     - **Skip these schema fields** (not relevant to Exa-sourced rows): `main` (deprecated), `person_permalink`, `organization_permalink`, `linkedin_company_id` (deprecated), `edge_uuid` (set by downstream FalkorDB edge-creation pass), `triples_created_at` (set by triples pass), `description`.
  5. Optionally persist the raw `$match` payload to `person_enrich_data.exa_data` (or equivalent column) so the next enrichment pass has it locally and doesn't re-call Exa.
  6. **Kick off async deep-research bio — LAST step in the loop.** Call `function.run "mvp/deep-research/deep-person-basic"` (#12833) with the just-created `master_person_id`. Placed at the end on purpose so every other write (work_experience, optional `person_enrich_data` payload, etc.) has landed before #12833 reads master_person + addons to build the deep-research context.
     - **Call shape:** `function.run "mvp/deep-research/deep-person-basic" { input = { master_person_id: $newPerson.id } }`. The function takes a single int input. No need to capture/use the response inline — it's a fire-and-kickoff (returns the `profile_enrichment_job` row after the external service accepts the job).
     - **What it does downstream:** #12833 reads master_person + addons (master_company, certifications, work_experience), YAML-encodes them as context, formats a long system prompt, then calls `mvp/enrich/deep-research-person-or-company` (#12747). #12747 fires a POST to the `orbiter-enrich` Cloud Run service and inserts a row into `profile_enrichment_job` (#686) with `processing: true` and the returned `job_id`.
     - **Completion is async via webhook.** When the Cloud Run job finishes (typically 30s – several minutes), it POSTs to `webhooks/enrich-profile` (endpoint #8303), which updates the `profile_enrichment_job` row, writes the parsed biography array to `master_person.deep_bio`, runs `mvp/bios/add-person-biography` (the same #2536 we already called inline) with the joined biography, runs `mvp/bios/create-llm-person-bios`, runs `mvp/social/social-insights/get-social-insights`, and runs `mvp/node/update-person-node`. We don't need to wire any of that here — it all happens off the webhook.
     - **Cost flag.** Model is `moonshotai/kimi-k2.6:online` with up to 16K output tokens + parallel web search — roughly **$0.10–$0.50 per person**. At ~6 c-suite hits per company × the full backfill queue, this becomes the dominant per-company cost of the v4 changes. Open question: gate this call (e.g., skip when `master_person.deep_bio` is already populated from a prior pass, or skip on `cascade_depth > 0`, or only run for the top N matches by some confidence signal).
- **Then queue for additional enrichment.** After the inline upsert, enqueue this `master_person_id` via the person-enrichment queue (likely `mvp/queue/upsert-enrich-person` — verify exact function name) so downstream workers run the deeper person flow (PDL backfill, Signal NFX person side, master-person v2 enrich, etc.) asynchronously.
- **Queue priority.** Set the queue priority **one tier below** the priority we use for investors (so investor enrichment stays ahead of these C-suite additions in the worker queue). Look up the investor tier value in the relevant queue function before wiring; pass `priority_tier: <investor_tier> + 1` (or whatever the queue's "lower priority" convention is — investigate whether higher numbers = lower priority or the inverse).

## Upstream improvement — populate `linkedinUrl` + `website` in #12988

<Note>
  **Built 2026-05-14.** Both fixes below are live on `mvp/enrich/get-exa-company-c-suite` (#12988). Test by running v4 against a master_company with `exa_c_suite` cleared, then inspect `company_enrich_data.exa_c_suite.matches[*].entities[0].properties.workHistory[*].company` — `linkedinUrl` and `website` should be populated on enriched companies (was null on every row pre-fix). Note: `educationHistory[*].institution.linkedinUrl` now uses the `/school/` form for schools/universities thanks to the unified regex.
</Note>

`mvp/enrich/get-exa-company-c-suite` (#12988) currently returns `linkedinUrl: null` and `website: null` on every enriched company inside `workHistory[*].company` and `educationHistory[*].institution`, even when `expand_companies: true`. Both fields *exist* in the response shape — they're just always null. Two bugs in the post-processing lambdas:

### Bug 1 — `linkedinUrl` regex matches the wrong path segment

In the "Build enriched lookup list" lambda, the LinkedIn URL is derived from the Exa company ID like this:

```js
const m = id.match(/library\/company\/([^\/?#]+)/);
if (m && m[1]) linkedinUrl = 'https://linkedin.com/company/' + m[1];
```

But the actual Exa company IDs use the `organization` segment, not `company`:

```
https://exa.ai/library/organization/jfjgkbqvgr9   ← Shepherd Center
https://exa.ai/library/organization/k7t465sztky   ← Google
https://exa.ai/library/organization/0jtkn73hfcj   ← Frame.io
```

The regex never matches → `linkedinUrl` stays null on every row.

**Fix options (pick one):**

- **(a)** Swap the regex literal `company` → `organization`. Minimal change; assumes the `<slug>` portion of Exa's URI always matches the LinkedIn company slug (true in the samples we've seen but not guaranteed by Exa).
- **(b) (Recommended)** Parse the LinkedIn URL out of the markdown `text` body Exa returns. The scraped company summary reliably contains a line like `- LinkedIn: linkedin.com/company/shepherd-center`. Regex against `text` for `LinkedIn:\s*(linkedin\.com\/(?:company|school)\/[^\s\n]+)` and prefix `https://`. Handles both companies (`/company/`) and educational institutions (`/school/`) in one shot — important because the same lambda enriches both `workHistory[].company` and `educationHistory[].institution`.

Lean: do **(b)**, fall back to **(a)** when `text` is empty or the line isn't present.

### Bug 2 — `website` source is the wrong field

The lambda asks `/contents` for `extras: { links: 5 }` and then picks the first http link that isn't `linkedin.com` or `exa.ai`. The problem: Exa's `extras.links` returns links *found inside the scraped content* (related companies, citations, references on the LinkedIn page) — not the company's homepage. For a LinkedIn-page scrape, the company's own homepage rarely surfaces there, which is why `website` is null.

The homepage *is* reliably present in the markdown `text` body, as a line like `- Homepage: shepherd.org`.

**Fix:** parse `text` for the `Homepage:` line. Regex something like `Homepage:\s*([^\s\n]+)` against `text`, then prefix `https://` if the captured value doesn't already start with `http`. Drop the `extras.links` strategy entirely (saves a per-row token cost on the `/contents` call too — can drop `extras: { links: 5 }` from the request body).

### Edge cases to handle in the parser

- `text` is empty / missing for a given company (Exa hasn't indexed it). Both fields stay null — acceptable.
- `text` contains `Homepage:` but the value is a relative path, slack channel, or some other non-URL (rare but seen in malformed scrapes). After regex capture, validate that the value looks like a domain (`/^[a-z0-9.-]+\.[a-z]{2,}$/i`) before prefixing `https://`. If invalid, leave `website` null.
- Multiple `LinkedIn:` lines (some scrapes include a personal LinkedIn for the founder + a company LinkedIn). Take the first match — Exa's markdown puts the company LinkedIn at the top of the Company Details section.

### Out of scope for this improvement

- Validating that the derived `linkedinUrl` actually resolves to a real LinkedIn page (would require an HTTP HEAD per company — too expensive).
- Resolving `website` against a canonical domain registrar (subdomains vs apex). Take whatever Exa printed in the Homepage line; downstream consumers can normalize.
