These are the foundation functions. They must exist before any ingest, scoring, or lifecycle code can run. Every write to the Known graph goes through one of these helpers — no direct Cypher `CREATE` from callers.

## Design principle

No caller writes raw Cypher against Known edges. All creates, updates, and invalidations flow through these helpers so that canonicalization, temporal rules, and derived-field recomputation are enforced in exactly one place.

## `known/helpers/canonicalize_pair`

<Note>
**Build status:** ✅ Built in Xano as function **#12767** on 2026-04-10. Inline tests pending (XanoScript test-block syntax needs verification).
</Note>

Takes two person UUIDs and returns them in canonical order (lower first, by lexicographic string comparison). Every write path uses this before any `CREATE` or `MERGE` against the graph.

**Inputs:** `person_a_id` (text), `person_b_id` (text)
**Outputs:** `{ lower_id: text, higher_id: text }`

**Guards:**
- **Self-pairs are rejected.** If `person_a_id == person_b_id`, the function throws `inputerror` with message `"Cannot canonicalize a pair of identical uuids"`. A self-pair would mean "a relationship between a person and themselves" which is meaningless for KNOWS and any facet, and silently accepting it would mask caller bugs. Upstream code (ingest paths, identity merge) MUST ensure it never calls this helper with the same uuid twice.
- **Empty strings are rejected.** Both inputs are required non-empty text. Xano's `filters=trim|min:1` handles this at the input layer.

**Implementation notes:**
- Comparison is lexicographic string comparison on the uuid values. UUIDs are strings so `<` works correctly.
- Pure function — no side effects, no database access, no graph writes.
- Idempotent and stateless — calling it twice with the same inputs returns the same output.

**Tests to verify:**
- `a_id < b_id` case: returns `{lower: a_id, higher: b_id}`
- `b_id < a_id` case: returns `{lower: b_id, higher: a_id}`
- Identical uuids: throws `inputerror`

## `known/helpers/create_knows_edge`

Creates a new `KNOWS` edge with the canonical pair order enforced. The only entry point for `KNOWS` creation — no callers should do raw Cypher `CREATE (a)-[:KNOWS]->(b)`.

**Inputs:** `person_a_id`, `person_b_id`, `evidence_rows`, `origin_context`, `origin_entity_id`, `origin_detail`
**Outputs:** `knows_edge_uuid`
**Side effects:**
- Runs initial `shared_context_cache` computation
- Sets initial `strength_score` (may hit the presence floor if no direct interaction evidence, or the historical floor if either endpoint has `deceased_at`)
- Derives `known_since`, `first_evidence_at`, etc. from the evidence array

**Retrospective freeze check on creation.** Before setting the initial `strength_score`, the function checks if either endpoint has `deceased_at` set:

```
if person_a.deceased_at is not null OR person_b.deceased_at is not null:
  knows.score_frozen_at     = max(person_a.deceased_at, person_b.deceased_at)
  knows.score_frozen_reason = 'deceased'
  knows.historical_edge     = true   # signals the historical floor should apply
```

This handles the **deceased-at-ingest** case (e.g., creating a KNOWS between two historical figures from digitized letters or press archives). The edge is born frozen — it never enters a "live, updatable" state. The `scoring/apply_freeze_if_deceased` function is idempotent and still runs on the next scoring pass, but the state is already correct from creation.

**When `historical_edge` is true**, `compute_strength_score` applies the [historical edge floor](/guides/known/scope/scoring#historical-edge-floor) of 15 instead of the normal presence floor of 8. This reflects that historical relationships with deceased endpoints can't be meaningfully scored by interaction-decay formulas and should carry at least a "contact" tier baseline to acknowledge the real-world relationship existed.

## `known/helpers/append_evidence`

Appends new evidence to an existing `KNOWS.evidence` array and recomputes derived fields.

**Inputs:** `knows_edge_uuid`, `evidence_row`
**Outputs:** updated `KNOWS` edge
**Recomputes on every call:**
- `confidence` (sum of weights, capped at 1.0)
- `known_since`, `first_evidence_at`, `last_evidence_at`
- `first_direct_interaction_at`, `last_direct_interaction_at`
- `connected_platforms`, `total_social_connections`

**Critical:** Every derived-from-evidence field must be recomputed on every append. Implementation should enumerate the list explicitly — a silent "forgot to recompute X" bug here would let derived fields drift from evidence over time.

## `known/helpers/compute_confidence`

Computes `confidence = min(1.0, sum(weights))` from an evidence array. Pure function.

**Inputs:** `evidence_rows[]`
**Outputs:** `confidence: float`

## `known/helpers/check_evidence_floor`

Evaluates an evidence array against the floor rule: `(strong_count ≥ 1) OR (weak_count ≥ 2)`.

**Inputs:** `evidence_rows[]`
**Outputs:**
```json
{
  "passes_floor": true,
  "strong_count": 1,
  "weak_count": 1,
  "total_confidence": 0.75
}
```

## `known/helpers/derive_known_since`

Derives temporal fields from an evidence array. Key rule: temporal derivations use `evidence_occurred_at`, not `first_seen_at`. This is what enables retroactive backdating when LinkedIn tells us "they connected in 2019" even though Orbiter only saw the evidence in 2024.

**Inputs:** `evidence_rows[]`
**Outputs:**
```json
{
  "known_since": 2019,
  "first_evidence_at": "2019-05-22T00:00:00Z",
  "last_evidence_at": "2024-08-21T00:00:00Z",
  "first_direct_interaction_at": null,
  "last_direct_interaction_at": null
}
```

## `known/helpers/create_facet`

Generic facet creation. Enforces canonical pair order, sets `valid_from`, `source`, `created_by`, `created_at`.

**Inputs:** `facet_type`, `person_a_id`, `person_b_id`, `source`, `properties`
**Outputs:** `facet_uuid`

## `known/helpers/invalidate_facet`

Sets `valid_to`, `invalidation_reason`, `invalidated_by` on an existing facet. **Never deletes.** Invalidation preserves history.

**Inputs:** `facet_uuid`, `invalidation_reason`, `invalidated_by`, `invalidated_at`
**Outputs:** updated facet
