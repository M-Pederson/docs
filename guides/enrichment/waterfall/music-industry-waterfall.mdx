## Source: the MusicBrainz API

| Property | Value |
|---|---|
| Base | `https://musicbrainz.org/ws/2/` |
| Format | append `?fmt=json` (or `Accept: application/json`) |
| Auth | none required |
| **Rate limit** | **1 request/second, global**, with a meaningful `User-Agent` (e.g. `Orbiter/1.0 ( mark@orbiter.io )`). Exceeding it returns `503`. |
| Lookup | `/{entity}/{MBID}?inc={subqueries}` |
| Browse | `/{result}?{linked}={MBID}&limit=100&offset=N&inc=...` |
| Search | `/{entity}?query={lucene}&limit=N` |
| Pagination | `limit` (default 25, **max 100**) + `offset`; increment by results returned |

**`inc=` is the efficiency lever.** A single lookup can pull aliases, tags, genres, ratings, and every relationship type (`artist-rels`, `recording-rels`, `work-rels`, `release-rels`, `label-rels`, `event-rels`, `place-rels`, `area-rels`, `url-rels`) in **one** request. One artist + two browse pages ‚âà 3 requests ‚âà 3 seconds. This is why the cascade must be tightly depth- and tier-capped.

<Warning>
**The 1 req/s ceiling is global, not per-cron.** Four independent queue crons each firing lookups would collectively blow the limit. The design routes **every** MusicBrainz call through one token-bucket throttle and a **single unified queue processor** ‚Äî see [Rate-limit architecture](#rate-limit-architecture). Realistic throughput is **~1 entity per 3‚Äì6 seconds**, so this pipeline is deliberately slow and bounded.
</Warning>

## Core entities ‚Üí graph nodes

MusicBrainz's 13 core entities map to Orbiter graph nodes as follows. **Seed entities** (minimum bootstrap set) are **artist, recording, label, event**; the rest are created via cascade.

|---|---|:--:|---|
| `artist` (type = Person) | `Entity:Person:Music_Artist` | ‚úÖ | **is** an Orbiter `Person` (dedup-reconciled, not a separate node) |
| `artist` (Group / Orchestra / Choir) | `Entity:Music_Group` | ‚úÖ | ‚Äî (new node type ‚Äî a band is never a `Person`) |
| `recording` | `Entity:Recording` | ‚úÖ | ‚Äî |
| `label` | `Entity:Music_Label` | ‚úÖ | `Company` via `SAME_AS` |
| `event` | `Entity:Music_Event` (+ type sub-label) | ‚úÖ | `Live_Event` (future reconcile) |
| `work` | `Entity:Work` (+`:Song` etc.) | cascade | ‚Äî |

## Edge designs

Weights follow the canonical [Edge Weights](/guides/ontology/edge-weights) scale (**lower = stronger**; hub-node edges sit Weak or below; never sub-1). Every edge stamps `uuid`, `created_at`, `updated_at` via `timestamp()`. The bands: Defining 1‚Äì5, Very strong 6‚Äì15, Strong 16‚Äì25, Moderate 26‚Äì40, Weak 41‚Äì65, Negligible 100.

<Note>
The tables below summarize each edge. For the **full parameter list per edge** ‚Äî including the standard envelope (`uuid`, `weight`, `created_at`, `updated_at`, `description`, `description_generated_at`) carried by every edge ‚Äî see the dedicated **[Music Edges](/guides/enrichment/waterfall/music-edges)** reference. **`Artist`** in a pattern means `Person:Music_Artist` or `Music_Group`; `MEMBER_OF` is always `(Person:Music_Artist)‚Üí(Music_Group)`.
</Note>

## Pipeline architecture (proposed functions)

Mirrors IMDB: **gates** ‚Üí **orchestrators** ‚Üí **cascade helpers** ‚Üí **queue infra** ‚Üí **unified cron**, all under a **namespace `mb/`** in Xano workspace 3. No LLM/scrape phases ‚Äî orchestrators map structured JSON straight to Cypher.

### Entry gates

<Card icon="play">

```
mb/get-add/artist          (proposed)
mb/get-add/recording       (proposed)
mb/get-add/label           (proposed)
mb/get-add/event           (proposed)
```

One gate per seed entity. Each: normalize MBID ‚Üí dedup on a 30-day `enriched_at` window ‚Üí ensure stub row (`mb_artist` / `mb_recording` / ‚Ä¶) ‚Üí branch: **sync-enrich** when `cascade_depth == 0 && queue == false` (invoke the orchestrator directly), else **upsert the queue row**. Seed-id propagation: when no caller seed is passed, the just-created stub is the seed. The MBID is the natural key ‚Äî no URL normalization needed (unlike IMDB).

</Card>

### Orchestrators (one structured lookup ‚Üí node + edges ‚Üí cascade)

<Card icon="layer-group">

```
mb/run-base-artist-enrich   (proposed)
```

**Phases:** 1 throttled lookup `artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels` ‚Üí 2 save raw JSON to `mb_artist` ‚Üí 3 `MERGE` Artist node (+`:Person`/`:Group`) + stamp `node_uuid` ‚Üí 4 `FROM_AREA` edge + `HAS_GENRE` edges ‚Üí 5 `MEMBER_OF` edges from `member of band` rels (queue the other artists) ‚Üí 6 cascade: browse `recording?artist=` + `release-group?artist=` (paginated, throttled) ‚Üí 7 stamp `enriched_at` / `enrich_success` / `cascade_depth`. Per-phase `try_catch` ‚Üí `log_crash`.

### Cascade helpers (depth-capped fanout)

<Card icon="wrench">

```
mb/cascade-from-artist     ‚Üí band members T1 ¬∑ collaborators T2 ¬∑ recordings T2 (cap 50) ¬∑ release-groups T2 ¬∑ events T3
```

<Card icon="wrench">

## Rate-limit architecture

The defining constraint. Every MusicBrainz call passes through a token bucket, and **one** cron drains all queues so total throughput never exceeds 1 req/s.

<Card icon="gauge-high">

```
mb/rate-limit/acquire   (proposed)
```

A token-bucket gate backed by a singleton `mb_rate_limit` row (`last_call_at`, `tokens`). Before any `api.request` to musicbrainz.org, the orchestrator calls `acquire`: if `now - last_call_at < 1000ms`, it **defers** (re-queues the entity with a short backoff and returns) rather than sleeping. Stamps `last_call_at = now` on grant. Centralizes the limit across every function that touches the API.

</Card>

<Card icon="clock-rotate-left">

```
process-queue-musicbrainz   (proposed Task)
```

**One unified cron**, not four. Each tick it selects the single highest-priority pending row **across all queues** ‚Äî order `priority_tier ASC, cascade_depth ASC, entity_rank, id ASC` (entity_rank biases seeds/artists ahead of bulk recordings) ‚Äî locks it, dispatches the matching `run-base-*` orchestrator, deletes on success / releases + `log_crash` on failure. **Schedule ~ every 2 s.** Because each orchestrator may need several lookups (entity + browse pages), real wall-clock is **~3‚Äì6 s per entity** once `acquire` paces the internal calls. A unified cron makes the global 1 req/s trivially enforceable; four parallel crons would not.

</Card>

<Warning>
**Throughput is intentionally low.** At ~1 entity / 3‚Äì6 s, a depth-0 artist with its top recordings + albums + members takes minutes, and a deep cascade takes hours ‚Äî by design. The depth cap (2), tier caps, and per-entity caps below keep total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.
</Warning>

## API errors, retries & rate limits

Reference for handling the MusicBrainz `ws/2` API. Every call to musicbrainz.org goes through `mb/rate-limit/acquire` (pacing) **and** a shared retry wrapper (`mb/api/request`, below) that applies this policy ‚Äî orchestrators never call `api.request` directly.

<Warning>
**A meaningful `User-Agent` is mandatory.** Requests without one (or with a generic library default) are throttled hard or blocked. Use `Orbiter/1.0 ( mark@orbiter.io )`. There is no API key.
</Warning>

### Status codes

| Status | Meaning | Retry? | Handling |
|---|---|:--:|---|
| `200` | OK | ‚Äî | parse JSON |
| `400` | Bad request ‚Äî malformed MBID / Lucene query | ‚ùå terminal | `log_crash` `phase: api_bad_request`; drop the queue row |
| `404` | Entity not found / merged away | ‚ùå terminal | stamp `enrich_success = false`, mark stub `gone`, drop row |
| `429` / `503` | **Rate limited / temporarily unavailable** | ‚úÖ backoff | honor `Retry-After`; exponential backoff ‚Äî the common case |
| `500` / `502` / `504` | Server / gateway error | ‚úÖ backoff | transient ‚Äî retry with backoff |
| timeout / network | no response | ‚úÖ backoff | retry with backoff |

<Note>
MusicBrainz returns **`503` for both** genuine overload **and** rate-limit violations (it does not consistently use `429`). Treat `503` as "back off and retry," never as a hard failure.
</Note>

### Retry policy

```text
max_attempts   = 5
base_delay     = 1000 ms                       # ‚â• the 1 req/s floor
backoff        = base_delay * 2^(attempt-1)    # 1s ‚Üí 2s ‚Üí 4s ‚Üí 8s ‚Üí 16s
jitter         = ¬± 250 ms                       # avoid herd on the shared cron
respect        = Retry-After header when present (overrides computed delay)
terminal       = {400, 404}                     # never retried
on_exhaustion  = release lock ¬∑ log_crash phase="api_exhausted" ¬∑ re-queue one tier lower
```

- **Defer, don't sleep.** Xano functions shouldn't block, so `acquire` and the retry wrapper **re-queue** the entity with a `next_attempt_at` backoff timestamp instead of sleeping in-process. The unified cron skips rows whose `next_attempt_at` is in the future.
- **Backoff is per-entity; pacing is global.** The 1 req/s token bucket governs *spacing* between any two calls; the retry backoff governs *re-attempts* of one failed call. Both invariants hold simultaneously.
- **Retries are idempotent.** Every write is `MERGE`-based with `coalesce`d `created_at`, so re-running a partially-applied enrich never duplicates nodes or edges.

<Card icon="cloud-arrow-up">

```
mb/api/request   (proposed)
```

Single choke-point for every MusicBrainz call: takes `{path, inc, params}`, calls `acquire` (pace) ‚Üí issues the request with the canonical `User-Agent` ‚Üí classifies the response per the table above ‚Üí applies the retry policy. Returns `{ok, status, json, attempts}`. Centralizing here means rate-limit + retry behavior lives in exactly one place, and the orchestrators stay simple.

</Card>

## Depth model

Identical to IMDB ‚Äî hard cap at depth 2.

| Depth | Meaning | Behavior |
|:--:|---|---|
| **0** | Seed (user / bootstrap) | Full lookup + all edges + cascade fanout |
| **1** | Direct discovery from a seed | Stub via `get-add`. Queued. Cron enriches + expands, but **only T1‚ÄìT2 children queue at depth 2** |
| **2** | Transitive | Stub queued. Cron enriches + writes edges, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why cap at 2?** MusicBrainz is ~2.5 M artists / ~30 M recordings. One prolific artist ‚Üí ~20 albums ‚Üí ~250 recordings ‚Üí each recording ‚Üí 1 work + ~5 credited people ‚Üí thousands of entities at depth 2. Combined with the 1 req/s limit, an uncapped cascade would never drain. Depth 2 + tier caps + per-entity caps (20 albums, 50 recordings) keep a seed's full expansion in the low thousands of API calls.
</Check>

## Loop prevention (six layers)

1. **Dedup gate in `get-add`:** `mbid` already enriched within 30 days ‚Üí return existing, no lookup.
2. **Queue upsert:** duplicate `mbid` ‚Üí `count++`, tier upgrade. One row per MBID.
3. **Depth cap:** depth 2 enriches but does not cascade; depth 3 never queued.
4. **Tier cap at depth ‚â• 1:** only T1‚ÄìT2 discoveries queue at depth 2.
5. **Kill switch:** `check-kill-switch-mb` zero-API blocker.
6. **Global rate limiter:** `mb/rate-limit/acquire` + unified cron enforce 1 req/s across the whole pipeline (the layer IMDB doesn't need).

## Seed flows

### Seed artist (depth 0)

<Steps>
  <Step title="Kill switch + dedup">
    `check-kill-switch-mb`; lookup `mb_artist` by `mbid`, return if enriched &lt; 30 days.
  </Step>
  <Step title="Throttled lookup">
    `acquire` token ‚Üí `GET artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels`. Save raw JSON.
  </Step>
  <Step title="Node + base edges">
    `MERGE (a:Entity:Artist {mbid})` (+`:Person`/`:Group`), stamp `created_at`/`updated_at`. Write `FROM_AREA` + `HAS_GENRE`. Reconcile to `Person` via `SAME_AS` if matched.
  </Step>
  <Step title="Membership edges">
    For each `member of band` rel ‚Üí `MEMBER_OF` edge (instrument/dates/original) + queue the member artist at **T1**.
  </Step>
  <Step title="Cascade (browse, throttled, paginated)">

  </Step>
  <Step title="Mark success">

  </Step>
</Steps>

## Queue tables (proposed schema)

One table per seedable entity, keyed on normalized `mbid`. Shared shape:

| Field | Type | Notes |
|---|---|---|
| `id` | int | PK |
| `created_at` | timestamp | |
| `mbid` | text (unique) | natural key |
| `name` / `title` | text | provisional, from discovery context |
| `entity_id` | int nullable | FK into `mb_artist` / `mb_recording` / ‚Ä¶ once the stub exists |

| `source_function` | text | e.g. `cascade-from-artist` |

## Open questions to resolve before build

<CardGroup cols={1}>
  <Card title="Artist ‚Üî Person reconciliation" icon="link">
    When does a `Person:Music_Artist` become a `SAME_AS` link to an existing Orbiter `Person`? Proposed: a separate reconciliation pass keyed on name + shared URLs (Wikidata/IMDb/social) + ISNI ‚Äî **not** at ingest, to avoid false merges. Same for `Music_Label` ‚Üí `Company`.
  </Card>
  <Card title="Area mapping" icon="earth-americas">
    Reuse Orbiter `Country`/`Region`/`City` (MERGE by name, stamp `mbid`) vs. dedicated `MusicArea`? Reuse is proposed so geography stays unified, but MusicBrainz areas (subdivisions, historical regions) may not match cleanly ‚Äî needs a mapping table.
  </Card>
  <Card title="Genre vs tag" icon="tags">
    MusicBrainz exposes both curated `genres` and free `tags`. Proposal: `HAS_GENRE` ‚Üí `Genre` nodes from `genres` only (weight 100); keep `tags` as a node property (csv) for search, not edges.
  </Card>
  <Card title="Recording / release explosion" icon="triangle-exclamation">
    A popular work has hundreds of recordings; an album has dozens of releases (per-country pressings). Caps (50 recordings, 20 albums) and Weak edge weights mitigate, but we may want to ingest only **canonical** releases (status = Official, earliest per country) rather than every pressing.
  </Card>
  <Card title="Cron cadence vs. backfill bursts" icon="gauge-high">
    A 2 s unified cron drains ~30 K entities/day. A large bootstrap (e.g. all Sub Pop artists) may warrant a dedicated higher-priority lane or an offline bulk import from the [MusicBrainz database dumps](https://musicbrainz.org/doc/MusicBrainz_Database/Download) instead of the live API.
  </Card>
</CardGroup>
