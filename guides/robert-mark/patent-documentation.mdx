## What is Orbiter.io?

Orbiter is an AI-powered relationship intelligence platform. Given any goal a user is trying to accomplish — hire a key employee, raise a round, find a co-founder, source an acquisition target, get advice — Orbiter surfaces the people in the user's professional network best positioned to help, with grounded, context-rich explanations of **why** each person matters and the path and connections to that person.

The platform serves **organizations, enterprises, and Networked Operators** — dealmakers, investors, and executives in media, technology, and enterprise — for whom relationship leverage is the primary asset.

## Core Value Proposition

The **Core Value Proposition** of Orbiter.io is its ability to leverage the context of the user's entire network to provide context-rich **suggestions** that help the user achieve any **OUTCOME** or assist any person in their network (referred to as a **LEVERAGE LOOP**). Orbiter.io's **Core Value Proposition** is a **suggestion engine** that analyzes the user's entire network to deliver targeted suggestions for **OUTCOMES** or **LEVERAGE LOOPS**.

For example: a user may have 20,000+ people in their network — Orbiter surfaces the six most meaningful matches, with grounded, context-rich explanations of the **why** and **how** each person could help the user achieve the outcome.

Users connect their email and calendar accounts and import contacts and network data from any source into Orbiter.io. Orbiter reconciles each contact against the **ORBITER UNIVERSE GRAPH** — its shared canonical layer of previously enriched entities. Contacts that already exist in the **ORBITER UNIVERSE GRAPH** are copied into the **USER GRAPH** fully enriched. Contacts that don't are first added to the **ORBITER UNIVERSE GRAPH**, enriched there, and then synced into the **USER GRAPH**.

<Note>
The **ORBITER UNIVERSE GRAPH** contains only public, verified information. None of the user's private data — emails, phone numbers, or other private context tied to a contact — is stored in the **ORBITER UNIVERSE GRAPH**. Private data from the user is stored only in the **USER GRAPH**.
</Note>

<Note>
Orbiter.io also supports **ORGANIZATION GRAPHS** — a third graph type scoped to a company or organization, synced with both the **ORBITER UNIVERSE GRAPH** and the **USER GRAPHS** of every member of that organization. An administrator controls granular permissions over what information is shared within the organization's network.
</Note>

## Intelligence Layers

Orbiter.io also generates **INTELLIGENCE LAYERS** on top of each graph type:

- The **ORBITER UNIVERSE GRAPH** produces **GENERAL INTELLIGENCE**.
- Each **ORGANIZATION GRAPH** produces **ENTERPRISE INTELLIGENCE**.
- Each **USER GRAPH** produces **USER INTELLIGENCE**.

These layers are generated by cron-scheduled analytics jobs that pull from their respective graphs, providing statistical analysis and insights that can be combined with graph-based queries when synthesizing the actual suggestions presented to the user.

## What is the IP we want to protect?

Schemas and ontologies are not patentable on their own — they describe data organization, which sits outside §101 subject-matter eligibility. Waterfall-style cascading enrichment, taken in isolation, also faces a moderate-to-dense prior-art landscape: multi-stage enrichment pipelines, priority-tiered queues, and depth-capped cascades all appear in published systems.

What we want counsel to claim is the **end-to-end process** — the user's journey from seed entity through enrichment, graph population, intelligence-layer generation, interview-driven goal capture, hybrid retrieval, and the synthesized suggestions returned with grounded explanations. Novelty sits in the combination and the coupling between stages; the §101 footing comes from the specific technical improvements the integrated method produces (lower per-record cost, lower latency, higher accuracy, higher query quality). Per-layer mechanics are best positioned as dependent claims that pin narrower fallback positions if the integrated independent claim is contested.

We also want counsel to consider protection over the **UI and user experience** — the step-by-step method by which a user moves from a stated goal to a curated set of network-grounded suggestions, with the user's entire network as the search surface. The novelty sits in the guided flow itself: eliciting an **OUTCOME** or **LEVERAGE LOOP**, applying the full graph as context mid-flow, and returning synthesized matches with grounded explanations of the **why** and **how**. The strongest §101 footing will likely tie this interaction model back to the underlying technical improvements rather than framing it as a pure UI method.

## Terminology: "Leverage Loops" vs. "Outcomes"

Two related but distinct concepts drive every suggestion Orbiter.io produces. They differ in *who* the user is trying to help and *what kind of entity* surfaces in the results.

| | **OUTCOME** | **LEVERAGE LOOP** |
|---|---|---|
| **Who is being helped** | The user themselves | Another person in the user's network |
| **What can be suggested** | Any entity type — people, companies, live events, funding rounds, etc. | People only |
| **Framing of the goal** | "Help me hire a key engineer." | "Help me help Sarah find investors." |

The same set of standard goal types — hiring key talent, finding investors, finding a co-founder, sourcing acquisition targets, getting advice, and so on — applies to both modes. In an **OUTCOME**, the goal is framed for the user; in a **LEVERAGE LOOP**, the same goal is framed for a target person in the user's network.

A **LEVERAGE LOOP** itself has two sub-modes, distinguished at the moment the user creates it:

- **Specific-help Leverage Loop.** The user defines what they want to help the target person with — e.g., "help Sarah find investors." This shares the **OUTCOME** backend pipeline; only the framing of the goal differs.
- **Generic Leverage Loop.** The user creates a leverage loop for the target person without a specific goal. The system analyzes the user's entire network — potentially thousands of contacts — and surfaces a concise set of the highest-impact introductions: people most likely to create mutual benefit with the target person.

All three paths — **OUTCOME**, Specific-help **LEVERAGE LOOP**, and Generic **LEVERAGE LOOP** — share the same underlying graph and intelligence layers. They diverge at the retrieval and synthesis stages.

## User Story: Account setup

<Steps>
  <Step title="Create an account">
    User signs up with email and password, OAuth, or single sign-on (SSO).
  </Step>
  <Step title="Connect email and calendar">
    User connects their email and calendar accounts via OAuth.
  </Step>
  <Step title="Import contact data">
    User imports their contacts from any data source — Apple Address Book, any CRM, a LinkedIn export, a CSV, or any other data source where their contacts are.
  </Step>
  <Step title="Reconcile and enrich">
    If any of the user's contacts already exist in the **ORBITER UNIVERSE GRAPH**, they are in an enriched state and are copied to the **USER GRAPH**. If they do not exist in the **ORBITER UNIVERSE GRAPH**, they are added, fully enriched, and then copied to the **USER GRAPH** along with additional connected nodes of entities that have a known relationship to that contact.
  </Step>
  <Step title="Real-time webhook updates">
    Any email or calendar event from a user's connected accounts automatically adds new contacts and logs the interactions. Additional third-party integrations enable real-time webhook updates from other data sources — CRMs, Slack, and more.
  </Step>
</Steps>

## User Story: Create Outcome

<Steps>
  <Step title="Create an outcome">
    The user describes what they want to achieve by text or voice input to an **AI interviewer**.
  </Step>
  <Step title="Interview for context">
    The **AI interviewer** — functioning as a **goal elicitation interface** — interviews the user for more context on the specific outcome until enough context exists. It provides a dynamically generated (**GenUI**) multiple-choice menu to make the interview process more efficient.
  </Step>
  <Step title="Choose result count and format">
    Once the **AI interviewer** is satisfied that it has enough context to dispatch the outcome request to a downstream retrieval-and-synthesis pipeline, it asks the user how many results they would like and what format the results should be returned in. Current result options include **Suggestions** (in-app suggestions), a CSV, a PDF, or a **COLLECTION** (a group of results stored inside Orbiter). Result options may be expanded in the future.
  </Step>
  <Step title="Dispatch and return results">
    The outcome request is dispatched to the downstream retrieval-and-synthesis pipeline, which returns the results to the user in the chosen format.
  </Step>
</Steps>

## Under the Hood

Orbiter.io is a multi-tenant typed property-graph operating system for relationships:

- A **typed property graph** with thirteen-plus node types and one-hundred-plus edge types capturing people, companies, schools, funding rounds, awards, geographies, patents, publications, and more, and the relationships between them — **autonomously expanding** through a depth-capped, priority-tiered enrichment cascade. We refer to this as the **ORBITER ENRICHMENT ENGINE**.
- A **backend orchestration layer** that ingests email and calendar signals to keep the graph alive — relationships carry validity intervals, last-confirmed timestamps, and source evidence on every edge.
- A **node-level vector index** over entity embeddings in the graphs, supporting semantic similarity search across every entity in the graphs.
- **Per-field vector indexes** on richer entity properties — narrative descriptions, full-document embeddings, structured profile fields, and LLM-derived narratives.
- A **multi-model LLM routing layer** that selects fast extraction models, reasoning synthesis models, and adjudication models based on task and cost.
