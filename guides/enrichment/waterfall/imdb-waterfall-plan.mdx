### Entry points

```text
mvp/imdb/get-add/imdb-person #12873
```

Gate for any IMDB person URL. Normalizes URL, dedups on a 30-day window, ensures stub, then either sync-enriches (depth 0 + `queue: false`) or upserts the queue row. **Primary entry point for new work.**

```text
mvp/imdb/get-add/imdb-title #12874
```

Gate for any IMDB title URL. Same shape as the person gate — stub, dedup, sync-call branch, queue upsert.

```text
mvp/imdb/add-imdb-to-master-person #10509
```

Attaches IMDB data to an **existing** `master_person_id`. Preserves inline scrape + LLM + `db.add imdb_person` because the caller owns the `master_person` link. Credits loop delegates to `get-add/imdb-title` (queued, T2, depth+1). **v1.2 (2026-04-24):** added Section 6 — non-fatal sync call to `#12887 add-award-records-from-imdb-person` after the credits loop. Closes the Phase-8 asymmetry vs orchestrator `#12880` (honorary / person-only awards now captured on main-pipeline entries, not just film-anchored awards via the title cascade). Called from `process-person-phase-8` **#12829**.

```text
mvp/imdb/verify-imdb-url #12677
```

Reverse lookup — given a `master_person_id`, search Tavily + disambiguate with LLM, return verified IMDB URL. Called from `process-person-phase-8` **#12829**.

### Orchestrators (full enrichment pipeline per entity)

```text
mvp/imdb/run-base-imdb-person-enrich #12880
```

7-phase person orchestrator: scrape → save → create `master_person` + `master_link` + `birthday_join` → bios + avatar + nodes → `add-award-records-from-imdb-person` (**#12887**, v1.1c) → `cascade-credits-from-person` (v1.2: passes `sync_top_credits: 5`) + `cascade-awards-from-person` → stamp `enriched_at` / `enrich_success` / `cascade_depth`. Per-phase `try_catch` → `log_crash`.

```text
mvp/imdb/run-base-imdb-title-enrich #12881
```

7-phase title orchestrator: scrape → tagline + save → `film_tv` node + stamp `node_uuid` → `add-award-records-from-title` → `cascade-crew-from-title` + `cascade-awards-from-title` → stamp `enriched_at` / `enrich_success` / `cascade_depth`.

### Cascade helpers (fanout from one entity to tier-queued related entities)

```text
mvp/imdb/cascade-credits-from-person #12875
```

Walks `imdb_person.json.credits[]`, tiers by role + rating, queues titles via `get-add/imdb-title` at `depth+1`. Hard cap: top 20 T3 entries by rating. v1.1 `sync_top_credits` input: when `> 0` AND `cascade_depth == 0`, top-N credits by `(tier ASC, rating DESC)` are sync-enriched inline (`queue: false, cascade_depth: 0`).

```text
mvp/imdb/cascade-crew-from-title #12876
```

Walks `film_tv.json`, tiers directors/writers/cinematographers/lead producer/top-3 cast, queues people via `get-add/imdb-person` at `depth+1`. Dedups by `imdb_url` — first tier wins.

```text
mvp/imdb/cascade-awards-from-person #12877
```

Walks `film_tv_award` rows for a person, queues project titles at T1 / `depth+1`.

```text
mvp/imdb/cascade-awards-from-title #12878
```

Walks `film_tv_award` rows for a title, queues co-winners (who have resolved `imdb_person_id`) at T2 / `depth+1`. `shared_with` names-only co-winners remain unresolved pending future URL-resolution work.

### Award scrapers (LLM extract → `film_tv_award` rows + queue primary nominees)

```text
mvp/imdb/add-award-records-from-imdb-person #12887
```

Scrapes `{imdb_url}/awards/` for a person. Accepts `cascade_depth`, delegates project-title creation to `get-add/imdb-title` (queued, T2, `depth+1`). Fixed entity is `imdb_person_id`; variable entity is the project. Written 2026-04-23 to replace the deleted **#12633** — same design, mirror of **#12879** with the sides flipped.

```text
mvp/imdb/add-award-records-from-title #12879
```

Mirror of **#12887** for titles. Resolves `primary_nominee_imdb_url` via `get-add/imdb-person` (queued, T2, `depth+1`). Writes `film_tv_award` rows with `project_title` / `project_imdb_url` echoed from the title.

### Graph writers (FalkorDB nodes + edges)

```text
mvp/edge/add-node-edge-imdb-award #2334
```

Writes `Film_TV` + `Film_TV_Award` nodes, `HAS_AWARD` edge, and (when a Person UUID resolves) `WORKED_ON` + `WON_AWARD` / `NOMINATED_FOR` edges to FalkorDB for each `film_tv_award` row. Two-query split so a missing Person stub never kills the whole write. Stamps `film_tv_award.node_uuid` on success.

```text
mvp/node/add-film-tv-node #2320
```

Creates a `Film_TV` node in FalkorDB with dynamic type label (`Movie`, `TV_Series`, `Short_Film`, etc.). Called from `run-base-imdb-title-enrich` Phase 3.

```text
mvp/imdb/tagline-generator #10516
```

LLM-generates a 96-char-or-less tagline from a film description. Called from `run-base-imdb-title-enrich` Phase 2.

```text
mvp/queue/upsert-imdb-title #12872
```

Reads `kill_switch_imdb` env var, counts `imdb_person` + `film_tv`, fail-open `try_catch`. Called from both `get-add` gates before any work.

    - **E.** `|length` is **not** a valid Xano filter (`Unable to locate func entry: length`). `|count` works on arrays/objects but there is no built-in string-length filter. Replaced both `$markdownLength` and `$rawResponseLength` assignments with `api.lambda` blocks returning `typeof $var.X === 'string' ? $var.X.length : 0`.
    - **F.** `api.request timeout: 90 → 300` seconds. A 580-row Scorsese awards page produces ~30k output tokens at OpenRouter's ~50 tok/s, breaching the 90s ceiling and forcing the server to return a 200 with an `.error` body (no `.choices`).

Previously `var $rawResponse { value = $api2.response.result.choices[0].message.content }` — which throws `Unable to locate var` when the response is an error body. Replaced with an `api.lambda` try/catch that returns `""` on any shape mismatch, guaranteeing Change D's empty-log fires instead of the function aborting mid-stack.

    **Third-round fix (v1.1e, #12887 only) — sandbox re-QA infra all green but content still empty:**
    - **H.** On Scorsese, the 283k markdown → 159,477-char LLM response → **still** parsed to zero awards. Added `api.lambda` that captures `first 2500 + last 2500 chars` of `$rawResponse` into the empty-log `data.raw_response_preview` field (5KB cap, well under the `log_crash.data` JSON limit). Next empty run will show the raw body in a single `searchTableContent` call.

    **Fourth-round diagnostic (v1.1f, #12887 only) — preview revealed `parsed_type: "string"` instead of `"array"`:**
    - **I.** The 159k raw response looked like valid JSON, and the parser appeared to succeed, but `typeof $awards === "string"` downstream. Added a `parsed_type` + `parsed_length` telemetry capture to the empty-log path to distinguish "parser returned string" (silent catch-as-value) from "parser returned array but empty".

    **Fifth-round fix (v1.1g, #12887 only) — parsedPreview surfaced the real culprit:**
    - **J.** v1.1f's `parsedPreview` (first 300 + last 300 chars) revealed the "parsed string" was actually the **JSON parse error message itself**: `"[Line 22:Char 31] Expected double-quoted property name in JSON at position 159633..."`. Root cause: Xano's `api.lambda` **silently catches thrown JS exceptions and returns the error message as the lambda's string return value** — rather than propagating the throw to the outer `try_catch`. The downstream code then treated `"[Line 22:Char 31] Expected..."` as a successful parse result. v1.1g added a defensive re-parse guard (if `typeof $awards === "string"` after parse, attempt `JSON.parse` once more and fall through on failure) and trimmed the output fields to 7 (per the stricter `add-award-records-from-title` schema).

    **Sixth-round fix (v1.1h, #12887 only) — 580 × verbose schema exceeded max_tokens:**
    - **K.** v1.1g's parse guard fired cleanly, confirming the response was truncated mid-object at char 159,633. Math: 10-field schema × ~500 chars/award × 580 awards ≈ 72k tokens > 65,536 `max_tokens` ceiling. Fixed by **slimming the LLM prompt schema from 10 fields to 7** (dropped redundant `award_name`, `award_event_url`, `award_ceremony_url` — all derivable from retained fields). New footprint: ~36k tokens for 580 awards, well under ceiling. Added belt-and-suspenders JSON salvage fallback that truncates at the last `},` and retries parse — handles any future edge-case truncation without reshipping.

    **Xano gotchas unlocked during Fix #5's five rounds:**
    1. `updateFunction` silently fails (returns "Unable to get successful response" with no server persistence) when `|set:"key":(some|filter|chain)` nests a filter call inside a `|set` arg. **Workaround: hoist nested filter results into a plain `var $name { value = ... }` block first, then reference the var.**
    2. `|length` is not a valid Xano filter. Use `api.lambda { code = "return ($var.X || '').length" }` for string length. `|count` is for arrays and objects.
    3. `$api2.response.result.choices[0].message.content` is a fragile path — on OpenRouter server-side timeouts or upstream errors, the response is a 200 with an `.error` body instead of `.choices`, and naive traversal aborts the whole function. **Always wrap LLM response extraction in a defensive `api.lambda` with try/catch that returns an empty fallback.**
    4. **(Big one)** `api.lambda` **silently catches thrown JS exceptions and returns the error message as the lambda's string return value** rather than propagating to outer `try_catch`. Surface: `JSON.parse("malformed")` inside an `api.lambda` doesn't throw upstream — it returns the V8 error message (`"[Line X:Char Y] Expected..."`) as a plain string, which downstream code treats as a successful parse result. **Workaround: always wrap risky ops in an **inner** explicit `try { } catch (e) { return { ok:false, error:e.message }; }` inside the lambda and check the `.ok` flag downstream.**
    5. Token-cost budgeting for LLM-extraction prompts: `chars_per_row × expected_row_count < (max_tokens × ~4)`. The Scorsese case — 10-field schema × ~500 chars × 580 awards = ~72k tokens — showed this is an easy-to-miss ceiling when porting a per-title prompt (usually 5–20 awards) to the person side (often 100s–1000s). Slim the schema proactively.

    **Close-out verification (Scorsese re-run after v1.1h):** 452 `film_tv_award` rows, 0 empty-log rows, 0 parse CRASH rows, canonical tracking-stripped URLs, `awards_json` populated.

### Implementation notes captured during build

- **URL normalization lives in get-add:** `trim | split:"/?" | first | rtrim:"/"`. All queue rows and `imdb_person` / `film_tv` rows are keyed on the normalized URL.
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` — NOT `function.call` (not valid in Xano).
- **`updateTableSchema` drops text `validators`:** the Xano MCP `updateTableSchema` tool silently strips `filters=trim` from text fields when replacing schema. Low impact when inputs are already trimmed upstream, but worth knowing.
- **Queue upsert is the single source of truth:** `get-add` always routes through `upsert-imdb-person` / `upsert-imdb-title`. No raw `db.add` into queue tables anywhere.
- **Kill switch is fail-open:** missing env var → `try_catch` logs to `log_crash` and defaults OFF.
- **`cascade_depth` + `queue` param on get-add:** when both `cascade_depth == 0` AND `queue == false`, `get-add` invokes the orchestrator synchronously after the stub exists and returns `action: "enriched_direct"` — no queue entry written.
- **Orchestrators are 7-phase with per-phase `try_catch` → `log_crash`:** each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` + stack trace.
- **Phase 1 guard:** if scrape+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early — no `master_person` gets created from a junk row.
- **Cascade helpers are pure dispatchers:** every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"`. Every `function.run` to `get-add` is wrapped in `try_catch` → `log_crash`.
- **Seed ID propagation:** a depth-0 seed invocation via `get-add` must set `seed_imdb_person_id` / `seed_film_tv_id` to the just-created stub's own id when no explicit seed is passed. Required for loop-prevention to work.
- **Top-N sync credits on person seeds (`sync_top_credits: 5`):** `cascade-credits-from-person` v1.1 sync-enriches the top 5 credits inline when the person is at `cascade_depth == 0`. Guarantees a seed returns with enriched films even before the title cron runs.
- **Cron single-row tick pattern:** modeled on `process-person-queue` **#126** / `process-company-queue` **#127**. Lock-on-pick, `try_catch` around orchestrator, release lock on failure (the company/person equivalents leave rows permanently locked on crash — the IMDB crons fix this).
- **Missing-stub guard:** if the queue row's `imdb_person_id` / `film_tv_id` is 0, write `log_crash` and `db.del` the queue row — prevents a bad row from monopolizing the lock forever.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **shared_with** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person + company waterfall solves this for LinkedIn/PDL/Enrich Layer. This plan applies the same pattern — **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** — to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits).
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) + top 3 cast.

---

## Depth model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
|:-----:|---------|----------|
| **0** | Seed (user-initiated) | Full scrape + LLM extract + awards + cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` + name/title, no scrape). Queued. Cron processor scrapes + expands, but **only T1–T2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes + writes data, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits + 9 acting + 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director + writer + DP + lead producer + 3 cast = ~7 people → ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000+.
</Check>

### Credit filter for person seed (cascade-credits-from-person)

```text
for each credit in imdb_person.json.credits:
    if role ∈ {director, writer}:                                → T1
    elif role = producer AND (rating ≥ 7 OR year = current_year): → T2
    elif role ∈ {executive producer, cinematographer, minor}
         OR rating ≥ 7:                                          → T3
    elif type ∈ {Video Game, Music Video}
         OR (type = Short AND rating < 8):                       → T4
    else: skip
    # T3 capped at top 20 by rating to prevent low-signal blowout
```

### Key crew filter for title seed (cascade-crew-from-title)

```text
from film_tv.json:
    directors[]                                      → T1
    writers[]                                        → T1
    cinematographer (if present in json)             → T2
    lead producer (first producer in credits list)   → T2
    top 3 cast (from cast[] sorted by bill order)    → T3
```

<Info>
**"Top cast = top 3" is a starting point** — may tune upward (e.g., top 5 or top 10) once we see real fanout numbers. The tier assignment is what matters; the count is a knob.
</Info>

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

### Depth-1 queued processing

When the cron picks up a queue entry, it sets `processing: true` and calls the same orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P6) only queue T1–T2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured — but no children are queued.

## Loop prevention (five layers)

1. **Dedup gate in `get-add`:** `imdb_url` already in `imdb_person` / `film_tv` with `enriched_at` within 30 days → return existing, no scrape.
2. **Queue upsert:** duplicate `imdb_url` in queue → `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth ≥ 1:** only T1–T2 discoveries are queued at depth 2 (T3–T4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.

| #10515 | `add-title-imdb-restricted` | Same scrape path, no cascade — used as cascade terminator |
| #12633 | `add-award-records-from-person` | Scrape `{imdb_url}/awards/` → write `film_tv_award` rows, call full `add-title-imdb` per project |
| #12632 | `add-imdb-credits` | Walked `imdb_person.json.credits[]`, called full `add-title-imdb` per credit |

**The loop:**

```text
person seed (#10506)
  ├─ add-award-records-from-person (#12633) → add-title-imdb (#10507) per award project
  │                                            └─ writer/director cascade → #10506 → ♾️
  └─ add-imdb-credits (#12632) → add-title-imdb (#10507) per credit
                                   └─ writer/director cascade → #10506 → ♾️
```

Observed in an early sandbox (Neveldine seed, `imdb_person.id = 1`): 13 credits scraped, `awards_json: []`. Stopped cleanly only because awards were empty and the writer/director cascade was `!`-disabled.

The new waterfall replaces this with `get-add` gates, `cascade-*` helpers, and a depth cap — no more brittle `!`-prefix safety.
