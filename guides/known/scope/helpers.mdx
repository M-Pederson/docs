These are the foundation functions. They must exist before any ingest, scoring, or lifecycle code can run. Every write to the Known graph goes through one of these helpers — no direct Cypher `CREATE` from callers.

## Design principle

No caller writes raw Cypher against Known edges. All creates, updates, and invalidations flow through these helpers so that canonicalization, temporal rules, and derived-field recomputation are enforced in exactly one place.

## `known/helpers/canonicalize_pair`

<Note>
**Build status:** ✅ Built in Xano as function **#12767** on 2026-04-10. Inline tests pending (XanoScript test-block syntax needs verification).
</Note>

Takes two person UUIDs and returns them in canonical order (lower first, by lexicographic string comparison). Every write path uses this before any `CREATE` or `MERGE` against the graph.

**Inputs:** `person_a_id` (text), `person_b_id` (text)
**Outputs:** `{ lower_id: text, higher_id: text }`

**Guards:**
- **Self-pairs are rejected.** If `person_a_id == person_b_id`, the function throws `inputerror` with message `"Cannot canonicalize a pair of identical uuids"`. A self-pair would mean "a relationship between a person and themselves" which is meaningless for KNOWS and any facet, and silently accepting it would mask caller bugs. Upstream code (ingest paths, identity merge) MUST ensure it never calls this helper with the same uuid twice.
- **Empty strings are rejected.** Both inputs are required non-empty text. Xano's `filters=trim|min:1` handles this at the input layer.

**Implementation notes:**
- Comparison is lexicographic string comparison on the uuid values. UUIDs are strings so `<` works correctly.
- Pure function — no side effects, no database access, no graph writes.
- Idempotent and stateless — calling it twice with the same inputs returns the same output.

**Tests to verify:**
- `a_id < b_id` case: returns `{lower: a_id, higher: b_id}`
- `b_id < a_id` case: returns `{lower: b_id, higher: a_id}`
- Identical uuids: throws `inputerror`

## `known/ingest/create_knows_edge` and `known/ingest/append_evidence`

These two functions used to be specified on this page but were moved under `known/ingest/` during the Phase 1 build — **they are no longer helpers**. Their canonical spec (including the directed-MERGE pattern on canonicalized pairs, the ON CREATE / ON MATCH branching, and the uniqueness-enforcement discussion) lives at [scope → ingest → Core ingest primitives](/guides/known/scope/ingest#core-ingest-primitives-built-2026-04-10).

Why the move: both functions write to the `:KNOWS` edge type directly (via `mvp/falkor/send-cypher`). Treating them as part of the ingest surface — alongside `lookup_knows` and the per-source ingest functions — gave callers a single "where do I find the thing that touches KNOWS?" answer. The helpers namespace is reserved for pure-compute primitives and facet writers.

## `known/helpers/compute_confidence`

<Note>
**Build status:** ✅ Built in Xano as function **#12768** on 2026-04-10.
</Note>

Computes `confidence = min(1.0, sum(weights))` from an evidence array. Pure function — no graph access, no side effects.

**Inputs:** `evidence_rows` (json — array of evidence row objects)
**Outputs:** `{ confidence: float }`

**Evidence row shape** (each element of the array):
```json
{
  "source": "email_direct",
  "weight": 0.5,
  "first_seen_at": "2024-03-15T10:42:00Z",
  "evidence_occurred_at": "2023-11-20T00:00:00Z",
  "raw_metadata": {}
}
```

Only the `weight` field is used by this function. Other fields are preserved on the KNOWS edge but do not influence confidence.

**Edge cases:**
- **Empty array** → `confidence = 0.0`. Valid input (e.g. before any evidence has been ingested for a pending pair). Not an error.
- **Missing `weight` field on a row** → throws `inputerror` with message `"Evidence row missing required weight field"`. Well-formed evidence rows MUST have weight.
- **Negative weight** → throws `inputerror` with message `"Evidence weight must be non-negative"`. Strong sources are 0.5 and weak sources are 0.25; negative values indicate a caller bug.
- **Sum exceeds 1.0** → capped at 1.0 by `min`. Not an error. This is the whole point of the cap.
- **Non-array input** → Xano's `json` type validation should catch most cases; if it reaches the function the iteration fails gracefully with an error.

**Called by:**
- [`create_knows_edge`](#known-helpers-create-knows-edge) — initial confidence at creation
- [`append_evidence`](#known-helpers-append-evidence) — recomputes confidence after appending

**Inputs:** `evidence_rows[]`
**Outputs:** `confidence: float`

## `known/helpers/check_evidence_floor`

<Note>
**Build status:** ✅ Built in Xano as function **#12769** on 2026-04-10. Calls `compute_confidence` (#12768) internally.
</Note>

Evaluates an evidence array against the floor rule: `(strong_count ≥ 1) OR (weak_count ≥ 2)`. Pure function — no graph access, no side effects.

**Inputs:** `evidence_rows` (json — array of evidence row objects)
**Outputs:**
```json
{
  "passes_floor": true,
  "strong_count": 1,
  "weak_count": 1,
  "total_confidence": 0.75
}
```

**Strong / weak classification.** The function classifies each row by its `source` field against a hardcoded list that mirrors the canonical enum in [Schema Design → Evidence sources](/guides/known/schema-design/edges#evidence-sources). Rows with unknown source values are treated as weak (conservative default) and logged for investigation.

**Strong sources** (alone sufficient to clear the floor):
- `crm_contact`, `phone_contact`, `email_direct`, `calendar_meeting`, `manual_user_add`, `sms_chat`

**Weak sources** (require ≥2 to clear the floor):
- `linkedin_connection`, `twitter_follow`, `instagram_follow`, `shared_event_attendance`, `shared_employment_inferred`, `mutual_friend_mention`, `press_co_mention`

**Floor rule:** `passes_floor = (strong_count >= 1) OR (weak_count >= 2)`

**Edge cases:**
- **Empty array** → `{ passes_floor: false, strong_count: 0, weak_count: 0, total_confidence: 0 }`. Valid, not an error.
- **Row missing `source` field** → throws `inputerror`. Well-formed evidence rows MUST have a source identifier.
- **Unknown source value** → counted as weak with a debug log entry. This lets ingest pipelines add new source types without immediately breaking the floor check; the source definition list in this function should be updated in a follow-up PR.

**Called by:** the ingest pipelines (`known/ingest/*`) and [`create_knows_edge`](#known-helpers-create-knows-edge) to determine whether a new pair has enough evidence to enter the graph.

**Calls:** `compute_confidence` internally to populate the `total_confidence` field.

## `known/helpers/derive_known_since`

<Note>
**Build status:** ✅ Built in Xano as function **#12770** on 2026-04-10.
</Note>

Derives temporal fields from an evidence array. Key rule: temporal derivations use `evidence_occurred_at`, not `first_seen_at`. This is what enables retroactive backdating when LinkedIn tells us "they connected in 2019" even though Orbiter only saw the evidence in 2024.

**Inputs:** `evidence_rows` (json — array of evidence row objects)
**Outputs:**
```json
{
  "known_since": 2019,
  "first_evidence_at": "2019-05-22T00:00:00Z",
  "last_evidence_at": "2024-08-21T00:00:00Z",
  "first_direct_interaction_at": null,
  "last_direct_interaction_at": null
}
```

**Direct interaction sources** (used to populate `first_direct_interaction_at` / `last_direct_interaction_at`):
- `email_direct`, `calendar_meeting`, `sms_chat`

Other strong sources (`phone_contact`, `crm_contact`, `manual_user_add`) are acknowledgment sources — they prove a person knows of another but are not themselves direct communication events. They do not populate the `*_direct_interaction_at` fields.

**Field derivations:**
- `known_since` = year component of `min(evidence_occurred_at)` across all rows (integer year)
- `first_evidence_at` = `min(evidence_occurred_at)` across all rows (ISO timestamp)
- `last_evidence_at` = `max(evidence_occurred_at)` across all rows
- `first_direct_interaction_at` = `min(evidence_occurred_at)` across rows where `source` is in the direct interaction list; null if no direct interaction evidence exists
- `last_direct_interaction_at` = `max(evidence_occurred_at)` across rows where `source` is in the direct interaction list; null if none

**Edge cases:**
- **Empty array** → all fields null except `known_since` which is also null. Valid input, not an error.
- **Row missing `evidence_occurred_at`** → throws `inputerror`. Every evidence row MUST have this field.
- **Evidence with timestamps in the future** → not rejected; included as-is. If a bug sends future timestamps, they'll show up on the edge and can be cleaned up downstream.

**Called by:**
- [`create_knows_edge`](#known-helpers-create-knows-edge) — computes initial temporal fields
- [`append_evidence`](#known-helpers-append-evidence) — recomputes after new evidence is appended

**Pure function** — no graph access, no side effects.

## `known/helpers/create_facet`

<Note>
**Build status:** ✅ Built in Xano as function **#12771** on 2026-04-10. First graph-writing helper — sends cypher to FalkorDB via `mvp/falkor/send-cypher` (#2815). **v1.1 FalkorDB-compat fix published 2026-04-10 21:26 UTC** — the initial v1 emitted `datetime()` for both `valid_from` and `created_at`, which would have silently failed inside FalkorDB (same class of bug as the v2.1–v2.5 MENTOR saga). v1.1 resolves `valid_from` to a plain ISO 8601 string via an `api.lambda` (matching the Known convention that all `valid_from`/`valid_to` fields are plain strings) and uses `timestamp()` for `created_at`. See [cyphers/known FalkorDB compatibility](/guides/cyphers/known#falkordb-compatibility-read-this-before-writing-cypher).
</Note>

Generic facet creation. Enforces canonical pair order, sets `valid_from`, `source`, `created_by`, `created_at`, and any facet-specific `properties`. Creates the edge in FalkorDB and returns its uuid.

**Inputs:**
- `facet_type` (text) — one of `BONDED`, `PROFESSIONAL_TIE`, `KINSHIP`, `ROMANTIC`, `ESTRANGED`, `MENTOR`
- `person_a_id` (text) — uuid of the first person (will be canonicalized)
- `person_b_id` (text) — uuid of the second person (will be canonicalized)
- `source` (text) — `inferred` or `declared`
- `properties` (json) — facet-specific additional properties (e.g. `bond_type` for BONDED, `relation` + `role_a_to_b` + `role_b_to_a` for KINSHIP)
- `created_by` (text) — cron name, user id, or LLM identifier (e.g. `scoring_cron_v3`, `user:sarah`)
- `valid_from` (text) — ISO 8601 timestamp; defaults to `now()` if null

**Outputs:** `{ facet_uuid: text }`

**Guards:**
- `facet_type` must be one of the allowed values; throws `inputerror` otherwise
- `source` must be `inferred` or `declared`; throws `inputerror` otherwise
- Delegates self-pair check to `canonicalize_pair` (which throws `inputerror` for identical uuids)
- For `MENTOR` (directed), canonicalization is NOT applied — `person_a_id` is the mentor, `person_b_id` is the mentee, and the edge is created with that exact direction

**Canonicalization:**
- For all facet types EXCEPT `MENTOR`, the pair is canonicalized via `canonicalize_pair` so `lower_id` becomes the source of the edge
- For `MENTOR`, the direction is preserved as provided (mentor → mentee)

**Cypher (templated):** see [Cyphers → Known → `create_facet` generic template](/guides/cyphers/known#helper-create-facet-generic-cypher-template).

The `FACET_TYPE` placeholder is substituted with the validated `facet_type` input value. Properties are spread into the `CREATE` clause after the base fields.

**Does NOT:**
- Check for an existing valid facet of the same type on the same pair — that's the caller's responsibility (e.g. `bonded_creator` cron checks before calling). This helper is the low-level primitive.
- Set `weight` — that's done by `compute_weight` on the next scoring pass
- Fire any lifecycle cascades (e.g. does not invalidate a BONDED when creating an ESTRANGED)

**Called by:** `bonded_creator`, `estrangement_detector`, `professional_tie_maintainer`, user/LLM declared-facet creation paths

**Calls:** `canonicalize_pair`, `mvp/falkor/send-cypher`

## `known/helpers/invalidate_facet`

<Note>
**Build status:** ✅ Built in Xano as function **#12772** on 2026-04-10. **v1.1 FalkorDB-compat fix published 2026-04-10 21:27 UTC** — the initial v1 emitted `datetime()` or `datetime("...")` for `valid_to` via a conditional string-builder, which would have silently failed inside FalkorDB. v1.1 resolves `valid_to` to a plain ISO 8601 string via an `api.lambda`, matching the `create_facet` convention. See [cyphers/known FalkorDB compatibility](/guides/cyphers/known#falkordb-compatibility-read-this-before-writing-cypher).
</Note>

Sets `valid_to`, `invalidation_reason`, `invalidated_by` on an existing facet. **Never deletes.** Invalidation preserves history.

**Inputs:**
- `facet_uuid` (text) — the uuid of the facet edge to invalidate
- `facet_type` (text) — one of `BONDED`, `PROFESSIONAL_TIE`, `KINSHIP`, `ROMANTIC`, `ESTRANGED`, `MENTOR` (needed to construct the MATCH clause since FalkorDB edge-label lookups require the type)
- `invalidation_reason` (text) — one of the enum values from [Invalidation reasons](/guides/known/schema-design#invalidation-reasons)
- `invalidated_by` (text) — cron name, user id, or LLM identifier
- `valid_to` (text, optional) — ISO 8601 timestamp; defaults to `now()` if null

**Outputs:** `{ success: bool, facet_uuid: text }`

**Guards:**
- `facet_type` must be one of the allowed values
- `invalidation_reason` must be one of the enum values; throws `inputerror` otherwise
- If the facet is already invalidated (`valid_to IS NOT NULL`), the function is a no-op and returns `success: true` with a debug log

**Cypher (templated):** see [Cyphers → Known → `invalidate_facet` generic template](/guides/cyphers/known#helper-invalidate-facet-generic-cypher-template).

**Does NOT:**
- Delete the edge — invalidation preserves history
- Fire any cascades (e.g. does not create ESTRANGED when invalidating BONDED). Cascades are the caller's responsibility (typically the lifecycle crons).
- Check uniqueness — if two rows with the same uuid exist (shouldn't happen), both get updated

**Called by:** every lifecycle cron, identity merge facet resolution, user-declared invalidation paths

**Calls:** `mvp/falkor/send-cypher`
