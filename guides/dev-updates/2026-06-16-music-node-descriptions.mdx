## The problem

Two things were wrong with music node descriptions:

1. **~99% garbage.** `create-node-description` #13112 built the LLM input by dumping the **raw MusicBrainz JSON sliced to 8000 chars** ‚ÄĒ which cut big records mid-object, so deepseek replied *"Please provide the MusicBrainz record."* On the staging Cinq fixture **275 of 279** recording descriptions were that echo; only small records (less-famous duos) came out clean.
2. **Persons had none.** `run-base-artist-enrich` #13094 only called the finalizer for **groups** ‚ÄĒ `Person:Music_Artist` nodes were skipped (0/146 had a description) and kept their name+disambiguation vectors.

And because **the description is the embedding source** for recordings/groups, those ~275 recordings had their `name_embedding` + Gemini `embeddings` computed from the string *"Please provide the MusicBrainz record"* ‚ÄĒ near-identical, useless vectors.

## GOOD-or-null invariant

**A music node `description` is EITHER a GOOD description OR exactly `null` ‚ÄĒ never anything in between.** A GOOD description is one short factual sentence, **‚Č§127 characters, beginning with the entity name**, derived only from the provided source fields. It must **never** be LLM meta-commentary, a refusal, a stall echo, or a placeholder. Because the description is the **embedding source**, anything other than GOOD-or-null poisons the node's `name_embedding` + Gemini `embeddings`. (This rule extends to all node and edge descriptions across the graph.)

The canonical artifact this rule exists to kill ‚ÄĒ observed on a live recording node and **mapped to `null` going forward**:

> *"Wait, the user hasn't provided the MusicBrainz record yet. I must wait for the structured data before I can write the description. Please provide the structured MusicBrainz record for the entity."*

Enforced in **code, not the prompt** (the prompt asks for clean output, but LLMs don't reliably comply), at **two independent checkpoints**:

1. **Input-empty short-circuit** (#13112) ‚ÄĒ if the record is missing/not an object or the name is falsy, return `null` *before* the LLM is ever called on an empty stub.
2. **Output reject-guard** (#13112 **and** re-checked in #13117 before persistence) ‚ÄĒ strip wrapping quotes/backticks, hard-truncate >127, and map any text containing a refusal/meta-commentary marker to `null`. The marker list is first-person + meta-vocabulary that never appears in a clean third-person description: `provide the`, `please provide`, `i need the`/`i need a`, `musicbrainz record`, `structured musicbrainz`/`structured data`/`structured record`, `no record`, `the user has`, `hasn't provided`/`has not provided`/`have not provided`, `i must wait`, `wait for the structured`, `before i can write`/`before i can generate`, `as an ai`, `i cannot`/`i can't`, `i am unable`/`i'm unable`, `i do not have`/`i don't have`.

A node that fails validation is left `description = null` and **unstamped** (`description_generated_at IS NULL`) so the next backfill retries it. **Null always beats garbage.**

## Design notes

- **Strict ‚Č§127.** Enforced in the prompt *and* hard-truncated (LLMs don't reliably count characters). One short factual sentence, starting with the entity name.
- **Description ‚Üí vectors.** Recordings, groups, and persons all embed on their description (persons switched from name+disambiguation). A clean description means a meaningful vector.
- **Stub-proof.** Only `enrich_success` rows with a `node_uuid` are processed. The ~165 collaborator **stub** nodes (graph endpoints of credit edges, no MB record) are correctly skipped ‚ÄĒ they can't be described without full enrichment.
