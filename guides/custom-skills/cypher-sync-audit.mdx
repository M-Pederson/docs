## Triggers on

`cypher audit`, `cypher sync`, `cypher drift`, `audit cyphers`, `sync cyphers`, `check cyphers`, `are my cypher docs in sync`, or after deploying any Xano function that contains a `util.template_engine` block.

## Why this skill exists

Mintlify cypher docs drift fast. Every time a Xano function's `util.template_engine` block changes — switching `datetime()` to `timestamp()`, swapping undirected `MERGE` for directed, renaming an input variable — the corresponding doc snippet silently goes stale unless someone manually re-syncs it. The cost of letting it drift is high: future engineers (and Claude) read the docs as the source of truth and reproduce broken cyphers from them. This skill is the periodic reconciliation pass.

## Status legend in the docs

🟢 live (in production), 🟡 built_not_live (function exists but hasn't been triggered end-to-end), 🔵 planned (spec only), ⚪ illustrative (pedagogical only). **Only 🟢 and 🟡 are audited.**

## Guardrails

- Read-only — never edits cypher docs, only reports drift. Doc edits are explicitly out of scope and require a separate user approval.
- Never tries to "render" Twig templates — both sides are templates and should match literally after whitespace normalization.
- Skips 🔵 planned and ⚪ illustrative snippets — those are intentionally untethered to a Xano function.
- Surfaces undocumented Xano cyphers by default (firehose) so engineers see graph writes that aren't documented.
- Intermediate JSON and the final report live in `/tmp/cypher-audit-*.{json,md}` and are overwritten on every run — no history is kept.

## Resource files

- `scripts/extract_cyphers.py` — walks the docs dir, builds the heading-level stack, and emits the doc cypher inventory as JSON.
- `scripts/compare.py` — normalizes and diffs the doc inventory against the Xano inventory, emits the markdown report.

# Cypher Sync Audit — FalkorDB Cypher Documentation Auditor

## Purpose

Audit every FalkorDB Cypher snippet in the Mintlify cypher pages against the
actual Cypher embedded inside the Xano functions they reference, then produce
a markdown drift report. Read-only by design — the skill never edits docs, it
only describes what's out of sync so the user can decide what to fix manually.

## When to Use This Skill

Use this skill when:
- You want to audit whether documented Cypher snippets match what's actually in
  Xano workspace 3
- You suspect cypher docs have drifted from live functions
- You want to find undocumented graph writes happening in Xano
- You've changed a Cypher block in Xano and want to know if docs need updating
- "audit cyphers", "cypher sync", "cypher drift", or similar phrases

---

## Key Concepts

### Status Legend

Status emojis in cypher doc headings:
- 🟢 **live** — function exists in production and has been triggered end-to-end
- 🟡 **built_not_live** — function exists but hasn't been triggered end-to-end
- 🔵 **planned** — spec only, no function yet (audits skip these)
- ⚪ **illustrative** — pedagogical, no backing function (audits skip these)

Only 🟢 and 🟡 are audited. 🔵 and ⚪ are intentionally untethered to Xano.

### Cypher Blocks in Mintlify

Cyphers are documented in fenced code blocks:

\`\`\`mdx
## MENTOR ingest — mvp/yc/process-yc-people (#12700) 🟢

### Single founder branch

\`\`\`cypher
MERGE (org:Organization {crunchbase_id: $cb_id})
  ON CREATE SET org.created_at = timestamp()
RETURN org
\`\`\`
\`\`\`

The heading includes:
- The function name
- The function ID in parentheses: \`(#12700)\`
- The status emoji: \`🟢\`, \`🟡\`, \`🔵\`, or ⚪

Sub-headings inherit the function ID and status from their parent — \`###
Single founder branch\` correctly inherits \`function_id: 12700\` and \`status:
🟢\`.

### Cypher Blocks in Xano

Cypher snippets are embedded inside functions in several forms:

1. **Template Engine blocks** (most common):

\`\`\`xanoscript
util.template_engine {
  value = """
    MERGE (org:Organization {crunchbase_id: {{$input.cb_id}}})
      ON CREATE SET org.created_at = timestamp()
    RETURN org
  """
} as $query_single_founder
\`\`\`

2. **Direct Cypher variables** (used by callers of \`mvp/falkor/send-cypher\`):

\`\`\`xanoscript
var $query_single_founder {
  value = """
    MERGE (org:Organization {crunchbase_id: {{$input.cb_id}}})
      ON CREATE SET org.created_at = timestamp()
    RETURN org
  """
}
\`\`\`

The \`as $var_name\` or variable name allows multi-block functions (like MENTOR's
multi-founder vs single-founder branches) to be matched to the right doc
snippet.

## Execution Protocol

### Phase 1 — Extract documented cyphers from Mintlify

Runs \`scripts/extract_cyphers.py\` over \`guides/cyphers/*.mdx\`:

1. Walk every \`\`\`cypher\` fenced block in the docs.
2. Use a heading-level stack to inherit the function ID and status emoji from
   the nearest parent heading.
   - Example: \`### Single founder branch\` under \`## MENTOR ingest —
     mvp/yc/process-yc-people (#12700) 🟢\` correctly inherits \`function_id:
     12700\` and \`status: 🟢\`.
3. Extract the function ID using regex: \`\\(#(\\d+)\\)\`
4. Skip 🔵 planned and ⚪ illustrative entries automatically (no backing
   function).
5. Fall back to a 30-line scan above the fence if heading inheritance fails.
6. Output a JSON inventory keyed by \`(source_file, source_line, function_id,
   status, cypher)\`.

### Phase 2 — Fetch live cyphers from Xano workspace 3

For each unique \`function_id\` in the doc inventory:

1. Call \`mcp__xano__getFunction\` (workspace_id: 3) with \`include_xanoscript:
   true\`. **Batch in parallel groups of 5–8.**
2. As a fallback, try \`mcp__xano__getAPI\` if the function is an endpoint.
3. Extract every \`util.template_engine { value = """...""" } as $var\` block
   from the XanoScript.
4. Also extract \`var $cypher { value = """...""" }\` shapes (used by callers of
   \`mvp/falkor/send-cypher\`).
5. For multi-block functions, capture the \`as $var_name\` so blocks can be
   matched by name first, then by order of appearance as a fallback.
6. Flag functions that 404 as \`xano_missing\`; flag functions with zero cypher
   blocks as \`stale_no_cypher\`.

### Phase 3 — Diff doc vs Xano

Runs \`scripts/compare.py\`:

1. Normalize both sides:
   - Collapse whitespace (leading/trailing, multiple newlines)
   - Strip C-style and SQL-style comments
   - Preserve Twig substitutions like \`{{$input.foo|json_encode}}\` literally
     (both sides are templates and should match character-for-character)
2. Compare line-by-line.
3. For multi-block functions, match by \`as $var_name\` first, then by order of
   appearance within the doc section as a fallback.
4. Classify each snippet as **in-sync**, **drifted**, **undocumented**, or
   **stale**.

### Phase 4 — Report categories

The output is a markdown report with five sections:

- **✅ In sync** — count only.
- **⚠️ Drifted** — per-snippet \`diff\` blocks showing the doc line vs the Xano
  line, with file:line pointers.
- **🆕 Undocumented Xano cyphers** — cypher blocks that exist in Xano but no
  doc snippet references them. Intentional firehose so engineers know when
  graph writes are happening that aren't documented.
- **💀 Stale doc references** — doc snippets referencing a function that no
  longer exists in Xano workspace 3 or that no longer contains any
  \`util.template_engine\` block.
- **❓ Unlinked doc cyphers** — doc snippets where no function ID could be
  found nearby. The fix is either to add \`(#NNNNN)\` to the section header or
  to mark the snippet ⚪ illustrative.

### Phase 5 — Present and offer handoff

1. Reads \`/tmp/cypher-audit-report.md\` and shows it verbatim.
2. Asks: *"Want me to update any of the docs to match Xano?"*
3. Doc edits themselves are out of scope — the skill hands off back to the main
   conversation if the user wants to apply fixes.

## Rules

- **Read-only by design.** Never edit cypher docs, only report drift. Doc edits
  are explicitly out of scope and require separate user approval.
- **Never render Twig templates.** Both sides are templates and should match
  literally after whitespace normalization. If they don't, that's drift.
- **Skip 🔵 planned and ⚪ illustrative snippets.** Those are intentionally
  untethered to a Xano function.
- **Surface undocumented Xano cyphers by default** (firehose) so engineers see
  graph writes that aren't documented.
- **Heading inheritance is primary.** Use the heading-level stack to find
  function IDs and status emojis. Fall back to a 30-line scan above the fence
  if inheritance fails.
- **Preserve Twig in both-sides diffs.** Template variables like
  \`{{$input.foo|json_encode}}\` are literal and must match exactly.
- **Match multi-block functions by name, then order.** If a function has
  multiple \`util.template_engine\` or \`var $cypher\` blocks, match doc snippets
  by \`as $var_name\` first, then by order of appearance as a tiebreaker.

## Edge Cases

### Function has multiple cypher blocks

Example: MENTOR's \`mvp/yc/process-yc-people\` function has both:
- \`util.template_engine { value = "..." } as $query_single_founder\`
- \`util.template_engine { value = "..." } as $query_multi_founder\`

Doc sections for the same function can appear in different sub-headings:
- \`### Single founder branch\` — matches \`$query_single_founder\`
- \`### Multi founder branch\` — matches \`$query_multi_founder\`

Matching is by \`as $var_name\` first, then by order of appearance as a fallback.

### Xano function 404s

Flagged \`xano_missing\` in the report. The doc section is marked as stale.

### Xano function exists but has no cypher blocks

Flagged \`stale_no_cypher\`. The cypher was removed from the function; the doc
should be too.

### Doc snippet has no nearby function ID

If heading inheritance fails and a 30-line scan above the fence finds no
\`(#NNNNN)\`, the snippet is marked \`unlinked\`. Fix by:
1. Adding \`(#NNNNN)\` to the section header, or
2. Marking the snippet ⚪ illustrative if it's intentional.

### Cypher uses Twig, Xano uses raw values (or vice versa)

Example mismatch:
- Doc: \`{crunchbase_id: {{$input.cb_id}}}\`
- Xano: \`{crunchbase_id: $cb_id}\`

Both are valid; the docs are **drifted** because they don't match literally.
The user must decide which form is correct and update one side.

### Doc fence contains broken Cypher

The normalize step will still diff it. If it drifts from Xano, it shows in the
drifted section. Syntax correctness is not validated.
