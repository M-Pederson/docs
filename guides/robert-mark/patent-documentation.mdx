## What is Orbiter.io?

Orbiter is an AI-powered relationship intelligence platform. Given any goal a user is trying to accomplish — hire a key employee, raise a round, find a co-founder, source an acquisition target, get advice — Orbiter surfaces the people in the user's professional network best positioned to help, with grounded, context-rich explanations of **why** each person matters and the path and connections to that person.

The platform serves **organizations, enterprises, and Networked Operators** — dealmakers, investors, and executives in media, technology, and enterprise — for whom relationship leverage is the primary asset.

## Core Value Proposition

The **Core Value Proposition** of Orbiter.io is its ability to leverage the context of the user's entire network to provide context-rich **suggestions** that help the user achieve any **OUTCOME** or assist any person in their network (referred to as a **LEVERAGE LOOP**). Orbiter.io's **Core Value Proposition** is a **suggestion engine** that analyzes the user's entire network to deliver targeted suggestions for **OUTCOMES** or **LEVERAGE LOOPS**.

For example: a user may have 20,000+ people in their network — Orbiter surfaces the six most meaningful matches, with grounded, context-rich explanations of the **why** and **how** each person could help the user achieve the outcome.

Users connect their email and calendar accounts and import contacts and network data from any source into Orbiter.io. Orbiter reconciles each contact against the **ORBITER UNIVERSE GRAPH** — its shared canonical layer of previously enriched entities. Contacts that already exist in the **ORBITER UNIVERSE GRAPH** are copied into the **USER GRAPH** fully enriched. Contacts that don't are first added to the **ORBITER UNIVERSE GRAPH**, enriched there, and then synced into the **USER GRAPH**.

<Note>
The **ORBITER UNIVERSE GRAPH** contains only public, verified information. None of the user's private data — emails, phone numbers, or other private context tied to a contact — is stored in the **ORBITER UNIVERSE GRAPH**. Private data from the user is stored only in the **USER GRAPH**.
</Note>

<Note>
Orbiter.io also supports **ORGANIZATION GRAPHS** — a third graph type scoped to a company or organization, synced with both the **ORBITER UNIVERSE GRAPH** and the **USER GRAPHS** of every member of that organization. An administrator controls granular permissions over what information is shared within the organization's network.
</Note>

## Intelligence Layers

Orbiter.io also generates **INTELLIGENCE LAYERS** on top of each graph type:

- The **ORBITER UNIVERSE GRAPH** produces **GENERAL INTELLIGENCE**.
- Each **ORGANIZATION GRAPH** produces **ENTERPRISE INTELLIGENCE**.
- Each **USER GRAPH** produces **USER INTELLIGENCE**.

These layers are generated by cron-scheduled analytics jobs that pull from their respective graphs, providing statistical analysis and insights that can be combined with graph-based queries when synthesizing the actual suggestions presented to the user.

## What is the IP we want to protect?

Schemas and ontologies are not patentable on their own — they describe data organization, which sits outside §101 subject-matter eligibility. Waterfall-style cascading enrichment, taken in isolation, also faces a moderate-to-dense prior-art landscape: multi-stage enrichment pipelines, priority-tiered queues, and depth-capped cascades all appear in published systems.

What we want counsel to claim is the **end-to-end process** — the user's journey from seed entity through enrichment, graph population, intelligence-layer generation, interview-driven goal capture, hybrid retrieval, and the synthesized suggestions returned with grounded explanations. Novelty sits in the combination and the coupling between stages; the §101 footing comes from the specific technical improvements the integrated method produces (lower per-record cost, lower latency, higher accuracy, higher query quality). Per-layer mechanics are best positioned as dependent claims that pin narrower fallback positions if the integrated independent claim is contested.

We also want counsel to consider protection over the **UI and user experience** — the step-by-step method by which a user moves from a stated goal to a curated set of network-grounded suggestions, with the user's entire network as the search surface. The novelty sits in the guided flow itself: eliciting an **OUTCOME** or **LEVERAGE LOOP**, applying the full graph as context mid-flow, and returning synthesized matches with grounded explanations of the **why** and **how**. The strongest §101 footing will likely tie this interaction model back to the underlying technical improvements rather than framing it as a pure UI method.

Five distinct candidate-novel families surface across this document. Each is summarized below with an honest read for counsel on **why it may be worth pursuing** — the strength genuinely varies, and the recommended posture throughout is to concede the generic building blocks and claim the specific *integrated combinations* plus their dependent mechanics.

| Claim family | What it protects | Why it may be worth pursuing |

## Terminology: "Leverage Loops" vs. "Outcomes"

Two related but distinct concepts drive every suggestion Orbiter.io produces. They differ in *who* the user is trying to help and *what kind of entity* surfaces in the results.

| | **OUTCOME** | **LEVERAGE LOOP** |
|---|---|---|
| **Who is being helped** | The user themselves | Another person in the user's network |
| **What can be suggested** | Any entity type — people, companies, live events, funding rounds, etc. | People only |
| **Framing of the goal** | "Help me hire a key engineer." | "Help me help Sarah find investors." |

The same set of standard goal types — hiring key talent, finding investors, finding a co-founder, sourcing acquisition targets, getting advice, and so on — applies to both modes. In an **OUTCOME**, the goal is framed for the user; in a **LEVERAGE LOOP**, the same goal is framed for a target person in the user's network.

A **LEVERAGE LOOP** itself has two sub-modes, distinguished at the moment the user creates it:

- **Specific-help Leverage Loop.** The user defines what they want to help the target person with — e.g., "help Sarah find investors." This shares the **OUTCOME** backend pipeline; only the framing of the goal differs.
- **Generic Leverage Loop.** The user creates a leverage loop for the target person without a specific goal. The system analyzes the user's entire network — potentially thousands of contacts — and surfaces a concise set of the highest-impact introductions: people most likely to create mutual benefit with the target person.

All three paths — **OUTCOME**, Specific-help **LEVERAGE LOOP**, and Generic **LEVERAGE LOOP** — share the same underlying graph and intelligence layers. They diverge at the retrieval and synthesis stages.

## Under the Hood

Orbiter.io is a multi-tenant typed property-graph operating system for relationships.

### The Graphs

The **Orbiter Universe Graph**, **Organization Graphs**, and **User Graphs** are typed property graphs with eighteen-plus node types and thirty-plus edge types capturing people, companies, schools, funding rounds, awards, geographies, patents, publications, and more, and the relationships between them — **autonomously expanding** through a depth-capped, priority-tiered enrichment cascade. We refer to this as the **ORBITER ENRICHMENT ENGINE**.

### Orbiter Enrichment Engine

The enrichment pipeline produces three profile types — **Persona Profiles** for people, **Organization Profiles** for companies, and **Investment Thesis Profiles** for people and companies that are investors — created based on LLM processing of narratives derived from the context generated in the enrichment process. These profiles have multiple narratives stored as natural language and also **Gemini vector embeddings** for the text of each narrative.

A **backend orchestration layer** ingests email and calendar signals along with any other data source connected to keep the graph alive — relationships carry validity intervals, last-confirmed timestamps, and source evidence on every edge.

A **node-level vector index** over entity **Gemini embeddings** in the graphs supports semantic similarity search across every entity in the graphs. Users can target specific graphs to query.

**Per-field vector indexes** on richer entity properties — narrative descriptions, full-document embeddings, structured profile fields, and LLM-derived narratives — used for semantic filtering and scoring via approximate nearest-neighbor search with vector quantization.

#### Prior art: the cascade itself is not novel

We concede this plainly, and counsel should treat it as conceded in any background section. The engine's acquisition strategy is a **depth-capped, priority-tiered enrichment cascade** — for a given entity and a target field, query an ordered list of third-party providers, falling through to the next source on a missing record, a thin result, or a below-threshold confidence, with each phase skipped when its target field is already populated and the whole traversal gated by a `cascade_depth` cap that prevents unbounded discovery loops. This pattern — "**waterfall enrichment**" — was popularized and is widely credited to **Clay** (clay.com), which advertises 150+ cascaded sources ordered by cost and coverage, conditional per-column run criteria, and corroboration across sources. The broader landscape includes **Clearbit** (now HubSpot Breeze Intelligence), **ZoomInfo**, **People Data Labs**, and **Apollo**, among others.

Multi-stage enrichment pipelines, priority-tiered queues, depth-capped cascades, confidence-thresholded fall-through, and even **attribute-conditioned source selection** (running a provider only when an entity attribute such as industry or title is true) are all conceded prior art. The generic technique is the dominant, named, commoditized pattern of the present era, and nothing in the cascade *as a cascade* should be presented to an examiner as a point of novelty.

### The Expertise Engine — the candidate-novel element

The candidate-novel contribution sits one layer above the cascade. We call it **THE EXPERTISE ENGINE**: a self-learning expertise layer that derives a structured, multi-label semantic understanding of an entity — its **DOMAINS OF EXPERTISE** — from data *already acquired and resident in the graph*, and turns that understanding into the control signal that steers what the cascade does next. The engine is designed to serve **both sides of the graph from one shared, self-learning taxonomy**: it detects expertise for **people** (writing `HAS_EXPERTISE` edges) and for **companies** (writing `SPECIALIZES_IN` edges), resolving every detected expertise against the *same* persisted node catalog. Crucially, that catalog is not static — when a detected expertise has no matching node, the engine **match-or-creates**: it mints a new `SubDomainExpertise` node on the fly, so the graph keeps *learning* new sub-domains as it enriches. Top-level **DOMAINS** stay curated and human-reviewed — they are never auto-created — which is precisely what keeps the self-learning growth high-quality rather than noisy. The derived understanding then drives **domain-specialized enrichment** in a closed feedback loop: **derive semantic understanding → steer subsequent data acquisition → write new information back to the graph → re-evaluate.**

**The expertise taxonomy is real and live.** Expertise is not a label slapped on a node — it is a persisted, two-level property-graph taxonomy:

- **DomainExpertise** — top-level domains (e.g. *AI Infrastructure*, *Venture Capital & Private Equity*); a curated set of ~120 nodes.
- **SubDomainExpertise** — specific areas beneath a domain (e.g. *Generative AI & LLMs*); ~2,400 nodes, the large majority parented to a domain.
- **`HAS_EXPERTISE`** — `Person → SubDomainExpertise`, carrying `confidence`, `evidence`, `depth` (*creator / expert / practitioner / familiar*), `recency` (*current / recent / historical*), `signal_type`, `match_score`, and a graph-path `weight` (10–50, lower = stronger).
- **`SPECIALIZES_IN`** — `Company → SubDomainExpertise`, the company-side analog (built and sandbox-validated; **0 edges live** as of this writing, pending a backfill decision).
- **`HAS_SUBDOMAIN`** — `DomainExpertise → SubDomainExpertise`, the fixed-weight taxonomy parent-child link enabling a free 2-hop rollup: `(Person)-[:HAS_EXPERTISE]->(SubDomainExpertise)<-[:HAS_SUBDOMAIN]-(DomainExpertise)`.

**The detection that produces this taxonomy is live and continuously written on the person side — and it is self-learning.** For each entity, an extraction step reads node properties, one-hop edge descriptions, and the deep biography, and emits expertise signals; the resolver then runs a **MATCH-OR-CREATE** decision against the live expertise-node catalog. When an incoming expertise is close enough to an existing node, the engine **MATCHES** it; when nothing in the catalog is close enough, it **CREATES** a brand-new `SubDomainExpertise` node — with an LLM-generated definitional description — and attaches the entity to it. This match-or-create is the engine's learning mechanism: the graph keeps acquiring new sub-domains as new kinds of people and companies flow through it. The same engine logic runs for **people** (emitting `HAS_EXPERTISE`) and **companies** (emitting `SPECIALIZES_IN`) over the one shared catalog. By contrast, top-level **DOMAINS are CURATED** — the engine attaches each new sub-domain to an *existing* `DomainExpertise` only, and if a sub-domain fits no curated domain it is flagged for human review rather than forced into a wrong parent or auto-created. We treat this asymmetry as a quality feature, not a limitation: sub-domains grow freely and automatically, while the human-reviewed domain set keeps the rollup taxonomy clean. The growth is real — as of 2026-06-19, the catalog holds ~2,435 `SubDomainExpertise` nodes, ~2,265 of them engine-created (93%), parented to a curated set expanded from 119 to 123 domains by human review on 2026-06-19. One precision note for counsel: the *behavior* described here (match-or-create, curated parents) is what runs today, but the live resolver matches on the OpenAI `expertise_embedding` vector at a single 0.45 threshold; the symmetric `name + blurb` / Gemini `expert_embeddings` guard-band mechanics in the flow below are the **locked target**, sandbox-validated and not yet deployed. The full mechanics — node/edge schemas, the vector-match decision tree, the match-or-create guard band, and the in-flight embedding migration — are documented at [Expertise](/guides/cyphers/expertise).

Internally, **THE EXPERTISE ENGINE** runs the following ordered flow — identical for people (`HAS_EXPERTISE`) and companies (`SPECIALIZES_IN`), differing only in how the candidate expertise list is sourced — and mirrored box-for-box in the figure built from it. This is the **locked target flow** (Gemini `expert_embeddings`, symmetric `name + blurb`, guard band); it is sandbox-validated end-to-end. The live resolvers today implement the same match-or-create *behavior* but read the OpenAI `expertise_embedding` vector at a single 0.45 threshold (no blurb, no guard band, no adjudicator):

<Steps>
  <Step title="Read graph-resident signals">
    The engine reads what the cascade has already resolved for the entity. For a **person**, an LLM extraction step reads node properties, one-hop edge descriptions, and the deep biography. For a **company**, the candidate specialties are sourced from the resolved company-specialty data. No new external fetch occurs here — the engine works from data already resident in the graph.
  </Step>
  <Step title="Emit candidate expertise">
    The engine emits a structured, multi-label set of candidate expertise items. Each carries `evidence` and `signal_type`; person-side items also carry `depth` (*creator / expert / practitioner / familiar*), `recency` (*current / recent / historical*), and `confidence`.
  </Step>
  <Step title="Generate a definitional blurb (target flow)">
    For each candidate, the engine generates a short definitional blurb of the expertise *concept* (not of the entity) — so two phrasings of the same concept resolve to the same place. A fast path short-circuits this: an exact / normalized-name hit against the catalog skips the blurb and embedding entirely and goes straight to MATCH.
  </Step>
  <Step title="Embed the candidate (target flow)">
    The engine embeds the candidate as `name + blurb`, producing a query vector directly comparable to the catalog nodes (which embed `name + description`). This symmetric representation is what lets synonyms collapse together while sibling concepts stay apart. (The live resolver instead embeds the name only against the OpenAI `expertise_embedding` vector.)
  </Step>
  <Step title="KNN match against the catalog">
    The query vector is run through a KNN search against the dedicated expertise vector index, which structurally returns only `SubDomainExpertise` / `DomainExpertise` nodes. The nearest existing node and its cosine distance (`score`) are returned.
  </Step>
  <Step title="MATCH-OR-CREATE on a guard band (target flow)">
    The engine decides on a guard band rather than a hard cutoff: a very-near hit is an **auto-MATCH**; a clearly-distant result is an **auto-CREATE** of a new `SubDomainExpertise` node (with an LLM-generated description and its own vectors); a gray-zone result is handed to a cheap LLM adjudicator ("same concept? Y/N"). A parallel name-only signal acts as a tripwire — near-identical names that KNN alone might over-split are routed to adjudication. (The live resolver uses a single 0.45 cutoff with no adjudicator; the guard band and tripwire are sandbox-validated.)
  </Step>
  <Step title="Attach a curated parent domain — or flag for review">
    The matched-or-created `SubDomainExpertise` is attached to an **existing** curated `DomainExpertise` via a `HAS_SUBDOMAIN` edge, resolved by a single LLM call against the full curated domain list. If no curated domain fits, the sub-domain is flagged `parent_unresolved` for human review — never forced into a wrong parent, and **never auto-creating a domain**. This is what keeps the free 2-hop domain rollup clean.
  </Step>
  <Step title="Write the expertise edge with full lineage">
    The engine writes the edge — `(Person)-[r:HAS_EXPERTISE]->(SubDomainExpertise)` or `(Company)-[r:SPECIALIZES_IN]->(SubDomainExpertise)` — carrying `r.weight` (graph path-cost, lower = stronger), `r.match_score`, `r.match_type` (*matched / created*), an inline natural-language `r.description`, and the evidence lineage (`r.confidence`, `r.evidence`, `r.signal_type`, and on the person side `r.depth` / `r.recency`, rolling into `expertise_strength`). The new edge becomes graph-resident data the loop can read on the next pass.
  </Step>
</Steps>

### Orbiter Intelligence Engine

Scheduled cron tasks continuously generate and update statistical analyses across each graph type, producing the **General Intelligence Layer** from the **Orbiter Universe Graph**, an **Enterprise Intelligence Layer** for each **Organization Graph**, and a **User Intelligence Layer** for each **User Graph**.

In practice, the same observation carries different meanings at different scopes. A new connection between two professionals may be globally unremarkable, strategically novel for an organization, and one warm hop away for an individual user — three actionable readings of one event, each surfaced to the appropriate consumer.

The engine computes six families of graph statistics — centrality and influence, brokerage and pathfinding, community and cohesion, temporal dynamics, semantic and domain composition, and affinity and similarity. The same algorithms run at every scope; only the underlying subgraph changes (the full universe graph, an organization's scoped traversal, or a single user's named subgraph).

The three layers are not parallel computations over the same data — they form a hierarchical cascade in which each layer's statistical distributions become the **priors** against which the next layer's metrics are normalized. The General Intelligence Layer produces *universe priors*. The Enterprise Intelligence Layer recomputes the same metrics inside an organization's scoped subgraph and pairs them with universe priors to produce a **two-channel score** per node and per edge — an absolute value plus a normalized deviation from the broader baseline (implemented variously as a z-score, log-likelihood ratio, or over-/under-indexing ratio depending on the metric family). The User Intelligence Layer does the same against enterprise priors, or against universe priors directly when no organization scope applies.

Because the universe priors themselves drift over time as the graph expands, every lower layer is continuously re-normalized — the cascade pushes updated population statistics downward without manual reconfiguration, so previously persisted scores are re-interpreted against current priors on each cycle.

The result is that a single observed relationship event — for example, the creation of a new edge between two nodes — produces three distinct scored interpretations: one at universe scope, one inside the organization the user belongs to, and one inside the user's own named graph. The same event can register as statistically unremarkable at one layer and as a high-percentile outlier at another, and the system surfaces that tension as actionable signal rather than collapsing it.

#### What each layer surfaces

The six families run identically at every scope — only the prior they are normalized against changes, and that changes what each metric *means*. Read in isolation, each layer has a characteristic failure mode; the composition is where the value lives.

- **General Intelligence Layer** (*Orbiter Universe Graph*) answers *"how does the world actually look — what is rare, who is globally central, what is the base rate?"* Alone, it yields **celebrity rankings**: globally significant people with no path to them.
- **Enterprise Intelligence Layer** (*Organization Graph*) answers *"where does this organization stand, relative to the world and to itself?"* Alone, it yields **gap reports**: problems with no route to a fix.
- **User Intelligence Layer** (*User Graph*) answers *"what is my highest-leverage move — whom can I actually reach, and which ties are compounding versus decaying?"* Alone, it yields **recency lists**: people the user already talks to.

The same metric family, scored against three different priors, produces three different *kinds* of insight:

| Metric family | General (Universe Graph) | Enterprise (Organization Graph) | User (User Graph) |
|---|---|---|---|
| **Centrality & influence** | Globally most authoritative people in a domain (domain-weighted PageRank) — the field's gravity wells | The contact carrying the most relational weight against *your* accounts — often a junior, not a partner (hidden internal influence) | Your true power players — contacts whose own networks most amplify *your* reach, not whom you message most |
| **Brokerage & pathfinding** | Who bridges otherwise-disconnected industries worldwide — structural-hole spanners at global scale | The handful of people who are the *sole* bridge between your capital network and your deal flow — concentration risk | Your shortest warm path to a target, through whom, with prior successful-intro history |
| **Community & cohesion** | Universe-scale cohorts that act as fixed baselines (e.g., a specific accelerator-year founder cohort) | Relationships clustered into market segments — one segment serviced entirely by a departing employee | Your network split into separate circles with no overlap — you are the only bridge between them |
| **Temporal dynamics** | The global deployment-velocity baseline (a typical investor's edge-creation rate) | Org edge-velocity into a category up **+3.4σ** vs your own baseline — a thesis forming bottom-up before anyone named it | A key tie decayed three weeks past your typical recontact cadence — the "act now" signal |
| **Semantic & domain composition** | Rarity priors — e.g., cinematography expertise on ~0.8% of people; an Emmy for it on ~0.6% of cinematographers | You over-index on one vertical **4.3×** vs the universe while under-indexing on another — a structural blind spot | You personally under-index on the domain your stated outcomes require — a portfolio gap |
| **Affinity & similarity** | The population distribution of similarity that defines what "similar" even means | *Account-like-this* — ranked lookalikes of your best account that you have zero relationship with (white space) | *People-like-this* in your own network — contacts most similar to someone you just clicked with |

Composed, these collapse into a single directional recommendation: the **universe prior** strips celebrity bias (the target matters because it is strategically relevant to *this* org, not merely globally famous), the **enterprise prior** strips "we have no relationship" pessimism (the user's personal graph holds a one-hop path the org graph never saw), and the **user prior** strips the echo chamber (a strong tie is elevated because its *destination* carries broad significance, not because it is the user's favorite contact). Each layer corrects a failure mode the others would otherwise lock the user into — the composite is a directional vector none of the three could produce alone.

<Note>
**Worked example — one edge, three readings.** A new `collaborated_with` edge is created between Producer X and Cinematographer Y. The same event is scored against three different priors:

- **Universe → `+0.2σ`** — against the global prior for this edge type, an ordinary industry pairing; statistically unremarkable.
- **Enterprise (a studio's Organization Graph) → `+3.4σ`** — relative to the cinematographers the studio has previously reached through known producers, Y is a high-percentile outlier: new surface area into the organization.
- **User (the producer's ego graph) → `+2.8σ`** — the user holds a strong, recent tie to Producer X, so Y is now one warm hop away.

Decomposed, the readings diverge for different reasons: the enterprise score is driven by **composition** (`+4.1σ` — the studio has historically under-indexed on this cinematographer cohort), while the user score shifts to **brokerage and community** (the strong tie to Producer X makes Y a structurally valuable new neighbor). One event, three readings — none derivable from the others without the prior-propagation step.
</Note>

### Suggestion Generation

Both request types — **OUTCOMES** and **LEVERAGE LOOPS** — share a **multi-model LLM routing layer** that selects fast extraction models, reasoning-synthesis models, and adjudication models by task and cost, and both reduce a large network to a small, synthesizable set. They diverge in what serves as the *query* and how that set is reduced, as detailed below.

#### Leverage Loop (Generic)

A **Generic Leverage Loop** is the one-click move: given a **target person**, surface the six people from the user's *entire* network who would be the most valuable to introduce to that target. There is no goal to elicit — the target's own persona *is* the query.

<Steps>
  <Step title="Anchor on the target's persona">
    The target person's eight persona vectors become the anchor of the search.
  </Step>
  <Step title="Score every contact, per dimension">
    Each of the user's contacts is scored against the target across all eight dimensions independently — yielding, for every contact, a **match signature**: a vector of eight semantic-similarity scores rather than a single blended number. The per-dimension breakdown preserves *how* two people relate (shared domain, complementary track record, overlapping community, divergent communication style), which a single similarity score would collapse.
  </Step>
  <Step title="Cluster to a working candidate set">
    The engine **clusters** contacts on the combination of their per-dimension scores. Clustering over this multi-dimensional match-signature space groups the network by the *shape* of each contact's relationship to the target and isolates the highest-value combinations — collapsing a network of thousands down to a **working candidate set of roughly two dozen (~24)**. This clustering step, not a flat similarity ranking, is what makes reducing twenty thousand contacts to a focused set both tractable and non-arbitrary.
  </Step>
  <Step title="Synthesize the why">
    The ~24 candidates are passed to a reasoning-synthesis model with the **full persona context** of the target and each candidate. A system prompt optimized for relationship reasoning drafts *why* the user should introduce each pair and the plausible **win-win outcomes** for both sides.
  </Step>
  <Step title="Rerank and deliver the top six">
    A second **LLM reasoning pass reranks** the synthesized candidates — scoring each introduction's strength and mutual benefit — and the **top six** are returned to the user as ranked, grounded suggestions.
  </Step>
</Steps>

##### The persona substrate

Every person in the graph that is known to a user is projected into a **PERSONA** — a synthesized, multi-dimensional profile composed of exactly **eight orthogonal narrative + vector pairs**. Each dimension answers a distinct question from a distinct source signal: who they are now (`identity_profile`), what they know (`domain_focus`), what they can do operationally (`soft_skills`), what they have done (`track_record`), what they discuss publicly (`public_engagement`), how they communicate (`communication_style`), the communities they belong to (`community_affiliation`), and what they care about beyond work (`passions_interests`). Each dimension is stored twice — as a natural-language **narrative** and as the **Gemini vector embedding** (3,072-dimensional) *of that narrative text*. The narrative is the human-readable artifact; **its embedding is what every match runs on**. All comparison is per-dimension approximate-nearest-neighbor (KNN) cosine similarity *between the embedded narratives* — a query's `domain_focus` narrative-vector is scored against candidates' `domain_focus` narrative-vectors, and so on across all eight dimensions independently.

Personas are **generated**, not declared. A rollup synthesizer assembles the entity's full **enrichment context from the graph** — work history and employment edges, `HAS_EXPERTISE` rows, honors and awards, public and social signals, community affiliations — and an **LLM writes each of the eight narratives from that context**, one per dimension. Each narrative is then embedded into its dimension's vector. So the pipeline is: *graph-resident enrichment context → LLM-written narrative (per dimension) → embedding of that narrative → the vector KNN matches on.* The same eight-dimension schema describes a real person (a corpus row) and, on the search side, a *synthesized ideal* (the query) — so the two are directly comparable in the same vector space, dimension for dimension. Full schema and synthesis detail live at [Personas](/guides/open-work/personas).

**An illustrative person Persona.** For a representative operator (fictional — "Dana Okafor"), the eight generated narratives read roughly as follows. Each is also stored as its own 3,072-dimensional Gemini embedding:

| Dimension | Narrative — illustrative person Persona |
|---|---|
| `identity_profile` | Co-founder and CEO of a Series B developer-tools company; a staff engineer turned product leader, now focused on bringing AI-agent infrastructure to enterprise engineering teams. |
| `domain_focus` | Deep in developer tooling, distributed systems, and applied LLM-agent orchestration, with working fluency in enterprise security and platform reliability — known for turning frontier AI research into shippable developer products. |
| `soft_skills` | A hands-on technical founder: recruits and leads senior engineering teams, runs enterprise design partnerships, and closes complex multi-stakeholder deals; turns ambiguous problems into shipped roadmaps. |
| `track_record` | Scaled the company from zero to ~80 people and eight-figure ARR; earlier shipped a widely adopted open-source CI framework and was an early engineer at two acquired startups. |

#### Outcomes

An **OUTCOME** covers the user helping *themselves* — and also the **Specific-help Leverage Loop**, where the user is helping another person with a specific request, which is framed and routed as an outcome pursued on that person's behalf. Unlike the Generic Leverage Loop's single fixed query, outcomes are answered by an **orchestrating agent**.

<Steps>
  <Step title="Capture the goal">
    The **AI interviewer** chat produces two artifacts: an **outcome summary** (the desired result in natural language) and an **outcome-type categorization** (which standard goal it is — find investors, hire key talent, find a co-founder, source an acquisition, get advice, and so on).
  </Step>
  <Step title="Route by outcome type">
    An **orchestrating agent** consumes both artifacts and **routes to the tool workflow specific to that outcome type**.
  </Step>
  <Step title="Reduce the network to a target cluster">
    The selected workflow reduces the user's entire network to a **target cluster** using a **combination of graph traversal** — typed-edge paths and relationship structure — **and semantic scoring** against the persona substrate. The **exact reduction strategy is specific to the outcome type**: each outcome-type tool workflow defines its own mix of traversal patterns and scoring weights, so a *find-investors* run clusters the network differently than a *hire-key-talent* or *find-a-co-founder* run.
  </Step>
  <Step title="Synthesize candidates">
    A **reasoning model** processes the target cluster under a **system prompt selected by the outcome type and grounded in the outcome summary** (the request captured in step 1), generating candidate suggestions with grounded explanations.
  </Step>
  <Step title="Re-score and deliver">
    A second **adjudication pass** scores the candidates, and the **top-scoring** suggestions are delivered in the **format the user requested** — in-app Suggestions, a CSV, a PDF, or a Collection.
  </Step>
</Steps>

The two modes differ chiefly in control flow. A **Generic Leverage Loop** anchors on one real person's persona and reduces by clustering on match signatures. ## Non-Obviousness: Ablation Contrasts

Each contrast below pairs the naive/baseline approach an examiner would expect against Orbiter's design choice and what that choice buys — isolating the specific decisions that make the claim families non-obvious. (Some right-hand approaches are the locked target design; the contrast is a design-rationale argument, independent of current deployment state.)

| Baseline (obvious) approach | Orbiter's approach | What the contrast buys |
|---|---|---|
| A single blended similarity score, ranked | **Eight-dimension match signature + clustering** | A blended score collapses *how* two entities relate; the per-dimension signature preserves the shape of the relationship (shared domain vs. complementary track record vs. overlapping community), and clustering on that multi-dimensional shape is not reducible to a single similarity rank. |
| Return the top-K nearest neighbors | **Cluster to ~24, synthesize, then rerank to six** | Top-K returns a flat list with an arbitrary cutoff; clustering to a working set makes "six out of twenty thousand" tractable and non-arbitrary, and grounded LLM synthesis over that set produces explained introductions rather than a ranked list. |
| Name-only expertise embeddings | **Name + definitional-text embeddings (symmetric query/corpus)** | Name-only over-splits synonyms and conflates siblings; embedding `name + definitional blurb` symmetrically on both the query and the catalog node collapses synonyms while keeping sibling concepts apart — e.g., "generative AI" resolves to *Generative AI & LLMs* instead of minting a duplicate. |
| Naive graph RAG (dump a *k*-hop subgraph) | **Prior-weighted graph traversal** | A *k*-hop dump returns mostly noise at graph scale; injecting the intelligence-layer two-channel priors biases traversal and retrieval toward the edges that actually matter, so the model sees signal rather than volume. |
| Fan every query out to every data provider | **Expertise-conditioned source routing** | Fanning out is costly, slow, and pollutes the typed graph with low-precision cross-domain data; routing only to the source specialized for the *detected domain of expertise* cuts API calls, rate-limit pressure, latency, and cross-domain noise. |

Each row maps to a claim family above and to the §101 technical-improvement hooks in the **Notes for patent counsel** blocks: the right-hand column is the concrete improvement; the left-hand column is the obvious alternative the claim language should distinguish over.
