<Note>
**Status: planned work - design decision recorded, implementation not yet built.** Orbiter Universe remains the canonical owner of enrichment tables, joins, source payloads, provenance, confidence, and graph writes. Orbiter Backend stores compact read-model snapshots for product rendering and only breaks fields back out when the app needs to query, filter, sort, segment, or edit them directly.
</Note>

## Decision

As the enrichment engine moves into the standalone Orbiter Universe backend, do not duplicate every enrichment join table into Orbiter Backend. Instead, sync stable display snapshots back to the App backend:

| App table | Field | Purpose |
| --- | --- | --- |
| `master_person` | `profile_snapshot` | Person side-panel display facts: skills, expertise, roles, interests, languages, experience, and education. |
| `master_company` | `profile_snapshot` | Company side-panel display facts that are descriptive rather than financial. |
| `master_company` | `financial_profile` | Company capital, funding, investment, valuation, and financial-metric display facts. |

This is a projection/read-model contract. The source of truth stays in Universe; Orbiter Backend stores the latest product-safe snapshot for fast UI reads.

## Boundary

Universe owns the normalized model:

- source-specific enrichment rows and raw payloads
- join tables for skills, languages, roles, expertise, interests, work experience, education, funding rounds, investors, and investments
- provenance, confidence, source freshness, dedupe, graph node ids, and replay/audit state

Orbiter Backend owns the product read model:

- `master_person.profile_snapshot`
- `master_company.profile_snapshot`
- `master_company.financial_profile`
- hot columns needed for search, sort, filtering, permissions, or card rendering
- user/org/private product data that Universe must not overwrite

Use the existing [Projection Pattern](/guides/open-work/orbiter-univers-standalone/projection-pattern) for idempotent sync packages, versioning, and replay.

## Person `profile_snapshot`

`profile_snapshot` on `master_person` should be the person side-panel snapshot. Keep the top-level keys stable and return empty arrays rather than omitted keys.

```json
{
  "skills": [
    {
      "label": "Go-to-market strategy",
      "confidence": 0.92,
      "source": "orbiter_universe"
    }
  ],
  "expertise": [],
  "roles": [],
  "interests": [],
  "languages": [
    {
      "label": "English",
      "proficiency": "native"
    }
  ],
  "experience": [
    {
      "title": "VP of Sales",
      "company_name": "Acme",
      "universe_company_uuid": "uuid",
      "start_date": "2021-04",
      "end_date": null,
      "is_current": true
    }
  ],
  "education": [
    {
      "school_name": "University of Michigan",
      "universe_school_uuid": "uuid",
      "degree": "MBA",
      "field": "Business",
      "start_year": 2015,
      "end_year": 2017
    }
  ],
  "source_system": "orbiter_universe",
  "synced_at": "2026-07-09T00:00:00Z",
  "version": 1
}
```

Keep common summary fields as normal columns when the product needs them outside the side panel:

| Column | Why it stays outside JSON |
| --- | --- |
| `current_title` | Card/header display, search, and sorting. |
| `current_company_name` | Card/header display and quick filters. |
| `current_company_id` / Universe company reference | Joins, relationship paths, and company drill-down. |
| `current_location` | Filtering, search, and side-panel header display. |
| `latest_school_name` / `highest_degree` | Optional only if education becomes a first-class filter or card signal. |

## Company `profile_snapshot`

Use `profile_snapshot` on `master_company` for descriptive company facts that are not primarily capital-market data. Examples:

```json
{
  "industries": [],
  "expertise": [],
  "markets": [],
  "technologies": [],
  "locations": [],
  "summary": null,
  "source_system": "orbiter_universe",
  "synced_at": "2026-07-09T00:00:00Z",
  "version": 1
}
```

Keep high-traffic company fields as normal columns or display-table fields: name, domain, logo, headline, location, company type, employee count range, and any values used by search or card ranking.

## Company `financial_profile`

Use `financial_profile` on `master_company` for funding rounds, investments, investors, valuation, and financial metrics. The name is intentionally broad so funding can expand into revenue, ARR, valuation history, acquisition details, or other financial facts later.

```json
    "total_raised_usd": 125000000,
    "latest_round_type": "Series B",
    "latest_round_amount_usd": 50000000,

    "investor_count": 12
  },
  "funding_rounds": [
    {
      "universe_round_uuid": "uuid",
      "round_type": "Series B",
      "announced_on": "2025-09-15",
      "amount_usd": 50000000,

  "synced_at": "2026-07-09T00:00:00Z",

```

Keep query-critical financial fields as normal columns when the product needs them:

| Column | Why it stays outside JSON |
| --- | --- |
| `total_funding_usd` | Sort/filter/rank companies by funding. |
| `latest_round_type` | Segment and filter by company stage. |
| `latest_round_date` | Freshness, triggers, and timeline sorting. |
| `latest_round_amount_usd` | Ranking and card summaries. |
| `latest_valuation_usd` | Optional only if valuation becomes a product filter or rank signal. |

## Sync Rules

- Treat both JSON fields as replaceable snapshots generated by Universe, not hand-edited App state.
- Include `source_system`, `synced_at`, and `version` on every snapshot.
- Include Universe UUIDs inside array objects when a UI action may need to drill into the canonical entity.
- Prefer arrays of objects over arrays of strings so confidence, source, provenance labels, and future ids can be added without reshaping the field.
- Keep keys stable; use empty arrays or empty objects for missing sections.
- Add normal columns or indexed projection tables only for fields the App must query, filter, sort, permission, or edit independently.

## Open Questions

| Question | Default answer |
| --- | --- |
| Should `profile_snapshot` contain source/provenance per item? | Yes for anything displayed with confidence/source labels; otherwise keep source detail in Universe and expose only drill-down ids. |
| Should App users edit snapshot values directly? | No. Edits should become proposed corrections or App-owned overrides, then flow back through an approval/correction path. |
| Should search use JSON directly? | Only for lightweight reads. Real search/filter/segment paths should use normal columns, search indexes, or projection tables. |
| Should person work experience and education stay normalized in App? | Only if the App needs to query, edit, or permission individual entries. For display, keep them in `profile_snapshot`. |
