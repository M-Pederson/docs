<Tip>
  **Empirical finding that shapes this page:** the card detail endpoints have almost **no
  direct SQL reads** â€” `get-person-details_v2` / `get-company-details-v5` / `get-person-buttons`
  assemble cards from the **FalkorDB graph** (`falkor/send-cypher`) plus thin helper functions.
  So the App backend needs far fewer SQL tables than the enrichment side writes. Most person/company
  *facts* (honors, awards, publications, expertise, skills) reach a card **through the graph**, not
  by reading the source-of-truth SQL table directly.
</Tip>

## Tables ONLY in Orbiter Backend

Backend display tables are product read models, not Universe source tables. Universe sends projection packages;
Orbiter Backend owns these tables, upserts them from projection packages, and reads them for cards, search rows,
side-panel headers, and compact display sections. These are the display tables from the
[Projection Pattern](/guides/open-work/orbiter-univers-standalone/projection-pattern#combined-display-tables).
