> **Status: built and smoke-tested (May 2026).** Functions deployed in Xano workspace 3; orchestrator verified end-to-end on 3 known orgs. One open caveat: OpenRouter's `:online` suffix triggers a non-deterministic Xano `ERROR_FATAL` on certain response shapes — see [Open issue: `:online` parser failure](#open-issue-online-parser-failure) below. Workaround in place; lambda-based fix planned.

> Scope is intentionally minimal because the broader Fundable → AlloyDB refactor is on the near-term roadmap; the proper queue / trigger / backfill machinery will be designed against that target. This doc covers only what's needed in the short term to call enrichment on demand.

## Goal

Resolve a sparse `fundable_organizations` row to canonical identifiers (domain, LinkedIn, Crunchbase, PitchBook, etc.) using the LLM identity-resolution prompt designed in [Co Enrichment Optimization](/guides/open-work/co-enrichment-optimization) — exposed as a single callable function that any caller can invoke when it knows an org needs more identity data.

## Locked decisions

| # | Decision | Choice |
|---|---|---|
| 1 | Where do `confidence ≥ 0.85` results land? | **Direct-write** to `fundable_organizations.{domain,linkedin_url,…}` via per-field `db.edit` (literal field name, variable value). Audit columns deferred — audit lives in the append-only `fundable_organization_enrichment_attempts` log. Rollback via the log: `UPDATE … WHERE id IN (SELECT fundable_org_id FROM attempts WHERE model_id='…' AND outcome='resolved' AND created_at > '…')`. |
| 2 | Tier-2 (escalation) model | **Deferred** — bake-off `claude-sonnet-4.6:online` vs `gemini-2.5-pro:online` vs `gpt-5:online` later. Pipeline will accept the model id as a config var so it's swappable without code changes. |
| 3 | Human review surface for `0.6 ≤ conf < 0.85` | **Xano table editor** with a saved view filtered to `status = 'needs_review'`. Zero new code. Reviewer sees attempts log + queue audit, manually approves or edits. |
| 4 | Pilot scope | **Skipped.** Function exists; called on demand. Real validation happens against the AlloyDB rebuild. |
| 5 | Worker concurrency | **N/A.** No worker. Function called inline by whoever needs it. |
| 6 | OpenRouter API key location | **`$env.openRouter`** |
| 7 | Auto-trigger on new investments / new orgs | **No triggers.** All calls are explicit; no Xano table triggers in this iteration. |

## Architecture overview

```
Caller
  │
  ▼
enrichment/resolve_org_identity(fundable_org_id)
  │
  ├──▶ enrichment/build_org_enrichment_context  (✅ exists)
  │       returns: { fundable_data, deal_context, *_yaml, deal_count_total, ... }
  │
  ├──▶ enrichment/classify_signal_tier
  │       deterministic — no LLM call
  │       returns: "has_name_signal" | "portco_only" | "no_signal"
  │
  ├──▶ if signal_tier == "no_signal":
  │       log attempt as soft_fail, return { outcome: "soft_fail" }
  │
  ├──▶ enrichment/run_llm_resolution(model_id, prompt, payload)
  │       single OpenRouter call ($env.openRouter), :online model
  │       returns: { parsed_json, raw_response, tokens_in, tokens_out, latency_ms, cost_usd }
  │
  ├──▶ Validate JSON shape + extract confidence
  │
  ├──▶ Route by confidence:
  │       conf ≥ 0.85   → write resolved fields to fundable_organizations
  │                       + audit columns; return { outcome: "resolved", ... }
  │       0.6 ≤ <0.85   → return { outcome: "needs_review", ... }; do NOT write
  │       conf < 0.6    → return { outcome: "soft_fail", ... }
  │
  └──▶ append row to fundable_organization_enrichment_attempts (always, every path)
```

## Schema changes

### New table — `fundable_organization_enrichment_attempts` (append-only)

Every call to `resolve_org_identity` writes one row here, regardless of outcome. This is the audit trail and the surface humans review for `needs_review` cases.

| Field | Type | Notes |
|---|---|---|
| `id` | int | primary |
| `created_at` | timestamp | |
| `fundable_org_id` | uuid (FK) | |
| `signal_tier` | text | `has_name_signal` / `portco_only` / `no_signal` |
| `prompt_version` | text | e.g. `v0.2` |
| `model_id` | text | e.g. `google/gemini-3.1-flash-lite:online` (null for `no_signal` short-circuit) |
| `fundable_data_snapshot` | json | exact YAML/JSON sent to the model |
| `deal_context_snapshot` | json | exact YAML/JSON sent |
| `deal_count_at_call` | int | for re-run gating |
| `llm_response_raw` | text \| null | exact string the model returned (pre-parse) |
| `llm_response_parsed` | json \| null | null on parse failure |
| `confidence` | decimal \| null | |
| `outcome` | text | `resolved` / `needs_review` / `soft_fail` / `parse_error` / `api_error` |
| `latency_ms` | int \| null | |
| `tokens_in` / `tokens_out` | int \| null | |
| `cost_usd` | decimal \| null | computed at call time |

### Columns added to `fundable_organizations`

For `confidence ≥ 0.85` results, the existing fields (`domain`, `linkedin_url`, `crunchbase_url`, `pitchbook_url`, `website`, `country_code`, `region`, `twitter`, `facebook`) get **direct-write** alongside these new audit columns:

| Field | Type | Notes |
|---|---|---|
| `enrichment_resolved_at` | timestamp \| null | |
| `enrichment_resolved_by_model` | text \| null | which model produced the canonical answer |
| `enrichment_confidence` | decimal \| null | |
| `enrichment_canonical_name_alt` | text \| null | for legacy / consolidated entities (e.g. "Pfizer R&D Innovate" → canonical Pfizer Ventures) |

Reversibility: `UPDATE fundable_organizations SET domain=null, … WHERE enrichment_resolved_by_model='…' AND enrichment_resolved_at > '…'` rolls back any batch.

## Functions built

| Name | Function ID | Purpose |
|---|---|---|
| `enrichment/build_org_enrichment_context` | **12980** | Assemble YAML context for one org (existing — design + recipe in [Co Enrichment Optimization](/guides/open-work/co-enrichment-optimization)) |

| `enrichment/resolve_org_identity` | **12984** | Orchestrator (public entry point) — ties everything together, writes back, logs attempt |

Plus one auxiliary API endpoint (`enrichment/build_org_enrichment_context`, API ID 8517) wrapping the context-builder for direct invocation outside Xano.

### `classify_signal_tier` — deterministic

Keeps `no_signal` cases from ever calling an LLM.

```
let signals = []

# Signal 1: has descriptive short_description (not just the name repeated)
if fundable_data.short_description exists
   AND length(short_description) > 50
   AND short_description != fundable_data.name
  → signals += "row_description"

# Signal 2: an investor mention inside any deal long_description
for each deal in deal_context:
  if deal.long_description contains fundable_data.name (case-insensitive)
    → signals += "deal_named_investor"
    break

# Signal 3 / 4: "kind of entity" or "founder" cues
keywords = ["founded by", "its brands", "subsidiary of", "parent of",
            "venture arm of", "family office", "state-owned"]
if any keyword appears in row OR any deal description
  → signals += "entity_phrase"

# Signal 5: at least 2 portcos with names (fingerprint)
if deal_context length >= 2 AND each has a non-null portco
  → signals += "portco_fingerprint"

# Tiering
if signals contains any of (row_description, deal_named_investor, entity_phrase)
  → "has_name_signal"
elif signals contains "portco_fingerprint"
  → "portco_only"
else
  → "no_signal"
```

Edge: if a non-Latin name has no romanization in any description, Signal 2 will silently miss. Acceptable — those will fall to `portco_only` and pay for the stronger model.

### `run_llm_resolution` — OpenRouter via `api.lambda`

POST to `https://openrouter.ai/api/v1/chat/completions` wrapped in `api.lambda` (Node fetch) rather than `api.request`. **Why a lambda:** Xano's `api.request` JSON response parser intermittently throws an uncatchable `ERROR_FATAL: Error parsing JSON: Syntax error` on OpenRouter `:online` responses — see [Open issue](#open-issue-online-parser-failure). The lambda parses with `JSON.parse` (more forgiving), extracts only the curated fields, and returns scalars across the boundary.

| Input | Type | Notes |
|---|---|---|
| `model_id` | text | e.g. `google/gemini-3.1-flash-lite:online` |
| `system_prompt` | text | v0.2 prompt |
| `user_payload` | text | concatenation of `fundable_data_yaml` + `deal_context_yaml` |
| `temperature` | decimal | 0.0 |
| `timeout_sec` | int | default 180 |
| `max_tokens` | int | default 4000 |

| Output | Type |
|---|---|
| `parsed_json` | json (or null on parse failure) |
| `raw_response` | text (the model's content string) |
| `tokens_in` / `tokens_out` | int |
| `latency_ms` | int |
| `cost_usd` | decimal |
| `error` | text \| null — `fetch:…` / `openrouter_parse:…` / `http_<code>` |
| `http_succeeded` | bool |

Even with the lambda, the `:online` boundary is still flaky — see retry logic in the orchestrator below.

### `resolve_org_identity` — orchestrator (public entry point)

| Input | Type | Default |
|---|---|---|
| `fundable_org_id` | uuid | — |
| `tier1_model_id` | text | `google/gemini-3.1-flash-lite:online` |
| `tier2_model_id` | text | `google/gemini-3.1-flash-lite:online` |
| `prompt_version` | text | `v0.2` (logged with every attempt) |
| `max_llm_attempts` | int | `3` (retry on ERROR_FATAL — see open issue) |

| Output | Type |
|---|---|
| `outcome` | text — `resolved` / `needs_review` / `soft_fail` / `parse_error` / `api_error` |
| `signal_tier` | text |
| `signals` | array of signal-type strings (which of the 5 signals fired) |
| `confidence` | decimal \| null |
| `resolved_fields` | json — what we wrote back (or proposed, for `needs_review`) |
| `attempt_id` | int — FK to the attempts row |
| `model_used` | text |
| `llm_attempts` | int — how many LLM attempts the retry loop ran |
| `last_caught_error` | text \| null — the most recent ERROR_FATAL message, if any |
| `latency_ms` / `tokens_in` / `tokens_out` / `cost_usd` | metrics from the successful call |

Logic:

1. Call `build_org_enrichment_context(fundable_org_id)` → context.
2. Call `classify_signal_tier(context)` → tier.
3. If `tier == "no_signal"`: log attempt with `outcome = soft_fail`, return.
4. Else: pick `model_id` based on tier (tier1 for `has_name_signal`, tier2 for `portco_only`).
5. **Retry loop:** call `run_llm_resolution` inside `try_catch`. On `ERROR_FATAL`, log the caught error and retry. Up to `max_llm_attempts` tries.
6. If all retries failed: `outcome = api_error`.
7. If LLM call returned an `error` field: `outcome = api_error`.
8. If `parsed_json == null` (JSON parse failure): `outcome = parse_error`.
9. Route by confidence:
   - `≥ 0.85` → per-field `db.edit` writes to `fundable_organizations` for each non-empty field. `outcome = resolved`.
   - `0.6..0.85` → do NOT write. `outcome = needs_review`.
   - `< 0.6` → no write. `outcome = soft_fail`.
10. Always log attempt row (regardless of outcome).

## Confidence routing

| Confidence band | Action | Where data lives |
|---|---|---|
| `≥ 0.85` | **Auto-write** to `fundable_organizations` | Direct merge with audit columns |
| `0.6 – 0.85` | **Human review (Xano table view)** — no write | In attempts log |
| `< 0.6` | **Soft fail** — no write | Attempts log only |

## Human review surface

Reviewer opens the Xano table editor for `fundable_organization_enrichment_attempts`, applies a saved filter `outcome = 'needs_review'` joined with the source `fundable_organizations` row, and inspects:

- The `llm_response_parsed` JSON (LLM's proposed values + reasoning + alternatives)
- The `fundable_data_snapshot` and `deal_context_snapshot` (what the model saw)

If the proposal looks right, the reviewer manually edits the `fundable_organizations` row to set the resolved fields + audit columns. The attempts row is left in place as the audit trail.

If wrong: reviewer takes no action; the row stays as `needs_review`. (We can add an explicit reviewer-decision column later if needed.)

## Cost — per-call, not backfill

The pipeline runs on demand. Per-call cost depends on tier:

| Tier | Model | Per-call estimate |
|---|---|---|
| 1 | `google/gemini-3.1-flash-lite:online` | ~$0.001 base + ~$0.015 web search = **~$0.016** |
| 2 | (TBD escalation `:online`) | ~$0.075 base + ~$0.015 web = **~$0.090** |
| n/a | `no_signal` | $0 |

For a hypothetical 2,360-org one-off backfill at 50/35/15 mix: **~$93 total**. But that backfill is **deferred** (see below). Day-to-day cost is essentially zero — only when callers explicitly invoke the function.

## Open issue: `:online` parser failure

OpenRouter's `:online` suffix (which enables built-in web search) intermittently triggers `ERROR_FATAL: Error parsing JSON: Syntax error` on the lambda→xanoscript boundary. Repro:

- Same input, same prompt: sometimes succeeds, sometimes fails (~50% rate on certain orgs like Pfizer R&D Innovate).
- The raw OpenRouter response (status 200, parseable by Node's `JSON.parse`) returns successfully — verified by debug API that returns `$api_response` directly.
- The failure is at Xano's response/lambda parser, not network or OpenRouter.
- The error is **catchable** by `try_catch` wrapping `function.run` (verified — see retry loop in orchestrator).

### Mitigation in place

1. `run_llm_resolution` uses `api.lambda` instead of `api.request`. The lambda parses with Node's `JSON.parse`, extracts only curated scalar fields (`raw_content`, `tokens_in`, `tokens_out`, `cost_usd`, etc.), and returns no nested objects across the boundary.
2. `resolve_org_identity` wraps `function.run "enrichment/run_llm_resolution"` in `try_catch` and retries up to `max_llm_attempts` times (default 3, configurable). With a per-attempt failure rate of ~50%, 3 retries gives ~87% reliability; 5 gives ~97%.
3. The `llm_attempts` field in the response tells callers how many retries it took. Spike in retries → file a follow-up.

### Lessons learned (XanoScript gotchas)

- `db.edit { data = $var }` rejects variable assignments — `data` requires a literal object. Workaround: per-field `db.edit` calls with literal field name + variable value.
- `db.direct_query { sql = $var }` rejects variable SQL — must be a literal string.
- `db.direct_query { arg = [a, b] }` doesn't accept arrays for multi-`?` placeholders — only single value.
- Direct path access (`$x.y.z[0]`) throws on missing fields; `|get:"y.z.0":default` is the safer pattern.
- `|get` deep paths support array indexing with `.0.`, `.1.`, etc.
- `ERROR_FATAL: Error parsing JSON: Syntax error` from `api.request` or `api.lambda` IS catchable by `try_catch` around `function.run` — retry works.

## How to call it

```bash
# Via the build_org_enrichment_context API + a function.run from another stack,
# or directly from any XanoScript:

function.run "enrichment/resolve_org_identity" {
  input = {
    fundable_org_id: "76af7202-b26d-41c5-81c4-862e874cd047"
  }
} as $result

// $result.outcome           → "resolved" | "needs_review" | "soft_fail" | "api_error" | "parse_error"
// $result.confidence        → 0..1 (or null)
// $result.resolved_fields   → what was (or would have been) written
// $result.attempt_id        → FK to the audit row
// $result.llm_attempts      → how many retries it took
```

For an `api_error` outcome with `llm_attempts == max_llm_attempts`, the caller can increase `max_llm_attempts` and re-run, or fall back to a non-`:online` model via `tier1_model_id` override.
