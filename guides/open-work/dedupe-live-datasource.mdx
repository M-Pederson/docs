### Can we add a unique index on `company_domain`?

**Not as-is.** No real duplicates block it, but two structural issues do:

1. **492 rows with empty-string `company_domain`.** Postgres unique indexes treat `""` as a value (not NULL), so 492 rows colliding on `""` would fail the constraint.
2. **Default is `""` and column is nullable.** Three viable fixes:
    - Update the 492 empty-string rows to `NULL`, drop the `""` default, then add a plain `UNIQUE` btree on `company_domain`. Postgres allows multiple NULLs in a unique index, so the empties stay valid.
    - Add a **partial unique index**: `UNIQUE (company_domain) WHERE company_domain <> '' AND company_domain IS NOT NULL`. Lets the empties stay as `""`.
    - Add a **functional unique index on `lower(company_domain)`** if we want case-insensitivity enforced going forward (today it doesn't matter â€” no case-variant dupes exist).

**Recommendation:** clean the 492 empty-strings â†’ `NULL`, then add a regular unique btree on `company_domain`. The data is already clean enough to support it immediately after that one-step normalization.

### Can we add a unique index on `linkedin_url`?

**Not as-is.** Three blockers:

1. **14 rows in 7 canonical-duplicate groups.** They pass the literal unique constraint today (the strings differ by `www.`) but they're the same person â€” adding a unique index now locks in the duplication. Fix the 7 groups first.
2. **609 `NULL` rows.** These are fine for a unique index (Postgres allows multiple NULLs), but only if the column actually stays `NULL` â€” the schema default is `""`, so any new row inserted without `linkedin_url` set will collide on `""` immediately.
3. **URL shape drift.** 4,191 rows use `https://www.linkedin.com/...`, 609 use `https://linkedin.com/...`, 2 use `https://uk.linkedin.com/...`, 1 uses `http://linkedin.com/...`, 1 uses `http://uk.linkedin.com/...`. The canonical scan above only matched the first two; the country-subdomain rows can hide more dupes.

**Recommendation:**
1. Merge the 7 canonical-duplicate groups (keep one row, redirect any FKs from the other, delete the loser).
2. Merge the 8 `mixed_null_and_url` groups.
3. Triage the 26 `different_urls` groups manually â€” these need a human (or a fuzzy/LLM match) to call true-dupe vs. homonym.
4. Normalize the surviving URLs to one canonical form (`https://www.linkedin.com/in/{slug}`, lowercased, no query, no trailing slash, no country subdomain).
5. Change the default from `""` to `NULL` so newly-inserted blank rows don't all collide.
6. Add a **functional unique btree on `lower(linkedin_url)`** (or a partial `UNIQUE (linkedin_url) WHERE linkedin_url IS NOT NULL`) so the normalization holds.
