## Orbiter Calendar — Architecture

The calendar is a **first-class UI lens over data Orbiter already syncs**, not a new integration. The architecture has three clean layers and one direction of authority. **Nylas v3 is the system of record and the sync bridge**: Orbiter talks to a user's real Google / Outlook / iCloud calendars *only* through Nylas, by `grant_id`, and Nylas pushes every change back through webhooks. **FullCalendar on TanStack Start + React is the render layer**: a stateless view surface initialized with plugins, each supplying views (month grid, time grid, agenda list, year grid, resource timeline). **Xano holds the queryable mirror**: the per-user Nylas grant, a fast-to-filter cache of events, the links from events to Orbiter graph entities (deals, tasks, people, companies), and the webhook receiver that keeps the mirror in near-real-time sync. We render and filter against the Xano mirror so FalkorDB joins and FullCalendar paint stay fast; the moment a write happens, it flows **out** through Nylas and **reconciles back** through the webhook — Nylas always wins.

The state model is intentionally asymmetric. The UI may hold an optimistic placeholder, Xano may hold a stale mirror row for a few seconds, and Nylas may accept or reject a write; only the Nylas-confirmed event is durable. That gives every divergence a simple convergence rule:

| Divergence | Allowed temporary state | Convergence rule |
| ---------- | ----------------------- | ---------------- |
| UI write before webhook | Pending block keyed to the proposed write / `external_id` | Replace with the webhook-confirmed mirror row, or roll back on Nylas failure / reconcile timeout. |
| Duplicate or out-of-order webhook | Multiple deliveries for the same provider event | Dedupe by delivery id, then last-writer-wins by provider `updated_at`/sequence, never by arrival order. |
| Missed webhook | Mirror lags provider | Bounded delta poll closes the gap using the same idempotent upsert path. |
| Provider delete / disconnect | Mirror still has cached content | Set `deleted_at`, hide from reads, then hard-purge within the retention window. |
| Recurrence exception | Client RRULE view disagrees with provider instances | Prefer Nylas-expanded instances; client RRULE is render-only fallback. |

The product philosophy is **"one entity, many lenses."** FullCalendar's killer feature is **multiple toggleable Event Sources** layered on a single canvas. Orbiter renders three color-coded layers at once: **Nylas-synced events** (the real calendar, mirrored in `event` 221), **`df_task` due/start dates** (work due, from the Deal Flow design), and **`df_deal` expected-close dates** (forecast milestones). They are not three calendars — they are three lenses on the same week, each a switchable layer in the same grid. Color is meaning, never decoration: a synced meeting, a task due, and a deal closing read as distinct, learn-once colors — and the layer palette is bound to the account theme (see [Theme-aware rendering](#theme-aware-rendering)), so the same meaning holds in light and dark.

The second philosophy is **AI-first, human-in-the-loop**: the calendar assumes the operator runs many relationships at once, so the AI carries the load and the human keeps the wheel. The Copilot proposes meeting times (Nylas Availability), drafts the invite, binds it to the right deal, and ranks the day's meetings into a prep-priority budget — but **nothing irreversible happens silently**. The AI is a *proposer*, never an actor: every scheduling action it takes is a reversible, attributable proposal the operator confirms, routed through the **same** write verb a human button-click uses. There is no separate "agent API" that can bypass the human; there is one autonomy-gated path, and the AI is just another caller of it.

Writes are **single-contract and reversible**. A "Book meeting" button in the UI, a drag-create on the grid, an external chip dropped onto a date, and an agent's scheduling action all resolve to the **same** `df_action_def` verb `schedule_meeting` (`source_engine='calendar'`) — there is no UI-vs-agent drift, and the AI proposes while the operator decides. Every event write goes to Nylas via `grant_id` (CRUD + RSVP on the Events API), then the resulting `event.created` / `event.updated` / `event.deleted` webhook reconciles the mirror. **The frontend never mutates `event` (221) directly** — there is exactly one write path, and the mirror is downstream of it. The append-only product timeline is **`df_event`** (record history on a date axis); the user-facing notification feed stays **`activity` (218)**; calendar nudges fire through **`df_reminder`** into `activity` (218), never directly. Everything is **`organization_id`-scoped** and access-gated through **`resolve_access()` / `df_grant`** — Xano has no DB-level RLS, so authorization is explicit on every read, write, and realtime broadcast.

Crucially, the calendar surface is **both a product and a context feed**. Email + calendar activity is an explicit input to the Deal Flow momentum engine (`df_deal.momentum_last_activity_at`, rotting, `df_health_score`), and a meeting bound to a deal resolves its attendees to `master_company` (142) / `master_person` (139) graph nodes. So the same synced event that paints a block on the week grid also moves a deal's momentum, ranks itself into the AI prep budget, and feeds [Meeting Prep](/guides/modes/meeting-prep).

## Operator job & core loop

The operator is not hiring this calendar to admire a grid. They are hiring it to answer three high-frequency, high-stakes questions without leaving Orbiter:

| Job | Frequency / stakes | Calendar answer |
| --- | ------------------ | --------------- |
| **What needs my attention next?** | Daily, often many times per day; missed meetings and stale follow-ups cost relationships. | A docked agenda and AI-ranked prep budget surface the next meeting, conflicts, and the few conversations worth preparing for. |
| **Can I safely schedule this without breaking trust?** | Every meeting creation/reschedule; wrong time, wrong attendee, or privacy leak is expensive. | Availability, timezone-correct NL entry, conflict repair, and the privacy ladder make the proposed slot inspectable before commit. |
| **How does this meeting move the work forward?** | Every deal/project touch; a meeting that is not tied to a next move becomes inert calendar noise. | `cal_event_link` binds the meeting to a deal/person/company/task so Meeting Prep, momentum, reminders, and the graph all read the same context. |

The core loop is therefore ruthless: **see the next commitment → understand why it matters → confirm or repair the next action → reconcile back to Nylas → feed the graph.** Month/week/year/team views, drag-and-drop, NL date entry, Scheduler pages, and AI proposals earn their place only when they shorten or de-risk that loop. Anything that is merely "a nice calendar feature" is deferred behind the loop: Phase 0 proves the surface, Phase 1 makes real commitments visible, Phase 2 makes the loop writable and graph-linked, and Phase 3 adds team scheduling only after privacy and org scope are enforceable.

## Non-negotiable invariants

The calendar is allowed to be fast, optimistic, and AI-assisted only because a small set of invariants stays boring. These are product requirements, not implementation preferences:

| Invariant | Enforcement point | Why it cannot move |
| --------- | ----------------- | ------------------ |
| Privacy ladder applies before serialization | Server projection via `redact_to_rung`, after `resolve_access()` / `df_grant` | Busy-only teammate data must not leak titles, attendees, locations, descriptions, URLs, or raw `event_data` over the wire. |
| Client state is never authorization | Every read, write, and realtime broadcast resolves access in Xano | Xano has no DB-level RLS, so org membership and grants must be checked explicitly on every calendar path. |
| No direct client writes to `event` (221) | `schedule_meeting` → Nylas → webhook → mirror | The mirror is downstream cache, not user-authored truth; bypassing Nylas creates unreconcilable provider drift. |
| Expired or revoked grants stop participation immediately | Write gate, read projection, purge sweep | A disconnected account must disappear or redact before cached detail is physically purged. |
| Time is UTC plus IANA timezone, never wall-clock alone | Parser, write contract, mirror read, FullCalendar adapter | DST and cross-zone rendering only stay correct when the original named zone survives storage. |
| All-day events are floating | Parser and render adapter | A date-only birthday or all-day block must not become midnight UTC and slide across days for another viewer. |
| AI proposes only through the same contract | Copilot proposal card and `schedule_meeting` | There is no hidden agent write lane; irreversible changes need confirmation or an explicit autonomy gate. |
| Recurrence authority stays with the provider | Nylas-expanded instances preferred over client RRULE | Overrides, cancellations, and moved instances are provider facts; client expansion is a render-only fallback. |

## Phase gates

The build sequence is valuable only if each phase buys down the next phase's risk. The gate for leaving a phase is therefore evidence, not feature count:

| Phase | Proves | Do not spend / build yet |
| ----- | ------ | ------------------------ |
| P0 | The calendar surface is useful over existing dated Orbiter work with no Nylas dependency and no writes. | OAuth, provider sync, write-back, recurrence editing, team rows, premium license. |
| P1 | Real provider events can be connected, indexed, mirrored, redacted for owner settings, and repaired after webhook disorder. | Drag writes, cross-teammate sharing, Scheduler pages, AI booking. |
| P2 | Every human or AI write uses one reversible contract and reconciles through Nylas without corrupting the mirror. | Premium timeline, Availability fan-out across teammates, public booking pages. |
| P3 | Team availability is safe, licensed, privacy-bounded, and economically worth the Nylas / FullCalendar spend. | Any broader automation that bypasses confirmation or creates a second write path. |

This keeps the free tier durable: P0–P2 remain a complete personal calendar and meeting-prep surface even if the organization never buys the premium resource timeline.

## Design principles

Seven principles govern the AI-first surface — each something to design *toward*, validated by current best-in-class products and recent research (full list in [Prior art & references](#prior-art-&-references) below). The throughline: **the AI is a proposer; the operator keeps the wheel.**

| # | Principle | In Orbiter | Validated by |
| - | --------- | ---------- | ------------ |
| 1 | **Natural-language & event extraction** | events/dates created from a typed phrase, an email, or a chat line — not only grid clicks; the [NL date picker](/guides/open-work/calendar/nl-date-picker) is the shared input | Gmail "Add events with Gemini"; ScheduleMe |
| 2 | **Suggested slots with transparent rationale** | a proposed time shows *why* — availability, working hours, time zones, conflicts — never an opaque pick | Gmail "Help me schedule"; Gemini in Calendar; Carbon explainability |
| 3 | **Preview-before-commit** | every AI action is an editable preview the operator confirms, through the single `schedule_meeting` verb | Morgen propose→tune→approve; Gmail side-panel confirm |
| 4 | **One-click conflict repair** | conflicts/declines surface an actionable fix on the same reversible write path, not just a warning | Gemini reschedule banner; Motion auto-reschedule |
| 5 | **Preference memory** | the system learns this operator's windows, buffers, and priorities and applies them consistently | PEARL (memory is *required* — LLMs err without it); Reclaim |
| 6 | **Visible confidence & error states** | low confidence prompts confirm/disambiguate instead of a silent guess; the AI fails legibly | Carbon AI-vs-manual variant; the picker's confidence chip |
| 7 | **AI recs are editable objects, not hidden actions** | every AI output is an inspectable, editable, undoable object on a calendar-first canvas — never a magic background mutation | the throughline; the Guardian/Reclaim caution on goal clarity |

## Implementation contracts

Decisions to preserve when this plan turns into Xano tables / endpoints / FullCalendar config:

| Contract | Decision |
| -------- | -------- |
| System of record | **Nylas is authoritative** for connected-calendar data. Orbiter never writes a provider calendar directly — all CRUD/RSVP goes to Nylas by `grant_id`; the Xano mirror is a cache, reconciled by webhook. |
| Single write contract | `schedule_meeting` is **one** `df_action_def` verb (`source_engine='calendar'`). The "book meeting" button, a grid drag-create, an external-chip drop, and the scheduling agent **all** invoke the identical autonomy-gated, reversible path → Nylas Events API by `grant_id` → `event.*` webhook → mirror upsert. No UI-vs-agent drift; **no direct `event` (221) mutation from the client.** |
| AI is a proposer, never an actor | The AI never writes the calendar on its own authority. Every AI scheduling/reschedule/bind action lands as a **reversible, attributable, autonomy-gated proposal** routed through the *same* `schedule_meeting` verb a human click uses, with **no silent irreversible action**: the operator confirms (or auto-confirm is an explicit per-account autonomy setting). Attribution (`source_engine`, actor, `df_suggestion` lineage) records that the AI proposed and the human confirmed, and the webhook-reconcile path makes the write undoable. When proposing across teammates, the AI sees **only** the Free/Busy each teammate's privacy ladder has shared — never raw detail it is not entitled to. |
| Copilot is the AI surface | The conversational AI is the existing **Copilot** — CrayonChat v2 in `src/features/copilot`, backed by the **Robert API group `Bd_dCiOz` (id 1261)** (Thesys C1 wrapping Claude). The Copilot reads calendar data through the **same `useCalendarEvents()`** (V2 group `LR2ywW7R`), not the Robert group, and supports voice (Gradium/ElevenLabs TTS, Web-Speech/Gradium STT). Calendar context handed to the AI inherits the same `resolve_access()` gates and the privacy ladder — it must never carry content the operator could not see in the UI. |

The UI shows a pending state until the matching webhook lands (or a short reconcile poll confirms). |
| Reuse before create | A full Nylas-synced model **already exists** — store/read events in `event` (221), calendars in `calendar` (223), recurrence in `master_event` (222), and the Nylas grant in `nylas_grant` (183). Do **not** introduce a parallel events table or a second data hook. New `cal_*` tables are only the small orthogonal gaps (graph linking, webhook idempotency, render preferences). |

The three per-account calendar toggles (`enable_calendar`, `share_conflicts`, `share_conflict_details`) are **new bool columns on `nylas_grant` (183)** alongside its existing precedent bool toggles (`ai_enabled`, `show_in_email_widget`, `signature_enabled`, `primary_for_sending`), read via `first_notnull`. Do not conflate with the APIANT meeting-intelligence stack (`integration_connection` 717 holds an `apiant_connection_id`, not a Nylas grant). |
| Privacy ladder | **Strict, monotonic:** `enable_calendar` (default ON) gates everything; `share_conflicts` (default ON) exposes Free/Busy to teammates; `share_conflict_details` (default OFF) reveals titles/attendees instead of an opaque "busy" block. | Week start | `firstDay` is **one user-level preference** on `user` (125) settings (`settings.week_start ∈ {0,1}`), mapped straight onto FullCalendar `firstDay` (0 = Sun, 1 = Mon). Not per-calendar, not per-grant. |
| Active-view registration | The calendar is a **new `active-view`** plus a **new file-based route** — two distinct registries. **(a)** Add `"calendar"` to the `ActiveView` union and a `viewTitles` entry in `src/types/active-view.ts` (the union is verified as `discover` / `collections` / `teams` / `outcomes` / `leverage-loops` / `serendipity` / `meeting-prep`), then surface it in the top-nav, side-nav, and network-canvas the way the existing views are. **(b)** Add the file-based route `/calendar` under `src/routes/_authed/`. | Namespace | Any genuinely new calendar table is `cal_*`, marked **NEW** with no id. The new client feature folder is `src/features/calendar/{api,components,hooks,types,constants}`. |
| Calendar is a view | Per the Deal Flow design, a calendar lens is `df_view.view_type='calendar'` (filters/sorts/group_by JSON over `df_task`/`df_deal`) — no new "view" table. The Nylas event layer is the only thing that needs persisted-event storage, and it already has it (`event` 221). |
| Access | Every read/write/broadcast passes through `resolve_access()` or the `df_grant_effective` cache — no DB-level RLS. Org↔user mapping resolves only through `organization_membership` (696) (needs the composite `(user_id, organization_id, status)` btree from the Deal Flow design). |
| Org scoping (the real gap) | `event` (221) has **no** `user_id`, **no** `grant_id`, and **no** `organization_id` — its only FKs are `calendar_id` → `calendar` (223) and `master_event_id` → `master_event` (222). `calendar` (223) carries `user_id` + `grant_id` but **no** `organization_id`. So org scope resolves **two hops**: `event → calendar.user_id → organization_membership` (696). Org-level sharing needs either a denormalized `organization_id` column on `event`/`calendar` (fast range scans) **or** a membership-resolved join on every read. Decide before Phase 3 team rows. |
| History vs notifications | `df_event` is the append-only product timeline (calendar history reads here); `activity` (218) is the notification feed only. Reminder firings fan into `activity` (218) via the `df_event.verb → activity_type` (217) map. |
| Recurrence | Recurring calendar **events** ride the existing `master_event` (222) recurrence master; recurring **tasks** ride `df_task.recurrence_rule` (RFC-5545 RRULE) + the spawn fn. Prefer letting **Nylas expand** the event series (it carries overrides/cancellations/EXDATEs the provider knows about); the client-side `rrule` plugin is the **render-only** fallback and is not source of truth. |
| Timezone storage & rendering | Store **UTC + the original IANA tz id**. App-side date math is **date-fns v4 + `@date-fns/tz` (`TZDate`)** via `toTZDate(value, tz)` / `nowTZ(tz)` from `src/lib/timezone.ts` — **the app does not use luxon.** FullCalendar gets `timeZone={useTimezone()}` plus a FullCalendar tz **adapter** (`@fullcalendar/luxon3` or `@fullcalendar/moment-timezone`); that adapter is the **only** reason luxon would appear in the bundle, and only for named-zone *rendering*. No `add_days_to_timestamp` exists in XanoScript — use `add_secs_to_timestamp(days * 86400)`, and compute DST-spanning ranges in the named tz, not by adding raw seconds. |
| Index reality (verified) | Only PK + buildable composite btree / unique / full-text / spatial / vector indexes exist (no GIN/partial). **Verified live:** `event` (221), `calendar` (223), `master_event` (222), and `nylas_grant` (183) **each carry exactly two indexes — `primary(id)` and `btree(created_at desc)`** — nothing on `start`/`end`/`calendar_id`/`master_event_id`/`grant_id`/`user_id`. Time-range scans need new btrees (see Build plan, Phase 1). |
| Webhook idempotency | Inbound Nylas webhooks dedupe through a new `cal_webhook_event` (NEW) ledger keyed on a **unique delivery id** — redelivery is a no-op upsert. Upserts are **last-writer-wins on the provider's reported state** (guarded by the event's `updated_at`/sequence), not on arrival order, so out-of-order delivery is safe. Raw bodies still land in `webhook_calendar_log` (618) for audit. |
| Theme | The calendar **observes** the account theme: `.dark`/`.light` flips FullCalendar's `--fc-*` vars (mapped onto app OKLch tokens) with **no JS theme switching**, and the selected accent (`THEME_COLORS`) drives today-highlight / selection / the primary "my events" layer. Category colors keep WCAG contrast in both modes and always pair color with a label/icon. |
| Premium licensing | Team overlays (resource Timeline + vertical resource) **and** `multiMonthYear` are **all FullCalendar premium** and require a **purchased `schedulerLicenseKey`** in production (the bundled `CC-Attribution-NonCommercial-NoDerivatives` eval key is non-commercial CC and not valid for Orbiter prod). Ship a free-tier fallback (month/week/day/agenda only, no resource rows) as a **permanent** tier, not a stopgap. |
| Privacy & purge | The mirror caches real calendar content (titles, descriptions, attendees, locations). Every read is `resolve_access()`-gated; `calendar.read_only` is respected; the team-overlay read redacts to the entitled ladder rung server-side. **Disconnect / `grant.expired` / revoke / account-delete must hard-purge the mirror** (soft `event.deleted_at` → purge sweep on the new `event(deleted_at)` btree) and stop sync. |
| Safe field access | Nylas responses are read with `\|get:"field":default` (mandatory) so a missing key never throws; defaulted bool flags read with `first_notnull` (false is "empty", so `first_notempty` would coerce it back to the default). |

### Full-screen canvas

- **Month** — `dayGridMonth` (`@fullcalendar/daygrid`): the classic month grid (also does multi-week), color-coded by Event Source, hover preview, click-to-drill into day/week.
- **Week** — `timeGridWeek` (`@fullcalendar/timegrid`): vertical time-of-day columns with business-hours shading and selectable ranges for drag-create.
- **Year** — `multiMonthYear` (`@fullcalendar/multimonth`, **premium**): a full-year grid for long-range planning (close-month forecast, project targets).
- **Team overlay** — the resource **Timeline** (`@fullcalendar/resource` + `@fullcalendar/resource-timeline`, **premium**): horizontal time axis, **one row per teammate/calendar**, `resourceGroupField` to group rows (by team), `resourceAreaColumns` to turn the left area into a data grid, and **drag an event between people** (`eventDrop` re-assigns through `schedule_meeting`). A vertical resource view (`@fullcalendar/resource-timegrid`, **premium**) puts a few teammates side-by-side as columns for one day/week. Teammate rows show only what the privacy ladder permits (`redact_to_rung`, server-side).

### Docked side-panel

- **Agenda** — `listWeek` / `listDay` (`@fullcalendar/list`): a chronological list for quick scanning and conflict spotting, rendered in a narrow always-docked panel built from the reused `event-card` / `date-divider` components. This is the surface a **single `0`** opens.
- **Single day** — `timeGridDay` (`@fullcalendar/timegrid`): today's hour-by-hour column for at-a-glance "what's next," sitting beside email / signals / the graph.

The side-panel and the canvas are two FullCalendar instances over the same Event Sources, so an event can be **dragged from the panel onto the canvas** (and back): the panel exposes its `event-card`s as `Draggable` items and the canvas accepts the drop through the same `schedule_meeting` path. The panel is the calendar's persistent presence — it stays docked while the operator is in email, the graph, or a deal, and a single `0` toggles it without leaving the current view. Each agenda row also carries a **"Prep with AI"** affordance that deep-links into the Copilot (see below).

### AI-first scheduling & meeting prep (human-in-the-loop)

The calendar is built for an operator running many relationships at once, so the AI carries the load and the human keeps the wheel. This is grounded in surfaces that **already exist** in `src/features/copilot`, not new infrastructure: the Copilot is CrayonChat v2 backed by the Robert API group `Bd_dCiOz` (id 1261, Thesys C1 wrapping Claude), it reads the same `useCalendarEvents()` data, it renders structured cards (`meeting_prep_card`, `meeting_goal_card`, `contact_card`) via its template registry, it speaks/listens through voice hooks, and its default-mode conversation starters already **rank the next meeting and auto-draft** a `Prep for {meeting title} — {relative time}` prompt with talking points (`use-conversation-starters.tsx`).

The loop is always **propose → confirm → reconcile**:

<Steps>
  <Step title="AI proposes">
    The Copilot proposes meeting times off **Nylas Availability** across participants, drafts the invite (title/body/attendees), and binds the meeting to the right deal (`cal_event_link` to a `df_deal`). It also **ranks** the day's meetings into the attention budget — `df_focus_entry` (`entry_type='meeting'`, `df_suggestion.source_engine='meeting_prep'`) — so the operator triages a prioritized list, not the raw grid. A proposal is a draft, never a write. Across teammates, the AI sees only the Free/Busy each privacy ladder has shared (`redact_to_rung`).
  </Step>
  <Step title="Operator confirms">
    Nothing irreversible happens silently. The operator reviews the proposed time/invite/binding in the Copilot card (or inline on the grid) and confirms. Confirmation is the only thing that commits — or, where an account has explicitly opted into higher autonomy, an auto-confirm setting stands in for the click, but the action is still attributed and reversible.
  </Step>
  <Step title="One verb writes it">
    Confirmation invokes the **single** `schedule_meeting` `df_action_def` verb (`source_engine='calendar'`) — the identical autonomy-gated, attributable path a human button-click, drag-create, or chip-drop uses. The write goes to Nylas by `grant_id`; the client paints an optimistic pending block. A write against an expired grant is blocked with a reconnect prompt, not silently dropped.
  </Step>
  <Step title="Webhook reconciles (and makes it undoable)">
    The authoritative `event.*` webhook upserts the mirror by `external_id`, replacing the optimistic block. Because the canonical state lives in Nylas and reconciles through the webhook, the action is **reversible** — an undo is just another `schedule_meeting` proposal (cancel/move) through the same path. `df_suggestion` lineage records that the AI proposed and the human confirmed.
  </Step>
</Steps>

### AI proposal contract

Trust lives in the proposal object, not in a vague promise that "AI helped." Every AI scheduling surface renders the same inspectable object before commit:

| Proposal field / state | Operator-visible behavior | Backend contract |
| ---------------------- | ------------------------- | ---------------- |
| Suggested slot | Shows time, timezone, attendees, calendar, conflicts, and why this slot was chosen | Computed from Nylas Availability and the same redacted Free/Busy projection the team timeline uses. |
| Draft invite | Title, body, attendees, conferencing, and linked deal/person/company are editable before confirm | Stored as proposal payload and written only after `schedule_meeting` confirmation. |
| Confidence / uncertainty | Low-confidence proposals ask a focused question or show alternatives instead of presenting a single false certainty | `df_suggestion` records model confidence, source context, and the chosen operator correction. |
| Operator edits | Editing a time, attendee, title, or binding removes any misleading "AI decided" provenance from that field | The final confirmed payload records both proposed values and accepted values for attribution. |
| Pending write | A confirmed proposal becomes a pending calendar block with a clear "syncing" state | Pending id resolves to Nylas provider id through webhook or rolls back on failure. |
| Failed proposal | Expired grant, read-only calendar, rate limit, no mutual availability, or privacy denial renders as a failed proposal with a retry path | Failure does not create or mutate `event` (221); it records the failed action result for audit and model tuning. |
| Undo / repair | Undo is shown beside the confirmed action while the operator still has context | Undo is another `schedule_meeting` proposal, not a direct mirror mutation. |

The AI is allowed to recommend, explain, and remember preference corrections; it is not allowed to hide uncertainty, invent availability, reveal hidden teammate detail in a rationale, or leave a half-created event behind when Nylas rejects a write.

How each AI capability maps to the surfaces that already ship:

| AI capability | Surface | Verb / contract |
| ------------- | ------- | --------------- |
| Propose times across participants | Copilot chat / inline grid card; Nylas Availability (over `redact_to_rung` Free/Busy) | `schedule_meeting` (proposal → confirm) |
| Draft an invite (title/body/attendees) | Copilot draft card | `schedule_meeting` (create) |
| Bind a meeting to a deal/person/company | Copilot binding card / external-chip drop | `cal_event_link` written under the same confirm |
| Rank "meetings to prep" | AI attention budget lane | `df_focus_entry` (`entry_type='meeting'`) |
| Generate a meeting briefing | Copilot **meeting mode** `MeetingPrepCard` (shipped [Meeting Prep](/guides/modes/meeting-prep)) | read-only; `/meeting-prep` Robert endpoint |
| Hands-free scheduling/prep | Copilot voice (Gradium/ElevenLabs TTS, Web-Speech/Gradium STT) | resolves to the same verb |

**Deep-link contract.** Any calendar surface hands work to the Copilot by dispatching `window.postMessage({ target: 'Copilot', data: { action: 'open', section?, prompt? } })` (verified in `copilot-manager.tsx`). The optional **`prompt`** is the load-bearing piece: clicking "Prep with AI" on an agenda row, an event card, or a `df_focus_entry` opens the Copilot with a **pre-drafted** prompt (e.g. `Prep for my meeting '{title}' with {attendees} {relative time} — pull context and give me talking points`), exactly the shape the default-mode starter already builds. Reschedule/booking flows pass a scheduling prompt the same way. The Copilot never writes the calendar through the Robert group — it reads via `useCalendarEvents()` and writes only by invoking the shared `schedule_meeting` verb.

**Natural-language is the input layer.** AI- and voice-proposed times, and every operator-typed date, resolve through the **same** shared NL date parser to a `TZDate` (in `useTimezone()`), so "next Tuesday 2pm" spoken to the Copilot, typed in the create dialog, or used in a filter all land on the identical resolution path. The parser + UX is designed on its own subpage, [NL date picker](/guides/open-work/calendar/nl-date-picker) — referenced here, not inlined.

### Theme-aware rendering

The calendar **observes** the account theme — it never runs its own theme switch.

The calendar maps FullCalendar's `--fc-*` CSS vars onto the app OKLch tokens in `styles.css`, **scoped so the `.dark` class flips them automatically** (`@custom-variant dark (&:is(.dark *))`) — no JS theme switching of the calendar. ### Interaction & data features

- **Drag-and-drop everywhere** via `@fullcalendar/interaction` — `select`/`dateClick` to create, `eventDrop`/`eventResize` to reschedule, **reassign across teammate rows** on the resource timeline (`eventDrop` changes the owning resource), the **`Draggable`** API to drop **external chips** (`df_task` / `df_deal` / `master_person` / `master_company`) from outside the grid onto a date to schedule (writing the `cal_event_link` edge), and dragging events **between the docked side-panel and the canvas**. Every drop routes through the single `schedule_meeting` write contract → Nylas by `grant_id` → webhook reconcile — **never a direct mirror mutation.** The drop optimistically paints a pending block keyed to the pending `external_id`, replaced when the `event.*` webhook lands; a failed write rolls it back with a toast (`@/components/ui/sonner`). A drop onto a teammate row the ladder does not expose (or a `read_only` calendar) is rejected before it leaves the client, with the same toast.
- **Natural-language date entry everywhere** — a shared NL date input ("next Tuesday 2pm", "in 3 days") across create / reschedule / filter / reminder, and the resolution layer for AI- and voice-proposed times. Parsing + UX is designed on its own subpage, [NL date picker](/guides/open-work/calendar/nl-date-picker); the calendar just consumes the resolved `TZDate` (resolved in `useTimezone()`).
- **Command palette** — add a Navigation command in `src/features/command-palette/constants/commands.tsx` (`getCommands(navigate, queryClient, currentUser)` returns `CommandDefinition[]`): `{ id:'calendar-view', label:'Calendar', category:'Navigation', icon:<CalendarIcon className='size-4'/>, action:()=>navigate({to:'/calendar'}), shortcut:'0 0' }`. The `shortcut` field is the display string (rendered in the palette); the `hotkey` field (react-hotkeys-hook combo) is intentionally left unset here because the `0` behavior is the custom double-tap layer below, not a single combo. It renders under the existing **Navigation** heading automatically (`command-palette.tsx` filters `c.category === "Navigation" && !c.hidden` into a cmdk `Group heading="Navigation"`).
- **Hotkeys (strict double-tap)** — `use-command-hotkeys.ts` already loops every command and calls `useHotkeys(cmd.hotkey ?? "", handler, { enabled: !!cmd.hotkey && !cmd.hidden })`. Add a **dedicated** `useHotkeys('0', …)` alongside that loop with react-hotkeys-hook v5 (`react-hotkeys-hook ^5.2.4`, verified). It disambiguates single vs double press with a ~300ms window: on the **first** `0`, start a ~300ms timer that opens/toggles the **agenda-day side panel**; if a **second** `0` arrives within the window, cancel that timer and open the **full-screen calendar** instead. Net: single `0` = side panel (deliberately delayed ~300ms to disambiguate), `0`-then-`0` = full screen. react-hotkeys-hook fires single combos only, so the double-tap timing is a small custom layer (track `lastZeroAt` / a `useRef` timeout) on top of `useHotkeys('0')`. react-hotkeys-hook by default does **not** fire inside inputs/textareas (`enableOnFormTags=false`), so a bare `0` is safe while typing in the NL date input, search, or any field. The palette shows `'0'` (panel) and `'0 0'` (full screen).
- **Timezone** — `FullCalendar timeZone={useTimezone()}` where `useTimezone()` (`src/lib/timezone.ts`) returns `user?.local_timezone ?? Intl.DateTimeFormat().resolvedOptions().timeZone` (user from `useCurrentUser()`, `src/features/auth/api/use-current-user`; `AuthUser.local_timezone` == `user` 125 `local_timezone`, captured during auth/onboarding). A FullCalendar tz **adapter** plugin (`@fullcalendar/luxon3` or `@fullcalendar/moment-timezone`) handles named-zone *rendering*; all app date math is `toTZDate(value, tz)` / `nowTZ(tz)` on `@date-fns/tz` (verified: `date-fns ^4.1.0`, `@date-fns/tz ^1.4.1`) — **the app does not use luxon for its own logic.**
- **Week start** — FullCalendar `firstDay` reads the one user preference `user` (125) `settings.week_start` (0 = Sun, 1 = Mon); both the canvas and the side-panel read the same value.
- **Recurrence** — render recurring series via `@fullcalendar/rrule`; prefer Nylas-expanded instances off `master_event` (222), client-side rrule expansion is the render-only fallback (it drifts on overrides/exceptions/DST — never source of truth).
- **Category color-coding** — multiple toggleable Event Sources, one per category, mapped to theme tokens; the Nylas, `df_task`, and `df_deal` layers each own a distinct color that holds contrast in both modes.
- **Attendees / RSVP** — render participants + RSVP responses from the Nylas event (`event` 221 `event_data` JSON) with the reused `event-participant-badge` / `attendance-status` components; accept/decline writes through `useSendRsvp()` (`src/features/events/api/use-send-rsvp.ts`) → the [Send RSVP](/api-reference/xano/v2-calendar-events/calendar-events-send-rsvp) endpoint → Nylas.
- **Reminders** — per-event nudges scheduled in `df_reminder`; a cron fires `remind_at` rows into `activity` (218) + push.
- **AI-first scheduling, human-in-the-loop** — the Copilot proposes times, drafts the invite, binds it to a deal, and ranks "meetings to prep" into `df_focus_entry`, all on the same reversible `schedule_meeting` verb that backs the button, the drag-create, and the chip-drop; operators triage the AI-ranked attention budget, not the raw grid, and each entry deep-links into the Copilot via `postMessage` target `'Copilot'` with a pre-drafted `prompt`. The full propose → confirm → reconcile loop, the capability→surface map, and the deep-link contract live in [AI-first scheduling & meeting prep](#ai-first-scheduling-meeting-prep-human-in-the-loop) above.

### Responsiveness, accessibility & keyboard

The calendar is keyboard-first and degrades cleanly to small screens — both are acceptance criteria, not polish.

- **Responsiveness** — on wide screens the canvas and docked side-panel sit side by side; below the app's panel breakpoint the side-panel collapses to an overlay (still opened by single `0`) and the canvas takes full width. On narrow widths the default view drops from `timeGridWeek` to `timeGridDay`/`listWeek` (week columns are unreadable on phones), the resource Timeline falls back to a scrollable single-team view, and the header controls wrap into a compact toolbar. Touch gestures map to the same `eventDrop`/`eventResize`/`select` handlers via the interaction plugin's pointer support.
- **Keyboard navigation** — arrow keys move the focused day/time cell, `Enter` opens create on the focused cell, `Tab` cycles focusable events, `Enter`/`Space` opens the focused event, and `Esc` closes popovers/dialogs. The view switcher, legend toggles, and date navigator are all reachable and operable by keyboard. The double-tap `0` enters from anywhere (form-tag-guarded); the command palette (`⌘K`) is the discoverable entry point listing the `'0'` / `'0 0'` shortcuts.
- **Accessibility** — the grid carries proper roles/labels (grid semantics, `aria-label` on cells and events with the human-readable date/time + title), focus is trapped and restored in dialogs/popovers, and the now-indicator and today-highlight are announced, not just colored. **Color is never the only signal** — every category and RSVP state pairs color with a label and icon, and the chosen token pairs pass WCAG AA contrast in both light and dark. Motion respects `prefers-reduced-motion` (the app's existing `motion-reduce:transition-none` convention carries into event transitions and drag affordances). Pending/optimistic blocks are marked with a visible + screen-reader "pending" state until the webhook reconciles. AI proposals also carry an explicit "proposed by AI — confirm to schedule" affordance so the human-in-the-loop boundary is visible, not implicit. Busy-only blocks announce as "Busy" with no leaked detail.

<Warning>
**FullCalendar team overlays are premium and cost money in production.** The resource **Timeline**, **vertical resource**, and **multiMonthYear** views all require a purchased `schedulerLicenseKey`. The bundled eval key `CC-Attribution-NonCommercial-NoDerivatives` is **non-commercial Creative Commons only** and is not valid for Orbiter production. Phases 0–2 use **only the free MIT plugins** (`daygrid`, `timegrid`, `list`, `interaction`, `rrule`, plus the tz adapter) — month/week/day/agenda with no resource rows — so the premium license becomes a deliberate Phase-3 cost decision, with the free-tier (no team overlay) as a **permanent** fallback. The AI prep surface (ranked `df_focus_entry`, Copilot deep-link, meeting briefings) rides the free tier; only AI *booking across teammates* (Availability over the resource timeline) waits on Phase 3.
</Warning>

## Build plan

Four phases, **dependency-ordered**, each shipping standalone value: **P0** read-only `df_*` overlays on free plugins + palette/hotkeys/theme/timezone, then **P1** connect/sync/mirror + core views + the settings ladder + the missing btrees, then **P2** write-back/recurrence/graph-linking + drag-everywhere + NL date entry, then **P3** premium team overlays + Free/Busy + Nylas Scheduler + AI cross-teammate scheduling. ## Open questions / risks

<AccordionGroup>
  <Accordion title="AI autonomy, confirmation & attribution (human-in-the-loop)">
    - **The AI is a proposer, never an actor.** Every AI scheduling/reschedule/bind action routes through the *same* `schedule_meeting` verb a human click uses — there is no separate agent write path that can bypass the operator. The boundary must be **visible** in the UI ("proposed by AI — confirm to schedule"), not implicit.
    - **No silent irreversible action.** Default is confirm-before-write. Higher autonomy (auto-confirm) is an explicit per-account setting on the autonomy gate, and even then the action is attributed (`source_engine`, actor, `df_suggestion` lineage) and reversible via the webhook-reconcile path (undo is another `schedule_meeting` proposal).
    - **The AI inherits the privacy ladder.** When proposing across teammates it sees only the Free/Busy `redact_to_rung` returns — never raw detail a teammate has not shared. It also cannot propose against an expired grant: a proposal touching a `grant_expired` account is blocked, not silently dropped.
    - Open: where does the autonomy threshold live (per-user, per-org, per-deal)? Default: conservative per-account confirm-on; revisit per workflow once usage data exists.
    - Risk: an AI proposal that fails Nylas (rate limit, expired grant, declined attendee) must surface as a *failed proposal*, not a silent no-op — same rollback + toast as a human write.
  </Accordion>
  <Accordion title="Per-account privacy ladder enforcement (busy vs detail)">
    - The three toggles (`enable_calendar` ON / `share_conflicts` ON / `share_conflict_details` OFF) are a **strict, monotonic** ladder on `nylas_grant` (183), read via `first_notnull` (false must not coerce to default).
    - **Enforcement is server-side, before serialization.** One projection function — `redact_to_rung(event, viewer, owner_grant)` — runs on every team-facing read and every Availability fan-out. Below the detail rung it strips title, attendees, description, location, URL, and `event_data` and returns an opaque "Busy" block at the real `start`/`end`; below the busy rung it returns nothing; with `enable_calendar = false` the account is absent from the overlay entirely. The redacted fields **never reach the wire** — the settings UI's parent-OFF-disables-children cascade is convenience, not a security boundary.
    - The **owner always sees their own calendar in full** — the ladder governs *team exposure*, not self-view; the viewer-vs-owner check is part of the read.
    - Both the human resource-timeline read and the AI Availability path call the **same** `redact_to_rung`, so there is no second, looser code path. Everything is `resolve_access()`-gated on top of the ladder, so a teammate outside the org never reaches even the busy-only rung.
    - Open: does an **org admin** override (compliance/legal hold) ever supersede an individual's OFF? Default: no — individual setting wins; revisit only if org policy demands it, and log any override into the audit trail.
  </Accordion>
  <Accordion title="Org scoping on legacy event / calendar (verified gap)">
    - **Verified:** `event` (221) has no `user_id`/`grant_id`/`organization_id` (only `calendar_id` + `master_event_id`); `calendar` (223) has `user_id`/`grant_id` but no `organization_id`. Org scope resolves two hops: `event → calendar.user_id → organization_membership` (696).
    - Decision before Phase 3: **denormalize `organization_id`** onto `event`/`calendar` (fast `(organization_id, start)` range scans, but a backfill + sync write-time cost, and a keep-in-sync burden when a calendar moves orgs) **vs.** membership-resolved join on every read (no schema change, slower team scans). Lean denormalize for the resource timeline.
    - Either way, team-overlay reads must pass `resolve_access()` **and** `redact_to_rung` — there is no DB-level RLS, and org scope alone does not imply detail entitlement.
  </Accordion>
  <Accordion title="Missing time-range indexes (verified)">
    - **Verified:** `event` / `calendar` / `master_event` / `nylas_grant` each carry only `primary(id)` + `btree(created_at desc)`. Every FullCalendar window read is currently an unindexed scan on `start`/`end`/`calendar_id`.
    - **Phase 1 must ship the new btrees before real data lands** (`event(calendar_id, start)`, `event(start)`, `event(end)`, `event(master_event_id)`, `event(deleted_at)`, `calendar(grant_id)`, `calendar(user_id)`, `nylas_grant(grant_id)` unique, `nylas_grant(user_id)`) — otherwise the canvas reads degrade with row count.
    - No GIN/partial indexes in Xano — composites only; pick column order to match the dominant filter (`calendar_id` then `start`). The `event(deleted_at)` btree doubles as the purge-sweep index (see Purge & retention).
  </Accordion>
  <Accordion title="Webhook reliability, idempotency & initial backfill">
    - Initial sync can be large (years of history) — bound it (`nylas_grant.timerange` / `oldest_sync`) and **page** it so a single backfill never stalls the receiver or hits Nylas rate limits.
    - Webhooks can arrive **out of order, duplicated, or be missed** — dedupe via `cal_webhook_event` (NEW, unique `delivery_id`, redelivery = no-op), upsert by `external_id`, and resolve order by comparing the payload's `updated_at`/sequence to the stored row (**last-writer-wins on provider state, not arrival order** — a stale delivery is discarded).
    - Missed webhooks are caught by a **periodic delta poll** bounded by the same sync window; treat the poll as the backstop, not the primary path.
    - `grant.expired` must set `nylas_grant.grant_expired`, surface a reconnect prompt, and **pause writes** for that grant (including AI proposals — a proposal against an expired grant is blocked, not silently dropped). On reconnect, re-run a bounded delta to close the gap.
  </Accordion>
  <Accordion title="Single write contract — no direct mirror mutation">
    - Every write surface (button, drag-create, resize, teammate-reassign, external-chip drop, cross-panel drag, AI Copilot) funnels through `schedule_meeting` (`df_action_def`, `source_engine='calendar'`) → Nylas → webhook → mirror upsert. The frontend never writes `event` (221); the client path is the `use-schedule-meeting` mutation only.
    - Optimistic UI shows a pending state (with a screen-reader "pending" label) until the matching `event.*` webhook lands (or a short reconcile poll confirms); a failed Nylas write rolls the optimistic state back with a toast.
    - Risk: webhook latency makes a just-created event "pop in" late — mitigate with the optimistic placeholder keyed to the pending `external_id`, replaced on reconcile.
    - Risk: a drop onto a `read_only` calendar or a teammate row the ladder does not expose must be rejected (client- and server-side) with the same rollback toast — not a half-written state.
  </Accordion>
  <Accordion title="Timezone storage & rendering (date-fns v4, not luxon)">
    - Store **UTC + the original IANA tz id**; never store wall-clock only. App date math is **date-fns v4 + `@date-fns/tz` (`TZDate`)** via `toTZDate`/`nowTZ` in `src/lib/timezone.ts`.
    - FullCalendar `timeZone` = `useTimezone()` (`user.local_timezone` ?? browser tz) with a FullCalendar tz **adapter** (`@fullcalendar/luxon3` or `@fullcalendar/moment-timezone`) — that adapter is the **only** reason luxon enters the bundle, and only for named-zone rendering. Do not migrate app logic to luxon.
    - Reconcile **three** tz sources: the event's tz (Nylas), the calendar's tz (`calendar` 223 `timezone`), and the user's (`user` 125 `local_timezone`). They can disagree (a meeting created in one zone, owned by a calendar in another, viewed by a user in a third) — the user's `useTimezone()` is the render authority, but the stored IANA id is what makes the conversion correct.
    - XanoScript has no `add_days_to_timestamp` — use `add_secs_to_timestamp(days * 86400)`, and compute **DST-spanning** ranges in the named tz (a "7 days from now" that crosses a DST boundary is not 7×86400 wall-clock seconds), never by adding raw seconds.
    - AI/voice-proposed and operator-typed dates all resolve through the same NL parser → `TZDate` in `useTimezone()`, so there is one resolution path, not a parallel one for spoken input. All-day events (no time component) must be treated as floating, not coerced into a tz.
  </Accordion>
  <Accordion title="Recurrence expansion — Nylas vs client-side rrule">
    - Prefer **Nylas-expanded** instances off `master_event` (222) so the mirror matches the provider exactly — the provider knows the **overrides** (a single moved instance), **cancellations** (EXDATE), and **exceptions** that an RRULE alone does not encode. Index `event(master_event_id)` to pull a whole series.
    - `@fullcalendar/rrule` client-side expansion is a **render-only fallback** and **drifts** on exceptions, overrides, and DST transitions — do not treat it as source of truth, and prefer the mirrored instances whenever they exist.
    - "This and following" / single-instance edits must route the same `schedule_meeting` verb to Nylas, which decides whether it touches the master or splits the series — the client must not try to rewrite the RRULE itself.
    - Recurring **tasks** are a **separate** model: `df_task.recurrence_rule` (RFC-5545) + spawn fn via `recurrence_parent_task_id`. Do not merge the two recurrence systems.
  </Accordion>
  <Accordion title="Theme-aware rendering parity">
    - FullCalendar `--fc-*` vars map onto app OKLch tokens, scoped so the `.dark` class flips them with **no JS theme switch** (`@custom-variant dark (&:is(.dark *))`). Selected accent (`THEME_COLORS`) drives today/selection/"my events".
    - Risk: category colors that pass contrast in light can fail in dark (and vice-versa) — pin each category to a `--chart-1..5` token pair that passes WCAG in **both**, and always pair color with a label/icon (never color-only). Test the legend, event blocks, the busy-only block, and the now-indicator in both modes and at the largest density.
  </Accordion>
  <Accordion title="Responsiveness, a11y & keyboard coverage">
    - The side-panel must collapse to an overlay below the panel breakpoint (still opened by single `0`), the canvas default view must drop from week to day/agenda on narrow widths, and the resource Timeline must fall back to a scrollable single-team view — none of these are optional polish.
    - Keyboard nav (arrow-to-cell, `Enter`-to-create/open, `Esc`-to-close), focus trap/restore in dialogs, ARIA grid semantics, and `prefers-reduced-motion` are acceptance criteria. The double-tap `0` must stay form-tag-guarded so it never fires while typing.
    - AI-proposed blocks and busy-only blocks must be distinguishable to assistive tech ("proposed by AI, confirm to schedule" / "Busy"), not color-only.
  </Accordion>
  <Accordion title="FullCalendar premium licensing cost">
    - Team overlays (resource Timeline + vertical resource) **and** `multiMonthYear` are all premium and need a **purchased** `schedulerLicenseKey` in production.
    - The bundled `CC-Attribution-NonCommercial-NoDerivatives` key is **non-commercial Creative Commons** — not valid for Orbiter prod, and shipping it would be a license violation, not just a watermark.
    - Decision: budget the license for Phase 3, or ship the free-tier fallback (month/week/day/agenda, no resource rows) **permanently**. Phases 0–2 stay 100% free MIT plugins, so the spend is a deliberate Phase-3 gate, not a launch blocker. The AI prep surface (ranked budget, Copilot deep-link, briefings) is free-tier; only AI cross-teammate booking is gated.
  </Accordion>
  <Accordion title="Nylas rate limits, provider quirks & team Free/Busy at scale">
    - Nylas rate-limits Events / Availability / Free-Busy — batch, cache against the mirror, back off; **do not call Nylas on every FullCalendar paint or scroll.** A resource Timeline of N teammates × a week is N Free/Busy lookups — cache aggressively, refresh on **webhook**, not on scroll, and let the mirror serve the grid while Free/Busy fills only the unbookable gaps.
    - AI scheduling amplifies this: a "propose times across 6 participants" call is an Availability fan-out — debounce it, cache the slot computation, and never re-run it on every keystroke or every panel re-render.
    - Provider quirks: **iCloud / IMAP** have limited calendar support, **no reliable push** (rely on the delta poll there), and slower sync; **Microsoft shared-calendar** access needs the right **delegated scopes** at connect time (a missing scope yields empty Free/Busy, not an error); conferencing (Meet/Zoom/Teams) differs per provider and may not round-trip identically.
    - Cross-org / external participants in Availability often have **no grant** — fall back to Free/Busy-only or a public booking page, and require explicit consent before sending invites; never assume a grant exists for an attendee email.
  </Accordion>
  <Accordion title="Privacy & data-residency of the mirror">
    - The mirror caches real calendar content (titles, descriptions, attendees, locations) in `event` (221) — gate **every** read through `resolve_access()`, respect `calendar.read_only` on writes, and let `redact_to_rung` collapse to busy-only for any team view below the detail rung.
    - Mirrored content is `organization_id`-scoped on `cal_*` tables; the **legacy** `event`/`calendar` tables need the org-scope decision above before any org-wide sharing — until then, team reads resolve org through the membership join and still redact per the ladder.
    - The Copilot sends calendar context to the AI provider (Thesys C1 wrapping Claude via Robert API `Bd_dCiOz`) — that data flow **inherits the same access gates and the ladder**; it must never carry content the operator could not see in the UI, and it must never carry a teammate's detail below their shared rung.
    - Data-residency: the mirror is a second copy of provider data — its retention, region, and purge behavior are an explicit policy decision, not an accident of caching (see Purge & retention).
  </Accordion>
  <Accordion title="Purge & retention (disconnect / revoke / delete)">
    - **Disconnect / `grant.expired` / revoke must stop sync and hard-purge the mirror** for that grant: mark `event.deleted_at`, then sweep-delete on the new `event(deleted_at)` btree so titles/attendees/locations do not linger as stale cached content after the user has cut the connection.
    - **Account / org deletion (right-to-erasure)** must purge `event` (221), `calendar` (223), `master_event` (222), `cal_event_link`, `cal_calendar_pref`, and the grant row — and decide retention for `webhook_calendar_log` (618) / `cal_webhook_event` (audit logs may need a shorter, separate retention than live mirror content).
    - A provider-side delete (`event.deleted`) sets `deleted_at`; the periodic sweep does the physical purge. Keep the soft-delete window short for privacy, but long enough to absorb out-of-order redelivery.
    - Open: do we retain the raw `webhook_calendar_log` (618) bodies after a purge for audit, or scrub them too? Default: scrub PII-bearing bodies on account deletion; keep a redacted delivery ledger for idempotency only.
  </Accordion>
  <Accordion title="cal_connection vs reusing nylas_grant (183)">
    - Default decision: **reuse `nylas_grant` (183)** — the grant, the three privacy toggles, and the default-calendar pointer all live there. Only create `cal_connection` (NEW) if calendar-specific connection state emerges that does not belong on the mailbox grant row.
    - Open: does org-shared calendar connect policy belong on the grant, on `organization` (695), or on a thin `cal_connection`? Resolve before Phase 3 team overlays.
  </Accordion>
</AccordionGroup>
