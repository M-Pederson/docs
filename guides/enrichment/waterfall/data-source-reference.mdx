The `data_source` table (Xano table **161**) is **deprecated**. It mapped enrichment providers to integer `data_source_id`s that were written across the graph and join tables. We are replacing every `data_source_id` with a plain‑text **`source`** string — pipeline by pipeline — so the table can be safely dropped.

This page is the **source of truth** for that migration: each legacy `data_source_id` and its canonical `source` string.

<Note>
**Status legend**

- ✅ **Confirmed** — locked; safe to write in code
- 🟡 **Proposed** — best guess, awaiting sign‑off
- 🔴 **Needs input** — string not yet decided
- ⬜ **Not migrated** — provider not yet swept (legacy table name shown for reference)

⚠️ marks a canonical string that **intentionally differs** from the legacy table's `source` name — the live data convention wins (e.g. `91` is "PDL" in the table but stored as **"People Data Labs"**).
</Note>

## Migration progress

- **Person Enrichment pipeline** — writer + caller waves **complete**: every site now writes/passes `source` (`data_source_id` kept transitionally). Remaining: backfill `enrich_history_person.source` (held), then switch the phase-0/phase-1 dedup queries + `resolve-temp-name` from `data_source_id` to `source`, then drop `data_source_id`. Active ids: `0, 3, 5, 7, 8, 11, 20, 59, 72, 73, 79, 86, 89, 91, 94, 97, 99, 101`.
- **Company / IMDB / Nylas / Merge / Activity pipelines** — not yet swept.

## Provenance principle

Process‑stage ids like **`Work Enrich` (73)** and **`Education Enrich` (72)** are **not** replaced by a fixed string. Instead, the **`source` rides on the data record**: when a `work_experience` / `education_experience` row is written, its `source` is set to the *provider that found it* (e.g. PDL → `People Data Labs`). Any company / link / position created from that record **inherits `record.source`** rather than hardcoding a stage label.

This requires a `source` text field on the record table:

- `work_experience` (147) — ✅ already has `source`
- `education_experience` (230) — ➕ `source` field **to be added** (decided), then same pattern

<Warning>**Schema change required:** add a `source` (text) field to `education_experience` (table 230) before migrating id 72.</Warning>

## Conventions & gotchas

- **`91` → "People Data Labs"** (not "PDL") and **`79` → "Master Person Enrich"** (not "Base Person Enrich"). The legacy table names are misleading; the data tables already store the longer strings, so those win.
- **`0`** is a sentinel written in a few places (e.g. `process-person-phase-3` → `add-person-biography`) and does **not** exist in table 161. Needs an explicit decision (a string, or leave `source` unset).
- **`null` `data_source_id` passthroughs** (many plumbing calls pass `null`) simply drop the field — no `source` is written.
- **Backfill required:** dedup `where` clauses that match `data_source_id == N` (e.g. `process-person-phase-0` on `enrich_history_person`) switch to matching `source`. Existing rows must have `source` backfilled first or dedup will miss them and create duplicates.
- **Shared writers** (`create-master-link`, `add-person-biography`, `add-position-record`, `add-education-record`, `replace-avatar`, the `*_join` helpers, and `get-add/master-company`) accept `data_source_id` as an input and are called by the company and ingest pipelines too — their signatures change to a `source` text param, so all callers must update together.
- **String standardizations:** `Exa.ai` → **`Exa`** (id 101) and `SIgnal.NFX.com` / `SIgnal NFX` → **`Signal NFX`** (id 64). Code fixed to match: `master-person-from-exa` (#12997 — a `create-master-link` call still wrote `"Exa.ai"`) and `replace-avatar` (#237 — avatar-priority logic compared `source == "SIgnal NFX"`; the rest of the codebase already used `Signal NFX`).
