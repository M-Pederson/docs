This page catalogs every existing Xano function that interacts with the FalkorDB graph and maps each one to its impact on Known. Known is primarily greenfield — most Known functions are new — but a handful of existing functions either **create edges Known reads from** (upstream dependencies) or **create edges Known unifies with** (like the `MENTOR` edge discovered during scenario stress tests).

**Nothing in this page has been changed in Xano yet.** This is the reference for a future implementation pass.

**Workspace:** `OrbiterV2` (workspace_id: 3)

---

## Categories

Each existing Xano function falls into one of four categories:

<CardGroup cols={2}>
  <Card title="🔴 Direct cypher update" icon="pen-to-square">
    The function contains an embedded cypher template that writes to an edge Known has redesigned. The cypher needs to be replaced with the Known-aware version.
  </Card>
  <Card title="🟡 Trigger hook needed" icon="bolt">
    The function writes to an edge Known reads from (upstream). No cypher changes needed, but Known's lifecycle crons need to be triggered when this function runs.
  </Card>
  <Card title="⚪ Legacy / deprecated" icon="trash-can">
    The function is part of an older surface that Known replaces entirely. Mark as deprecated and route callers to Known's new functions. Do not update the cypher — it's going away.
  </Card>
  <Card title="🟢 No changes" icon="circle-check">
    The function writes to edges or nodes that Known does not touch (film/TV, expertise, locations, funding rounds, etc.). No action required.
  </Card>
</CardGroup>

## 🔴 Direct cypher updates

### `mvp/yc/process-yc-people` (#12700)

<Note>
**Status:** ✅ **v2.6 live** on 2026-04-10 at 20:36 UTC. The YC → Known MENTOR ingest is actually writing edges to FalkorDB now — confirmed with a direct cypher run in the sandbox browser that created both the FOUNDED and MENTOR edges for Bu Kinoshita / Diana Hu / Resend. **Earlier v2.5 claim of "working end-to-end" was wrong** — v2.5 was producing zero FOUNDED and zero MENTOR edges because the MENTOR `ON CREATE SET` used `m.created_at = datetime()`, which is a Neo4j-only function that throws inside FalkorDB. `mvp/falkor/send-cypher` silently swallowed the error, `vector_logs` kept getting written because it runs AFTER `send-cypher`, and no crash log surfaced — the saga was hidden by the wrapper swallowing the real error. v2.6 fixes this by replacing `datetime()` with `timestamp()` in both the multi-founder and single-founder MENTOR branches. Zeno Rocha's pre-existing `mvp/get-add/master-person` (#12553) crash is still isolated and out of scope.
</Note>

**Previous behavior (v1):** Created `MENTOR` edges from YC partners to batch founders during company enrichment. Used a cypher template with `description`, `company_uuid`, `company_name` properties only. No lifecycle tracking, no weight, no source/formality/domain.

**Current behavior (v2.1, published):** The embedded cypher template was replaced with the [v2 MENTOR cypher](/guides/known/schema-design/edges#v2-cypher-unified-known-aware-ingest) in both the multi-founder and single-founder branches. The v2 version adds `valid_from`, `valid_to`, `source`, `formality`, `domain`, `weight`, `created_by`, `created_at`, `invalidation_reason`, `invalidated_by` fields with YC-appropriate defaults.

**Implementation details:**
- MERGE key is now `(mentor, mentee, company_uuid, valid_from)` for idempotent re-runs across time windows
- `valid_from` is stored as an **ISO string** (not `datetime()`) derived from the company's founded year as `"YYYY-01-01T00:00:00Z"`; falls back to `"1970-01-01T00:00:00Z"` if the founded year is unknown
- `ON CREATE SET` populates all v2 fields with YC-appropriate defaults (`source='inferred'`, `formality='formal'`, `domain='accelerator'`, `weight=14`)
- `ON MATCH SET` only updates the mutable display fields (`description`, `company_name`) on re-runs; lifecycle state is immutable after creation
- **No Xano stack helper vars** — all substitution logic is inlined in the Twig template using `??` for defensive null handling

**No migration needed.** Known is being rolled out on a **fresh graph** — no existing v1 MENTOR edges exist in production. The v1 → v2 migration cypher documented in [Schema Design → MENTOR → Migration](/guides/known/schema-design/edges#migration-v1-v2) is retained as a reference for the pattern, but was not part of the Known rollout path.

**Testing (2026-04-10):** Smoke tested against resend.com (master_company_id=30) in a wiped sandbox. Result: the function completes, the YC partner and all resolvable founders get real v2 MENTOR edges with every field populated, and the one problematic founder (Zeno Rocha) is logged and skipped via the per-founder `try_catch`. `valid_from` is written as an ISO string (not `datetime()`), and re-running the function produces `ON MATCH` behavior (no duplicate edges). The only outstanding failure is the pre-existing #12553 bug — see the changelog below for the full debugging history.

**v2 → v2.1 → v2.2 → v2.3 → v2.4 → v2.5 → v2.6 changelog:**

**v2 (published 2026-04-10 18:11 UTC, then rolled forward by v2.1):**
- Added three stack helpers (`$foundedYearStr`, `$validFromIso`, `$companyNameJson`) that invoked XanoScript filters (`|is_empty`, `|to_string`, `|json_encode`) on `$masterCompany.founded`. Crashed with `"Text filter requires an int"` when `founded` was null.

**v2.1 (published 2026-04-10 18:22 UTC):**
- Removed all three helpers. Moved the valid_from derivation inside the `util.template_engine` block using Twig's `??` operator: `"{{$var.masterCompany.founded ?? 1970}}-01-01T00:00:00Z"`. Twig handles null and int gracefully.
- MERGE-key `valid_from` changed from `datetime()` to plain ISO string. Safer for key-matching and consistent with how Orbiter's existing ontology stores `WORKS_AT.valid_from` as a string.
- **Still crashed** on the sandbox re-run against resend.com (master_company_id=30). The `Text filter requires an integer...` error persisted, but the parent `run-base-company-process` catch block swallowed it without any section-level context. No actionable stack trace.

**v2.2 (published 2026-04-10 18:51 UTC) — diagnostic instrumentation, no cypher changes:**
- Wrapped each section of `mvp/yc/process-yc-people` (Sections 2–7) in its own `try_catch` with `log_crash` writes.
- Each `log_crash` write includes `master_company_id` populated (addresses the meta-point that every crash log row should carry a traceable entity id), a `function_name` of `mvp/yc/process-yc-people` (no more hiding under the parent's name), and a `note` field tagged with `"v2.2 SECTION N"` plus a short description of what that section does.
- The `data` field captures relevant context at the point of failure: founder counts, YC partner name, whether `founded` is null, etc.
- Section 4 (the founder master-person resolve loop) and Section 5 (the YC partner resolve) have an additional inner `try_catch` around the `mvp/get-add/master-person` call so each founder's crash is logged individually with their name.
- The v2 MENTOR cypher in Section 7 is unchanged from v2.1.
- **Result:** Crash localized to Section 4 (founder resolve loop) on founder **Zeno Rocha**. Log row had `function_name = "mvp/yc/process-yc-people"`, `note = "v2.2 SECTION 4: founder master-person resolve loop"`, and the data blob showed the founder's exact input payload at the moment of failure.

**v2.3 (published 2026-04-10 19:04 UTC) — conditional email push:**
- Diagnosis of v2.2 log: Section 4 was building the `other_profiles` list with `email: []|push:(null)|filter_empty:""`. The `filter_empty` filter only strips empty strings — it does **not** strip `null`. Result: `other_profiles` contained `[null]` which failed `trim` downstream.
- Fix: Changed the email push to be conditional — only push `email` into the list if the value is non-empty, using a Twig `{% if %}` guard in the template.
- **Still crashed.** Same symptom on Zeno. v2.3 instrumentation showed the email field was now clean, but the same `"Text filter requires..."` error hit on a different field.

**v2.4 (published 2026-04-10 19:12 UTC) — conditional link pushes:**
- Diagnosis: the same filter_empty gap existed on the three link fields (`linkedin`, `crunchbase`, `x_url`). Each was pushed unconditionally and nulls were slipping through.
- Fix: Made each of the three link pushes independently conditional, same pattern as v2.3 email fix.
- **Still crashed.** New instrumentation confirmed the `other_profiles` list was now fully clean: `other_profiles_ct: 3` (email + 2 valid URLs), no nulls. But the crash persisted inside `mvp/get-add/master-person` (#12553) itself.

**v2.5 (published 2026-04-10 19:21 UTC) — scalar null coercion:**
- Diagnosis: Section 4 was passing nullable text scalars (`first_name`, `last_name`, `name`, `headline`, `description`, `location`, `image_url`) directly to `mvp/get-add/master-person`. The downstream function applies `|trim` and similar text filters on its inputs, which crash on `null`.
- Fix: Every nullable text input to `mvp/get-add/master-person` is now coerced to `""` before the call using `?? ""`. `user_id` (nullable int) is coerced to `0`.
- **Result:** Zeno Rocha still crashes. Instrumentation now shows the input to `mvp/get-add/master-person` is completely clean — every text scalar is either a real string or `""`, every int is a real int or `0`, `other_profiles` is `[]` or a list of non-null values. The `"Text filter requires..."` error persists **inside** `mvp/get-add/master-person` (#12553) on some condition we can't trigger from the outside by scrubbing inputs alone.

**Decision (2026-04-10 19:25 UTC) — accept partial success:**
- The pre-existing `mvp/get-add/master-person` (#12553) has a bug that triggers on Zeno Rocha's specific combination of inputs. That bug is **out of scope for Known** — it existed before the Known work started, has nothing to do with MENTOR edges or the Known schema, and happens inside a function that Known doesn't own.
- The v2.5 inner `try_catch` around the `mvp/get-add/master-person` call in Section 4 **isolates the failure per founder**: Zeno's row logs a crash to `log_crash` with full context (`master_company_id`, `function_name = "mvp/yc/process-yc-people"`, `note = "v2.5 SECTION 4: founder master-person resolve loop"`, and the founder name in the data blob) and the loop continues. All other founders and the YC partner resolve in Section 5 complete successfully, and the v2 MENTOR cypher in Section 7 writes real Known-aware edges for every resolved pair.
- **Net effect (claimed at the time):** YC → Known MENTOR ingest is working end-to-end for 99% of YC companies. The only remaining failure mode is specific founders whose input payload triggers the pre-existing #12553 bug; those founders are logged, skipped, and recoverable later if #12553 is fixed.
- **Net effect (actual reality, discovered in v2.6 retest):** The v2.5 decision was wrong. The MENTOR cypher was producing **ZERO edges** for every YC company, not just Zeno. The failure was being masked by Section 7's wrapper — see v2.6 below.

**v2.6 (published 2026-04-10 20:36 UTC) — the real fix:**
- Retest of v2.5 against a wiped sandbox running resend.com showed exactly **4 FOUNDED edges** in the graph total, and **0 MENTOR edges**. The 4 FOUNDED edges were all from YC partners' or angels' own work-history enrichment (Diana Hu → Escher Reality, Paul Copplestone → Supabase, etc.) — none of them were from Section 7 of `mvp/yc/process-yc-people`.
- Diagnosis: Section 7's cypher template contained `m.created_at = datetime()` in both the multi-founder and single-founder MENTOR branches. `datetime()` is a Neo4j function, **not supported by FalkorDB**. When `mvp/falkor/send-cypher` executed the cypher, FalkorDB rejected it with a "date/time format error", but the wrapper swallowed the error, `vector_logs` still got written (because that `db.add` runs AFTER `function.run "mvp/falkor/send-cypher"`), and no `log_crash` row surfaced. The saga had been silently zero-output from v2.1 all the way through v2.5.
- Root-cause proof: ran the exact Section 7 cypher manually in the FalkorDB sandbox browser with `datetime()` → immediate "date/time format error". Changed to `timestamp()` → cypher succeeded and created both the FOUNDED edge (Bu → Resend) and the MENTOR edge (Diana Hu → Bu) with all v2 Known fields populated.
- Fix: replaced `m.created_at = datetime()` with `m.created_at = timestamp()` in both MENTOR MERGE branches. Nothing else changed. Published as v2.6 at 20:36 UTC.
- Also audited all other Known cyphers in the Mintlify docs and found 7 more `datetime()` / `duration()` usages in not-yet-live cypher blocks (`create_facet` helper, `release_stale_merge_locks` janitor, identity-merge `MergeEvent` creation, identity-merge tombstone, residency overlap query, and the duplicate stale-lock query in `scope/identity-merge.mdx`). All fixed to use `timestamp()` + integer math in the same commit.
- Added a **FalkorDB compatibility section** to the top of [guides/cyphers/known](/guides/cyphers/known#falkordb-compatibility-read-this-before-writing-cypher) documenting which Neo4j temporal functions are forbidden and what to use instead. Future Known cyphers must follow the convention.
- **What the v2.5 saga really was, in retrospect:** every attempt from v2.1 through v2.5 was fixing the wrong bug. v2.1–v2.5 addressed real issues in Section 4 (the founder master-person resolve loop) — null scalars, empty lists, filter_empty gaps — and the fixes were correct. But Section 7 had been broken from the moment v2 shipped, and no telemetry surfaced it because the wrapper ate the error. Testing against a fully-wiped sandbox + direct cypher inspection in the FalkorDB browser was what finally exposed it.

**Net effect (v2.6):** YC → Known MENTOR ingest is **actually writing edges now** — FOUNDED for every resolvable founder, MENTOR for every YC-partner → founder pair, with full v2 Known-aware schema populated. Zeno Rocha still fails inside pre-existing #12553 and is still out of scope. Manual sandbox cypher run confirmed v2.6 works. Awaiting one more full-run retest to see the edges appear via the real function path rather than manual cypher.

## 🟡 Trigger hooks needed (no cypher changes)

These functions write to edges Known reads from. Known's lifecycle and cache-refresh crons need to be triggered when these functions run, but the functions themselves don't change.

### `mvp/edges/create-work-edges` (#2800)

**Current behavior:** Creates `FOUNDED`, `WORKS_AT`, `BOARD_MEMBER_OF`, `ADVISOR_TO`, `INVESTED_IN (Person)`, and `MEMBER_OF` edges for a master_person based on their work history.

**Known dependency:** `PROFESSIONAL_TIE` in Known is inferred from `WORKS_AT` overlap between two persons at the same company. When this function creates, updates, or invalidates a `WORKS_AT` edge, Known needs to:

1. Fire [`professional_tie_maintainer`](/guides/known/scope/lifecycle#known-lifecycle-professional-tie-maintainer-trigger) to re-evaluate all `PROFESSIONAL_TIE` edges involving the affected person
2. Fire [`cache/invalidate_on_upstream_change`](/guides/known/scope/cache#known-cache-invalidate-on-upstream-change) to mark dependent `shared_context_cache.employments` entries stale

**Implementation approach:** Add trigger calls at the end of `create-work-edges` (after cyphers are executed). Alternatively, use a deferred queue — mark affected KNOWS edges as "needs recompute" in a Xano table, and let the nightly cron pick them up.

**No cypher changes required in this function.** The existing cyphers that create `WORKS_AT` etc. are unchanged.

### `mvp/edges/create-education-edges` (#1898)

**Current behavior:** Creates `ATTENDED`, `ATTENDED_TOGETHER`, `STUDIED_UNDER` edges for a master_person's education history.

**Known dependency:** `shared_context_cache.educations` on KNOWS reads from `ATTENDED` edges and computes overlap between two persons. When this function creates or updates an `ATTENDED` edge, Known needs to fire `cache/invalidate_on_upstream_change` for all KNOWS edges touching the affected person.

**Implementation approach:** Same as `create-work-edges` — append trigger calls at end.

**No cypher changes required.**

### `mvp/node/update-person-node` (#2758)

**Current behavior:** Syncs `master_person` properties + embedding to the FalkorDB `Person` node. Derives graph labels (e.g. `Angel`) from `is_angel` bool on `master_person`.

**Known dependency:** Known adds new properties to the `Person` node: `deceased_at`, `merged_into`, `deleted_at`, `tombstone`. When these fields change on `master_person`, the graph node must be synced.

**Required change:** Extend the cypher in this function to also set the new Known-specific properties when they're non-null. See [Cyphers → Known → `update-person-node` extension for Known properties](/guides/cyphers/known#identity-merge-update-person-node-extension-for-known-properties).

**Additionally:** when `deceased_at` is first set on a person, fire the [`deceased_freezer`](/guides/known/scope/lifecycle#known-lifecycle-deceased-freezer-trigger) cron to cascade death handling across all KNOWS edges touching this person.

**Impact:** Low. One SET clause addition, one trigger fire on deceased_at set.

### `mvp/funding/add-all-fundable-deals` (#12703) and `mvp/funding/add-fundable-deal-node` (#12701)

**Current behavior:** Creates funding round nodes and `INVESTED_IN (Funding)`, `LEAD_INVESTED_IN`, `FOLLOW_ON_INVESTED_IN`, `RAISED`, `INVESTMENT_PARTNER_IN`, `INVESTMENT_PARTNER_AT` edges from fundable deals.

**Known dependency:** Known's `PROFESSIONAL_TIE(relationship_role='investor_founder')` is inferred from the combination of investment relationships and founder relationships at the same company. Specifically: if person A has `INVESTED_IN` at company X, and person B has `FOUNDED` at company X, they should have a `PROFESSIONAL_TIE(investor_founder)`.

**Required change:** Same trigger pattern as `create-work-edges` — fire `professional_tie_maintainer` and `cache/invalidate_on_upstream_change` when funding edges are created or updated. The existing cyphers in these functions are unchanged.

**Impact:** Low. Trigger hook only.

### `mvp/expertise/resolve-person-expertise` (#12668) and `mvp/expertise/resolve-company-specialties` (#12746)

**Current behavior:** Create `HAS_EXPERTISE` and `SPECIALIZES_IN` edges via vector-matching + MERGE.

**Known dependency:** None currently. Known does not use expertise/specialty edges in its scoring or facet derivation. If a future version of Known wants to use "shared expertise domain" as a bonding signal, these functions would gain a trigger hook.

**Required change:** None for the initial Known build.

### Nylas webhook pipeline (log-parse functions)

**Current behavior:** `message.created`, `event.created`, `event.updated`, `parse-from`, `parse-to`, `parse-cc` handlers ingest email and calendar activity into Xano tables.

**Known dependency:** Known's [ingest functions](/guides/known/scope/ingest) (`email_direct`, `calendar_meeting`, `sms_chat`) read from these activity tables to extract evidence rows and create/update `KNOWS` edges.

**Required change:** At the end of each Nylas handler, call the appropriate Known ingest function with the new activity data. This is an **append operation**, not a cypher replacement — the existing Nylas handlers remain as the data source, and Known's ingest becomes a downstream consumer.

**Impact:** Medium. Multiple handlers, all append-only changes. See [scope/ingest](/guides/known/scope/ingest) for the full pattern.

### `mvp/imdb/add-imdb-to-master-person` (#10509) and related IMDB functions

**Current behavior:** Enrich a master_person with IMDB data, create film/TV nodes and credits.

**Known dependency:** None. Known does not currently model film/TV relationships. IMDB collaborators are not represented in Known's facet model.

**Required change:** None.

### `mvp/node/sync-domain-expertise-nodes` (#12663)

**Current behavior:** Syncs `domain_expertise` and `sub_domain_expertise` table rows to FalkorDB nodes.

**Known dependency:** None.

**Required change:** None.

## ⚪ Legacy / deprecated surfaces that Known replaces

These functions are **deprecated** and should NOT have their cyphers updated. Known provides the replacement path; existing callers should be migrated to the Known equivalents, and these functions can be removed once migration is complete.

### `mvp/merge/master-merge` (#12736) — deprecated

**Current behavior:** Was a placeholder for merging `master_person` records that share an exact identifier. Marked as deprecated — not in active use.

**Known replacement:** [Identity Merge](/guides/known/scope/identity-merge) — `known/merge/execute_merge` and the full merge algorithm. Known's merge is **greenfield**, not an extension of this function. Do NOT treat `master-merge` as an implementation target.

### `mvp/merge/detect-suspects` (#12709) — uses deprecated flow

**Current behavior:** Detects suspected duplicate contacts using embedding similarity + LLM, writes to the `merge_suspect` table, and calls the deprecated `master-merge` function for deterministic exact-match cases.

**Known replacement:** Known's [identity resolution trigger surface](/guides/known/scope/identity-merge#merge-trigger). The detect-suspects logic (embedding-based candidate detection) could be preserved or rewritten, but the downstream call to `master-merge` must be replaced with `known/merge/execute_merge` from Known.

**Decision needed:** keep `detect-suspects` as the identity resolver (feeding candidates into Known's merge) OR replace it entirely with a new Known-aware identity resolver. Deferred to implementation time.

### `merge/person` (#2269) — tagged "master merge"

**Current behavior:** Older merge/person function.

**Known replacement:** Same as above — fully superseded by `known/merge/execute_merge`. Deprecate once Known is live.

### `merge/mp-merge` (#159) — tagged "refactored" + "depreciated" (sic)

**Current behavior:** Already marked as deprecated in the Xano tags. Legacy hard-merge function.

**Known replacement:** `known/merge/execute_merge`. Safe to remove after Known is live.

### `mvp/archive/merge-founders-v2` (#4630)

**Current behavior:** Archive function for merging founder records. Path includes `/archive/`, indicating it's already archived.

**Known replacement:** None needed — the archive path is a signal this is already end-of-life.

### `mvp/dupe/merge-company-nodes` (#10505)

**Current behavior:** Deduplicates company nodes in the graph.

**Known scope:** **Out of scope** for Known's initial build. Known only merges `Person` nodes. Company deduplication remains the responsibility of this existing function (not deprecated, not updated — just unrelated to Known).

**Note:** If a future version of Known wants to handle company identity merges, the patterns from `known/merge/execute_merge` (audit log, tombstone, reparent edges) would transfer. Deferred.

### What needs to happen in order

**Starting from a fresh graph** — Known is being rolled out in a new environment where no existing v1 edges need backfill. The "Phase 0" migration step from earlier versions of this doc is not applicable here; it has been removed.

**Phase 1 — update the YC ingest cypher:**
