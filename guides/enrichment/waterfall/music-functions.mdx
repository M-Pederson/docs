## Foundation

<Card icon="gauge-high">

```
mb/rate-limit/acquire ‚Äî 13079
```

The global 1 req/s token gate, backed by the `mb_rate_limit` singleton (#736). If the last granted call was less than 1100 ms ago it sleeps 1 s, then stamps `last_call_at`. **Fail-open** ‚Äî if the gate itself errors, the call proceeds and the failure is logged, so a broken limiter never halts the pipeline.

</Card>

<Card icon="cloud-arrow-up">

```
mb/api/request ‚Äî 13080
```

Single choke-point for **every** MusicBrainz call. Calls `acquire` for pacing, sends the canonical `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`), appends `fmt=json`, and retries up to 5 attempts with backoff sleeps between attempts (none after the final one). `400`/`404` are terminal ‚Äî no retry. Returns `{ok, terminal, status, attempts, json}`; failures `log_crash` with phases `api_bad_request` / `api_exhausted`.

</Card>

## Queue upserts

<Card icon="wrench">

```
mb/queue/upsert-artist ‚Äî 13081
mb/queue/upsert-recording ‚Äî 13082
mb/queue/upsert-label ‚Äî 13083
mb/queue/upsert-event ‚Äî 13084
mb/queue/upsert-release-group ‚Äî 13085
mb/queue/upsert-work ‚Äî 13086
```

Single source of truth for queue writes, keyed on `mbid`. Duplicate discovery ‚Üí `count++`, **lowest tier wins**, and the source-table FK is backfilled once the stub exists.

Inputs default via `first_notnull` (tier 4 / depth 1) so an explicitly passed `0` survives ‚Äî this is what lets bootstrap-lane rows (`priority_tier = 0`, `cascade_depth = 0`) be queued without the defaults coercing them.

## Orchestrators

Every phase is wrapped in `try_catch` ‚Üí `log_crash` (phases `mb_lookup` / `mb_node` / `mb_edges` / `mb_browse` / `mb_cascade`). A terminal `404`/`400` marks the source row `gone`. All edges stamp `e.uuid` (plus the full [standard envelope](/guides/enrichment/waterfall/music-edges#standard-edge-envelope)). Cascading is **inlined** in each orchestrator's `mb_cascade` phase ‚Äî the standalone `cascade-from-*` helpers from the design were not built.

<Card icon="layer-group">

```
mb/run-base-label-enrich ‚Äî 13091
```

Lookup `label/{mbid}?inc=url-rels+label-rels+ratings+tags+genres` ‚Üí `Company:Music_Label` node ‚Üí `SUBSIDIARY_OF` (w15, stub-MERGEs the other label). Then browses `release?label=`, capped at **3 pages / 300 releases** (`inc=artist-credits+release-groups`), queueing release-groups T2, credited artists T2, and ownership labels T1.

</Card>

<Card icon="layer-group">

```
mb/run-base-artist-enrich ‚Äî 13094
```

Lookup `artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels+event-rels` ‚Üí artist/group node ‚Üí `MEMBER_OF` edges (w5 current / w12 past, with `instruments`, `begin`/`end`/`ended`, `original`). Cascade: members T1, Albums capped at 20 chronological T2, recordings capped at 50 T2, and (v1.1, 2026-06-12) **main-performer events T3 ‚Äî queued only when the artist itself is a depth-0 seed** (the events enter the queue as depth-1 children, so their own artist fanout stops at depth 2, per the depth model). Event discovery rides the same lookup: `inc=event-rels` embeds the full event objects, so no extra browse call is needed. Collaborators are still deferred to v2. **v1.3 (2026-06-14):** a depth-0 `Person:Music_Artist` is enqueued for [person-resolution](#person-resolution) via `mb/queue/upsert-person-resolution` (13122) ‚Äî the same depth-0 gate as the event cascade, so only seed artists run it. **v1.4 (2026-06-15):** those same depth-0 Person rows now also get their serper avatar **inline at ingest** via `fetch-person-avatar` (13120, non-fatal) ‚Äî previously the avatar only came from the person-resolution drain, which ships inactive, so directly-seeded artists came in avatar-less.

</Card>

(The artist-credit `PERFORMED` handler already split by type; the rel-credit handler didn't ‚Äî caught on Nirvana, see the [sandbox test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run).)

## Unified queue drain

<Card icon="wrench">

```
mb/queue/process-next ‚Äî 13102
```

Picks the **single best row across all 6 queues** ordered by `priority_tier ASC, cascade_depth ASC, entity_rank ASC, id ASC` ‚Äî `entity_rank` is a constant per queue (artist 1, label 2, **event 3**, work 4, release_group 5, recording 6) ‚Äî skipping rows whose `next_attempt_at` is in the future. Locks the row, dispatches the matching orchestrator, deletes on success. v1.1 (2026-06-12) wired `queue_mb_event` into the scan, the candidate ranking, and dispatch (‚Üí 13105), filling the rank-3 slot reserved at v1.

On failure: `retry_count++` with a `next_attempt_at` backoff of 5 min √ó retries; the row is dropped after 5 failures.

</Card>

## LLM edge descriptions

Mirrors the IMDB `WORKED_ON` trio (#13016‚Äì#13018), but **one generic trio covers all music edge types** ‚Äî keyed on `e.uuid` rather than one trio per edge type. Model: **`deepseek/deepseek-v4-flash`** (temp 0.3, `reasoning:{enabled:false}` + `max_tokens:200`), producing a single factual 15‚Äì30-word sentence. All descriptions obey **[GOOD-or-null](/guides/dev-updates/2026-06-16-music-node-descriptions#good-or-null-invariant)** ‚Äî a clean sentence or `null`, never a refusal/meta echo.

<Card icon="cloud-arrow-up">

```
mb/edge/create-music-edge-description ‚Äî 13095  (v1.4)
```

Given one edge's structured properties (opaque `via_release` mbid stripped) + endpoint node names, calls **`deepseek-v4-flash`** (reasoning off + `max_tokens:200`) and returns the single-sentence description **after cleanup**: strip wrapping quotes + leading meta-preambles, normalize U+2011/U+2010 ‚Üí `-`, and a GOOD-or-null reject guard (refusal/meta ‚Üí empty). **v1.4 (2026-06-17)** swapped the model from `gpt-oss-120b`, which dropped spaces ~15% of the time ("Stevensis", "Themusicgroup"). Sandbox sample: *"Sufjan Stevens is a member of the music group Danielson."*

</Card>

<Card icon="wrench">

```
mb/edge/write-music-edge-description ‚Äî 13096
```

Stamps `e.description` + `e.description_generated_at` (ISO) onto the edge via Cypher, matched on `e.uuid`.

</Card>

<Card icon="clock-rotate-left">

```
mb/tool/backfill-music-edge-descriptions ‚Äî 13097  (v1.6)
```

Batch drainer (1‚Äì100 per run, default 10). Selects edges where `description_generated_at IS NULL`, filtered by edge type (allowlist ‚Äî `ON_RECORD_LABEL` added v1.3, **`OWNED_BY` added v1.5** by the [label-owner](/guides/enrichment/waterfall/music-reconciliation#build-spec--label-owner-resolution-owned_by) session) + `labels()` membership, and runs the create ‚Üí write pair per edge. Idempotent via the timestamp guard. **`mode:"reset"` (v1.4)** nulls already-stamped music-edge descriptions (scoped to the `X-Data-Source` graph) so a normal drain re-describes them ‚Äî used to swap the whole graph from gpt-oss to deepseek output. **v1.6 (2026-06-17) perf:** the SELECT now anchors on the relationship types (`MATCH (a)-[e:T1|T2|‚Ä¶]->(b)`) so FalkorDB scans only the music-edge matrices, not the entire people+company edge set ‚Äî cut a live batch SELECT from >280 s (timeout) to ~3 s.

</Card>

## Cover art & group images

Built 2026-06-12 (second session). Two parallel image pipelines hang media off the source tables: **recording cover art** from the [Cover Art Archive](https://coverartarchive.org) and **group photos** from a serper.dev Google Images search. Both store the original URL *and* a re-hosted GCS WebP URL ‚Äî `cover_art_url` / `gcp_cover_art_url` on `mb_recording` (#738) and `music_group_image_url` / `gcp_music_group_image_url` on `mb_artist` (#737). Neither is cron-wired ‚Äî they run via the [operator endpoints](#operator-endpoints) below.

<Card icon="wrench">

```
mvp/music/save-cover-art ‚Äî 13104
mvp/music/save-music-group-image ‚Äî 13107
```

GCS converters (the IMDB poster-saver pattern, #13044). Each regex-extracts the **MusicBrainz MBID (UUID)** from its identifier input (`recording_identifier` / `music_group_identifier` ‚Äî callers pass the bare `mbid`), converts the source image to WebP via `mvp/avatar/convert-image-url-to-webp` (12756), and uploads via `mvp/google-storage/upload-file` (12725) to `music-cover-art/<mbid>.webp` / `music-group-image/<mbid>.webp`. Returns the public GCS URL.

</Card>

<Card icon="cloud-arrow-up">

```
mvp/music/fetch-recording-cover-art ‚Äî 13108
```

One recording, end-to-end: skip if `gcp_cover_art_url` already set ‚Üí **1 throttled MB lookup** `recording/{mbid}?inc=releases` ‚Üí Cover Art Archive JSON (`coverartarchive.org/release/{id}`) for up to 5 releases, `Official` first ‚Üí saves `cover_art_url` (500px thumbnail preferred, `https`-normalized) ‚Üí 13104 ‚Üí saves `gcp_cover_art_url`. **v1.3 (2026-06-14):** after stamping the GCS URL it writes `SET n.image_url` on the `Recording` graph node via `send-cypher` (non-fatal; the `already_done` skip path *also* self-heals the node, so re-driving an imaged row catches it up). `no_art` ‚Üí `SET n.image_url = null` on the node and the row is re-checked on later runs.

</Card>

<Card icon="cloud-arrow-up">

```
mvp/music/fetch-group-image ‚Äî 13109
```

One group, end-to-end (**groups only** ‚Äî `artist_type = Person` rows are skipped): skip if `gcp_music_group_image_url` set ‚Üí serper.dev images search (`q = "<name> band <disambiguation>"`, key from env `SERPER_DEV`, same call shape as `mvp/enrich/find-avatar` 12846) ‚Üí saves `music_group_image_url` (first result) ‚Üí 13107 ‚Üí saves `gcp_music_group_image_url`. **v1.3 (2026-06-14):** writes `SET n.image_url` on the `Music_Group` graph node after the GCS stamp (non-fatal; self-heals on the `already_done` path); `no_image` ‚Üí `SET n.image_url = null`. `serper_error` is transient and leaves the node untouched for retry.

## Node descriptions

Built 2026-06-12 (second session). Every **`Recording`**, **`Music_Group`**, and **`Music_Artist`** node gets a `description` ‚Äî and its `name_embedding` is generated **from that description** (overwriting the name-based embedding the artist node writer stamps at ingest). All descriptions obey the **[GOOD-or-null invariant](/guides/dev-updates/2026-06-16-music-node-descriptions#good-or-null-invariant)**: a clean ‚â§127-char factual sentence beginning with the entity name, or exactly `null` ‚Äî never an LLM refusal/meta-commentary echo (which would poison the embedding).

<Card icon="cloud-arrow-up">

```
mvp/music/create-node-description ‚Äî 13112  (v2.2)
```

Generate (don't write) the LLM description for one node from a **slim, type-aware field-set** (not raw JSON ‚Äî raw-JSON@8000 was what made deepseek echo *"please provide the record"*). One OpenRouter call ‚Äî **`deepseek/deepseek-v4-flash`**, temp 0.2, **`reasoning: {enabled: false}` + `max_tokens: 200`** ‚Äî producing **one factual sentence ‚â§127 chars** that **begins with the entity name**, so the description-based embedding keeps the name signal. The field-set is passed as a JSON *user* message built in a lambda (not `util.template_engine` ‚Äî avoids the markdown-autolinker URL burn from the IMDB build). **Two GOOD-or-null guards (v2.2):** an **input-empty short-circuit** returns `null` before the LLM is called on a missing/empty record, and an **output reject-guard** strips quotes, hard-truncates to 127, and maps any refusal/meta-commentary (first-person + `the user`/`provide`/`structured`/`musicbrainz` markers) to `null`. The guard lives in the lambdas, **not** the prompt.

<Warning>
The reasoning flag is load-bearing: deepseek-v4-flash is a reasoning model, and without it the completion budget goes to thinking tokens and `message.content` comes back **empty** on larger records ‚Äî found live during the Cinq smoke test (3 of 4 rows silently skipped) and fixed the same day. CLAUDE.md trap #13.
</Warning>

</Card>

<Card icon="clock-rotate-left">

```
mvp/music/backfill-node-descriptions ‚Äî 13113
```

Batch drainer (default 10, clamped 1‚Äì50; `kind` = `recording` / `music_group` / empty for both). **Source-table-driven selection** (stub-proof ‚Äî graph-side `description IS NULL` would loop forever on un-enriched stub nodes): rows with `enrich_success`, a `node_uuid`, and `description_generated_at IS NULL`. Per node: use MB `annotation`/`disambiguation` when ‚â• 40 chars (as `"<name> ‚Äî <text>"`), else 13112 ‚Üí embed the **final** description via **both** `vectors/create-vectors-string` (#4676 ‚Üí `name_embedding`, 1536) **and** `vectors/create-gemini-vectors` (#13114 ‚Üí `embeddings`, 3072) ‚Üí one Cypher `SET n.description, n.description_generated_at (ISO), n.updated_at, n.name_embedding [, n.embeddings]` ‚Üí stamp the source row. `send-cypher` string returns are thrown as `CYPHER_ERROR`; failures log phase `node_description`. **v1.2 (2026-06-14):** dual-writes the Gemini `embeddings`; both fns surface as a JS array in lambda context, so each is `JSON.stringify`'d into `vecf32(...)` (the old `String(...).startsWith('[')` check was removed ‚Äî it would now throw on every row).

</Card>

## Node embeddings

Built 2026-06-14. Dual-write of the Gemini `embeddings` vector (and OpenAI `name_embedding` where missing) so every music node carries **both** vectors during the OpenAI ‚Üí Gemini migration. The four ingest-time writers (13090, 13092, 13113) handle the forward path for `Company:Music_Label` / `Music_Artist` / `Music_Group` / `Recording`; this backfill catches up existing nodes and the five types with no ingest-time embedding.

<Card icon="clock-rotate-left">

```
mvp/music/backfill-node-embeddings ‚Äî 13115
```

Graph-driven, idempotent (gated on `embeddings IS NULL`). Selects `:Entity` nodes carrying one of the 9 music labels (`label` input optionally restricts to one), builds embed text ‚Äî the node `description` for `Recording`/`Music_Group`, else `name + disambiguation` ‚Äî then computes `embeddings` via #13114 (always) and `name_embedding` via #4676 (only when the node lacks it, so it never clobbers the description-based one on `Recording`/`Music_Group`). One `SET n.embeddings [, n.name_embedding]` Cypher per node, matched on `uuid`. Both embedding fns return a JS array in lambda context ‚Üí `JSON.stringify`'d into `vecf32(...)`. Per-row `try_catch` ‚Üí `log_crash` phase `node_description` (`node_embedding` is not in the strict phase enum). `Genre` is intentionally excluded because genres are properties, not nodes. **No vector index is created** ‚Äî the `:Entity(embeddings)` 3072 index is deferred to OpenAI deprecation.

</Card>

## ON_RECORD_LABEL edges

Built 2026-06-17. A **graph-derived** backfill that materializes the [`ON_RECORD_LABEL`](/guides/enrichment/waterfall/music-edges#structural-music-hub-aware) edge (`(Recording)‚Üí(Company:Music_Label)`, weight **55**) ‚Äî a one-hop convenience shortcut of the existing `(Recording)-[:APPEARS_ON]->(Release)-[:RELEASED_ON]->(Company:Music_Label)` path. **No MusicBrainz fetch** ‚Äî the recording lookup (#13100) never pulls label/release data, so the edge is derived purely in-graph from the canonical release's label (only canonical releases materialize as `Release` nodes, so the path *is* the canonical-release label).

**Forward-wired (2026-06-17).** No longer backfill-only: the release-group orchestrator `mb/run-base-release-group-enrich` (**#13099 v1.2**) calls #13131 with the new `release_mbid` scope right after writing the canonical Release + `RELEASED_ON` + `APPEARS_ON` edges ‚Äî guarded on a non-empty canonical release mbid, in its own non-fatal `try_catch` ‚Äî so every newly-enriched release-group **auto-derives** `ON_RECORD_LABEL` for its tracks.

<Card icon="clock-rotate-left">

```
mvp/music/backfill-on-record-label-edges ‚Äî 13131  (v1.1)
```

Single idempotent Cypher via `mvp/falkor/send-cypher`: `MATCH (rec:Recording)-[:APPEARS_ON]->(rel:Release)-[ro:RELEASED_ON]->(lbl:Company:Music_Label) MERGE (rec)-[r:ON_RECORD_LABEL]->(lbl)` ‚Äî `ON CREATE SET` stamps `uuid` + `created_at`; every run sets `weight = 55`, `updated_at`, `via_release = rel.mbid`, and `catalog_number = ro.catalog_number` (carried from the canonical release's `RELEASED_ON`). `MERGE` per `(recording, label)` pair ‚Üí idempotent, no duplicates. Optional scopes: `recording_mbid` (one recording) or **`release_mbid` (v1.1 ‚Äî all recordings on one release, used by the #13099 forward-wire)**; omit both for a full graph sweep. Operator endpoint `tool/backfill-on-record-label-edges` (#8688) forwards both. `description` is left `null` until the edge describer runs. **Sandbox-verified 2026-06-17:** full sweep created 12 edges (weight 55, `via_release`/`catalog_number` present), re-run produced 0 duplicates, scoped mode returned 1 for a labeled recording and 0 for one with no label path.

</Card>

## Person-resolution

Built 2026-06-14. The person-side mirror of the label‚ÜíCompany bridge ‚Äî resolves a `Person:Music_Artist` node to a `master_person` and unlocks the social/people enrichment that MB artists otherwise miss (they carry no `linkedin_url`). A depth-0 `Person:Music_Artist` is enqueued at enrich time; a **standalone cron** drains the queue and, per artist, runs Exa social discovery cross-checked against the MB url-rels + aliases, mints/links a `master_person` carrying **all** verified socials as `master_link` rows (which auto-fire the twitter/instagram/youtube/linkedin scrapers + `social_insights` + influence), `SAME_AS`-bridges `Music_Artist ‚Üí master_person`, stamps `mb_artist.master_person_id`, annotates the node `social_links`, and sets a serper.dev avatar. Full writeup: [dev update](/guides/dev-updates/2026-06-14-person-resolution). It builds the person half of [Music Reconciliation](/guides/enrichment/waterfall/music-reconciliation) **as a bridge, not a merge**.

```
mb/run-person-resolution ‚Äî 13121
```

The orchestrator. Builds a verification set from `mb_artist.json` (known social URLs from `relations[]`, aliases / legal names from `aliases[]`, area / birth-year / genres) ‚Üí calls Exa (13118) ‚Üí **cross-checks** candidates against the MB set to defeat same-name false positives ‚Üí mints/links a `master_person` via `mvp/get-add/master-person-new` (#13039) passing **all verified socials in `links`** (each written as a `master_link` row, auto-firing the platform scrapers) ‚Üí stamps `mb_artist.master_person_id` ‚Üí writes the `SAME_AS` bridge `Music_Artist -w100-> master_person Person` ‚Üí annotates the node (`SET n.social_links`, `n.social_resolved_at`) + fetches a serper avatar (13120). `hold_downstream = false` (run the full PDL‚ÜíEnrichLayer people pipeline) **only when a LinkedIn is verified**; `true` otherwise ‚Äî the social scrapers still fire from the link writes.

A LinkedIn is accepted **only by content** (its profile text shows a music-identity signal ‚Äî a music-role keyword, the artist's stage name, or a genre), which defeats same-name strangers (v2.2). **v2.3 (2026-06-14):** a `deceased` flag derived from the MB `life-span` (`ended == true` or an `end` date) **skips the LinkedIn-accept branch entirely** for dead artists ‚Äî a deceased musician cannot run a LinkedIn, so any hit is an impersonator / fan / namesake. Caught live on Kurt Cobain (died 1994) during the [sandbox node-type test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run); socials still flow, only LinkedIn is blocked. **v2.4 (2026-06-14):** if the url-rels include an `imdb.com/name/nm‚Ä¶` page, a dedicated `try_catch` phase calls `mvp/imdb/add-imdb-to-master-person` (#10509) **synchronously** (inherits the request data source ‚Äî avoids the async‚Üílive leak) to enrich the **existing** master_person with IMDb bio/credits/awards (no duplicate; adds a Firecrawl + LLM of latency per IMDb-carrying artist). Verified on Cobain ‚Äî `imdb_person` landed on sandbox, FK-linked, no dup, and the master_person's graph `Person` node gained the IMDb `roles` / `short_biography` / `long_biography` / `avatar` (credits queue as `imdb_title` rows). See the [handoff](/guides/dev-updates/2026-06-14-person-resolution).

## Deviations & deferrals

Differences between the [design](/guides/enrichment/waterfall/music-industry-waterfall) and what was built:

| Deviation / deferral | Detail |
|---|---|
| ~~**Events deferred**~~ **Events built 2026-06-12** | `mb/get-add/event` (13106) + `mb/run-base-event-enrich` (13105) now exist and the drain (13102 v1.1) fills the reserved `entity_rank = 3` slot. Event-specific deviations: events are discovered **only from depth-0 artists** (`inc=event-rels`, main-performer rels, T3 ‚Äî no event browse pagination); `guest performer` (a third live rel type) maps to `PERFORMED_AT` with `role`; event `genres`/`tags`/`rating` are written but empty in practice on MusicBrainz. Smoke-tested on Reading Festival 1992 (all four phases green: node + sub-label, `HEADLINED`, `HELD_AT` place inline, headliner cascade). |
| **Collaborators deferred to v2** | The artist orchestrator (13094) does not queue collaborators ‚Äî members, albums, recordings, and (v1.1) depth-0 main-performer events only. |
| **Image pipelines un-cron'd** | Cover art (13110) and group images (13111) run only via the operator endpoints; `no_art`/`no_image` rows stay null and are re-attempted on every run (each attempt costs external calls). |
| **Node descriptions overwrite `name_embedding`** | 13113 regenerates `name_embedding` from the description for `Recording`/`Music_Group` nodes, replacing the name+disambiguation embedding 13092 writes at ingest. LLM-empty rows stay unstamped and retry next run. |
| **Cascade helpers inlined** | The standalone `mb/cascade-from-*` functions were not built; each orchestrator carries its own `mb_cascade` phase. |
| **`FROM_AREA` country-only** | The area backfill (13103) maps areas at country granularity only; no city/region mapping in v1. |
| **Genre nodes retired** | Genres remain `genres` properties on music nodes for facets, badges, filters, and search text. Do not write `Entity:Genre` nodes or `HAS_GENRE` edges. |
| **Cron 60 s √ó 10 batch** | Design called for a ~2 s tick; Xano cron granularity is 60 s, so Task 180 drains up to 10 rows per tick instead ‚Äî approximately equivalent throughput. |
| **Recordings browse unsorted** | The artist cascade takes the **first 50** recordings as returned, not the top 50 by length/date. |
| **Label browse cap** | `release?label=` browse is capped at 3 pages / 300 releases; hitting the cap is logged. |
