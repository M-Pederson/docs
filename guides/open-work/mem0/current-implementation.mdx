This page documents what exists in Orbiter today for AI chat memory and context: the server-side OpenUI05 dispatch path (Zep-backed), the context blocks injected into each turn, the deterministic Xano context functions `mvp/context/get-all-user-context_v2` (`#13076`) and `mvp/meeting-prep/classify-participants-get-context` (`#13070`), the conversation tables, and an audit confirming there are no live Mem0 consumers yet.

For the target Mem0 architecture and the steps to get there, see [Migration Plan](/guides/open-work/mem0/migration-plan).

<Warning>
  **Update (June 10, 2026):** the Outcomes surface has migrated. API `POST /openui05/dispatch` (`#8497`) is now **Mem0-backed** in production — Mem0 V3 search fills the `[NETWORK MEMORY]` slot and a `claude-haiku-4-5` extractor writes durable memories at end of turn. The Zep description below is preserved as the pre-cutover reference: the Zep blocks remain in the endpoint, dormant behind `$env.MEM0_API_KEY == ""` gates (unset `MEM0_API_KEY` to roll back). The "no live Mem0 consumers" audit below predates the cutover. See the shipped-implementation callout in [Migration Plan](/guides/open-work/mem0/migration-plan#outcomes-ai-chat).
</Warning>

## Current Implementation Snapshot

The existing OpenUI AI chat path is already server-side. The frontend calls one Xano endpoint and does not call an LLM, Zep, or Mem0 directly.

| Item | Current state |
| --- | --- |
| Frontend entrypoint | API `POST /openui05/dispatch` (`#8497`) |
| Xano API | API `POST /openui05/dispatch` (`#8497`) in `Robert OpenUI 0.5` group `#1276` (`api:C5i2nPpF`) |
| Auth | `user` table auth (`#125`) |
| Memory provider today | Mem0 (since June 10, 2026; Zep dormant behind env gate) |
| Frontend inputs | `query`, stable `thread_id`, `suggestion_request_id`, and YAML context blocks |
| Server output | OpenUI response payload plus `mem_used` and `mem_ingested` flags |

API `POST /openui05/dispatch` (`#8497`) does the memory work inside Xano:

1. Builds a Zep user id as `user-{auth.id}`.
2. If `thread_id` and `$env.zep` exist, ensures the Zep user and thread exist.
3. Reads Zep thread context from `GET /api/v2/threads/{thread_id}/context`.
4. Builds `$mem_context` from frontend-provided YAML blocks plus Zep's `[NETWORK MEMORY]`.
5. Routes the turn through classification and the appropriate OpenUI tool endpoint.
6. Writes back to Zep with `POST /api/v2/threads/{thread_id}/messages`.

The current Zep write stores the user's raw query and an assistant route summary like `Routed to find_investors. ...`; it does not appear to store the full rendered OpenUI response as a Xano transcript inside endpoint `#8497`.

### Context Blocks

The frontend-provided YAML blocks are injected into `$mem_context` in this order when present:

1. `[YOU]` from `self_yaml_context`.
2. `[NETWORK MEMORY]` from Zep.
3. `[HELPING TARGET -- master_person #{for_master_person_id}]` from `target_yaml_context`.
4. `[YOUR RELATIONSHIP WITH TARGET]` from `target_relationship_summary`.
5. `[YOUR COMPANY]` from `self_company_yaml_context`.
6. `[TARGET COMPANY -- master_person #{for_master_person_id}'s current company]` from `target_company_yaml_context`.

### User Context Snapshot: function `mvp/context/get-all-user-context_v2` (`#13076`)

Use function `mvp/context/get-all-user-context_v2` (`#13076`) to create the user's frozen self-context at the start of a suggestion request draft.

<Warning>
  **Deprecated function:** `mvp/context/get-all-user-context` (`#13069`) is deprecated. Do not wire new flows to it; use function `mvp/context/get-all-user-context_v2` (`#13076`) instead.
</Warning>

Function contract:

```txt
function: mvp/context/get-all-user-context_v2
id: 13076
input: { user_id: auth.id }
response: markdown string
```

This function returns compact Markdown for the authenticated user. It includes person signals, experience, education, and current work context. It intentionally excludes noisy/internal context such as user IDs, master person IDs, raw YAML IDs, `bio_500`, and current company rows where `master_company_id` is null.

#### When to call it

Call function `mvp/context/get-all-user-context_v2` (`#13076`) once immediately after the `suggestion_request` draft row is created and before the first OpenUI chat turn.

Current table:

```txt
suggestion_request (#529)
- id
- user_id
- copilot_mode
- request_context
- master_person_id
- status
- llm_summary_context
- number_of_results
- deliver_format
```

Do not store the user context in `request_context`. Keep `request_context` for the user's actual ask. Store the generated Markdown snapshot in the JSON field `suggestion_request.llm_summary_context` under a namespaced key:

```json
{
  "openui_context": {
    "self_context_function_id": 13076,
    "self_context_function": "mvp/context/get-all-user-context_v2",
    "self_context_format": "markdown",
    "self_context_generated_at": "2026-06-08T00:00:00Z",
    "self_context_user_id": 15,
    "self_markdown": "# User Context\n\n## Robert Boulos\n\n..."
  }
}
```

Implementation sketch:

```txt
1. API `POST /suggestion-requests` (`#8130`) creates `suggestion_request` table `#529` with `status = draft`.
2. Run function `mvp/context/get-all-user-context_v2` (`#13076`) with `{ user_id: auth.id }`.
3. Edit the new suggestion_request row.
4. Merge the function result into llm_summary_context.openui_context.self_markdown.
5. Use the same stored snapshot for every chat turn on that suggestion_request.
```

If we later add `ai_chat_context_snapshot`, store the full Markdown there and keep only `context_snapshot_id` plus metadata in `llm_summary_context`. Until then, `llm_summary_context.openui_context.self_markdown` is the least-invasive storage location because the field already exists on `suggestion_request`.

#### How it feeds back into chat

Every call to API `POST /openui05/dispatch` (`#8497`) for the same `suggestion_request_id` should include the stored Markdown snapshot as `self_yaml_context`.

The input name is still `self_yaml_context` for backwards compatibility, but the value can be Markdown. API `POST /openui05/dispatch` (`#8497`) treats it as text and injects it under `[YOU]`.

Current flow:

```txt
suggestion_request.llm_summary_context.openui_context.self_markdown
  -> dispatch input self_yaml_context
  -> [YOU] block in $mem_context
  -> downstream OpenUI tool endpoint context
```

Preferred server-side cleanup: update API `POST /openui05/dispatch` (`#8497`) so that when `suggestion_request_id` is present and `self_yaml_context` is empty, it loads `suggestion_request.llm_summary_context.openui_context.self_markdown` itself. That keeps the frontend contract smaller and ensures Xano owns context assembly.

Do not rerun function `mvp/context/get-all-user-context_v2` (`#13076`) on every chat turn. Refresh it only when:

- creating the draft `suggestion_request`
- the user explicitly asks to refresh their context
- the product intentionally restarts the draft with a new context snapshot

Do not write this full context snapshot to Mem0. Mem0 should receive only extracted durable memories, not the whole profile/work context.

### Meeting Prep Context Builder: function `mvp/meeting-prep/classify-participants-get-context` (`#13070`)

Use function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) as the canonical meeting-prep participant context builder.

Function contract:

```txt
function: mvp/meeting-prep/classify-participants-get-context
id: 13070
input:
  user_email: email
  participant_emails: email[]
response: participant JSON with embedded Markdown context
```

`user_email` is not just a filter input. Function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) seeds `my_team` from `user_email` first, then adds any other same-domain attendees from `participant_emails`. This keeps one-on-one external meetings from returning `my_team: null` when the caller excludes the signed-in user from the attendee list.

Meeting Prep does not need function `mvp/context/get-all-user-context_v2` (`#13076`). Function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) is the Meeting Prep context builder, owns the user's team context through `my_team`, and is called by API `POST /meeting-prep-context/create` (`#8325`) during the normal prep-initiation path.

The function expands the older participant classification response with Markdown context fields:

| Location | Context field | Source helper |
| --- | --- | --- |
| `my_team` | `my_team_company_context` | function `mvp/context/get-master-company-markdown-context` (`#13072`) |
| `my_team.members[]` | `person_context` | function `mvp/context/get-master-person-markdown-context` (`#13071`) |
| `participating_companies[]` | `company_context` | function `mvp/context/get-master-company-markdown-context` (`#13072`) |
| `participating_companies[].members[]` | `person_context` | function `mvp/context/get-master-person-markdown-context` (`#13071`) |
| `personal_participants[]` | `person_context` | function `mvp/context/get-master-person-markdown-context` (`#13071`) |

#### When to call it

Call function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) only when the user chooses meeting prep for a calendar event.

Preferred lifecycle: keep Nylas webhook ingestion and calendar sync focused on calendar/event freshness. Do not run function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) for every incoming meeting by default. When the user actively opens or starts meeting prep, a prep-initiation path creates or loads the `meeting_prep_context`, runs function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) only if the stored participant package is missing or stale, and stores the packaged participant context there.

Use the stable meeting key that already exists in Xano:

```txt
meeting_prep_context.id        -> internal meeting prep context id
meeting_prep_context.nylas_event_id -> external calendar event id
```

Use `user_id + nylas_event_id` as the idempotency key. Reopening or refreshing prep for the same user's event updates the existing `meeting_prep_context` row instead of creating a duplicate.

If the product introduces a separate `meeting_id`, store that alongside `nylas_event_id`, but do not lose the Nylas event id because it is needed for webhook reconciliation.

Implementation sketch:

```txt
1. User selects "meeting prep" for a calendar event.
2. Frontend calls API `POST /meeting-prep-context/create` (`#8325`) with `nylas_event_id`, optional `desired_outcome`, and optional `force_refresh`, or the widget bridge detects a "Meeting with ...: {event title}" message in API `POST /conversations/{conversation_id}/messages` (`#8154`).
3. API creates or loads meeting_prep_context for user_id + nylas_event_id.
4. API loads the calendar event, user's calendar email, and attendee emails.
5. If participant_context exists and the source hash still matches the current attendee/organizer set, reuse it.
6. Otherwise, skip cancelled events and events with no useful participants.
7. Run function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) with `user_email` and `participant_emails`. `participant_emails` can exclude `user_email`; function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) still seeds `my_team` from `user_email`.
8. Store the full JSON result in meeting_prep_context.participant_context.
9. Derive a single meeting prep memo Markdown string from the embedded context fields.
10. Store the derived memo in meeting_prep_context.meeting_prep_memo_markdown.
11. Store lightweight metadata such as participant_context_generated_at, participant_context_source_hash, participant_context_source_event_updated_at, participant_context_function_id, and participant_context_status.
12. Use that stored memo for every meeting-prep chat turn for the same event.
```

Rerun function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) idempotently only when the user chooses meeting prep and the stored participant context is missing or stale because the event participant list, attendee emails, organizer, or linked company/person identities changed. Do not rerun it on every chat turn.

#### How the memo feeds chat

For meeting prep, the LLM should receive all participant Markdown, but as deterministic Xano context rather than Mem0 memory.

Build a single memo from:

```txt
# Meeting Prep Context

## My Team
{my_team.my_team_company_context}

### My Team Members
{my_team.members[].person_context}

## Participating Companies
{participating_companies[].company_context}

### External Participants
{participating_companies[].members[].person_context}

## Personal Participants
{personal_participants[].person_context}
```

Then pass the stored memo into the meeting-prep chat input before retrieved memories:

```txt
System instructions

Meeting prep memo Markdown from meeting_prep_context.meeting_prep_memo_markdown

Relevant durable user memories from Mem0

Relevant run memories from Mem0

Recent meeting-prep chat transcript

Latest user message
```

This gives the model the full participant context every turn without polluting Mem0 with large profile/company snapshots.

#### Mem0 processing

Do not POST the full function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) participant result, full meeting memo, or raw participant Markdown to Mem0.

Use Mem0 around meeting prep this way:

1. **Before the LLM turn:** search Mem0 with a query built from the latest user message, meeting title, desired outcome, and participant names.
2. **During the LLM turn:** include the function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) meeting memo as deterministic Xano context and the Mem0 search results as retrieved memory.
3. **After the assistant response:** run memory extraction asynchronously against the latest user message, assistant response, meeting flow state, and retrieved memories.
4. **Write to Mem0 only when extracted facts are durable:** relationship facts confirmed by the user, reusable meeting preferences, strong strategy decisions, or meeting-specific commitments.
5. **Use run scope for event-specific facts:** `run_id = "meeting_prep:{meeting_prep_context_id}"`.
6. **Use user scope for durable facts that should follow the user across meetings:** `user_id = "{auth.id}"`.

Recommended metadata for meeting prep writes:

```json
{
  "flow_type": "meeting_prep",
  "thread_id": "meeting_prep:88",
  "source_type": "meeting_prep_chat",
  "source_id": "ai_chat_message:456",
  "meeting_prep_context_id": "88",
  "nylas_event_id": "calendar-event-id",
  "category": "meeting_strategy"
}
```

3. If an active memory exists, updates that row's `value`, `category`, and `source`.
4. If no active memory exists, creates a new `copilot_memory` row.
5. Writes a create/update entry into `copilot_memory_log`.
6. Returns `{ memory, action }`.

Mem0 status: **no external Mem0 call**; this is local Xano memory storage.
