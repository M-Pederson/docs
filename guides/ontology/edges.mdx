**32 edge relationship types** across 10 logical groups.

Every edge in the Orbiter graph represents a typed relationship between two nodes. Edges carry properties including a **weight** (lower = more important), temporal bounds (`valid_from`/`valid_to`), and a `uuid` for tracking.

**Weight convention:** Lower numbers indicate stronger/more important relationships. This is consistent across the entire graph.

---

## Work Relationships

Edges connecting people to companies through employment and professional roles. Created by `mvp/edges/create-work-edges` (#2800).

### FOUNDED

A person founded or co-founded a company.

| | |
|---|---|
| **Pattern** | `(Person)-[FOUNDED]->(Company)` |
| **Weight** | **5** (highest importance) |
| **Created by** | `mvp/edges/create-work-edges` (#2800) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `title` | string | Founder title (e.g. "Co-Founder & CEO") |
| `description` | string | Auto-generated sentence |
| `valid_from` | string | Start date |
| `valid_to` | string | End date (null = still active) |
| `weight` | integer | Always `5` |

<Accordion title="Real example">
```json
{
  "uuid": "a1b2c3d4-5678-9abc-def0-112233445566",
  "title": "Co-Founder",
  "description": "Josh Diamond co-founded Orbiter.io.",
  "valid_from": "2023-01",
  "valid_to": null,
  "weight": 5
}
```
`(Josh Diamond)-[FOUNDED]->(Orbiter.io)`
</Accordion>

### ATTENDED_TOGETHER

Derived edge: two people overlapped at the same school during the same time period.

| | |
|---|---|
| **Pattern** | `(Person)-[ATTENDED_TOGETHER]-(Person)` |
| **Weight** | **35-70** (varies by shared context) |
| **Created by** | `mvp/edges/create-education-edges` (#1898) — derived |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `school` | string | School name |
| `overlap_start` | string | Start of overlap period |
| `overlap_end` | string | End of overlap period |
| `same_degree` | boolean | Whether they shared the same degree |
| `same_major` | boolean | Whether they shared the same major |
| `same_field` | boolean | Whether they shared the same field |
| `relationship_strength` | integer | 1-3 (3 = same major, strongest) |
| `context` | string | Description of the connection |
| `description` | string | Auto-generated sentence |
| `weight` | integer | `35` (same major), `50` (same field), `70` (overlap only) |

<Note>
This is an **undirected** relationship — `MERGE (person)-[at:ATTENDED_TOGETHER]-(peer)` uses a non-directional pattern.
</Note>

## Expertise

Edges connecting people to expertise domains. Created by `mvp/expertise/resolve-person-expertise` (#12668) and `mvp/node/sync-domain-expertise-nodes` (#12663).

### HAS_EXPERTISE

Person has expertise in a domain or sub-domain. Created via vector search matching or when a new SubDomainExpertise node is created.

| | |
|---|---|
| **Pattern** | `(Person)-[HAS_EXPERTISE]->(DomainExpertise\|SubDomainExpertise)` |
| **Weight** | N/A (uses `confidence` and `match_score` instead) |
| **Created by** | `mvp/expertise/resolve-person-expertise` (#12668) |

| Property | Type | Description |
|----------|------|-------------|
| `confidence` | number | `1.0` (directly evidenced) or `0.7` (inferred). ON MATCH keeps max. |
| `signal_type` | string | How the expertise was detected: `job_title`, `company_role`, `project`, `skill_mention`, `education`, `achievement`, `tool_proficiency`, or `industry_experience`. Pipe-separated when multiple signals merge. |
| `depth` | string | Expertise depth: `creator` > `expert` > `practitioner` > `familiar`. ON MATCH keeps strongest. |
| `recency` | string | Temporal currency: `current` > `recent` > `historical`. ON MATCH keeps most current. |
| `evidence` | string | Pipe-separated evidence strings (e.g. `"Founded AI startup \| Led ML team"`) |
| `identified_as` | string | Pipe-separated LLM-identified expertise area names (the raw `expertise_area` before vector matching) |
| `match_score` | number | Cosine distance from vector search (lower = better match, threshold < 0.45) |
| `match_type` | string | `"matched"` (existing node) or `"created"` (new node) |

<Accordion title="Real example">
```json
{
  "confidence": 1.0,
  "signal_type": "company_role",
  "depth": "creator",
  "recency": "current",
  "evidence": "Co-Founder and CEO of Orbiter.io",
  "identified_as": "entrepreneurship",
  "match_score": 0.38,
  "match_type": "matched"
}
```
`(Josh Diamond)-[HAS_EXPERTISE]->(Entrepreneurship Education [SubDomainExpertise])`
</Accordion>

### SPECIALIZES_IN

Company specializes in a domain or sub-domain. Created by running each item in the company's `specialties` array through the expertise vector search pipeline — same embedding + matching logic as `HAS_EXPERTISE`, but with a distinct edge type for semantic clarity.

| | |
|---|---|
| **Pattern** | `(Company)-[SPECIALIZES_IN]->(DomainExpertise\|SubDomainExpertise)` |
| **Weight** | N/A (uses `match_score`) |
| **Created by** | `mvp/expertise/resolve-company-specialties` (#12746) |

| Property | Type | Description |
|----------|------|-------------|
| `specialty` | string | The original specialty string from the Company's `specialties` array that triggered this match |
| `weight` | number | Path weight derived from match_score: `min(round(10 + score×80), 50)`. Lower = stronger path. Range 14–50. |
| `match_score` | number | Cosine distance from vector search (lower = better match, threshold < 0.45) |
| `match_type` | string | `"matched"` (existing node) or `"created"` (new node) |

<Accordion title="Real example">
```json
{
  "specialty": "AI/ML",
  "match_score": 0.18,
  "match_type": "matched"
}
```
`(Company)-[SPECIALIZES_IN]->(Artificial Intelligence [SubDomainExpertise])`
</Accordion>

**Cross-entity query** — find people whose expertise matches a company's specialties:

```cypher
MATCH (c:Company {uuid: $company_uuid})-[:SPECIALIZES_IN]->(d:SubDomainExpertise)<-[:HAS_EXPERTISE]-(p:Person)
```

### HAS_SUBDOMAIN

Links a SubDomainExpertise node to its parent DomainExpertise node. Forms the expertise taxonomy tree.

| | |
|---|---|
| **Pattern** | `(DomainExpertise)-[HAS_SUBDOMAIN]->(SubDomainExpertise)` |
| **Weight** | N/A (structural/taxonomy edge) |
| **Created by** | `mvp/node/sync-domain-expertise-nodes` (#12663) |

## Funding

Edges connecting companies, investors, and funding rounds. Created by `mvp/funding/add-fundable-deal-node` (#12701) and `mvp/fundable/resolve-investors-edges` (#12702).

### RAISED

A company raised a funding round.

| | |
|---|---|
| **Pattern** | `(Company)-[RAISED]->(Funding_Round)` |
| **Weight** | **10** |
| **Created by** | `mvp/funding/add-fundable-deal-node` (#12701) |

| Property | Type | Description |
|----------|------|-------------|
| `weight` | integer | Always `10` |
| `announced_date` | string | Date announced |
| `company_name` | string | Company name |
| `description` | string | Round description |

<Accordion title="Real example">
```json
{
  "weight": 10,
  "announced_date": "2019-11-25",
  "company_name": "Frame.io",
  "description": "Frame.io raises $50M in Series C funding led by Insight Partners."
}
```
`(Frame.io)-[RAISED]->(Frame.io Series C)`
</Accordion>

### INVESTED_IN (Funding)

An institutional investor or angel participated in a funding round (not as lead).

| | |
|---|---|
| **Pattern** | `(Company:VC_Firm\|Person:Angel)-[INVESTED_IN]->(Funding_Round)` |
| **Weight** | **20** (institutional) or **25** (angel) |
| **Created by** | `mvp/fundable/resolve-investors-edges` (#12702) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Composite: `investor_uuid-round_uuid` |
| `description` | string | Auto-generated investor sentence |
| `is_lead_investor` | boolean | Always `false` |
| `weight` | integer | `20` (institutional) or `25` (angel) |

<Note>
`INVESTED_IN` is used in two contexts: (1) person-to-company from work/title detection (weight 15), and (2) investor-to-funding-round from funding data (weight 20/25). The source pattern and target node type distinguish them.
</Note>

<Accordion title="Real example">
```json
{
  "uuid": "accel-uuid-frameio-seed-uuid",
  "description": "Accel invested in the Frame.io Seed funding round.",
  "is_lead_investor": false,
  "weight": 20
}
```
`(Accel)-[INVESTED_IN]->(Frame.io Seed)`
</Accordion>
