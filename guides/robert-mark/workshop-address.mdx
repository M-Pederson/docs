## Points of comparison

A few asymmetries between the two tables worth discussing:

- **Coordinate types differ** — `master_address` stores `longitude` / `latitude` as `text`; `primary_location` uses `decimal`.
- **Normalization differs** — `master_address` stores city / region / country as free text plus codes; `primary_location` references the normalized `city` / `region` / `country` lookup tables.
- **`updated_at`** exists on `master_address` but not on `primary_location`.
- **No uniqueness constraints** on either table beyond the primary key — nothing prevents duplicate addresses or locations for the same entity.

## Proposed: `natural_key` on master\_address — scope

<Note>
**Workshop scope only — no Xano changes made.** This section scopes the canonical-address dedupe key. It reuses the `natural_key` column already planned for `master_address` in the [Universe ↔ Backend sync design](/guides/open-work/backend-refactor-tables) (upsert key `(node_uuid, natural_key)`, one of the four sync columns), so we get one dedupe concept, not two.
</Note>

### Why — today's dedupe is broken in both directions

The two live writer paths use different, conflicting keys (verified against live XanoScript, 2026-07-15):

- `mvp/address/add-company-address` (2451) dedupes on **`(master_company_id, city, state_region)`** — massively **over-dedupes**: one row per company per city. A second office in the same city is silently swallowed (or overwrites the first via the update branch when `address_line_1` is empty).
- `mvp/address/create-person-address-join` (4654) dedupes on **exact raw `(master_person_id, formatted_address)`** — **under-dedupes**: "123 Main St" and "123 Main Street" both insert, since one string is user-typed and the other geocoder-formatted.

A computed `natural_key` with a unique index replaces both with one correct rule.

### Key definition

`natural_key` is a **computed, tier-prefixed normalized string** (not the raw `formatted_address`):

| Tier | Prefix | Value | When |
| --- | --- | --- | --- |
| 1 | `pid:` | geocoder place id | **Not available today** — Radar forward geocoding returns no durable place id (the retired `archive/details/add-address` 228 took a Google `place_id` input). Keep the optional `place_id` column from the sync design for a future geocoder upgrade. |
| 2 | `cmp:` | `country_code\|region_code\|city\|postal_code\|street\|number\|unit` | Row has geocoded components (the normal Radar path) |
| 3 | `fmt:` | normalized `formatted_address` string | Row was never geocoded (user-typed only) |

Normalization for tiers 2–3: lowercase, trim, collapse whitespace, strip punctuation and diacritics, null/missing component → empty string. Tier prefixes prevent cross-tier false collisions.

**Recommendation:** store the readable pipe-joined string, not a sha256 of it — deterministic either way, but debuggable when a merge goes wrong. Hash only if key length becomes a problem.

**Uniqueness:** composite unique index on `(master_person_id, master_company_id, natural_key)` — two people at the same address must remain legal. The null-vs-`0` FK defaults must be coalesced consistently (today `master_person_id` defaults to `0` but `master_company_id` to null) so blank owners actually collide.

### Backfill & rollout order

1. **Add columns** `natural_key` (text) + optional `place_id` (text) — via the Xano UI, *not* the MCP full-schema replace (drops `trim` validators) and never `xano workspace push`.
2. **Ship the helper** and wire all writers to compute-on-write. No constraint yet — old and new rows coexist.
3. **Backfill** existing rows in batches (rows with geocoded components → `cmp:` tier; rest → `fmt:` tier). Row volume: TBD, measure before running.
4. **Collision report** — group by `(owner, natural_key)`; merge duplicates keep-oldest and repoint `my_person_address_join` (310), `my_company_address_join` (314), `my_address_notes` (293).
5. **Create the unique index** only after the collision report is clean — index creation fails on existing dupes.
6. **Flip writers to upsert semantics** (early-return paths still bump counters, per the queue upsert convention).

### Open questions

- Keep `unit` in the key? (Recommend yes — two units in one building are distinct addresses.)
- Countries without postal codes and PO boxes — does the component normalization need per-country rules, or is best-effort fine at current data quality?
- Backfill geocoding for never-geocoded rows (costs Radar calls per row) or accept `fmt:` keys until a row is next touched?
- Does this land before or with the Universe sync's `ALTER TABLE` for the other three sync columns (`content_hash`, `source_run_id`, `locked`)?
