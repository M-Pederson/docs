## Overview

APIANT is the integration middleware between meeting-source providers (Google Meet, Granola, Zoom, future) and Orbiter. APIANT owns OAuth, source webhooks/polling, token refresh, artifact discovery, and normalization. Orbiter receives normalized data through a small set of generic endpoints and handles calendar matching, storage, and graph processing.

This page is the live build log for the integration. It is updated as each milestone lands in Xano workspace 3.

## Authentication

### Inbound (APIANT → Orbiter)

Workspace-scoped Bearer API key from `integration_api_key`.

```http
Authorization: Bearer orb_live_wsk_<workspace_id>_<random>
Content-Type: application/json
Idempotency-Key: <uuid-v4>
X-Orbiter-Source: google_meet
```

- One key per Orbiter workspace. Hot rotation supported.
- Keys are scoped to a single workspace and cannot read or write other workspaces' data.
- Compromised keys can be revoked individually; APIANT will receive `401 invalid_api_key` and must alert.

### Outbound (Orbiter → APIANT)

Standard APIANT API key, held in Orbiter Xano environment variables. Used for: starting OAuth flows, triggering backfill, revoking connections.

## Helper functions

Every inbound endpoint calls these two helpers as the first two steps in its stack. They are not endpoints themselves — they're reusable Xano functions invoked via `function.run`.

### `integrations/auth_check` (function id 12937)

```xs
function.run "integrations/auth_check" as $auth
```

- Reads `Authorization` header from `$env.$http_headers`, case-tolerant.
- Strips the `Bearer ` prefix.
- Looks up the token in `integration_api_key` where `status = active`.
- Updates `last_used_at` on success.
- Throws `accessdenied` with `{error:{code:"invalid_api_key"}}` body if missing/invalid/revoked.
- Returns `{ workspace_id, api_key_id }`.

### `integrations/idempotency_check` (function id 12938)

```xs
function.run "integrations/idempotency_check" {
  input = {
    workspace_id: $auth.workspace_id
    endpoint: "POST:/v1/integrations/meetings"
    raw_body: $body
  }
} as $idem
```

- Reads `Idempotency-Key` header. Throws `invalid_payload` if missing.
- SHA256 hashes the canonicalized body.
- Looks up `(workspace_id, idempotency_key)` in `integration_idempotency`.
- **Same key + same hash** → returns `{ duplicate: true, response_body, ... }` so the endpoint can replay the original response verbatim.
- **Same key + different hash** → throws `idempotency_conflict`.
- **New key** → inserts a row with `expires_at = now + 48h` and returns `{ duplicate: false, key_row_id }`. The endpoint backfills `response_body` + `record_id` after writing.

### POST /v1/integrations/connections

Register or upsert a connection. Idempotent on `apiant_connection_id` (creates if not found, updates if found).

**Request body**

```json
{
  "user_id": 123,
  "workspace_id": 3,
  "source": "google_meet",
  "external_account_id": "mark@orbiter.io",
  "external_account_label": "mark@orbiter.io (Workspace)",
  "apiant_connection_id": "conn_abc123",
  "status": "active",
  "scopes": ["meetings.space.readonly", "drive.readonly"],
  "connected_at": "2026-04-30T14:00:00Z"
}
```

**Response 200**

```json
{ "id": 42, "status": "created" }
```

`status` is `"created"`, `"updated"`, or `"duplicate"` (idempotency replay).

### PATCH /v1/integrations/connections/{apiant_connection_id}

**Request body** — only fields you want to change:

```json
{ "status": "needs_reauth" }
```

**Response 200**

{ "external_id": "conn_abc123", "status": "updated", "connection_status": "needs_reauth" }
```

Mark connection revoked. Does not delete historical meetings.

**Response 200**

```

### POST /v1/integrations/meetings

Push one normalized meeting. Single most important endpoint. Both real-time webhook deliveries and backfill batches use this same path with the same Idempotency-Key.

**Dedup strategy** — idempotent on **two** levels:
1. `Idempotency-Key` header (48h window)
2. Natural key `(workspace_id, source, source_meeting_id)` — second write to the same meeting upserts (extends `artifacts`, `participants`, etc.) and returns `status: "duplicate"`.

**Calendar match** — performed inline against the existing `event` table:
1. If `calendar_match_hints.ical_uid` provided → match against `event.external_id`. Confidence: high.
2. Else if `calendar_match_hints.conference_url` provided → match against `event.url`. Confidence: medium.
3. Else: `calendar_match` is null. (Granola fuzzy-match path: time + organizer + attendee — TODO, run as background pass.)

**Request body** — see [meeting envelope](#integration_meeting-table-id-718) field reference.

**Response 200**

```json
{
  "id": 42,
  "status": "accepted",
  "calendar_match": {
    "matched_event_id": 7891,
    "confidence": "high",
    "method": "ical_uid"
  }
}
```

`status` is `"accepted"`, `"duplicate"`, or `"partial"` (stored but artifacts incomplete).

### POST /v1/integrations/events

Generic event log. Use cases: `transcript_not_yet_available`, `meeting_too_short_to_process`, `user_revoked_at_source`. Resolves `apiant_connection_id` → `integration_connection_id` and `(source, source_meeting_id)` → `integration_meeting_id` for join convenience when known.

**Request body**

```json
{
  "user_id": 123,
  "workspace_id": 3,
  "source": "zoom",
  "apiant_connection_id": "conn_abc123",
  "source_meeting_id": "uuid-from-zoom",
  "event_type": "transcript_not_yet_available",
  "severity": "info",
  "message": "Zoom AI Companion still processing; will retry in 5min",
  "payload": { "retry_after_sec": 300 }
}
```

**Response 200**

```json
{ "id": 99 }
```

## Idempotency & retries

Every POST/PATCH/DELETE request must include `Idempotency-Key: <uuid-v4>`.

- Orbiter stores the key for 48 hours.
- Duplicate key + identical payload → `200` with `status: "duplicate"` and the original `id`.
- Duplicate key + different payload → `409 idempotency_conflict`.
- APIANT retries on `5xx` and network failures with exponential backoff (start 30s, max 1h, max 10 attempts), reusing the same Idempotency-Key.
- APIANT does not retry `4xx` except `429`. On `429`, honor `Retry-After`.

## Error contract

```json
{
  "error": {
    "code": "invalid_payload",
    "message": "Field 'started_at' must be ISO8601",
    "field": "started_at",
    "request_id": "req_abc123"
  }
}
```

| HTTP | Code | Meaning | APIANT action |
|---|---|---|---|
| 400 | `invalid_payload` | Malformed/missing required field | Do not retry; log + alert |
| 401 | `invalid_api_key` | Bad bearer token | Do not retry; alert |
| 403 | `forbidden_workspace` | Key doesn't match `workspace_id` in body | Do not retry; alert |
| 404 | `connection_not_found` | `apiant_connection_id` unknown | Do not retry; resync connection state |
| 409 | `idempotency_conflict` | Same key, different payload | Do not retry; investigate |
| 422 | `unprocessable` | Valid shape, semantic violation (e.g. `ended_at` before `started_at`) | Do not retry; log |
| 429 | `rate_limited` | Workspace quota exceeded | Retry per `Retry-After` |
| 500 | `internal_error` | Orbiter bug | Retry with backoff |
| 503 | `service_unavailable` | Maintenance / dependency down | Retry with backoff |

## Validation rules (Orbiter-side)

These rules are enforced by Orbiter; understanding them helps APIANT avoid `400`s.

1. `started_at < ended_at` (returns `422` otherwise).
2. `duration_sec >= 0`.
3. `source_meeting_id` non-empty and `< 512` chars.
4. `participants[].email` must validate as email or be null.
5. Artifact `content` `<= 5MB` inline. Larger transcripts must be split or sent as `url`.
6. Artifact `url` must be HTTPS.
7. `summary_doc.url` recommended even when `content` provided (preserves source link for users).
8. `raw_payload <= 5MB`.
9. Total request body `< 10MB`.

## Backfill expectations

- After connection created, Orbiter calls APIANT to trigger 30-day backfill.
- APIANT delivers backfill via the same `POST /v1/integrations/meetings` endpoint.
- APIANT may parallelize up to 10 concurrent requests per workspace.
- Backfill ordering does not matter; meetings are independent.

## Open questions for APIANT

1. **Webhook dedup behavior** — when Zoom fires `recording.completed` and `meeting.summary_completed` separately, does APIANT consolidate into one Orbiter call, or push twice? Orbiter is idempotent on natural key, so either works — preference for one consolidated push when feasible.
2. **Backfill kickoff** — APIANT exposes a "trigger backfill" API that Orbiter calls? Confirm endpoint shape.
3. **Connection re-auth flow** — when APIANT detects token expiry, does APIANT prompt the user directly, or signal Orbiter to prompt via `PATCH /connections` with `status: needs_reauth`? Orbiter UI will show banner from connection status either way.
4. **Per-user rate limits at source** — does APIANT smooth source-side rate limits, or pass through? Orbiter's `429` will only apply to Orbiter API.
5. **PII handling in `raw_payload`** — some sources include emails/IPs. Confirm APIANT's encryption-at-rest posture before populating `raw_payload`.
