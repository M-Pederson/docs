Every enrichable MusicBrainz entity type has its own **get-add gate** — the six functions below are how new work enters the waterfall, whether from a cascade discovery, a bootstrap batch, or a manual seed. The operator seed endpoints that feed them are listed below the gates. Architecture spec: [Music Industry Waterfall overview](/guides/enrichment/waterfall/music-waterfall) · function roster: [Music Waterfall Functions](/guides/enrichment/waterfall/music-functions) · queue drain: [Music Waterfall Crons](/guides/enrichment/waterfall/music-crons).

## Entry gates

<Card icon="play">
  ```text
  mb/get-add/label — 13087           → run-base-label-enrich (13091)
  mb/get-add/artist — 13088          → run-base-artist-enrich (13094)
  mb/get-add/release-group — 13089   → run-base-release-group-enrich (13099)
  mb/get-add/recording — 13093       → run-base-recording-enrich (13100)
  mb/get-add/work — 13098            → run-base-work-enrich (13101)
  mb/get-add/event — 13106           → run-base-event-enrich (13105)
  ```

  One gate per enrichable entity (the event gate landed 2026-06-12, completing the set). Each gate: 30-day `enriched_at` dedup → skip `gone`-tombstoned rows → ensure a stub row in the matching `mb_*` source table → seed propagation (when no caller seed is passed, the just-created stub is its own seed).

  Branching: **sync-enrich** (invoke the orchestrator directly) when `cascade_depth == 0 && queue == false`; otherwise upsert the queue row and let the drain handle it.
</Card>

<Note>
  Bootstrap seed batches queue at **`priority_tier = 0`** — the reserved lane that drains ahead of all cascade traffic ([decision 5](/guides/enrichment/waterfall/music-waterfall#resolved-design-decisions-2026-06-11)). The queue upserts default via `first_notnull` (tier 4 / depth 1), so an explicitly passed `0` survives.
</Note>

## Seed endpoints (operator)

Manual seeding rides the `musicbrainz` API group (1283, canonical `P8PNYYKM`, base URL `https://xh2o-yths-38lt.n7c.xano.io/api:P8PNYYKM`) — each endpoint routes straight into the matching get-add gate. None are pipeline-wired. The QA/tooling endpoints of the same group live on [Music Waterfall Functions → Operator endpoints](/guides/enrichment/waterfall/music-functions#operator-endpoints).

| Endpoint | Verb | ID | Purpose |
| --- | :-: | --: | --- |
| `seed/label` | POST | 8643 | Seed a label MBID (→ `mb/get-add/label`) |
| `seed/artist` | POST | 8644 | Seed an artist MBID (→ `mb/get-add/artist`) |
| `seed/event` | POST | 8649 | Seed an event MBID (→ `mb/get-add/event`) |
| `seed/recording` | POST | 8666 | Seed a recording MBID (→ `mb/get-add/recording`) — cascade-only type |
| `seed/release-group` | POST | 8667 | Seed a release-group MBID (→ `mb/get-add/release-group`); also materializes the canonical `Release` |
| `seed/work` | POST | 8668 | Seed a work MBID (→ `mb/get-add/work`) — cascade-only type |

Every call requires the standard Xano headers — `X-Data-Source` and `X-Branch`:

```bash
curl -X POST https://xh2o-yths-38lt.n7c.xano.io/api:P8PNYYKM/seed/artist \
  -H "Content-Type: application/json" \
  -H "X-Data-Source: sandbox" \
  -H "X-Branch: v1" \
  -d '{"mbid": "5b11f4ce-a62d-471e-81fc-a69a8278c7da"}'
```
