This is the executable build plan for **Scope A** of the copilot orchestration migration.

<Note>
  This page is a **plan only** — execute it phase by phase, verifying each acceptance check before moving on. It covers `cards.ts`, the route, typed data tools over the Robert API, `emitCard`, `prompts.ts`, `toCrayonSSE`, auth forwarding, and persistence: replace the blocking Xano `/chat` call with a streaming Vercel AI SDK (`ai@^6`) agent loop in a TanStack Start route, while Xano stays as data, auth, and tools. The Crayon renderer and all 18 template cards are **out of scope** and must not change beyond the client repoint + explicit TTS shim. For the rationale, architecture diagrams, scope boundaries, and the [current-state audit](/guides/open-work/ai-sdk-migration#current-state-verified), see the parent design page: [/guides/open-work/ai-sdk-migration](/guides/open-work/ai-sdk-migration).
</Note>

<Warning>
  The client-side SSE event names are load-bearing. The existing `processStreamedMessage` (from `@crayonai/react-core`, imported dynamically at `copilot-app.tsx:1160`) parses exactly `event: text` and `event: tpl`. The server adapter `toCrayonSSE` must emit those **exact** event names or the renderer fails silently. Do not "improve" the event names in Scope A.
</Warning>

<Warning>
  **Not every mode is an ordinary agent-loop chat turn.** Verified in `copilot-app.tsx`: **meeting** mode never calls `/chat` — it is a deterministic 2-step client interceptor (lines ~542-665). The first meeting message returns a synthetic `meeting_goal_card`; selecting a goal sends a `[MEETING_GOAL] ...` prompt that calls `/meeting-prep` directly and returns a `meeting_prep_card`. Likewise the **quick-leverage** dispatch (`"in every possible way"`, lines ~480-540) is a deterministic dispatch, not an LLM turn. The phrase "all 4 modes served by the route" does **not** mean a like-for-like agent-loop migration for these paths — see Phase 8 Step 3 for what "served" means per mode.
</Warning>

### Critical details to not miss

<Warning>
  These are the highest-risk implementation details. Re-check this list before every coding session and before opening the PR.
</Warning>

- **New endpoints only:** Scope A is additive. Create the new frontend route `/api/copilot/chat`; wrap existing Robert endpoints as tools; if a different backend contract is required, add a new versioned/additive Robert endpoint instead of changing an existing one.
- **TanStack route primitive:** use `createFileRoute("/api/copilot/chat")({ server: { handlers: { POST }}})`. Do not use `createServerFileRoute`.
- **Exact SSE events:** Crayon only understands `event: text` and `event: tpl`. Do not rename them or add a required third event.
- **No renderer edits:** `templates.ts`, the 18 template components, `copilot-crayon.css`, and voice hook internals stay untouched in Scope A.
- **Card schema gotchas:** `outcome_card.createdAt/dispatchedAt/completedAt` are numbers, not strings; several avatar/result fields accept `null`; `dispatch_confirmation.master_person_id` may be `null` from the existing fallback.
- **Search tool gotcha:** `searchPersons` uses `/search?q=&mode=&limit=`, not `scope`; preserve the `/person-search?query=&limit=` fallback.
- **Route body must carry context:** send `conversation_id`, `master_person_id`, `person_context`, and `network_data` from the client, or route-side persistence/prompt parity will regress.
- **Conversation id comes back in a header:** the route creates/reuses the conversation before streaming and returns `X-Copilot-Conversation-Id`; the client must copy it into `conversationIdRef.current`.
- **TTS needs a stream clone:** `processStreamedMessage` consumes the `Response`; clone/tee it first if you need to collect text for `voiceRef.current.speak`.
- **Build-time flag:** `VITE_COPILOT_AI_SDK` is not instant rollback. Use a runtime flag source if no-deploy rollback is required.

### Tools to build

Each typed tool wraps an existing Robert API endpoint (group `Bd_dCiOz`, id 1261). The `execute()` forwards the user's bearer token. These are the data tools for the agent loop — start with the first three behind the flag (Phase 8 rollout step 1). Wrapping an endpoint is allowed; changing an existing endpoint contract is not. If parity requires a different payload or response, add a new Robert endpoint and call that from the tool.

| Tool (typed, Zod `inputSchema`) | Wraps Robert API endpoint | Method | Notes |
| --- | --- | --- | --- |
| `searchPersons(q, mode, limit?)` | `/search` + `/person-search` fallback | GET | Match `src/features/copilot/lib/xano-copilot.ts:305-348`: `/search` uses `q`, `mode`, `limit`; fallback `/person-search` uses `query`, `limit` and normalizes the response. |
| `getPersonContext(master_person_id)` | `/person-context/{id}` | GET | Enriched person context (currently fetched client-side, passed as `person_context` string). |
| `getConnectionPath(target_master_person_id)` | `/connection-path` | GET | 2nd/3rd-degree path to a target. |
| `getNetwork(query?, page?, per_page?)` | `/network` | GET | Network summary (currently fetched client-side, passed as `network_data` string). See Phase 5 / Risks for parity. |
| `getMeetingPrep(...)` | `/meeting-prep` | POST | Meeting prep generation (mode `meeting`). May stay behind the client interceptor — see Phase 8 Step 3. |
| `memory(op, ...)` | `/memory`, `/memory/{id}` | GET/POST/PATCH/DELETE | Memory read/write. Prefer separate tools (`getMemories`, `createMemory`, `updateMemory`, `deleteMemory`) unless the model quality tests show an `op` discriminator is easier to control. |
| `triggerGeneration(suggestion_request_id, mode)` | `/generate-leverage-loop`, `/generate-outcome`, `/generate-serendipity` | POST | Async generation. `mode` is `loop`, `outcome`, or `serendipity`; body is `{ suggestion_request_id }`. Pair with `/process-status` polling. |
| `emitCard(card)` | none (local) | n/a | Validates a `CardEnvelope` against `cards.ts` before SSE. The only non-network tool. |

<Tip>
  `emitCard` is not a Robert API wrapper — it is the validation gate. The model calls it (or the route calls it on every model-emitted card) to run the `CardEnvelope` discriminated union through Zod. Reject/retry malformed cards **before** they reach `toCrayonSSE`.
</Tip>

    ```ts
    // src/features/copilot/server/to-crayon-sse.ts
    import { CardEnvelope, type CardEnvelope as CardEnvelopeType } from "../contracts/cards";

    type CrayonCallbacks = {
      onText?: (delta: string) => void;
      onCard?: (card: CardEnvelopeType) => void;
      onFinish?: () => Promise<void> | void;
      onError?: (error: unknown) => void;
    };

    export function formatCopilotError(error: unknown): string {
      const raw = error instanceof Error ? error.message : String(error ?? "Unknown error");
      if (raw.includes("Authentication expired") || raw.includes("Unauthorized")) {
        return "Session expired. Please refresh the page and try again.";
      }
      if (raw.includes("permission") || raw.includes("403")) {
        return "You don't have permission to perform this action.";
      }
      if (raw.includes("Too many requests") || raw.includes("429")) {
        return "Too many requests. Please wait a moment and try again.";
      }
      if (raw.includes("Failed to fetch") || raw.includes("NetworkError")) {
        return "Network error. Check your connection and try again.";
      }
      return `Something went wrong: ${raw}`;
    }

    export function toCrayonSSE(
      result: { fullStream: AsyncIterable<any> },
      callbacks: CrayonCallbacks = {},
      headers: HeadersInit = {},
    ): Response {
      const encoder = new TextEncoder();
      const stream = new ReadableStream({
        async start(controller) {
          const emit = (event: "text" | "tpl", data: string) => {
            controller.enqueue(encoder.encode(`event: ${event}\ndata: ${data}\n\n`));
          };

          try {
            for await (const part of result.fullStream) {
              if (part.type === "text" && part.text) {
                callbacks.onText?.(part.text);
                emit("text", part.text);
              }

              if (part.type === "tool-result" && part.toolName === "emitCard") {
                const card = CardEnvelope.parse(part.output);
                callbacks.onCard?.(card);
                emit("tpl", JSON.stringify({
                  name: card.name,
                  templateProps: card.templateProps,
                }));
              }
            }
            await callbacks.onFinish?.();
          } catch (error) {
            callbacks.onError?.(error);
            emit("text", formatCopilotError(error));
          } finally {
            controller.close();
          }
        },
      });

      return new Response(stream, {
        headers: {
          "Content-Type": "text/event-stream",
          "Cache-Control": "no-cache, no-transform",
          Connection: "keep-alive",
          "X-Accel-Buffering": "no",
          ...headers,
        },
      });
    }
    ```

    async function collectCrayonText(response: Response): Promise<string> {
      if (!response.body) return "";
      const reader = response.body.getReader();
      const decoder = new TextDecoder();
      let buffer = "";
      let text = "";

      while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        buffer += decoder.decode(value, { stream: true });
        const events = buffer.split("\n\n");
        buffer = events.pop() ?? "";
        for (const event of events) {

            .split("\n")
            .filter((line) => line.startsWith("data:"))
            .map((line) => (line.startsWith("data: ") ? line.slice(6) : line.slice(5)))
            .join("\n");
          text += data;
        }
      }
      return text.trim();

  </Step>

  <Step title="Flagged rollout (default first, then mode-by-mode)">

          - The other three modes are unaffected.
        </Check>
      </Step>
      <Step title="Step 2 — Validate quality, latency, parity; turn on telemetry">

        <Check>
          - For the comparison prompt set, the route produces cards that pass `CardEnvelope` validation and match the Xano path's intent.
          - p95 latency is within the agreed budget.
          - Traces are flowing to the sink.
        </Check>
      </Step>
      <Step title="Step 3 — Roll mode-by-mode: leverage → meeting → outcome">
        Widen the flag condition one mode at a time, in this order, adding mode-specific tools as needed. **But "served by the route" does not mean the same thing for every mode** — some modes are deterministic, non-LLM paths and may legitimately stay client-side.

        <AccordionGroup>
          <Accordion title="meeting — decide: keep the interceptor, or replicate the 2-step flow">

            Either way the agent/interceptor must read `selectedModeRef.current` (meeting is not in the `selectedMode` state union, `:213-215`).
          </Accordion>

These **cannot** move server-side (they are not pure SSE). Keep quick-leverage as a **permanent client interceptor**, and scope only the **conversational** leverage turns to the route. Do not try to fold the deterministic dispatch into the agent loop.
          </Accordion>
        </AccordionGroup>

        For the conversational portions: `leverage`/`outcome` need `triggerGeneration` (`/generate-*`) + `memory`; `meeting` (if path (b)) needs `getMeetingPrep` (`/meeting-prep`). Re-run the parity/latency check after each mode.

        <Warning>

As each mode moves to the route, decide per interceptor: formalize as a server tool/card, keep as an explicit **declared** shim, or create a new additive wrapper endpoint.   </Step>

    <Note>Phase 9.</Note>

## Risks & watch-items

<AccordionGroup>
  <Accordion title="Additive endpoint discipline">
    Scope A must not mutate existing Robert/Xano endpoint contracts. Existing endpoints may be wrapped by tools exactly as-is. If route-side persistence, idempotency, or deterministic shims need a different contract, create a new additive endpoint and leave the old one callable until Phase 9+ retirement.
  </Accordion>
  <Accordion title="Route primitive resolved">
    TanStack Start server routes use `createFileRoute("/api/copilot/chat")({ server: { handlers: { POST }}})`, which gives the handler a raw `Request` for `Authorization` / `X-Data-Source` / `X-Branch` forwarding. Do not use the old draft's `createServerFileRoute` placeholder.
  </Accordion>
  <Accordion title="Message-shape mismatch (Crayon vs ModelMessage)">
    Crayon thread messages are `{ id, role, message? }`; `streamText` requires `{ role, content }`. The client sends raw Crayon messages and `shapeHistory` (Phase 3) maps them before `streamText` (Phase 5) — porting `copilot-app.tsx:667-711` (assistant text-part extraction, the `"I provided a card response."` fallback, last-10, text-only). No `message`/`id` may leak into the model call. Check: a thread with assistant card parts converts to valid `ModelMessage[]`.
  </Accordion>
  <Accordion title="Voice TTS dead on the streamed path">
    TTS fires inside `processMessage` (`:941-953`), which the flag-on branch bypasses. Without the Phase 6 shim (`response.clone()` before `processStreamedMessage`, collect `event: text` chunks from the clone, then `voiceRef.current.speak(fullText)`), voice silently stops working in default mode once flagged on.
  </Accordion>
  <Accordion title="Flag mechanism vs instant rollback">
    `VITE_COPILOT_AI_SDK` is build-time inlined by Vite — flipping it requires a rebuild/redeploy, so the "instant, no-deploy rollback" promise does not hold. If instant rollback is required, source the flag at runtime (copilot_config row, bootstrap/me endpoint, or localStorage/query-param). See *Before you start*.
  </Accordion>
  <Accordion title="Server-readable Xano base URL">
    `VITE_API_URL` is client-only (`import.meta.env`) and `undefined` server-side. Tools must read `env.XANO_API_URL` (added in Phase 0, schema + runtimeEnv) — never `process.env.VITE_API_URL`, which yields the host-less base `:Bd_dCiOz`.
  </Accordion>
  <Accordion title="Extra hop latency (client → server → Xano per tool call)">
    Each tool call now traverses client → the new route → Robert API, instead of the client fetching context and passing it inline to one blocking `/chat`. Measure p95 per-turn latency in Phase 8 step 2 against the Xano baseline. Mitigations: parallelize independent tool calls, cap `stopWhen: stepCountIs(6)`, and pre-resolve cheap context (e.g. `getPersonContext`) where the prompt already implies it.
  </Accordion>
  <Accordion title="Prompt parity + injected context (person_context / network_data)">
    The four system prompts move from [`copilot_config`](/guides/open-work/ai-sdk-migration#current-state-verified) to `prompts.ts`. Any paraphrase changes behavior — copy verbatim (Phase 2). Critically, today the client pre-fetches **person_context** (`personContextRef`, set at `:1310/1321`) and **network_data** (`networkSummary`, set at `:453`) and passes them as load-bearing `/chat` string params. Enumerate every Xano-injected context (memory / person / network) and map each to a tool result or interpolation — do **not** drop them silently. Add a `getNetwork` tool (wrapping `/network`) or pass `networkSummary` as initial context; inject selected-person context as a first message/system suffix or via `getPersonContext`.
  </Accordion>
  <Accordion title="Preserving client-side enrichment / fallbacks / interceptors">
    Today the client does network-avatar enrichment of `button_group` (lines ~796-847), a `dispatch_confirmation` fallback via regex on the last user messages (lines ~900-939), `BLOCKED_DURING_INTERVIEW` filtering, the meeting-goal interceptor (`:542-665`), and the quick-leverage interceptor (`:480-540`, with `handlePersonClear` + `chatKey` bump + grid removal side effects that cannot move server-side). Each must be either formalized as a server tool/card or kept as an **explicit, declared** shim — never quietly duplicated across client and server. Track each one to a decision before Phase 9.
  </Accordion>
  <Accordion title="Error-path SSE parity">
    The current `processMessage` has a client-side error mapper (`:1010-1051`) that converts auth/permission/429/network errors into friendly `event: text` messages. On the AI-SDK path a thrown tool/model error would surface differently. Port the classification into `toCrayonSSE`/the route (Phase 4) so failures emit a friendly `event: text` (or `error_card`) instead of dropping the stream.
  </Accordion>
  <Accordion title="SSE event-name + granularity coupling">
    `event: text` / `event: tpl` are the only names the existing Crayon parser understands. `toCrayonSSE` must emit them exactly. The current emitter is word-by-word (`:963-998`); token deltas are acceptable if framing/newlines hold — confirm `processStreamedMessage` tolerates partial-word chunks. Changing event names is a [Scope B](/guides/open-work/ai-sdk-migration#scope-b-renderer-crayon-openui) concern, not Scope A.
  </Accordion>
  <Accordion title="Historical conversation hydration after lambda retirement">
    Thread-recovery (`:1065-1139`) handles both stored `template_name`/`template_props` and raw JSON in `content` from lambda-era rows. Retiring the lambda (Phase 9) must keep the raw-JSON-in-`content` recovery path so old conversations still render.
  </Accordion>
</AccordionGroup>
