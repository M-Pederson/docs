**Session (2026-06-16).** All changes live in **Xano workspace 3, branch `v1`**, validated in **sandbox**, then backfilled on the **staging** data source (live not deployed yet). The graph node `description` is the embedding source, so fixing it also repairs the vectors.

## The problem

Two things were wrong with music node descriptions:

1. **~99% garbage.** `create-node-description` #13112 built the LLM input by dumping the **raw MusicBrainz JSON sliced to 8000 chars** — which cut big records mid-object, so deepseek replied *"Please provide the MusicBrainz record."* On the staging Cinq fixture **275 of 279** recording descriptions were that echo; only small records (less-famous duos) came out clean.
2. **Persons had none.** `run-base-artist-enrich` #13094 only called the finalizer for **groups** — `Person:Music_Artist` nodes were skipped (0/146 had a description) and kept their name+disambiguation vectors.

And because **the description is the embedding source** for recordings/groups, those ~275 recordings had their `name_embedding` + Gemini `embeddings` computed from the string *"Please provide the MusicBrainz record"* — near-identical, useless vectors.

## What shipped

<Card icon="filter">

```
mvp/music/create-node-description — 13112  (v2)
```

Builds a **slim, type-aware field-set** (name/type/country/life-span/genres/tags for artists+groups; name/artists/first-release/genres for recordings) instead of raw-JSON@8000 — small, complete, parseable. Prompt enforces a **strict ≤127 characters**; output is hard-truncated to 127 and **refusal echoes are rejected**. Handles recording / group / **artist**.

Adds the **`music_artist`** path (persons now get a description); **always** calls #13112 (no MB-annotation short-circuit, so ≤127 always holds); passes the real type through; writes the description text to the **source table + node**; and embeds `name_embedding` (#4676) + `embeddings` (#13114) **on the description**.

Forward-path wiring — calls `finalize-music-node` for **persons** (was groups-only), at all depths.

## Design notes

- **Strict ≤127.** Enforced in the prompt *and* hard-truncated (LLMs don't reliably count characters). One short factual sentence, starting with the entity name.
- **Description → vectors.** Recordings, groups, and persons all embed on their description (persons switched from name+disambiguation). A clean description means a meaningful vector.
- **Stub-proof.** Only `enrich_success` rows with a `node_uuid` are processed. The ~165 collaborator **stub** nodes (graph endpoints of credit edges, no MB record) are correctly skipped — they can't be described without full enrichment.

## Staging backfill

Re-ran every **describable** node on the staging Cinq fixture through #13113 v2 (`redo=true`) — replacing the garbage descriptions with clean ≤127 text on the node + source table and rebuilding `name_embedding` + `embeddings` from it. The 165 stubs are excluded by design.

**Result (verified on staging):** **280 / 280 recordings** + **14** enriched artists/groups re-described, **0 errors**. Recording descriptions went from **275/279 garbage → 0 garbage**, all 280 with description-based vectors and the text in `mb_recording.description`. Sample: *"Adictiva by Daddy Yankee and Anuel AA, released in 2020, is a reggaeton recording."*

| Node type | Total | Described | Garbage | Desc-based vectors |
| --- | --- | --- | --- | --- |
| Recording | 280 | 280 | **0** | 280 |
| Music_Artist (enriched) | 10 | 10 | 0 | 10 |
| Music_Group (enriched) | 4 | 4 | 0 | 4 |

(The remaining 136 `Music_Artist` + 29 `Music_Group` are collaborator stubs with no MB record — not describable without enrichment.)
