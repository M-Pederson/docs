## The problem

The contacts that matter most to a power operator — celebrity cell numbers, private direct lines earned over years — are the ones that end a career if they leak with the operator named as the source. Standard "encrypted at rest" does nothing for this: AlloyDB decrypts transparently for any authorized query, so an attacker with valid app or DB credentials gets plaintext. At-rest encryption protects against a stolen disk, not a breached application.

The fix is **application-layer field encryption plus tokenization**, gated by step-up auth, applied selectively to the crown-jewel fields — not a blanket control that taxes every contact.

## What it does

For a contact field marked sensitive (a VIP phone or email), Orbiter:

- Stores the value as **ciphertext**, never as a readable number in the database.
- Keeps a **token** (not the number) on the graph node, so relationship intelligence keeps working without the live value ever being exposed.
- Requires a **passkey-confirmed reveal** to show the real value — one field, one access, one audit record.
- Supports **crypto-shred**: destroy the key, the value becomes permanently unreadable.

The operator-facing claim: *if Orbiter is breached, your celebrity numbers are ciphertext, not a spreadsheet — and every reveal is logged, so nobody browses your book, not even Orbiter.*

<Info>
  **This is the operator claim, not BYOK.** The solo operator does **not** hold their own key — "only you hold the key" is the enterprise (Atlas) claim. The operator's claim is gated, logged, encrypted, no bulk-read. Keep the two separate; a security reviewer will catch drift. See [the BYOK boundary](#open-decisions).
</Info>

## The padlock — a three-state model

The control is a padlock icon on the contact side panel. A real padlock is **asymmetric** — you snap it shut with no key, but need the key to open it — and that asymmetry *is* the security model. There are three states, not two:

```mermaid
stateDiagram-v2
    [*] --> Unlocked: contact created (plaintext)
    Unlocked --> LockedMasked: click padlock — free, no key
    LockedMasked --> LockedRevealed: tap reveal + passkey (logged)
    LockedRevealed --> LockedMasked: auto-hide / navigate away
    LockedMasked --> Unlocked: remove lock (step-up + confirm)
    LockedRevealed --> Unlocked: remove lock (step-up + confirm)
    LockedMasked --> [*]: crypto-shred (destroy DEK)
```

<CardGroup cols={3}>
  <Card title="Locked + masked" icon="lock">
    Closed padlock, value hidden (`•••• ••••`). The resting state for a crown jewel.
  </Card>
  <Card title="Locked + revealed" icon="eye">
    Passkey-confirmed peek. Value shows for that moment, logged. **Padlock stays closed** — data is still tokenized, you just looked through the window.
  </Card>
  <Card title="Unlocked" icon="lock-open">
    Protection actually removed; plaintext returns to the field. Rare, deliberate, gated.
  </Card>
</CardGroup>

<Warning>
  **The footgun: reveal ≠ unlock.** Clicking the closed padlock must **reveal** (state 2), not **unlock** (state 3). If a single click strips protection, an attacker in a live session clicks every padlock open and walks out with the book — the leaked-spreadsheet outcome, delivered by our own UI.

  The rule that matches the physical padlock: **closing is free; peeking needs the key but stays locked; removing the lock is gated and confirmed.** Unlock lives behind a menu action with step-up + a "you sure?" confirm.
</Warning>

## Operator control model

The operator's real question is *"what do I create, and what do I hold?"* — and the disciplined answer is **the operator controls the reveal, never the key.** There is no user-held token or key on this tier: the token is a system artifact, and every key lives in Cloud KMS under Orbiter's control. What the operator owns is the *decision surface* — when a field is protected, when it's looked at, and when it's destroyed.

<Info>
  **The operator never authors or holds a token.** The "token" is the blind index — `HMAC(normalize(value), index_key)` — that the backend derives on lock. It is a system-managed stand-in, not a credential the operator types, stores, copies, or can lose. "Only you hold the key" is the [enterprise/Atlas claim](#open-decisions), deliberately *not* this tier.
</Info>

### Create — locking is the only path

A token comes into existence exactly one way in P1: the operator **locks** a field that already exists as plaintext. The click is free — no key, no challenge — and the backend does the rest (normalize → blind-index → encrypt → null the plaintext → write the token to the graph node). The operator's act is a single intent — *"protect this"* — not a key ceremony. P2 adds a second, automatic path — [born-locked ingestion](#phasing) that tokenizes known VIPs at the source — but that is system-authored too, never operator-authored.

### Control — four verbs, escalating gates

The operator's entire control vocabulary is four verbs, and the gate rises with the consequence:

| Verb | Gate | Effect on the value | The token |
| --- | --- | --- | --- |
| **Lock** | none — free | plaintext → ciphertext | **created** |
| **Reveal** | step-up (passkey), logged | decrypted for that moment | unchanged — peeked through the window |
| **Unlock** | step-up **+ confirm** | plaintext re-materializes | removed |
| **Shred** | step-up **+ confirm** | permanently unreadable | DEK destroyed; blind index may remain |

Protecting is frictionless, **looking** costs a passkey, and **removing or destroying** costs a passkey *plus* an explicit "you sure?". That gradient — cheap to protect, costed to expose — *is* the control model.

## Key custody — why Orbiter holds the key

The crown-jewel question for this tier is *who holds the key*. The answer is **Orbiter holds it** — the operator holds **reveal authority** (their passkey), not the key material. This is the deliberate, defensible choice, and it rests on one distinction security reviewers will test: **custody is not authority.**

<Info>
  **The decision in one line.** The operator's passkey *gates* every reveal; Orbiter's KMS-held KEK *performs* the decrypt. So "operator-controlled" is true and "Orbiter cannot decrypt" is **false** — and conflating the two is the exact claim-drift the [BYOK boundary](#open-decisions) exists to prevent. We hold the key on purpose, because moving it off-server would delete the platform value the $100 tier is sold on.
</Info>

### Why not the operator, and not a hybrid (yet)

| Option | Who holds the key | Honest claim | Intelligence on locked fields | Key-loss posture |
| --- | --- | --- | --- | --- |
| **A — Orbiter-custody** *(current)* | Orbiter (per-tenant KEK in Cloud KMS) | "Gated, logged, per-record, no bulk — nobody browses your book, not even Orbiter." **Never** "cannot decrypt." | **Full** — enrichment, dedup, shared-number edges, suggestions, search all survive (the blind index stays server-side) | Recoverable — re-enroll a passkey; the key never lived on the device |
| **B — Operator-custody (device E2EE)** | The operator's device / passkey-derived key | "Orbiter literally cannot decrypt" — and that is genuinely true | **None** — no server enrich, no value-dedup, no shared-number edge, drops from suggestions/search | **Irreversible** — lost device = permanently lost data, unless a recovery party is added (which dilutes the claim back toward A) |
| **C — Hybrid vault mode** | Per-field: default Orbiter, opt-in operator | Per-field — each field carries the claim it can actually back | Full by default; **none** on opted-in vault fields | Operator chooses per field |

Option B earns the strongest *literal* claim but pays for it by making the most valuable contacts the least intelligent — a direct inversion of the product. It is also the wrong key-loss posture for a solo operator with no HSM or escrow, where a lost or replaced phone is routine. So B is the right **primitive** for a narrow use, not the right **default**.

<Note>
  **"Like enterprise" does not mean device-held keys.** Enterprise [BYOK](#open-decisions) is **EKM-backed and server-reachable**: the tenant controls and can revoke the key, but Orbiter's servers still decrypt to run intelligence. The enterprise property a buyer covets is *only-you-can-revoke + every access is gated and logged* — **not** vendor-blindness. Option A is the honest solo-tier analogue of that property. A human-device-held key (Option B) is a different, strictly more brittle thing, and we never market the two as equivalent.
</Note>

### The vault-mode exception

True "we cannot decrypt" has exactly one honest home: a **per-field vault mode** (Option C) that promotes a specific crown jewel to operator-held E2EE. It applies when **both** conditions hold:

1. The operator's stated threat is an **Orbiter-side compromise or rogue insider** — the one scenario default custody does *not* cover (see below) — and
2. They **explicitly accept irreversible key-loss** for that field: they would rather destroy the number than have Orbiter able to surrender it.

That is the only case where "cannot decrypt" is both true and worth its cost. Vault mode is a **demand-gated fast-follow, not a launch posture** — it ships only once pilot data proves the paranoid micro-segment exists, and only with airtight routing-integrity tests.

<Warning>
  **What default custody does NOT cover — say it plainly.** Option A protects excellently against the three likely breaches (stolen disk, stolen backup, leaked DB creds). It does **not** protect against app-server RCE, full compromise (KMS + DB + app), or a privileged insider with KMS+DB IAM — all of which can decrypt these fields. The step-up gate is an application-layer control enforced by the same backend an attacker would control; there is no cryptographic floor beneath it. The operator's worst fear — *leaked with me named as the source* — lands in this uncovered zone. That gap is precisely what the [vault-mode exception](#the-vault-mode-exception) exists to close, per field, for the buyer who needs it.
</Warning>

  <Accordion title="The hard rule for copy">

Market the **capability** ("gated, logged, per-record, no bulk; nobody browses your book, not even Orbiter"), **never** the impossibility ("Orbiter cannot decrypt" / "not even we can read it"). ## Architecture

### Where the data lives

<Tabs>
  <Tab title="AlloyDB">
    Authoritative store for the encrypted field. Holds **ciphertext**, the **wrapped DEK**, the **GCM nonce**, and the **blind index** (HMAC for dedup/linking). The plaintext column on `contacts` is nulled when a field is locked and replaced with a reference to the secret row.
  </Tab>
  <Tab title="FalkorDB">
    Stores only the **token / blind index** on the contact node — never the raw value. Graph traversal, shared-number edges, and suggestions all operate on the token. A breach of the graph yields nothing recoverable.
  </Tab>
  <Tab title="Cloud KMS">
    Holds the **KEK** (key-encrypting key) that wraps per-record DEKs. Separately holds the **blind-index key** (Orbiter-managed, distinct from the KEK — see below). For the enterprise tier, the KEK is EKM-backed.
  </Tab>
  <Tab title="WorkOS">
    Mints the short-lived **step-up session** on a passkey/MFA challenge. The backend demands a valid step-up before it will call KMS unwrap or return plaintext.
  </Tab>
</Tabs>

### Key hierarchy

Two levels. A **per-tenant KEK** lives in Cloud KMS and is the encryption boundary. **Per-record DEKs** (AES-256-GCM via Tink) encrypt each field, are wrapped by the KEK, and stored alongside the ciphertext. The unwrapped DEK caches in app memory with a short TTL; wrap/unwrap is the only operation that touches KMS.

<Info>
  **The blind-index key is decoupled from the KEK — deliberately.** Dedup and shared-number graph linking run on `HMAC(normalize(value))`. If that HMAC used the tenant KEK, revoking the KEK would stall dedup and enrichment. So the blind-index key is **Orbiter-managed in Cloud KMS, separate from the encryption KEK**. Consequence: when a key is revoked the value is unreadable but HMAC matching still works structurally — Orbiter can *confirm two records share a phone* but cannot *recover the phone*. For Tier 2 threat models that's the right trade; it's also the one place the "Orbiter can derive nothing" claim carries an asterisk.
</Info>

### The blind index is not a secret — and what that costs

The token on the graph node and the `blind_index` column is **deliberately unencrypted** — it has to be, or it couldn't do its job.

<Info>
  **Why it sits in the clear.** The token is `HMAC-SHA256(normalize(value), index_key)`, and it lives **unencrypted in two places**: as a property on the FalkorDB contact node and in the AlloyDB `blind_index` column. That is the point — dedup, shared-number edges, and suggestions are all equality joins on this value, so it must stay directly comparable. The real number lives elsewhere, as AES-256-GCM ciphertext (per-record DEK wrapped by the per-tenant KEK).
</Info>

Finding the token is usually **harmless**, and for the three likely breaches (stolen disk, stolen backup, leaked DB creds) it is. The index is **one-way** (HMAC is not reversible), it is **keyed** (the `index_key` lives in Cloud KMS, not the database), and the value it stands for is stored separately as ciphertext. An attacker who dumps the column gets a list of opaque tags and no way to turn them back into numbers.

<Warning>
  **The low-entropy dictionary attack — the one breach this column does not survive.** Phone numbers are low-entropy: ~10^10 globally, far fewer in a single region. A deterministic keyed index over a domain that small is offline dictionary-attackable **the moment the `index_key` leaks** — and the realistic way it leaks is app-server RCE, because the app *must* hold the key to compute matches. With the key, an attacker computes `HMAC(every plausible number, index_key)` offline and matches against the stolen tokens, recovering numbers **without ever touching the ciphertext or the DEK**. This side-channel bypasses the field encryption entirely.

  The two first-instinct mitigations — truncate the HMAC for k-anonymity, and treat the column as Tier-2 — are real layers but **not the fix**. Truncation buys almost nothing once the key leaks (the domain is trivially enumerable, so the attacker still recovers each token's small k-bucket) and it degrades the dedup precision you need. Column-gating helps against passive DB leaks, but the same RCE that steals the key usually also reads the column.
</Warning>

The strongest defensible fix changes **where the key lives**, so there is no offline attack at all. Ship the cheap, real layers in P1; reserve the heavier options.

| Layer | What it buys | The tradeoff |
| --- | --- | --- |
| **1. KMS-MAC blind index** *(P1 keystone)* | Move the HMAC into a Cloud KMS **MAC key** (`purpose=MAC`, `HMAC_SHA256`, `MacSign`). The raw `index_key` never enters the app process, so RCE can no longer grind offline. | Converts the offline attack to a **rate-limited online oracle**, not zero — pair with tight KMS quota + audit alerts. One round-trip per uncached lookup. |
| **2. Per-tenant pepper / domain separation** *(P1)* | Distinct index sub-key per tenant+field; kills cross-tenant correlation and global precomputed dictionaries; clean per-tenant crypto-shred. | Necessary, not sufficient on its own; conflicts with *cross-tenant* shared-number edges if you ever want them. |
| **3. Three separate KMS keys** *(P1, already done)* | Keep `index_key`, per-record DEK, and per-tenant KEK distinct so KEK rotation/crypto-shred never disturbs dedup. | None — preserve the current design. |
| **4. Truncation + column-gating** *(P1, as layers)* | Blurs frequency/fingerprint leakage and shrinks the token-exfil surface. | Mitigation only; does **not** defeat a leaked key. Frame honestly, never as the fix. |
| **5. OPRF / POPRF service** *(reserve)* | **Eliminates** the offline attack with input-privacy from the key holder; POPRF adds bulk key-rotation via update tokens (the one thing KMS-MAC lacks). | Self-hosting + hardening a **second trust domain** is real ops overhead, and solves input-hiding you don't need (you own both endpoints). |
| **6. Vaulted random tokenization** *(nuclear)* | Random keyless token → **no key, no function to invert** → offline attack eliminated unconditionally. | A **vault round-trip on every insert** and a new single crown-jewel datastore to guard as hard as the ciphertext. |

<Info>
  **Ship this for P1: KMS-MAC as the keystone.** Replace the in-process `HMAC(value, index_key)` with a Cloud KMS `MacSign(keyVersion, normalize(value))`. HMAC stays deterministic, so the returned tag is byte-identical for identical inputs — dedup, shared-number edges, and suggestions work **unchanged**, with no change to the query model. The win: an app-server RCE that steals process memory now gets **no key**, only throttled `MacSign` calls that are IAM-scoped, per-second rate-limited, fully recorded in Cloud Audit Logs, and instantly revocable. The unbounded offline 10^10 grind becomes a throttled, observable, killable online channel. Cost is single-digit cents at this tier (~$0.06/mo per software key version, $0.03/10k ops; ~1,000 QPS default quota). This is the productized version of the OPRF idea, in-stack, with zero extra services to run.
</Info>

<Warning>
  **Rotation breaks equality if you're naive about it.** `MacSign` signs with the key's **primary** version, so rotating the key changes the tag for the same input and breaks matching against old rows. Pin the `keyVersion`, store it alongside each `blind_index`, and lazy-migrate on next touch — or treat the index key as long-lived and rotate only on suspected compromise via a one-time full re-index. Unlike the KEK, the **index key should rotate rarely**, precisely because rotation forces a re-index. (This is why decoupling `index_key` from the KEK is correct: the KEK can crypto-shred and rotate freely without disturbing dedup.)
</Warning>

<AccordionGroup>
  <Accordion title="Strongest crypto answer (reserve): an OPRF/POPRF service">
    An RFC 9497 (V)OPRF — pairing-free 3HashSDHI via Cloudflare CIRCL in Go — derives the index as `F(skS, value)` where `skS` lives in a separate trust domain the app never holds. The app blinds the input, the key server applies its secret, the app unblinds a deterministic output. Because the app can never evaluate the PRF offline, RCE yields at most a rate-limited online oracle, and determinism/dedup are fully preserved. POPRF's standout property is **bulk key rotation via update tokens** — re-key every stored value without re-reading plaintext — which KMS-MAC lacks.

    The catch for a solo operator: there is **no managed OPRF on GCP**, so you must run and isolate the key-holding server yourself. If *that* server is compromised the key leaks and you're back to the offline attack. The obliviousness (hiding the input from the evaluator) is defense-in-depth you don't strictly need here — you already trust your own backend with plaintext at ingest. Reserve as the high-assurance upgrade, not the v1.

    **Do not** cite RFC 9497 §7.2.3 as authority that OPRFs "offer no protection against low-entropy inputs" — §7.2.3 is about recovering the *server's* private key via accumulated queries. The correct general principle (a queryable oracle still permits *online* low-entropy guessing, so rate-limit and rotate) holds independently.
  </Accordion>
  <Accordion title="Nuclear alternative: vaulted random tokenization">
    Generate a **random** token per distinct value and store the value↔token mapping in a separate hardened vault. The token has **zero mathematical relationship** to the number — there is no key and no function to invert — so the offline dictionary attack is eliminated even with unlimited compute. Dedup survives because the vault does deterministic insert-or-return-existing.

    The cost is structural: every dedup/edge now needs a **vault round-trip** instead of a local compare, and the vault becomes the single most valuable target — PCI's "most lucrative target." You've relocated the crown jewel, not removed it, and an RCE'd app with vault read access can dump mappings directly unless you gate it. Justified only if you must contractually prove tokens are unconditionally irreversible.
  </Accordion>
  <Accordion title="Explicitly rejected">
    - **Format-Preserving Encryption (FF1/FF3-1):** the key still lives in app memory (same leak path), it's **reversible** (a leaked key decrypts directly — worse than one-way HMAC), and small phone domains sit near or below NIST's safe bounds with known cryptanalysis (Durak–Vaudenay broke FF3). Rejected.
    - **Searchable Symmetric Encryption (SSE):** research-grade, no GCP fit, leaks search/access patterns, and doesn't match graph node-equality. Rejected.
    - **Frequency-hiding (FH-OPE):** equal values no longer collide to one token, which **breaks** the deterministic dedup and shared-number edges we depend on. Skip unless frequency analysis on shared celebrity numbers becomes an explicit threat.
  </Accordion>
</AccordionGroup>

<Note>
  **Net.** Truncation alone is not a real fix for the RCE/key-leak threat. The defensible P1 stack is **KMS-MAC blind index + per-tenant domain separation + tight quota/audit alerting**, with truncation and column-gating kept as honest secondary layers. It defeats the *offline* attack while preserving every dedup, rotation, and crypto-shred property — on the stack we already pay for. OPRF/Pythia is the documented north star; vaulted tokenization is the nuclear option that eliminates the attack but recreates a single crown-jewel store.
</Note>

### The reveal flow

```mermaid
sequenceDiagram
    actor U as Operator
    participant FE as TanStack/React
    participant API as Go Backend
    participant WOS as WorkOS
    participant KMS as Cloud KMS
    participant DB as AlloyDB

    U->>FE: Tap reveal on locked field
    FE->>API: POST /vip/reveal (contact_id, field)
    API->>API: Check active step-up session
    alt No valid step-up
        API-->>FE: 401 step_up_required
        FE->>WOS: Trigger passkey / MFA challenge
        WOS-->>FE: Step-up token
        FE->>API: Retry reveal + step-up token
        API->>WOS: Verify step-up token
    end
    API->>DB: Fetch ciphertext + wrapped_dek + nonce
    API->>KMS: Unwrap DEK (tenant KEK)
    KMS-->>API: Plaintext DEK (cached, short TTL)
    API->>API: AES-256-GCM decrypt
    API->>DB: Write reveal audit row
    API-->>FE: Plaintext — single field, single use
    FE->>U: Display value; padlock stays closed
```

**Step-up session window.** A passkey tap mints a short elevated session (default proposed 5–15 min). Inside the window, reveals don't re-prompt — so working a list of ten artists isn't ten passkey challenges — but **each reveal is still individually logged**. This is the answer to "auth once, get everything": not unlock-all-forever, but a bounded working session, with per-field audit intact. Mental model is Face ID, not a vault you crank open each time.

### What stays intact

Locking does **not** break the graph. On lock, the field moves to ciphertext + blind index; dedup and shared-number edges keep running on the HMAC. A locked contact still links, still dedupes, still surfaces in suggestions — the only change is the field is reveal-gated. Locking is cheap: no intelligence lost.

Locking does **not** scrub the past. If a contact lived as plaintext before being locked, prior backups still hold the plaintext until they age out. Lock protects forward in the live system. The fully-clean version is contacts **born locked** — tokenized at ingestion when the source is a known VIP (see [Open decisions](#open-decisions)).

</CodeGroup>

**Crypto-shred semantics**

- **Per-record (live):** null `wrapped_dek` on the secret row → the DEK is gone, ciphertext is unrecoverable in the live DB. Orbiter does this unilaterally — clean GDPR erasure, no tenant involvement. Does **not** cover backups (the wrapped DEK still sits in snapshots until they age out).
- **Per-tenant (everywhere):** destroy the KEK → every record under it goes dark, *including backups*, because the wrapped DEKs inside them can never be unwrapped again. This is the only mechanism that makes "erased everywhere, including backups" literally true.

Default topology: **per-tenant KEK + per-record DEK** gives both live per-subject shred and tenant-wide backup shred. Per-VIP KEKs are reserved for tenants who contractually demand subject-level *backup* erasure (key sprawl is the cost).

### 2 — Backend (Go)

Use `tink-go` + `tink-go-gcpkms`. Don't hand-roll GCM — nonce-reuse bugs live there. `KmsEnvelopeAead` wraps DEKs through KMS; EKM routing is transparent once the key URI points at an EKM-backed key.

```go service.go (sketch)
// Lock: plaintext field -> ciphertext + blind index + token. No key needed to lock.
func (s *VIPService) Lock(ctx context.Context, contactID uuid.UUID, field, plaintext string) error {
    norm := normalize(plaintext)                       // E.164 for phone
    bi   := s.blindIndex.HMAC(norm)                    // Orbiter-managed index key
    ct, dek, nonce, err := s.envelope.Encrypt(ctx, []byte(plaintext), s.kekURI(tenantOf(ctx)))
    // ... persist secret row, null contacts.phone, set phone_locked=true,
    //     write token (blind index) to FalkorDB node, audit 'lock'
}

// Reveal: requires a valid step-up session. Single field, single use, always audited.
func (s *VIPService) Reveal(ctx context.Context, contactID uuid.UUID, field string) (string, error) {
    if !s.stepUp.Valid(ctx) {
        return "", ErrStepUpRequired                   // -> 401 step_up_required
    }
    sec := s.repo.GetSecret(ctx, contactID, field)
    if sec.WrappedDEK == nil {
        return "", ErrShredded                          // crypto-shredded
    }
    pt, err := s.envelope.Decrypt(ctx, sec.Ciphertext, sec.WrappedDEK, sec.Nonce, sec.KEKURI)
    s.audit.Write(ctx, sec.ID, "reveal", s.stepUp.ID(ctx))
    return string(pt), err
}

// Shred (live): destroy the DEK. Unlock: deliberate, step-up + confirm, re-materializes plaintext.
```

Endpoint surface:

<ResponseField name="POST /vip/lock" type="auth: session">
  Encrypts and tokenizes a field. Free — no step-up required to lock.
</ResponseField>
<ResponseField name="POST /vip/reveal" type="auth: step-up">
  Returns plaintext for one field. `401 step_up_required` if no valid elevated session. Always writes an audit row.
</ResponseField>
<ResponseField name="POST /vip/unlock" type="auth: step-up + confirm">
  Removes protection, returns the field to plaintext. Rare, gated, confirmed.
</ResponseField>
<ResponseField name="POST /vip/shred" type="auth: step-up + confirm">
  Destroys the per-record DEK. Irreversible in the live store.
</ResponseField>

**Operational guardrails.** Rate-limit and alert on decrypt volume — anomalous bulk decrypt is the breach signal. Cache the unwrapped DEK with a short TTL only. The enrichment waterfall (PDL → Proxycurl → …) needs plaintext transiently; make the enrichment service the sole component authorized to dereference and call external APIs, and never write plaintext back to the primary store — rather than widening decrypt rights across general workers.

### 3 — Frontend (TanStack / React)

Component tree on the contact side panel:

```
<VIPField>                     // wraps a sensitive field; renders masked or revealed
 ├─ <PadlockToggle/>           // three-state icon: open / closed-masked / closed-revealed
 ├─ <MaskedValue/>             // •••• rendering when locked + not revealed
 └─ <RevealedValue/>           // shown value + auto-hide timer

Hooks:
  useReveal()                  // TanStack mutation: reveal + step-up challenge handling
  useStepUpSession()           // elevated-session state: isElevated, elevate(), msRemaining
  useLockField() / useUnlockField() / useShredField()
```

Key behaviors:

<Steps>
  <Step title="Padlock click = reveal, not unlock">
    On a closed padlock, the click fires `useReveal()`. **Never** demote to plaintext on a single click. Unlock is a separate menu item wired to `useUnlockField()` behind a confirm dialog.
  </Step>
  <Step title="Reveal handles the step-up round-trip">
    `useReveal()` calls `/vip/reveal`. On `401 step_up_required`, it triggers the WorkOS passkey challenge, then retries with the step-up token. On success, the value renders and an auto-hide timer starts (returns to masked on timeout or navigation).
  </Step>
  <Step title="Session window suppresses re-prompts, not logging">
    `useStepUpSession()` tracks the elevated window. While `isElevated`, subsequent reveals skip the challenge — but each still hits `/vip/reveal` and writes its own audit row. Surface `msRemaining` as a subtle countdown so the operator knows the window is open.
  </Step>
  <Step title="Masked by default, locked is sticky">
    A locked field renders masked on every load. The component never holds plaintext beyond the reveal lifecycle — no plaintext in component state after auto-hide, no plaintext persisted to any client cache.
  </Step>
</Steps>

<Warning>
  **No client-side bulk reveal.** Do not build a "reveal all" affordance, a copy-all, or anything that batches reveals into an export. That capability *is* the breach this feature exists to prevent. The absence is intentional and is itself part of the trust story.
</Warning>

### Phasing

| Phase | Scope | Operator claim |
|-------|-------|----------------|
| **P1** | Manual padlock, per-tenant KEK (Orbiter-managed in Cloud KMS), reveal-on-demand, step-up session, per-record + per-tenant shred, audit log | "Tokenized, encrypted, gated, logged — nobody browses your book, not even Orbiter." |
| **P2** | Born-locked ingestion: auto-tokenize known VIPs at ingest; "this looks high-profile — lock it?" auto-suggest | Forward + backward clean for VIPs onboarded after P2. |
| **P3** | Per-VIP external keys + customer-facing key-lifecycle / reveal-audit dashboard | Subject-level backup erasure; provable access history. |

Rotation is cheap throughout — rotating a KEK rewraps DEKs, it does not re-encrypt data; Tink handles KEK versioning, so it's a background rewrap job, not a migration.

## Open decisions

<AccordionGroup>
  <Accordion title="Session window length — the UX-critical dial">
    Too short and a power operator working a list feels nagged; too long and "step-up" becomes theater. Proposed default 5–15 min; tune from pilot feedback rather than fixing it now. This is the single parameter that determines whether the feature feels secure-normal or clunky.
  </Accordion>
  <Accordion title="Blind-index key ownership">
    Confirmed Orbiter-managed (separate from KEK) so dedup survives KEK revocation. Accept the asterisk: Orbiter can confirm two records share a value but cannot recover it. Decide how literal the marketing claim must be before locking copy.
  </Accordion>
  <Accordion title="Born-locked ingestion (P2)">
    Auto-tokenize at ingest when the source flags a known VIP, avoiding the "plaintext lived in yesterday's backup" gap. Needs a VIP-detection signal — roster match, manual flag, or an enrichment heuristic. Manual padlock ships first; assisted later.
  </Accordion>
  <Accordion title="KEK granularity — per-tenant vs per-VIP">
    Per-tenant KEK + per-record DEK is the default (both live and backup shred covered at their respective levels). Per-VIP KEKs only for tenants who contractually require subject-level *backup* erasure. Don't pay the key-sprawl cost by default.
  </Accordion>
  <Accordion title="BYOK boundary (operator vs enterprise)">
    The operator tier is **not** BYOK — the operator does not hold a key. "Only you hold the key, Orbiter literally cannot decrypt" is the **enterprise/Atlas** claim (EKM-backed tenant KEK). Same crypto spine, two narratives: enterprise buys control of the *key*; the operator buys control of the *reveal*. Keep copy disciplined so the gap never appears on the $100 tier.
  </Accordion>
</AccordionGroup>

## Out of scope (deliberate non-features)

- **Bulk reveal / export of VIP fields** — the capability this feature exists to prevent.
- **At-rest CMEK as the differentiator** — CMEK is the Tier 0/1 baseline; it is instance-scoped and decrypts transparently, so it is not the "Orbiter can't decrypt" control.
- **Enterprise BYOK/HYOK** — tracked separately as the Atlas upsell; this page is the Tier 2 operator surface.
- **Blockchain / on-chain storage of VIP data** — confidentiality and deletability are the whole point; a chain offers replication, transparency, and immutability — the opposite on every axis. On-chain ciphertext (or the low-entropy blind index) would be public and brute-forceable *forever*, and immutability breaks crypto-shred, GDPR erasure, and the kill switch. The data never goes on a chain.

<Note>
  **What's worth borrowing from the blockchain pitch.** The one safe, valuable on-chain-shaped artifact is a **tamper-proof reveal ledger** — hashes of reveal-audit events only, never PII — backing the claim *"even Orbiter can't edit or erase the record of who looked at your number,"* which targets the operator's real fear (Orbiter as the silent leak vector). Under the hood that's a **transparency log** (Merkle tree); a public-chain anchor of the Merkle root is optional and only earns its keep as a crypto-native vanity SKU. Because only event hashes are anchored, it stays erasure-safe — the number itself is still crypto-shreddable. **Sell the property, never the buzzword.** The genuinely "decentralized" upgrade is **threshold / MPC key custody** — no single party (including Orbiter) holds the whole key, the [BYOK boundary](#open-decisions) without the device-loss footgun — tracked as an enterprise/research item, not a chain of contacts.
</Note>
