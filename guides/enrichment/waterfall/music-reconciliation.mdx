## Proposed flow â€” Person:Music_Artist merge

1. **Select:** `mb_artist` rows with `artist_type = 'Person'`, `node_uuid` set, not yet reconciled (needs a tracking column â€” see open questions).
2. **Tier 1 â€” hard keys:** `url-rels` contains `linkedin.com/in/` â†’ `mvp/resolve/match-master-person` (13038). Cheapest definitive match.
3. **Tier 2 â€” shared links:** any `url-rels` URL hits `master_link` (the 13039 pre-API pattern, adapted).
4. **Tier 3 â€” name embedding:** KNN via 12566 against `Person` nodes. Candidates below the distance threshold go to Tier 4; no candidate â†’ no match, stop.
5. **Tier 4 â€” LLM confirmation:** two-tier confirm pattern from the merge-detection pipeline before any merge fires.
6. **On confirmed match:** `graph-merge-person` (12949) + `resync-person-node-uuid` (12964); the surviving `Person` node keeps/gains the `Music_Artist` label and music properties.

## Proposed flow â€” Music_Label â†’ Company SAME_AS

1. **Select:** `mb_label` rows with `node_uuid` set and `master_company_id = 0`.
2. **Tier 1 â€” domain:** `url-rels` `official homepage` â†’ domain â†’ `master-company-new` (12558, `hold_downstream=true`).
3. **Tier 2 â€” name embedding:** KNN via 12566 against `Company` nodes, LLM-confirmed.
4. **On match:** stamp `mb_label.master_company_id` + write `SAME_AS` w1 with `match_method` (`domain` / `embedding+llm` / â€¦) and `confidence` (0â€“1) â€” the [edge spec](/guides/enrichment/waterfall/music-edges#identity-bridges) already carries both params for auditability.

## Open design questions

These need decisions before build (the same design-first gate as the [original six](/guides/enrichment/waterfall/music-industry-waterfall#resolved-design-decisions-2026-06-11)):

1. **Auto-merge vs review queue.** Person merges are destructive-ish (edges repoint). Do Tier 3/4 (embedding + LLM) matches merge automatically, or land in a review table for human confirmation with only Tier 1/2 (hard-key) matches auto-merging?
2. **Match-state tracking on `mb_artist`.** `mb_label` has `master_company_id`; `mb_artist` has no equivalent. Add `master_person_id` (tableref, `0` = unreconciled) â€” and a `reconcile_status` (`pending` / `matched` / `no_match` / `review`) so no-match rows aren't rescanned forever?
3. **Embedding threshold.** 12566's general threshold is 0.65. Person-name collisions (common names) argue for a stricter bar + mandatory LLM confirm; what's the floor?
4. **Create-on-miss for companies.** 12558 *creates* a `master_company` when nothing matches. Should an unmatched `Music_Label` create a new company (Sub Pop deserves one) or only link to existing companies in v1?
5. **Embedding source mismatch.** Node descriptions (13113) now overwrite `name_embedding` with description-based vectors for `Music_Group`/`Recording` â€” person artists keep name+disambiguation embeddings from 13092. Reconciliation KNN should run against `Person` embeddings, which are name-based â€” consistent today, but worth pinning before descriptions expand to more node types.
