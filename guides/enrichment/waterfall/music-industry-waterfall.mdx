## Source: the MusicBrainz API

| Property | Value |
|---|---|
| Base | `https://musicbrainz.org/ws/2/` |
| Format | append `?fmt=json` (or `Accept: application/json`) |
| Auth | none required |
| **Rate limit** | **1 request/second, global**, with a meaningful `User-Agent` (e.g. `Orbiter/1.0 ( mark@orbiter.io )`). Exceeding it returns `503`. |
| Lookup | `/{entity}/{MBID}?inc={subqueries}` |
| Browse | `/{result}?{linked}={MBID}&limit=100&offset=N&inc=...` |
| Search | `/{entity}?query={lucene}&limit=N` |
| Pagination | `limit` (default 25, **max 100**) + `offset`; increment by results returned |

**`inc=` is the efficiency lever.** A single lookup can pull aliases, tags, genres, ratings, and every relationship type (`artist-rels`, `recording-rels`, `work-rels`, `release-rels`, `label-rels`, `event-rels`, `place-rels`, `area-rels`, `url-rels`) in **one** request. One artist + two browse pages ≈ 3 requests ≈ 3 seconds. This is why the cascade must be tightly depth- and tier-capped.

<Warning>
**The 1 req/s ceiling is global, not per-cron.** Four independent queue crons each firing lookups would collectively blow the limit. The design routes **every** MusicBrainz call through one token-bucket throttle and a **single unified queue processor** — see [Rate-limit architecture](#rate-limit-architecture). Realistic throughput is **~1 entity per 3–6 seconds**, so this pipeline is deliberately slow and bounded.
</Warning>

## Core entities → graph nodes

MusicBrainz's 13 core entities map to Orbiter graph nodes as follows. **Seed entities** (minimum bootstrap set) are **artist, recording, label, event**; "cascade" entities get their own queued lookup; inline entities/values are handled from data embedded in other lookups — no queue, no extra API call. Places can materialize inline as nodes; areas reuse Orbiter geography; genres stay properties on the source music node.

| MusicBrainz entity | Graph node label | Seed? | Bridges to existing Orbiter node |
|---|---|:--:|---|
| `artist` (type = Person) | `Entity:Person:Music_Artist` | ✅ | Person-shaped music node; depth-0 person-resolution can bridge to canonical `master_person` via `SAME_AS` |
| `artist` (Group / Orchestra / Choir) | `Entity:Music_Group` | ✅ | — (new node type — a band is never a `Person`) |
| `recording` | `Entity:Recording` | ✅ | — |
| `label` | `Entity:Company:Music_Label` | ✅ | Company-shaped; optional `SAME_AS` to canonical `Company` |
| `event` | `Entity:Event` (+ `Music_*` type sub-label) | ✅ | Shared Event model |
| `work` | `Entity:Work` (+`:Song` etc.) | cascade | — |
| `release-group` | `Entity:Release_Group` (+`:Album`/`:Single`/`:EP`) | cascade | — |
| `release` | `Entity:Release` | cascade | — |
| `place` | `Entity:Place` (+`:Studio`/`:Venue`) | inline | — |
| `area` | reuse `Country` / `Region` / `City` (MERGE by name; stamp `mbid`) | inline | Orbiter geography |
| `genre` | property on source music node | inline | — |
| `series` / `instrument` / `url` | properties / deferred | — | — |

Every node follows Orbiter conventions: base `Entity` label, `uuid` (random) **plus `mbid`** as the natural MERGE key, `display_label`, `visibility`, dual vectors (`name_embedding` from `openai/text-embedding-3-small` via #4676 and `embeddings` from `google/gemini-embedding-001` via #13114), and `created_at` / `updated_at` stamped via Cypher `timestamp()` on create and update (see [Node Types](/guides/ontology/nodes) and the [timestamp convention](/guides/ontology/nodes#timestamp-coverage-audit)). Generated `description` currently exists only for `Recording`, `Music_Group`, and `Music_Artist`; other music node types vectorize `name + disambiguation`.

## Edge designs

Weights follow the canonical [Edge Weights](/guides/ontology/edge-weights) scale (**lower = stronger**; hub-node edges sit Weak or below; never sub-1). Every edge stamps `uuid`, `created_at`, `updated_at` via `timestamp()`. The bands: Defining 1–5, Very strong 6–15, Strong 16–25, Moderate 26–40, Weak 41–65, Negligible 100.

<Note>
The tables below summarize each edge. For the **full parameter list per edge** — including the standard envelope (`uuid`, `weight`, `created_at`, `updated_at`, `description`, `description_generated_at`) carried by every edge — use the canonical ontology section: [MusicBrainz Edges](/guides/ontology/edges#musicbrainz). **`Artist`** in a pattern means `Person:Music_Artist` or `Music_Group`; `MEMBER_OF` is always `(Person:Music_Artist)→(Music_Group)`. The docs now list 25 edge types because that is the description allowlist; 21 are actively forward-emitted today, while `COLLABORATED_WITH`, `SUPPORTING_MUSICIAN_FOR`, `TRIBUTE_TO`, and `FROM_AREA` need forward writers.
</Note>

## Pipeline architecture

Mirrors IMDB: **gates** → **orchestrators** → **inlined cascades** → **queue infra** → **unified cron**, all under a **namespace `mb/`** in Xano workspace 3. No LLM/scrape phases in the structural path — orchestrators map structured JSON straight to Cypher (an LLM pass fills edge `description`s afterwards).

<Note>
**Built and live-audited 2026-07-01.** The forward-path function inventory lives on the dedicated **[Music Waterfall Functions](/guides/enrichment/waterfall/music-functions)** page. Backfill-only functions/endpoints/tasks are intentionally excluded there too.
</Note>

### Entry gates

<Card icon="play">

```
mb/get-add/label — 13087
mb/get-add/artist — 13088
mb/get-add/release-group — 13089
mb/get-add/recording — 13093
mb/get-add/work — 13098
mb/get-add/event — 13106
```

One gate per enrichable entity, including event. Each: normalize MBID → dedup on a 30-day `enriched_at` window → skip `gone`-tombstoned rows → ensure stub row (`mb_artist` / `mb_recording` / …) → branch: **sync-enrich** when `cascade_depth == 0 && queue == false` (invoke the orchestrator directly), else **upsert the queue row**. Seed-id propagation: when no caller seed is passed, the just-created stub is the seed. The MBID is the natural key — no URL normalization needed (unlike IMDB).

</Card>

### Cascade helpers (depth-capped fanout)

<Card icon="wrench">

```
cascading — inlined in each orchestrator's mb_cascade phase (no standalone functions)
```

**The standalone `mb/cascade-from-*` helpers were not built.** Each orchestrator carries its own `try_catch`-wrapped `mb_cascade` phase instead. Depth guarding (`cascade_depth + 1 >= 3` → no queue) and within-run MBID dedup (first tier wins) work as designed; collaborator fanout is deferred to v2.

</Card>

## Rate-limit architecture

The defining constraint. Every MusicBrainz call passes through a token bucket, and **one** cron drains all queues so total throughput never exceeds 1 req/s.

<Card icon="gauge-high">

```
mb/rate-limit/acquire — 13079
```

A token gate backed by the singleton `mb_rate_limit` row (#736: `last_call_at`, `tokens`). Every MusicBrainz call (via `mb/api/request`) calls `acquire` first: if `now - last_call_at < 1100ms` it **sleeps 1 s** (a simplification of the designed defer-and-requeue — requeue backoff still exists at the queue layer via `next_attempt_at`), then stamps `last_call_at = now`. **Fail-open**: if the gate errors, the call proceeds and the failure is logged. Centralizes the limit across every function that touches the API.

</Card>

<Card icon="clock-rotate-left">

```
process-queue-musicbrainz — Task 180   (INACTIVE)  ·  mb/queue/process-next — 13102
```

**One unified cron**, not four. Each invocation of `mb/queue/process-next` (13102, v1.1) selects the single highest-priority pending row **across all six queues** — order `priority_tier ASC, cascade_depth ASC, entity_rank ASC, id ASC` (entity_rank: artist 1 · label 2 · event 3 · work 4 · release_group 5 · recording 6), skipping rows with a future `next_attempt_at` — locks it, dispatches the matching `run-base-*` orchestrator, deletes on success / `retry_count++` with a 5 min × retries backoff on failure (dropped after 5). **Schedule deviation:** the design called for a ~2 s tick, but Xano cron granularity is 60 s — so Task 180 runs every **60 s and drains up to 10 rows per tick**, roughly the same throughput. The task **shipped INACTIVE**; pausing it is the pipeline's manual kill. A unified cron makes the global 1 req/s trivially enforceable; four parallel crons would not.

</Card>

<Warning>
**Throughput is intentionally low.** At ~1 entity / 3–6 s, a depth-0 artist with its top recordings + albums + members takes minutes, and a deep cascade takes hours — by design. The depth cap (2), tier caps, and per-entity caps below keep total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.
</Warning>

## API errors, retries & rate limits

Reference for handling the MusicBrainz `ws/2` API. Every call to musicbrainz.org goes through `mb/rate-limit/acquire` (pacing) **and** a shared retry wrapper (`mb/api/request`, below) that applies this policy — orchestrators never call `api.request` directly.

<Warning>
**A meaningful `User-Agent` is mandatory.** Requests without one (or with a generic library default) are throttled hard or blocked. Use `Orbiter/1.0 ( mark@orbiter.io )`. There is no API key.
</Warning>

### Status codes

| Status | Meaning | Retry? | Handling |
|---|---|:--:|---|
| `200` | OK | — | parse JSON |
| `400` | Bad request — malformed MBID / Lucene query | ❌ terminal | `log_crash` `phase: api_bad_request`; drop the queue row |
| `404` | Entity not found / merged away | ❌ terminal | stamp `enrich_success = false`, mark stub `gone`, drop row |
| `429` / `503` | **Rate limited / temporarily unavailable** | ✅ backoff | honor `Retry-After`; exponential backoff — the common case |
| `500` / `502` / `504` | Server / gateway error | ✅ backoff | transient — retry with backoff |
| timeout / network | no response | ✅ backoff | retry with backoff |

<Note>
MusicBrainz returns **`503` for both** genuine overload **and** rate-limit violations (it does not consistently use `429`). Treat `503` as "back off and retry," never as a hard failure.
</Note>

### Retry policy

```text
max_attempts   = 5
base_delay     = 1000 ms                       # ≥ the 1 req/s floor
backoff        = base_delay * 2^(attempt-1)    # 1s → 2s → 4s → 8s → 16s
jitter         = ± 250 ms                       # avoid herd on the shared cron
respect        = Retry-After header when present (overrides computed delay)
terminal       = {400, 404}                     # never retried
on_exhaustion  = release lock · log_crash phase="api_exhausted" · re-queue one tier lower
```

- **Live implementation uses bounded sleeps inside the API helper.** `mb/rate-limit/acquire` sleeps when the global 1 req/s floor has not elapsed, and `mb/api/request` sleeps between its internal retry attempts. Queue-level failures still use `next_attempt_at`, so the unified cron skips rows whose full enrich retry window is in the future.
- **Backoff is per-entity; pacing is global.** The 1 req/s token bucket governs *spacing* between any two calls; the API retry backoff governs *re-attempts* of one failed HTTP call; the queue backoff governs failed enrich rows. All three invariants hold simultaneously.
- **Retries are idempotent.** Every write is `MERGE`-based with `coalesce`d `created_at`, so re-running a partially-applied enrich never duplicates nodes or edges.

<Card icon="cloud-arrow-up">

```
mb/api/request — 13080
```

Single choke-point for every MusicBrainz call: calls `acquire` (13079) for pacing → issues the request with the canonical `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`) and `fmt=json` appended → classifies the response per the table above → retries up to **5 attempts with backoff sleeps** between attempts (none after the final), `400`/`404` terminal. Returns `{ok, terminal, status, attempts, json}`; failures `log_crash` with phases `api_bad_request` / `api_exhausted`. Centralizing here means rate-limit + retry behavior lives in exactly one place, and the orchestrators stay simple.

</Card>

## Priority tiers

<Note>
**Tier 0 — bootstrap lane (decided).** Large manual seed batches (e.g. a full label roster) are queued at `priority_tier = 0`, a reserved lane that drains ahead of all cascade traffic. There is no offline dump import — everything rides the live 1 req/s API, and multi-day bootstraps are accepted (see [Resolved](#resolved-design-decisions-2026-06-11)). The tiers below (1–4) apply to cascade-discovered entities.
</Note>

### `queue_mb_artist` (1 = highest)

| Tier | Rule |
|:--:|---|
| **1** | Band member of a seed group; main performer on a seed recording/event; composer of a seed work |
| **2** | Collaborator; producer/engineer on a seed recording; supporting musician |
| **3** | Co-performer at a seed event; artist credited on a label-discovered release |
| **4** | Everyone else mentioned |

### `queue_mb_recording` (1 = highest)

| Tier | Rule |
|:--:|---|
| **1** | Recording where the seed artist is the **main** performer; title track of a seed release-group |
| **2** | Other recordings credited to the seed artist |
| **3** | Recording on a release discovered via a seed label |
| **4** | Low-signal (alternate takes, untitled, video) |

### `queue_mb_release_group` / `queue_mb_event`

| Tier | Release-group rule | Event rule |
|:--:|---|---|
| **1** | Album by seed artist; award-winning | Event the seed artist **headlined** |
| **2** | EP / single by seed artist | Event the seed artist performed at (support) |
| **3** | Compilation / live / soundtrack | — |
| **4** | Other appearance | — |

### Cascade filter — artist seed

```text
from artist lookup + browse:
    member of band (each)                          → T1 (artist)
    release-group type=Album (cap 20 by date)      → T2 (release-group)
    recording main-credit (cap 50 by length/date)  → T2 (recording)
    collaborator / supporting musician             → T2 (artist)
    event main performer                           → T3 (event)
    # depth ≥ 1: keep only T1–T2
```

### Cascade filter — recording seed

```text
from recording lookup:
    work (performance)                 → T1 (work)
    main performer artist-credit       → T1 (artist)
    producer / engineer / instrument   → T2 (artist)
    recorded-at place                  → Place node inline (no queue)
```

## Depth model

Identical to IMDB — hard cap at depth 2.

| Depth | Meaning | Behavior |
|:--:|---|---|
| **0** | Seed (user / bootstrap) | Full lookup + all edges + cascade fanout |
| **1** | Direct discovery from a seed | Stub via `get-add`. Queued. Cron enriches + expands, but **only T1–T2 children queue at depth 2** |
| **2** | Transitive | Stub queued. Cron enriches + writes edges, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why cap at 2?** MusicBrainz is ~2.5 M artists / ~30 M recordings. One prolific artist → ~20 albums → ~250 recordings → each recording → 1 work + ~5 credited people → thousands of entities at depth 2. Combined with the 1 req/s limit, an uncapped cascade would never drain. Depth 2 + tier caps + per-entity caps (20 albums, 50 recordings) keep a seed's full expansion in the low thousands of API calls.
</Check>

## Loop prevention (six layers)

1. **Dedup gate in `get-add`:** `mbid` already enriched within 30 days → return existing, no lookup.
2. **Queue upsert:** duplicate `mbid` → `count++`, tier upgrade. One row per MBID.
3. **Depth cap:** depth 2 enriches but does not cascade; depth 3 never queued.
4. **Tier cap at depth ≥ 1:** only T1–T2 discoveries queue at depth 2.
5. **Manual kill (no dedicated switch — decided 2026-06-11):** layers 1–4 make the cascade structurally non-recursive (depth is monotonic with a hard cap; cycles collapse into `count++` on existing rows), so the planned `check-kill-switch-mb` env-var gate was skipped. Pausing the single unified cron task in Xano stops the entire pipeline.
6. **Global rate limiter:** `mb/rate-limit/acquire` + unified cron enforce 1 req/s across the whole pipeline (the layer IMDB doesn't need).

## Seed flows

### Seed artist (depth 0)

<Steps>
  <Step title="Dedup gate">
    Lookup `mb_artist` by `mbid`, return if enriched &lt; 30 days. (No kill-switch check — skipped by design; pause the unified cron to stop the pipeline.)
  </Step>
  <Step title="Throttled lookup">

  </Step>

  </Step>
  <Step title="Membership edges">
    For each `member of band` rel → `MEMBER_OF` edge (instrument/dates/original) + queue the member artist at **T1**.

  </Step>
  <Step title="Mark success">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

## Resolved design decisions (2026-06-11)

The open questions from the original design, each resolved before build (the sixth — places — surfaced while writing the [table spec](/guides/enrichment/waterfall/music-tables)):

<CardGroup cols={1}>
  <Card title="Artist ↔ Person reconciliation — separate post-ingest pass" icon="link">
    **Decided; built.** Ingest creates `Person:Music_Artist` and `Company:Music_Label` nodes keyed on `mbid` — no reconciliation inline. The built person-resolution path (#13118–#13123) bridges depth-0 person artists to a canonical `master_person` with `SAME_AS` and verified social links; label-owner resolution (#13132–#13134) writes `OWNED_BY`. Label→Company `SAME_AS` exists in live code as well. Fixed-forward 2026-07-01: both live `SAME_AS` writers use identity weight `1`.
  </Card>
  <Card title="Area mapping — reuse Orbiter geography" icon="earth-americas">
    **Decided.** MusicBrainz areas MERGE into the existing `Country` / `Region` / `City` nodes by name, stamping `mbid` on match. No dedicated `Music_Area` node type and no mapping table. Unmappable or historical areas fall back to the nearest modern parent (e.g. a historical region maps to its present-day country).
  </Card>
  <Card title="Genre vs tag — genres as properties" icon="tags">
    **Decided.** Curated `genres` and free-form `tags` stay as node properties for search, filtering, facets, and badges. Do not create `Genre` hub nodes or `HAS_GENRE` edges.
  </Card>
  <Card title="Canonical release — one Official release per release-group" icon="compact-disc">
    **Decided.** Each `Release_Group` materializes exactly **one** `Release` node: `status = Official`, earliest release date (any country). Tracklists and `APPEARS_ON` edges hang off that canonical edition. All other pressings remain in the raw JSON on the `mb_*` source tables, so the decision is reversible without re-fetching.
  </Card>
  <Card title="Bootstrap — live API only, priority lane" icon="gauge-high">
    **Decided.** No offline import from the [MusicBrainz database dumps](https://musicbrainz.org/doc/MusicBrainz_Database/Download). Large bootstraps (e.g. a full label roster) ride the same 1 req/s live API, queued at **`priority_tier = 0`** — a reserved bootstrap lane that drains ahead of all cascade traffic. One code path, no dump tooling; multi-day seed loads are accepted.
  </Card>
  <Card title="Places — inline materialization, no queue" icon="location-dot">
    **Decided.** `Place` nodes are written directly during recording/event enrich from the place object embedded in `place-rels` (name, type, address, coordinates, area — everything the node spec needs). No `queue_mb_place`, no place orchestrator, zero extra API calls at the 1 req/s ceiling. The `mb_place` source table stores the embedded JSON and `mbid`, so a future scoped lookup can add URL rels without re-discovery.
  </Card>
</CardGroup>
