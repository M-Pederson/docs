<Warning>
**Matching for primary\_location — logic exists in Xano today, verified 2026-07-15, two gaps found.** `primary_location` needs **no `natural_key` column**: its natural key is already the `(city_id, region_id, country_id)` FK triple, and the matching logic is live in `mvp/node/primary-location-nodes-edges_v2` (4690), called by `add-person-primary-location` (2766) from `process-person-phase-2` (12823). A bare "London" never reaches the table un-disambiguated — Radar geocodes it into components first. Current predicates: country matches on name or alpha2; region is scoped by `country_id`; `primary_location` is looked up on the exact triple and inserted if missing.

Two gaps to fix (planned — not yet applied in Xano):

1. **City lookup same-country collision bug.** The predicate is `city_name matches AND (region_id == $region OR country_id == $country)`. The `OR` collapses same-name cities within one country: London **Kentucky** reuses London **Ohio**'s city row (both match on `country_id == US`), and every Springfield/Portland/Columbus does the same. Fix: match on region when both sides have one; fall back to country only when region is absent.
2. **No unique index on the triple** — verified: `primary_location` (385) and `city` (332) carry only a primary key + `created_at` btree. The query-then-insert dedupe is race-prone (two concurrent phase-2 runs for the same city can double-insert). Add a unique index on `(city_id, region_id, country_id)` after a collision check.
</Warning>

## Points of comparison

A few asymmetries between the two tables worth discussing:

- **Coordinate types differ** — `master_address` stores `longitude` / `latitude` as `text`; `primary_location` uses `decimal`.
- **Normalization differs** — `master_address` stores city / region / country as free text plus codes; `primary_location` references the normalized `city` / `region` / `country` lookup tables.
- **`updated_at`** exists on `master_address` but not on `primary_location`.
- **No uniqueness constraints** on either table beyond the primary key — nothing prevents duplicate addresses or locations for the same entity.

## Proposed: `natural_key` on master\_address — scope

<Note>
**Workshop scope only — no Xano changes made.** This section scopes the canonical-address dedupe key. It reuses the `natural_key` column already planned for `master_address` in the [Universe ↔ Backend sync design](/guides/open-work/backend-refactor-tables) (upsert key `(node_uuid, natural_key)`, one of the four sync columns), so we get one dedupe concept, not two.
</Note>

### Why — today's dedupe is broken in both directions

The two live writer paths use different, conflicting keys (verified against live XanoScript, 2026-07-15):

- `mvp/address/add-company-address` (2451) dedupes on **`(master_company_id, city, state_region)`** — massively **over-dedupes**: one row per company per city. A second office in the same city is silently swallowed (or overwrites the first via the update branch when `address_line_1` is empty).
- `mvp/address/create-person-address-join` (4654) dedupes on **exact raw `(master_person_id, formatted_address)`** — **under-dedupes**: "123 Main St" and "123 Main Street" both insert, since one string is user-typed and the other geocoder-formatted.

A computed `natural_key` with a unique index replaces both with one correct rule.

### Key definition

`natural_key` is a **computed, tier-prefixed normalized string** (not the raw `formatted_address`):

| Tier | Prefix | Value | When |
| --- | --- | --- | --- |
| 1 | `pid:` | geocoder place id | **Not available today** — Radar forward geocoding returns no durable place id (the retired `archive/details/add-address` 228 took a Google `place_id` input). Keep the optional `place_id` column from the sync design for a future geocoder upgrade. |
| 2 | `cmp:` | `country_code\|region_code\|city\|postal_code\|street\|number\|unit` | Row has geocoded components (the normal Radar path) |
| 3 | `fmt:` | normalized `formatted_address` string | Row was never geocoded (user-typed only) |

**Recommendation:** store the readable pipe-joined string, not a sha256 of it — deterministic either way, but debuggable when a merge goes wrong. Hash only if key length becomes a problem.

### Normalization spec (global) — v2

What makes the key work globally is that the **geocoder is the canonicalizer** (Radar expands "Str." → "Street", resolves ambiguous city names into components) and that **pipe positions are fixed** — a missing component leaves an empty slot and never shifts the others, so sparse address formats degrade gracefully instead of breaking.

Rules for tiers 2–3 (`cmp:` / `fmt:`):

1. **Fixed field order** `country_code|region_code|city|postal_code|street|number|unit`; missing component → empty string, position preserved.
2. **Lowercase everything;** trim; collapse internal whitespace to a single space in `street` / `city`.
3. **Strip punctuation and diacritics** (Unicode NFKD, drop combining marks) — applies to Latin scripts; CJK and other non-Latin scripts pass through intact (see rule 4).
4. **Pin the Radar response language** (query with `language=en`) so components always come back in one script. Without this, the same Beijing address geocoded from English input vs `中关村大街1号` returns different-script components → two keys for one building.
5. **`postal_code`: strip ALL whitespace**, keep alphanumerics only — UK postcodes ("SW1A 2AA") are written inconsistently with/without the internal space.
6. **No-postal-code countries** (and no-street-number regions): slots stay empty; the key degrades to street- or city-level granularity, which is acceptable because uniqueness is scoped per owner.
7. **`fmt:` tier gets rules 2–3 and 5 only** — no abbreviation expansion (that's the geocoder's job). A `fmt:` key never matches a `cmp:` key for the same address; convergence happens only when the row is geocoded.

| Address | natural\_key |
| --- | --- |
| 133 West 22nd Str. New York, NY 10011 | `cmp:us\|ny\|new york\|10011\|west 22nd street\|133\|` |
| Same string, never geocoded | `fmt:133 west 22nd str new york ny 10011` |
| Kenyatta Avenue, Nairobi 00100, Kenya | `cmp:ke\|30\|nairobi\|00100\|kenyatta avenue\|\|` |
| 1 Zhongguancun Street, Haidian, Beijing 100080 | `cmp:cn\|bj\|beijing\|100080\|zhongguancun street\|1\|` |
| 10 Downing Street, London SW1A 2AA | `cmp:gb\|eng\|london\|sw1a2aa\|downing street\|10\|` |
| 10 Bayfront Avenue, Singapore 018956 | `cmp:sg\|\|singapore\|018956\|bayfront avenue\|10\|` |

### Completeness floor & upgrade rule

Dedupe alone doesn't stop junk rows: live example on `master_address` today — `neural frames` has row 1 `formatted_address = "DE"` (country only, from a sparse Fundable HQ concat) next to row 2 `"Donaustraße 44, 12043 Berlin, DE"`. Under `natural_key` those are *different* keys (`cmp:de||||||` vs the full Berlin key), so both would survive. Two additional rules in `add-company-address` (2451), enforced at the same choke point as the key:

1. **Completeness floor (reject at write).** If the geocode resolves to no city — country-only or region-only components — skip the insert and log it. A row like "DE" carries no address information; it should never be written.
2. **Upgrade in place (same address only).** A new write **supersedes** an existing row for the same owner when the existing row's non-empty components all match the new address AND the new address has strictly more populated slots (e.g. `cmp:de|be|berlin|||` upgraded by `cmp:de|be|berlin|12043|donaustrasse|44|`). The existing row is updated in place — `id` preserved, joins intact — not duplicated.
3. **Downgrade guard (ignore).** The reverse direction is a no-op: a less complete address never overwrites or joins alongside a more complete row it's consistent with.

"Consistent with" is strict: every non-empty slot of the coarser key must equal the finer key's value — a city-level Berlin row is upgradeable by a Berlin street address, but not by a Munich one, and a country-only row (which the floor now prevents) matches nothing.

**Backfill addition:** the collision report gains a completeness sweep — for each owner, delete coarse rows that are consistent with a more complete surviving row (screenshot case: row 1 "DE" dies because row 2 exists), repointing any joins first.

**Uniqueness:** composite unique index on `(master_person_id, master_company_id, natural_key)` — two people at the same address must remain legal. The null-vs-`0` FK defaults must be coalesced consistently (today `master_person_id` defaults to `0` but `master_company_id` to null) so blank owners actually collide.

### Field name — keep `natural_key`? (open, recommendation: keep)

- **Keep `natural_key` (recommended):** it is the locked column name in the [Universe ↔ Backend sync design](/guides/open-work/backend-refactor-tables) across all four facet tables (`master_email` / `master_phone` / `master_link` / `master_address`) — the generic upsert machinery keys on `(node_uuid, natural_key)`, and one uniform name is what makes the apply endpoint generic. It's also the standard DB term for a domain-derived key (vs the surrogate `id`).
- **`canonical_address`:** self-describing on this table, but per-table naming breaks the generic sync apply, and "canonical" already means something else on `master_link` (canonical URL normalization).
- **`dedupe_key` / `identity_key`:** clearer intent for app developers — viable only as a uniform rename across all four facet tables in the sync design, not an address-only name.

### Writer inventory — enrichment waterfalls only

</Note>

And the person waterfall never writes `master_address` at all: `mvp/address/add-person-primary-location` (2766) geocodes via Radar and writes graph location nodes / `primary_location` via `mvp/node/primary-location-nodes-edges_v2` — so this change is **company-pipeline-only**.

Implementation surface for this pass: **one new helper + one function edit.**

### Backfill & rollout order

1. **Add columns** `natural_key` (text) + optional `place_id` (text) — via the Xano UI, *not* the MCP full-schema replace (drops `trim` validators) and never `xano workspace push`.
2. **Ship the helper** and wire 2451 to compute-on-write. No constraint yet — old and new rows coexist.
3. **Backfill** existing rows in batches (rows with geocoded components → `cmp:` tier; rest → `fmt:` tier). Row volume: TBD, measure before running.
4. **Collision report** — group by `(owner, natural_key)`; merge duplicates keep-oldest and repoint `my_person_address_join` (310), `my_company_address_join` (314), `my_address_notes` (293).
5. **DB unique index: deferred.** With user-facing writers out of scope, a table-wide unique constraint would make `PUT /master-addresses/{id}` (3112) and the join helpers fail on violation. This pass runs `natural_key` as an **application-level upsert key inside 2451** plus a recurring collision report; the DB constraint lands when the app writers are updated in the later phase.

### Open questions

- Field name: keep `natural_key` (recommended, see above) or rename uniformly across the four facet tables?
- Keep `unit` in the key? (Recommend yes — two units in one building are distinct addresses.)
- PO boxes — normalize as `street="po box", number=N`, or leave to the `fmt:` tier? (Countries without postal codes are now handled by spec rule 6 — empty slots degrade gracefully.)
- Backfill geocoding for never-geocoded rows (costs Radar calls per row) or accept `fmt:` keys until a row is next touched?
- Does this land before or with the Universe sync's `ALTER TABLE` for the other three sync columns (`content_hash`, `source_run_id`, `locked`)?
