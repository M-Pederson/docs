title: "Person Enrichment Cascade"
description: "Four-tier fallback waterfall (PDL → Hunter → EnrichLayer → Exa) for resolving a master_person from sparse inputs — especially email-only seeds where PDL returns 404."
---

<Note>
Lives entirely inside `mvp/get-add/master-person` (#12553). Shipped 2026-05-24 as v3.8 → v3.10. The tiers are gated such that each cheaper provider runs first and the next only fires when the cheaper one failed to surface a `linkedin_url`. Every API response is stashed onto `person_enrich_data` so downstream phases can re-process without re-paying.
</Note>

## Why this exists

The seed inputs to `mvp/get-add/master-person` can be sparse — sometimes we have a `linkedin_url`, sometimes just an email, sometimes a phone, sometimes a name + company. People Data Labs is the cheapest first call; it accepts email + name + linkedin and returns a 200 with profile data or a 404 with `{"data":{},"error":{"type":"not_found"},"status":404}`.

Before v3.8, a PDL 404 produced a junk placeholder row: `master_person.name = "Someone"` (derived from the email local-part), no `linkedin_url`, no profile. That's the failure mode this cascade fixes.

## The cascade

```
PDL  ($input.email + name + linkedin if any)
 │
 ├─ status 200 ─────────────► proceed normally
 │
 └─ status 404 ─┐
                │
                ▼
Tier 2 — Hunter.io /v2/combined/find  (gate: linkedin_url empty AND email present)
  ├─ returns linkedin → set $linkedinUrl  ─────────┐
  └─ no linkedin                                    │
        │                                           │
        ▼                                           │
Tier 3 — EnrichLayer /v2/profile/resolve/email     │
  (lookup_depth=deep, enrich_profile=enrich)        │
  ├─ public_identifier  → build                     │
  │   https://www.linkedin.com/in/{slug}            │
  │   → set $linkedinUrl ────────────────────────┐  │
  └─ no linkedin                                 │  │
        │                                        │  │
        ▼                                        ▼  ▼
   thin record fallback                Tier 4 — Exa get-exa-profile
   (no linkedin, name from email)      (gate: $linkedinUrl present)
```

Each tier only fires when the prior tier didn't resolve a `linkedin_url`. The Exa block is reused unmodified — it's still gated on `$linkedinUrl|is_empty == false`, so whether the URL came from `$input.links`, Hunter, or EnrichLayer doesn't matter.

## Tier-by-tier

### Tier 1 — People Data Labs

Always fires (when `cascade_depth == 0`). The existing PDL call hasn't changed. Returns a profile or a 404.

```
GET https://api.peopledatalabs.com/v5/person/enrich
  ?email=...&first_name=...&last_name=...&profile=...
```

Stored at `person_enrich_data.people_data_labs`.

### Tier 2 — Hunter.io Combined Enrichment

**Function:** `mvp/enrich/get-hunter-combined` (#13032)
**Env var:** `$env.HUNTER_IO`
**Gate:** `$api1.status == 404 && $linkedinUrl is empty && $input.email|first is non-empty`

```
GET https://api.hunter.io/v2/combined/find?email={email}&api_key={key}
```

Returns person object (name, location, employment, **linkedin handle**, twitter/github/facebook handles, phone) plus a company object (industry, HQ, size, revenue, tech stack, funding). Hunter's `linkedin` field may be a bare handle (`"johndoe"`), a `/in/handle` path, or a full URL — the helper normalizes all three to `https://www.linkedin.com/in/{slug}`.

404 from Hunter is a clean miss — no charge, no error log.

Stored at `person_enrich_data.hunter` (json, added in v3.9).

### Tier 3 — EnrichLayer Reverse Email Lookup

**Function:** `api/enrich_layer/person-email` (#12734) — *already existed* prior to v3.10
**Env var:** `$env.ENRICH_LAYER`
**Gate:** `$api1.status == 404 && $linkedinUrl is still empty && $input.email|first is non-empty`

```
GET https://enrichlayer.com/api/v2/profile/resolve/email
  ?email={email}&lookup_depth=deep&enrich_profile=enrich
```

The `lookup_depth=deep` parameter is what makes EnrichLayer a useful Tier 3 — it goes beyond their cache and runs heuristic resolution, which is specifically recommended for work emails (the case where Hunter often whiffs). `enrich_profile=enrich` then inlines the cached LinkedIn profile data, so we get both the resolved URL AND the profile payload in one round-trip.

The helper extracts `response.result.profile`, which has the same shape as `api/enrich_layer/person-linkedin` — meaning downstream code that already knows how to consume `enrich_layer_data` can consume this too. We rebuild the URL from `public_identifier` (`https://www.linkedin.com/in/{slug}`) and assign it to `$linkedinUrl`.

Stored at `person_enrich_data.enrich_layer_email` (json, added in v3.10 — distinct from `enrich_layer_data` which is from the linkedin-url path).

### Tier 4 — Exa Profile Search

**Function:** `mvp/enrich/get-exa-profile` (#12936)
**Gate:** `$api1.status == 404 && $linkedinUrl is non-empty`

```
POST https://api.exa.ai/search
  { query: <name>, category: "people", numResults: 5, contents.summary, contents.highlights }
```

Exa runs a "people"-category search keyed off the candidate's name, then filters results to the one whose URL contains the LinkedIn slug. It returns a rich `matched` object with `summary`, `highlights[]`, `image`, and `entities[0].properties.workHistory[]`.

The Exa response is **stashed only here** — actual processing of `workHistory` into `work_experience` rows happens later in `mvp/enrich/run-base-person-enrich` Phase 4b via the helper `mvp/enrich/process-person-exa-profile` (#13031). That phase reads `person_enrich_data.exa`, iterates `workHistory`, resolves each company via `get-add/master-company`, and adds `master_person` biographies + avatar.

Stored at `person_enrich_data.exa` (json, added in v3.8).

## What lands on `person_enrich_data`

| Column | Source | When populated |
|---|---|---|
| `people_data_labs` | PDL `/v5/person/enrich` envelope | Always (cascade_depth == 0) |
| `enrich_layer_data` | EL `/v2/profile/{linkedin_url}` | When linkedin_url present (any tier) |
| `enrich_layer_email` | EL `/v2/profile/resolve/email` profile | Tier 3 — PDL 404 + no linkedin upstream |
| `hunter` | Hunter `/v2/combined/find` envelope | Tier 2 — PDL 404 + email present |
| `exa` | Exa `/search` matched object | Tier 4 — PDL 404 + linkedin URL (from any tier) |

All five columns are `json`, `nullable: false`, default `""`. The writes happen at two persist sites:

- **Section 7** of `mvp/get-add/master-person` (existing person path) — five separate gated `db.edit person_enrich_data` blocks, each conditional on the corresponding stash being non-empty.
- **Section 9** (new person path) — a single `db.edit person_enrich_data` with all five fields in the data block.

## Resilience

Every tier is wrapped in its own `try_catch`. A vendor outage on Hunter doesn't block EnrichLayer; an EnrichLayer failure doesn't block Exa. Failures write a `log_crash` row with the email/url payload and continue.

## What's NOT in the cascade

- **No name/phone-only resolution.** All three new tiers require an email. A phone-only seed still falls back to the thin-record path.
- **No second-pass.** Tiers don't retry. If Hunter returns a linkedin URL that's actually wrong, Exa happily searches for it and we end up with a polluted `master_person.linkedin_url`. Watch `log_crash` for "PDL 404 ... fallback threw" rows in the first weeks of production traffic.
- **No company-side cascade.** Hunter's `combined/find` returns rich company data but we don't currently feed it to `get-add/master-company`. The `hunter.company.*` fields sit in `person_enrich_data.hunter` for later use.
