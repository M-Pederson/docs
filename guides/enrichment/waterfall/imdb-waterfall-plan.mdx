### Entry points

<Card icon="play">

```
mvp/imdb/get-add/imdb-person — #12873
```

Gate for any IMDB person URL. **v1.3 (2026-04-22):** normalizes URL, dedups on a 30-day window, ensures stub, then either sync-enriches (depth 0 + `queue: false` → invokes `run-base-imdb-person-enrich` directly) or upserts the queue row. Seed-id propagation: when no caller seed is passed, the just-created stub IS the seed. **Primary entry point for new work.**

</Card>

<Card icon="play">

```
mvp/imdb/get-add/imdb-title — #12874
```

Gate for any IMDB title URL. **v1.3 (2026-04-22):** same shape as the person gate — stub, dedup, sync-call branch, queue upsert, seed-id propagation. **v1.4 (2026-05-27):** pre-API local dedup — drops the 30-day staleness short-circuit in favor of an explicit existing-row check before any scrape. **v1.5 (2026-05-27):** adds a `refresh` bool input (default `false`); when `true`, forces a re-scrape even if the `film_tv` row is fresh. _(The person gate #12873 remains at v1.3 — it did not receive the dedup/refresh upgrade.)_

</Card>

<Card icon="play">


Attaches IMDB data to an **existing** `master_person_id`. Preserves inline scrape + LLM + `db.add imdb_person` because the caller owns the `master_person` link. **v1.5 (2026-05-17):** model is `google/gemini-3.1-flash-lite` (the Pro variant from v1.3 was returning empty bodies). **v1.4 (2026-05-17):** added 30-day `enriched_at` staleness check — re-scrapes existing `imdb_person` rows when `enriched_at` is null or > 30 days old, db.edit-overwriting `json` / `markdown` / `image_url` / `name`. db.add path now sets `enriched_at: now` so future runs can dedup correctly. **v1.2 (2026-04-23):** Section 6 — non-fatal sync call to `#12887` after credits closes the Phase-8 honorary-award asymmetry. ## What changed in May 2026

April's eight-fix sandbox-found bug sweep closed the original waterfall (see `git log` for the full history; the prior version of this doc preserved them all). May's changes were architectural extensions, not bug fixes.

<CardGroup cols={1}>
  <Card title="WORKED_ON edge writers added (Bug 2 closure)" icon="cloud-arrow-up">
    The original architecture had no writer for `(Person)-[:WORKED_ON]->(Film_TV)` edges outside the award context. **#2334** only fires when a `film_tv_award` row exists with both endpoints resolved — so a person's plain credits never produced WORKED_ON edges.
    
    **Fix:** built two symmetric writers.
    - **#13013** `add-credit-edges-from-person` (2026-05-16) — wired into **#12880** as Phase 4c.
    - **#13015** `add-credit-edges-from-title` (2026-05-17) — wired into **#12881** as Phase 3.5.
    
    Each silently skips endpoints whose other side hasn't been enriched (`node_uuid` null). Symmetric: once both sides exist, edges appear regardless of which entity processed first. MERGE-based, idempotent. Plus **#13016** `create-worked-on-edge-description-from-graph` (2026-05-17) for LLM-generated edge descriptions, keyed on `(person_uuid, film_uuid)` since the credit writers don't stamp `e.uuid`.
  </Card>

## Depth model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
|:-----:|---------|----------|
| **0** | Seed (user-initiated) | Full scrape + LLM extract + awards + cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` + name/title, no scrape). Queued. Cron processor scrapes + expands, but **only T1–T2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes + writes data, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits + 9 acting + 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director + writer + DP + lead producer + 3 cast = ~7 people → ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000+.
</Check>

## Priority tiers

### `queue_imdb_person` (1 = highest)

| Tier | Rule | Example (seed = title "Gamer") |
|:----:|------|-------------------------------|
| **1** | Director OR Writer of seed title; award-winning/nominated person on seed | Mark Neveldine, Brian Taylor |
| **2** | Cinematographer OR lead Producer of seed title; co-winner / co-nominee on seed's award (via `shared_with`) | Ekkehart Pollack (DP), lead producers |
| **3** | Top 3 cast of seed title; other `shared_with` people | Gerard Butler, Michael C. Hall, Amber Valletta |
| **4** | Everyone else mentioned | Minor cast, thanks, video-game voice actors |

### `queue_imdb_title` (1 = highest)

| Tier | Rule | Example (seed = person "Neveldine") |
|:----:|------|------------------------------------|
| **1** | Seed person was Director OR Writer on this title; award-winning/nominated title | Crank, Gamer, Ghost Rider, Crank: High Voltage, Jonah Hex, Pathology |
| **2** | Seed person was lead Producer (not exec) AND (rating ≥ 7 OR year = current_year) | Kid Revolver (8.2 Short), Great Pretenders (8.9 Short), The Olympians (upcoming) |
| **3** | Seed person was executive producer / cinematographer / minor role; OR rating ≥ 7 regardless of role | Officer Downe, Urge, Pathology (producer), Vatican Tapes (director) |
| **4** | Video games / music videos / shorts without high rating | Jonah Hex (Video Game, tt38040358) |

### Credit filter for person seed (`cascade-credits-from-person`)

```text
for each credit in imdb_person.json.credits:
    if role ∈ {director, writer}:                                → T1
    elif role = producer AND (rating ≥ 7 OR year = current_year): → T2
    elif role ∈ {executive producer, cinematographer, minor}
         OR rating ≥ 7:                                          → T3
    elif type ∈ {Video Game, Music Video}
         OR (type = Short AND rating < 8):                       → T4
    else: skip
    # T3 capped at top 20 by rating to prevent low-signal blowout
```

### Key crew filter for title seed (`cascade-crew-from-title`)

```text
from film_tv.json:
    directors[]                                      → T1
    writers[]                                        → T1
    cinematographer (if present in json)             → T2
    lead producer (first producer in credits list)   → T2
    top 3 cast (from cast[] sorted by bill order)    → T3
```

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

### Depth-1 queued processing

When the cron picks up a queue entry, it sets `processing: true` and calls the same orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P6) only queue T1–T2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured — but no children are queued.

## Loop prevention (five layers)

1. **Dedup gate in `get-add`:** `imdb_url` already in `imdb_person` / `film_tv` with `enriched_at` within 30 days → return existing, no scrape.
2. **Queue upsert:** duplicate `imdb_url` in queue → `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth ≥ 1:** only T1–T2 discoveries are queued at depth 2 (T3–T4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.

## Implementation notes captured during build

- **URL normalization lives in get-add:** `trim | split:"/?" | first | rtrim:"/"`. All queue rows and `imdb_person` / `film_tv` rows are keyed on the normalized URL.
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` — NOT `function.call` (not valid in Xano).
- **Queue upsert is the single source of truth:** `get-add` always routes through `upsert-imdb-person` / `upsert-imdb-title`. No raw `db.add` into queue tables anywhere.
- **Kill switch is fail-open:** missing env var → `try_catch` logs to `log_crash` and defaults OFF.
- **`cascade_depth` + `queue` param on get-add:** when both `cascade_depth == 0` AND `queue == false`, `get-add` invokes the orchestrator synchronously after the stub exists and returns `action: "enriched_direct"` — no queue entry written.
- **Orchestrators are per-phase `try_catch` → `log_crash`:** each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` + stack trace. Phase enums: `llm_extract`, `save_scrape`, `master_person`, `node_stamp`, `bios_optional`, `credit_edges`, `awards`, `cascades`.
- **Phase 1 guard:** if scrape+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early — no `master_person` gets created from a junk row.
- **Cascade helpers are pure dispatchers:** every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"`. Every `function.run` to `get-add` is wrapped in `try_catch` → `log_crash`.
- **Seed ID propagation:** a depth-0 seed invocation via `get-add` must set `seed_imdb_person_id` / `seed_film_tv_id` to the just-created stub's own id when no explicit seed is passed. Required for loop-prevention to work.
- **Top-N sync credits on person seeds (`sync_top_credits: 5`):** `cascade-credits-from-person` v1.1 sync-enriches the top 5 credits inline when the person is at `cascade_depth == 0`. Guarantees a seed returns with enriched films even before the title cron runs.
- **Cron single-row tick pattern:** modeled on `process-person-queue` **#126** / `process-company-queue` **#127**. Lock-on-pick, `try_catch` around orchestrator, release lock on failure. Priority pick order: `priority_tier` asc, `cascade_depth` asc, `id` asc.
- **Missing-stub guard:** if the queue row's `imdb_person_id` / `film_tv_id` is 0, write `log_crash` and `db.del` the queue row — prevents a bad row from monopolizing the lock forever.
- **Symmetric credit-edge writers:** #13013 (person side) and #13015 (title side) each silently skip endpoints whose other side hasn't been enriched. Whichever entity processes first stamps its `node_uuid`; the other side picks up the edge on its own run. No reconciler cron needed.
- **Honorary awards have no Film_TV anchor:** `#2334 v1.5` handles `film_tv_id == null` by gating the Film_TV MERGE on `hasFilm` and swapping the Film_TV_Award MERGE key from `film_tv_imdb_url` → `person_uuid` so honoraries remain unique per-person.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **shared_with** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person + company waterfall solves this for LinkedIn/PDL/Enrich Layer (and now, in v4, for the post-PDL LLM stack). This plan applies the same pattern — **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** — to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits). Write WORKED_ON edges for every credit whose film is already in the graph.
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) + top 3 cast. Write WORKED_ON edges for every cast/crew whose person is already in the graph.
