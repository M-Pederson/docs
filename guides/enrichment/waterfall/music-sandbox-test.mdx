## The driving surface

Three of the seedable types had no HTTP seed endpoint (only `seed/artist|label|event` existed), and there was no generic node-assertion endpoint. The 2026-06-14 run added four permanent operator endpoints to the `musicbrainz` API group (#1283) to close those gaps:

| Endpoint | Verb | ID | Purpose |
|---|:--:|--:|---|
| `tool/run-cypher` | POST | 8665 | **Read-only** Cypher тЖТ `mvp/falkor/send-cypher`; write keywords (`CREATE`/`MERGE`/`DELETE`/`DETACH`/`SET`/`REMOVE`/`DROP`) blocked by a token-split guard. The generic assertion surface. |
| `seed/recording` | POST | 8666 | Seed a recording MBID (тЖТ `mb/get-add/recording`); default sync depth-0 enrich |
| `seed/release-group` | POST | 8667 | Seed a release-group MBID (тЖТ `mb/get-add/release-group`); also materializes the canonical `Release` |
| `seed/work` | POST | 8668 | Seed a work MBID (тЖТ `mb/get-add/work`) |

They join the existing seed/tool endpoints (`seed/artist` 8644, `seed/label` 8643, `seed/event` 8649, `tool/run-person-resolution` 8660, the backfill drivers, and `tool/peek-person-node` 8661) on the [Music Waterfall Functions](/guides/enrichment/waterfall/music-functions#operator-endpoints) page.

<Warning>
**Sandbox pinning is 100 % header-driven.** There is **no** `data_source` body input on any of these endpoints тАФ the data source is selected entirely by the `X-Data-Source: sandbox` request header (every endpoint has `cache.datasource = true`). **Omit the header and you write to the LIVE graph + LIVE rows.** Always send `X-Data-Source: sandbox` and `X-Branch: v1`. See [Data Source Reference](/guides/enrichment/waterfall/data-source-reference).
</Warning>

## Runbook

Every call: base URL `https://xh2o-yths-38lt.n7c.xano.io/api:P8PNYYKM`, headers `X-Data-Source: sandbox` + `X-Branch: v1`, no auth. Crons #180 (MB queue) and #182 (person-resolution) stay **inactive** тАФ the whole runbook is direct/synchronous.

<Steps>
  <Step title="Seed the 7 entities (sync)">
    `seed/artist` ├Ч2 (Nirvana, Cobain) тЖТ `seed/label` (Sub Pop, runs the inline company bridge) тЖТ `seed/event` (Reading, materializes the Place inline) тЖТ `seed/recording`, `seed/release-group`, `seed/work`.   </Step>
  <Step title="Run the person bridge">
    Look up Cobain's `mb_artist` **row id** in the sandbox source, then `POST /tool/run-person-resolution {"mb_artist_id": <id>, "hold_downstream": true}`. `hold_downstream:true` suppresses the FullEnrich people pipeline (data-source-safe).
  </Step>
  <Step title="Run the completeness backfills, in order">
    `backfill-node-descriptions` (desc + both vectors for `Recording`/`Music_Group`) тЖТ `backfill-genre-area-edges` (legacy name; now writes `FROM_AREA` only) тЖТ `backfill-node-embeddings` (both vectors for the remaining music graph nodes) тЖТ `backfill-cover-art` + `backfill-group-images` (`image_url`). Each is idempotent; re-call until `written`/`processed` reach 0.
  </Step>
  <Step title="Assert">
    Run per-node-type envelope + edge checks via `tool/run-cypher` (below).
  </Step>
</Steps>

## Findings from the first run

1. **ЁЯФ┤тЖТтЬЕ Deceased-LinkedIn false positive (fixed).** Person-resolution bridged Kurt Cobain (died 1994) to an impersonator LinkedIn тАФ the v2.2 content-verify can't separate a real artist-exec from a music-themed namesake/fan profile. Fixed as `mb/run-person-resolution` **v2.3**: a `deceased` flag (from MB `life-span.ended`/`end`) now skips the LinkedIn-accept branch entirely; socials still flow. Re-test: `linked_social`, `verified_linkedin: ""`, `deceased: true`, no duplicate person. See the [dev update](/guides/dev-updates/2026-06-14-person-resolution).
2. **Description coverage.** Only `Recording` + `Music_Group` receive a generated `description` today; the other 8 types carry `description = null` (see the canonical [standard music node envelope](/guides/ontology/nodes#music-standard-node-envelope)).
3. **Stub-MERGE neighbors.** Cascade neighbors are MERGE-created as stub nodes immediately (Nevermind's 12 tracklist `Recording`s appeared as stubs and were caught up by the embeddings backfill), so a per-label count grows past the hand-seeded fixture тАФ assert by mbid, not by global counts.
4. **ЁЯФ┤тЖТтЬЕ Recording credit type-collision (fixed).** A second run that seeded every album track **fully** (a `seed/recording` per track) exposed that `mb/run-base-recording-enrich` (#13100) labeled *all* `PRODUCED`/`ENGINEERED`/`PLAYED_ON` credits `Person:Music_Artist` тАФ so Nirvana (a Group) credited as producer/engineer spawned a **duplicate `Person` node** alongside its `Music_Group`. Fixed in **#13100 v1.2** (type-aware: split rel-credits into Person vs Group buckets). The partial first run hid it because tracklist recordings were title-only, so their credits never wrote.

<Tip>
**A `Recording` is only fully enriched** (`length_ms`, ISRCs, cover art, and the `PERFORMED`/`RECORDING_OF`/`PRODUCED`/`ENGINEERED`/`PLAYED_ON` credit edges) via a **direct `seed/recording`** or a **queue drain**. A `seed/release-group` materializes its tracklist as **title-only partials** (`title` + `APPEARS_ON` + embeddings, but no `length_ms` or credit edges). Seed per-track when you need full recordings without draining.
</Tip>
