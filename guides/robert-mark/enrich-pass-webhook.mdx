## TL;DR

FullEnrich (fullenrich.com) is a reverse-email-lookup vendor. We use it as a **last-resort tier** in the person enrichment cascade — fired only when PDL and EnrichLayer have both failed to produce a LinkedIn URL for an email-only seed.

Because the FullEnrich API is async (bulk endpoint that POSTs results back to a webhook minutes-to-days later), the response arrives **after** `mvp/get-add/master-person` (#12553, deprecated 2026-06-01; callers migrated to #13039) has already created the master_person and kicked off the downstream orchestrator `#13040 mvp/enrich/enrich-master-person`. By the time FullEnrich's callback lands here, three things may be true simultaneously:

1. The `master_person` exists.
2. `#13040` may still be in flight, mid-phase, writing to overlapping rows (master_link, work_experience, biography, avatar, master_person.linkedin_url).
3. FullEnrich finally has data that should be applied to those same rows.

**The webhook's job is therefore three steps:**

1. **Stash** the FullEnrich payload onto `person_enrich_data.full_enrich` by id. (Done today, v1.6.)
2. **Wait** for `#13040` to finish processing this master_person, so we don't race against the orchestrator. (**Not yet built — proposed v1.7.**)
3. **Process** the FullEnrich payload into `master_link` / `avatar` / `biography` / `work_experience` / `master_email` / `master_phone` / social links via a new helper, `mvp/enrich/process-full-enrich`. (**Not yet built.**)

## When FullEnrich gets called

`#13036 mvp/enrich/full-enrich-reverse-email-lookup` is the dispatch function. Today it's wired up so that it fires when:

- `mvp/get-add/master-person` (#12553, deprecated 2026-06-01; callers migrated to #13039) is invoked with an email-only seed (no LinkedIn URL upstream)
- PDL returns 404 on the email
- EnrichLayer's email-path fallback (#12734) returns no LinkedIn URL either
- We have an email but still no usable identity

`#13036` then POSTs the email to FullEnrich's bulk reverse-email endpoint with a `name` and `custom` carrier encoding `person_enrich_data_id` + `data_source`, plus our webhook URL. FullEnrich returns an `enrichment_id` immediately. The actual lookup happens on FullEnrich's side, often delayed.

## The race condition

```
T=0    #12553 creates master_person M (or matches an existing one)
T=0    #12553 dispatches #13036 (FullEnrich bulk POST, returns immediately)
T=0    #12553 dispatches #13040 async-shared (downstream orchestrator)

T+1s   #13040 Phase 0 starts (PDL backfill)
T+5s   #13040 Phase 4b (Exa processing) — writes master_link, biography, work_experience
T+8s   #13040 Phase 5 (Fundable) — writes more
T+10s  ...

T+30s  ←── FullEnrich callback arrives at this webhook
            (in this window, #13040 may still be writing to M)
```

Without a guard, the webhook + `process-full-enrich` would write to `master_link` / `work_experience` / `master_email` etc. while `#13040` is concurrently writing to the same rows. Three failure modes:

1. **Duplicate inserts** — both processes insert "current title at current employer" biographies / work_experience entries from different sources.
2. **Overwrite races** — both set `master_person.linkedin_url`, last write wins. If FullEnrich's URL is wrong, we get polluted state.
3. **Avatar/url thrash** — `master_person.avatar` flip-flops between Exa's image and FullEnrich's image.

These are all *avoidable* by simply waiting for #13040 to finish before applying FullEnrich's data.

### Step 5: Wait for orchestrator (locked-in design)

**Decision (2026-05-30):** Use `enrich_history_person.processing` flag, keyed by `master_person_id + source == "Master Person Enrich"`. Webhook polls directly — **no cron**.

The orchestrator (`#13040`) gets two small additions:

1. **At the very top of `mvp/enrich/enrich-master-person`** (right after the `master_person` guard, before any phase fires) — upsert an `enrich_history_person` row:
   - `master_person_id = $input.master_person_id`
   - `source = "Master Person Enrich"`
   - `processing = true`
   - `updated_at = now`
2. **At the very end** (after the "all phases passed" `log_crash` block, in BOTH success and failure exit paths) — set the same row's `processing = false`.

This replaces the current `data_source_id == 79` keyed upsert in #13040's setup block. The text-based `source` field is more readable and decouples the orchestrator from the `data_source` lookup table.

The webhook then polls this exact row:

```
db.query enrich_history_person {
  where = master_person_id == $masterPersonId AND source == "Master Person Enrich"
  return = {type: "single"}
} as $history

If ($history is empty OR $history.processing == false):
  → proceed to process-full-enrich
Else:
  → sleep N seconds, re-query, repeat until processing == false (or timeout)
```

### Wait policy (locked-in)

| Setting | Value |
| --- | --- |
| Poll interval | **5 seconds** (fixed) |
| Hard cap | **30 minutes** (orchestrator's worst-case runtime is ~30 min; typical is 1–10 min) |
| Behavior on cap | Proceed to `process-full-enrich` anyway, write `log_crash` row with `note: "wait-for-orchestrator timed out at 30min"` |
| Behavior when no `enrich_history_person` row exists at first poll | Wait **30 seconds grace** (give orchestrator dispatch time to land), then proceed if still absent |

### Wait mechanism (verified 2026-05-30)

XanoScript has **native `while` + `util.sleep`** constructs — no `api.lambda` setTimeout hack needed. Verified via POC API `test/while-sleep` (function `#13048`):

```xanoscript
var $startedMs { value = now|to_int }
var $deadlineMs { value = $startedMs + 1800000 }   // 30 min cap
var $graceMs { value = $startedMs + 30000 }        // 30s grace for missing row
var $done { value = false }
var $timedOut { value = false }

while (`$done == false`) {
  each {
    db.query enrich_history_person {
      where = $db.enrich_history_person.master_person_id == $masterPersonId && $db.enrich_history_person.source == "Master Person Enrich"
      return = {type: "single"}
    } as $history

    conditional {
      if ($history|is_empty) {
        // No row yet — orchestrator may not have started. Grace window.
        conditional {
          if (`(now|to_int) > $graceMs`) {
            var.update $done { value = true }       // give up waiting, proceed
          } else {
            util.sleep { value = 5 }
          }
        }
      }

      elseif ($history.processing == false) {
        var.update $done { value = true }           // orchestrator finished, proceed
      }

      elseif (`(now|to_int) >= $deadlineMs`) {
        var.update $done { value = true }
        var.update $timedOut { value = true }       // hit 30 min cap, proceed anyway
      }

      else {
        util.sleep { value = 5 }
      }
    }
  }
}

conditional {
  if ($timedOut == true) {
    db.add log_crash {
      data = {
        function_name: "webhook/contact-finished"
        master_person_id: $masterPersonId
        note: "wait-for-orchestrator timed out at 30min — proceeding to process-full-enrich anyway"
      }
    }
  }
}
```

**Runtime verification results:**

| Test | Expected | Actual elapsed | Result |
| --- | --- | --- | --- |
| 3 iter × 1s sleep | ~3s | 3.01s | ✅ |
| 6 iter × 5s sleep | ~30s | 30.10s | ✅ |
| 1 iter × 60s sleep | ~60s | 60.06s | ✅ — Xano holds requests open beyond 60s |
| 1 iter × 180s sleep | ~180s | 180.04s | ✅ — clean response at 3min, no 504 / timeout |

**XanoScript syntax notes (gotchas from the docs):**

- The `while` condition uses **backticks around the expression**: `` while (`$count < $target`) `` — not bare parentheses.
- Inside `while`, the body lives in an `each { ... }` block.
- Same backtick convention for any expression that needs runtime re-evaluation (e.g. `value = \`$count + 1\``).

### Webhook-timeout caveat (FullEnrich's HTTP client)

Xano lets us hold the request open indefinitely. The harder constraint is **FullEnrich's HTTP client timeout** — if their side gives up before our wait completes, they'll retry the callback, causing duplicate webhook invocations.

We don't know FullEnrich's exact timeout (likely 30s or 60s). Two mitigations:

1. **`process-full-enrich` MUST be rigorously idempotent.** Each helper (`create-master-link`, `add-person-biography`, etc.) should upsert / no-op on existing rows. Duplicate invocations cost cycles, not correctness.
2. **Common-case observation:** FullEnrich's bulk reverse-email lookup takes minutes-to-hours on their side, while the orchestrator finishes in 1–10 min. So in the **typical case**, by the time the callback arrives, the orchestrator is already done — wait loop exits in 1 iteration (~5s). The 30-min cap only matters for the rare "FullEnrich-faster-than-orchestrator" race.

If duplicate webhooks become a real problem in practice, the fallback design is to invert: webhook returns 200 immediately after stashing, then orchestrator's end-of-pipeline step also runs `process-full-enrich` if `person_enrich_data.full_enrich` is non-empty. That deferred-by-orchestrator approach was Option C in the original Q1 question; we'd revisit if needed.

## process-full-enrich (new helper)

A new function — **not** a port of the retired `#13033 process-person-rocket-reach`. The field mapping differs significantly (see [Response shape](#fullenrich-response-shape-verified-against-one-real-sample)). Sketch:

```
function "mvp/enrich/process-full-enrich" {
  input { int master_person_id, int? cascade_depth? }

  stack {
    db.get person_enrich_data by master_person_id → $ped
    var $full = $ped.full_enrich
    var $profile = $full|get:"profile":{}

    if ($profile|is_empty) → log "FullEnrich returned no profile" + return

    // (a) backfill master_person.linkedin_url
    //     from $profile.social_profiles.professional_network.url
    //     when master_person.linkedin_url is empty

    // (b) master_link insert
    //     create-master-link { link_url: $profile.social_profiles.professional_network.url,
    //                          source: "FullEnrich" }

    // (c) [SKIPPED v1] avatar — FullEnrich does not return one in known samples

    // (d) biography add
    //     add-person-biography {
    //       title: $profile.employment.current.title,
    //       company: $profile.employment.current.company.name,
    //       data_source_id: 103
    //     }

    // (e) work_experience iter
    //     foreach $profile.employment.all[]:
    //       resolve company via mvp/get-add/master-company
    //         (preferring company.domain → company.social_profiles.professional_network.url → company.name)
    //       insert work_experience row with data_source_id: 103,
    //         carrying title, seniority, start_at, end_at, is_current

    // (f) emails iter — gate on $full.contact_info.emails[] being populated
    //     foreach email → add-person-email { source: "FullEnrich" }

    // (g) phones iter — gate on $full.contact_info.phones[] being populated
    //     foreach phone → add-person-phone { source: "FullEnrich" }

    // (h) social links iter
    //     foreach key in $profile.social_profiles (except professional_network):
    //       create-master-link { link_url: ...url, source: "FullEnrich" }
  }
}
```

Idempotent on re-runs. Each sub-step in its own try_catch — one bad job_history entry doesn't poison the rest.

## FullEnrich response shape (verified against two real samples)

Confirmed against two real `person_enrich_data.full_enrich` payloads — Leon Silverman (ped_id=2, originally misaligned) and Carl Bressler (sandbox session 3). FullEnrich's response shape is **substantially different** from the retired RocketReach helper — process-full-enrich is a re-mapping, not a port. Both samples agree on:

- `profile.avatar` / `profile.image` / `profile.profile_pic` — **all absent**. Avatar for Carl id=1 comes from a different vendor (ScrapeCreator), pinned at `person_enrich_data.scrapecreator_person.image`. Step (c) in `process-full-enrich` is correctly SKIPPED.
- `profile.social_profiles` — **only `professional_network` (LinkedIn)** present. No twitter, github, facebook, instagram entries in either sample. Step (h) remains DEFERRED until a counter-example appears.
- `contact_info: {}` — **always empty** for our endpoint (`/api/v2/contact/reverse/email/bulk`). Phone & Email Finder is a separate FullEnrich product on a different endpoint with separate credits.

### `full_enrich.contact_info` — intentionally unused

The reverse-email endpoint we call (`/api/v2/contact/reverse/email/bulk`) **never populates** `contact_info`. Populating it would require FullEnrich's separate **Phone & Email Finder** add-on — different endpoint, different credits.

And we don't actually need it. **The LinkedIn URL is the only field we need from FullEnrich.** Once `master_person.linkedin_url` is backfilled in step (a), downstream enrichment (Exa, PDL-by-linkedin, ScrapeCreators via `create-master-link`'s domain hook) picks up emails, phones, biographies, and more via the LinkedIn path.

`process-full-enrich` v1.2 removed the aspirational `(f)` emails and `(g)` phones blocks accordingly. Task #15 (verify `contact_info` shape) is **WONTFIX** for the same reason.

### Fields NOT returned (vs RocketReach)

- **Avatar / profile image** — not present in the sample's `profile`. `master_person.avatar` will NOT be backfilled from FullEnrich (unlike the retired RocketReach helper that wrote from `raw.profile_pic`).
- **City / region** — only `location.country` + `country_code` available. No granular location.
- **Direct `linkedin_url` field** — must go through `social_profiles.professional_network.url`.

### Production incident — Carl Bressler → Robert Boulos data corruption (2026-05-30)

End-to-end test on Carl Bressler exposed a routing bug in `#13039` (`mvp/get-add/master-person-new`). The function accepted `data_source` as an optional input that callers were expected to pass when running in non-default data sources. The sandbox dispatch invoked it without that input, so `#13036` was called with `data_source: null`. FullEnrich's callback then arrived at the webhook with empty `custom.data_source`, fell through the per-branch `db.set_datasource` conditional, and **wrote to LIVE's `person_enrich_data.id=3` row (Robert Boulos, real Orbiter user) instead of sandbox's row**.

Compounded by FullEnrich's HTTP client retrying — their timeout is shorter than the v1.7 webhook's 95–180s wait+process cycle — the callback fired 5 times, each one adding fresh contaminated rows. Final damage on Robert: 60 fake `work_experience` rows (Carl's job history at MovieLabs, Hallstone Ventures, etc.), 1 `about_person` biography ("Co Executive Producer / Founder at Stealth Startup"), and Carl's FullEnrich payload pinned to Robert's enrich row. **All cleaned up** via `deleteTableContentBulk` + `deleteTableContentItem` + wiping `full_enrich`. master_link was protected by `create-master-link`'s built-in dedup — no contamination there.

Fix shipped in two parts:

**Webhook v1.8 — emergency safety guard.** Refuses any callback item with empty `custom.data_source`. Logs to `log_crash`, returns ok=true (so FullEnrich stops retrying), counts as `refused_no_data_source` in the response. Stopped the bleeding within seconds.

**`#13039` v1.7 — data_source auto-detect via marker table.** New table `runtime_metadata` (id=733) with schema `{id, key, value, updated_at}` and a `UNIQUE` btree on `key`. Each data source holds its own copy of one row: `{key:"data_source_name", value:"live"}` (or staging/sandbox). At FullEnrich dispatch time, `#13039` now:

1. Prefers caller-supplied `$input.data_source` if non-empty (explicit override still wins).
2. Otherwise queries `runtime_metadata where key=="data_source_name"` and uses `value` as the data source.
3. Failed lookups `log_crash` and pass empty — webhook v1.8 will then refuse it. Belt and braces.

`#12553 mvp/get-add/master-person` (the older sibling) was **not yet updated** — webhook v1.8's guard provides safety until it's patched the same way. (Deprecated 2026-06-01; all callers migrated to #13039; pending retirement.)

#### v1.5 — temp_ name backfill via #2649 (step b.4)

Email-only seeds create `master_person` with `first_name = "temp_<hash>"` since no real name is known at create-time. Before session 3, the FullEnrich callback never wrote the resolved name back, so:

- `master_person.first_name` stayed as `temp_<hash>` indefinitely
- Cross-source merges (e.g. Exa firing later via `#12997`) silently preserved the placeholder, because `#12997`'s `first_name|first_notempty:$parsedName.first_name` only swaps when the existing value is *empty*, not when it's `temp_`

v1.5 added step (b.4): if `$masterPerson.first_name` starts with `temp_`, run `profile.full_name` (fallback: `first_name + " " + last_name`) through `#2649 mvp/format/name-format`, JSON.parse the result, and write `name`, `first_name`, `middle_name`, `last_name`, `suffix`, `nickname` back. If the LLM returns something unparseable, fall back to FullEnrich's own pre-split `profile.first_name` / `profile.last_name` (it provides both separately). #### v1.6 — PDL retry (b.5) + EnrichLayer fallback (b.6)

FullEnrich is the last-resort tier *because PDL and EnrichLayer already failed on the email*. But those failures were against `email=<seed>`. Once FullEnrich resolves the canonical LinkedIn URL, the *same* PDL endpoint hit with `profile=<url>` is dramatically more likely to succeed.

Both writes overwrite the prior 404 entries (`db.edit person_enrich_data` by master_person_id). Same retry-safety story as the FullEnrich stash: if the webhook retries, the second pass writes the same payload again — idempotent.

#### v1.7 — cosmetic relabel

The temp_ name step was labeled `(a.5)` but actually runs **after** step (b). Renamed to `(b.4)` so the label matches execution order. No behavior change.

#### #2649 v3.4 — structured-output fix

Direct call to `#2649` with "Carl Bressler" was returning the dict-literal string `full_name: "Carl Bressler", first_name: "Carl", ...` instead of JSON `{"full_name":"Carl Bressler","first_name":"Carl",...}` — making `JSON.parse` throw at every caller.

Three issues, all fixed:

1. **`response_format.json_schema` was a string template, not a nested object.** The original v3.0 code wrapped the schema in backticks with literal `\n` chars, which XanoScript serialized to a multi-line *string* rather than an object expression. OpenRouter received unparseable schema and silently fell back to free-form output. Rewrote as plain nested `{} |set: "type": "json_schema" |set: "json_schema": (...)` object construction.
2. **`provider.allow_fallbacks: true` without `require_parameters: true`.** OpenRouter was free to route to a provider that doesn't honor `response_format.json_schema`. Added `require_parameters: true` so the router refuses non-conforming providers.
3. **Prompt examples used dict-literal syntax.** Even when structured output isn't enforced, models imitate the few-shot examples in the system prompt. Reshaped every example from `→ full_name: "X", first_name: "Y"` to `→ {"full_name":"X","first_name":"Y"}` and added an explicit "Return STRICT JSON only" directive.

Two intermediate versions before landing on v3.4:

- **v3.2** introduced the response_format-as-object fix and JSON examples — but used `(["a","b"])` for array literals, which XanoScript silently strips when saving. Schema saved as `|set:"required":` and `|set:"type":` with no RHS — model produced just `"{"` (one brace) and stopped.
- **v3.3** dropped `reasoning.effort` and added `max_tokens: 500` while debugging the truncation — but the broken array literals were still the actual root cause.
- **v3.4** rebuilt all arrays via the `[] |push:"x" |push:"y"` chain (the pre-v3.2 pattern), which XanoScript preserves correctly. Verified by reading the saved XanoScript back: `required` has all 6 fields, each `*_name` property has the `["string","null"]` union type intact.

The same JSON.parse-only pattern in `#12997 master-person-from-exa` had been silently writing empty `first_name` for every Exa-sourced row that went through `#2649` — visible on sandbox `master_person.id=34` (Carl from Exa), `first_name: ""`. v3.4 of `#2649` fixes both callers without any change to `#12997` itself.
