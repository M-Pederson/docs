### Implementation notes captured during build

- **URL normalization lives in get-add:** `trim | split:"/?" | first | rtrim:"/"`. All queue rows and `imdb_person`/`film_tv` rows are keyed on the normalized URL, so callers can pass raw IMDB URLs with trailing slashes or query strings.
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` — NOT `function.call` (not valid in Xano). Burned one compile on this.
- **`updateTableSchema` drops text `validators`:** the Xano MCP `updateTableSchema` tool silently strips `filters=trim` from text fields when replacing schema. After adding `enriched_at`/`cascade_depth`/`enrich_success` to `imdb_person` and `film_tv`, existing text columns lost their auto-trim. Low impact (inputs upstream are already trimmed), but flagging — never use `updateTableSchema` on a table with `filters=` unless you plan for validator loss. Restore via the Xano UI if needed.
- **Queue upsert is the single source of truth:** `get-add/imdb-person` and `get-add/imdb-title` always route through `upsert-imdb-person`/`upsert-imdb-title`. No raw `db.add` into the queue tables anywhere.
- **Kill switch is fail-open:** missing `kill_switch_imdb` env var → try_catch logs to `log_crash` and defaults to OFF. Matches `check-kill-switch-company` behavior.
- **`cascade_depth` + `queue` param on get-add (v1.1, 2026-04-22):** when both `cascade_depth == 0` AND `queue == false` are passed, `get-add` invokes the orchestrator synchronously after the stub exists and returns `action: "enriched_direct"` — no queue entry is written. Any other combination (including `queue: null` / unset) still falls through to queue-upsert. Mirrors the `master-company` v3 behavior for a single depth-0 seed.
- **Orchestrators (Step 6) are 7-phase with per-phase `try_catch` → `log_crash`:**
  - `run-base-imdb-person-enrich` **#12880** — Phase 1 scrape + preprocess + LLM extract; Phase 2 save `json` / `markdown` / `name` / `image_url` onto `imdb_person`; Phase 3 create `master_person` + `master_link` + `birthday_join` (all dedup-guarded, idempotent on re-run); Phase 4 `avatar/replace-avatar` + `bios/add-person-biography` + `bios/create-llm-person-bios` + `node/add-person-node` + `node/update-person-node`, then stamp `node_uuid` + `master_person_id` back; Phase 5 `add-award-records-from-person`; Phase 6 `cascade-credits-from-person` + `cascade-awards-from-person`; Phase 7 stamp `enriched_at` / `enrich_success: true` / `cascade_depth`.
  - `run-base-imdb-title-enrich` **#12881** — same shape without master_person creation: Phase 1 scrape/LLM; Phase 2 tagline + save `json` / `markdown` / `title` / `year` / `imdb_rating` / etc onto `film_tv`; Phase 3 `node/add-film-tv-node`; Phase 4 re-fetch `film_tv` to capture `node_uuid`; Phase 5 `add-award-records-from-title`; Phase 6 `cascade-crew-from-title` + `cascade-awards-from-title`; Phase 7 stamp `enriched_at` / `enrich_success` / `cascade_depth`.
  - **Phase 1 guard:** if scrape+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early — no master_person gets created from a junk row, and the queue processor won't retry forever.
  - Each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` + stack trace — matches the company/person `run-base-*-enrich` observability convention.
- **Cascade helpers (Step 4) are pure dispatchers.** Each reads a source entity, applies tier rules via a lambda, and fans out `get-add/imdb-{person,title}` calls per tiered result. Key properties:
  - Every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"` — the hard cascade cap from the [Depth Model](#depth-model) lives in the cascade layer, not in `get-add`. This matters because `get-add` will eventually serve sync-call depth-0 seeds too.
  - Every `function.run` to `get-add` is wrapped in `try_catch` → `log_crash`, so one rotten credit / crew member does not abort the whole cascade.
  - Tier rules are all inside the lambda. Tweaking thresholds (e.g. "T3 when rating >= 6.5") is a single-line edit in the lambda body — no restructuring needed.
- **`cascade-credits-from-person` (#12875):** lambda produces tier-4 for video games / music videos / shorts with rating \<8, tier-1 for `director` (non-exec) or `writer`, tier-2 for `producer` with rating ≥ 7 **or** current-year, tier-3 for `executive producer` / `cinematographer` / rating ≥ 7, tier-4 otherwise. Hard cap: if more than 20 T3 entries, keep only the top 20 by rating (drops noisy mid-career cinematography / acting credits on prolific people).
- **`cascade-crew-from-title` (#12876):** dedups by `imdb_url` across directors / writers / cinematographers / lead producer / top-3 cast — first tier wins per URL. This means if someone directs AND is in the top-3 cast, they get queued at T1 (director wins). Upsert semantics on the queue still bump `count` across multiple titles.
- **`add-award-records-from-title` (#12879) mirrors #12633 structure** but uses a title-specific LLM schema. Instead of per-person fields `project_title` / `project_imdb_url` the schema has `primary_nominee_name` / `primary_nominee_imdb_url` — the project is fixed (input title) and the nominee is what varies per row. For each award row:
  1. Call `get-add/imdb-person` on `primary_nominee_imdb_url` at T2/depth+1 (queued) and capture the `imdb_person_id` it returns.
  2. Dedup on (`film_tv_id`, `imdb_person_id`, `year`, `award_type`, `award_description`) — a null `imdb_person_id` is a valid bucket (e.g. Best Picture with no single named nominee).
  3. Insert `film_tv_award` with `project_title` / `project_imdb_url` echoed from the title itself.
  4. After all rows are inserted, sweep `film_tv_award.node_uuid == null` and call `mvp/edge/add-node-edge-imdb-award` per row.
- **Awards cascade — asymmetric by design:**
  - `cascade-awards-from-person` (#12877) queues award **projects** via `project_imdb_url` at T1 (an award-nominated project is a tier-1 signal about the person).
  - `cascade-awards-from-title` (#12878) queues **co-winners** already linked to that title at T2. `film_tv_award.shared_with` is names-only (no IMDB URLs), so we cannot resolve unknown co-winners today. If/when award scraping is enhanced to capture `shared_with` URLs, add a second loop — documented inline in the function description.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **shared_with** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person + company waterfall solves this for LinkedIn/PDL/Enrich Layer. This plan applies the same pattern — **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** — to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits).
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) + top 3 cast.

## Depth Model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
|:-----:|---------|----------|
| **0** | Seed (user-initiated) | Full scrape + LLM extract + awards + cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` + name/title, no scrape). Queued. Cron processor scrapes + expands, but **only T1–T2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes + writes data, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits + 9 acting + 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director + writer + DP + lead producer + 3 cast = ~7 people → ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000+.
</Check>

### Credit filter for person seed (Phase 5)

```
for each credit in imdb_person.json.credits:
    if role ∈ {director, writer}:                  → T1
    elif role = producer AND (rating ≥ 7 OR year = current_year):  → T2
    elif role ∈ {executive producer, cinematographer, minor} OR rating ≥ 7:  → T3
    elif type ∈ {Video Game, Music Video} OR (type = Short AND rating < 8):  → T4
    else: skip
    # T3 capped at top 20 by rating to prevent low-signal blowout
```

### Key crew filter for title seed (Phase 4)

```
from film_tv.json:
    directors[]   → T1
    writers[]     → T1
    cinematographer (if present in json) → T2
    lead producer (first producer in credits list) → T2
    top 3 cast (from cast[] sorted by bill order) → T3
```

<Info>
**"Top cast = top 3" is a starting point** — may tune upward (e.g., top 5 or top 10) once we see real fanout numbers. The tier assignment is what matters; the count is a knob.
</Info>

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

## Queue Tables

Two new tables, keyed on normalized `imdb_url` (not `imdb_person_id` / `film_tv_id`) because many discoveries happen before the underlying row exists. URL normalization: strip query params (`|split:"/?"|first`) and trailing slash (`|rtrim:"/"`), matching the existing convention in `#10506` / `#10507`.

### `queue_imdb_person`

| Field | Type | Notes |
|-------|------|-------|
| `id` | int | PK |
| `created_at` | timestamp | |
| `imdb_url` | text (unique) | Normalized canonical URL |
| `name` | text | Provisional name from discovery context (nullable) |

Identical shape with `imdb_url`, `title` (instead of `name`), `film_tv_id` (instead of `imdb_person_id`).

## Seed Flows

### Seed person flow (depth 0)

<Steps>
  <Step title="Kill switch (P1)">
    `check-kill-switch-imdb`. If active, return existing `imdb_person` if found; otherwise log to `kill_switch_blocked_imdb` and return empty.
  </Step>
  <Step title="Dedup check + scrape (P2)">
    Lookup `imdb_person` by normalized `imdb_url`. If `enriched_at` within last 30 days → return existing. Otherwise scrape via Firecrawl, run LLM extractor (existing `#10506` logic), write `imdb_person` row.
  </Step>
  <Step title="Awards scrape (P3)">
    Existing `add-award-records-from-person` logic: scrape `{imdb_url}/awards/`, LLM extract, write `film_tv_award` rows.
  </Step>
  <Step title="Cascade — awards fanout (P4)">
    For each `film_tv_award`:
    - Queue the award's project title → `get-add/imdb-title` at depth 1, T1, `source_function: cascade-awards-from-person`
    - For each name in `shared_with[]` → queue as person at depth 1, T2, `source_function: cascade-awards-shared-with` (no `imdb_url` — name-only, must be resolved at processing time via IMDB search)
  </Step>
  <Step title="Cascade — credits fanout (P5)">
    Walk `imdb_person.json.credits[]`. Apply the credit filter rules above to assign tier. For each kept credit, call `get-add/imdb-title` at depth 1 with computed tier, `source_function: cascade-credits-from-person`, `source_entity_id: imdb_person.id`.
  </Step>
  <Step title="Person enrichment side-effects (P6)">
    Existing logic: `master_person` creation or update, bio, avatar, roles, birthday_join, person node, `update-person-node`. Unchanged.
  </Step>
  <Step title="Mark success (P7)">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

### Seed title flow (depth 0)

<Steps>
  <Step title="Kill switch (P1)">
    Same as person flow.
  </Step>
  <Step title="Dedup check + scrape (P2)">
    Lookup `film_tv` by normalized `imdb_url`. If `enriched_at` within 30 days → return existing. Otherwise scrape + LLM extract (existing `#10507` logic). Writes `film_tv` row.
  </Step>
  <Step title="Awards scrape (P3)">
    **New function:** `add-award-records-from-title` — reuses the `#12633` pattern but scrapes `{title_imdb_url}/awards/`. Writes `film_tv_award` rows linked to `film_tv_id` (and no `imdb_person_id`).
  </Step>
  <Step title="Cascade — key crew (P4)">
    Walk `film_tv.json` / `film_tv.directors` / `film_tv.writers` / `film_tv.json.cast[]`:
    - Directors → T1
    - Writers → T1
    - Cinematographer → T2
    - Lead producer (first in producer list) → T2
    - Top 3 cast → T3
    Each queued via `get-add/imdb-person` at depth 1, `source_function: cascade-crew-from-title`.
  </Step>
  <Step title="Cascade — award shared_with (P5)">
    For each `film_tv_award` row with a `shared_with[]` list, queue those names as persons at depth 1, T2, `source_function: cascade-awards-shared-with`.
  </Step>
  <Step title="Title node update (P6)">
    Existing `add-film-tv-node` logic. Unchanged.
  </Step>
  <Step title="Mark success (P7)">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

### Depth-1 queued processing

When the cron picks up a queue entry (either table), it sets `processing: true` and calls the same `run-base-imdb-person-enrich` / `run-base-imdb-title-enrich` orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P4/P5) only queue T1–T2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured — but no children are queued.

## Loop Prevention (Five Layers)

1. **Dedup gate in `get-add`:** `imdb_url` already in `imdb_person` / `film_tv` with `enriched_at` within 30 days → return existing, no scrape. Matches the multi-layer dedup in company/person waterfall.
2. **Queue upsert:** duplicate `imdb_url` in queue → `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth ≥ 1:** only T1–T2 discoveries are queued at depth 2 (T3–T4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.

## Open / Deferred

- **Acting-heavy seeds** — if a user seeds an actor (not a director/writer), primary credits are their highest-billed cast roles, not writer/director slots. The P5 credit filter above doesn't handle this cleanly. May need a `seed_profession` input to `run-base-imdb-person-enrich` that changes the tier rules.
- **Name-only `shared_with` queue entries** — award `shared_with[]` gives names without `imdb_url`. The queue processor must resolve name → IMDB URL via search before scraping. Parking this; may end up skipping name-only entries.
- **Cinematographer cascade** — DPs often have 50+ credits. At depth 1 a DP seed would cascade their full filmography at T2–T3. May need a profession-aware reduction.
- **Kill switch table** — use existing `kill_switch_blocked_people` pattern or add `kill_switch_blocked_imdb`? TBD.
