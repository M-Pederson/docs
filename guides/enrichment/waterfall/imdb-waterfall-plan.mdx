### Entry points

```text
mvp/imdb/get-add/imdb-person #12873
```

Gate for any IMDB person URL. Normalizes URL, dedups on a 30-day window, ensures stub, then either sync-enriches (depth 0 + `queue: false`) or upserts the queue row. **Primary entry point for new work.**

```text
mvp/imdb/get-add/imdb-title #12874
```

Gate for any IMDB title URL. Same shape as the person gate вЂ” stub, dedup, sync-call branch, queue upsert.

```text
mvp/imdb/add-imdb-to-master-person #10509
```

Attaches IMDB data to an **existing** `master_person_id`. Preserves inline scrape + LLM + `db.add imdb_person` because the caller owns the `master_person` link. Credits loop delegates to `get-add/imdb-title` (queued, T2, depth+1). **v1.2 (2026-04-24):** added Section 6 вЂ” non-fatal sync call to `#12887 add-award-records-from-imdb-person` after the credits loop. Closes the Phase-8 asymmetry vs orchestrator `#12880` (honorary / person-only awards now captured on main-pipeline entries, not just film-anchored awards via the title cascade). Called from `process-person-phase-8` **#12829**.

```text
mvp/imdb/verify-imdb-url #12677
```

Reverse lookup вЂ” given a `master_person_id`, search Tavily + disambiguate with LLM, return verified IMDB URL. Called from `process-person-phase-8` **#12829**.

### Cascade helpers (fanout from one entity to tier-queued related entities)

```text
mvp/imdb/cascade-credits-from-person #12875
```

Walks `imdb_person.json.credits[]`, tiers by role + rating, queues titles via `get-add/imdb-title` at `depth+1`. Hard cap: top 20 T3 entries by rating. v1.1 `sync_top_credits` input: when `> 0` AND `cascade_depth == 0`, top-N credits by `(tier ASC, rating DESC)` are sync-enriched inline (`queue: false, cascade_depth: 0`).

```text
mvp/imdb/cascade-crew-from-title #12876
```

Walks `film_tv.json`, tiers directors/writers/cinematographers/lead producer/top-3 cast, queues people via `get-add/imdb-person` at `depth+1`. Dedups by `imdb_url` вЂ” first tier wins.

```text
mvp/imdb/cascade-awards-from-person #12877
```

Walks `film_tv_award` rows for a person, queues project titles at T1 / `depth+1`.

```text
mvp/imdb/cascade-awards-from-title #12878
```

Walks `film_tv_award` rows for a title, queues co-winners (who have resolved `imdb_person_id`) at T2 / `depth+1`. `shared_with` names-only co-winners remain unresolved pending future URL-resolution work.

### Graph writers (FalkorDB nodes + edges)

```text
mvp/edge/add-node-edge-imdb-award #2334
```

Writes `Film_TV` + `Film_TV_Award` nodes, `HAS_AWARD` edge, and (when a Person UUID resolves) `WORKED_ON` + `WON_AWARD` / `NOMINATED_FOR` edges to FalkorDB for each `film_tv_award` row. Two-query split so a missing Person stub never kills the whole write. Stamps `film_tv_award.node_uuid` on success.

```text
mvp/node/add-film-tv-node #2320
```

**v2 (2026-05-16):** uses `MERGE (f:Film_TV { imdb_url }) ON CREATE SET ... ON MATCH SET ... SET f:Entity SET f:Movie/TV_Series/...` вЂ” idempotent against existing nodes (typically those created by #2334 in award context during a prior cascade). Cypher built via `api.lambda` (no template-engine markdown autolinker risk). Adds `display_label` + `source` properties for consistency with #2334's writes. Always re-stamps `film_tv.node_uuid` on success. Called from `run-base-imdb-title-enrich` Phase 3.

```text
mvp/node/add-person-node #2601
```

**v1.6 (2026-05-17):** uses `MERGE (p:Person { master_person_id }) ON CREATE SET ... ON MATCH SET ... SET p:Entity` вЂ” idempotent. Pre-v1.6 used `CREATE` and was creating duplicate Person nodes on every `#12880` Phase 4a re-invocation, orphaning the previous node. ON CREATE seeds `uuid` + `visibility=false`; ON MATCH refreshes `name` + `embedding` only. `update-person-node` flips visibility to true once `bio_500` is ready (unchanged). Mirror of #2320 v2.

```text
mvp/node/update-person-node #2758
```

Read-only вЂ” `MATCH (n:Entity {uuid})` + SET properties + embedding + dynamic labels (`:Person`, `:Angel` if is_angel, `:C_Suite` if is_c_suite). No duplicate risk (MATCH-based, no creation).

  <Card title="Fix #1 вЂ” add-node-edge-imdb-award (#2334) v1.4: JSON.stringify Cypher literals" icon="bug">

### Implementation notes captured during build

- **URL normalization lives in get-add:** `trim | split:"/?" | first | rtrim:"/"`. All queue rows and `imdb_person` / `film_tv` rows are keyed on the normalized URL.
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` вЂ” NOT `function.call` (not valid in Xano).
- **`updateTableSchema` drops text `validators`:** the Xano MCP `updateTableSchema` tool silently strips `filters=trim` from text fields when replacing schema. Low impact when inputs are already trimmed upstream, but worth knowing.
- **Queue upsert is the single source of truth:** `get-add` always routes through `upsert-imdb-person` / `upsert-imdb-title`. No raw `db.add` into queue tables anywhere.
- **Kill switch is fail-open:** missing env var в†’ `try_catch` logs to `log_crash` and defaults OFF.
- **`cascade_depth` + `queue` param on get-add:** when both `cascade_depth == 0` AND `queue == false`, `get-add` invokes the orchestrator synchronously after the stub exists and returns `action: "enriched_direct"` вЂ” no queue entry written.
- **Orchestrators are 7-phase with per-phase `try_catch` в†’ `log_crash`:** each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` + stack trace.
- **Phase 1 guard:** if scrape+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early вЂ” no `master_person` gets created from a junk row.
- **Cascade helpers are pure dispatchers:** every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"`. Every `function.run` to `get-add` is wrapped in `try_catch` в†’ `log_crash`.
- **Seed ID propagation:** a depth-0 seed invocation via `get-add` must set `seed_imdb_person_id` / `seed_film_tv_id` to the just-created stub's own id when no explicit seed is passed. Required for loop-prevention to work.
- **Top-N sync credits on person seeds (`sync_top_credits: 5`):** `cascade-credits-from-person` v1.1 sync-enriches the top 5 credits inline when the person is at `cascade_depth == 0`. Guarantees a seed returns with enriched films even before the title cron runs.
- **Cron single-row tick pattern:** modeled on `process-person-queue` **#126** / `process-company-queue` **#127**. Lock-on-pick, `try_catch` around orchestrator, release lock on failure (the company/person equivalents leave rows permanently locked on crash вЂ” the IMDB crons fix this).
- **Missing-stub guard:** if the queue row's `imdb_person_id` / `film_tv_id` is 0, write `log_crash` and `db.del` the queue row вЂ” prevents a bad row from monopolizing the lock forever.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **shared_with** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person + company waterfall solves this for LinkedIn/PDL/Enrich Layer. This plan applies the same pattern вЂ” **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** вЂ” to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits).
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) + top 3 cast.

---

## Depth model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
|:-----:|---------|----------|
| **0** | Seed (user-initiated) | Full scrape + LLM extract + awards + cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` + name/title, no scrape). Queued. Cron processor scrapes + expands, but **only T1вЂ“T2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes + writes data, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits + 9 acting + 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director + writer + DP + lead producer + 3 cast = ~7 people в†’ ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000+.
</Check>

## Priority tiers

### `queue_imdb_person` (1 = highest)

| Tier | Rule | Example (seed = title "Gamer") |
|:----:|------|-------------------------------|
| **1** | Director OR Writer of seed title; award-winning/nominated person on seed | Mark Neveldine, Brian Taylor |
| **2** | Cinematographer OR lead Producer of seed title; co-winner / co-nominee on seed's award (via `shared_with`) | Ekkehart Pollack (DP), lead producers |
| **3** | Top 3 cast of seed title; other `shared_with` people | Gerard Butler, Michael C. Hall, Amber Valletta |
| **4** | Everyone else mentioned | Minor cast, thanks, video-game voice actors |

### `queue_imdb_title` (1 = highest)

| Tier | Rule | Example (seed = person "Neveldine") |
|:----:|------|------------------------------------|
| **1** | Seed person was Director OR Writer on this title; award-winning/nominated title | Crank, Gamer, Ghost Rider, Crank: High Voltage, Jonah Hex, Pathology |
| **2** | Seed person was lead Producer (not exec) AND (rating в‰Ґ 7 OR year = current_year) | Kid Revolver (8.2 Short), Great Pretenders (8.9 Short), The Olympians (upcoming) |
| **3** | Seed person was executive producer / cinematographer / minor role; OR rating в‰Ґ 7 regardless of role | Officer Downe, Urge, Pathology (producer), Vatican Tapes (director) |
| **4** | Video games / music videos / shorts without high rating | Jonah Hex (Video Game, tt38040358) |

### Credit filter for person seed (cascade-credits-from-person)

```text
for each credit in imdb_person.json.credits:
    if role в€€ {director, writer}:                                в†’ T1
    elif role = producer AND (rating в‰Ґ 7 OR year = current_year): в†’ T2

```

### Key crew filter for title seed (cascade-crew-from-title)

```text
from film_tv.json:
    directors[]                                      в†’ T1
    writers[]                                        в†’ T1
    cinematographer (if present in json)             в†’ T2

```

<Info>
**"Top cast = top 3" is a starting point** вЂ” may tune upward (e.g., top 5 or top 10) once we see real fanout numbers. The tier assignment is what matters; the count is a knob.
</Info>

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

### Depth-1 queued processing

When the cron picks up a queue entry, it sets `processing: true` and calls the same orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P6) only queue T1вЂ“T2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured вЂ” but no children are queued.

## Loop prevention (five layers)

1. **Dedup gate in `get-add`:** `imdb_url` already in `imdb_person` / `film_tv` with `enriched_at` within 30 days в†’ return existing, no scrape.
2. **Queue upsert:** duplicate `imdb_url` in queue в†’ `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth в‰Ґ 1:** only T1вЂ“T2 discoveries are queued at depth 2 (T3вЂ“T4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.
