## Why this exists

Every node in the graph holds property values that have been merged from multiple sources (PDL, Enrich Layer, Fundable, user-entered, customer-uploaded, public web). For DSAR fulfillment under GDPR Article 15 and source disclosure under Article 14, we need to be able to answer two questions on demand:

1. **For any node, where did each property value come from?**
2. **For any source, which property values across the graph came from it?** (e.g., "show me everything sourced from PDL")

The granular source data is preserved in the enrichment landing tables. This page describes the lightweight provenance layer that sits **on the node itself** — so the reveal tool can answer the above questions without walking back through join tables or doing code archaeology over the merge logic.

<Note>
  This is Phase 1 scope — sufficient for **manual DSAR fulfillment**. Property-level revision history (what a value used to be before it was replaced) is deferred to Phase 2+.
</Note>

## The pattern: JSONB provenance blob

A single `field_provenance` JSONB column on each node table that holds third-party-sourced data, plus three node-level summary columns.

### Schema additions

Every node table (`person_node`, `company_node`, and any other node type holding sourced PII or third-party data) gets:

| Column | Type | Purpose |
|---|---|---|
| `created_at` | `timestamptz` | When the node was first created in Orbiter |
| `updated_at` | `timestamptz` | When the node was last touched (any field) |
| `last_modified_source` | `text` | Which source last contributed to this node |
| `field_provenance` | `jsonb` | Per-property source metadata (see below) |

### Structure of `field_provenance`

One key per property on the node. Value is a `{ source, source_id, updated_at }` object describing where the **current value** of that property came from and when it landed.

```json
{
  "<property_name>": {
    "source": "<source_name>",
    "source_id": "<id_in_source_landing_table>",
    "updated_at": "<ISO 8601 timestamp>"
  }
}
```

<Tip>
  `source_id` is the foreign key back to the granular landing table for that source (e.g., `pdl_records.id`, `enrich_layer_records.id`). This is how the reveal tool drills from "node-level provenance summary" to "full source record" when a DSAR or audit demands it.
</Tip>

### Reading the example

- The top of the record is the **merged current state** — what every consumer of the node (UI, intelligence engine, scoring) sees and reads.
- `field_provenance` is the parallel structure: one key per property, with provenance for the *current* value.
- In this example: PDL gave us firmographics and identifiers, Enrich Layer gave us descriptive/visual fields, Fundable gave us funding and ownership data.
- `last_modified_source` is `"EnrichLayer"` because Enrich Layer's update at `2026-05-15` was the most recent merge into the node.

## Merge-time write logic

The enrichment pipeline must write `field_provenance` entries at merge time. This is the single most important implementation requirement on this page.

<Steps>
  <Step title="Determine which source wins each property">
    When merging source data into a node, the merge logic decides which source contributes the final value for each property (by recency, source priority, confidence score, or whatever rule applies).
  </Step>
  <Step title="Write the provenance entry">
    For each property whose value was contributed or replaced in this merge:
    ```
    node.field_provenance[property] = {
      source: <winning_source_name>,
      source_id: <id_in_source_landing_table>,
      updated_at: <merge_timestamp>
    }
    ```
  </Step>
  <Step title="Update node-level summaries">
    - Set `node.updated_at = <merge_timestamp>`
    - Set `node.last_modified_source = <source_with_most_recent_property_contribution>`
  </Step>
</Steps>

<Warning>
  If today's merge logic does not record which source contributed each final property value, **that is the gap to close before pilot.** Source landing tables alone cannot reconstruct provenance for non-trivial conflicts — only the merge step knows which value won and why. If we don't capture that decision at merge time, we cannot answer DSAR Article 15 source disclosure without code archaeology.
</Warning>

## Indexing

GIN index on `field_provenance` enables fast JSONB queries across the node table:

```sql
CREATE INDEX idx_company_node_provenance
  ON company_node
  USING GIN (field_provenance jsonb_path_ops);
```

Use `jsonb_path_ops` rather than the default GIN operator class — it's smaller and faster for containment queries, which is most of what we'll be running against this column.

## Query examples for DSAR / audit

### "Where did each field on this Frame.io node come from?"

```sql
SELECT
  key AS field,
  value->>'source'      AS source,
  value->>'source_id'   AS source_id,
  value->>'updated_at'  AS updated_at
FROM company_node, jsonb_each(field_provenance)
WHERE id = 'company_node_frameio_uuid'
ORDER BY key;
```

### "Which fields on this node came from PDL?"

```sql
SELECT key AS field
FROM company_node, jsonb_each(field_provenance)
WHERE id = 'company_node_frameio_uuid'
  AND value->>'source' = 'PDL';
```

### "Show me every company node that has any field sourced from Fundable"

Useful for source-level retraction scenarios (e.g., if a source retracts data or we need to re-run a particular enrichment).

```sql
SELECT id, name
FROM company_node
WHERE field_provenance @> '{"funding_total_usd": {"source": "Fundable"}}'::jsonb
   OR field_provenance @> '{"ownership_status":  {"source": "Fundable"}}'::jsonb;
```

For broader "any field from Fundable" queries, we can extend the reveal tool to iterate keys, or store a precomputed `sources_present` text array on the node as a denormalized index.

## Reveal tool

Mark builds a function (call it `getNodeProvenance(node_id)`) that returns the `field_provenance` object plus, for any property the caller asks to drill into, the full source landing record via `source_id` lookup. This is the function the manual DSAR operator uses to assemble the Article 15 response.

Minimum interface:

```
getNodeProvenance(node_id)
  → returns {
      node_id,
      node_type,
      properties: [
        { field, value, source, source_id, updated_at }
      ],
      created_at,
      updated_at,
      last_modified_source
    }

getSourceRecord(source, source_id)
  → returns the full record from the corresponding landing table
```

## Phase 1 scope

In scope, must ship before pilot:

- `field_provenance`, `created_at`, `updated_at`, `last_modified_source` columns on every node table holding third-party-sourced data (start with `person_node` and `company_node`).
- Merge pipeline writes provenance at merge time, including the source identity and source landing-table `source_id`.
- GIN index on `field_provenance` per node table.
- `getNodeProvenance` / `getSourceRecord` reveal functions exposed for internal use (operator-facing, not user-facing yet).

## Out of scope for Phase 1

- **Property-level revision history** — capturing what a value used to be before it was replaced by a better source. Deferred to Phase 2; we'll either extend `field_provenance` values to arrays, or add a `node_property_history` table.
- **Self-service DSAR portal** — manual fulfillment is sufficient for pilot volume.
- **Cross-graph entity resolution** — different scope (the question of "is Jane Doe in User Graph the same Jane Doe as in Universe Graph and Acme's Org Graph"). Tracked separately.
- **Retention-driven auto-deletion** — retention schedule is a Phase 2 deliverable from MoFo, no point automating it yet.

## Open design question: edge provenance

Edges with `valid_from` / `valid_to` need the same provenance pattern, but the mechanics differ depending on what's changing:

- **Relationship change** (Jane left Acme on Jan 15, joined NewCo on Jan 16) → tombstone the old edge with `valid_to`, create a new edge with `valid_from`. Each edge has its own `field_provenance`.
- **Property correction on an existing edge** (Jane's title at Acme was "Engineer" per PDL, corrected to "Senior Engineer" per Enrich Layer — same relationship, same time period, better source) → in-place update with new `field_provenance` entry for the corrected property.

These two mechanics need to be cleanly distinguished in the merge logic. Worth a design pass before shipping edge provenance — going back and untangling them later is painful.

## Dependencies on legal work

Two things land via MoFo's Phase 1 and feed back into how this gets used (but don't block the build):

- **Lawful basis tagging.** Once LIAs are documented, we may want to add a `lawful_basis` field to each `field_provenance` entry (or as a separate node-level tag). Mark can stub with a placeholder enum value now and populate once MoFo confirms.
- **Universe Graph deletion rules.** Different controller posture, may need different deletion semantics than User Graph or Org Graph. Build the deletion *capability* in the reveal tool, gate execution behind operator decision until the legal framework lands.
