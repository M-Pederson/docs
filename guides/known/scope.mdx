This is the implementation scope for the Known relationship model. Every Xano function, cron job, and trigger is organized by block. Each block has its own sub-page detailing the functions, inputs/outputs, cadences, and implementation notes.

Nothing below has been built yet. This is the build-order reference.

See [Schema Design](/guides/known/schema-design) for field definitions and [Documentation](/guides/known/documentation) for concepts.

## Build order (priority)

**Phase 1 — foundation (cannot build anything else without this):**
1. [Helper functions](/guides/known/scope/helpers) — canonicalize_pair, create_knows_edge, append_evidence, check_evidence_floor, create_facet, invalidate_facet, derive_known_since
2. [Staging table](/guides/known/scope/staging) — insert, lookup, consume, expire
3. [Shared context cache](/guides/known/scope/cache) — compute_shared_context (without cron yet — just the function)

**Phase 2 — ingest (data starts flowing in):**
4. [Strong-source ingest functions](/guides/known/scope/ingest) — email_direct, calendar_meeting, phone_contact, manual_user_add
5. [Weak-source ingest functions](/guides/known/scope/ingest) — linkedin_connection, shared_employment_inferred, press_co_mention
6. [Interaction count maintenance](/guides/known/scope/counts) — per-modality count functions
7. [Household node functions](/guides/known/scope/households) — create_household, add_resident, end_residency, match_or_create_by_address
8. [Introduction node functions](/guides/known/scope/introductions) — create_introduction, infer_from_email_cc
9. Ingest crons (hook into Nylas pipelines, LinkedIn sync, etc.)

**Phase 3 — scoring:**
10. [compute_reciprocity, compute_decay_half_life, compute_strength_score](/guides/known/scope/scoring)
11. [compute_cohabitation_overlap](/guides/known/scope/households) (used by shared context cache)
12. [update_peak_watermarks, apply_freeze_if_deceased](/guides/known/scope/scoring)
13. [recompute_all orchestrator](/guides/known/scope/scoring)
14. Scoring cron

**Phase 4 — lifecycle automation:**
15. [bonded_creator + bonded_invalidator](/guides/known/scope/lifecycle) with hysteresis
16. [professional_tie_maintainer](/guides/known/scope/lifecycle) (trigger)
17. [kinship_divorce_cascader](/guides/known/scope/lifecycle) (trigger — handles in-law + step-relation cascades)
18. [mentor_invalidator](/guides/known/scope/lifecycle) (nightly cron — invalidates stale inferred MENTOR edges)
19. [deceased_freezer](/guides/known/scope/lifecycle) (trigger)

**Phase 5 — LLM layer:**
20. [llm/sentiment_pass](/guides/known/scope/llm)
21. [llm/relationship_classifier](/guides/known/scope/llm) (proposes inferred MENTOR edges for asymmetric interaction patterns)
22. [estrangement_detector](/guides/known/scope/lifecycle) (depends on sentiment pass)
23. [reconciliation_detector](/guides/known/scope/lifecycle) (depends on sentiment pass)
24. [llm/estrangement_reviewer, llm/reconciliation_reviewer](/guides/known/scope/llm)

**Phase 6 — query + UI:**
25. [Query helpers](/guides/known/scope/queries) — get_shortest_path, get_super_connectors, get_household_history, etc.
26. Frontend integration

**Phase 7 — identity merge (highest-risk, build last):**
27. [Identity merge algorithm](/guides/known/scope/identity-merge) — `execute_merge`, `merge_knows_pair`, `merge_facet_pair`, `reparent_edge`, tombstone logic
28. `MergeEvent` node type creation + indexes
29. `redirect_person_uuid` query helper (resolves tombstoned uuids to canonical)
30. Integration with the external identity resolution pipeline

**Note on migrations:** Known is being rolled out on a **fresh graph**. No v1 → v2 migration is needed because no v1 edges exist in the target environment. The v1 → v2 MENTOR backfill cypher at [Schema Design → MENTOR → Migration](/guides/known/schema-design/edges#migration-v1-v2) is retained as a reference pattern only, not as a prerequisite step.

## Open implementation questions

Questions deferred from the design workshop. Should be resolved during implementation, not before.

- **Scoring formula weights:** The strength_score formula has coefficients `w1` through `w7`. Initial values need to be tuned against real data — the design doc uses illustrative scoring but does not commit to specific weights.
- **Shared experience bonus time decay:** The `shared_experience_bonus` term in the scoring formula is currently time-independent — shared employment 20 years ago contributes as much as shared employment 2 years ago. Recent shared experiences should probably count more. Needs a decay function applied to historical shared context, with its own half-life. Scoring tuning decision, not a schema change.
- **Presence floor value:** Currently proposed as 8 (out of 100). Tune against real edges once scoring runs on production data.
- **`shared_context_cache` refresh strategy:** Full nightly recompute vs incremental on upstream trigger. Start with full nightly and optimize if performance warrants.
- **LLM sentiment cost budget:** Sentiment passes over every recent interaction for every `KNOWS` edge may be expensive. Cron should batch and prioritize — only run sentiment on edges flagged by other signals (e.g. recent context terminus).
- **Evidence coverage granularity:** `full | partial | none` may need finer resolution for calendar (per-account) and email (per-mailbox).
- **`bond_type` LLM fallback:** When the cron creates an inferred `BONDED` but none of the derivation rules match (no spouse, no co-parent, no co-founder), what `bond_type` does it get? Default to `['friendship']` or defer to LLM classifier?
- **Hysteresis thresholds (70/55):** Initial values. Tune against real data.
- **Staging expiry window (730 days):** Initial value. Tune based on how often retroactive matches actually occur.
- **Origin context tiebreaker:** Current rule is "earliest-in-time wins." When timestamps tie or are equally valid, need a secondary tiebreaker.
- **Household merging:** When ingest realizes two separate `Household` nodes refer to the same physical residence, how are they merged? Residency edges, overlap windows, address_hash collisions all need a strategy.
- **Identity merge concurrency:** If two merge operations are in flight simultaneously touching overlapping persons, the second should detect the in-flight state and abort. Needs a distributed lock or optimistic concurrency control. See [Identity Merge — Known limitations](/guides/known/scope/identity-merge#known-limitations).
- **Identity resolver match threshold:** At what confidence does the identity resolver automatically call `execute_merge` vs flagging for review? Start conservative (≥0.95 auto, 0.8–0.95 review queue).

## Not in scope (yet)

These were discussed during design but are explicitly out of scope for initial implementation:

- **`CoFoundingGroup` node type** — a bipartite hyperedge for "this set of people co-founded this company together as a unit." Would enable queries like "everyone who has ever co-founded anything with Alice, regardless of company" in one hop. Currently handled (lossy) via pairwise `PROFESSIONAL_TIE` edges + the existing `Company` node. Worth adding later if the query pattern becomes important.
- **`PolyamorousGroup` node type** — a bipartite hyperedge for "this set of people is in a committed polyamorous group as a unit." Currently handled via pairwise `ROMANTIC` + `BONDED` edges (which work correctly, since Known's uniqueness constraint is per-pair, not per-person). A `PolyamorousGroup` node would enable one-hop queries like "find all polyamorous triads/quads in the graph" without clique-detection traversal. Defer unless the query pattern becomes important.
- **Evidence records as nodes** — currently stored as a JSON array on `KNOWS.evidence`. Promoting each evidence row to a node would enable queries like "how many KNOWS edges were created from LinkedIn evidence this month" but ~3-10x the node count. Defer unless needed.
- **Graph visualization** beyond basic UI display
- **Relationship strength time-series charts** — would require storing historical strength_score snapshots, which we do not currently plan to do (peak watermarks are the only historical fields)
- **Multi-hop path scoring** — e.g. "how many degrees of separation" — infrastructure is in place via the compositional edge model, but scoring formulas and queries are not yet defined
- **Relationship prediction** (predicting future estrangement, future bond formation) — ML layer, separate project
- **Cross-workspace relationship sharing** — relationships currently scoped to the single Orbiter graph
