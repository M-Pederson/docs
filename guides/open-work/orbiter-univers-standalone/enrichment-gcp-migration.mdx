## Intent

Rebuild the four enrichment pipelines — **Person**, **Company**, **Music**, and **Film & TV** (IMDb-sourced) — as **Go services on GCP**, writing into a **greenfield [Orbiter Universe](/guides/open-work/orbiter-univers-standalone)**: a brand-new FalkorDB graph \+ AlloyDB, standalone from day one. This is **not an in-place migration** — there is no backfill of the existing graph. The Go pipelines populate a clean Universe; the current Xano stack keeps serving the product, in parallel, until each pipeline is cut over.

The hard part is not the Go — it's keeping the rebuild **faithful**. We reproduce the pipeline as-is — the **data-correctness contracts** (graph node/edge shape & MERGE keys, dedup/identity, provenance, provider contracts, idempotency) **and** the **orchestration / queueing / LLM flow**. The default is parity; a developer may redesign part of the flow only after discussing it with Mark and getting sign-off. To make "faithful" measurable, two things are front-loaded before any Go is written: **hard-freeze the Xano enrichment** as a stable reference, and **turn that frozen XanoScript into a language-neutral spec \+ golden fixtures**.

<Note>
  **Why not "just rewrite it."** The pipelines are deep call graphs (the Person orchestrator alone is 13 phases; Company is a 15-step gated orchestrator) wired to ~8 external providers, an LLM layer, and a graph with strict MERGE/idempotency semantics. Rewriting from reading XanoScript would lose the load-bearing details — `first_notnull` zero-defaults, `coalesce(created_at)` on re-enrich, the 1 req/s MusicBrainz gate, the bounded-cascade fanout rules. The methodology below captures those _as contracts_ — the flow around them is fair game to redesign.
</Note>

## Target architecture

Settled in the AlloyDB docs and adopted here unchanged — except the graph is **new** (Universe), not the existing one:

- **Compute:** Go on **Cloud Run**. Thin authenticated HTTP API \+ Pub/Sub-push workers (Cloud Run jobs for the bulk Fundable seed). Idempotent workers via `idempotency_key` \+ row-lock `UPDATE … RETURNING`; DLQ \+ bounded retries. No Xano orchestration, no Python/Node workers.
- **Storage:** **AlloyDB Postgres** is canonical. Vectors as native `pgvector` columns indexed with **ScaNN** (`alloydb_scann`, cosine) — chosen org-wide for inline metadata filtering. JSONB for raw provider payloads, GCS for files.
- **Graph (new):** a **standalone Universe FalkorDB graph**, built fresh. AlloyDB owns **vector retrieval \+ structured state \+ scoring**; FalkorDB owns **graph structure \+ RAG reasoning** (the "why"). The existing FalkorDB vector-writing functions define the **node/edge envelope contract** the Go writers must reproduce (decision #3) — but they write into the clean graph, no migration of old data.
- **Embeddings:** Universe nodes carry **Gemini `gemini-embedding-001` (1536, L2-normalized)** only (decision #1; standard updated 2026-07-06). OpenAI 1536 is not written into Universe.
- **Read path:** fused NL query (LLM hard-filters \+ ScaNN top-K) in AlloyDB, then graph RAG over FalkorDB (decision #6).
- **Go stack:** `pgx/v5` \+ `pgxpool` \+ `pgvector-go` \+ the AlloyDB connector with IAM auth (Workload Identity, no passwords); two pools (write / read-replica).
- **Template:** the [YuNet Build Plan](/guides/open-work/yunet-avatar-pipeline/build-plan) is the canonical Go-native build-plan shape (Intent → Contracts → Architecture → Go boundaries → AlloyDB model → Queue flow → Observability → IAM → runbook). Each pipeline's build plan follows it.

## The migration, step by step

<Steps>

    - **Verify master\_person creation from a non-LinkedIn profile URL** — confirm a person record is created correctly when the source profile URL is not LinkedIn (e.g. IMDb / MusicBrainz / a company team page), so that path is in a known-good state before freeze.

  <Step title="Capture golden data-contract fixtures">
    While the frozen pipeline is still live, record representative **real inputs and the exact Xano outputs** they produce — the target the Go rebuild diffs against. Run against an **isolated data source** (sandbox / the staging Cinq fixture), never live.

    Per pipeline, a corpus of **10–30 cases** spanning happy-path, thin-data, dedup-hit, explicit-zero defaults, and known edge cases (Zeno Rocha for Person; Cinq/Sub Pop/Nirvana for Music; a seed company with Exa founders; an IMDb title\+person with awards). For each: the input, the resulting **graph subgraph** dumped via Cypher (nodes\+edges\+properties), the key **table write-set** fields, and the **raw LLM \+ provider request/response pairs** captured at fixture time. Snapshot provider responses so fixtures are **replayable without re-hitting rate-limited APIs**. Version-pin everything to the Step 1 frozen reference.

    The four pipelines share one substrate. Building it inside each pipeline produces **four divergent clients** — exactly the graph-envelope, `vecf32` array-vs-string, and reasoning-empty-content traps that already bit the Xano side. This is the **Stage-0 foundations sprint** (decision #4): all four devs build these versioned internal Go packages **with tests, before any pipeline work**.

    | --- | --- |
    | `internal/graph` | Universe FalkorDB client; standard MERGE envelope helper (`coalesce(created_at)`, `timestamp()`, `randomUUID`, weight); upsert-by-key; `'Label' IN labels(a)` convention; vector helper accepting array **and** bracket-string |
    | `internal/llm` | OpenRouter client; model registry from the model-summary pages; strict JSON-mode parsing; retry-on-empty-content (reasoning-budget trap) |
    | `internal/embeddings` | **Gemini `gemini-embedding-001` (1536, L2-normalized)** embed against the AlloyDB ScaNN index \+ the Universe graph (decision #1 — no OpenAI 1536) |
    | `internal/datasource` | One client per provider behind a shared **fail-open rate limiter**; recorded-response replay mode for tests |
    | `internal/provenance` | Canonical source-string constants \+ the "source rides the record" rule |
    | `internal/queue` | Depth / tier / upsert primitives |
  </Step>
  <Step title="Rebuild each pipeline in Go, parity-test the data contracts">
    Reimplement each pipeline against the Step 2 spec, **continuously validated by the Step 3 fixtures**. Per decision #3 this is **parity-first** — reproduce the flow as specified; a dev takes any proposed redesign to Mark _before_ building it, not after. The harness diffs:

    1. **Field-level table diff** — _exact_ for structured/identity fields (ids, mbid, canonical domain, dedup keys, tiers, source strings); _tolerant_ for LLM prose (assert presence, length ceilings, JSON well-formedness, optional cosine/judge similarity).
    2. **Graph-shape assertions** — node/edge sets by label\+key and `(type, endpoints, weight)`; envelope invariants. Run each fixture **twice** and assert idempotency: counts don't grow, uuids stable, `created_at` unchanged on re-run while `updated_at` moves.
    3. **Outcome assertions** — dedup-wins, no runaway fanout, tier resolution — plus the orchestration order where it's observable; an unexplained deviation flags for review unless Mark has signed it off.

    Provider clients run in **replay mode** against the Step 3 cache, so CI runs are deterministic and free.
  </Step>
  <Step title="Seed the new Universe">
    The Universe graph starts empty and there's no backfill (decision #2). It's populated from two explicit sources (decision #8), driven through the Go pipelines into the clean graph \+ AlloyDB:

    - **Mark-prepared list** — a curated text list of **people & companies** Mark supplies; fed in as enrichment requests.
    - **Bulk seed from the Fundable tables** — the existing Fundable company/investor data, drained through a **Cloud Run job** paced by the shared fail-open rate limiter (so the provider re-hits don't blow MusicBrainz/PDL/Exa quotas).

    This doubles as a **scale validation**: run the same seed entities through the frozen Xano reference and diff the data-contract outputs in aggregate (node/edge counts, dedup collision rate, null-description rate, embedding presence) before the product depends on Universe. Forward product activity continues to enrich into Universe on top of the seed.
  </Step>
  <Step title="Cut the product over per pipeline (projection sync)">
    Switch the product to Universe **one pipeline at a time** (decision #5), lowest-blast-radius first, with the frozen Xano stack retained as instant rollback. The product reads via **projection sync** (decision #9): Universe projects its read-model card tables (`imdb_person_card`, `film_tv_card`, `music_entity_card`, …) one-way into the App's AlloyDB, and the App reads those — it never couples to Universe internals.

    Recommended order: **Film & TV or Music first** (newer, best-documented, most self-contained) → **Company** → **Person** last (deepest cascade, most callers). Each cutover is a feature-flag/percentage rollout with a rollback procedure and post-cutover verification queries. Retire each Xano pipeline only after its Universe replacement holds data-contract parity in production for a defined window.
  </Step>
</Steps>

## Documenting the pipelines: comment the XanoScript, or generate a spec?

You asked whether to run a **skill \+ agent to add comments into all the Xano function code**. Recommendation: **don't comment the live XanoScript. Auto-generate a read-only spec instead.**

<Warning>
  **Why not inline comments.** Commenting the functions means _writing to the baseline you just hard-froze in Step 1_. Every `updateFunction` round-trip risks the documented serializer traps — inline `{…}`/`[…]` literals silently rewritten into dead backtick template strings, `function.run` targets blanked, `filters=trim` stripped. Touching ~100\+ functions just to annotate them is a high-probability way to corrupt the very thing you're trying to preserve. And the comments would live where the 4 Go devs will never look — they read Go and a spec, not XanoScript.
</Warning>

Instead, build a **read-only extraction skill** that mirrors the proven pattern of [`cypher-sync-audit`](/guides/cyphers) and `check-models` — _Xano is source of truth, docs are the mirror, scan → report → confirm → write-docs → verify_:

- Fetch each frozen function's XanoScript via `getFunction` (`include_xanoscript: true`, batched 5–8 in parallel — the same substrate the existing audit skills use).
- **Emit a structured per-function spec sheet** into the docs tree (the Step 2 deliverable) — I/O contract with `first_notnull`/`first_notempty` flags, call graph, data-source calls, the graph edge/node catalog, cross-links into the existing prompt/model pages.
- **Never write back to Xano.** Zero blast radius. The output is _the hand-off artifact_, it's diffable, and it can be drift-audited later exactly like `cypher-sync-audit` audits Cypher.

<Note>
  If a handful of genuinely cryptic functions need clarification, capture it in the spec's **notes column** — not as a source edit. Inline `//` comments remain available only as an explicit, opt-in, **sandbox-first, one-function-at-a-time** mode (additive-only, routed through `updateFunction`, round-trip-verified by re-fetch\+diff) — not the default, and not for a bulk pass.
</Note>

## Splitting the work across 4 developers

Per decision #4 — **do not start one-dev-per-pipeline.** That cut builds the graph/LLM/embeddings/data-source substrate four times and diverges. Use a two-stage model:

<Steps>
  <Step title="Stage 0 — Shared-foundations sprint (~1–2 weeks, all 4 devs converging)">
    The two strongest engineers own the `internal/*` packages (Step 4); the other two contribute the provider clients they'll most depend on (Person dev builds the PDL/Enrich-Layer/Exa client; Music dev builds the MusicBrainz client \+ the fail-open rate limiter). **All four pair through this** so everyone internalizes the shared contracts and the Xano traps before splitting. Exit criterion: all `internal/*` packages versioned, unit-tested, frozen-interface. The spec (Step 2) and fixtures (Step 3) can run in parallel here since they only read the frozen Xano reference.
  </Step>
  <Step title="Stage 1 — One pipeline per dev (now safe)">
    Each dev owns spec → rebuild → seed → cutover for one domain, building strictly on the Stage-0 packages.

    | Developer | Pipeline | Rationale |
    | --- | --- | --- |
    | Graph-package co-owner | **Person** | Deepest cascade, most callers, the `master-person-new` migration — give it to whoever knows the graph envelope best |
    | Dev B | **Company** | Mid-complexity; needs the missing table schema authored during spec; the Fundable bulk seed lands here |
    | Dev C | **Music** | Newer, design-first, cleanly documented — most self-contained |
    | Dev D | **Film & TV** | Newer, self-contained, good docs — natural first cutover |
  </Step>
</Steps>

<Tip>
  Keep one package owner as a part-time **"foundations steward"** through Stage 1, so a cross-cutting change (a new edge-envelope field, an OpenRouter model swap) lands once in the shared package rather than four times in pipelines. Sequence the cutovers (Step 7) — never simultaneous — so the team reuses the runbook on the smallest pipeline first.
</Tip>

### What actually gates the schedule

The dev clock starts only once Step 1 is done. After that, the critical path is no longer dev velocity — it's the **short serial prefix** plus **calendar-bound validation**:

```text
[Step 1 gate] → spec + fixtures → [foundations + parity harness] → pipeline rebuild → cutover soak
```

<Warning>
  **The parity harness \+ fixtures are the AI multiplier — build them first (Steps 3–4).** An objective, automated acceptance gate is exactly what lets an AI agent iterate to green autonomously. Without it, AI development on Step 5 degrades back toward human-review-bound velocity — and the 4-week estimate goes with it. This is the highest-leverage early investment, more than any pipeline code.
</Warning>

<Tip>
  **The 4 weeks assumes the calendar-bound items are kicked off on day 1 of the dev clock (start of Step 2), not deferred to the foundations step.** AlloyDB provisioning, IAM/security review, the ScaNN org-flip, and Fundable-seed sizing all provision in parallel with the spec \+ fixtures work — so by the time the AI has pipelines passing parity, the infra is ready instead of being the thing you wait on. In the AI model, the Go/devops engineer owns this infra kickoff plus the harness and the merge/cutover discipline that keeps 4 parallel AI-driven streams coherent.
</Tip>

## Key risks

<AccordionGroup>
  <Accordion title="LLM non-determinism breaks naive parity">
    Bios, descriptions, expertise, awards, edge descriptions return free text that will never byte-match. Strict diffing makes every run "fail" and devs learn to ignore parity. **Mitigation:** field-type-aware tolerance (exact for structured, semantic/shape tolerance for prose); assert the _prompt\+model contract_ from the fixture, not the exact words.
  </Accordion>

  <Accordion title="Graph write-semantics divergence in the new Universe">
    The reference relies on `MERGE` \+ `coalesce(r.created_at, timestamp())`, stable `randomUUID()`, the shared `:Entity`-base vector index (not per-node-type), and `'Label' IN labels(a)`. A fresh Go writer can silently reset timestamps, double-create edges, or write embeddings that aren't KNN-searchable. **Mitigation:** encode the envelope in one shared helper (Step 4); assert graph _shape_ \+ `created_at` stability across a re-run.
  </Accordion>

  <Accordion title="Data-source rate limits, contracts & terminal-status rules">
    MusicBrainz = 1 req/s, required User-Agent, 400/404 terminal. PDL/Exa/Enrich-Layer/FullEnrich have quotas, async webhooks, pagination. The **bulk Fundable seed** (Step 6) can blow quotas if unpaced; contract drift looks like a Go bug. **Mitigation:** the shared fail-open rate limiter \+ recorded-response replay; pace the seed via Cloud Run jobs; pin provider API versions; snapshot contracts in the spec.
  </Accordion>

  <Accordion title="Xano-specific default coercion (first_notnull vs first_notempty)">
    Defaulted bools and ints-that-can-legitimately-be-0 (`priority_tier 0`, `cascade_depth 0`, bootstrap rows) **must** use `first_notnull` or an explicit 0 is coerced back to the default. A dev reading "defaults to 4" silently breaks the bootstrap lane and tier-upgrade logic. **Mitigation:** the spec flags every defaulted field with its semantics; fixtures include an explicit-zero case.
  </Accordion>

  <Accordion title="Unapproved flow deviation introduces a regression">
    Parity-first (decision #3) means the flow is reproduced as specified — the risk is a dev quietly "improving" orchestration/queueing and dropping an outcome the old flow guaranteed: dedup-wins, lowest-tier-wins, bounded (non-runaway) fanout. These only break at scale, not in single-fixture tests. **Mitigation:** any flow change goes to Mark _first_; spec the order \+ outcomes explicitly; assert them with multi-entity fixtures; validate aggregate distributions during the Step 6 seed.
  </Accordion>

  <Accordion title="Provenance migration is half-done in the frozen reference">
    `source` rides on the record (`work_experience.source` → inherited by created company/link) while legacy `data_source_id` inputs remain transitionally. The Go rebuild must reproduce this hybrid faithfully or mislabel provenance. **Mitigation:** the hard freeze pins the migration state; encode canonical strings \+ the inheritance rule in `internal/provenance`.
  </Accordion>
</AccordionGroup>

## Deferred to the per-pipeline build plans

Second-order questions the decisions above raise — build-plan detail, not PRELIM-level forks:

- **Projection-sync cadence & event format** — push vs poll, and the Universe→App update-event schema for the card tables.
- **Forward-enrichment trigger** — exactly which product actions enqueue enrichment into Universe after the initial seed.
- **Fundable bulk-seed volume & pacing** — row count, the Cloud Run job's concurrency, and the per-provider budget for the one-time drain.
- **ScaNN tuning** — quantizer choice beyond `MODE=AUTO` and per-query `scann.*` GUCs, from first-benchmark numbers.
- **Provider set review** — confirm the Go rebuild keeps the same providers (PDL, Enrich Layer, Exa, Fundable, MusicBrainz, IMDb scrape, Serper) or swaps any.
