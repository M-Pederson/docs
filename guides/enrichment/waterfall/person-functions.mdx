## Entry-gate stages \+ gate helpers

<Card icon="cloud-arrow-up">
  ```text
  mvp/enrich/cascade-person-data — #13037
  ```

  **v1.4 (2026-06-01) — the pure data-fetch cascade** (stage 1 of the gate; runs only when the pre-API dedup misses AND `cascade_depth == 0`). Order: PDL `/v5/person/enrich` → Enrich Layer Tier 2 (email path `api/enrich_layer/person-email` #12734, LinkedIn path `api/enrich_layer/person-linkedin` #12612) → PDL retry with the EL-derived LinkedIn URL (promoted only on `status == 200` AND `likelihood > 5`) → Exa Tier 3 `get-exa-profile` #12936. Returns one object: `pdl`, `pdl_status`, `pdl_email_404`, `el`, `el_linkedin_url`, `el_links` (LinkedIn \+ Twitter \+ Facebook URLs, deduped), `el_profile_pic`, `exa`, `linkedin_url`, `input_email_source`, and `needs_full_enrich` (true when the EL-via-email path surfaced no LinkedIn URL — the caller's trigger for the async [FullEnrich tier](#fullenrich-async-contact-tier)). **v1.4** treats a PDL 200 with empty `data.experience` as thin data, so it continues through EL / retry / Exa instead of stopping. Zero DB writes except `log_crash` on a caught exception; every tier is try\_catch-isolated so an EL failure never blocks Exa.
</Card>

<Card icon="filter">
  ```text
  mvp/resolve/match-master-person — #13038
  ```

  **v1.0 (2026-05-26, unchanged since creation) — the 4-tier dedup resolver** (stage 2). Walks `master_link.link_url` (normalized) → `master_email.email_address` → `master_phone.phone_number` (per-number LLM formatting via `llm-phone-format` #2540, **name-guarded** — first OR last name must agree — with a phone-conflict error path) → direct `master_person.linkedin_url` fallback (catches people who have the column set but no `master_link` row). Returns `{master_person_id, match_via, status, message, formatted_phones}` — every supplied phone is formatted even after a match lands, so the caller reuses `formatted_phones` in the write loops without double-LLMing. No graph writes. The `match_via` values are the observability contract on every dedup `log_crash` row (see [Entry Points](/guides/enrichment/waterfall/person-entry-points)).
</Card>

<Card icon="cloud-arrow-up">
  ```text
  mvp/format/name-format — #2649
  ```

Called early in #13039 (so the cascade gets structured name parts for PDL), from `resolve-temp-name` #12890, and from `process-full-enrich` #13049's temp-name repair. _(The old phases page's "Groq Llama 3.3 with Gemini 2.5 Pro fallback" claim was two model generations stale — the live call is the v3.4 structured-output Gemini path.)_

<Card icon="wrench">
  ```text
  mvp/stop/check-kill-switch-v2 — #4617
  ```

  **Headerless (description comment only; unchanged since 2026-03-19) — the person-creation kill switch.** Returns `true` when the live `master_person` row COUNT is \>= the `kill_switch_person` row in the `environment_variables` TABLE (per-datasource threshold — not a Xano `$env` var; live threshold 10,000 per the #12997 inline note); any failure in the check is caught, logged to `log_crash`, and defaults the switch **OFF** — creation proceeds (fail-open). When ON, #13039 answers from local-only lookups (links / emails / phones / `linkedin_url`) and blocks new-person creation with zero external API spend; #12997 calls it twice (company-resolve containment \+ person-create gate); #12816 skips its person section entirely. Since #13039 **v1.13 (2026-06-30)** blocked people are audited in `log_crash` #542 **only** — the `kill_switch_blocked_people` #611 side-write is gone and the table is deprecated (no drainer ever existed; see [deviations](#deviations-deprecated--orphaned)). _(The old core-concepts name `check-kill-switch-person` was wrong — the live name is `check-kill-switch-v2`; the legacy predecessor `mvp/stop/check-kill-switch` #2774 still exists but is not the pipeline gate.)_
</Card>

<Card icon="wrench">
  ```text
  mvp/format/set-person-names — #12820
  ```

  **v1.0 (2026-04-13, unchanged):** tiny isolation helper — `db.edit` of the `master_person` name fields, but only when `person_id > 0` and at least first or last name is non-empty. Its inputs are named `fn` / `ln` / `mn` / `sfx` / `nick` instead of `first_name` / `last_name` to dodge the Xano naming-collision bug where data keys that shadow a parent function's input parameters silently resolve to the (empty) inputs. **Xano artifact — do not reimplement:** in a port this is a plain guarded UPDATE; the helper exists only for the collision workaround. Single writer for person name fields — #13039 and #12890 both route through it.
</Card>

## Orchestrator

<Card icon="layer-group">
  ```text
  mvp/enrich/enrich-master-person — #13040
  ```

  **v3.23 (2026-06-30) — the person orchestrator** (phases 0–12, with phase 9 run twice). Dispatched async-shared by `master-person-new` #13039 (new, non-queued depth-0 people), `process-full-enrich` #13049 (`force_run: true` re-run after a FullEnrich writeback), the company orchestrator's C-suite mode (#12992 Step 7 via #12997), and the queue drains (Task #126 / #12816). Inputs: `master_person_id`, `deep_research`, `cascade_depth`, `force_run` (v3.10), `current_company_only` (v3.13), `allow_investor_network_cascade` (v3.14). Pre-flight: a **stale-dispatch guard** returns early with a `log_crash` entry when the `master_person` row was deleted since the async fire; then the **60 s source-keyed debounce** — any `enrich_history_person` row with the same `master_person_id` \+ `source == "Master Person Enrich"` \+ `updated_at` inside the window logs `"DUPLICATE TRIGGER, debounced (60s window)"` and returns null (debounce rows log `qa_passed: true` since v3.19); `force_run: true` bypasses it. The setup row is upsert-keyed on `(master_person_id, source)` — source **text**, not the legacy numeric `data_source_id` (v3.8/v3.9 source sweep) — so the debounce key is `updated_at` (touched every re-run), not `created_at`.

  **Live chain (reordered vs the old page):** 0 → 1 → 2 → 3 → 4 → 4b (Exa) → 5 → **EARLY phase 9** (`build_investment_thesis: false`, immediately after Fundable linking) → 6 → **best-current-role sync** via `mvp/work/choose-best-current-role` #12621 → safety close → 7 → safety close → 8 → **phase 9 AGAIN** (`build_investment_thesis: true`) → 10 (no-op stub, still called) → 11 → 12 → final `enrich_history_person.processing = false` close → closeout aggregate → **v3.23 post-thesis best-effort retry**: re-runs `create-llm-person-bios` #2537 and `update-person-node` #2758 so a thesis built late in the run still reaches the bio and the graph node. Every phase runs in its own `try_catch` writing per-phase `qa_passed` / `CRASH: phase-N` rows to `log_crash` #542 (strict `phase` enum — see [Core Concepts → Per-Phase Crash Logging](/guides/enrichment/waterfall/core-concepts#per-phase-crash-logging)); failures are logged but never block later phases.

  **v3.14–v3.23 ladder:** **v3.14** threads the `allow_investor_network_cascade` opt-in into phase 9; **v3.15** moves the early phase-9 pass to right after Fundable linking; **v3.16** best-effort current-role sync after phase 6 via #12621; **v3.17** closeout ignores optional deep-research launch warnings (pairs with #4578 v1.1 — a launch miss never fails the run); **v3.18** phase-9 investment-thesis pass; **v3.19** debounce rows log `qa_passed: true`; **v3.20/v3.21** safety-close `enrich_history_person.processing` after phases 7 and 6 respectively (v3.20 post-phase-7, v3.21 post-phase-6 — a mid-run crash can no longer strand the FullEnrich webhook's wait loop); **v3.22** threads `current_company_only` into Phase 1 so Exa C-suite leaves skip the paid EL lookup; **v3.23** post-thesis bio/node retry. History: v3.5 added phase 4b; v3.7 removed RocketReach (see [deviations](#deviations-deprecated--orphaned)); the deleted twin #12554 is history too — #13040 is the **only** person orchestrator.
</Card>
