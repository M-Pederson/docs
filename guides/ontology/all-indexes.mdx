## The label-scoping rule (read before proposing indexes)

FalkorDB indexes are **per-label**: a query uses an index only when the indexed label appears in the MATCH/MERGE pattern. `MATCH (p:Person {uuid: $u})` does **not** use `Entity(uuid)` even though the node carries both labels ŌĆö this is why `C_Suite(uuid)` exists, and why the 2026-07-06 additions below add per-label coverage for `Person`, `Company`, and the legacy non-`Entity` labels. When writing new Cypher, prefer including `Entity` in the pattern (`MATCH (p:Entity:Person {uuid})`) so the base-label indexes apply; when a pattern can't, make sure the label it does use is indexed on the lookup key.

# Vector search

Everything below was formerly the _Vector Indexes_ page.

<Info>
  **­¤öÆ Embedding standard ŌĆö read this before writing any vector code.** Every Orbiter vector, on every surface, is **`gemini-embedding-001` ┬Ę 3072-dim ┬Ę cosine**. The stored vectors (AlloyDB) and the query vector (generated by AlloyDB _or_ OpenRouter) **must** match on model, dimension, and normalization ŌĆö mismatch doesn't error, it silently returns wrong results. This applies to the AlloyDB vector layer; the FalkorDB indexes documented on this page are a separate store.

  ---

  **1 ┬Ę AlloyDB \+ ScaNN ŌĆö one-time setup (put this in your migrations)**

  ```sql
  -- Extension: alloydb_scann cascades the pgvector `vector` type automatically
  CREATE EXTENSION IF NOT EXISTS alloydb_scann CASCADE;
  
  -- Column: full 3072-dim. It's 12 KB (> the 8 KB page) so it is ALWAYS TOASTed ŌĆö
  -- set STORAGE EXTERNAL to store it out-of-line WITHOUT compression
  -- (float data is high-entropy; the default EXTENDED just wastes CPU).
  ALTER TABLE people ADD COLUMN biography_embedding vector(3072);
  ALTER TABLE people ALTER COLUMN biography_embedding SET STORAGE EXTERNAL;
  
  -- ScaNN index ŌĆö cosine distance. Build it AFTER the table has data.
  CREATE INDEX people_bio_scann ON people
    USING scann (biography_embedding cosine)
    WITH (num_leaves = 1000, quantizer = 'SQ8');
  ```

  - **Distance keyword is `cosine`** ŌĆö _not_ `vector_cosine_ops` (that's pgvector HNSW/IVFFlat syntax, which we do **not** use).
  - **`num_leaves` Ōēł ŌłÜ(row count)** is the starting rule (1000 Ōēł 1M rows); `quantizer = 'SQ8'` is the default (\< 1ŌĆō2 % recall loss).
  - **Build after loading rows** ŌĆö ScaNN partitions the data, so `num_leaves` must be Ōēż row count. On large tables use `CREATE INDEX CONCURRENTLY` (non-blocking, slower) and raise `maintenance_work_mem` first.
  - Never store the raw 3072 vector without `STORAGE EXTERNAL`, and never index it with pgvector HNSW/IVFFlat (they cap at 2000 dims ŌĆö ScaNN is why we can keep the full 3072).

  **2 ┬Ę Generate a vector inside GCP (Go / pgx)** ŌĆö AlloyDB embeds server-side, no external call:

  ```go
  const embedSQL = `SELECT embedding('gemini-embedding-001', $1)::vector`
  
  var vec pgvector.Vector // []float32, len 3072
  if err := pool.QueryRow(ctx, embedSQL, text).Scan(&vec); err != nil {
      return fmt.Errorf("embed %q: %w", text, err)
  }
  // ŌĆ”then INSERT vec into the vector(3072) column
  ```

  **3 ┬Ę Query ŌĆö cosine KNN with ScaNN re-ranking on** (do it inside a txn so the GUCs are `SET LOCAL`):

  ```go
  // re-ranking matters at 3072-dim: gather candidates on quantized vectors,
  // then re-score the finalists on full precision.
  tx.Exec(ctx, `SET LOCAL scann.pre_reordering_num_neighbors = 200`) // default 50├ŚLIMIT
  tx.Exec(ctx, `SET LOCAL scann.num_leaves_to_search       = 100`)  // recall vs latency
  
  const q = `
    SELECT id, biography_embedding <=> $1 AS distance
    FROM people
    WHERE country = $2                       -- inline metadata filter (ScaNN-only!)
    ORDER BY biography_embedding <=> $1
    LIMIT 20`
  rows, err := tx.Query(ctx, q, queryVec, "US")
  ```

  - **`<=>`** is cosine distance (lower = closer). Match it to the index's `cosine` build.
  - **`scann.pre_reordering_num_neighbors`** is the re-ranking knob ŌĆö recommended for Ōēź 500-dim vectors; raise it for higher recall. **`scann.num_leaves_to_search`** trades recall for latency (ratio to `num_leaves` Ōēł recall).
  - Add `WHERE` filters freely ŌĆö ScaNN pushes them _into_ the index traversal. That inline-filtering ability is the whole reason we're on ScaNN and not HNSW.

  ---

  **Creating vectors via OpenRouter (must stay search-compatible with ┬¦2)**

  When a vector is produced **outside GCP** ŌĆö Xano, an app server, or embedding a user's search query ŌĆö call OpenRouter's OpenAI-compatible endpoint with the **same model and dimension** so it lands in the same space as the AlloyDB-stored vectors:

  ```bash
  curl https://openrouter.ai/api/v1/embeddings \
    -H "Authorization: Bearer $OPENROUTER_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{
      "model": "google/gemini-embedding-001",
      "input": "text to embed",
      "dimensions": 3072
    }'
  ```

  Response is OpenAI-shaped ŌĆö the vector is `data[0].embedding`:

  ```json
  { "data": [ { "embedding": [0.013, -0.041, "ŌĆ”3072 floats" ], "index": 0 } ],
    "model": "google/gemini-embedding-001", "usage": { "prompt_tokens": 5 } }
  ```

  **Compatibility rules ŌĆö break any one and search silently degrades:**

  | Rule | Why |
  | --- | --- |
  | **Same model** ŌĆö `google/gemini-embedding-001` (OpenRouter) ŌēĪ `gemini-embedding-001` (AlloyDB) | Same underlying model = same vector space |
  | **`dimensions: 3072`** (or omit ŌĆö 3072 is the default) | Must equal the `vector(3072)` column. If you ever store a truncated dim, stored **and** query vectors must use the identical `dimensions`. |
  | **Do NOT send `task_type`** | OpenRouter's OpenAI-compatible endpoint doesn't expose it and AlloyDB's `embedding()` doesn't set one ŌĆö setting it on one side only pushes queries into a different sub-space |
  | **Normalize consistently** | At full 3072 the model returns unit-normalized vectors on both paths, so cosine is apples-to-apples. If you truncate dims, L2-normalize on **both** sides. |
  | **Metric = cosine end-to-end** | Index built `cosine`, query uses `<=>` |

  ---

  **Never mix** `gemini-embedding-001` vectors with `-002` or `gemini-embedding-2` ŌĆö different space; switching models means re-embedding the whole corpus.
</Info>

## How vector search works in FalkorDB

FalkorDB uses **HNSW (Hierarchical Navigable Small World)** indexes under the hood. Indexes are created explicitly via `CREATE VECTOR INDEX` (see the copy-paste block above) and maintained automatically as you write `vecf32` properties to nodes.

Queries go through the `db.idx.vector.queryNodes()` procedure:

```cypher
CALL db.idx.vector.queryNodes(
  'Entity',              // label to search
  'name_embedding',      // vecf32 property
  3,                     // k ŌĆö number of nearest neighbors
  vecf32($embedding)     // query vector
)
YIELD node, score
```

- **`score`** is cosine distance ŌĆö lower is better (0 = identical, 1 = orthogonal).
- The procedure returns up to `k` nodes sorted by ascending score.
- Filtering (e.g. by node type) happens **after** the vector search in a `WHERE` clause.

## Keeping the index healthy

<Warning>
  **A node can have a valid `vecf32` and still be invisible to search.** Writing a `vecf32` property only adds the node to the HNSW index if a healthy index for that label/property **already exists** in the data source at write time. A node written into a source whose index is missing/unhealthy gets a perfectly good vector that is **never indexed** ŌĆö `IS NOT NULL` is true, `vec.cosineDistance` works, but `queryNodes` never returns it. This was the root cause of the 2026-06-17 incident where music nodes had embeddings but `"who is ŌĆ”"` returned nothing on `staging`.
</Warning>

**Incremental maintenance is reliable ŌĆö once the index exists.** With a healthy index, every write path indexes on write (verified across `MERGE ŌĆ” SET vector ŌĆ” SET :Entity`, `:Entity` in the MERGE label set, and stub-then-backfill). So this class of failure is a **provisioning/state gap, not a writer-code bug.** The fixes below all target index _state_.

<Steps>
  <Step title="Provision the index before seeding">
    Run **`tool/create-indexes` (#4669)** on every data source (`sandbox` / `staging` / `live`) **before** any node writes, and at the top of any seed/migration. It is idempotent. New data source ŌåÆ indexes first, then seed.
  </Step>
  <Step title="Verify (or rebuild) after every bulk operation">
    Snapshot/restore, data-source copy, and large backfills can land `vecf32` properties **without** populating HNSW. After any seed-to-new-source or copy, run a coverage check or a one-time rebuild as a gate ŌĆö don't assume the copy carried the index.
  </Step>
  <Step title="Detect drift with a per-node self-probe ŌĆö never full enumeration">
    `queryNodes` is **approximate** (HNSW, `efRuntime` Ōēł 100). Trying to list every member from a single probe vector silently omits the farthest nodes, producing **false "missing" counts**. The ground-truth membership test is to query a node with **its own** vector ŌĆö an indexed node returns itself at score Ōēł 0:

    ```cypher
    // is THIS node in the index? indexed ŌćÆ returns itself@~0
    CALL db.idx.vector.queryNodes('Entity', 'name_embedding', 1, vecf32($node_own_vector))
    YIELD node, score
    WHERE node.uuid = $node_uuid
    RETURN score
    ```
  </Step>
  <Step title="Repair">
    **Drop \+ recreate** the index for a full backfill (Ōēł1s for ~25k vectors; brief search-degradation window on live):

    ```cypher
    DROP VECTOR INDEX FOR (n:Entity) ON (n.name_embedding)
    CREATE VECTOR INDEX FOR (n:Entity) ON (n.name_embedding)
      OPTIONS {dimension: 1536, similarityFunction: 'cosine', M: 16, efConstruction: 200, efRuntime: 100}
    ```

    For a few stragglers, force a per-node re-index by `null`-then-restoring the **same** value (a same-value re-set is a no-op and won't trigger reindexing):

    ```cypher
    MATCH (n {uuid: $uuid}) SET n.name_embedding = null;
    MATCH (n {uuid: $uuid}) SET n.name_embedding = vecf32($same_vector)
    ```

    Backfill is async ŌĆö `queryNodes` returns empty for a few seconds after `CREATE`, then populates.
  </Step>
</Steps>

<Note>
  A rare residual: a handful of otherwise-valid nodes can resist both rebuild and re-set (a FalkorDB HNSW edge case ŌĆö observed on 2 old `VC_Firm` nodes, 2 of ~25k). If business-critical, recreate the node (re-enrich) so it inserts fresh; otherwise it's safe to leave.
</Note>

## Live usage ŌĆö expertise matching

The primary vector search in production matches incoming expertise signals against existing `SubDomainExpertise` nodes (locked spec, #12926 person / #12746 company). It searches the Gemini **`expert_embeddings`** vector:

```cypher
CALL db.idx.vector.queryNodes(
  'Entity', 'expert_embeddings', 3, vecf32({{$var.embedding}})
)
YIELD node, score
WHERE score < 0.45
  AND 'SubDomainExpertise' IN labels(node)
RETURN node.uuid AS uuid, node.name AS name, score
ORDER BY score ASC
```

**Key details:**

- Searches across the `Entity` label (all node types), then filters to `SubDomainExpertise` in the `WHERE` clause ŌĆö use `'Label' IN labels(node)`, never `labels(node) = 'ŌĆ”'`.
- Returns the 3 nearest neighbors with a cosine distance threshold of **\< 0.45**.
- The `score` feeds the `match_score` / weight formula on the resulting `HAS_EXPERTISE` / `SPECIALIZES_IN` edge (see [Edge Weights](/guides/ontology/edge-weights)).

## Common patterns

### Deduplication via vector similarity

Before creating a new node, query for existing nodes with similar embeddings. If the top result scores below a threshold, merge instead of creating a duplicate.

```cypher
CALL db.idx.vector.queryNodes('Entity', 'name_embedding', 1, vecf32($embedding))
YIELD node, score
WHERE score < 0.15
  AND 'Person' IN labels(node)
RETURN node.uuid AS existing_uuid, score
```

### Skills and specialties matching

Person `skills` and Company `specialties` are matched against the `SubDomainExpertise` taxonomy via the same vector pipeline ŌĆö each signal is embedded and searched against `expert_embeddings`; the resulting edges carry the original source text.

- **Person skills** ŌåÆ `HAS_EXPERTISE` edge
- **Company specialties** ŌåÆ `SPECIALIZES_IN` edge

Both edge types target the same `SubDomainExpertise` nodes, enabling cross-entity queries:

```cypher
// People whose expertise matches a company's specialties
MATCH (c:Company {uuid: $company_uuid})-[:SPECIALIZES_IN]->(d:SubDomainExpertise)<-[:HAS_EXPERTISE]-(p:Person)
RETURN p.name, d.name AS shared_domain
```

## Distance metric

FalkorDB vector indexes use **cosine distance**. Score interpretation (Orbiter conventions, not FalkorDB defaults):

| Score Range | Meaning |
| --- | --- |
| `< 0.15` | Near-duplicate ŌĆö likely the same entity |
| `0.15 ŌĆō 0.30` | Very similar ŌĆö strong semantic match |
| `0.30 ŌĆō 0.45` | Related ŌĆö used as the expertise matching threshold |
| `> 0.45` | Weak or no match ŌĆö typically filtered out |

## FalkorDB index gotchas

- **Label scoping (the big one).** An index binds to one label; the pattern must name that label. `MATCH (p:Person {uuid})` cannot use `Entity(uuid)`. Prefer `(p:Entity:Person {ŌĆ”})` patterns in new Cypher.
- **Not-equal filters (`<>`) are not index-accelerated.** Prefer positive predicates (`WHERE f.source = 'declared'`).
- **`IS NULL` / `IS NOT NULL` ARE accelerated** by btree indexes ŌĆö this is why indexing `valid_to` directly speeds up "currently valid" filters.
- **Composite edge indexes split** into independent single-property indexes on our FalkorDB version.
- **Vector indexes require populated embeddings** ŌĆö creating against an empty label is fine; queries return nothing until vectors land.
- **Btree indexes only contain nodes that carry the property** ŌĆö per-label indexes on sparse properties are cheap; the real cost of over-indexing is write amplification and planner noise, not storage.
- **Edge direction in index syntax:** always `()-[r:EDGE]-()` undirected.
