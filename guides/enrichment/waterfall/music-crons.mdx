Four scheduled tasks serve the MusicBrainz waterfall. The unified drain (Task 180) is the single queue consumer for all six MB queues — **pausing it is the pipeline's manual kill** (there is no dedicated kill switch, [by design](/guides/enrichment/waterfall/music-waterfall#loop-prevention-six-layers)). Drain internals: [Music Waterfall Functions → Unified queue drain](/guides/enrichment/waterfall/music-functions#unified-queue-drain) · person-resolution: [Music Waterfall Functions → Person-resolution](/guides/enrichment/waterfall/music-functions#person-resolution).

<Card icon="clock-rotate-left">
  ```text
  process-queue-musicbrainz — Task 180   (INACTIVE)
  ```

  The unified cron. Runs every **60 s** and makes up to **10 `process-next` (13102) calls per tick** (the design's ~2 s cadence isn't possible at Xano's 60 s cron granularity — see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals)). **Pausing this task is the pipeline's manual kill.** Because `tool/drain-queue` also calls 13102, manual drains follow the same tier-0 gate as the cron.
</Card>

<Card icon="clock-rotate-left">
  ```text
  backfill-music-edge-descriptions — Task 181   (ACTIVE — legacy name)
  ```

  Async safety-net task for music edge descriptions. Runs every **300 s** on the **`sandbox` datasource**, processing up to **20** undescribed music edges per tick through `mb/tool/backfill-music-edge-descriptions` (#13097), which uses the same `deepseek/deepseek-v4-flash` create/write path (#13095 → #13096) as the live enrichers.

  This task is **not the primary forward path** after the 2026-07-01 update and no longer drains live leftovers. The primary path is now in-band: each music edge-writing orchestrator calls #13158 immediately after its edge phase. Task 181 catches sandbox edges still missing `description_generated_at`: historical rows, inline LLM failures, bounded-limit overflow, and manual graph writes. It is self-maintaining: once no pending music edges exist, each tick is a cheap no-op.
</Card>

<Card icon="clock-rotate-left">
  ```text
  process-queue-person-resolution — Task 182   (INACTIVE)
  ```

  The [person-resolution](/guides/enrichment/waterfall/music-functions#person-resolution) drain — a **standalone cron** (decoupled from the MB queue drain so Exa/serper pacing is independent of MB's 1 req/s budget). Drains **5 rows per tick** via `mb/queue/process-next-person-resolution` (13123). Shipped `active = false`; pausing it is the kill switch.

  **Pending activation hook (2026-07-02):** when this task is enabled, the deterministic MB-credit expertise emitter `mvp/music/mb-credits-to-expertise` (13160, `write: true`) should be wired in after each successful `SAME_AS` bridge, so bridged music people get `HAS_EXPERTISE` edges \+ `person_has_expertise` rows straight from their MusicBrainz credits (no LLM; sandbox-verified idempotent).
</Card>

<Card icon="clock-rotate-left">
  ```text
  vector-index-health-check — Task 183   (ACTIVE — live, daily)
  ```

  Daily vector-index health monitor, born from the 2026-06-17 staging incident where music vectors were written before the `Entity.name_embedding` index was healthy and became silently unsearchable. Samples **300** `:Entity` nodes that carry `name_embedding` and self-probes each for index membership via `vectors/audit-vector-index` (13128) — the reliable test, since `queryNodes` enumeration is approximate. **Read-only** (`repair = false`): it never mutates graph data, and only writes a `log_crash` alert row when more than 5 of the sample are unindexed (tolerating known HNSW edge-case stragglers). Runs against the workspace default data source (= live). For on-demand healing, call the vector-index-health endpoint with `repair: true`, or rebuild the index for systemic drift.
</Card>
