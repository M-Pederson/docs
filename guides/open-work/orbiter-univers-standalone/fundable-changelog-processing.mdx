**Status: open work / plan.** Nothing on this page is built. Fundable is a living database in BigQuery; the Xano `fundable_*` tables are currently a static import. The lane being built: a sync cron pulls new/changed rows from BigQuery into the mirror tables and writes a **changelog**; after the cron completes, a **worker** loops the changelog and dispatches graph/table updates. The dispatch targets are almost entirely **existing** workspace-3 functions — this page maps each change case to the function that already handles it, and calls out the small set of genuinely new pieces.

<Note>
  Sibling pages: [Fundable table updates](/guides/open-work/fundable-table-updates) owns the sync/changelog/queue infrastructure design and its AlloyDB target runtime; [Fundable investor enrich](/guides/open-work/fundable-investor-enrich) is the on-demand sparse-investor identity resolver. This page owns the piece between them — the **changelog → action dispatch map**. The map is runtime-agnostic: built in Xano first, every action below is a direct `function.run` target; moved to the AlloyDB backend later, the same map is the behavioral contract for the port (the way the [Company Go Rebuild Spec](/guides/enrichment/waterfall/company-go-rebuild-spec) treats Xano functions as reference behavior).
</Note>

```text
BigQuery (living)
  --cron sync-->  fundable_* mirror tables
  --changelog-->  fundable_change_log rows
  --worker------> existing enrichment functions
  --------------> graph nodes/edges + master tables + queue rows
```

## The three graph-linked tables

Only three fundable tables link directly to canonical rows and node UUIDs; everything else rolls up through them. The link columns already exist in the schemas — no migration needed to start.

| Mirror table | Link columns (already in schema) | Graph identity |
| --- | --- | --- |
| `fundable_organizations` #666 | `master_company_id` → `master_company` #142, plus `master_company_node_uuid` | `Company` node (uuid mirrored on `master_company.node_uuid`) |
| `fundable_people` #667 | `master_person_id` → `master_person` #139, plus `master_person_node_uuid` | `Person` node |
| `fundable_deals` #671 | `funding_round_node_uuid` plus `processed_at` | `Funding_Round` node — MERGE-keyed `uuid = fundable_deals.id`, so the deal id **is** the node uuid |

Child tables carry no node UUIDs of their own; a change to any of them means "re-replay the parent deal":

| Child table | Rolls up to |
| --- | --- |
| `fundable_institutional_investments` #678 | parent `deal_id` |
| `fundable_institutional_investments_person` #681 | grandparent deal via `institutional_investment_id` |
| `fundable_angel_investments` #679 | parent `deal_id` |

Changelog-only in v1 (no graph action): `fundable_articles` #680, `fundable_valuations` #672, `fundable_organization_associations` #673, the industry tables #670/#674, and the `fundable_linkedin_*` family #668/#669/#675/#676/#677.

## The workhorse: the existing deal cascade

The hard part of changelog processing already exists, is already idempotent, and was end-to-end validated in the 2026-07-02 Deel smoke tests (run 5: 52 PASS / 0 FAIL). The worker mostly just needs to aim it.

<Card icon="play">
  ```text
  mvp/investor/cascade-deal-participants — 12856
  ```

  **The per-deal replay unit.** Inputs: `fundable_deal_id` (required), `cascade_depth` (pass an explicit `0`), `origin_master_company_id` / `origin_master_person_id`. One call: ensures the portfolio company exists as a `master_company` (sync get-add, `hold_downstream: true`, local-only, not queued) → IPO signal (`company_financial.is_public` escalate when `ipo_status == "public"`) → `Funding_Round` \+ `RAISED` via #12701 → investor/partner/angel edges \+ co-investor queueing via #12702 (depth-0 gated) → stamps `fundable_deals.processed_at` **only when the node actually materialized** (v1.8).
</Card>

<Card icon="wrench">
  ```text
  mvp/funding/add-fundable-deal-node — 12701
  ```

  `Funding_Round` node \+ `RAISED` edge (weight 10, `created_at` floored to the announced date's UTC midnight). MERGE-keyed on the deal id; sizes/counts written only when greater than 0, never zeroed on refresh. v1.9 hardening: JSON-encoded string props (apostrophe-safe), guarded embedding splice, and a missing `node_uuid` logs a crash and leaves `funding_round_node_uuid` falsy so #12856 retries.
</Card>

<Card icon="wrench">
  ```text
  mvp/fundable/resolve-investors-edges — 12702
  ```

  The investor half, depth-0 only. Resolves institutional investor orgs, deal partners, and co-angels (unknown participants are created as **local-only leaves**, then queued), and MERGEs the edge set: `LEAD_INVESTED_IN` w15 · `INVESTED_IN` w20 institutional / w25 angel · `FOLLOW_ON_INVESTED_IN` w20 · `INVESTMENT_PARTNER_IN` w15 · `INVESTMENT_PARTNER_AT` w20, plus `VC_Firm` / `Angel` label adds. Since the 2026-07-03 investor-queue redesign, **all investors enter the queues at `priority_tier: 1`** via #12854/#12855.
</Card>

<Card icon="layer-group">
  ```text
  mvp/funding/add-all-fundable-deals — 12703
  ```

  Org-level fan-out: walks a fundable org's rounds **raised** (chronological, `date asc`) and rounds **invested in**, dispatching each deal into #12856 inside a per-deal `try_catch`. Capped 25 target \+ 25 investor deals per invocation. Use it when a whole org needs (re)processing; use #12856 directly when the changelog names specific deals.
</Card>

Properties the worker leans on:

- **Self-gating staleness check (#12856 v1.2).** A replay is skipped unless the deal, its institutional investments, or their partner links carry an `updated_at` newer than `processed_at`. The worker can afford to over-dispatch — replaying an unchanged deal is a cheap no-op.
- **Idempotent MERGE writes.** Re-running a changed deal updates the `Funding_Round` node and edges in place; no duplicates.
- **Retry-safe failure marker (v1.8).** A node-write failure leaves `processed_at` null and logs a crash, so the changelog row can simply be retried.
- **The angel blind spot.** The staleness gate does **not** read `fundable_angel_investments` — a new angel on an already-processed deal will not re-trigger on its own. The worker must null `fundable_deals.processed_at` first (the documented escape hatch), or the gate gets the small extension proposed below.
- **Bounded participant creation.** Portfolio companies are created `hold_downstream` local-only; investors/partners/angels are local leaves plus tier-1 queue rows. A deal replay never launches recursive enrichment on its own.

## Eligibility gate

Same Stage-1 principle as [Fundable table updates](/guides/open-work/fundable-table-updates#graph-eligibility-gate): a changed row earns graph work only when it is already connected to the canonical layer.

| Change on | Dispatch only when | Otherwise |
| --- | --- | --- |
| a deal (or its children) | the portfolio org (`fundable_deals.organization_id` → `fundable_organizations`) has `master_company_id` \+ `master_company_node_uuid` | mark `ignored` with a skip reason |
| an organization | `master_company_id` \+ `master_company_node_uuid` present | `ignored` |
| a person | `master_person_id` \+ `master_person_node_uuid` present | `ignored` |

This gate matters because #12856 has get-add semantics — pointed at a deal whose company we've never adopted, it will **create** that company. Against a full BigQuery mirror, ungated dispatch turns the sync into unbounded company minting. Unlinked rows are not lost: they link naturally the next time the entity crosses an entry gate (`master-company-new` #12558 §8 Fundable lookup, `master-person-new` #13039 §5b, person phase-5 #12826, company phase-6 #12810 Block-1 domain repair) — and each of those linkings is itself a changelog-worthy event that can then release the held deal replays.

`mvp/query-get-all-invetsors` #12730 returns a deal's investor org \+ angel arrays **with** their master ids and node UUIDs — a ready-made helper if the worker wants per-participant eligibility introspection.

| # | Change (example) | Changelog signal | Action | Notes |
| --- | --- | --- | --- | --- |
| 1 | **New funding round on a linked company** (Deel raises a new round) | `fundable_deals` insert; portfolio org linked | `#12856` with `fundable_deal_id`, `cascade_depth: 0`, `origin_master_company_id` | One call materializes the round, edges, labels, and tier-1 co-investor queue rows. |
| 2 | **Deal fields corrected** (size, date, financing type, descriptions) | `fundable_deals` update | Same #12856 call | Staleness gate passes (`updated_at` newer than `processed_at`); MERGE refreshes node properties. |
| 3 | **New investor joins an existing round** | `fundable_institutional_investments` insert/update | #12856 on the parent `deal_id` | Explicitly covered by the staleness gate. |
| 4 | **Partner attributed to an investment** | `fundable_institutional_investments_person` insert/update | #12856 on the grandparent deal | Covered by the staleness gate. |
| 5 | **Existing person newly linked to a deal as an angel** | `fundable_angel_investments` insert | **Null `fundable_deals.processed_at`, then** #12856 | Angel rows are the staleness blind spot — forced replay required until the gate is extended (see new-build #5). An unlinked angel is created as a local leaf \+ tier-1 queue row by #12702; a linked one MERGEs against the existing Person node. |
| 6 | **New company appears in Fundable** | `fundable_organizations` insert | **Log-only by default.** Opt-in link-only reconcile: #12558 with `allow_create: false` \+ `link-fundable-org` #12794 (catches orgs we already track). All creation paths respect company kill switch #12789. |
| 7 | **Linked company's org row changes** (descriptions, counts, `total_raised`, `ipo_status`, employees) | `fundable_organizations` update | Conservative today: `process-company-phase-6` #12810 with `master_company_id` (link repair \+ fill-empty/escalate-only financials, incl. the `is_public` escalate) \+ node re-sync `update-company-node` #4659. **Gap:** a targeted changed-field refresh needs a new function | #12810 never overwrites non-empty `funding_total` and never writes it for VC firms (deliberate #12989 suppression). | 8 | **New person appears in Fundable** | `fundable_people` insert | **Log-only by default.** Opt-in seeding: `batch-seed-fundable-person` #12853 with the `fundable_person` row (the Task #155 pattern: creates `master_person`, writes the linkback, queues via #12854) | Person kill switch #4617 applies. Mass-seeding every Fundable person is a deliberate decision, not a default. |

New targeted function, then `update-person-node` #2758 to re-sync the node | Full re-run `enrich-master-person` #13040 works but is heavy. |

## Worker mechanics

1. **Collapse child changes to one replay per deal.** Dedupe key `deal_replay:<fundable_deal_id>` — a new deal arriving with 3 investments and 2 partner rows is six changelog rows but **one** #12856 dispatch.
2. **Chronological order within an org.** When several deals for one company are pending, dispatch oldest-first (`date asc`) — the #12703 v2.3 lesson: `FOLLOW_ON_INVESTED_IN` detection needs earlier rounds in the graph before later ones.
3. **Org link events before deal replays.** Soft ordering — the cascade self-heals — but processing a `link-fundable-org` event first converts that org's held deal rows from `ignored` to eligible within the same run.
4. **Per-action `try_catch`** (the #12703 pattern): one bad deal never aborts the batch. Failures stay `failed` on the changelog for retry; #12856 v1.8 guarantees a failed node write is retryable.
5. **Always pass an explicit `cascade_depth: 0`.** The depth inputs are nullable and read via `first_notnull`; an **omitted** depth fails closed to 1, and the investor-edge half (#12702) silently never runs.
6. **Cap batch size per run** and record overflow, mirroring the 25\+25 fan-out governor thinking in #12703.
7. **Expect kill-switch outcomes.** Company #12789 / person #4617 — when ON, creation paths return local-lookup-only or blocked results. Record the outcome; don't treat it as failure.
8. **Crash logging:** `log_crash` #542 has a **strict `phase` enum** — the worker must reuse an existing phase value or extend the enum first, or its own crash logger will crash.
9. **Known side effect to inherit knowingly:** #12702 unconditionally overwrites a deal partner's `master_person.master_company_id` / `current_title` with the deal's firm context on every replay.

<Warning>
  **Queue effects.** Every deal replay writes tier-1 rows into `queue_enrich_person` / `queue_enrich_company` via #12854/#12855 — and the scheduled drains (Task #126 `process-person-queue`, Task #127 `process-company-queue`, both filtering `priority_tier` ≤ 1) still ship `active = false`. Until they are enabled (or the operator drainer `process-enrichment-queue` #12816 is run), changelog processing will **accumulate** queue rows without enriching the investors it discovers. Decide drain activation as part of go-live.
</Warning>

## Worked example — Deel raises a new round

<Steps>
  <Step title="Sync cron lands the change">
    One new `fundable_deals` row for Deel's org, three `fundable_institutional_investments`, one partner link, one angel row → six changelog rows in the same sync run.
  </Step>
  <Step title="Worker dedupes and gates">
    All six collapse to one `deal_replay` for the new deal id. Deel's org row carries `master_company_id` \+ `master_company_node_uuid` → eligible.
  </Step>
  <Step title="One dispatch">
    `#12856` with the deal id, `cascade_depth: 0`, `origin_master_company_id` = Deel's id. The staleness gate passes trivially (`processed_at` is null on a new deal — the angel blind spot only matters for angels added to already-processed deals).
  </Step>
  <Step title="The cascade does the rest">
    Portfolio ensure no-ops (Deel already linked) → #12701 MERGEs the `Funding_Round` (uuid = deal id) \+ `RAISED` edge → #12702 writes the lead/follow-on/angel/partner edge set, adds `VC_Firm` / `Angel` labels, and queues the co-investors at tier 1 → `processed_at` stamped, `funding_round_node_uuid` written back.
  </Step>
  <Step title="Close out">
    Worker marks all six changelog rows `complete` and records the dispatched function \+ result. The tier-1 queue rows wait for Task #126/#127 (or #12816).
  </Step>
</Steps>

## Sync-side reuse (BigQuery plumbing that already exists)

The cron itself is new build, but not from scratch:

<Card icon="cloud-arrow-up">
  ```text
  mvp/funding/_bq-execute — 12690
  ```

  Generic self-authenticating BigQuery SQL runner (fetches its own service-account token, returns typed rows). Currently caller-less — the natural fetch primitive for the new watermark queries.
</Card>

<Card icon="layer-group">
  ```text
  mvp/funding/get-add-fundable-records — 12689
  ```

  The per-org subgraph ingester: given a domain / LinkedIn / Crunchbase input, pulls ONE organization plus its full funding subgraph from `orbiter-io.fundable_data_feed_production` and mirrors it into the `fundable_*` tables via #12691/#12692. Keep it as the **targeted repair / on-demand pull** tool. Known caveats before leaning on it harder: no `try_catch` (a mid-run failure leaves a silently partial mirror), the PHASE-13 dangling institutional-investment-id bug, and #12692's blind-insert duplicate risk for id-less people.
</Card>

<Card icon="wrench">
  ```text
  mvp/funding/_upsert-org — 12691   ·   mvp/funding/_upsert-person — 12692
  ```

  Natural-key upsert helpers (org: crunchbase permalink; person: `crunchbase_person_id`, else `linkedin_person_id`). Both preserve the `master_*` backlinks across re-upserts — exactly the property the incremental sync needs. #12692's fallback blind insert (person with neither id) and its unique-index collision failure mode should be fixed as part of the sync build.
</Card>

Also available: `mvp/enrich/big-query-person-profile` #12555 (BQ person lookup by LinkedIn/Crunchbase URL) and `mvp/funding/fundable-lookup` #12698 (mirror-only org lookup used by the entry gates — never BigQuery).

What the plumbing does **not** do today: table-level incremental windows. Everything is per-org. The watermark loop ("all rows changed since X, across all mirrored tables") plus changelog writes is the new part.

## New pieces to build

Tables (Xano twins of the sibling page's control-table design — keep the field names aligned so the AlloyDB port is mechanical):

1. `fundable_sync_runs` — one row per cron run: status, window, watermark before/after, per-table counts.
2. `fundable_change_log` — one row per changed source record: `sync_run_id`, `source_table`, `source_id`, `change_type` (insert / update / remove), `changed_fields`, `diff_json`, `processing_status` (logged / queued / complete / ignored / failed), `skip_reason`, `dispatched_function`, `result_json`.

Functions:

1. **`fundable/sync-from-bigquery`** (\+ scheduled task, ship `active = false`) — watermark window per table via #12690, diff against the mirror, upsert (reuse #12691/#12692; add thin upsert helpers for deals / investments / angels / articles, whose upserts today exist only inline inside #12689), write changelog rows.
2. **`fundable/process-change-log`** (\+ scheduled task, ship `active = false`) — the worker implementing the dispatch map and mechanics above.
3. **`fundable/apply-org-field-updates`** — closes case 7: map changed `fundable_organizations` fields onto `master_company` / `company_financial` (respecting the #12989 VC `funding_total` suppression and #12810's escalate-only philosophy), then re-sync the node via #4659.
4. **`fundable/apply-person-field-updates`** — closes case 9: Fundable profile URLs as `master_link` rows (`source: "Fundable"`), name/title/avatar/contact deltas, then re-sync via #2758.
5. **#12856 staleness-gate extension** (small, optional) — include `MAX(fundable_angel_investments.updated_at)` alongside the existing three checks, retiring the forced `processed_at` null in case 5.
6. **Deferred:** edge retirement / orphan-edge audit for `remove` tombstones.

Nothing else is new. The graph writers (#12701/#2601/#2602/#2758/#4659), the cascade (#12856/#12702/#12703), the queue upserts (#12854/#12855), the entry gates (#12558/#13039), the linkers (#12793/#12794), the seeders (#12853/#13066), the kill switches (#12789/#4617), and the merge/repair tooling (#12945/#12956) all exist and are current.

## Open decisions

1. **Sync scope.** Mirror the full BigQuery dataset, or only the already-mirrored slice plus opt-in expansions? The eligibility gate protects the graph either way, but mirror size and changelog volume differ enormously.
2. **Runtime.** The sibling page mandates the final build lands in the AlloyDB backend. If an interim Xano worker ships first (this page makes that directly possible), say so explicitly and treat it as reference behavior for the port.
3. **Queue drain activation.** Enable Task #126/#127 when changelog processing goes live, or accept queue growth and drain via #12816 manually?
4. **Child-table v1 scope.** Confirm articles / valuations / associations / industries stay changelog-only in v1 (valuations arguably feed case 7's financial refresh later).
