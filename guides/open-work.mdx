This page is the **org-wide canonical queue** for everything that isn't a committed spec and isn't live production code. If you find yourself adding a `TODO`, an "open question", a "pending decision," a "not yet built" note, a drift callout, or a "needs verification after X" to any Orbiter docs page, **add it here instead** and leave a one-line pointer on the source page.

<Note>
  **Onboarding developers:** this is where you look first. Every area of the codebase has a section below. "Is there anything I should know about before working on X?" is answered here. If a section is empty, that area currently has no known TODOs — check the scope/build pages in that area for what's in active development.
</Note>

## What belongs on this page

| Category | Example |
|---|---|
| **Spec-only functions/crons** | `dedupe_knows_edges` — design is frozen, Xano implementation not yet built |
| **Open decisions** | "What's the presence floor value?" — needs a call before we build the scoring cron |
| **Drift and cleanup** | `Entity.name_embedding` running `efRuntime: 10` vs declared 100 |
| **Verification tasks** | Run `CALL db.indexes()` after the fresh-graph bring-up to confirm `efRuntime: 100` |
| **Pending retirements** | `mvp/merge/detect-suspects` (#12709) — keep rewired or replace entirely |

## What does NOT belong here

- **Per-phase build-status tables inside scope sub-pages** (`✅ Built` / `⏳ Pending` per function). Those are reader context at the point where a spec is being read — they answer "is this built yet?" for someone navigating the spec. Leave them inline.
- **"Not in Scope (Yet)" product deferrals** — `CoFoundingGroup`, `PolyamorousGroup`, graph visualization, etc. Those are architectural decisions that frame the current scope, not work waiting to be done. They stay on the relevant scope page.
- **Committed architectural decisions** that look like TODOs but aren't. If the workshop decided "we'll use canonicalized MERGE + a dedupe cron instead of a mutex table" — that's a decision, not a TODO. It goes in the spec, not here. Only the *unbuilt cron itself* lives here.

## Discipline for adding new items

When a new TODO appears:

1. **Pick the right domain section** (Known, Enrichment, Log-parse, Ontology, Third-party APIs, API Reference, Skills & Tech Stack)
2. **Pick the right sub-category within that domain** (spec-only / open decision / drift / verification / retirement)
3. **Add a row to that section's table** with `Item`, `Context/Trigger`, `Links`, `Status`
4. **If you're editing a spec page that triggered the discovery**, leave a one-line pointer on the source page — never leave the full TODO body on both pages

When an item closes, mark it `✅ Done` with the date and a one-line resolution note. Keep resolved items in place for one sprint cycle (useful for retrospectives), then delete. Permanent "decisions" graduate from here into the appropriate spec page — they should never just disappear.

**Split rule:** when a single domain crosses ~30 open items, split that domain into its own sub-page (e.g. `guides/open-work/known.mdx`) and keep this page as the index. Right now everything fits on one page.
