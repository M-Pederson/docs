Every node in the Orbiter graph carries the base label **`Entity`**. Most node types store one or more `vecf32` embedding properties, enabling semantic similarity search via FalkorDB's built-in vector index API.

All indexes (scalar and vector) are created by **`tool/create-indexes`** — **#4669**. As of 2026-04-11 this function is idempotent and creates **37 property-level indexes**: 35 scalar (btree) indexes on node properties and relationship properties, plus 2 vector (HNSW) indexes on embedding properties. It finishes by calling `CALL db.indexes()` and returning the full index list as confirmation. Run it after provisioning a new FalkorDB instance or after a graph reset.

See [All Indexes](/guides/ontology/all-indexes) for the canonical list of every scalar + vector index, the 22 Known-model additions live since 2026-04-11, and the KNOWS composite-split note.

## How vector search works in FalkorDB

FalkorDB uses **HNSW (Hierarchical Navigable Small World)** indexes under the hood. Indexes are created explicitly via `CREATE VECTOR INDEX` (see `tool/create-indexes` #4669) and maintained automatically as you write `vecf32` properties to nodes.

Queries go through the `db.idx.vector.queryNodes()` procedure:

```cypher
CALL db.idx.vector.queryNodes(
  'Entity',              // label to search
  'name_embedding',      // vecf32 property
  3,                     // k — number of nearest neighbors
  vecf32($embedding)     // query vector
)
YIELD node, score
```

- **`score`** is cosine distance — lower is better (0 = identical, 1 = orthogonal).
- The procedure returns up to `k` nodes sorted by ascending score.
- Filtering (e.g. by node type) happens **after** the vector search in a `WHERE` clause.

## Keeping the index healthy

<Warning>
  **A node can have a valid `vecf32` and still be invisible to search.** Writing a `vecf32` property only adds the node to the HNSW index if a healthy index for that label/property **already exists** in the data source at write time. A node written into a source whose index is missing/unhealthy gets a perfectly good vector that is **never indexed** — `IS NOT NULL` is true, `vec.cosineDistance` works, but `queryNodes` never returns it. This was the root cause of the 2026-06-17 incident where music nodes had embeddings but `"who is …"` returned nothing on `staging`.
</Warning>

**Incremental maintenance is reliable — once the index exists.** With a healthy index, every write path indexes on write (verified across `MERGE … SET vector … SET :Entity`, `:Entity` in the MERGE label set, and stub-then-backfill). So this class of failure is a **provisioning/state gap, not a writer-code bug.** The fixes below all target index *state*.

<Steps>
  <Step title="Provision the index before seeding">
    Run **`tool/create-indexes` (#4669)** on every data source (`sandbox` / `staging` / `live`) **before** any node writes, and at the top of any seed/migration. It is idempotent. New data source → indexes first, then seed.
  </Step>
  <Step title="Verify (or rebuild) after every bulk operation">
    Snapshot/restore, data-source copy, and large backfills can land `vecf32` properties **without** populating HNSW. After any seed-to-new-source or copy, run a coverage check or a one-time rebuild as a gate — don't assume the copy carried the index.
  </Step>
  <Step title="Detect drift with a per-node self-probe — never full enumeration">
    `queryNodes` is **approximate** (HNSW, `efRuntime` ≈ 100). Trying to list every member from a single probe vector silently omits the farthest nodes, producing **false "missing" counts**. The ground-truth membership test is to query a node with **its own** vector — an indexed node returns itself at score ≈ 0:

    ```cypher
    // is THIS node in the index? indexed ⇒ returns itself@~0
    CALL db.idx.vector.queryNodes('Entity', 'name_embedding', 1, vecf32($node_own_vector))
    YIELD node, score
    WHERE node.uuid = $node_uuid
    RETURN score
    ```
  </Step>
  <Step title="Repair">
    **Drop + recreate** the index for a full backfill (≈1s for ~25k vectors; brief search-degradation window on live):

    ```cypher
    DROP VECTOR INDEX FOR (n:Entity) ON (n.name_embedding)
    CREATE VECTOR INDEX FOR (n:Entity) ON (n.name_embedding)
      OPTIONS {dimension: 1536, similarityFunction: 'cosine', M: 16, efConstruction: 200, efRuntime: 100}
    ```

    For a few stragglers, force a per-node re-index by `null`-then-restoring the **same** value (a same-value re-set is a no-op and won't trigger reindexing):

    ```cypher
    MATCH (n {uuid: $uuid}) SET n.name_embedding = null;
    MATCH (n {uuid: $uuid}) SET n.name_embedding = vecf32($same_vector)
    ```

    Backfill is async — `queryNodes` returns empty for a few seconds after `CREATE`, then populates.
  </Step>
</Steps>

<Note>
  A rare residual: a handful of otherwise-valid nodes can resist both rebuild and re-set (a FalkorDB HNSW edge case — observed on 2 old `VC_Firm` nodes, 2 of ~25k). If business-critical, recreate the node (re-enrich) so it inserts fresh; otherwise it's safe to leave.
</Note>

## Live usage — expertise matching

The primary vector search in production matches incoming expertise signals against existing `SubDomainExpertise` nodes. From the [expertise pipeline](/guides/cyphers/expertise):

```cypher
CALL db.idx.vector.queryNodes(
  'Entity', 'expertise_embedding', 3, vecf32({{$var.embedding}})
)
YIELD node, score
WHERE score < 0.45
  AND labels(node) = 'SubDomainExpertise'
RETURN node.uuid AS uuid, node.name AS name, score
ORDER BY score ASC
```

**Key details:**

- Searches across the `Entity` label (all node types), then filters to `SubDomainExpertise` in the `WHERE` clause.
- Returns the 3 nearest neighbors with a cosine distance threshold of **< 0.45**.
- The `score` is used downstream as the `match_score` property on the resulting `HAS_EXPERTISE` edge (see [edges → HAS_EXPERTISE](/guides/ontology/edges)).

## Common patterns

### Deduplication via vector similarity

Before creating a new node, query for existing nodes with similar embeddings. If the top result scores below a threshold, merge instead of creating a duplicate.

```cypher
CALL db.idx.vector.queryNodes('Entity', 'name_embedding', 1, vecf32($embedding))
YIELD node, score
WHERE score < 0.15
  AND labels(node) = 'Person'
RETURN node.uuid AS existing_uuid, score
```

### Multi-property search

For node types with two embedding properties (e.g. `DomainExpertise`), choose the property based on intent:

- **`name_embedding`** — match by name / label (e.g. "Machine Learning").
- **`expertise_embedding`** — match by semantic meaning of the expertise area (e.g. an embedding derived from a description of skills, not just the label).

### Skills and specialties matching

Person `skills` and Company `specialties` arrays are matched against the existing `SubDomainExpertise` taxonomy via the same vector search pipeline. Each array item is embedded and searched against `expertise_embedding` — the resulting edges carry the original source text.

- **Person skills** → `HAS_EXPERTISE` edge with `skill` property
- **Company specialties** → `SPECIALIZES_IN` edge with `specialty` property

Both edge types target the same `SubDomainExpertise` nodes, enabling cross-entity queries:

```cypher
// People whose expertise matches a company's specialties
MATCH (c:Company {uuid: $company_uuid})-[:SPECIALIZES_IN]->(d:SubDomainExpertise)<-[:HAS_EXPERTISE]-(p:Person)
RETURN p.name, d.name AS shared_domain
```

```cypher
// Companies that specialize in the same domains as a person's skills
MATCH (p:Person {uuid: $person_uuid})-[:HAS_EXPERTISE]->(d:SubDomainExpertise)<-[:SPECIALIZES_IN]-(c:Company)
RETURN c.name, d.name AS shared_domain
```

See [edges → SPECIALIZES_IN](/guides/ontology/edges) and [edges → HAS_EXPERTISE](/guides/ontology/edges) for full property definitions.

## Distance metric

FalkorDB vector indexes use **cosine distance** by default. Score interpretation:

| Score Range | Meaning |
|---|---|
| `< 0.15` | Near-duplicate — likely the same entity |
| `0.15 – 0.30` | Very similar — strong semantic match |
| `0.30 – 0.45` | Related — used as the expertise matching threshold |
| `> 0.45` | Weak or no match — typically filtered out |

These thresholds are conventions used across the Orbiter pipelines, not FalkorDB defaults.
