## Temp Xano ↔ GCP flow (pre-migration)

Until the full migration lands, persona creation and scoring run as a **Xano-orchestrated, GCP-backed** flow. **GCP owns the persona store** — the eight `(narrative, vector)` records and the full persona record live there (the AlloyDB `persona_vector` substrate from the [scoring pipeline](/guides/open-work/suggestion-recipes/loop-generic)); Xano assembles context and exposes the entry points.

```
Xano  ──  assemble person context  ──▶  GCP
GCP   ──  generate 8 persona narratives  →  embed  →  write full record
          (narratives + vectors, "investment-thesis"-style)  ──▶  GCP persona store
Xano  ──  persona/get-scores(person_id, user_id, params)  ──▶  GCP  ──▶  scores JSON
```

## `persona/get-scores` — proposed input contract

<Note>
**Proposed — Mark to confirm. This is the v1 (my-contacts) contract.** Derived from the `loop_generic` scoring constants (`wDrivers`, `wValence`, `wSoft`, `wComm`, `softmaxTemp`) and the tuner controls. Identity inputs are required; every scoring knob defaults to the current pipeline constant, so a call with only `person_id` + `user_id` reproduces today's behavior. | Parameter | Type | Default | Range | What it controls |
|-----------|------|---------|-------|------------------|

| `w_soft` | float | `0.03` | `0`–`1` | Modifier: soft-skills similarity (inverted in `complement`). |

| `candidate_ids` | uuid[] | _null_ → derive from the user's contacts | — | Optional explicit candidate set; the harness pins a fixed set for repeatable eval. |
| `mutuality` | enum `off` \| `min` \| `harmonic` | `off` | — | Combine target→candidate and candidate→target scores (open question — production should enable). |

## Njogu / Denis — `persona/get-scores` build plan

The implementation spec for the GCP function. Everything runs in **Go against AlloyDB** (Postgres + `pgvector`). There is **no FalkorDB in v1** — the candidate set is resolved *relationally* from the requesting user's contacts, not by graph traversal. The function takes the [input contract](#persona-get-scores-proposed-input-contract) above and returns the ranked score JSON that Robert's harness evaluates.

### Data model & the candidate-set rule

All personas live in one AlloyDB table, **eight `(narrative, vector)` rows per person**, linked to a `master_person`. A `contact` row links a `master_person` to a `user_id`. The join that defines the candidate pool is the heart of this function:

<Info>
**The rule:** given the **target** (`master_person`) and the **`user_id`** (the introducer), the candidate set is **every *other* persona belonging to that user's contacts** — i.e. `contact.user_id = $user_id → contact.master_person_id → persona`, minus the target. The user is the connector; we rank the target against the rest of *their* network.
</Info>

```
user_id ──(contact.user_id)──▶ contact ──(contact.master_person_id)──▶ master_person ──(persona.master_person_id)──▶ persona  (8 vectors)
                                                                                    └── exclude the target's master_person_id
```

### Steps to implement

<Steps>
<Step title="Resolve the target">
`person_id` (input) → the target's `master_person_id` (the `persona` key). Confirm the mapping — see open questions.
</Step>
<Step title="Resolve the candidate pool (the contact join)">
Select the `master_person_id` of every contact of `user_id` that has a persona, excluding the target. This is the only place the candidate set is defined — keep it one query.
</Step>
<Step title="Score in SQL: per-dimension cosine + topical cross-cells">
One AlloyDB query returns, per candidate, the seven embedding-scored diagonal dims (z-scored across the pool) plus the six raw `3×3` off-diagonal topical cells. Dim 7 (`community_affiliation`) is **not** embedding-scored.
</Step>
<Step title="Compose in Go: role-typed aggregation + serendipity">
Apply the input params — soft-max the drivers, sign the valence by `mode`, add modifiers; compute the serendipity lift over the driver pool. (No graph-bridge term in v1.)
</Step>
<Step title="Rank & return JSON">
Two lanes (diagonal / serendipity by lift floor), min-max normalized to a single overall order; emit the harness payload.
</Step>
</Steps>

### SQL — candidate resolution + per-dimension scoring (AlloyDB)

<CodeGroup>

```sql schema
-- Eight (narrative, vector) rows per person, keyed by master_person.
CREATE TABLE persona (
  master_person_id bigint       NOT NULL,
  dim              smallint     NOT NULL,         -- 1..8 (see "What a persona is")
  narrative        text,
  embedding        vector(3072) NOT NULL,         -- gemini-embedding, L2-normalized → cosine = 1-(<=>)
  updated_at       timestamptz  NOT NULL DEFAULT now(),
  PRIMARY KEY (master_person_id, dim)
);

-- contact (existing): a user's relationship to a master_person.
--   contact(id, user_id, master_person_id, ...)
CREATE INDEX IF NOT EXISTS contact_user_idx ON contact (user_id, master_person_id);
```

```sql get-scores query  ($1 = target master_person_id, $2 = user_id)
WITH cand AS (                               -- candidate pool = the user's contacts that have a persona, minus the target
  SELECT DISTINCT c.master_person_id AS mp
  FROM contact c
  WHERE c.user_id = $2
    AND c.master_person_id <> $1
    AND EXISTS (SELECT 1 FROM persona p WHERE p.master_person_id = c.master_person_id AND p.dim = 1)
),
tgt AS (                                     -- target's 8 vectors pivoted to columns (dim 7 handled outside the embedding)
  SELECT
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=1) AS v1,
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=2) AS v2,
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=3) AS v3,
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=4) AS v4,
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=5) AS v5,
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=6) AS v6,
    (SELECT embedding FROM persona WHERE master_person_id=$1 AND dim=8) AS v8
),
sims AS (                                    -- cosine sim = 1-(<=>), valid because 3072-d is normalized
  SELECT cand.mp AS master_person_id,
    1-(p1.embedding<=>t.v1) AS s_identity,
    1-(p2.embedding<=>t.v2) AS s_domain,
    1-(p3.embedding<=>t.v3) AS s_soft,
    1-(p4.embedding<=>t.v4) AS s_track,
    1-(p5.embedding<=>t.v5) AS s_public,
    1-(p6.embedding<=>t.v6) AS s_comm,
    1-(p8.embedding<=>t.v8) AS s_passion,
    -- 3×3 topical off-diagonal family {domain=2, public=5, passion=8}: target_dim × candidate_dim
    1-(p5.embedding<=>t.v2) AS x_dom_pub,    -- tgt domain  × cand public
    1-(p8.embedding<=>t.v2) AS x_dom_pas,    -- tgt domain  × cand passion
    1-(p2.embedding<=>t.v5) AS x_pub_dom,    -- tgt public  × cand domain
    1-(p8.embedding<=>t.v5) AS x_pub_pas,    -- tgt public  × cand passion
    1-(p2.embedding<=>t.v8) AS x_pas_dom,    -- tgt passion × cand domain
    1-(p5.embedding<=>t.v8) AS x_pas_pub     -- tgt passion × cand public
  FROM cand
  CROSS JOIN tgt t
  JOIN persona p1 ON p1.master_person_id=cand.mp AND p1.dim=1
  JOIN persona p2 ON p2.master_person_id=cand.mp AND p2.dim=2
  JOIN persona p3 ON p3.master_person_id=cand.mp AND p3.dim=3
  JOIN persona p4 ON p4.master_person_id=cand.mp AND p4.dim=4
  JOIN persona p5 ON p5.master_person_id=cand.mp AND p5.dim=5
  JOIN persona p6 ON p6.master_person_id=cand.mp AND p6.dim=6
  JOIN persona p8 ON p8.master_person_id=cand.mp AND p8.dim=8
)
SELECT s.master_person_id,
  -- standardize each diagonal dimension across THIS candidate pool (the user's network)
  COALESCE((s_identity-avg(s_identity)OVER())/NULLIF(stddev_pop(s_identity)OVER(),0),0) AS z_identity,
  COALESCE((s_domain  -avg(s_domain)  OVER())/NULLIF(stddev_pop(s_domain)  OVER(),0),0) AS z_domain,
  COALESCE((s_soft    -avg(s_soft)    OVER())/NULLIF(stddev_pop(s_soft)    OVER(),0),0) AS z_soft,
  COALESCE((s_track   -avg(s_track)   OVER())/NULLIF(stddev_pop(s_track)   OVER(),0),0) AS z_track,
  COALESCE((s_public  -avg(s_public)  OVER())/NULLIF(stddev_pop(s_public)  OVER(),0),0) AS z_public,
  COALESCE((s_comm    -avg(s_comm)    OVER())/NULLIF(stddev_pop(s_comm)    OVER(),0),0) AS z_comm,
  COALESCE((s_passion -avg(s_passion) OVER())/NULLIF(stddev_pop(s_passion) OVER(),0),0) AS z_passion,
  x_dom_pub, x_dom_pas, x_pub_dom, x_pub_pas, x_pas_dom, x_pas_pub   -- raw cross cells; z-scored per type in Go
FROM sims s;
```

</CodeGroup>

### Go — composition & scoring

Reads the rows above, applies the input params, returns ranked `Score`s. This is the `loop_generic` `ScoreBatch` adapted to the contact-resolved pool, parameterized, and with the graph-bridge term removed for v1.

```go
package persona

import (
	"math"
	"sort"
)

// Params == the persona/get-scores input contract (graph-only knobs omitted in v1).
type Params struct {
	Mode                 string  // "peer" | "complement"
	SoftmaxTemperature   float64 // 2.0
	WDrivers, WValence   float64 // 0.55, 0.20
	WSoft, WComm         float64 // 0.03, 0.02
	SerendipityLiftFloor float64 // τ, 0.50
	Limit                int     // 20
	// WBridge, NodeNudge, MaxHops belong to a future reachable-network mode — unused when the pool is the user's direct contacts.
}

// One row from the SQL query above.
type CandRow struct {
	MasterPersonID                                              int64
	ZIdentity, ZDomain, ZSoft, ZTrack, ZPublic, ZComm, ZPassion float64
	XDomPub, XDomPas, XPubDom, XPubPas, XPasDom, XPasPub         float64 // raw cross cells, SQL order
}

type Score struct {
	MasterPersonID int64   `json:"master_person_id"`
	Rank           int     `json:"rank"`
	Lane           string  `json:"lane"` // "diagonal" | "serendipity"
	Diagonal       float64 `json:"diagonal"`
	SerendipityZ   float64 `json:"serendipity_z"`
	Lift           float64 `json:"lift"`
	SerTgtDim      int     `json:"ser_tgt_dim"`  // provenance: the two narratives that explain the match
	SerCandDim     int     `json:"ser_cand_dim"`
}

// cross-cell provenance (targetDim, candidateDim), same order as the SQL columns.
var crossDims = [6][2]int{{2, 5}, {2, 8}, {5, 2}, {5, 8}, {8, 2}, {8, 5}}

func GetScores(rows []CandRow, p Params) []Score {
	n := len(rows)
	if n == 0 {
		return nil
	}

	// z-score each of the 6 cross-cell TYPES across the pool (baselines differ per cell type).
	var cols [6][]float64
	for j := range cols {
		cols[j] = make([]float64, n)
	}
	for i, r := range rows {
		cells := [6]float64{r.XDomPub, r.XDomPas, r.XPubDom, r.XPubPas, r.XPasDom, r.XPasPub}
		for j := 0; j < 6; j++ {
			cols[j][i] = cells[j]
		}
	}
	for j := range cols {
		cols[j] = zscore(cols[j])
	}

	out := make([]Score, 0, n)
	for i, r := range rows {
		drivers := softMax(p.SoftmaxTemperature, r.ZDomain, r.ZPublic, r.ZPassion)

		var valence, softSign float64 = 0, 1
		switch p.Mode {
		case "complement":
			valence, softSign = r.ZDomain-r.ZIdentity, -1 // complementary skills beat matched skills
		default: // "peer"
			valence = r.ZIdentity + r.ZTrack
		}
		modifiers := p.WSoft*softSign*r.ZSoft + p.WComm*r.ZComm

		// v1: embedding-only composite (no structural-hole bridge — see open questions).
		diagonal := p.WDrivers*drivers + p.WValence*valence + modifiers

		// serendipity: argmax over the z-scored topical cross-cells; lift vs the driver pool.
		best, bestJ := math.Inf(-1), -1
		for j := 0; j < 6; j++ {
			if cols[j][i] > best {
				best, bestJ = cols[j][i], j
			}
		}
		s := Score{MasterPersonID: r.MasterPersonID, Diagonal: diagonal, SerendipityZ: best, Lift: best - drivers}
		if bestJ >= 0 {
			s.SerTgtDim, s.SerCandDim = crossDims[bestJ][0], crossDims[bestJ][1]
		}
		if s.Lift >= p.SerendipityLiftFloor {
			s.Lane = "serendipity"
		} else {
			s.Lane = "diagonal"
		}
		out = append(out, s)
	}

	rankByOverall(out) // overall = max(minmax(diagonal), minmax(lift)), desc
	if p.Limit > 0 && len(out) > p.Limit {
		out = out[:p.Limit]
	}
	for i := range out {
		out[i].Rank = i + 1
	}
	return out
}

func rankByOverall(ss []Score) {
	rng := func(get func(Score) float64) (lo, hi float64) {
		lo, hi = math.Inf(1), math.Inf(-1)
		for _, s := range ss {
			v := get(s)
			lo, hi = math.Min(lo, v), math.Max(hi, v)
		}
		return
	}
	dlo, dhi := rng(func(s Score) float64 { return s.Diagonal })
	llo, lhi := rng(func(s Score) float64 { return s.Lift })
	norm := func(v, lo, hi float64) float64 {
		if hi <= lo {
			return 0
		}
		return (v - lo) / (hi - lo)
	}
	overall := make(map[int64]float64, len(ss))
	for _, s := range ss {
		overall[s.MasterPersonID] = math.Max(norm(s.Diagonal, dlo, dhi), norm(s.Lift, llo, lhi))
	}
	sort.SliceStable(ss, func(i, j int) bool {
		return overall[ss[i].MasterPersonID] > overall[ss[j].MasterPersonID]
	})
}

func softMax(temp float64, xs ...float64) float64 {
	if temp <= 1e-6 { // temp 0 → mean
		var s float64
		for _, x := range xs {
			s += x
		}
		return s / float64(len(xs))
	}
	var num, den float64
	for _, x := range xs {
		w := math.Exp(temp * x)
		num, den = num+w*x, den+w
	}
	if den == 0 {
		return 0
	}
	return num / den
}

func zscore(xs []float64) []float64 {
	n := float64(len(xs))
	if n == 0 {
		return xs
	}
	var mean float64
	for _, x := range xs {
		mean += x
	}
	mean /= n
	var v float64
	for _, x := range xs {
		v += (x - mean) * (x - mean)
	}
	sd := math.Sqrt(v / n)
	out := make([]float64, len(xs))
	if sd == 0 {
		return out
	}
	for i, x := range xs {
		out[i] = (x - mean) / sd
	}
	return out
}
```

### Open questions

<Info>
**No graph (FalkorDB) is needed for this scope — it's fully relational + vector on AlloyDB.** The candidate set comes from `contact` (`user_id → master_person → persona`) and the ranking is persona similarity; both live in AlloyDB. The graph signals in the full [scoring pipeline](/guides/open-work/suggestion-recipes/loop-generic) only matter for the broader *"rank everyone reachable"* mode and **collapse here**: when the pool is the user's **direct contacts**, there is no multi-hop distance to score, and the user is themselves the bridge between any two of their contacts. So `w_bridge` / `node_nudge` / `max_hops` are **out of scope for v1** (they belong to a future reachable-network mode), and the Go composite is embedding-only **by design** — not a limitation.

**The one thing it can't do:** detect whether the **target and a candidate already know each other** (to skip a redundant intro). `contact` encodes *user→person* edges, not *person↔person*, so v1 won't suppress already-connected pairs. Acceptable for the harness; if we want it later it needs a person↔person source (a relationship table, or a co-contact proxy across users) — **not** necessarily FalkorDB. See open questions.
</Info>

- **`person_id` → `master_person_id`.** Confirm the input `person_id` *is* the `master_person_id`, or define the resolution (persons table?).
- **`persona` table shape.** The SQL assumes **8 rows per person** keyed `(master_person_id, dim)`. If Mark's reference builder writes one row with 8 vector columns (or a vector array), the query changes — lock the shape before coding.
- **Small-pool standardization.** Z-scores are computed across *the user's contacts*. A user with few contacts → noisy/degenerate z-scores. Define a **minimum pool size** and a fallback (e.g. network-wide μ/σ per dimension precomputed offline).
- **Dim 7 (`community_affiliation`).** Graph-gated in the full method; with no graph it's just **another embedding dimension** here. Decide: score it on the diagonal like dims 1–6/8 (it has a narrative + vector), or leave it out of v1.
- **Already-connected detection (deferred).** If we later want to skip target↔candidate pairs that already know each other, add a person↔person source — a relationship/edge table, or a co-contact proxy (count of users who have *both* as contacts). Not needed for the harness; not a FalkorDB dependency.
- **Versioned personas.** If `persona` is bitemporal/versioned, add a `valid_to IS NULL` (current) filter to every join.
- **Embedding consistency.** Every persona must share one embedding model + dimensionality (gemini, 3072, normalized) or `1-(<=>)` is invalid. Reject/await mixed-model rows.
- **Mutuality.** v1 scores one direction (target→candidate). The method notes production should combine both directions (`min` / harmonic) — out of scope for v1, flag for v2.
- **Performance.** For large contact lists the per-dim cross-join + window z-scores are fine to a few thousand candidates; beyond that, index `persona(master_person_id, dim)`, consider an ANN prefilter, or precompute the pool. Confirm expected pool sizes with Mark.
