> **Status: planned.** This page is a parking lot for future enrichment engine improvements. The current sandbox waterfall can pass while still surfacing useful raw evidence that is intentionally not materialized yet.

## Why this exists

The enrichment waterfall now captures much richer raw evidence than it always materializes into first-class rows, queues, and graph edges.

During the 2026-07-05 Hazel Savage sandbox QA, the core waterfall and graph passed: people, schools, work-history edges, queue behavior, and graph metadata all settled cleanly. The non-blocking observation was that raw enrichment and deep-bio evidence appeared to contain extra older work-history signals beyond what the current PDL-led materialization wrote into `work_experience`.

That is not a data-quality failure by itself. It is a product and pipeline design question: when high-signal evidence is captured by Exa, Firecrawl, Crunchbase, or deep-bio extraction, should it remain source evidence only, become a reviewable candidate, or be materialized into the graph?

## Current behavior

`person_enrich_data` stores raw person source evidence from APIs and scrape/extraction helpers. For person waterfall QA, this table is the fastest place to see what was sourced but not necessarily used.

| Source area | Current role |
| --- | --- |
| `people_data_labs` | Primary structured person profile source for education and work-history materialization. |
| `enrich_layer_data` / `enrich_layer_email` | Fallback profile evidence and profile-photo/link evidence. |
| `exa` | Rich person profile evidence, biography inputs, and additional work-history style signals. |
| Crunchbase and scrape payloads | Stored source evidence for profiles, biographies, investment context, and downstream extraction work. |
| Deep-bio outputs | Human-readable biography and expertise context; may mention additional historical roles. |

The current materialization contract is conservative:

- PDL remains the primary source for structured work-history rows.
- Generic role strings such as "Board Member" or "Angel Investor" must not become company nodes without an explicit organization.
- Social-profile expanded URLs must not become company graph nodes without an explicit employer, education, or investment relationship.
- Past companies can get company nodes and `WORKS_AT` edges, then queue for later enrichment at bounded depth.
- Past companies must not trigger founders or C-suite fanout during the same person waterfall.
- Schools should full-enrich inline instead of waiting in a company queue.

## Hazel QA note

Hazel's clean sandbox pass still raised one useful v1.1 note: source evidence appeared to include older roles or references that were not all written into `work_experience`.

Examples to review in future runs:

- Shazam
- Universal Music Group
- HMV
- other earlier career references captured in Exa/deep-bio style evidence

These should not automatically become graph nodes just because they appear in prose. They should first become explicit, auditable candidates with source, confidence, and status.

## Proposed v1.1 candidate layer

Add a small candidate step between raw source evidence and materialized rows.

Possible home:

- a dedicated `person_enrichment_signal_candidate` table, if these signals need status, review, or retry lifecycle; or
- a structured JSON field on the relevant source row, if the first version remains source-local.

Recommended candidate fields:

| Field | Purpose |
| --- | --- |
| `master_person_id` | Person the signal belongs to. |
| `source_table` | Source table, usually `person_enrich_data` or `permalink_person`. |
| `source_row_id` | Source record id. |
| `source_name` | `People Data Labs`, `Exa`, `Crunchbase Person`, `Firecrawl`, `Deep Bio`, etc. |
| `signal_type` | `work_history`, `education`, `investment`, `profile_link`, `bio_fact`, etc. |
| `organization_name` | Raw organization name when present. |
| `title` | Raw role/title when present. |
| `start_date` / `end_date` | Normalized dates when present. |
| `source_url` | Exact URL supporting the signal when available. |
| `evidence_text` | Short extracted text span or source snippet. |
| `confidence` | Parser or matcher confidence. |
| `status` | `captured`, `matched`, `materialized`, `needs_review`, `skipped`. |
| `skip_reason` | Why the signal was not materialized. |

## Materialization rules

Before a v1.1 candidate writes rows, queues, or graph edges, it should pass explicit gates.

| Gate | Rule |
| --- | --- |
| Identity | Organization identity must resolve to a canonical company, school, or investment target. |
| Relationship | The relationship must be explicit: employer, school, investment, founder role, board role, etc. |
| Source quality | Prefer structured source fields over prose. Prose-only signals need higher confidence or review. |
| Duplicate safety | Match existing `work_experience`, `education_experience`, company, and graph edge records first. |
| Recursion control | Any newly queued company must stay depth-bounded and non-recursive. |
| Graph metadata | Every created node and edge must include `uuid` and `description`. |

## QA requirements

When this v1.1 work is built, `qa-the-sandbox` should add a captured-but-unused section:

- count raw work-history, education, investment, and profile-link candidates found in source evidence
- count candidates materialized into rows or graph edges
- count candidates skipped with `skip_reason`
- distinguish expected sparse/deferred companies from true missed materialization
- warn when high-confidence current work or education evidence remains unused
- keep prose-only historical references as warnings or candidates, not hard failures by default
- verify no social/profile URL became a company node without an explicit relationship
- verify queue fanout remained bounded after settlement polling

## Open decisions

| Decision | Default recommendation |
| --- | --- |
| Candidate storage | Use a dedicated table once these signals affect materialization. |
| Historical work fanout | Do not auto-materialize prose-only old roles until matching confidence is high. |
| QA severity | Treat captured-but-unused historical references as warnings, not failures, unless they are current roles or education. |
| Human review | Require review for low-confidence org matches and prose-only evidence. |
| Graph writes | Only write graph edges after the candidate is matched to an explicit relationship and canonical entity. |
