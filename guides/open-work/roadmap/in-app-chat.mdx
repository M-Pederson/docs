In-app chat for Orbiter, built on **[Stream](https://getstream.io)** for real-time messaging (plus video and audio) and **[Zep](https://getzep.com)** for long-term conversational memory. Stream provides the transport, channels, and UI primitives; Zep ingests those conversations into searchable, summarized memory that downstream Orbiter LLM agents can recall and reason over.

## Why this stack

### Stream — transport, UI, video, audio

- Battle-tested chat APIs and pre-built React components — channels, threads, reactions, typing indicators, read receipts, mentions, file uploads, search
- First-party SDKs across React, React Native, iOS, Android, and Flutter — same backend powers web and mobile
- Built-in moderation, role-based permissions, push notifications
- **Stream Video & Audio** SDKs share the same auth and user model as chat, so a thread can promote into a 1:1 call or group video room without re-onboarding participants
- Voice notes inside threads, group huddles, screen share, recording — all on the same SDK
- Webhooks and server-side hooks let us mirror every event into Xano or pipe directly into Zep on send

### Zep — memory layer for LLM context

- Long-term conversational memory: vector + keyword + recency hybrid search over message history
- Auto-extracts facts, entities, and summaries from raw chats; summaries stay current as the conversation grows
- Per-user, per-session, and group memory across multiple participants
- Hosted or self-hosted; drop-in Python and TypeScript SDKs

### What the combination unlocks

1. **Context-aware AI participants.** A bot or copilot in a Stream channel pulls memory from Zep before responding — so "what did Mark and I decide about the merge spec last week?" gets a real answer, not just whatever's in scrollback.
2. **Cross-channel intelligence.** Zep stores all chat history globally indexed by user. Discover/paths/suggest agents can pull conversational signal ("this person mentioned they were hiring a designer") without scraping every channel by hand.
3. **Persistent personas.** Zep's facts-about-a-user (preferences, ongoing projects, communication style) feed every Orbiter LLM call, not just chat. Today we re-derive person context on every call; with Zep, that context is built once and stays warm.
4. **Voice and video as first-class memory.** Stream Video records and transcribes; push transcripts into Zep with the same ingest API and audio/video moments become indexable alongside text.

## Suggested rollout

<Steps>
  <Step title="Stand up Stream first">
    Ship the Stream-only happy path on one use case (team chat or 1:1 DMs) — channels, messages, mentions, reactions, threads. Use Stream's React components verbatim; theming comes later.
  </Step>
  <Step title="Mirror messages into Zep">
    Add a Stream webhook that posts every new message into Zep, keyed by Stream user ID and channel ID. No agent yet — just build the memory store and validate ingest at volume.
  </Step>
  <Step title="Add the first AI participant">
    Pick one channel type (e.g. deal-flow). Add a bot user in Stream, back it with Zep memory plus the Orbiter graph, and let users @-mention it for recall and suggestions. Iterate on the prompt before scaling out.
  </Step>
  <Step title="Layer in video & audio">
    Enable Stream Video on existing channels — same users, same auth. Capture transcripts, push them into Zep, and the AI participant inherits memory of meetings too.
  </Step>
</Steps>

### What Stream provides out of the box

- **Channels** — public, private, hidden, password-protected; per-channel roles (owner / moderator / member); custom channel types with their own permission matrix
- **Threads** — full reply trees with parent-message context and "view in channel" jumps
- **Direct messages** — 1:1 and small-group DMs sharing the same primitives as channels
- **Reactions & custom emoji** — unicode plus server-side custom reactions
- **Mentions & notifications** — `@user` and `@channel` mentions with push, email, and webhook fallbacks
- **Typing indicators, read receipts, online presence** — wired into the React SDK
- **File and image uploads** with previews and inline media rendering
- **Search** — full-text across messages, channels, and members; scopable per channel
- **Slash commands** — built-in (`/giphy`, `/mute`) and custom; ideal surface for Orbiter agent triggers (`/find-investors`, `/draft-intro`, `/who-knows`)
- **Message permissions** — edit, delete, pin, flag — configurable per role
- **Custom message types** — render any structured payload (a person card, a deal card, a chart, a path-to-X graph) inline in conversation
- **Moderation tools** — flag, ban, mute, automod — all server-side enforced

### What we layer on top

- **Entity mentions** — `@person`, `@company`, `@deal` that resolve to graph nodes and render rich cards inline
- **AI agents as members** — with Zep-backed memory and live graph context; invoked by mention or slash command
- **Cross-channel suggestions** — "this conversation looks like a deal-flow update; want to log it?"
- **Voice notes and video huddles** via Stream Video, with auto-transcripts piped to Zep

## Open questions

- **Zep hosting.** Hosted (zero ops, less control) vs. self-hosted alongside Xano and FalkorDB on our infra. Affects cost, data residency, and how memory ingest scales.
- **Identity model.** Map Orbiter `master_person` IDs directly to Stream user IDs, or use a separate stable ID? Affects displayName, avatar, and presence sync.
- **Channel taxonomy.** One workspace per Orbiter org with freeform channels, or channels auto-keyed to graph entities (one per deal, one per project)?
- **Bot permissions.** What can an AI participant read/write — same as a member, or a privileged read-only mode by default? Affects compliance posture.
- **Audio/video scope.** Start with 1:1 calls only, or jump straight to group huddles and recordings?
