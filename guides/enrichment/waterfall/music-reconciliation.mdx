## Built ‚ÄĒ what actually shipped (2026-06-14)

Both reconciliation jobs are live. The person side **departed from the scoped node-merge** because MB artists carry no `linkedin_url` (and rarely match an existing business `Person`), so there is usually nothing to merge *into*. Instead we **bridge**:

| Job | Built mechanism | Handoff |
|---|---|---|
| `Company:Music_Label` ‚Üí `Company` | depth-0 `SAME_AS` (w100) inline in `mb/run-base-label-enrich` (#13091): MB `official homepage` rel ‚Üí else Exa `find-label-domain` (#13116) ‚Üí `master-company-new` (#12558) ‚Üí stamp `mb_label.master_company_id` | [label‚ÜíCompany](/guides/dev-updates/2026-06-14-label-company-inline-orchestration) |
| `Person:Music_Artist` ‚Üí `Person` | async queue + worker (`mb/run-person-resolution` #13121): Exa `find-person-social` (#13118) cross-checked vs MB url-rels + aliases ‚Üí **mint/link a `master_person`** (#13039) carrying all verified socials as `master_link` rows (auto-fires the twitter/instagram/youtube/linkedin scrapers + `social_insights` + influence) ‚Üí stamp `mb_artist.master_person_id` ‚Üí `SAME_AS` (w100). Verified **LinkedIn** additionally runs the full people pipeline. Plus node `social_links` annotation + serper.dev avatar. | [person-resolution](/guides/dev-updates/2026-06-14-person-resolution) |

Why a bridge, not a merge: a music identity (`Music_Artist`) and a business identity (`master_person`) are distinct records of the same human ‚ÄĒ keep both, link them ‚ÄĒ and the social/people enrichment runs off `master_link`, so even a no-LinkedIn artist's socials get scraped/enriched once they hang off a `master_person`. The scoped Tier-1‚Üí4 KNN merge below was **not built** for persons.

## Build spec ‚ÄĒ label owner resolution (`OWNED_BY`)

<Note>
**Status: BUILT + sandbox-verified, forward-path-live 2026-06-17.** Resolver `mvp/music/resolve-label-owner` (**#13132**, Exa + `deepseek/deepseek-v4-flash`) + linker `mb/resolve-and-link-label-owner` (**#13133**) ship live; the linker is wired into `mb/run-base-label-enrich` (**#13091**) as a depth-0 phase after the company `SAME_AS`, so seeding a label now auto-resolves and links its owner. `OWNED_BY` was added to the edge-description allowlist (**#13097**), and those edges get LLM descriptions via the deepseek describer (**#13095**). Verified end-to-end on sandbox: `Hardly Art ‚ÄĒOWNED_BY‚Üí Sub Pop Records` and `Matador ‚ÄĒOWNED_BY‚Üí Beggars Group` (both `mb_subsidiary`, w15), Tier-1‚ÜíTier-2 fallback (Def Jam ‚Üí Universal Music Group), idempotency, and graceful degradation. Manual link endpoint: `tool/link-label-owner` (#8689).
</Note>

**Goal.** When a `Company:Music_Label` is created, resolve **two** company concepts: (1) the label's canonical company profile ‚ÄĒ `SAME_AS` (done) ‚ÄĒ and (2) the company that *owns* it ‚ÄĒ `OWNED_BY`. Owner resolution is **tiered** so it works whether or not MusicBrainz records the parent.

**Why a new edge.** An Exa-discovered corporate parent (e.g. Sub Pop ‚Üí Warner Music Group) usually has **no MB mbid**, so a clean label‚Üílabel `SUBSIDIARY_OF` isn't writable. Decision (2026-06-17): `OWNED_BY` `(:Company:Music_Label)‚Üí(:Company)` is **the** canonical owner edge, written in *both* tiers. `SUBSIDIARY_OF` stays for music-label lineage only. See the canonical [identity and ownership bridge spec](/guides/ontology/edges#music-identity-ownership-bridges).

### 1. Schema ‚ÄĒ add to `mb_label`

Add via the Xano **UI** (the MCP full-replace strips `filters=trim`; **never** `xano workspace push` ‚ÄĒ it wipes the data source):

| Column | Type | Notes |
|---|---|---|
| `owner_company_id` | int ‚Üí `master_company` | `0` = unresolved |
| `owner_match_method` | text | `mb_subsidiary` / `exa_llm` / `none` |
| `owner_confidence` | float | 0‚Äď1 |
| `owner_resolved_at` | timestamp | null until resolved |

### 2. New resolver `mb/resolve-label-owner` (side-effect-free)

Mirrors `find-label-domain` (#13116). **In:** `label_name`, `label_domain?`, `label_area?`, `label_mbid`. Exa search ‚Üí LLM extract ‚Üí `{owner_name, owner_domain, confidence, source_url}` or `null`. **Accept only** when `source_url` is present **and** `confidence ‚Č• 0.7` (anti-hallucination, same cross-check ethos as person-resolution). LLM call sets `reasoning:{enabled:false}` + a `max_tokens` cap (reasoning models otherwise return empty `content`). Writes nothing ‚ÄĒ pure resolver.

### 3. Hook into `mb/run-base-label-enrich` (#13091)

New `try_catch` phase **`mb_owner`** (append the value to the `log_crash.phase` enum *first* via the whole-schema Metadata PUT ‚ÄĒ it's a strict enum). Runs after the existing `SAME_AS` block; **skips when `owner_company_id != 0`** (idempotent re-enrich).

1. **Tier 1 ‚ÄĒ MB-native:** find a *parent* in `label-rels` ‚ÄĒ an **incoming `label ownership`** relation **or** a **backward `imprint`** relation (‚ö†ÔłŹ the common *forward* `label ownership` points at labels this label **owns**, i.e. children ‚ÄĒ do not invert the hierarchy). Keep the existing `SUBSIDIARY_OF` (w15) + parent-queued-T1; take the parent's **name**, resolve its **domain** via `find-label-domain` (#13116). `match_method = mb_subsidiary`.
2. **Tier 2 ‚ÄĒ Exa:** no MB parent ‚Üí `mb/resolve-label-owner`. `match_method = exa_llm`.
3. **Converge:** `(owner_name, owner_domain)` ‚Üí `master-company-new` (#12558, `hold_downstream=true`, create-on-miss) ‚Üí stamp the four `mb_label` fields ‚Üí write the edge via `mvp/falkor/send-cypher` (#2815; guard `typeof r === 'string'` before reading `.data`):

```cypher
MERGE (l:Company:Music_Label {uuid: $label_uuid})
MERGE (c:Company {uuid: $owner_company_uuid})
MERGE (l)-[r:OWNED_BY]->(c)
SET r += {match_method: $mm, confidence: $conf, source: $src, weight: 15}
// + standard edge envelope: uuid, created_at, updated_at, description, description_generated_at
```

### 4. Coverage gate ‚ÄĒ result (probed 2026-06-17)

The graph is too sparsely seeded to measure from (live 17 labels / 0 `SUBSIDIARY_OF`, staging 1, sandbox 5 / 2 `SUBSIDIARY_OF`), so coverage was probed against the **live MusicBrainz API** on the named cases:

| Label | `label-rels` returned | Owner found by Tier 1? |
|---|---|---|
| **Sub Pop Records** | 2√ó `label ownership` **forward** ‚Üí Hardly Art, Next Ambiance (labels Sub Pop *owns*) | ‚ĚĆ ‚ÄĒ corporate parent (Warner Music Group) **absent from MB** |
| **Matador** | `imprint` **backward** ‚Üí Beggars Group; `label distribution` (both ways); `label ownership` forward ‚Üí True Panther | ‚úÖ ‚ÄĒ but via **`imprint`/backward**, not `label ownership` |

**Findings that shaped the spec:**
- **Direction is load-bearing** ‚ÄĒ MB's prevalent `label ownership` **forward** edge points at *children*, not the parent. The owner is an **incoming `label ownership`** or **backward `imprint`** (see Tier 1 above).
- **Ownership spans ‚Č•2 relation types** (`label ownership`, `imprint`, arguably `business association`).
- **MB structurally misses the ultimate corporate parent** (Sub Pop‚ÜíWarner is simply not modeled). ‚Üí **Tier 2 (Exa) is required, not optional** ‚ÄĒ it's the only path to the major-label/holding-co parent for a large fraction of labels.

**Before scaling:** seed a representative batch of labels through #13091 and re-run the graph counts to size the real Tier-1 hit-rate; the MB probe above already confirms Tier 2 must exist.

## The two reconciliation jobs

| Job | Mechanism | Why different |
|---|---|---|
| `Person:Music_Artist` ‚Üí existing `Person` | **Node merge** (graph dedup ‚ÄĒ edges repoint onto the surviving node, no bridge edge) | A person artist *is* an Orbiter Person; two nodes for one human is a dedup bug |
| `Company:Music_Label` ‚Üí `Company` | **`SAME_AS` edge** (w1, `match_method` + `confidence` params) + stamp `mb_label.master_company_id` | A label and a company profile can be distinct records of the same entity ‚ÄĒ keep both, link them |

## Reusable infrastructure (audited 2026-06-12)

<Card icon="magnifying-glass">

```
mvp/resolve/match-master-person ‚ÄĒ 13038
```

Resolves a `master_person_id` from links / emails / phones, with a `linkedin_url` direct fallback; returns `match_via`. **Conditionally reusable**: only when the MB artist's `url-rels` contain a `linkedin.com/in/` URL. It requires link/email-grade keys ‚ÄĒ MB artists have neither by default, so this is Tier 1, not the workhorse.

</Card>

<Card icon="cloud-arrow-up">

```
mvp/falkor/send-cypher-with-embeddings ‚ÄĒ 12566
```

Takes a 1536-float vector + Cypher, inlines it as a `vecf32()` literal, and runs `db.idx.vector.queryNodes` KNN with score thresholds (0.65 general / 0.85 expertise). **Directly reusable** as the name-similarity workhorse ‚ÄĒ KNN the artist's `name_embedding` against `Person` nodes, the label's against `Company` nodes. This is the canonical name-matching pattern in the codebase (`resolve-person-expertise` 12668/12926).

</Card>

<Card icon="wrench">

```
merge/helpers/graph-merge-person ‚ÄĒ 12949
merge/helpers/resync-person-node-uuid ‚ÄĒ 12964
merge/helpers/graph-merge-company ‚ÄĒ 12958
```

The merge machinery. `graph-merge-person` repoints edges from the orphan node onto the survivor; `resync-person-node-uuid` sweeps the **13 tables** that store `person_node_uuid` after a merge. **Directly reusable** for the artist‚ÜíPerson merge once a match is confirmed. The company variant exists if a label turns out to *be* the company node (full merge instead of `SAME_AS`).

</Card>

<Card icon="building">

```
mvp/get-add/master-company-new ‚ÄĒ 12558
```

Get-or-create `master_company` by domain + profile URLs (LinkedIn / Crunchbase), with `hold_downstream=true` to skip enrich dispatch. **Directly reusable** for the label side: the MB label's `url-rels` `official homepage` relation yields the domain that feeds 12558's match path. On match/create: stamp `mb_label.master_company_id` (already in the schema, `0` = unreconciled) and write the `SAME_AS` edge via send-cypher (2815).

</Card>

<Warning>
**Not reusable:** `mvp/get-add/master-person-new` (13039) and `mvp/enrich/enrich-master-person` (13040). 13039 requires email/LinkedIn keys and unconditionally runs the PDL‚ÜíEnrichLayer‚ÜíExa cascade (wrong/expensive for MB artists); 13040 enriches known persons ‚ÄĒ it neither matches nor merges. The piece to *copy* (not call) from 13039 is its pre-API `master_link` lookup: query `master_link` for any of the MB artist's `url-rels` URLs (Wikipedia, Discogs, socials).
</Warning>

## Proposed flow ‚ÄĒ Person:Music_Artist merge

1. **Select:** `mb_artist` rows with `artist_type = 'Person'`, `node_uuid` set, not yet reconciled (needs a tracking column ‚ÄĒ see open questions).
2. **Tier 1 ‚ÄĒ hard keys:** `url-rels` contains `linkedin.com/in/` ‚Üí `mvp/resolve/match-master-person` (13038). Cheapest definitive match.
3. **Tier 2 ‚ÄĒ shared links:** any `url-rels` URL hits `master_link` (the 13039 pre-API pattern, adapted).
4. **Tier 3 ‚ÄĒ name embedding:** KNN via 12566 against `Person` nodes. Candidates below the distance threshold go to Tier 4; no candidate ‚Üí no match, stop.
5. **Tier 4 ‚ÄĒ LLM confirmation:** two-tier confirm pattern from the merge-detection pipeline before any merge fires.
6. **On confirmed match:** `graph-merge-person` (12949) + `resync-person-node-uuid` (12964); the surviving `Person` node keeps/gains the `Music_Artist` label and music properties.

## Proposed flow ‚ÄĒ Company:Music_Label ‚Üí Company SAME_AS

1. **Select:** `mb_label` rows with `node_uuid` set and `master_company_id = 0`.
2. **Tier 1 ‚ÄĒ domain:** `url-rels` `official homepage` ‚Üí domain ‚Üí `master-company-new` (12558, `hold_downstream=true`).
3. **Tier 2 ‚ÄĒ name embedding:** KNN via 12566 against `Company` nodes, LLM-confirmed.
4. **On match:** stamp `mb_label.master_company_id` + write `SAME_AS` w1 with `match_method` (`domain` / `embedding+llm` / ‚Ä¶) and `confidence` (0‚Äď1) ‚ÄĒ the canonical [identity and ownership bridge spec](/guides/ontology/edges#music-identity-ownership-bridges) already carries both params for auditability.

## Open design questions

These need decisions before build (the same design-first gate as the [original six](/guides/enrichment/waterfall/music-industry-waterfall#resolved-design-decisions-2026-06-11)):

1. **Auto-merge vs review queue.** Person merges are destructive-ish (edges repoint). Do Tier 3/4 (embedding + LLM) matches merge automatically, or land in a review table for human confirmation with only Tier 1/2 (hard-key) matches auto-merging?
2. **Match-state tracking on `mb_artist`.** `mb_label` has `master_company_id`; `mb_artist` has no equivalent. Add `master_person_id` (tableref, `0` = unreconciled) ‚ÄĒ and a `reconcile_status` (`pending` / `matched` / `no_match` / `review`) so no-match rows aren't rescanned forever?
3. **Embedding threshold.** 12566's general threshold is 0.65. Person-name collisions (common names) argue for a stricter bar + mandatory LLM confirm; what's the floor?
4. **Create-on-miss for companies.** 12558 *creates* a `master_company` when nothing matches. Should an unmatched `Company:Music_Label` create a new company (Sub Pop deserves one) or only link to existing companies in v1?
5. **Embedding source mismatch.** Node descriptions (13113) now overwrite `name_embedding` with description-based vectors for `Music_Group`/`Recording` ‚ÄĒ person artists keep name+disambiguation embeddings from 13092. Reconciliation KNN should run against `Person` embeddings, which are name-based ‚ÄĒ consistent today, but worth pinning before descriptions expand to more node types.
6. **Owner-company resolution for labels MB doesn't link.** ‚úÖ *Resolved 2026-06-17* ‚ÄĒ decided to build a tiered owner resolver writing a direct `(:Company:Music_Label)-[:OWNED_BY]->(:Company)` edge. Promoted to its own [Build spec ‚ÄĒ label owner resolution](#build-spec--label-owner-resolution-owned_by) section above.
