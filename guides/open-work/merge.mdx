## Why merge is hard

A duplicate is not just two rows in `master_person`. It's two rows that are each pointed at by
**86 other tables for person** and **65 other tables for company**, plus a node in FalkorDB with
N edges, plus join rows under `my_person` for every user who has either copy in their CRM.

A correct merge has to:

1. Pick a survivor and aggregate scalar fields from the duplicates onto it
2. Repoint every FK in every dependent table from each duplicate ŌåÆ survivor
3. Collapse rows that become duplicates after repoint (e.g. two `master_email` rows with the same address)
4. Re-route every graph edge from the duplicate Person/Company node ŌåÆ survivor node, deduping on type
5. Delete the duplicate rows + duplicate nodes
6. Re-derive computed state (employment, embeddings, person/company nodes)
7. Be idempotent ŌĆö re-running with the same input is a no-op
8. Be audited ŌĆö every merge writes a snapshot row that can roll back

Inputs follow the same shape for both:

```ts
type MergeInput = {
  survivor_id: number;          // master_person_id or master_company_id we keep
  duplicate_ids: number[];      // array of master IDs that collapse into the survivor
  initiated_by: 'user' | 'cron' | 'admin';
  merge_suspect_id?: number;    // when triggered from merge_suspect (#284)
};
```

### Phase flow

<Steps>
  <Step title="Validate & lock">
    Reject if `survivor_id` Ōłł `duplicate_ids`, if any duplicate is `null`, or if any `master_person`
    row is missing. Lock all involved rows for the duration of the merge by writing a row to
    `merge_lock` (new table ŌĆö see [Open questions](#open-questions)).
  </Step>
  <Step title="Snapshot for rollback">
    Write a `merge_audit` row (new table) with the JSON of the survivor + each duplicate's
    `master_person` row, plus the counts of dependent rows in every table touched. This is the
    rollback artifact and the audit log.
  </Step>
  <Step title="Aggregate scalar fields onto the survivor">
    For each scalar column on `master_person` (139), apply the column-specific merge rule below.
  </Step>
  <Step title="Repoint master_person_id FKs across all 86 dependent tables">
    `UPDATE ŌĆ” SET master_person_id = survivor_id WHERE master_person_id IN (duplicate_ids)`.
  </Step>
  <Step title="Reconcile my_person + its joins">
    For each user that has a `my_person` (128) row pointing at one of the duplicates, fold it
    into their existing `my_person` row pointing at the survivor (if any), or repoint it.
  </Step>
  <Step title="Dedup multi-row tables">
    After the repoint, several tables now contain rows that are duplicates of each other on a
    natural key. Collapse them per the rules below.
  </Step>
  <Step title="Regenerate derived state">
    Re-run `mvp/work/process-work-experience` (rebuilds `employment` 263) and
    `mvp/node/update-person-node` (#2758, syncs FalkorDB Person node properties + embedding).
  </Step>
  <Step title="Graph merge in FalkorDB">
    Re-route every edge from each duplicate Person node ŌåÆ survivor Person node, dedup on
    `(type, target_uuid)`, then `DETACH DELETE` the duplicates. See [Graph merge cypher](#graph-merge-cypher).
  </Step>
  <Step title="Delete duplicate master_person rows">
    Only after every dependent FK is repointed, every join is reconciled, and the graph is in sync.
  </Step>
  <Step title="Update merge_suspect + re-enqueue enrichment">
    Set `merged=true` on every `merge_suspect` (284) row that referenced any of the merged IDs.
    Push the survivor to `queue_enrich_person` (582) so search indexes refresh.
  </Step>
</Steps>

### Scalar field aggregation rules (master_person, table 139)

| Field group | Rule |
| --- | --- |
| `name`, `first_name`, `middle_name`, `last_name`, `suffix`, `nickname` | Survivor wins if non-null. Else longest non-null from duplicates. Re-run `mvp/format/set-person-names` after. |
| `headline`, `bio_500`, `bio_2000`, `tagline` | Longest non-null wins. |
| `is_angel`, `is_c_suite`, `is_investor`, `is_founder`, any `is_*` boolean | Logical OR across survivor + all duplicates. |
| `node_uuid` | Survivor's node_uuid is canonical. Duplicates' node_uuids drop out via the graph merge. |
| `master_company_id` (primary current company) | Survivor wins if non-null; else most recent non-null from duplicates by `updated_at`. |
| `visibility`, `enriched_at`, `qa_passed`, `enrichment_complete_at` | Survivor wins; if any duplicate has a more recent `enriched_at`, copy that timestamp + the related fields together as a unit. |
| `created_at` | Min across survivor + duplicates (preserve oldest provenance). |
| `updated_at` | `now`. |

### my_person reconciliation

`my_person` (128) is per-user. Different users can each have their own `my_person` row pointing at
the same `master_person`. After repointing, multiple rows for the same `(user_id, master_person_id)`
are illegal ŌĆö collapse them.

For each user that owns a `my_person` row pointing at any duplicate:

1. Look up the user's existing `my_person` row pointing at `survivor_id` (if any).
2. **If none**: just `UPDATE my_person SET master_person_id = survivor_id WHERE id = duplicate_my_person.id`. Done.
3. **If one exists**: fold the duplicate's joins into the survivor's joins, then delete the duplicate `my_person` row.

Tables that hold `my_person_id` and need fold-in handling:

| Table | ID | Rule when folding |
| --- | --- | --- |
| `my_person_email_join` | 312 | Repoint to survivor's `my_person.id`, then dedup on `(my_person_id, master_email_id)` |
| `my_person_phone_join` | 313 | Repoint, dedup on `(my_person_id, master_phone_id)` |
| `my_person_address_join` | 310 | Repoint, dedup on `(my_person_id, master_address_id)` |
| `my_person_link_join` | 311 | Repoint, dedup on `(my_person_id, master_link_id)` |
| `my_note_person_join` | 318 | Repoint, dedup on `(my_person_id, my_note_id)` |
| `activity` | 218 | Repoint, then dedup on `(my_person_id, message_id)` ŌĆö keep oldest, merge non-null metadata (`read_at`, `replied_at`, etc.) from losers onto the winner |
| `email_activity` | 597 | Repoint, then dedup on `(my_person_id, message_id)` ŌĆö same metadata-merge rule |
| `calendar_event_activity` | 598 | Repoint, then dedup on `(my_person_id, event_id)` ŌĆö same metadata-merge rule |
| `important_date` | 242 | Repoint, dedup on `(my_person_id, date_descriptor_id, date_value)` |
| `possible_duplicates` | 400 | Repoint; delete rows where both sides become the same `my_person_id` |

```text
function: merge/company-v2  (TO BUILD ŌĆö workspace 3, branch v1)
```

### Phase flow

Same shape as person merge:

<Steps>
  <Step title="Validate & lock">
    Reject self-merge, missing rows; lock all `master_company` rows.
  </Step>
  <Step title="Snapshot for rollback">
    Write `merge_audit` row.
  </Step>
  <Step title="Aggregate scalar fields onto the survivor">
    See [scalar rules below](#scalar-field-aggregation-rules-master-company-table-142).
  </Step>
  <Step title="Repoint master_company_id FKs across all 65 dependent tables">
    `UPDATE ŌĆ” SET master_company_id = survivor_id WHERE master_company_id IN (duplicate_ids)`.
  </Step>
  <Step title="Reconcile my_company">
    `my_company` (149) is per-user; repoint with the same logic as `my_person`.
  </Step>
  <Step title="Repoint master_person.master_company_id">
    Anyone whose primary current company points at a duplicate gets repointed to the survivor.
  </Step>
  <Step title="Dedup multi-row tables">
    Identity records, joins, and any other table that becomes duplicate-keyed after repoint.
  </Step>
  <Step title="Regenerate derived state">
    Re-run `mvp/node/add-update-company-node` to refresh the survivor's FalkorDB Company node
    properties + embedding. Re-run `process-work-experience` for any affected master_persons
    (their `employment` rows now reference the survivor).
  </Step>
  <Step title="Graph merge in FalkorDB">
    Re-route every edge from duplicate Company nodes ŌåÆ survivor Company node, dedup on type +
    target. See [Graph merge cypher](#graph-merge-cypher). The existing
    `mvp/dupe/merge-company-nodes` (#10505) is a starting point for the property-merge logic
    but only handles `profiles` ŌĆö the new version must handle every property and every edge type.
  </Step>
  <Step title="Delete duplicate master_company rows">
    Only after every dependent FK is repointed and the graph is in sync.
  </Step>
  <Step title="Update merge_suspect + re-enqueue enrichment">
    Add a company-side `merge_suspect` table or extend #284 with a `kind` column. Push survivor
    to `queue_enrich_company` (583).
  </Step>
</Steps>

## Graph merge cypher

The graph merge runs **after** every Xano FK is repointed but **before** the duplicate
`master_*` rows are deleted (so we still know which `node_uuid` to delete from the graph).

FalkorDB does not support APOC, so edge re-routing must enumerate relationship types. We have
~25 edge types in production ŌĆö see [Edges](/guides/ontology/edges) ŌĆö so the cypher template
below loops over them in the orchestrator.

### Step 1 ŌĆö Aggregate properties onto the survivor

<CodeGroup>
```cypher Person
MATCH (s:Person {uuid: $survivor_uuid})
UNWIND $duplicate_uuids AS dup_uuid
MATCH (d:Person {uuid: dup_uuid})

SET s.full_name      = coalesce(s.full_name, d.full_name),
    s.first_name     = coalesce(s.first_name, d.first_name),
    s.last_name      = coalesce(s.last_name, d.last_name),
    s.headline       = coalesce(s.headline, d.headline),
    s.bio_500        = coalesce(s.bio_500, d.bio_500),
    s.bio_2000       = coalesce(s.bio_2000, d.bio_2000),
    s.is_angel       = coalesce(s.is_angel, false) OR coalesce(d.is_angel, false),
    s.is_c_suite     = coalesce(s.is_c_suite, false) OR coalesce(d.is_c_suite, false),
    s.is_investor    = coalesce(s.is_investor, false) OR coalesce(d.is_investor, false),
    s.is_founder     = coalesce(s.is_founder, false) OR coalesce(d.is_founder, false),
    s.profiles       = CASE
      WHEN d.profiles IS NULL THEN s.profiles
      WHEN s.profiles IS NULL THEN d.profiles
      ELSE s.profiles + ',' + d.profiles
    END

RETURN s.uuid AS survivor_uuid, count(d) AS duplicates_seen
```

```cypher Company
MATCH (s:Company {uuid: $survivor_uuid})
UNWIND $duplicate_uuids AS dup_uuid
MATCH (d:Company {uuid: dup_uuid})

SET s.company_name   = coalesce(s.company_name, d.company_name),
    s.company_domain = coalesce(s.company_domain, d.company_domain),
    s.tagline        = coalesce(s.tagline, d.tagline),
    s.summary_500    = coalesce(s.summary_500, d.summary_500),
    s.summary_2000   = coalesce(s.summary_2000, d.summary_2000),
    s.is_public      = coalesce(s.is_public, false) OR coalesce(d.is_public, false),
    s.is_acquired    = coalesce(s.is_acquired, false) OR coalesce(d.is_acquired, false),
    s.is_school      = coalesce(s.is_school, false) OR coalesce(d.is_school, false),
    s.profiles       = CASE
      WHEN d.profiles IS NULL THEN s.profiles
      WHEN s.profiles IS NULL THEN d.profiles
      ELSE s.profiles + ',' + d.profiles
    END

RETURN s.uuid AS survivor_uuid, count(d) AS duplicates_seen
```
</CodeGroup>

After this query returns, normalize comma-separated `profiles` (split ŌåÆ trim ŌåÆ unique ŌåÆ join) in
a follow-up `SET` to keep the column compact. The pattern is borrowed from
`mvp/dupe/merge-company-nodes` (#10505) but generalized to all string-list properties.

### Step 2 ŌĆö Re-route edges per relationship type

For each edge type, run the template below with `$rel_type` substituted in. The orchestrator
loops over the edge type list from [Edges](/guides/ontology/edges).

```cypher
// $rel_type is substituted in by the template engine ŌĆö Cypher does not allow parameterized rel types
MATCH (d:Person)-[r:KNOWS]-(other)
WHERE d.uuid IN $duplicate_uuids AND other.uuid <> $survivor_uuid

WITH d, other, r, properties(r) AS props,
     CASE WHEN startNode(r) = d THEN 'out' ELSE 'in' END AS direction

MATCH (s:Person {uuid: $survivor_uuid})

// Branch on direction so we preserve edge orientation
FOREACH (_ IN CASE WHEN direction = 'out' THEN [1] ELSE [] END |
  MERGE (s)-[new_r:KNOWS]->(other)
  SET new_r += props,
      new_r.merged_from = coalesce(new_r.merged_from, []) + d.uuid
)
FOREACH (_ IN CASE WHEN direction = 'in' THEN [1] ELSE [] END |
  MERGE (other)-[new_r:KNOWS]->(s)
  SET new_r += props,
      new_r.merged_from = coalesce(new_r.merged_from, []) + d.uuid
)

DELETE r
RETURN count(*) AS edges_rerouted
```

**Per-edge-type aggregation rules** when the `MERGE` lands on an existing edge:

| Edge type | Aggregation |
| --- | --- |
| `KNOWS` | Apply `known/ingest/append_evidence` (#12779) semantics: union evidence array, max confidence, min `known_since`, max `last_evidence_at` |
| `HAS_EXPERTISE` | Apply `mvp/expertise/upsert-person-has-expertise` (#12925) semantics: max confidence, strongest depth, most current recency, pipe-concat evidence |
| `WORKED_AT`, `EDUCATED_AT` | Keep both edges if `start_date` differs; merge if same `start_date` (sum tenure-derived weight) |
| `INVESTED_IN`, `RAISED` | Keep both ŌĆö investments are time-stamped events, not duplicates |
| `REGISTERED_FOR`, `ATTENDED` | Dedup on target uuid (one event per person) |
| `WON_AWARD`, `NOMINATED_FOR` | Dedup on target Award uuid |
| Any other relationship | Default: `SET new_r += props` (last write wins on scalar props) |

### Step 3 ŌĆö Detach delete the duplicate nodes

```cypher
MATCH (d:Person)
WHERE d.uuid IN $duplicate_uuids
DETACH DELETE d
RETURN count(d) AS nodes_deleted
```

For company side, swap `:Person` ŌåÆ `:Company`.

### Step 4 ŌĆö Re-sync the survivor node

After the merge, call `mvp/node/update-person-node` (#2758) for person or the equivalent
`mvp/node/add-update-company-node` for company. This recomputes the embedding from the now-merged
`master_person` / `master_company` row and re-applies any dynamic labels (e.g. `:Angel`, `:C_Suite`).

## Idempotency & safety guarantees

1. **Idempotent**: re-running `merge/person-v2` with the same `(survivor_id, duplicate_ids)` after a
   successful merge is a no-op ŌĆö the duplicate IDs no longer exist, so all `WHERE id IN duplicate_ids`
   queries match zero rows. The orchestrator should detect this and return `{status: 'already_merged'}`
   instead of failing.
2. **Atomic per-table**: each table's repoint+dedup runs inside a single transaction. The whole merge
   is *not* one mega-transaction (Xano can't hold one open across 80+ tables) ŌĆö instead, the
   `merge_audit` snapshot is the rollback point. A `merge/rollback` function uses the snapshot to
   un-merge.
3. **Fail-loud**: any FK constraint violation or unexpected row count writes a `log_crash` row and
   sets `merge_audit.status = 'failed'`. No silent partial merges.
4. **Survivor lock**: while a merge is in flight, the survivor + duplicate IDs are written to a
   `merge_lock` table. Other writers (enrichment crons, user actions) check this table and either
   wait or skip.

## Conflict resolution (decided)

Scalar field conflicts (survivor and a duplicate both have non-null values that disagree) split
into two tiers:

### High-stakes fields ŌĆö surface to the user when interactive

- `headline`, `bio_500`, `bio_2000`, `tagline`, `summary_500`, `summary_2000`
- `name`, `first_name`, `middle_name`, `last_name`, `suffix`, `nickname`
- `legal_name` (company)
- `domain`, `linkedin_url`, `twitter_url`, `website_url` (company)

When a **user-initiated** merge hits a conflict on any of these, the merge orchestrator pauses
and surfaces a side-by-side conflict-resolution UI. The user picks per field, or accepts the
"longest non-null" auto-pick default. Then the merge proceeds with the user's choices.

### Everything else ŌĆö auto-pick silently

All other scalar fields auto-pick by the rules in the [scalar aggregation tables](#scalar-field-aggregation-rules-master-person-table-139)
without surfacing anything.

### Cron-initiated merges ŌĆö never block on a human

Cron paths (deterministic exact-match via `mvp/merge/master-merge` #12736, future scheduled
re-merges of high-confidence suspects) auto-pick high-stakes fields too, AND write each
overridden field to `merge_audit.conflicts_resolved[]`:

```json
{
  "conflicts_resolved": [
    {
      "field": "headline",
      "winner_value": "Founder & CEO at Acme",
      "winner_source_master_id": 12345,
      "loser_values": ["Co-founder, Acme Inc."],
      "loser_source_master_ids": [67890],
      "rule_applied": "longest_non_null"
    }
  ]
}
```

A "review recent auto-merges" feed reads from `merge_audit` where `conflicts_resolved` is
non-empty, letting the user override after the fact (the override writes a follow-up
`merge_field_override` row but does NOT re-run the full merge).

### Confidence escape hatch ŌĆö deterministic matches skip even the logging

When the merge was triggered by a deterministic exact-match (same email or normalized phone),
the records are by definition the same person. Conflicts on scalar fields are just stale data
from one source ŌĆö auto-pick "longest non-null" without writing to `conflicts_resolved`. This
keeps the after-the-fact review feed focused on actually ambiguous merges.

The trigger source is recorded in `merge_audit.initiated_by` plus a new
`merge_audit.match_kind` column with values `'deterministic'` (email/phone exact-match),
`'llm_confirmed'` (Tier 1/Tier 2 from `mvp/merge/detect-suspects` #12709), or
`'user_picked'` (manual merge from the suspect UI).

## Suspect lifecycle after merge (decided)

When a merge completes, the orchestrator finalizes `merge_suspect` (284) in three steps:

1. **Repoint** ŌĆö `UPDATE merge_suspect SET master_person_id = survivor_id WHERE master_person_id IN duplicate_ids` and same for `merge_with_master_person_id` (or the company-side equivalent FKs once `kind` is added).
2. **Delete collapsed pairs** ŌĆö after repoint, delete any row where both sides now equal `survivor_id`. The pair is meaningless (the suspect was for two records that are now one).
3. **Flag for re-score** ŌĆö on every remaining row touching the survivor, set `needs_rescore=true`. A cron (new or extension to `mvp/merge/detect-suspects` #12709) picks up flagged rows and re-runs Tier 1 / Tier 2 scoring just for the affected pairs. The new score writes back to the same row and clears the flag.

UI behavior while `needs_rescore=true`: hide the row from the active suspect feed, or show with a "re-evaluating" tag. Don't let users action a stale-score suspect.

**Why deferred re-score, not synchronous**: scoring at merge time would burn Gemini Flash + Pro tokens on every merge ŌĆö most of which are routine deterministic matches that don't need their downstream suspects re-evaluated. Deferring lets the cron batch them.
