<Note>
  **Draft / working classification.** Derived by tracing the live Xano function stacks
  (workspace 3): the four enrichment pipelines for the **Universe** side, and the
  card-serving endpoints (`get-person-details_v2`, `get-company-details-v5`,
  `get-person-buttons`, financial / social / news / view functions) for the **App** side.
  IDs are live Xano table IDs. Validate before treating as the migration contract — this
  refines the [ownership boundary]( /guides/open-work/orbiter-univers-standalone#ownership-boundary)
  on the parent page into concrete table lists.
</Note>

<Tip>
  **Empirical finding that shapes this page:** the card detail endpoints have almost **no
  direct SQL reads** — `get-person-details_v2` / `get-company-details-v5` / `get-person-buttons`
  assemble cards from the **FalkorDB graph** (`falkor/send-cypher`) plus thin helper functions.
  So the App backend needs far fewer SQL tables than the enrichment side writes. Most person/company
  *facts* (honors, awards, publications, expertise, skills) reach a card **through the graph**, not
  by reading the source-of-truth SQL table directly.
</Tip>

## Tables SHARED in both Orbiter Universe & Orbiter Backend

Tables used on **both** sides of the boundary. Three kinds — identity masters, entity projections, and display facts. 25 tables.

**Product master / identity records** — the canonical entity + contact records. The FalkorDB graph
nodes are built **directly from them** (verified: `add-person-node` #2601 reads `master_person`,
`add-company-node` #2602 reads `master_company`; across all graph writers `master_person`, `master_company`,
`master_email`, `master_link`, `master_address`, `master_avatar`, `master_phone` feed node properties and edges),
and the App reads them for cards. *Target nuance ([decision #1](/guides/open-work/orbiter-univers-standalone#working-decisions)):*
the Universe keeps its own canonical entity keyed by `universe_entity_uuid` and the App upserts `master_*` from
Universe packages — so the identity lives on both sides, bridged by `universe_*_uuid`, rather than one shared row.
`master_person`(139) · `master_company`(142) · `master_link`(166) · `master_email`(155) · `master_phone`(151) · `master_address`(164) · `master_avatar`(227) · `master_banner`(228)

**Entity source-of-truth ↔ card projection** — the same table in **both** backends ([decision #2](/guides/open-work/orbiter-univers-standalone#working-decisions)).
Universe side = enrichment source-of-truth (raw scrape/LLM payloads, provider natural keys like IMDB URL / MBID,
freshness gates, cascade depth, graph node UUIDs, credit/award replay state); App side = card read-model
projection (display-safe fields, images, summaries, `universe_*_uuid`). Sync is one-way (Universe → App); product
edits never mutate the Universe source row. In greenfield AlloyDB the App side takes `*_card` names
(`imdb_person_card`, `film_tv_card`, `music_entity_card`).

*IMDB / Film & TV* — `imdb_person`(394) · `film_tv`(395) · `film_tv_award`(396)

*MusicBrainz* — `mb_artist`(737) · `mb_recording`(738) · `mb_label`(739) · `mb_event`(740) · `mb_release_group`(741) · `mb_release`(744) · `mb_work`(742) · `mb_place`(743)

**Display facts (enrichment-written ↔ card-read & side panel read)** — populated by the enrichment waterfalls and
surfaced in the App's cards / person side panel. `work_experience`, `skills`, `company_financial`, `primary_location`
are confirmed read directly by a card endpoint (full-corpus audit: ← get-person/company-details,
get-company-financial-data). `role`/`role_join` (current title & positions) drive the **person side panel** — sourced
via the FalkorDB person node that `role_join` feeds (`update-person-node` #2758), rather than a direct SQL read.
`work_experience`(147) · `skills`(130) · `company_financial`(287) · `primary_location`(385) · `role`(179) · `role_join`(329)

## Tables for Card Views ONLY in Orbiter Backend

Written and read **only** by the product — never touched by enrichment, no Universe copy. 12 tables.

`user`(125) · `contacts`(128) · `my_company`(149) · `my_note`(156) · `my_person_email_join`(312) · `collection`(347) · `orbiter_connect_request`(172) · `orbiter_connection_join`(257) · `linkedin_connection_join`(258) · `calendar`(223) · `person_view`(403) · `company_view`(402)

<Warning>
  **Three-way split.** *Tables ONLY on Orbiter Universe* have no App copy; *Tables for Card Views ONLY
  in Orbiter Backend* have no Universe copy; the **Shared** tables live on both sides — masters and
  display-facts as enrichment-written ↔ product-read, and the entity tables as a one-way projection
  (Universe → App) — see the [projection pattern](/guides/open-work/orbiter-univers-standalone#source-tables-vs-card-tables).
  Note `master_*` are classed App-owned ([decision #1](/guides/open-work/orbiter-univers-standalone#working-decisions)):
  enrichment populates them today, but the target has the App backend upsert them from Universe packages.
</Warning>
