### Entry points

```text

```

Gate for any IMDB person URL. Normalizes URL, dedups on a 30-day window, ensures stub, then either sync-enriches (depth 0 + `queue: false`) or upserts the queue row. **Primary entry point for new work.**

```text
mvp/imdb/get-add/imdb-title #12874
```

Gate for any IMDB title URL. Same shape as the person gate ŌĆö stub, dedup, sync-call branch, queue upsert.

```text
mvp/imdb/add-imdb-to-master-person #10509
```

Attaches IMDB data to an **existing** `master_person_id`. Preserves inline scrape + LLM + `db.add imdb_person` because the caller owns the `master_person` link. Credits loop delegates to `get-add/imdb-title` (queued, T2, depth+1). Called from `process-person-phase-8` **#12829**.

Reverse lookup ŌĆö given a `master_person_id`, search Tavily + disambiguate with LLM, return verified IMDB URL. Called from `process-person-phase-8` **#12829**.

<CardGroup cols={1}>
  <Card title="Fix #1 ŌĆö add-node-edge-imdb-award (#2334) v1.4: JSON.stringify Cypher literals" icon="bug">
    The edge writer produced malformed Cypher for descriptions containing an apostrophe (Gamer's `'Doom'-type environment`). The manual escape chain emitted `\\'` in the Cypher output, which FalkorDB parsed as a literal backslash + terminating quote, silently returning 0 rows. v1.4 switches to `JSON.stringify` for double-quoted Cypher literals ŌĆö no escape-level ambiguity.

    **Symptom:** 2 of 6 `film_tv_award` rows had `node_uuid = null` despite the orchestrator completing clean.
    **Fix:** replace the `esc` helper with `const jstr = v => JSON.stringify(String(v).replace(/[\x00-\x1f\x7f]/g, " "))`. Templates drop the surrounding `'...'` wrappers ŌĆö JSON provides the quotes.
  </Card>

  <Card title="Fix #2 ŌĆö cascade-awards-from-title (#12878) v1.1: filter-chain hoisting" icon="bug">

    **Pattern:** any time a filter-chain result is consumed by another filter chain, hoist the intermediate result into a `var $...` first.
  </Card>

### Implementation notes captured during build

- **URL normalization lives in get-add:** `trim | split:"/?" | first | rtrim:"/"`. All queue rows and `imdb_person` / `film_tv` rows are keyed on the normalized URL.
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` ŌĆö NOT `function.call` (not valid in Xano).
- **`updateTableSchema` drops text `validators`:** the Xano MCP `updateTableSchema` tool silently strips `filters=trim` from text fields when replacing schema. Low impact when inputs are already trimmed upstream, but worth knowing.
- **Queue upsert is the single source of truth:** `get-add` always routes through `upsert-imdb-person` / `upsert-imdb-title`. No raw `db.add` into queue tables anywhere.
- **Kill switch is fail-open:** missing env var ŌåÆ `try_catch` logs to `log_crash` and defaults OFF.
- **`cascade_depth` + `queue` param on get-add:** when both `cascade_depth == 0` AND `queue == false`, `get-add` invokes the orchestrator synchronously after the stub exists and returns `action: "enriched_direct"` ŌĆö no queue entry written.
- **Orchestrators are 7-phase with per-phase `try_catch` ŌåÆ `log_crash`:** each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` + stack trace.
- **Phase 1 guard:** if scrape+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early ŌĆö no `master_person` gets created from a junk row.
- **Cascade helpers are pure dispatchers:** every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"`. Every `function.run` to `get-add` is wrapped in `try_catch` ŌåÆ `log_crash`.
- **Seed ID propagation:** a depth-0 seed invocation via `get-add` must set `seed_imdb_person_id` / `seed_film_tv_id` to the just-created stub's own id when no explicit seed is passed. Required for loop-prevention to work.
- **Top-N sync credits on person seeds (`sync_top_credits: 5`):** `cascade-credits-from-person` v1.1 sync-enriches the top 5 credits inline when the person is at `cascade_depth == 0`. Guarantees a seed returns with enriched films even before the title cron runs.
- **Cron single-row tick pattern:** modeled on `process-person-queue` **#126** / `process-company-queue` **#127**. Lock-on-pick, `try_catch` around orchestrator, release lock on failure (the company/person equivalents leave rows permanently locked on crash ŌĆö the IMDB crons fix this).
- **Missing-stub guard:** if the queue row's `imdb_person_id` / `film_tv_id` is 0, write `log_crash` and `db.del` the queue row ŌĆö prevents a bad row from monopolizing the lock forever.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **shared_with** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person + company waterfall solves this for LinkedIn/PDL/Enrich Layer. This plan applies the same pattern ŌĆö **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** ŌĆö to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits).
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) + top 3 cast.

## Depth model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
|:-----:|---------|----------|
| **0** | Seed (user-initiated) | Full scrape + LLM extract + awards + cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` + name/title, no scrape). Queued. Cron processor scrapes + expands, but **only T1ŌĆōT2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes + writes data, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits + 9 acting + 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director + writer + DP + lead producer + 3 cast = ~7 people ŌåÆ ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000+.
</Check>

### Credit filter for person seed (cascade-credits-from-person)

```text
for each credit in imdb_person.json.credits:
    if role Ōłł {director, writer}:                                ŌåÆ T1
    elif role = producer AND (rating Ōēź 7 OR year = current_year): ŌåÆ T2
    elif role Ōłł {executive producer, cinematographer, minor}
         OR rating Ōēź 7:                                          ŌåÆ T3
    elif type Ōłł {Video Game, Music Video}
         OR (type = Short AND rating < 8):                       ŌåÆ T4

    # T3 capped at top 20 by rating to prevent low-signal blowout
```

### Key crew filter for title seed (cascade-crew-from-title)

```text
from film_tv.json:
    directors[]                                      ŌåÆ T1
    writers[]                                        ŌåÆ T1
    cinematographer (if present in json)             ŌåÆ T2
```

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

### Depth-1 queued processing

When the cron picks up a queue entry, it sets `processing: true` and calls the same orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P6) only queue T1ŌĆōT2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured ŌĆö but no children are queued.

## Loop prevention (five layers)

1. **Dedup gate in `get-add`:** `imdb_url` already in `imdb_person` / `film_tv` with `enriched_at` within 30 days ŌåÆ return existing, no scrape.
2. **Queue upsert:** duplicate `imdb_url` in queue ŌåÆ `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth Ōēź 1:** only T1ŌĆōT2 discoveries are queued at depth 2 (T3ŌĆōT4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.

## Historical reference

<Accordion title="Pre-waterfall function graph + loop risk (before 2026-04-22)">
The waterfall replaces a set of interconnected scrape functions whose loop prevention relied on three `!`-prefix-disabled `foreach` blocks ŌĆö brittle because one accidental edit re-enabled infinite recursion.

| # | Function | Role (pre-waterfall) |

| #10509 | `add-imdb-to-master-person` | Attach IMDB data to existing `master_person`. Tail-looped credits ŌåÆ `add-title-imdb` |
| #10507 | `add-title-imdb` | Scrape title ŌåÆ create `film_tv`. Had writer/director cascade (`!`-disabled) |


<Accordion title="Step 7 refactor details (2026-04-22)">
- `create-master-person-from-imdb` **#10506** and `add-title-imdb` **#10507** became **thin shims**: call `get-add/imdb-{person,title}` with `cascade_depth: 0, queue: false` (hits the sync-call branch), then `db.get` the resulting row and return it. - `add-imdb-to-master-person` **#10509** is a **targeted patch**, not a shim: preserves the inline scrape + LLM + `db.add imdb_person` flow because the orchestrator always creates a fresh `master_person`, and this function's contract is linking `imdb_person` to an **existing** `master_person_id`.
