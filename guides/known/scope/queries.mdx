Query helpers are pre-built Cypher query functions exposed to the frontend and to other Orbiter services. They wrap the raw query patterns in [Documentation — Query patterns](/guides/known/documentation#query-patterns) with input validation, pagination, and consistent output shapes.

---

## `known/query/get_current_relationship_summary`

Returns a flat summary for a single pair: the `KNOWS` edge plus all currently-valid facets. Used by the relationship detail UI.

**Inputs:** `person_a_uuid`, `person_b_uuid`
**Outputs:**
```json
{
  "knows": {"strength_score": 78, "closeness_tier": "close", ...},
  "facets": {
    "romantic": {"type": "married", "valid_from": "..."},
    "bonded": {"bond_type": ["spouse", "co_parent"], ...},
    "kinship": null,
    "professional_tie": null,
    "estranged": null,
    "mentor": null
  },
  "introductions_into_this_relationship": [...]
}
```

## `known/query/get_relationship_history`

Returns all facets ever existing between a pair, including invalidated ones, ordered by time. Used by the relationship history timeline UI.

**Inputs:** `person_a_uuid`, `person_b_uuid`
**Outputs:** array of facet rows with `facet_type`, `valid_from`, `valid_to`, `invalidation_reason`, and facet-specific properties

## `known/query/get_person_circle`

For a given person, returns all currently-valid `KNOWS` edges with their facets and strength scores. Supports filtering.

**Inputs:**
- `person_uuid`
- `filter.closeness_tier?` — e.g. `['close', 'inner_circle']`
- `filter.facet_types?` — e.g. `['BONDED', 'KINSHIP']`
- `filter.context?` — e.g. `'professional'`
- `pagination` — standard
- `sort_by` — `strength_score` (default), `last_direct_interaction_at`, `valid_from`

**Outputs:** paginated list of relationships

## `known/query/get_mutual_connections`

Given two people, returns the set of people both have `KNOWS` edges to. Used for "who do we both know" UIs.

**Inputs:** `person_a_uuid`, `person_b_uuid`, `pagination`
**Outputs:** paginated list of mutual persons with both `KNOWS` edges

## `known/query/get_shortest_path`

Given two people, returns the shortest path through the graph. Prefers introduction-backed hops: when two candidate paths are the same length, the one traversing the most `Introduction` nodes (via `MADE_INTRODUCTION` / `CONNECTS`) wins, because introductions are semantically stronger connectors than co-incidental shared context.

**Inputs:** `from_uuid`, `to_uuid`, `max_hops` (default 6)
**Outputs:**
```json
{
  "path": [
    {"person": {...}, "edge": {"type": "KNOWS", "strength_score": 82}, "introduction": {...}},
    ...
  ],
  "length": 3,
  "introductions_along_path": 2
}
```

## `known/query/get_super_connectors`

Given a person, returns the top N people who have introduced them to the most others (via `MADE_INTRODUCTION` → `Introduction` → `CONNECTS`). Used for "who are my best introducers" queries.

**Inputs:** `person_uuid`, `limit` (default 10)
**Outputs:** ranked list of introducers with introduction counts

## `known/query/get_household_history`

Given a person, returns all `Household` nodes they have `RESIDED_AT` and who else was at each household during their residency window. Used for "show me everyone I've lived with" queries.

**Inputs:** `person_uuid`
**Outputs:** list of households with residency windows and co-resident lists

## `known/query/get_merge_history_for_canonical`

Given a canonical person uuid, returns all `MergeEvent` nodes where this person was the target (absorbed other persons into this one). Used for audit/debugging and for "how many duplicates has this person absorbed" queries.

**Inputs:** `person_uuid`
**Outputs:** array of `MergeEvent` records with `absorbed_person_id`, `merged_at`, `initiated_by`, `confidence`, `rationale`, ordered by `merged_at` descending.

```cypher
MATCH (merge:MergeEvent)-[:CANONICAL_NOW]->(p:Person {uuid: $person_uuid})
OPTIONAL MATCH (merge)-[:ABSORBED]->(absorbed:Person)
RETURN merge, absorbed
ORDER BY merge.merged_at DESC
```

## `known/query/get_merge_count_for_canonical`

Given a canonical person uuid, returns the count of merges that have resulted in this person as the canonical. Cheap existence query for UI badges or audit summaries.

**Inputs:** `person_uuid`
**Outputs:** `{ merge_count: int }`

```cypher
MATCH (merge:MergeEvent)-[:CANONICAL_NOW]->(p:Person {uuid: $person_uuid})
RETURN count(merge) AS merge_count
```

## Design guidelines

- **All query helpers are read-only.** They should use `GRAPH.RO_QUERY` in FalkorDB to reject any accidental writes.
- **Always paginate.** Every list-returning query helper MUST support pagination. No unbounded result sets.
- **Consistent output shapes.** `KNOWS` returns use the same field set; facet returns use the same field set. The UI should be able to render any query result without special-casing the query.
- **Filter at the query level, not in application code.** Push `closeness_tier` filters, date ranges, and facet-type filters into Cypher `WHERE` clauses to use indexes.
- **Cache aggressively at the application layer.** Many of these queries are hot for individual users (their own circle, their own history). Cache with invalidation on any `KNOWS`/facet change touching the user.
