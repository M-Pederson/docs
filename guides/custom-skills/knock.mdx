## Core Concepts

### Workflows

A **workflow** is the primary unit of notification-as-code in Knock. It's
identified by a unique \`key\` and contains:

- **Trigger conditions** — when should this workflow fire? (e.g., "user signs up",
  "billing alert")
- **Steps** — sequences of actions (send email, send SMS, wait, branch logic)
- **Channels** — which notification channel each step uses (email, in-app, push, etc.)
- **Templates** — the content for each channel (subject, body, rich HTML, etc.)
- **Tenant context** — which tenant/workspace owns this workflow (multi-tenant support)

Workflows are defined as JSON files on disk (when using workflow-as-code via the CLI).

### Environments

Knock uses a three-tier environment structure:

- \`development\` — your sandbox. Safe for experimentation.
- \`staging\` — pre-production. Resembles production closely.
- \`production\` — live. Real users, real notifications.

**Flow:** edits happen in \`development\`, then \`commit\` to make them promotable,
then \`commit promote\` to flow \`development → staging → production\`. You never
push directly to \`production\`.

### Channels

A **channel** is a configured notification provider:

- Email (SMTP, SendGrid, AWS SES, etc.)
- SMS (Twilio, etc.)
- Push (Firebase Cloud Messaging, Apple Push, etc.)
- In-app (Knock's hosted UI)
- Slack (via OAuth or webhook)
- Chat (Discord, Teams, etc.)

Channels are configured once in the Knock dashboard and referenced by workflow
steps.

### Recipients & Users

A **recipient** is anyone who receives a notification. In Knock terminology:

- Recipient = any entity (user, account, device) you want to notify
- Users are a special case of recipients with identity (email, phone, external ID)
- User properties (preferences, do-not-disturb, timezone) are stored by Knock and
  used to personalize delivery

### Tenants (Multi-Tenancy)

A **tenant** is an isolated workspace within Knock where you scope:

- User preferences and subscriptions
- Branding (sender name, colors, logo)
- Channel routing (which channels are enabled for this tenant)

Useful for SaaS products where each customer is a tenant with their own
notification preferences.

## Authentication & Tokens

Knock uses three token types. Always use the least-privileged token for your use case.

### \`pk_*\` — Publishable Keys

**Scope:** Client-side only. Identify account but no sensitive actions.

**Use case:** embedding Knock's in-app notification UI in a browser. Safe to
expose in frontend code because it only lets you read subscription preferences.

**Example:**
\`\`\`javascript
import { Knock } from '@knocklabs/client'

const client = new Knock('pk_12345abc...')
\`\`\`

### \`sk_*\` — Secret Keys

**Scope:** Server-side API calls. Can trigger workflows, upsert users, manage
subscriptions. DO NOT commit or expose.

**Use case:** triggering workflows from your backend, updating user properties.

**Storage:** Use environment variables. Never hardcode. Example:

\`\`\`bash
export KNOCK_SECRET_KEY='sk_...'
\`\`\`

### \`kst_*\` — Service Tokens

**Scope:** CLI, Management API, and agent-toolkit (MCP server). Full account
access unless scoped via permissions.

**Use case:** the Knock CLI (\`knock workflow pull\`, \`knock push\`), the
agent-toolkit MCP server, CI/CD pipelines.

**Storage:** Use \`~/.config/knock/config.json\` to avoid shell history:

\`\`\`json
{
  "service_token": "kst_..."
}
\`\`\`

Then reference it:

\`\`\`bash
knock workflow pull --config ~/.config/knock/config.json
\`\`\`

## Workflow-as-Code: CLI & File Layout

### Knock CLI Commands

#### \`knock workflow pull\`

Download all workflows from a Knock account to your local disk.

\`\`\`bash
knock workflow pull --config ~/.config/knock/config.json
\`\`\`

Creates a directory structure:
\`\`\`
workflows/
├── my-workflow-1.json
├── my-workflow-2.json
├── notify-on-signup/
│   ├── workflow.json
│   ├── email/
│   │   ├── digest.json (if multi-step)
│   │   ├── template.html
│   │   └── subject.txt
│   ├── sms/
│   │   └── template.txt
│   └── in-app/
│       └── template.json
\`\`\`

#### \`knock workflow push\`

Push local workflow files back to Knock. This uploads to \`development\` only.

\`\`\`bash
knock workflow push my-workflow-1.json
\`\`\`

#### \`knock workflow validate\`

Dry-run a workflow without actually pushing it.

\`\`\`bash
knock workflow validate my-workflow-1.json
\`\`\`

#### \`knock workflow run\`

Trigger a workflow for testing (emits a test notification).

\`\`\`bash
knock workflow run my-workflow-1 --recipient user@example.com
\`\`\`

#### \`knock commit\`

Commit changes in \`development\` so they become eligible for promotion.

\`\`\`bash
knock commit -m "Add SMS step to signup workflow"
\`\`\`

#### \`knock commit promote\`

Promote a committed change from \`development → staging → production\`.

\`\`\`bash
knock commit promote
\`\`\`

This is interactive: choose which environment to promote to. Never push directly
to \`production\`; always flow through \`staging\` first.

### Workflow File Structure

A workflow JSON file looks like:

\`\`\`json
{
  "key": "notify-on-signup",
  "name": "Notify on Signup",
  "description": "Send welcome email and SMS on user signup",
  "triggers": [
    {
      "type": "event",
      "event_key": "user.signup"
    }
  ],
  "steps": [
    {
      "id": "send-email",
      "channel_id": "email",
      "template": "welcome-email",
      "conditions": []
    },
    {
      "id": "send-sms",
      "channel_id": "sms",
      "template": "welcome-sms",
      "conditions": [
        {
          "operator": "not_empty",
          "argument": "phone_number"
        }
      ]
    }
  ]
}
\`\`\`

**Key fields:**
- \`key\` — unique identifier, used when triggering from code
- \`triggers\` — what events or conditions start this workflow
- \`steps\` — ordered list of notification actions
- \`conditions\` — per-step filters (only send SMS if phone_number is populated)

### Template Files

Templates live in per-channel directories. Example:

**email/subject.txt:**
\`\`\`text
Welcome, {{firstName}}!
\`\`\`

**email/template.html:**
\`\`\`html
<h1>Hello, {{firstName}}!</h1>
<p>Thanks for joining us.</p>
\`\`\`

Templates use Liquid-style substitution: \`{{fieldName}}\` is replaced with data
passed when triggering the workflow.

## @knocklabs/agent-toolkit Integration

The \`@knocklabs/agent-toolkit\` is a pre-built library for AI agents to
interact with Knock workflows.

### As an MCP Server

Run the toolkit as an MCP server that Claude Code can call:

\`\`\`bash
npx -y @knocklabs/agent-toolkit -p local-mcp \\
  --service-token kst_... \\
  --tools get_workflow,list_workflows,run_workflow \\
  --workflows notify-on-signup,notify-on-billing
\`\`\`

This exposes Knock tools to your Claude Code session. Flags:
- \`--tools\` — which tools the agent can use (read/run only, not write)
- \`--workflows\` — filter to specific workflows

**Permission model:** always start read-only and widen deliberately. Never give an
agent \`manage\` scope on production.

### As a Vercel AI SDK Tool Set

If you're building an AI app with Vercel AI SDK:

\`\`\`javascript
import { createKnockToolkit } from '@knocklabs/vercel-ai'

const tools = await createKnockToolkit({
  serviceToken: 'kst_...',
  permissions: {
    workflows: 'read',  // can list/get workflows
    recipients: 'run',  // can trigger workflows for recipients
  }
})
\`\`\`

Then pass \`tools\` to your \`generateText()\` or \`streamText()\` call.

### Scoping Permissions

Always specify permissions explicitly:

\`\`\`javascript
{
  workflows: 'read',     // list, get (no write)
  recipients: 'run',     // trigger workflows, upsert users
  subscriptions: 'read', // read preferences (no change)
  // Never: 'manage' on production
}
\`\`\`

**Permission levels:**
- \`read\` — list and get; no modifications
- \`run\` — trigger workflows, send notifications
- \`write\` — upsert users, update preferences (rarely needed for agents)
- \`manage\` — delete, archive, full control (only for trusted admin flows)

## Triggering Workflows from Server Code

Use \`@knocklabs/node\` to trigger workflows from your backend.

### Basic Workflow Trigger

\`\`\`javascript
import { Knock } from '@knocklabs/node'

const client = new Knock('sk_...') // secret key

await client.workflows.trigger('notify-on-signup', {
  recipients: ['user@example.com'],
  data: {
    firstName: 'Alice',
    lastName: 'Smith',
    signupDate: '2024-01-15'
  }
})
\`\`\`

### Data Becomes Template Context

The \`data\` object is passed to workflow templates. In the template:

\`\`\`html
<h1>Welcome, {{firstName}} {{lastName}}!</h1>
<p>You signed up on {{signupDate}}.</p>
\`\`\`

**Keep payloads small and stable:** Knock persists \`data\` for retry logic. Large
or frequently-changing data structures can inflate storage and cause template
mismatches on retry.

### Multiple Recipients

\`\`\`javascript
await client.workflows.trigger('notify-on-alert', {
  recipients: [
    { id: 'user-1', email: 'alice@example.com' },
    { id: 'user-2', email: 'bob@example.com' }
  ],
  data: { alertLevel: 'critical' }
})
\`\`\`

## Debugging Checklist

### "Workflow not found" Error

**Cause:** You triggered a workflow key that doesn't exist in the current
environment.

**Fix:**
1. Verify the workflow key is spelled correctly
2. Confirm the workflow is deployed to the environment you're targeting
   (\`development\` vs \`staging\` vs \`production\`)
3. Use \`knock workflow run\` to manually test the workflow first

### Template Variable Is Undefined

**Cause:** Your template references \`{{foo}}\` but the \`data\` object passed
to trigger didn't include \`foo\`.

**Fix:**
1. Log the \`data\` object before calling \`client.workflows.trigger()\`
2. Verify all template variables are in the data payload
3. Use Liquid conditionals for optional fields: \`{% if foo %}{{foo}}{% endif %}\`

### Permission-Scope Write Error (Agent-Toolkit)

**Cause:** The agent tried to update a workflow or user but only has \`read\`
permissions.

**Fix:**
1. Widen the permission scope: change \`workflows: 'read'\` to \`workflows:
   'write'\` if agents need to update workflows
2. Or narrow the agent's instructions to only read operations
3. Never grant \`manage\` scope unless absolutely necessary

### Commit Promote Is a No-Op

**Cause:** You pushed a workflow to \`development\`, but you forgot to \`commit\`.

**Fix:**
\`\`\`bash
knock workflow push my-workflow.json
knock commit -m "Updated my-workflow"
knock commit promote  # NOW it works
\`\`\`

### Duplicate Notifications Sent

**Cause:** Batch or throttle settings on the workflow are too lenient, or the
workflow is being triggered multiple times.

**Fix:**
1. Check the workflow's batch/throttle settings in the Knock UI
2. Verify you're not triggering the same workflow multiple times in a loop
3. Use \`recipient_id\` + timestamp deduplication if applicable
