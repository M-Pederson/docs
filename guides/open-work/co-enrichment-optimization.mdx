## Why this exists

Roughly 14% of `fundable_people` (72,174 of 528,045) are linked to ≥1 deal. On the org side, many investor records arrive nearly empty: a name and not much else — no `domain`, no `linkedin_url`, no `crunchbase_url`, no `website`. These records are unusable for downstream graph merging, profile rendering, or warm-intro discovery until we resolve them to a real-world entity.

This page tracks the work to fill that gap with a small, focused LLM enrichment loop.

## LLM Identity Enrichment — design

### Goal

Given a sparse `fundable_organizations` record, return any of:

- **`domain`** — primary website domain (no protocol)
- **`website`** — full URL
- **`linkedin_url`** — LinkedIn company page
- **`crunchbase_url`** — Crunchbase organization page
- **`twitter`** / **`facebook`** — social handles or URLs
- **`canonical_name`** — official name if it differs from `name` (e.g. romanization, full legal name)
- **`country_code`** / **`region`** if missing

Plus a **confidence score** (0–1) and a **reasoning** string the human reviewer can audit.

### Input shape — ONE YAML field, not many

**Recommendation: a single input `fundable_data` (YAML) with null/empty fields stripped, plus a separate `deal_context` input for surrounding signals.**

Why one consolidated YAML beats individual inputs:

1. **Sparse rows.** A typical sparse record has ~3 non-null fields out of ~30. Individual inputs would ship 27 explicit `null`s to the model every call — token waste, noise.
2. **Variable schema is free.** YAML lets each call carry whatever's actually present without a fixed contract.
3. **Token cost.** Directly impacts per-call price. For the sparse-record case, the YAML prompt is roughly 4× smaller than the equivalent labeled-field prompt.
4. **Model attention.** The LLM does better when it sees only the signal. Fewer distractor "null" lines = fewer pattern-match misfires.
5. **Easier prompt evolution.** Adding a new field to the org table doesn't require updating the function signature.

The org-level YAML alone is often not enough though — for the China/Jiazhifund case the **deal descriptions** were the data point that surfaced the romanized name. So:

- Input 1: **`fundable_data`** (YAML) — non-null fields from the org row
- Input 2: **`deal_context`** (YAML) — list of deals this org participated in, each with portfolio company name, financing type, size, deal description, source article URLs

### Building `deal_context` — the SQL recipe

A useful `deal_context` entry combines fields from **four tables**:

| Source table | Fields used | Why |
|---|---|---|
| `fundable_institutional_investments` | `lead_investor`, `personnel`, `fund` | Org's role on this deal. `fund` sometimes carries the actual fund name (e.g. "Sequoia Capital Growth Fund III"). |
| `fundable_deals` | `financing_type`, `size_usd`, `date`, `short_description`, `long_description`, `canonical_deal_id` | Descriptions are where romanizations / alt names / sector hints live (the Jiazhifund case, the Capital Venture Investment Fund case). |
| `fundable_organizations` (joined on `deals.organization_id`) | `name`, `country_code` | **Portco name** — 5 known portcos = a near-unique fingerprint. |
| `fundable_articles` (joined on `canonical_deal_id`) | `link`, `date`, `source_priority` | Web-fetchable URLs. Critical for any model with browse access. |

**Cap at 8 deals**, prioritized by:

1. Deals where the org was `lead_investor` (descriptions are far more likely to name them by canonical / transliterated form when they're the lead).
2. Longest `short_description` (more text = more chance of an alt-name surfacing).
3. Most recent first.

**Cap at 5 articles per deal**, ordered by `source_priority DESC, date DESC`. Some deals (esp. high-profile US Series A/B) have 16+ articles in `fundable_articles`; shipping all of them is pure token waste — the top 5 by source quality is plenty.

**Strip nulls and the empty-array sentinels.** `personnel` and `fund` are stored as `["{}"]` when empty — drop those.

The query (Postgres, runnable via `db.direct_query`):

```sql
WITH inv AS (
  SELECT *
  FROM x3_678_fundable_institutional_investments
  WHERE organization_id = ?
)
SELECT COALESCE(jsonb_agg(deal_obj ORDER BY ord), '[]'::jsonb) AS deal_context
FROM (
  SELECT
    jsonb_strip_nulls(jsonb_build_object(
      'portco',            NULLIF(po.name, ''),
      'portco_country',    NULLIF(po.country_code, ''),
      'role',              CASE WHEN inv.lead_investor THEN 'lead_investor' ELSE 'co_investor' END,
      'financing_type',    NULLIF(d.financing_type, ''),
      'size_usd',          NULLIF(d.size_usd::text, '0')::numeric,
      'date',              CASE WHEN d.date > 0 THEN to_char(to_timestamp(d.date / 1000), 'YYYY-MM-DD') END,
      'short_description', NULLIF(d.short_description, ''),
      'long_description',  NULLIF(d.long_description, ''),
      'articles', (
        SELECT jsonb_agg(a.link ORDER BY a.source_priority DESC NULLS LAST, a.date DESC NULLS LAST)
        FROM x3_680_fundable_articles a
        WHERE a.canonical_deal_id = d.canonical_deal_id
          AND a.link IS NOT NULL AND a.link <> ''
      ),
      'fund',      CASE WHEN inv.fund      IS NOT NULL AND inv.fund::text      NOT IN ('null','[]','["{}"]') THEN inv.fund      END,
      'personnel', CASE WHEN inv.personnel IS NOT NULL AND inv.personnel::text NOT IN ('null','[]','["{}"]') THEN inv.personnel END
    )) AS deal_obj,
    ROW_NUMBER() OVER (
      ORDER BY
        CASE WHEN inv.lead_investor THEN 0 ELSE 1 END,
        COALESCE(LENGTH(d.short_description), 0) DESC,
        d.date DESC NULLS LAST
    ) AS ord
  FROM inv
  JOIN x3_671_fundable_deals d ON d.id = inv.deal_id
  LEFT JOIN x3_666_fundable_organizations po ON po.id = d.organization_id
  LIMIT 8
) ranked;
```

The org-level `fundable_data` payload is one extra line:

```sql
SELECT jsonb_strip_nulls(to_jsonb(o.*)) AS fundable_data
FROM x3_666_fundable_organizations o
WHERE o.id = ?;
```

### Xano function — `enrichment/build_org_enrichment_context`

Both queries are wrapped in a reusable Xano function that returns a single payload containing JSON + ready-to-prompt YAML.

| | Value |
|---|---|
| **Function ID** | `12980` |
| **Name** | `enrichment/build_org_enrichment_context` |
| **API endpoint** | `GET https://xh2o-yths-38lt.n7c.xano.io/api:rlSvC4qe/enrichment/build_org_enrichment_context` |
| **API ID** | `8517` (in `Admin Backfill` group) |
| **Auth** | None (public — invoked by enrichment workers) |

**Input**

| Field | Type | Required | Description |
|---|---|---|---|
| `fundable_org_id` | uuid | yes | UUID of the `fundable_organizations` row |

**Output**

| Field | Type | Description |
|---|---|---|
| `fundable_data` | object | Org row, null/empty fields stripped |
| `deal_context` | array | Up to 8 deals, prioritized as described above |
| `fundable_data_yaml` | string | Same as `fundable_data` but pre-serialized to YAML — drop straight into a prompt |
| `deal_context_yaml` | string | Same as `deal_context` but pre-serialized to YAML |
| `deal_count_total` | int | Total deals for this org (so caller knows if context was capped) |
| `deal_count_returned` | int | How many deals are in `deal_context` (≤ 8) |

**Example call**

```bash
curl -sS "https://xh2o-yths-38lt.n7c.xano.io/api:rlSvC4qe/enrichment/build_org_enrichment_context?fundable_org_id=c7ca6f59-f58b-41c7-b461-b1d4ff2df672" \
  -H "X-Data-Source: live" \
  -H "X-Branch: v1" | jq
```

**Headers**

| Header | Required | Value |
|---|---|---|
| `X-Data-Source` | yes | `live` (or `staging`/`sandbox`) |
| `X-Branch` | yes | `v1` |
| `Content-Type` | optional | `application/json` |

**Calling from another XanoScript stack**

```xs
function.run "enrichment/build_org_enrichment_context" {
  input = {
    fundable_org_id: $org_id
  }
} as $context

// $context.fundable_data_yaml   → drop into LLM prompt
// $context.deal_context_yaml    → drop into LLM prompt
// $context.deal_count_total     → log if capped (> 8)
```

**Behavior**

- Throws `notfound` if the `fundable_organizations` row doesn't exist.
- Returns an empty `deal_context: []` if the org has no rows in `fundable_institutional_investments`.
- The 8-deal cap is hard-coded. If you want a different cap, fork the function — don't surface it as input (keeps the LLM prompt size predictable).

### System prompt (v0.2)

Updated based on live testing against real `fundable_organizations` rows. Changes from v0.1:

- **Explicit web-tool directive** (was implicit "if you have web tools").
- **Search strategy section** with specific query patterns to try.
- **Disambiguation guidance** using portco fingerprints — the v0.1 case where searching `Synaire` alone returned a respiratory-device company instead of the PLA-materials Synaire from our deal context.
- **Legacy / consolidated entity handling** — the Pfizer R&D Innovate case, where the canonical entity changed (now Pfizer Ventures) but historical Crunchbase rows still use the old name.
- **Strict JSON output** — no preamble, no markdown fences.
- **Confidence calibration table** — concrete anchors so models don't drift to 0.5 by default.
- **Reasoning depth tier** — high-confidence answers can be terse; low-confidence answers must enumerate evidence.

```text
You are an identity-resolution assistant. Given a sparse record from a venture
capital database, identify the real-world entity behind it and return canonical
identifiers (domain, LinkedIn, Crunchbase, etc.).

You have access to a web search tool and a web fetch tool. Use them — do not
answer from prior knowledge alone.

INPUT
You will receive two YAML payloads:
- `fundable_data`: non-null fields from the database row (organization-level).
  Names may be in any language. Treat the `name` field as a starting point, not
  the truth — it can be a romanization, abbreviation, partial name, or a legacy
  name that has since been consolidated under a different parent entity.
- `deal_context`: up to 8 deals where this organization appears as an investor,
  prioritized by lead-investor role + richer descriptions. Each deal carries
  the portfolio company name, financing type, USD size, date, descriptions,
  and up to 5 source article URLs. Deal descriptions are often the strongest
  signal — they sometimes name the investor in their canonical / transliterated
  / legal form, name co-launching parent entities, or describe fund structure.

SEARCH STRATEGY
1. First, scan `deal_context.*.short_description` and `deal_context.*.long_description`
   for any romanization, alt name, parent entity, fund family, or descriptive
   phrase about the investor. Extract these before searching the web.
2. Web search with the most specific name + a portco name. The combination
   disambiguates common-sounding names (e.g. "Synaire" alone could be a
   medical device company; "Synaire" + "Jiazhifund" + "PLA bio-materials"
   pins down the right company).
3. If you find candidate Crunchbase / LinkedIn / PitchBook / CB Insights pages,
   fetch them to confirm sector + portco overlap with the deal_context.
4. For non-Latin names: search both the original name AND any romanization you
   extracted from the descriptions.
5. For sparse name-only inputs: pivot off the portco set. Investing fingerprint
   (which companies + when + what stage) is often enough to resolve the
   investor when name search fails.

DISAMBIGUATION
When the same name returns multiple plausible entities:
- Cross-check against the deal_context's portcos: which candidate has the most
  portfolio overlap?
- Cross-check the deal date(s) against each candidate's known activity period.
- Cross-check sector signal — investor focus must match the deals' sectors.
- If you still can't disambiguate, populate `alternatives` with the top 2-3
  candidates and set `confidence < 0.6`. Do not coin-flip.

LEGACY / CONSOLIDATED ENTITIES
If the original entity has been renamed, merged, or consolidated:
- `canonical_name` = current canonical entity name
- `canonical_name_alt` = original name from `fundable_data.name` (for audit)
- Note the consolidation in `reasoning`
- Use the current entity's domain/LinkedIn/Crunchbase, not the legacy ones

OUTPUT
Return a single JSON object. No preamble, no markdown fences, no commentary
outside the JSON.

{
  "domain": string | null,
  "website": string | null,
  "linkedin_url": string | null,
  "crunchbase_url": string | null,
  "pitchbook_url": string | null,
  "twitter": string | null,
  "facebook": string | null,
  "canonical_name": string | null,
  "canonical_name_romanized": string | null,
  "canonical_name_alt": string | null,
  "country_code": string | null,        // ISO 3166-1 alpha-2 (e.g. "US", "CN")
  "region": string | null,
  "confidence": number,                  // 0..1, see calibration below
  "reasoning": string,                   // see depth requirements below
  "alternatives": [                      // populate when confidence < 0.85
    {
      "canonical_name": string,
      "domain": string | null,
      "evidence": string,
      "confidence": number
    }
  ]
}

CONFIDENCE CALIBRATION
- 0.95+  : direct hit — the entity has a public web page that confirms name +
           sector + at least one portco from deal_context.
- 0.85-0.94 : strong hit — multiple corroborating sources but one or two fields
           are inferred (e.g. country from address, not explicit).
- 0.60-0.84 : moderate — entity probably identified, some uncertainty (rare-name
           overlap, partial portco match, or consolidation history).
- 0.30-0.59 : weak — name and one signal point to a candidate, but no strong
           confirmation. Populate `alternatives`.
- < 0.30 : no plausible candidate found. Most fields should be null.

REASONING DEPTH
- For confidence ≥ 0.85: 1 sentence, naming the primary source(s).
- For confidence 0.6-0.84: 2-3 sentences listing the corroborating signals
  and any remaining uncertainty.
- For confidence < 0.6: enumerate every signal you considered and explain
  why none was decisive. Future re-runs depend on this reasoning.

RULES
- NEVER invent a domain, LinkedIn, or Crunchbase URL. If you can't verify via
  search or fetch, set the field to null and put the unverified candidate in
  `reasoning`.
- Prefer the entity's own LinkedIn / Crunchbase / website over third-party
  listings (Tracxn, Bloomberg, Premier Alts, goldi.ai, etc.).
- For non-English / non-Latin names, always populate `canonical_name_romanized`.
- A confidence < 0.6 means a human will review before merging — be honest;
  false-positives at high confidence are far worse than false-negatives.
```

### v0.3 addition: RESPONSE FORMATTING block

Appended to the end of the v0.2 prompt. Targets intermittent `ERROR_FATAL` from the Xano response parser by constraining the model's output to ASCII-safe, control-character-free, terse strings:

```text
RESPONSE FORMATTING (reliability constraints — DO NOT VIOLATE)
- The entire response must be valid JSON. No text before, after, or around it. No markdown code fences. No comments.
- All string values must use only printable ASCII (U+0020 to U+007E), with these
  specific exceptions allowed where genuinely needed for accuracy: Latin-extended
  letters with diacritics, and CJK characters in canonical_name /
  canonical_name_romanized fields ONLY.
- NEVER include line separator (U+2028), paragraph separator (U+2029),
  zero-width characters (U+200B-U+200F, U+FEFF), or any control characters
  (U+0000-U+001F, U+007F).
- Reasoning field must be one paragraph, no embedded newlines, no quoted
  multi-line snippets from web sources. Paraphrase only.
- Keep the entire JSON output under 2000 characters total when possible. Prefer
  concise over comprehensive in reasoning.
- Do not paste raw text excerpts from web search results into any string field.
  Summarize in your own words.
- Field values must not contain backticks, triple-quotes, or any markdown-style
  formatting characters.
```

Combined with the retry-on-ERROR_FATAL loop in the orchestrator and the lambda-based `run_llm_resolution`, v0.3 brought the legacy online-mode request shape into a usable range. As of the 2026-06-04 OpenRouter migration, the same function uses plain model IDs plus `openrouter:web_search`. **Hit rate: 8 of 9 tested resolve at confidence ≥ 0.85.** The single soft-fail (Jiazhifund) is genuinely obscure, not a prompt failure. Tier B name-only records resolve about as easily as Tier A — the `fundable_data` row's emptiness is a poor predictor of difficulty; the `long_description` of even *one* deal is usually enough.

**Pattern observation across all 9:** every successful resolution drew on at least one of these signals:

1. A descriptive phrase in `fundable_data.short_description` or `long_description` (Gates Frontier, Eleven Prime, Yuan Yi).
2. The investor named in canonical / romanized form inside a deal `long_description` (Capital Venture Investment Fund, Corecam, Orchid India, Yuan Yi, Jiazhifund).
3. A description phrase about *what kind* of entity it is (Schestowitz "across its brands" → corp-venture arm, not a VC).
4. A founder name (Eleven Prime → Bubba Murarka).
5. The portco fingerprint disambiguating a common name (Yuan Yi vs other "Yuanyi" funds; the failed Synaire-respiratory-vs-PLA case in v0.1 testing).

If `fundable_data` and `deal_context` between them carry **none** of these five signal types, the case is genuinely unresolvable and should land at `confidence < 0.6`. Jiazhifund is the only example we've seen so far.

## Open questions

- **Web tools.** Does the LLM get web fetch access at enrichment time, or only at human-review time? Affects prompt + cost dramatically — without web access the model is doing pure pattern-matching from prior knowledge, which won't catch China-specific funds or smaller regional firms.
- **Deal context cost.** Pulling deal + article rows for every sparse org is cheap (we already paginate by org_id), but bloats the prompt for orgs with 18+ deals (Gates Frontier). Should we cap to the top N most-informative deals, or trim by article-presence?
- **Confidence threshold for auto-merge.** Anything ≥ 0.9 auto-merge with audit trail? 0.7–0.9 queue for human review? < 0.7 mark as `unresolved` and re-try later when more deals accumulate?
- **Model choice.** Haiku 4.5 for first pass (cheap, fast), escalate to Sonnet 4.6 only on ambiguous cases? Or Opus 4.7 for everything since the volume is small (per-org, not per-row)?
- **Failure mode logging.** Where do we store the model's `reasoning` + `alternatives` so we can post-mortem the misses? New `fundable_org_enrichment_attempts` table?
- **Re-run policy.** When new deals land for an unresolved org, is that a trigger to re-enrich? Probably yes — more deal_context = better fingerprint.
