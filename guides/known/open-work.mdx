This page is the canonical queue for everything that isn't a **committed spec** and isn't **live Xano code**. If you find yourself adding a `TODO`, an "open question", a "pending decision", or a "not yet built" note to a Known scope page, **add it here instead** and leave a one-line pointer on the source page.

## What belongs on this page

| Category | Example |
|---|---|
| **Spec-only functions/crons** | `dedupe_knows_edges` — design is frozen, Xano implementation not yet built |
| **Open decisions** | "What's the presence floor value?" — needs a call before we build the scoring cron |
| **Drift and cleanup** | `Entity.name_embedding` running `efRuntime: 10` vs declared 100 |
| **Verification tasks** | Run `CALL db.indexes()` after the fresh-graph bring-up to confirm `efRuntime: 100` |
| **Pending retirements** | `mvp/merge/detect-suspects` (#12709) — keep rewired or replace entirely |

## What does NOT belong here

- **Per-phase build-status tables inside scope sub-pages** (`✅ Built` / `⏳ Pending` per function). Those are reader context at the point where a spec is being read — they answer "is this built yet?" for someone navigating the spec. Leave them inline.
- **Not in Scope (Yet)** product deferrals — `CoFoundingGroup`, `PolyamorousGroup`, graph visualization, etc. Those are architectural decisions that frame the current scope, not work waiting to be done. They stay in [scope → Not in scope (yet)](/guides/known/scope#not-in-scope-yet).
- **Committed architectural decisions** that look like TODOs but aren't. If the workshop decided "we'll use canonicalized MERGE + a dedupe cron instead of a mutex table" — that's a decision, not a TODO. It goes in the spec, not here. Only the *unbuilt cron itself* lives here.

## Discipline for adding new items

When a new TODO appears:

1. **Pick a section below** based on the category table above
2. **Add a row to that section's table** with `Item`, `Context/Trigger`, `Links`, `Status`
3. **If you're editing a spec page that triggered the discovery**, leave a pointer like:
   > *See [Open Work → Drift and cleanup](/guides/known/open-work#drift-and-cleanup) for the `name_embedding` efRuntime follow-up.*
4. **Never leave the full TODO body on both pages.** One source of truth. If context matters, the pointer carries it.

When an item closes, mark it `✅ Done` with the date and a one-line resolution note. Keep resolved items in place for one sprint cycle (useful for retrospectives), then delete. Permanent "decisions" graduate from here into the appropriate spec page — they should never just disappear.

## How this page stays current

- When a spec page gains a new "open question," "pending decision," or "not yet built" flag, **the author adds a row here first** and leaves a pointer on the source page. Never both.
- When an item closes, it moves to `✅ Done` inline with a one-line resolution note and a date. Resolved items stay in place for one sprint, then are deleted. Architectural decisions that arose from resolved items move into the relevant scope/schema-design page.
- When this page crosses ~50 open items, split by category into sub-pages (`open-work/decisions.mdx`, `open-work/drift.mdx`, etc.) and keep this page as an index.
- When reading any Known scope page, **any TODO-like callout you see that isn't on this page is a bug** — either the callout belongs here and wasn't moved, or it's stale and should be deleted.
