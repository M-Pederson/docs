`````markdown
# QA the Sandbox

## Objective

Given a pasted enrichment waterfall entry-point response or smoke-test request fixture, audit the sandbox run it represents. Start from the fixed Orbiter sandbox configuration below instead of rediscovering workspace/profile/branch/table IDs on every run. Inspect only sandbox data/logs, keep polling until the waterfall has stopped changing people/company/queue counts, identify errors, trace records created in queue tables, query the graph around the resolved root node after settlement, and finish with a practical repair report.

## Hard Boundaries

- Use workspace `3`.
- Assume the stable target is Xano profile `orbiter`, workspace `3` (`OrbiterV2`), branch `v1`, and data source/header `sandbox`. Do not spend time revalidating these constants on every run unless a command errors, the user reports a config change, or a needed tool is unavailable.
- Use sandbox data source/environment only. Do not query live/production data unless the user explicitly overrides this boundary.
- Treat the task as read-only. Do not run `xano workspace push`, `xano sandbox push`, `xano sandbox reset`, `xano sandbox delete`, function edits, schema edits, queue mutation, graph writes, or reruns unless the user explicitly asks.
- Before using a broad `xano sandbox ...` command, verify the active/profiled workspace and branch target workspace `3`; sandbox commands inherit profile context. Narrow MCP table/API calls with `x-data-source: sandbox` do not require this repeated CLI preflight.
- If sandbox data access is unavailable from the current tools, stop and report the blocker plus the exact safe access needed. Do not silently fall back to production.
- Do not use Robert Lab APIs for this workflow. The sandbox QA graph runner belongs in the `QA` API group, not `Robert Lab`.

### Optional Deep Bio Boundary

Deep Bio is an optional branch, not part of the core person-enrichment
completion contract. Establish whether the run explicitly enabled Deep Bio
before auditing provider jobs, callbacks, payload validators, or biography
content.

- When Deep Bio is explicitly disabled, require no Deep Bio dispatch, provider
  job, callback, terminal receipt, retry, or wait. Report those surfaces as
  `EXCLUDED`, not `PENDING`, missing, or failed.
- A Deep Bio call or job created while the flag is explicitly false is a
  dispatch-gating regression. It is not evidence that Deep Bio became required.
- Continue to require the complete non-biography waterfall: structured source
  capture, current-company selection, all allowed resolver phases, table/graph
  materialization, UUID/date/tense parity, queue policy, lifecycle cleanup, and
  independent graph readback.
- Core non-bio completion must not depend on a Deep Bio status, receipt, webhook,
  or callback. Do not wait for provider timeout/quiescence when no Deep Bio job
  was launched.
- When Deep Bio is enabled, audit it as a separate optional branch. A failure in
  that branch must not erase or invalidate independently verified non-bio
  results, although the Deep Bio branch itself still receives its normal QA
  classification.
- Do not infer that Deep Bio was disabled merely because no job exists. Use an
  explicit request/flag, published control-flow evidence, or a documented entry
  point default; otherwise report the branch intent as unknown.

## Intake

1. Preserve the pasted response as evidence.
2. Parse JSON if possible. If malformed, extract identifiers by text search.
3. Classify the input:
   - Entry-point response: contains a returned entity, `node_uuid`, master IDs, queue IDs, status, errors, or response metadata.
   - Smoke-test request fixture: contains source inputs such as `imdb_url`, `name`, `queue`, `cascade_depth`, `priority_tier`, `source_function`, `source_entity_type`, and seed IDs but no returned entity.
4. Identify or resolve the root entity to QA, preferring:
   - `node_uuid`
   - response entity object with `node_uuid`
   - `master_person_id`, `master_company_id`, `company_id`, `person_id`
   - for IMDB request fixtures, normalized `imdb_url` in `imdb_person` table `394`, then `name` plus `source_function`/timestamp evidence
   - queue id, run id, request id, or timestamp if no node id exists
5. Note the entry point/function name, visible status, returned errors, queue flags, and any IDs in the response or fixture.
6. Do not mark graph QA blocked merely because the pasted object lacks `node_uuid`. First attempt sandbox record resolution from the fixture fields. If the root graph node still cannot be determined after narrow table/log lookup, continue Xano/log QA and call out that graph QA is blocked pending a `node_uuid`.

## Fast Preflight And Conditional Discovery

Use the fastest safe path that can answer the question:

1. Prefer Xano MCP table tools with `workspace_id: 3` and `x-data-source: "sandbox"` for record counts, searches, and item reads. They avoid broad sandbox pulls and make the sandbox boundary explicit.
2. Use the fixed table/API registry above before calling `getWorkspaceContext`, API group lists, table lists, or other broad discovery.
3. Use CLI preflight only when MCP is unavailable, when a command will use broad `xano sandbox ...` state, or when a config mismatch is suspected.
4. Do not run `xano sandbox pull` by default. It is a fallback for missing narrow tools, unknown table IDs, or a user request for a local dump.

If CLI preflight is needed, use these current CLI forms:

```sh
xano profile list
xano workspace get -w 3 -o json
xano sandbox get
xano sandbox workflow_test list -o json
```

Use `xano function get <id> -w 3 -o xs --include_xanoscript` only for functions needed to understand the entry point, queue inserts, log writes, graph/Cypher execution, or error handling. Do not fetch full function lists unless the relevant function ID cannot be found from the registry, docs, API metadata, or targeted search.

### Reset And Async Quiescence

A sandbox wipe clears records but does not cancel API requests, provider jobs,
Cloud Run work, retries, or callbacks that were already in flight. Treat a
post-wipe run as potentially contaminated until the evidence window proves
otherwise:

- Record the test start time and the initial identity tuple for every related
  job: table row id, external `job_id`, request type, master person/company id,
  processing state, and creation timestamp.
- During every poll, compare those tuples, not only row counts. A row whose id
  remains constant but whose `job_id`, entity owner, or request type changes is
  an overwrite and a hard audit-integrity failure.
- Correlate late callbacks to their external job ids. A callback for a job id
  absent from the initial/current-run job set is stale or superseded unless a
  current-run launch request proves otherwise.
- Compare callback response shape and request duration with the currently
  published API source and its `updated_at`. If behavior contradicts current
  source and a long-running request spans a publish or wipe, identify it as
  likely pre-fix in-flight contamination instead of attributing it blindly to
  the current version. State clearly when this is an inference.
- Treat fields owned by an asynchronous job as `PENDING` until that job reaches
  a terminal state and the final settlement window completes. Do not call a
  missing deep bio, supplemental source array, callback-owned field, or graph
  write a landing defect while its producer can still write. After terminal
  success, re-read the supported destination: classify a missing optional value
  as `UNEXPLAINED_MISS` and a missing required value or claimed-success write as
  a failure. A terminal job error is failure evidence regardless of earlier
  partial materialization.
- Do not tell the user to wipe/retest while related jobs are processing or
  retrying, while old-version callback requests may still be running, or before
  the configured provider/API timeout window has drained. If active-request
  visibility is unavailable, report that reset quiescence is unproven.
- Treat `person_alias` as reset-sensitive identity state. After a claimed wipe
  and before the first person entry point, require table `759` to contain zero
  rows. During a run, require every alias to reference an existing `master_person`
  and fall within the evidence window. Otherwise report a hard reset failure.

When local files or pulled function output are already available, search them with `rg` for the response identifiers and likely infrastructure:

```sh
rg "node_uuid|crash_log|error|queue_|enrich_history|Falkor|GRAPH\\.QUERY|GRAPH\\.RO_QUERY|cypher" -n
```

If sandbox CLI profile context is unclear and the intended next command depends on profile context, do not continue into broad sandbox commands. Ask for the profile or exact sandbox access to use. Narrow MCP calls with `x-data-source: sandbox` may continue.

## Error And Log QA

Inspect sandbox records related to the test run:

- Crash/error log tables: names containing `crash`, `error`, `log`, `request`, or `history`.
- Enrichment history tables for person/company/source status.
- Function/request logs if available through Xano tooling or sandbox-only endpoints.
- Any error objects returned in the pasted response.

Correlate records by `node_uuid`, master IDs, queue IDs, run/request IDs, data source IDs, and timestamps around the test run. Look for:

- Exceptions, stack traces, failed function sections, and non-2xx external/API responses.
- Success status with empty/null output.
- Visibility or completion flags stuck false/in progress.
- Duplicate/repeated error records from retries and duplicate entities created by concurrent branches. Compare normalized natural keys (name plus company/context for people; name, domain, canonical profile URL, and permalink for companies) across rows created in the evidence window. Treat same-identity inserts only seconds apart as a race even when a later link writer makes both rows look valid; trace both create requests to identify the missing final pre-insert identity recheck.
- Audit `master_link` LinkedIn identity globally, not per owner. Normalize every
  LinkedIn regional host to `www.linkedin.com`, lowercase the host/path, and
  remove query strings, fragments, and trailing slashes. Require exactly one
  row per normalized LinkedIn URL across Person and Company owners, a non-empty
  `canonical_linkedin_url` equal to that normalized value, and exactly one
  owner (`master_person_id` xor `master_company_id`). A duplicate canonical
  value, null canonical value on a LinkedIn row, or dual owner is a hard
  failure. Apply the same rule to `/in/`, `/company/`, `/school/`, and other
  LinkedIn paths. Ordinary non-LinkedIn website URLs are explicitly exempt:
  their `canonical_linkedin_url` must remain null and their `link_url` may
  legitimately repeat across multiple records.
- Audit company identity across overlapping work sources, not only by exact
  `master_company_id`. For each person, group settled work rows by normalized
  title and start/end year-month. When two rows have the same employment
  signature but different company IDs, compare normalized company names,
  canonical organization URLs, domains, and source strength. A conservative
  name variant (for example one token differing only by a trailing plural
  `s`) plus the same role/dates and no conflicting domain/entity-class evidence
  is a likely duplicate-company failure, especially when one record owns the
  canonical `master_link` and the other carries a stale provider URL. Require
  one surviving company node and one employment edge; exact LinkedIn uniqueness
  alone does not prove company deduplication.
- Extend company-identity comparison across every organization-producing
  surface, including work, volunteering, education, certifications, and honor
  issuers. Treat an acronym, legal name, and full display name as the same
  company when an official domain, canonical LinkedIn URL, or equally strong
  alias evidence agrees and there is no conflicting entity evidence. A split
  where employment targets the full-name company but `ISSUED_BY` targets a
  separate acronym company is a hard semantic failure. Require one canonical
  company/table/graph identity, repointed source foreign keys and relationship
  endpoints, and no stale duplicate node. Prefer the earlier work-linked
  canonical company when evidence strength is otherwise equal; a newly created
  honor issuer must not displace it.
- When a cross-surface repair merges companies referenced by work,
  volunteering, education, certification, or honor rows, capture every
  relationship row/edge UUID before replay. After the supported merge/repoint
  path runs, require the source row to target the canonical company, exactly one
  canonical company/node to remain, and the original relationship edge UUID
  and edge properties to survive. A correct canonical company ID with a newly
  generated edge UUID is still a regression.
- Distinguish a same-batch identity-visibility race from a simultaneous insert.
  Compare each candidate company's `created_at` with the matching work rows'
  `created_at`, first non-null `master_company_id` write, and `updated_at`. If
  company A existed before company B was inserted but company A's work row did
  not receive its company ID until after both company inserts, the resolver
  could not safely rely only on already-linked work rows. Inspect the ordered
  resolver loop and its accumulator: require either an in-memory batch identity
  map, a company-level semantic candidate lookup, or immediate persistence of
  each resolved `master_company_id` before the next unresolved work row. Treat
  a later duplicate insert in this shape as a same-batch visibility defect even
  when requests were not concurrent.
- A conservative cross-company collision found by terminal work reconciliation
  is not a successful repair merely because it is logged or raised as a
  precondition. Require the terminal reconciler to choose one canonical company,
  invoke the supported company-merge/repoint path, re-read committed state, and
  fail only if the collision remains. The survivor decision should prefer a
  canonical `master_link`, stronger verified identity evidence, and an existing
  earlier company when there is no conflicting domain/entity evidence.
- After terminal company collision recovery, require exactly one semantic
  company row, one employment row/signature, one company queue identity, one
  company graph node, and one employment edge. Also verify canonical-ID
  propagation through volunteering, `master_link`, company enrichment data,
  resolver accumulators, and downstream child inputs. A unique `master_link`
  combined with a second company row that carries only a stale provider URL is
  still a hard failure.
- Treat URL-less synthetic C-suite candidates such as
  `source=Company_LLM` or `company_seed_fallback:*` as insufficient evidence
  for a new person. They may reuse one existing canonical exact-name or explicit
  alias match, but otherwise should fail closed before insert. If late verified
  profile evidence shows the seed-company role is historical or the current
  employer differs, fail the run even when the synthetic work history marked
  the seed role current.
- After settlement, group certifications by person plus normalized
  case-insensitive certification name and issuer. Exact duplicate rows fail,
  including duplicates produced by title-casing differences, overlapping
  source transactions, or replay. Verify the final resolver reconciled
  committed duplicates rather than relying only on the add-record lookup.
- Downstream steps skipped after an upstream failure.
- Use function request history as a call-chain trace before proposing a fix. If
  a parent function ran but its expected child has no matching request, the
  failure occurred before dispatch; inspect the parent statements preceding
  that call instead of changing the child. Scope request history to the test
  start time plus matching entity IDs/inputs; requests from earlier sandbox
  runs do not prove that a child ran in the current waterfall. Compare
  timestamps and inputs when retries or callbacks overlap.
- When a function that normally returns data or a success marker instead
  returns `null`, descend through same-window child request histories until the
  first unexpected `null` or missing child is found. Fetch that function's
  published XanoScript and scan the executed branch for active placeholders
  such as `function.run ""`, `db.get ""`, `db.edit ""`, or incomplete table/
  function references. A disabled placeholder begins with `!`; an active blank
  reference is a runtime failure even when the function published cleanly.
- When child work completed but the parent response is still `null`, inspect
  response-variable scope in the published XanoScript. A value declared only
  inside `try`, `if`, `foreach`, or another nested block may be unavailable to a
  later top-level `response = $value`. Confirm this against the executed branch;
  recommend hoisting the declaration or re-fetching the final record before the
  response rather than changing successful child functions.
- Diagnose `Unsupported parameter reference - xdo.<field>` schema-first. Check
  every in-scope `db.query` sort, predicate, and output against the live table
  schema before blaming Cypher. A missing relational column such as
  `created_at` can produce the same-looking error as a graph property rewrite.
- Treat `mvp/enrich/enrich-master-company` failures in `add-company-locations`
  as required-stage failures even when the company or school node and its
  relationship edge survived. In particular, `Unable to locate var:
  falkorNode.data[0].p.properties` means the location writer dereferenced an
  empty or structurally different Falkor response. Require a clean terminal
  success or an explicit no-location-evidence `SKIP` with `qa_passed=true`,
  empty `error_message`, and `warning=false`; a surviving node does not recover
  a `qa_passed=false` location crash.
- Warning-class logs even when `qa_passed=true`. A final `PASS` requires
  zero `qa_passed=false` rows and zero warning rows such as notes beginning
  with `WARN:`, optional-source warnings, or `qa_passed=true` rows that still
  carry a non-empty `error_message`. A handled optional miss that emits a clean
  `SKIP`/no-op with `qa_passed=true`, empty `error_message`, closed processing
  state, and explicit `warning=false` is expected behavior and does not lower
  a run from `PASS`. Apply the same contract to enrichment history: an optional miss with `enrich_success=false`, empty/implicit status, or no explicit reason is warning-class even when it is non-fatal and processing is closed.
- Audit `master_address` identity by owner and normalized natural key. New
  enrichment-owned rows require a non-empty natural key and exactly one owner
  (`master_person_id` xor `master_company_id`); duplicate
  `(owner type, owner id, natural_key)` rows fail. Compatible richer evidence
  must upgrade the keeper in place, sparse evidence must not downgrade it, and
  exact replay must be a no-op.
- Resolve `master_person.primary_location_id` only against `primary_location`
  table `385`. It is a foreign key to the canonical geography tuple
  (`city_id`, `region_id`, `country_id`), not to `master_address`. Numeric IDs
  can overlap across these tables, so never infer a person/address ownership
  failure by joining `master_person.primary_location_id` to `master_address.id`.
  Audit the referenced `primary_location` row and the Person `LOCATED_IN` graph
  edge separately from owner-scoped `master_address` natural-key QA.
- A failed provider job followed by a separate successful job is recovered but
  still warning-class because the failed terminal row remains part of the run.
  Treat it as a clean recovery only when the same logical job retains attempt
  audit details, closes with `processing=false`, has terminal `error=null`, and
  explicitly records a successful/recovered terminal outcome.
- Audit every related profile/provider job as a required terminal contract, not
  merely as background evidence. Preserve and compare row id, external `job_id`,
  owner entity, request type, `processing`, terminal status, terminal error,
  retry count, and same-row attempt history. A terminal JSON/response parse
  error is a failure even when the person or company row was created by an
  earlier phase. Read back fields owned by that job, such as `deep_bio`, so a
  nominally successful callback cannot hide missing downstream materialization.
- For Cloud Run deep-research callbacks, treat the external runner as the
  owner of provider retries within one logical `job_id`. A malformed, empty, or
  truncated provider response is retryable only when the callback explicitly
  carries `retry_scheduled`, `will_retry`, or an equivalent runner-owned retry
  disposition; an error string alone is terminal. Require each scheduled
  attempt to remain on the same job row, preserve an attempt entry with the
  error kind and callback time, and emit one terminal callback only after the
  runner exhausts its bounded attempts. A second job row for the same logical
  request, or a terminal parse error with no retry proof, is warning/failure
  evidence rather than a clean recovery.
- For every person deep-bio job, audit the complete provider boundary before
  judging the landed biography. Read these exact `profile_enrichment_job`
  surfaces when present:
  - launch input: `data._cloud_run_request`,
    `data._cloud_run_launch_response`, and `data._payload_audit`;
  - callback input: `response.provider_callback_payload` and
    `response.provider_attempt_payloads`;
  - trust decision: `response.provider_payload_validation` and
    `response.payload_audit`;
  - raw committed provider result: `llm_response`;
  - recovery state: `response.retry_history`, `response.retry_count`,
    `response.recovered`, `processing`, terminal status, and terminal error.
  Compare request, callback, and result character counts plus SHA-256 values.
  Report validator format, subject match, biography count/character count,
  parse error/position, and the bounded `tail_preview`. Require the successful
  callback receipt's `result.payload_validation.valid=true` and non-empty
  committed digest, then require that digest to match the independently hashed
  committed `llm_response`, the callback runner's canonical-payload digest, and
  the terminal/finalizer digest.
- Keep deep-bio digest meanings distinct. `provider_raw_input_sha256` (or the
  equivalent raw-attempt field) identifies the unmodified Cloud Run/provider
  result; `sanitized_payload_sha256` identifies validator-normalized JSON; and
  `committed_payload_sha256` identifies the exact text stored in
  `profile_enrichment_job.llm_response`. Compute the stored text digest
  independently rather than trusting a recorded field. Require the successful
  provider receipt's committed/exact digest, terminal digest, and independently
  computed `llm_response` digest to equal the committed digest. For the callback
  runner's independent revalidation, require `sanitized_payload_sha256` (the
  deterministic canonical JSON serialization) to equal that committed digest.
  Its `raw_input_sha256`/`exact_input_sha256` may legitimately differ because a
  Xano `json` function input decodes stored JSON text into an object before the
  validator runs. Still require `valid=true`, `sanitized=false`, matching
  `current_positions_sha256`, and the same validator version. A mismatch in the
  runner canonical digest is a hard failure; a runner raw/exact mismatch alone
  is not.
- Do not require a pre-commit JSON "fixed point" as proof of stored-byte
  integrity. Xano `json` function inputs can coerce a valid JSON string into an
  object and reserialize the same content differently. Serialize a validated
  canonical payload once, bind the receipt to that exact text, then independently
  read and hash `profile_enrichment_job.llm_response`. In the callback gate,
  compare those exact stored bytes with the revalidator's canonical
  `sanitized_payload_sha256`, not its representation-sensitive
  `exact_input_sha256`. Fail on a stored-text/readback mismatch, canonical digest
  mismatch, or semantic revalidation failure; do not reject valid content merely
  because object decoding changed whitespace, key order, or representation.
- Never paste a complete provider request/callback into the final QA report.
  When full-payload inspection is needed, save the exact JSON under
  `/private/tmp`, report its path, digest, and byte/character count, and include
  only the minimum diagnostic fragment. This preserves forensic evidence
  without flooding the report or exposing unrelated source material.
- A same-job provider retry can be a clean recovery only when every malformed

## Work History And School Materialization

When a person is enriched, distinguish raw experience capture from graph/company materialization:

- `work_experience` rows can be created without creating or linking `master_company` rows. Count both separately.

If the current work edge exists but `master_person.master_company_id` is null, fail the run and inspect `mvp/work/choose-best-current-role`, `mvp/enrich/enrich-master-person`, and late completion/webhook functions that edit `master_person`.
- Compare every settled `volunteering` row with its `VOLUNTEERED_AT` edge. A row with a non-null `end_year` must have graph `isCurrent=false`; a row with no end year must have `isCurrent=true` unless a stronger explicit source flag says otherwise. Require replay to repair existing edges, not only create missing ones, and re-read the committed edge properties. Count/UUID parity does not pass a stale recency flag.
- For every materialized `WORKS_AT` edge, verify both temporal representations
  against the authoritative grouped work rows: `valid_from`/`valid_to` plus
  `startYear`/`startMonth`/`endYear`/`endMonth`, and both `is_current` plus
  `isCurrent`. Year-only source precision must remain year-only rather than
  inventing January. Current roles require null end values and present-tense
  descriptions; ended roles require matching end values and past-tense
  descriptions. Replay must preserve the existing edge UUID while repairing
  either representation.
- Treat the settled table state as authoritative. A log such as "current role synced" is not enough to pass the current-company check; if the log shows `master_company_id` was repaired but the final `master_person.master_company_id` is null after settlement, call out a late-writer wipe.

Do not expect this person orchestrator to drain company queues as a side effect; school enrichment is now owned by the education resolver.

School graph nodes without `ATTENDED` edges are orphaned education materialization, not a pass.

  pair for a materialized education row. A surviving `edge_uuid` does not pass
  when `master_company_id` is null/zero; it is evidence of a late source-replay
  ownership wipe. Re-run the terminal education resolver, require a unique
  LinkedIn/domain/name school match, and verify the existing UUID is preserved.
- After a school relationship resolves, replay the same structured education
  identity with `master_company_id` absent. The committed school ID, row ID, and
  edge UUID must remain stable. If the replay clears the school ID, inspect
  `mvp/education/add-education-record` for a stale read/write race and require a
  just-in-time readback plus a post-write invariant before accepting the fix.
- Apply school-materialization expectations only to structured education evidence: rows in `education_experience`, structured education arrays from person-source payloads, LinkedIn/PDL/Enrich Layer/Exa education history, or another explicitly trusted structured credential extraction. Do not fail school QA because `deep_bio` narrative text mentions schools, degrees, academies, fellowships, prizes, patents, or honors that were never promoted into structured education evidence.
- Fail school QA when school-looking rows are captured but skipped as "not school-like". School evidence includes LinkedIn `/school/` URLs, `.edu` domains, `.ac.*`/`.ac.uk` style domains, and names containing terms such as University, College, School, Academy, Institute, Polytechnic, Business School, Graduate School, High School, or Secondary School.

Flag company-looking education candidates, especially LinkedIn `/company/` URLs with no school/domain/name evidence, entertainment/media/company names, or non-education `.com` domains. These should stay captured in `education_experience` but should not create `is_school=true` company rows or `ATTENDED` graph edges.

- Audit education duplicates after school resolution using the canonical
  `master_company_id`, dates, degree, and field—not raw school name alone.
  Same-school rows with identical dates, or a sparse source variant compatible
  with a dated/detail-rich row, should reconcile to one table row. Preserve
  distinct dated degrees at the same school; they may legitimately remain

  a degree or field from one structured education item while retaining

## Fix Recommendation Heuristics

When recommending or later verifying a XanoScript fix, remember that a successful `xano function edit --publish` only proves the function parsed. It does not prove embedded code blocks were serialized the way the author intended.

- For fixes touching `api.lambda`, `util.template_engine`, Cypher strings, or other embedded code/text blocks, recommend a fetch-back verification: `xano function get <id> -w 3 -o xs`, then scan the returned XanoScript for the expected `code = """ ... """`, `timeout`, Twig, or Cypher shape.
- Treat whitespace at a `util.template_engine` directive boundary as executable syntax. Twig may trim the newline immediately after `{% endif %}` and render invalid tokens such as `falseRETURN`. Keep an explicit literal separator before the following `RETURN`, `WITH`, or other clause, fetch the function back, and exercise both boolean branches at runtime.
- Fetch-back scan compound conditions for serialized backtick expressions such as ``if (`...`)`` or ``elseif (`...`)``. Xano can publish these successfully and then fail at runtime with a filter-parentheses error. Scalarize `get`, `is_empty`, and other filtered values into variables first, then make the conditional operate only on plain booleans; fetch back again and require zero active backtick conditionals.
- In particular, verify `api.lambda.timeout` sits outside the lambda `code` string. If `timeout = 5` appears inside the JavaScript payload or a stray quote delimiter appears inside the code, the function may publish but still return malformed values at runtime.
- For Cypher property rewrites, do not derive one property from another property assigned earlier in the same `SET` clause. FalkorDB may evaluate the derived expression against the pre-update value. Derive both values from the authoritative source row/parameters or split them into ordered clauses, then re-read the committed properties. Require the replay path to update existing relationships as well as newly created ones.
- Do not assume a graph wrapper throws on invalid Cypher. A provider may return an error string instead of a `{data:[...]}` result. Validate the response shape, include the raw graph response in sandbox failure logs, and fail the semantic gate when the expected returned row is absent.
- Use live Xano function fetch-back as the source of truth after publish. Local temp files, generated Swagger/OpenAPI descriptions, or CLI success output are supporting evidence only.
- Scan every fetched-back function touched by the fix for active blank
  references such as `function.run ""`, `db.get ""`, and `db.edit ""`.
  Disabled debug placeholders prefixed with `!` are different; active blanks
  must be removed or replaced before runtime verification.
- For multi-stage materializers, write all captured source rows before invoking
  shared resolvers, then resolve once. A legacy resolver call placed between a
  primary-row insert and a structured-source loop can abort the function and
  silently strand the remaining captured candidates.
- For a work/company identity-race fix, require controlled regressions in both
  source orderings (for example Enrich Layer then PDL, and PDL then Enrich
  Layer), plus an idempotent replay and a negative control with conflicting
  domains. Each positive ordering must settle to one semantic company/work/
  queue/graph identity; the negative control must preserve distinct companies.
  Run the existing LinkedIn-uniqueness, work-runtime, terminal-contract, and
  company/article regression workflows alongside these new cases so the repair
  cannot regress previously passing person-leaf behavior.
- For company classification repairs, require deterministic provider-free
  controls for explicit non-investors, a verified investor, passive
  `backed by` language, and an affiliated-entity mismatch. Exercise the
  published Xano helper live in addition to local tests and publish/fetch-back;
  XanoScript can publish successfully while embedded JavaScript still fails at
  runtime because of declaration order or payload shape.
- Require terminal cleanup on every exit path. Success, clean optional skip,
  and caught failure must all close processing flags and reconcile the matching
  history row; a downstream resolver failure must not leave history open.
`````

`````markdown
  current or have no end date. Exclude every role with an end date.
- Include all legitimate concurrent current roles in `currentPositions`, while
  allowing `master_person.current_title` and `master_company_id` to select one
  best primary current role.

  work reconciliation. The system prompt should default present-tense claims to
  those positions only.
- Require `bio`, `bio_500`, the leading/current section of `deep_bio`, the Person

- Historical roles may provide background evidence but must not be described as
  current. If the biography callback runs after current-role selection, require a
`````
