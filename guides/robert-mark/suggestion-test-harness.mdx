## Purpose

Central workspace for testing suggestion generation, ranking, and UX behavior across Robert / Mark scenarios.

<Info>
The test harness has two jobs: first, confirm whether the ranking score is returning the right people; second, tune the system prompt and model that generate the best "why" explanation for each suggestion.
</Info>

## Mark's Idea

This should become a hosted test page for reviewing generated suggestions. The page should load suggestion records that already exist, not suggestion requests, so reviewers are looking at the actual people the system produced.

<CardGroup>
  <Card title="Load Suggestions" icon="database">
    Primary action at the top of the harness. Fetches generated suggestions across the selected test scope and hydrates the table in the active tab.
  </Card>
  <Card title="Refresh Why" icon="rotate">
    Row-level action. Regenerates only the "why" for that suggestion using the release-candidate prompt, release-candidate model, suggestion summary, suggestion data, user context, and suggested person context.
  </Card>
</CardGroup>

### Prompt And Model Slots

Each tester needs four named slots for both the why prompt and the why model. The `Release Candidate` slot is the only slot used by `Refresh Why`; `Option A`, `Option B`, and `Option C` are working drafts for comparison.

| Slot | Prompt behavior | Model behavior |
| ---- | --------------- | -------------- |
| Option A | Editable system prompt draft. | Editable OpenRouter model name. |
| Option B | Editable system prompt draft. | Editable OpenRouter model name. |
| Option C | Editable system prompt draft. | Editable OpenRouter model name. |
| Release Candidate | Current candidate prompt used by row refreshes. | Current candidate model used by row refreshes. |

<Warning>
`Refresh Why` should not change the ranking score or the suggested person. It only regenerates the why text for the existing generated suggestion.
</Warning>

### Why Cost

Every why generation should persist token usage and cost next to the generated why text so reviewers can compare quality against spend.

| Metric | Definition |
| ------ | ---------- |
| Input tokens | Tokens sent to the model, including the selected system prompt, suggestion summary, suggestion data, user context, and suggested person context. |
| Output tokens | Tokens returned by the model for the generated why. |
| Total tokens | `input_tokens + output_tokens`. |
| Calculated cost | `(input_tokens / 1,000,000 * input_price_per_million) + (output_tokens / 1,000,000 * output_price_per_million)` for the selected OpenRouter model. |

Display this as a compact row value, for example `1,820 tokens / $0.0124`, with an expandable detail view for input tokens, output tokens, model pricing, and timestamp.

### Other Ideas

- Add a side-by-side compare mode that shows Option A, Option B, Option C, and Release Candidate why outputs for the same suggestion.
- Track prompt/model lineage on every refresh so ratings can be traced back to the exact prompt and model used.
- Add a cost-quality scatter plot so reviewers can compare rating against why cost by prompt/model pair.
- Add filters for suggestion type, ranking-score range, rating status, deliver format, and "needs review".
- Add a keyboard-first review flow: next row, previous row, star rating, copy why, and refresh why.
- Add a "golden set" toggle for benchmark suggestions that should always rank highly.
- Store reviewer notes alongside the 1 to 5 star rating so the subjective score has a short explanation.
- Add a diff view for refreshed why text so reviewers can see exactly what changed after prompt/model edits.
- Add export to CSV/JSON for offline review, sharing with Robert/Mark, or prompt-tuning analysis.

The notes above are the **product design**. This section is the **backend grounding** from the June 18, 2026 Mark/Robert sync (transcript 1186–1334): what the harness is really testing, which Xano endpoints it reuses, what is net-new, and where it sits in the build order.

<Note>
  **Sequencing: build this after How I Know.** Mark was explicit — the harness starts once How I Know X is "as good as it can be." It is the next thing after the relationship-interview punch list, not a parallel track. Treat the design above as locked-enough to build; treat the backend below as the part to align on before writing endpoints.
</Note>

### Two problems, not one

The harness exists to separate two failure modes that today are tangled together. This matches the Purpose callout at the top of this page.

| Problem | Question | What you tune | Where it lives today |
| --- | --- | --- | --- |
| **WHO** | Does the ranking surface the *right people*? | The candidate set + ranking score | The classify + per-class graph-match pipeline |
| **WHY** | Is the generated *explanation* good? | The system prompt + the model | Hardcoded per-class inside the synthesis endpoints |

A reviewer should be able to judge WHO **before** reading the WHY (is this a plausibly useful person at all?), then judge the WHY separately for specificity and evidence. The Option A/B/C/Release-Candidate slots and `Refresh Why` above are the WHY-tuning surface; the ranking score column is the WHO surface.

### Reuse — endpoints that already exist

| Endpoint | ID | Role in the harness |
| --- | --- | --- |
| `openui05/classify` | `#8490` | Classify a request into an outcome class (the WHO entry point). |
| `openui05/dispatch` | `#8497` | Run the full classified pipeline; returns `openui_lang` / `tool_data`. |
| Per-class synthesis | `#8491`+ | The class-specific candidate match + why generation. |
| `suggestion_request` | `#529` | The run container; the harness **loads generated suggestion records**, not raw requests (Mark's note above). |

### Net-new — the gaps to build

<Steps>
  <Step title="Override-dispatch endpoint">
    A new endpoint accepting `{ query, class, system_prompt, model, suggestion_request_id }` that runs synthesis with a **caller-supplied** system prompt and model, and returns `openui_lang` / `tool_data` plus eval metadata (token usage + cost). Needed because the system prompt and model are currently hardcoded per class — `suggestion_request` (`#529`) has no prompt/model columns. This is the backend behind `Refresh Why` and the A/B/C slots.
  </Step>
  <Step title="suggestion_eval_run table + endpoints">
    A new table to persist each evaluated run — `{ system_prompt, model, results snapshot, rating, latency, why_cost }` — plus endpoints to save a run and to browse/compare runs. This is what makes ratings durable and lineage traceable (the "track prompt/model lineage" and "cost-quality scatter" ideas above depend on it).
  </Step>
</Steps>

<Warning>
  **Rating scale — reconcile before building the table.** The June 18 call described a coarse **awesome / mediocre / bad** triage; the notes above specify a **1–5 star** rating with reviewer notes. Recommended resolution: persist the **1–5 star** scale as the source of truth, and offer **awesome / mediocre / bad** as a fast keyboard-first shortcut that maps to 5 / 3 / 1. Confirm with Mark so `suggestion_eval_run.rating` is defined once.
</Warning>

### Design for the GCP swap

<Note>
  Per the July 6 GCP migration ("every endpoint we make in Xano we remake in GCP"), build the override-dispatch and eval-run endpoints **thin and portable**, and give the harness an optional `backend_url` / proxy so its Xano endpoints can be repointed after the migration without a rebuild. The harness is an eval tool, so it can lag the migration — but it must not block it. Keep paths stable so a repoint is a config change, not a rewrite.
</Note>
