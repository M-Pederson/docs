## Backend — data model & integration

The first move is **reuse**. Orbiter already runs a full Nylas-synced calendar model in Xano workspace 3; the calendar module hangs off it rather than rebuilding it. This pass nails three things the existing model does **not** give us for free: the **missing time-range indexes**, the **org-scoping gap** on the legacy tables, and the **per-account privacy settings** (with their server-side enforcement) that gate team sharing — those privacy toggles and the `redact_to_rung` redaction live on [Settings & privacy](/guides/open-work/calendar/settings-privacy).

The data model has one product test: does it make the operator's scheduling loop correct, fast, and explainable? A table earns its place only if it supports one of four jobs: render a trustworthy time window, reconcile provider truth, enforce sharing/privacy, or bind the meeting to Orbiter work. That is why the plan adds `cal_webhook_event` and indexes before write-back, adds `cal_event_link` only when meetings need graph context, and defaults away from `cal_connection` unless a real connection-policy job appears. A parallel event store, calendar-specific task table, or speculative preference table would increase drift without serving the loop.

## Invariant placement

The model keeps each invariant at the layer that can actually enforce it. The settings UI can explain a toggle, but only the read projection can redact; the client can show a pending block, but only webhook ingest can make the mirror durable.

| Invariant | Lives in | Data-model implication |
| --------- | -------- | ---------------------- |
| Privacy ladder | `nylas_grant` flags plus `redact_to_rung` projection | Phase 1 may persist the toggles; the first team-facing read or Availability fan-out waits until Phase 3 server-side redaction exists. |
| Org access | `resolve_access()` / `df_grant`, with org resolved through membership or denormalized org scope | A calendar row being in the same org is not enough; every team read still needs the grant check and redaction. |
| Provider authority | Nylas event id plus `cal_webhook_event` delivery ledger | `event` (221) is upserted by provider state, not edited directly by the client. |
| Time correctness | `event.start` / `event.end` UTC plus IANA timezone on calendar/user/source | Named-zone math happens before storage and render; DST is not repaired by indexes later. |
| Grant lifecycle | `nylas_grant.grant_expired`, `event.deleted_at`, purge sweep | Revoke hides immediately, marks cached rows deleted, then physically purges detail inside the retention window. |
| Audit without PII hoarding | `webhook_calendar_log` plus `cal_webhook_event` | Raw logs need a scrub decision; the idempotency ledger can retain redacted delivery facts after mirror purge. |

## AI proposal lineage

AI scheduling does not need a parallel calendar table. The durable objects already exist in the Deal-Flow layer: `df_suggestion` for the proposal and rationale, `df_action_def` for the verb, `df_event` for the product timeline, and `cal_event_link` for the graph binding once confirmed. The model should record enough lineage to answer four audit questions later:

| Audit question | Stored on / derived from |
| -------------- | ------------------------ |
| What did the AI propose? | `df_suggestion` payload: candidate slot, invite draft, binding target, confidence, rationale, and redacted context summary. |
| What did the operator change? | Accepted payload vs proposed payload, including edited time, attendees, title, body, calendar, or graph link. |
| What actually wrote? | `schedule_meeting` action result and Nylas provider id after webhook reconciliation. |
| What failed or was undone? | Failed action result, rollback reason, retry metadata, and any later repair or undo proposal. |

The lineage stores redacted context summaries, not raw hidden teammate event detail. A failed AI proposal is still useful audit data, but it must not create an `event` (221) row until Nylas accepts a write and the webhook reconciles the mirror.

## Data-model build economics

The cheapest safe data model is the one that avoids durable state until a phase can prove it needs that state:

| Decision | Default | Promote only when |
| -------- | ------- | ----------------- |
| Event mirror | Reuse `event` (221) / `master_event` (222) | Never create a parallel `cal_event`; add indexes and edges around the existing mirror. |
| Connection state | Extend `nylas_grant` (183) | Create `cal_connection` only for a real org-level connection policy that cannot live on the grant. |
| Calendar render prefs | Prefer existing `calendar` prefs or thin `cal_calendar_pref` | Add the table only if per-user color, visibility, or order cannot fit safely on `calendar` (223). |
| Org scope | Resolve through membership until Phase 3 | Denormalize `organization_id` only when team-resource reads need the range-scan speed. |
| Graph linking | Add `cal_event_link` in Phase 2 | Link only after write-back and Meeting Prep need a persisted meeting-to-work edge. |
| Availability cache | Cache against the mirror in Phase 3 | Do not introduce cache tables before team fan-out and Nylas rate limits make them necessary. |

The per-user Nylas `grant_id` lives here and nowhere else. **It also gains the three NEW per-account calendar toggle bool columns** (`enable_calendar` / `share_conflicts` / `share_conflict_details`) that form the privacy ladder — see [Settings & privacy](/guides/open-work/calendar/settings-privacy) for the ladder, defaults, and server-side enforcement. |
| `calendar` (223) | **Per-user calendar container.** One row per connected calendar: `user_id` → `user` (125), `grant_id` (text), `external_id`, `name`, `timezone`, `is_primary`, `active`, `read_only`, `is_owned_by_user`, `is_holiday_calendar`, plus existing UI prefs `event_widget_preference` and `agenda_panel`. Drives the visibility/toggle list through `useCalendars()` and `useUpdateCalendarPreference()` in `src/features/events/api/`. **No `organization_id`** — org scope resolves via `user_id` → `organization_membership` (696). |
| `event` (221) | **The queryable event mirror.** Concrete instances: `start` / `end` timestamps (both nullable), `calendar_id` → `calendar` (223), `master_event_id` → `master_event` (222), `title`, `description`, `external_id` (provider event id), `provider`, `url`, `organize` bool, `event_data` JSON (attendees/RSVP/conferencing), soft-delete `deleted_at`. This is where FullCalendar's time-range reads run via `useCalendarEvents()`. **Note: `event` has NO `grant_id`, NO `user_id`, NO `organization_id`** — the grant/owner/org are reached only through `calendar_id`. |

| `activity` (218) / `activity_type` (217) | The unified **notification feed** (already merges calendar `event_id` and email thread/message; `unread` bool). Calendar reminder firings fan in here; it is **not** the timeline. |
| `webhook_calendar_log` (618) | Raw inbound calendar webhook payload log (`raw_body` JSON, `status`) — the existing raw-body landing spot; kept as the audit log behind the new idempotency ledger. |
| `account_provider` (224) | Provider list + logos for the connect-account UI. |
| `scheduling_link` (601) | External booking URL (Calendly-style) per `master_person` (139) — the existing booking pointer. |
| `important_date` (242) | Recurring personal dates (birthday/anniversary) + reminder lead-days — a non-Nylas dated source the calendar can render. |
| `user` (125) | Owner anchor + timezone (`local_timezone`, `settings.working_timezone`, `settings.event_widget_range`) **+ the new `settings.week_start`** (0/1); `tiptap_collab_jwt` for shared-calendar presence. `AuthUser.local_timezone` (read client-side via `useCurrentUser`) maps to this `local_timezone`. |
| `organization` (695) / `organization_membership` (696) | Org tenant + the **only** user↔org mapping (role admin/member, status). |
| `team` (613) / `team_member` (614) | Team grouping → the **rows of the resource Timeline** team overlay. |
| `master_company` (142) / `master_person` (139) | Graph nodes a meeting/attendee resolves to. |
| `integration_connection` (717) / `integration_meeting` (718) | The **separate** APIANT meeting-intelligence stack (`apiant_connection_id`) — explicitly **not** a Nylas grant; do not cross the wires. |
| `df_task` / `df_deal` / `df_event` / `df_reminder` / `df_action_def` / `df_view` / `df_focus_entry` / `df_suggestion` | Deal-Flow design tables (all NEW in that doc, `df_` prefix): the task/deal Event Sources, the product timeline, the reminder engine, the `schedule_meeting` write verb, the `view_type='calendar'` lens, the AI attention budget, and the AI proposal lineage. |

## New tables (`cal_*` — NEW, no id)

A small, orthogonal set that fills the gaps the existing model does not cover. Each is **NEW**, `cal_` prefix, no Xano id, `organization_id` → `organization` (695) NOT NULL, `updated_at` nullable, `created_at` default now, int PK, bool flags read via `first_notnull`.

- **`cal_event_link`** (**NEW**) — event ↔ graph/Deal-Flow entity. `event_id` → `event` (221), polymorphic `resource_type` (`df_deal`/`df_task`/`master_company`/`master_person`) + `resource_id`, `link_kind` (binds/about/attendee), `organization_id`. This is the persisted "this meeting is about this deal" edge that Meeting Prep + momentum read, and the target of an **external-chip drop** (and of the **AI binding a meeting to a deal**). Btrees on `(event_id)` and `(resource_type, resource_id)`. *Lands in Phase 2 — see [Build plan](/guides/open-work/calendar/build-plan).*
- **`cal_calendar_pref`** (**NEW**) — per-user, per-calendar render state FullCalendar needs that `calendar` (223) does not carry: assigned `color` (resolved against `THEME_COLORS`), `is_visible` (toggle), Event-Source order (LexoRank text). Keyed `(user_id, calendar_id)`. Read/written by the existing `useUpdateCalendarPreference()` hook (`src/features/events/api/use-update-calendar-preference.ts`). (Alternatively add `color`/`is_visible` columns to `calendar` 223, which already holds `event_widget_preference`/`agenda_panel`.) *Optional in Phase 1 — see [Build plan](/guides/open-work/calendar/build-plan).*
- **`cal_webhook_event`** (**NEW**) — inbound Nylas webhook **idempotency + dedupe** ledger, richer than the raw `webhook_calendar_log` (618). Stores the Nylas **delivery id** (`UNIQUE`), `event_type` (`event.created`/`updated`/`deleted`, `grant.expired`), `grant_id`, `external_id`, the provider's reported `updated_at`/sequence (the last-writer-wins guard), `processed_at`, `status`. The unique index makes redelivery a no-op upsert. Keep `webhook_calendar_log` (618) as the raw-body audit log behind it. *Lands in Phase 1 — see [Build plan](/guides/open-work/calendar/build-plan).*
- **`cal_connection`** (**NEW — default: do NOT create**) — the Nylas grant per user **already lives in `nylas_grant` (183)** and the three privacy toggles + default-calendar pointer extend that row. Create `cal_connection` **only** if calendar-feature state emerges that genuinely does not belong on the mailbox-scoped grant (e.g. a per-org connect policy). Default decision: extend `nylas_grant` (183) instead.

<Note>
**No new event-mirror table.** The verified schema already has `event` (221) (with `start`/`end`/`calendar_id`/`master_event_id`/`deleted_at`) as the queryable mirror and `master_event` (222) as the recurrence master. A second `cal_event` would duplicate them. The only persisted-event work is **adding indexes** (Phase 1), the org-scope decision (Phase 3 prerequisite), and the `cal_event_link` edge (Phase 2) — not a parallel table. See [Build plan](/guides/open-work/calendar/build-plan) for the phase sequencing.
</Note>

## New btree indexes (the real gap)

Verified live: every legacy calendar table carries **only** `primary(id)` + `btree(created_at desc)`. FullCalendar's reads are time-range scans (`start`/`end` within a visible window, filtered by `calendar_id`), which today are unindexed table scans. The exact new btrees to add (using the verified column names), with the phase each lands in:

| Table (id) | New index | Phase | Why |
| ---------- | --------- | ----- | --- |
| `df_task` (NEW) | `btree(due_date)`, `btree(start_date)` | **P0** | Read-only task layer window scans (from the Deal Flow design) |
| `df_deal` (NEW) | `btree(expected_close_date)` | **P0** | Read-only deal close-milestone layer |
| `event` (221) | `btree(calendar_id, start)` | **P1** | The primary read: "events on these calendars in this window." Composite covers the common filter + range in one. |
| `event` (221) | `btree(start)` and `btree(end)` | **P1** | Cross-calendar window scans and overlap/conflict checks. |
| `event` (221) | `btree(master_event_id)` | **P1** | Pull all instances of a recurring series. |
| `event` (221) | `btree(deleted_at)` | **P1** | Soft-delete filtering + the hard-purge sweep on disconnect/delete. |
| `calendar` (223) | `btree(grant_id)` and `btree(user_id)` | **P1** | List a grant's calendars / a user's calendars; the join hop for org-scope resolution. |
| `nylas_grant` (183) | `unique(grant_id)` and `btree(user_id)` | **P1** | Webhook routes by `grant_id`; the user → grants lookup feeds the team overlay. |
| `cal_webhook_event` (NEW) | `unique(delivery_id)` | **P1** | Idempotency — redelivery is a no-op. |
| `cal_event_link` (NEW) | `btree(event_id)`, `btree(resource_type, resource_id)` | **P2** | The two directions of the graph edge. |
| `df_reminder` (NEW) | `btree(remind_at)` | **P2** | The reminder firing sweep. |
| `event`/`calendar` (221/223) | `btree(organization_id, start)` | **P3** | Only if `organization_id` is denormalized for the resource-timeline org-wide feed (see Org-scope decision). |

> If org-level sharing denormalizes an `organization_id` onto `event`/`calendar`, add the composite `btree(organization_id, start)` for org-wide range scans (the resource-timeline feed) — otherwise that scan resolves through the `calendar.user_id → organization_membership` join and indexes the membership composite instead.

## Sync & idempotency flow

The mirror is kept fresh by one inbound path and one outbound path, both reconciling on the same `event` rows:

| Path | Source of truth | Idempotency key | Ordering rule | Failure behavior |
| ---- | --------------- | --------------- | ------------- | ---------------- |
| Initial backfill | Nylas list response | `(grant_id, external_id)` | Provider `updated_at` / sequence | Page and resume inside `timerange` / `oldest_sync`; do not block webhook receiver. |
| Provider webhook | Nylas webhook payload | `delivery_id` in `cal_webhook_event` | Provider `updated_at` / sequence, not arrival time | Duplicate is no-op; stale delivery is discarded; malformed payload lands in raw log with failed status. |
| UI / AI write | Nylas Events API response + later webhook | Pending client write id, then provider `external_id` | Webhook-confirmed provider state wins over optimistic state | Roll back optimistic UI on Nylas failure, expired grant, read-only calendar, or reconcile timeout. |
| Delta poll | Nylas list / delta response | `(grant_id, external_id)` | Same upsert comparator as webhook | Repairs missed webhooks; never creates a second ingest code path. |
| Disconnect / revoke | Grant lifecycle event | `grant_id` | Revoke beats all later stale event deliveries | Pause writes, hide rows, mark `deleted_at`, then hard-purge. |

<Steps>
  <Step title="Outbound write (the only write path)">
    `schedule_meeting` (`df_action_def`, `source_engine='calendar'`) → Nylas Events API by `grant_id` (create/update/delete + RSVP). The client gets an optimistic pending state; **no direct `event` (221) mutation.** Whether the caller is a button, a drag, or the AI Copilot, this is the identical path. A write against a grant whose `grant_expired = true` is **blocked**, not queued silently — including AI proposals.
  </Step>
  <Step title="Nylas emits the change">
    Nylas processes the write and emits `event.created` / `event.updated` / `event.deleted` (and `grant.expired` on token failure) to Orbiter's webhook endpoint. Provider-originated changes (a teammate edits the meeting in Google) arrive the same way.
  </Step>
  <Step title="Idempotent ingest">
    The receiver records the raw body in `webhook_calendar_log` (618), then checks `cal_webhook_event(delivery_id)` (unique). If already present → **no-op** (redelivery). Otherwise insert the ledger row and proceed.
  </Step>
  <Step title="Upsert the mirror (last-writer-wins by provider state)">
    Upsert `event` (221) by `external_id` (create / update / set `deleted_at`), expanding recurrence off `master_event` (222) as Nylas reports it. Out-of-order delivery is handled by comparing the payload's `updated_at`/sequence against the stored row — a stale (older) delivery is discarded, so reconciliation is **last-writer-wins on the provider's reported state, not on arrival order**.
  </Step>
  <Step title="Backstop reconcile">
    A periodic delta poll (bounded by `nylas_grant.timerange` / `oldest_sync`, paged) catches **missed** webhooks; `grant.expired` sets `nylas_grant.grant_expired`, surfaces a reconnect prompt, and **pauses writes** for that grant (button and AI alike).
  </Step>
</Steps>

## Event Source → data origin map

Each toggleable FullCalendar Event Source maps to exactly one data origin (and the phase it first renders in):

| Event Source (layer) | Data origin | First phase | Notes |
| -------------------- | ----------- | ----------- | ----- |
| Task due / start dates | `df_task.due_date` / `df_task.start_date` | P0 | A project-less task is a personal to-do; becomes draggable as an external chip in P2 |
| Deal close milestones | `df_deal.expected_close_date` | P0 | Close-month forecast lens buckets vs `df_target` windows; draggable as an external chip in P2 |
| Project / milestone dates | `df_project.start_date` / `target_date`, `df_project_section.target_date` (`is_milestone`) | P0 | Dated markers |
| Personal recurring dates | `important_date` (242) (`date`, `annual_recurring`, `reminder`) | P0 | Birthdays/anniversaries |
| Synced calendar events | `event` (221) (mirror of Nylas), filtered by `start`/`end` + `calendar_id` | P1 | The real meetings; writes go out via Nylas `grant_id` (P2). The "my events" layer takes the selected accent (`THEME_COLORS`). |
| Recurring series instances | `master_event` (222) expanded (Nylas-side) → `event` (221) instances | P1 | `@fullcalendar/rrule` is render-only fallback (added P2) |
| Reminders / nudges | `df_reminder.remind_at` | P2 | Fired into `activity` (218), not a paint layer by default |
| Meetings-to-prep (AI budget) | `df_focus_entry` (`entry_type='meeting'`) ranked off `event` (221) | P1 (read) / P2 (deep-link) | The AI attention budget; a non-grid "what needs prep" lane that deep-links into the Copilot |
| Team rows (resource overlay) | `team_member` (614) → per-teammate `nylas_grant` (183) → `event` (221) **via `redact_to_rung`** | P3 | **Premium** Timeline rows; **exposure gated by the privacy ladder** (`enable_calendar` → busy via `share_conflicts` → detail via `share_conflict_details`), redacted server-side (see [Settings & privacy](/guides/open-work/calendar/settings-privacy)); Free/Busy fills unbookable blocks. N teammates × a week = N Free/Busy fan-outs — cache against the mirror, refresh on webhook, never on scroll |
