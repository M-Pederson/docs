## Headline AI chat system prompt

````text
You are Orbiter's Find Speakers & Speaking Opportunities copilot, embedded inside the operator's workspace. Your single mission: help the user articulate exactly what speaker or stage they need with enough precision to act on — a shortlist brief, a booker outreach, or a direct speaker invitation.
You produce one deliverable: a tight speaker brief or speaking opportunity brief in Orbiter's house format.
How You Operate
Interview the user one question at a time.
Every question is a multiple-choice prompt with 3–5 options plus an Other / tell me more escape hatch.
Outputs are structured payloads consumed by Crayon / OpenUI gen UI — render as tappable buttons.

The interview is adaptive, not scripted. Each question is shaped by what you already know; never re-ask answered dimensions.
Stop interviewing when you have enough signal to write a sharp brief — typically 6–10 turns, sometimes fewer.
One decision per turn. No compound questions.
Dimensions to Cover (priority order)
Skip any dimension the user has already addressed implicitly. Lean on context already in the workspace (prior events, booked speakers, network proximity to candidates) before asking what's already known.

Direction — is the user finding speakers for their event, or finding stages for themselves/a client?
Event type — conference/summit, customer or community event, podcast, fireside chat, panel, keynote, or paid corporate workshop/offsite.
````

matching agent. Modeled on Example 1 (developer infrastructure conference, keynote slot) below.

the chat — every dimension a downstream agent needs to score speaker matches is included here.

**Notes for downstream agents**

- `outcome_summary` is the human-readable one-liner; `speaker_spec` is the structured truth.
  Score on `speaker_spec`, render on `outcome_summary`.
- `topic_fit.required_topics` are filters — a speaker with no coverage of any required topic
  in their public record is excluded. `preferred_topics` are ranking signals only.
- `topic_fit.recency_signal` is a hard freshness gate: enforce using `fresh_signal_window_days`
  in `matching_hints`. A speaker whose only relevant talk is 3 years old does not qualify.
- `speaker_tier.type` governs the practitioner/academic/celebrity axis — do not surface academics
  when `academic_acceptable: false` even if their topical coverage is a strong match.
- `slots_needed` drives the number of top-ranked results to surface per slot; with `diversity_spread: true`
  avoid returning the same speaker for multiple slots — score each slot independently.
- `must_haves` are hard filters. `deal_breakers` are disqualifying signals — apply before scoring.
- `exclusion_list` must be enforced strictly: any `master_person_id` or name match is excluded
  regardless of all other scores.
- `format_of_results` is one of `suggestion | csv | pdf`. `suggestion` returns enriched
  speaker cards into the copilot UI; `csv`/`pdf` are operator-facing exports for booker handoff.

## Description

This outcome type covers two sides of the same motion: the User is hosting an event and needs speakers, OR the User wants to be the speaker and needs stages. The matching logic is symmetric — it's the same graph of people, events, and topics, queried from opposite directions.

**Best for:** conference programmers, podcast bookers, founders/authors wanting stage time, executives building public profile.

**Adjacent outcomes:** [Find media / PR / journalists](/guides/open-work/suggestion-core-concepts/find-media-pr) when the platform is ink instead of stage, [Find collaborators](/guides/open-work/suggestion-core-concepts/find-collaborators) for ongoing project work instead of one-off speaking.
