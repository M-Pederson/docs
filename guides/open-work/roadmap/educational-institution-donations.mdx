**Status: scope & planning only.** Nothing on this page is built — no functions, no tables, no prompts deployed. Goal: find donors to universities and educational institutions and materialize **`DONOR_TO`** edges in the graph, with donor identities resolved to `master_person` / `master_company` records (LinkedIn URL or enough identity data points to find one via Exa).

## Why

Philanthropy to an institution is a triple signal: **wealth** (capacity), **affinity** (the donor cares enough to give), and very often an **alumni tie** the graph may not yet know about. For an operator-at-scale, "who gave to my alma mater / my target's alma mater" is a warm-path generator — `DONOR_TO` composes naturally with `ATTENDED` for intros, fundraising, and board-level networking.

## The key structural fact (researched 2026-07-05)

Donor data splits hard by **donor type**, because of IRS disclosure rules:

- **Individuals → public charities (universities): NOT in IRS data.** A 501(c)(3)'s Schedule B contributor names/addresses are withheld from public inspection (and redactable by the filer). There is no government database of individual gifts to universities.
- **Private foundations: fully public.** Foundations get no Schedule B protection — their 990-PF filings publicly disclose **both** their grant lists (every grant, grantee, amount) **and** their officers/directors/trustees.

So the design is two-track: a **free, structured foundation backbone** (990-PF data — zero LLM needed) plus an **agent-driven long tail** for individual donors (announced gifts, honor rolls, named buildings).

<Tip>
  **The family-foundation bridge.** Most large "individual" giving flows through family foundations — and a 990-PF lists its trustees **with addresses**. "The Smith Family Foundation gave \$5M to Stanford" \+ trustees John & Jane Smith = a person-level `DONOR_TO` (via foundation) with government-grade identity data. The free tier resolves far more _people_ than it first appears.
</Tip>

## Proposed `DONOR_TO` edge (draft)

| Aspect | Proposal |
| --- | --- |
| Pattern | `(Person)-[:DONOR_TO]->(Company:School)` and `(Company)-[:DONOR_TO]->(Company:School)` for foundations/corporations |
| Cardinality | One edge per donor–institution pair; aggregate properties (`gift_count`, `total_usd_known`, `first_gift_year`, `last_gift_year`). Per-gift detail stays relational. |
| Weight (§3.6 bands, lower = closer) |
| Metadata envelope | Required per the 2026-07-04 house standard: `uuid`, `created_at`, `updated_at`, `weight`, deterministic `description` ("X is a donor to Y"), `description_generated_at`. |
| Other properties | `amount_usd` (when public), `gift_years`, `purpose/campaign`, `gift_type` (cash / pledge / bequest / in-kind), `via_foundation_uuid` (family-foundation bridge), `source_urls`, `confidence`, `is_alumni` (when `ATTENDED` corroborates). |

Open sub-question: do foundations become their own nodes (`Company` with a foundation marker) with trustee `Person` edges, or do we collapse family-foundation gifts straight onto the trustees? Recommendation: **both** — foundation node \+ `DONOR_TO`, trustees linked to the foundation, and a derived person-level `DONOR_TO {via_foundation_uuid}` so path queries work either way.

## Agent design — three stages

### Stage A — Gift discovery agent (per institution)

Input: one institution (node uuid, name, domain, aliases). Sweeps, in order of precision:

1. Foundation backbone lookup (SOURCE / 990-PF rows for this institution) — structured, no LLM.
2. Curated mega-gift lists (Chronicle, Million Dollar List, annual roundups) filtered to this institution.
3. Open-web sweep: institution press releases, news ("\$X gift to Y"), campaign pages — web-search-grounded LLM.
4. Honor-roll / donor-wall pages: Firecrawl fetch → extraction LLM per page.

Output: structured **gift mentions** (contract below), never edges directly.

| Role | Model (recommendation) | Why |
| --- | --- | --- |
| Open-web discovery (step 3) | `moonshotai/kimi-k2.6` \+ `openrouter:web_search`/`web_fetch`, provider-pinned, `allow_fallbacks: false` | The house deep-research config (#12833) — already proven for multi-source web synthesis with citations. |
| Per-page extraction (step 4) | `google/gemini-3.1-flash-lite`, `reasoning: {enabled: false}`, strict JSON | The #12984 tier-1 economics (~\$0.001/call) — honor rolls are high-volume, low-ambiguity extraction. |
| Weak-signal retry | `deepseek/deepseek-v4-flash` \+ web search | The #12984 tier-2 pattern for pages/mentions the cheap model fumbles. |

### Stage B — Donor identity resolution (per mention)

Deterministic first, LLM last:

1. **Existing-graph match**: `master_person` by normalized name where an `ATTENDED` edge to the same institution exists (strong prior); existing `master_link` URL match if the mention carried one.
2. **Foundation-trustee path**: mention is a foundation → 990-PF trustee list → person candidates with addresses.
3. **Exa LinkedIn search** using the mention's identity points (location, title/employer, class year, spouse) — the #13121-style socials search.
4. **LLM adjudication**: `google/gemini-3-flash-preview` (temp 0.1) with the full mention context \+ candidate profile — the #12677 v5 config — with explicit same-name/different-person rejection. **Auto-link only at `confidence: high` / ≥0.85** (house convention); 0.6–0.85 → `needs_review`; below → park. Name-only matches are capped below the auto-link threshold no matter what the model says (the #13168 corroboration guard).

Minimum to _attempt_ auto-link: name \+ **two** independent corroborating identity points (e.g. class year \+ city, employer \+ spouse name).

### Stage C — Edge writer

MERGE `DONOR_TO` with the metadata envelope; idempotent per donor–institution pair. New-person policy mirrors the investor cascade: created **local-only leaves** \+ queue rows (tier TBD), kill switches respected, no recursive enrichment from a donation sweep.

## Draft system prompt v0.1 — gift extraction (not deployed)

```text
You extract charitable gifts to ONE educational institution from the provided page content.

Institution: {{institution_name}} (also known as: {{aliases}}). Domain: {{domain}}.

Return ONLY valid JSON: {"gifts": [...]}. Each gift object:
- donor_name: exactly as written (couples: keep as written, e.g. "John and Jane Smith")
- donor_type: "person" | "couple" | "foundation" | "company" | "anonymous"
- amount_usd: number | null  (NEVER infer or convert vague language into a number)
- gift_year: number | null
- purpose: string | null  (building, scholarship, professorship, campaign name)
- gift_type: "cash" | "pledge" | "bequest" | "in_kind" | "unknown"
- giving_level: string | null  (honor-roll band when stated, e.g. "President's Circle $50,000+")
- evidence_quote: verbatim sentence(s) naming the donor and the gift
- identity_points: {location, title, employer, class_year, degree, spouse_name, other} (nulls allowed)
- confidence: "high" | "medium" | "low"

Rules:
1. Only gifts TO this institution (or its named schools/colleges/foundations). Gifts BY the institution, grants it received from government, and sponsorships are out of scope.
2. Anonymous donors: record the gift with donor_type "anonymous"; never guess identity.
3. Class-year notations ("Jane Smith '88", "MBA '02") go into identity_points.class_year/degree.
4. Honor-roll listings without amounts are valid gifts: amount_usd null + giving_level when the page states the band.
5. Do not deduplicate across pages; emit every mention with its own evidence_quote.
6. If the page contains no qualifying gifts, return {"gifts": []}.
```

The Stage-B adjudication prompt follows the #12677 v5 shape (full mention context \+ candidate profile → `match: yes/no/none`, `confidence`, reasoning; explicit instruction that a same-name candidate in a different profession/era/geography is a **no**) — to be drafted against real cases in the LLM Test Harness before deployment.

## Rollout order

1. **Foundation backbone** (SOURCE / 990-PF import for institutions already in the graph) — highest precision, zero LLM cost, and the trustee bridge resolves people with address-grade identity.
2. **Curated mega-gift lists** — hundreds of named donors at the top of the wealth pyramid.
3. **Honor-roll sweeps** for the institutions with the most `ATTENDED` overlap — the agent-driven volume tier.
4. **Commercial API evaluation** (DonorSearch trial) only if 1–3 leave a coverage gap worth paying for.

Gating mirrors the Fundable Stage-1 rule: edges are written only for donors who resolve to an existing record or pass the auto-link threshold; everything else parks as `needs_review` with its evidence — releasable later, never lost.

## Open questions

1. **Gift floor** — \$1M\+ announced gifts only, or include honor-roll tiers (huge volume, weaker signal, weight 30–40)?
2. **Foundation & corporate donors in v1?** Recommended yes — they are the _free, precise_ tier and the trustee bridge is the best individual-identity source we have.
3. **Edge granularity** — aggregate per pair (proposed) vs one edge per gift.
4. **Weight scaling** — fixed bands (proposed) vs amount-scaled formula like `HAS_EXPERTISE`.
5. **Commercial licensing appetite** — DonorSearch quote worth requesting now, or after the free tiers land?
6. **International scope** — UK (Charity Commission accounts), Canada (T3010) have analogous public filings; v2.
7. **Runtime** — same decision as the Fundable lane: Xano interim vs the AlloyDB Universe backend. The agent design above is runtime-agnostic.
