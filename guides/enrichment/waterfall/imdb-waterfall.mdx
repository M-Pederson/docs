## Source contract ‚Äî IMDb via Firecrawl

Unlike the [Music waterfall](/guides/enrichment/waterfall/music-waterfall)'s structured JSON API, IMDb has no public API ‚Äî every entity is a **Firecrawl scrape of a public IMDb page to markdown, followed by an LLM extraction to structured JSON**.

| Property | Value |
| --- | --- |
| Person source | `https://www.imdb.com/name/nm‚Ä¶` ‚Üí `imdb_person.markdown` ‚Üí LLM ‚Üí `imdb_person.json` |
| Title source | `https://www.imdb.com/title/tt‚Ä¶` ‚Üí `film_tv.markdown` ‚Üí LLM ‚Üí `film_tv.json` |
| Awards source | `{imdb_url}/awards/` ‚Üí `award_markdown` ‚Üí LLM ‚Üí `awards_json` \+ `film_tv_award` rows |
| Posters | IMDb image CDN (`@._V1_` size transform recomputed) ‚Üí re-hosted GCS WebP `imdb-posters/<imdb_id>` |
| Natural key | **normalized `imdb_url`** ‚Äî \`trim | split:"/?" | first | rtrim:"/"\` (queues \+ source tables) |
| Staleness gate | `enriched_at` within **30 days** ‚Üí no re-scrape (`refresh: true` on the title gate forces one) |
| Rate limiting | no provider hard limit ‚Äî pacing comes from the single-row cron tick (600 s per queue) |
| Firecrawl wrapper | `mvp/firecrawl/scrape` #4515 calls Firecrawl v2 `/scrape` for markdown \+ links. **v1.6 (2026-07-07):** treats Xano optional-input `0`/empty-string sentinels as missing for `timeout_ms`, `request_timeout_seconds`, and `proxy`, restoring the intended defaults of 60000 ms, 60 s, and `auto`. Required person/title page scrapes stay hard-fail; optional awards-page scrapes pass `soft_fail: true` and skip cleanly on no markdown. |
| LLM extraction | models, temps, and token ceilings per function: [Film & TV model summary](/guides/enrichment/film-tv-pipeline/model-summary) ¬∑ prompts: [Film & TV system prompts](/guides/enrichment/film-tv-pipeline/llm-system-prompts) ‚Äî linked, not duplicated |

<Note>
  **Rebuild-spec gaps (per the [GCP migration plan](/guides/open-work/orbiter-univers-standalone/enrichment-gcp-migration#what-already-exists-vs-what-the-spec-must-add)):** the exact Firecrawl request contract, the LLM extraction JSON schemas (`credits[]`, `film_tv` shapes), and the award-scraper output schema are not yet written down ‚Äî they are Step-2 spec-extraction deliverables, to be pulled from the frozen XanoScript rather than authored by hand.

## Depth model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
| :-: | --- | --- |
| **0** | Seed (user-initiated) | Full scrape \+ LLM extract \+ awards \+ cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` \+ name/title, no scrape). Queued. Cron processor scrapes \+ expands, but **only T1‚ÄìT2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes \+ writes data, **no further cascade** |
| **3\+** | Never queued | Hard cap |

<Check>
  **Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits \+ 9 acting \+ 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director \+ writer \+ DP \+ lead producer \+ 3 cast = ~7 people ‚Üí ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000\+.
</Check>

## Priority tiers

### `queue_imdb_person` (1 = highest)

| Tier | Rule | Example (seed = title "Gamer") |
| :-: | --- | --- |
| **1** | Director OR Writer of seed title; award-winning/nominated person on seed | Mark Neveldine, Brian Taylor |
| **2** | Cinematographer OR lead Producer of seed title; co-winner / co-nominee on seed's award (via `shared_with`) | Ekkehart Pollack (DP), lead producers |
| **3** | Top 3 cast of seed title; other `shared_with` people | Gerard Butler, Michael C. Hall, Amber Valletta |
| **4** | Everyone else mentioned | Minor cast, thanks, video-game voice actors |

### `queue_imdb_title` (1 = highest)

| Tier | Rule | Example (seed = person "Neveldine") |
| :-: | --- | --- |
| **1** | Seed person was Director OR Writer on this title; award-winning/nominated title | Crank, Gamer, Ghost Rider, Crank: High Voltage, Jonah Hex, Pathology |
| **2** | Seed person was lead Producer (not exec) AND (rating ‚â• 7 OR year = `current_year`) | Kid Revolver (8.2 Short), Great Pretenders (8.9 Short), The Olympians (upcoming) |
| **3** | Seed person was executive producer / cinematographer / minor role; OR rating ‚â• 7 regardless of role | Officer Downe, Urge, Pathology (producer), Vatican Tapes (director) |
| **4** | Video games / music videos / shorts without high rating | Jonah Hex (Video Game, tt38040358) |

### Credit filter for person seed (`cascade-credits-from-person`)

```text
for each credit in imdb_person.json.credits:
    if role ‚àà {director, writer}:                                ‚Üí T1
    elif role = producer AND (rating ‚â• 7 OR year = current_year): ‚Üí T2
    elif role ‚àà {executive producer, cinematographer, minor}
         OR rating ‚â• 7:                                          ‚Üí T3
    elif type ‚àà {Video Game, Music Video}
         OR (type = Short AND rating < 8):                       ‚Üí T4
    else: skip
    # T3 capped at top 20 by rating to prevent low-signal blowout
```

### Key crew filter for title seed (`cascade-crew-from-title`)

```text
from film_tv.json:
    directors[]                                      ‚Üí T1
    writers[]                                        ‚Üí T1
    cinematographer (if present in json)             ‚Üí T2
    lead producer (first producer in credits list)   ‚Üí T2
    top 3 cast (from cast[] sorted by bill order)    ‚Üí T3
```

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

  </Step>

  </Step>

### Depth-1 queued processing

When the cron picks up a queue entry, it sets `processing: true` and calls the same orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P6) only queue T1‚ÄìT2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured ‚Äî but no children are queued.

External calls by depth (Firecrawl scrapes \+ LLM extracts):

- **Depth 0:** two Firecrawl scrapes \+ two LLM extracts for the seed itself ‚Äî the person page (P2) and `{imdb_url}/awards/` (P5) ‚Äî plus up to five inline title enrichments via `sync_top_credits: 5`, each carrying its own scrape \+ extraction cost before the seed call returns.
- **Depth 1:** queueing itself is free ‚Äî stubs are created via `get-add` with no scrape. External calls fire only when the cron picks the row: the same full flow (page scrape \+ LLM extract, awards scrape \+ LLM extract) per entity.
- **Depth 2:** same per-entity scrape \+ extraction cost, but cascade helpers are skipped ‚Äî no new queue rows, so external calls stop here (depth 3 is never queued).
- **Pacing:** no provider hard limit ‚Äî the external call rate is bounded by the single-row cron tick (600 s per queue), and both queue crons ship `active = false`.

## Loop prevention (five layers)

1. **Dedup gate in `get-add`:** `imdb_url` already in `imdb_person` / `film_tv` with `enriched_at` within 30 days ‚Üí return existing, no scrape.
2. **Queue upsert:** duplicate `imdb_url` in queue ‚Üí `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth ‚â• 1:** only T1‚ÄìT2 discoveries are queued at depth 2 (T3‚ÄìT4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.

## Implementation notes captured during build

- **URL normalization lives in get-add** ‚Äî all queue rows and `imdb_person` / `film_tv` rows are keyed on the normalized URL (the chain is specified on [IMDB Tables](/guides/enrichment/waterfall/imdb-tables#url-normalization)).
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` ‚Äî NOT `function.call` (not valid in Xano).
- **Queue upsert is the single source of truth:** `get-add` always routes through `upsert-imdb-person` / `upsert-imdb-title`. No raw `db.add` into queue tables anywhere.
- **Kill switch is fail-open:** missing env var ‚Üí `try_catch` logs to `log_crash` and defaults OFF.
- **`cascade_depth` \+ `queue` param on get-add:** when both `cascade_depth == 0` AND `queue == false`, `get-add` invokes the orchestrator synchronously after the stub exists and returns `action: "enriched_direct"` ‚Äî no queue entry written.
- **Orchestrators are per-phase `try_catch` ‚Üí `log_crash`:** each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` \+ stack trace, using the IMDB `phase` enum (strict ‚Äî listed on [IMDB Tables](/guides/enrichment/waterfall/imdb-tables#table-inventory)).
- **Phase 1 guard:** if scrape\+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early ‚Äî no `master_person` gets created from a junk row.
- **Cascade helpers are pure dispatchers:** every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"`. Every `function.run` to `get-add` is wrapped in `try_catch` ‚Üí `log_crash`.
- **Seed ID propagation:** a depth-0 seed invocation via `get-add` must set `seed_imdb_person_id` / `seed_film_tv_id` to the just-created stub's own id when no explicit seed is passed. Required for loop-prevention to work.
- **Top-N sync credits on person seeds (`sync_top_credits: 5`):** `cascade-credits-from-person` v1.1 sync-enriches the top 5 credits inline when the person is at `cascade_depth == 0`. Guarantees a seed returns with enriched films even before the title cron runs.
- **Cron single-row tick pattern:** modeled on `process-person-queue` **#126** / `process-company-queue` **#127**. Lock-on-pick, `try_catch` around orchestrator, release lock on failure. Priority pick order: `priority_tier` asc, `cascade_depth` asc, `id` asc.
- **Missing-stub guard:** if the queue row's `imdb_person_id` / `film_tv_id` is 0, write `log_crash` and `db.del` the queue row ‚Äî prevents a bad row from monopolizing the lock forever.
- **Symmetric credit-edge writers:** #13013 (person side) and #13015 (title side) each silently skip endpoints whose other side hasn't been enriched. Whichever entity processes first stamps its `node_uuid`; the other side picks up the edge on its own run. No reconciler cron needed.
- **Honorary awards have no `Film_TV` anchor:** `#2334 v1.5` handles `film_tv_id == null` by gating the `Film_TV` MERGE on `hasFilm` and swapping the `Film_TV_Award` MERGE key from `film_tv_imdb_url` ‚Üí `person_uuid` so honoraries remain unique per-person.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **`shared_with`** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person \+ company waterfall solves this for LinkedIn/PDL/Enrich Layer (and now, in v4, for the post-PDL LLM stack). This waterfall applies the same pattern ‚Äî **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** ‚Äî to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits). Write `WORKED_ON` edges for every credit whose film is already in the graph.
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) \+ top 3 cast. Write `WORKED_ON` edges for every cast/crew whose person is already in the graph.
