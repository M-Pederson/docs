### FalkorDB graph-merge-person v0.7 — incident, recovery, fix (2026-05-05)

**Incident:** First attempt at `merge/helpers/graph-merge-person` (#12949) used a single complex
cypher per edge type with `OPTIONAL MATCH` chains and a `FOREACH (CASE WHEN ... | CREATE ...)`
trick. In FalkorDB, when the inner `OPTIONAL MATCH` returned NULL for the target node, the FOREACH
CREATE failed with `"Failed to create relationship; endpoint was not found"` — but the outer
DETACH DELETE on the orphan still ran. Result: Traver's two orphan Person nodes (1cadc19b...,
12329e15...) were deleted with all 28 of their edges still attached. **Edges were lost.**

**Recovery:**
- The only truly unique lost edge (WORKS_AT to Unknown Company `34951bf7-5d54-46e9-83cf-2bfb0e2c05fe`,
  master_company_id 10170) was restored via direct cypher MERGE with the original properties
  recovered from inspection logs
- The other ~26 lost edges were almost certainly duplicates of survivor's existing edges (HAS_EXPERTISE,
  WORKS_AT, FOUNDED, LOCATED_IN to common targets), so the loss was effectively absorbed by
  survivor's intact edges
- **Acknowledged real loss:** ~2 KNOWS edges from orphans to other Persons. KNOWS isn't
  re-derivable without re-running the original ingest. Documented as accepted loss.

**Fix — v2.1 graph-merge-person:** completely restructured cypher pattern:

1. **Per-orphan, per-edge-type cypher** instead of one big chained query. 5 edge types × N orphans.
2. **MATCH (not OPTIONAL MATCH)** — if no edges exist, returns 0 rows, no-op.
3. **MERGE on survivor** with `ON CREATE SET r_new = properties(r_old)` — copies orphan props
   to new edge if survivor didn't have one. On collision (survivor already has edge to same target):
   survivor's edge wins, orphan's props discarded. **v0.8 will add per-type smart merge.**
4. **DELETE orphan edge** after MERGE.
5. **Safety gate:** after all 5 cyphers run, query `MATCH (o)-[r]-() RETURN count(r) AS remaining`.
   `DETACH DELETE` only fires if `remaining == 0`. Otherwise orphan stays alive and the function
   reports it in `orphans_skipped`.
6. **try_catch around every cypher** — failure on one edge type doesn't kill the whole orphan.

**Verification on Sam Seely orphans (still alive at this point):**

- Isolated test: moved 1 FOUNDED edge from orphan 6cfa7efd to survivor 2f883678 cleanly. Other
  edges intact. Cypher pattern verified.
- Full v2.1 run on both Seely orphans: **24 edges moved (HAS_EXPERTISE 13, WORKS_AT 8, FOUNDED 1,
  LOCATED_IN 2, KNOWS 0), 2 orphan nodes DETACH DELETEd, zero errors.** Survivor 2f883678 ended
  with 10 HAS_EXPERTISE, 5 WORKS_AT, 3 KNOWS, 1 FOUNDED, 1 LOCATED_IN.

**Lessons:**
- Test cypher in isolation before wrapping it in a delete. The first attempt would have failed
  silently in dry_run if I'd separated the cypher generation from the execution — but I'd jumped
  straight to running it.
- "Defense in depth" via try_catch around the master_person delete (v0.5) was the wrong instinct.
  It masked the orphan-leaving bug instead of preventing it. The fix is to gate destructive ops
  on positive confirmation that the migrate succeeded — not to swallow errors.
- FalkorDB's `OPTIONAL MATCH` chains with `FOREACH (CASE | CREATE...)` patterns are dangerous.
  Prefer simple `MATCH ... MERGE ... DELETE` per-edge-type sequences.

### v0.8 — merge_suspect lifecycle

After the master_person delete (so suspects reflect the final state), the orchestrator now
processes `merge_suspect` (#284) rows touching either side of the merge:

1. **Query**: `WHERE master_person_id IN $all_ids OR merge_with_master_person_id IN $all_ids`
   — fetches every suspect involving the survivor or any duplicate.
2. **Per row**: compute `new_a = (master_person_id IN duplicate_ids) ? survivor_id : master_person_id`,
   same for `new_b`. If `new_a == new_b` (the pair collapses to survivor↔survivor) → **delete**
   the suspect. Else → **`db.edit`** with the repointed FKs and `merged: true`.
3. **Rescore flag**: query remaining `merge_suspect` rows where survivor is on either side AND
   `merged == false` (these are suspects involving the survivor that weren't part of THIS merge
   but might be stale because the survivor's profile changed), set `needs_rescore: true` so a
   downstream cron can re-evaluate.

`suspect_plan` in the response reports `touched_count`, `collapsed_deleted` (ids), `marked_merged`
(ids), and `rescore_flagged` (ids).

Note: `merge_suspect` is currently empty in live (`mvp/merge/detect-suspects` #12709 hasn't been
crontab-scheduled yet). The block runs cleanly when invoked but reports all-zero counts.

- BOARD_MEMBER_OF, AUTHORED, VOLUNTEERED_AT, MEMBER_OF, REGISTERED_FOR

- CONTRIBUTED_TO, WORKED_ON, WON_AWARD, NOMINATED_FOR

edge type appears in the future. Adding it is a 1-line change to the helper's array.

**v0.5 (2026-05-05) — duplicate `master_person` rows are now deleted at the end.**

After all the repoint+dedup blocks and the `person_enrich_data` collapse, the orchestrator
attempts to delete each ID in `duplicate_ids` from `master_person`. Each delete is wrapped in
`try_catch`:

- **Success** → the ID is appended to `delete_plan.deleted`
- **Failure** (FK constraint violation from any of the ~74 person-side tables we haven't yet
  repointed) → the ID + JSON-encoded error is appended to `delete_plan.errors`. The orchestrator
  continues; the audit row still finalizes with `status='success'` (or `'dry_run'`).

The `try_catch` guard is the pragmatic answer to: "we only cover 12 of 86 dependent tables, so
the duplicate row may still be referenced". As coverage grows in v0.6+ we expect the
`errors` array to shrink to empty.

What v0.5 does NOT yet handle: `my_person` reconciliation (CRM rows + `my_person_*_join`s +
activity dedup), `master_address` dedup (needs canonical address hash), the long tail of ~70
remaining person-side tables, `merge_suspect` lifecycle, FalkorDB graph merge, conflict-resolution
UI/logging, `merge_lock` acquisition. Tracked for v0.6+.

## Xanoscript gotchas (learned the hard way)

These are runtime-vs-docs mismatches discovered during the v0.1 build. Apply them when extending
the orchestrator or building per-table dedup helpers.

| Don't write | Write instead | Why |
| --- | --- | --- |
| `\|filter:$$.field == X` | `\|filter:"return $this.field == X;"` | The bare `$$.field` syntax compiles but matches zero rows at runtime. Legacy quoted JS form is the only one that actually works. |
| `\|some:$$.field == true` | `\|some:"return $this.field === true;"` | Same reason. |
| `$result.field` (when `$result` came from `function.run`) | `$result\|get:"field"` | `.field` access fails on function.run results with "Unable to locate var: result.field". `\|get:"field"` works. (`.field` does work on `db.add`/`db.get` results.) |
| `$input.X\|default:false` | `($input.X == true)` or ternary | `\|default` is not a real filter. Using it produces a misleading `"Unable to locate func entry: default"` runtime error. |
| `db.direct_query { sql = $var }` or `sql = "..." ~ $var ~ "..."` | Use static `db.query <table_name>` blocks per table | `db.direct_query.sql` only accepts a string literal. Means generic FK repointing across N tables can't be one helper — has to be one block per table (in the orchestrator) or per-table helper functions. |
| `db.edit table { data = $merged_object }` | `data = { name: $merged\|get:"name", ... }` (one key per field) | `data` field rejects whole-object var refs. Have to spread the object into individual keys. |
| Multi-line `description = "..."` | One-line description | Newlines inside `description` strings break the parser with `unexpected newline`. |

## Why merge is hard

A duplicate is not just two rows in `master_person`. It's two rows that are each pointed at by
**86 other tables for person** and **65 other tables for company**, plus a node in FalkorDB with
N edges, plus join rows under `my_person` for every user who has either copy in their CRM.

A correct merge has to:

1. Pick a survivor and aggregate scalar fields from the duplicates onto it
2. Repoint every FK in every dependent table from each duplicate → survivor
3. Collapse rows that become duplicates after repoint (e.g. two `master_email` rows with the same address)
4. Collapse 1-row-per-entity caches by merging columns first-not-empty (`person_enrich_data`, `company_enrich_data`)
5. Re-route every graph edge from the duplicate Person/Company node → survivor node, deduping on type
6. Delete the duplicate rows + duplicate nodes
7. Re-derive computed state (employment, embeddings, person/company nodes)
8. Be idempotent — re-running with the same input is a no-op
9. Be audited — every merge writes a snapshot row that can roll back
10. Be dry-runnable — every `db.edit` / `db.del` / Cypher write gates on a `dry_run` flag so the function can be tested against live data without mutating it

Inputs follow the same shape for both:

```ts
type MergeInput = {
  survivor_id: number;          // master_person_id or master_company_id we keep
  duplicate_ids: number[];      // array of master IDs that collapse into the survivor
  initiated_by: 'user' | 'cron' | 'admin';
  merge_suspect_id?: number;    // when triggered from merge_suspect (#284)
  dry_run?: boolean;            // when true, every db.edit/db.del/Cypher write is logged but skipped
};
```

When `dry_run=true`, the function still writes a `merge_audit` row (with `status='dry_run'`) and
populates `pre_merge_snapshot` and `conflicts_resolved` exactly as it would for a real merge,
so the operator can inspect what *would* have changed. No FK repoint, dedup, graph mutation, or
row deletion happens. The full read path runs — including FK count probes — so the audit row's
counts reflect what's actually in the database.

### Test loop

1. Run `merge/person-v2 { survivor_id, duplicate_ids, initiated_by: 'admin', dry_run: true }`.
2. Inspect the resulting `merge_audit` row: `pre_merge_snapshot`, `conflicts_resolved`, and the
   per-table row counts that *would* change.
3. If the dry-run plan looks right, re-run with `dry_run: false`.
4. Inspect the post-merge `master_person` row, the `merge_audit` row, the FalkorDB Person node,
   and a spot-check of dependent tables.
5. Roll back via `merge/rollback` (or restore from `pre_merge_snapshot` manually) if anything's wrong.

## Merging a Person

```text
function: merge/person-v2  (TO BUILD — workspace 3, branch v1)
```

### Phase flow

<Steps>
  <Step title="Validate & lock">
    Reject if `survivor_id` ∈ `duplicate_ids`, if any duplicate is `null`, or if any `master_person`
    row is missing. Lock all involved rows for the duration of the merge by writing a row to
    `merge_lock` (new table — see [Open questions](#open-questions)).
  </Step>
  <Step title="Snapshot for rollback">
    Write a `merge_audit` row (new table) with the JSON of the survivor + each duplicate's
    `master_person` row, plus the counts of dependent rows in every table touched. This is the
    rollback artifact and the audit log.
  </Step>
  <Step title="Aggregate scalar fields onto the survivor">
    For each scalar column on `master_person` (139), apply the column-specific merge rule below.
  </Step>
  <Step title="Repoint master_person_id FKs across all 86 dependent tables">
    `UPDATE … SET master_person_id = survivor_id WHERE master_person_id IN (duplicate_ids)`.
  </Step>
  <Step title="Reconcile my_person + its joins">
    For each user that has a `my_person` (128) row pointing at one of the duplicates, fold it
    into their existing `my_person` row pointing at the survivor (if any), or repoint it.
  </Step>
  <Step title="Dedup multi-row tables">

### my_person reconciliation

`my_person` (128) is per-user. Different users can each have their own `my_person` row pointing at
the same `master_person`. After repointing, multiple rows for the same `(user_id, master_person_id)`
are illegal — collapse them.

For each user that owns a `my_person` row pointing at any duplicate:

1. Look up the user's existing `my_person` row pointing at `survivor_id` (if any).
2. **If none**: just `UPDATE my_person SET master_person_id = survivor_id WHERE id = duplicate_my_person.id`. Done.
3. **If one exists**: fold the duplicate's joins into the survivor's joins, then delete the duplicate `my_person` row.

## Graph merge cypher

The graph merge runs **after** every Xano FK is repointed but **before** the duplicate
`master_*` rows are deleted (so we still know which `node_uuid` to delete from the graph).

FalkorDB does not support APOC, so edge re-routing must enumerate relationship types. We have
~25 edge types in production — see [Edges](/guides/ontology/edges) — so the cypher template
below loops over them in the orchestrator.

### Step 1 — Aggregate properties onto the survivor

<CodeGroup>
```cypher Person
MATCH (s:Person {uuid: $survivor_uuid})
UNWIND $duplicate_uuids AS dup_uuid
MATCH (d:Person {uuid: dup_uuid})

SET s.full_name      = coalesce(s.full_name, d.full_name),
    s.first_name     = coalesce(s.first_name, d.first_name),
    s.last_name      = coalesce(s.last_name, d.last_name),
    s.headline       = coalesce(s.headline, d.headline),
    s.bio_500        = coalesce(s.bio_500, d.bio_500),
    s.bio_2000       = coalesce(s.bio_2000, d.bio_2000),
    s.is_angel       = coalesce(s.is_angel, false) OR coalesce(d.is_angel, false),
```

### Step 2 — Re-route edges per relationship type

For each edge type, run the template below with `$rel_type` substituted in. The orchestrator
loops over the edge type list from [Edges](/guides/ontology/edges).

```cypher
// $rel_type is substituted in by the template engine — Cypher does not allow parameterized rel types
MATCH (d:Person)-[r:KNOWS]-(other)
WHERE d.uuid IN $duplicate_uuids AND other.uuid <> $survivor_uuid

WITH d, other, r, properties(r) AS props,
     CASE WHEN startNode(r) = d THEN 'out' ELSE 'in' END AS direction

MATCH (s:Person {uuid: $survivor_uuid})

// Branch on direction so we preserve edge orientation
FOREACH (_ IN CASE WHEN direction = 'out' THEN [1] ELSE [] END |
  MERGE (s)-[new_r:KNOWS]->(other)
  SET new_r += props,
      new_r.merged_from = coalesce(new_r.merged_from, []) + d.uuid
)
FOREACH (_ IN CASE WHEN direction = 'in' THEN [1] ELSE [] END |
  MERGE (other)-[new_r:KNOWS]->(s)
  SET new_r += props,
      new_r.merged_from = coalesce(new_r.merged_from, []) + d.uuid
)

DELETE r
RETURN count(*) AS edges_rerouted
```

**Per-edge-type aggregation rules** when the `MERGE` lands on an existing edge:

| Edge type | Aggregation |
| --- | --- |
| `KNOWS` | Apply `known/ingest/append_evidence` (#12779) semantics: union evidence array, max confidence, min `known_since`, max `last_evidence_at` |
| `HAS_EXPERTISE` | Apply `mvp/expertise/upsert-person-has-expertise` (#12925) semantics: max confidence, strongest depth, most current recency, pipe-concat evidence |
| `WORKED_AT`, `EDUCATED_AT` | Keep both edges if `start_date` differs; merge if same `start_date` (sum tenure-derived weight) |
| `INVESTED_IN`, `RAISED` | Keep both — investments are time-stamped events, not duplicates |
| `REGISTERED_FOR`, `ATTENDED` | Dedup on target uuid (one event per person) |
| `WON_AWARD`, `NOMINATED_FOR` | Dedup on target Award uuid |
| Any other relationship | Default: `SET new_r += props` (last write wins on scalar props) |

## Idempotency & safety guarantees

1. **Idempotent**: re-running `merge/person-v2` with the same `(survivor_id, duplicate_ids)` after a
   successful merge is a no-op — the duplicate IDs no longer exist, so all `WHERE id IN duplicate_ids`
   queries match zero rows. The orchestrator should detect this and return `{status: 'already_merged'}`
   instead of failing.
2. **Atomic per-table**: each table's repoint+dedup runs inside a single transaction. The whole merge
   is *not* one mega-transaction (Xano can't hold one open across 80+ tables) — instead, the
   `merge_audit` snapshot is the rollback point. A `merge/rollback` function uses the snapshot to
   un-merge.
3. **Fail-loud**: any FK constraint violation or unexpected row count writes a `log_crash` row and
   sets `merge_audit.status = 'failed'`. No silent partial merges.
4. **Survivor lock**: while a merge is in flight, the survivor + duplicate IDs are written to a
   `merge_lock` table. Other writers (enrichment crons, user actions) check this table and either
   wait or skip.

## Conflict resolution (decided)

Scalar field conflicts (survivor and a duplicate both have non-null values that disagree) split
into two tiers:

### High-stakes fields — surface to the user when interactive

- `headline`, `bio_500`, `bio_2000`, `tagline`, `summary_500`, `summary_2000`
- `name`, `first_name`, `middle_name`, `last_name`, `suffix`, `nickname`
- `legal_name` (company)
- `domain`, `linkedin_url`, `twitter_url`, `website_url` (company)

When a **user-initiated** merge hits a conflict on any of these, the merge orchestrator pauses
and surfaces a side-by-side conflict-resolution UI. The user picks per field, or accepts the
"longest non-null" auto-pick default. Then the merge proceeds with the user's choices.

### Everything else — auto-pick silently

All other scalar fields auto-pick by the rules in the [scalar aggregation tables](#scalar-field-aggregation-rules-master-person-table-139)
without surfacing anything.

### Cron-initiated merges — never block on a human

Cron paths (deterministic exact-match via `mvp/merge/master-merge` #12736, future scheduled
re-merges of high-confidence suspects) auto-pick high-stakes fields too, AND write each
overridden field to `merge_audit.conflicts_resolved[]`:

```json
{
  "conflicts_resolved": [
    {
      "field": "headline",
      "winner_value": "Founder & CEO at Acme",
      "winner_source_master_id": 12345,
      "loser_values": ["Co-founder, Acme Inc."],
      "loser_source_master_ids": [67890],
      "rule_applied": "longest_non_null"
    }
  ]
}
```

A "review recent auto-merges" feed reads from `merge_audit` where `conflicts_resolved` is
non-empty, letting the user override after the fact (the override writes a follow-up
`merge_field_override` row but does NOT re-run the full merge).

### Confidence escape hatch — deterministic matches skip even the logging

When the merge was triggered by a deterministic exact-match (same email or normalized phone),
the records are by definition the same person. Conflicts on scalar fields are just stale data
from one source — auto-pick "longest non-null" without writing to `conflicts_resolved`. This
keeps the after-the-fact review feed focused on actually ambiguous merges.

The trigger source is recorded in `merge_audit.initiated_by` plus a new
`merge_audit.match_kind` column with values `'deterministic'` (email/phone exact-match),
`'llm_confirmed'` (Tier 1/Tier 2 from `mvp/merge/detect-suspects` #12709), or
`'user_picked'` (manual merge from the suspect UI).

## Suspect lifecycle after merge (decided)

When a merge completes, the orchestrator finalizes `merge_suspect` (284) in three steps:

1. **Repoint** — `UPDATE merge_suspect SET master_person_id = survivor_id WHERE master_person_id IN duplicate_ids` and same for `merge_with_master_person_id` (or the company-side equivalent FKs once `kind` is added).
2. **Delete collapsed pairs** — after repoint, delete any row where both sides now equal `survivor_id`. The pair is meaningless (the suspect was for two records that are now one).
3. **Flag for re-score** — on every remaining row touching the survivor, set `needs_rescore=true`. A cron (new or extension to `mvp/merge/detect-suspects` #12709) picks up flagged rows and re-runs Tier 1 / Tier 2 scoring just for the affected pairs. The new score writes back to the same row and clears the flag.

UI behavior while `needs_rescore=true`: hide the row from the active suspect feed, or show with a "re-evaluating" tag. Don't let users action a stale-score suspect.

**Why deferred re-score, not synchronous**: scoring at merge time would burn Gemini Flash + Pro tokens on every merge — most of which are routine deterministic matches that don't need their downstream suspects re-evaluated. Deferring lets the cron batch them.
