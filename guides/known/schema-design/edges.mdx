**10 edge types** across 4 logical groups.

Known is primarily an edge-based model. Every facet between two persons is a direct Cypher edge, canonicalized (lower `uuid` first for undirected facets). The exceptions — introductions and households — are [node types](/guides/known/schema-design/nodes) for reasons documented on that page.

## Summary

### Relationship facets (Person ↔ Person)

| Edge type | Direction | Weight range | Purpose |
|---|---|---|---|
| [`KNOWS`](#knows-the-envelope-edge) | Undirected | 25–60 | Envelope edge. Interaction counts, evidence, scoring, shared context cache. |
| [`PROFESSIONAL_TIE`](#professional-tie-work-relationship-overlay) | Undirected | 5–35 (3 tiers by role) | Work relationship scoped to a company — co-founder, manager/report, peer, board, etc. |
| [`BONDED`](#bonded-closeness-overlay) | Undirected | 7–25 (by source) | Close tie overlay. Declared (7–18) or inferred (10–25). |
| [`KINSHIP`](#kinship-family-overlay) | Undirected | 3–25 (2 tiers) | Family relationships. Immediate (3–10) or extended (10–25). |
| [`ROMANTIC`](#romantic-romantic-relationship-overlay) | Undirected | 4–20 (2 tiers) | Romantic relationships. Committed (4–10) or dating (10–20). |
| [`ESTRANGED`](#estranged-negative-valence-overlay) | Undirected | — (penalty on KNOWS) | Negative-valence overlay. Adds `severity × 5` penalty to the KNOWS weight. |
| [`MENTOR`](#mentor-directed-coaching-relationship) | **Directed** | 8–25 (by formality + source) | The only directed facet. Mentor → mentee. Unified with the existing YC `MENTOR` edge in production. |

### Structural edges (for Introduction and Household nodes)

| Edge type | Direction | Weight range | Purpose |
|---|---|---|---|
| `MADE_INTRODUCTION` | Directed | 25–40 | `(Introducer:Person)-[:MADE_INTRODUCTION]->(Introduction)`. Who made the introduction. |
| `CONNECTS` | Directed | 30–45 | `(Introduction)-[:CONNECTS]->(Person)`. Each person who was introduced. |
| `RESIDED_AT` | Directed | 8–35 (by current/past) | `(Person)-[:RESIDED_AT]->(Household)`. Residency window with move-in/move-out per person. |

Full details for each edge type follow below, grouped by facet vs structural.

## Weight convention

Every weighted edge carries a `weight` field (float) with the convention **lower = more important**. Weight ranges are declared per edge type (floor/ceiling) and the specific value per instance is computed from `strength_score` by the scoring cron:

```
weight = floor + (ceiling - floor) * (1 - strength_score / 100)
```

The full weight table is in [Schema Design → Weight model](/guides/known/schema-design#weight-model). Each edge section below repeats its own range for reference.

## `KNOWS` — the envelope edge

The baseline edge representing "these two people know each other." Carries interaction counts, evidence, confidence, scoring, and cached shared context. Every other facet is an overlay on top of an existing `KNOWS`.

| | |
|---|---|
| **Pattern** | `(Person)-[:KNOWS]->(Person)` (canonicalized, lower id first) |
| **Directionality** | Undirected (canonicalized) |
| **Cardinality** | At most one valid `KNOWS` per pair at any given time |
| **Created when** | Evidence floor is met (see [Evidence model](/guides/known/documentation#evidence-model)) |
| **Weight range** | Baseline **25–60**, derived from `strength_score`. Plus `ESTRANGED` penalty (`severity × 5`) when an ESTRANGED facet is currently valid on the same pair. |

### Cached shared context

Populated by a cron that traverses each person's `WORKS_AT`, `STUDIED_AT`, `ATTENDED_EVENT`, `KINSHIP`, and `RESIDED_AT` edges and intersects.

```json
"shared_context_cache": {
  "employments": [{"company_id": "...", "overlap_start": "2015-04-01", "overlap_end": "2022-06-30", "overlap_days": 2647}],
  "educations":  [{"school_id":  "...", "overlap_start": "...", "overlap_end": "...", "overlap_days": 0}],
  "events":      [{"event_id":   "...", "date": "2015-03-15"}],
  "children":    [{"person_id":  "...", "shared_since": "2010-03-22", "biological_both": true}],
  "households":  [{"household_uuid": "...", "overlap_start": "2008-06-14", "overlap_end": "2020-04-02", "overlap_days": 4309}],
  "ventures":    [{"venture_id": "...", "role": "co-founder"}],
  "computed_at": "2026-04-10T00:00:00Z"
}
```

The `households` array references `Household` nodes by uuid rather than storing address data inline. The overlap window is computed from the two persons' `RESIDED_AT` edges to the same `Household`.

|---|---|---|

Estrangement does NOT freeze the score. Peak watermarks preserve the "used to be close" signal while the live score decays normally.

### Evidence coverage

Records which data sources are imported for each person. Used to compute `adjusted_reciprocity` and to distinguish "one-sided relationship" from "incomplete data."

```json
"evidence_coverage": {
  "person_a": {"email": "full", "calendar": "full", "chat": "full", "phone": "full"},
  "person_b": {"email": "none", "calendar": "none", "chat": "none", "phone": "none"}
}
```

Values: `full`, `partial`, `none`.

### Evidence row shape

Every entry in the `evidence` array has this shape:

```json
{
  "source": "linkedin_connection",
  "first_seen_at": "2024-03-15T10:42:00Z",
  "evidence_occurred_at": "2019-05-22T00:00:00Z",
  "weight": 0.25,
  "raw_metadata": {"linkedin_connection_id": "xyz"}
}
```

- `first_seen_at` — when Orbiter ingested the evidence (system time)
- `evidence_occurred_at` — when the real-world event occurred (domain time). Used for `known_since` derivation.
- `weight` — contribution to confidence. See [Evidence sources](#evidence-sources) below.
- `raw_metadata` — source-specific payload for debugging / auditing.

### Evidence sources

Every source is tiered as **strong** or **weak**. Strong sources alone are sufficient to create a `KNOWS` edge. Weak sources require at least one additional source (strong or weak) to clear the floor.

**Strong sources (weight 0.5):**

| Source | Description |
|---|---|
| `crm_contact` | User explicitly tracked this person in their CRM |
| `phone_contact` | User has them in phone contacts |
| `email_direct` | Two-way email exchange (not newsletter) |
| `calendar_meeting` | Both on a calendar event together |
| `manual_user_add` | User explicitly stated "I know this person" |
| `sms_chat` | Direct messaging history |

**Weak sources (weight 0.25):**

| Source | Description |
|---|---|
| `linkedin_connection` | LinkedIn connection |
| `twitter_follow` | Public follow (mutual or one-directional) |
| `instagram_follow` | Public follow |
| `shared_event_attendance` | Both at same conference/event |
| `shared_employment_inferred` | Overlapped at a company — may not have known each other |
| `mutual_friend_mention` | Third-party mention in content |
| `press_co_mention` | Named in the same article |

**Floor rule:** `create_knows = (strong_count ≥ 1) OR (weak_count ≥ 2)`

**Confidence calculation:** `confidence = min(1.0, sum(weights))`

## `BONDED` — closeness overlay

| | |
|---|---|
| **Pattern** | `(Person)-[:BONDED]->(Person)` (canonicalized) |
| **Directionality** | Undirected |
| **Cardinality** | At most one valid per pair at a time |
| **Created when** | Inferred: `strength_score > 70` AND `shared_context_cache` has real overlap. Declared: user/LLM action. |
| **Invalidated when** | Inferred: `strength_score < 55` (hysteresis prevents flapping) OR an ESTRANGED facet is created. Declared: only by user/LLM action. |
| **Weight range** | Modulated by `source`. Declared: **7–18**. Inferred: **10–25**. `bond_type` does not modulate weight. |

| Property | Type | Description |
|---|---|---|
| `uuid` | string | Edge identifier |
| `valid_from` | timestamp | |
| `valid_to` | timestamp \| null | |
| `source` | enum | `inferred` \| `declared` |
| `bond_type` | array\<enum\> | Array of bond categories: `friendship`, `romantic`, `spouse`, `familial`, `cofounder`, `mentor_mentee`, `co_parent` |
| `confidence` | float | Derived for inferred; 1.0 for declared |
| `peak_strength_during_bond` | float | Max `strength_score` observed while this bond was valid |
| `created_by` | string | |
| `created_at` | timestamp | |
| `invalidation_reason` | enum \| null | |
| `invalidated_by` | string \| null | |

**Hysteresis rule (cron logic):** Inferred BONDED is created when `strength_score > 70`. It is only invalidated when `strength_score < 55`. The 15-point gap prevents thrashing near the threshold.

**Derivation of `bond_type` (inferred only):** Derived from currently-valid facets on the same pair at creation time:
- `ROMANTIC(type='married')` present → include `'spouse'`
- `ROMANTIC(type='dating' | 'domestic_partner')` present → include `'romantic'`
- `KINSHIP` present → include `'familial'`
- `PROFESSIONAL_TIE(relationship_role='co_founder')` present → include `'cofounder'`
- Both persons have `KINSHIP(relation='parent')` to the same third person → include `'co_parent'`
- `MENTOR` present → include `'mentor_mentee'`
- Otherwise → `'friendship'`

### Creation rules

**Direct biological relations** (`parent`, `child`, `sibling`): created from declared family tree data or inferred from public records. No automatic cascade creation.

**Grandparent / grandchild / aunt / uncle / niece / nephew / cousin:** auto-created when the graph contains the intermediate `parent` / `child` / `sibling` chain. E.g. if Alice has `parent` to Bob, and Bob has `parent` to Carol, then Alice's `grandparent` KINSHIP to Carol is created automatically by the KINSHIP maintenance cron.

**In-law KINSHIPs on marriage (and committed-tier ROMANTIC):** when a `ROMANTIC` facet of any committed tier (`married`, `engaged`, `domestic_partner`, `separated`) is created between persons A and B, the cron auto-creates:

- `parent_in_law` KINSHIPs: A ↔ each of B's parents (and vice versa)
- `sibling_in_law` KINSHIPs: A ↔ each of B's siblings (and vice versa)
- `child_in_law` KINSHIPs: A ↔ each of B's children from prior relationships (and vice versa)

Each auto-created in-law has `via_chain = [{person_id: B, relationship: 'spouse'}]` (or `'domestic_partner'` etc. depending on the ROMANTIC type).

**Step-parent / step-child KINSHIPs on marriage:** when the committed ROMANTIC forms and one party has existing `parent` KINSHIPs to minor or adult children, the new spouse automatically gains `step_parent` KINSHIPs to those children. Each child reciprocally gains `step_child`. The `via_chain` references the biological parent who is the bridge.

**Step-sibling KINSHIPs:** when two persons each have a biological `parent` KINSHIP, and their parents share a committed ROMANTIC, a `step_sibling` KINSHIP is created between them. The `via_chain` lists **both** biological parents as anchors (see [`via_chain` structure](#via-chain-structure) below).

**Adoptive relations:** `adoptive_parent`, `adoptive_child`, `adoptive_sibling` are only created via **declared action** — a user marks "I adopted X" or an LLM review pass detects a formal adoption record. Adoptive relations are **direct** (no `via_chain`) because adoption creates a legal parent-child relationship not mediated by an intermediate person.

**Adoption can supersede step-relations.** If a user declares an adoption that replaces an existing step-relation (e.g. step-father formally adopts step-son), the existing `step_parent` KINSHIP is invalidated with `invalidation_reason='superseded'` and a new `adoptive_parent` KINSHIP is created. The old row remains as history.

### via_chain structure

The `via_chain` field represents the path by which an indirect KINSHIP was established. Each entry is `{person_id, relationship}`.

- **Single-anchor chains** (in-laws): one entry pointing at the bridging person.
  ```json
  // Alice ↔ her mother-in-law Patricia, via her spouse Bob
  via_chain: [{person_id: "bob_uuid", relationship: "spouse"}]
  ```

- **Multi-anchor chains** (step-siblings): multiple entries, each pointing at one of the bridging persons. The marriage connecting the bridges is implicit from the graph state.
  ```json
  // Noah ↔ his step-sister Olivia, via Noah's mom (1000) and Omar (1002)
  // who are married to each other
  via_chain: [
    {person_id: "1000_uuid", relationship: "parent"},  // Noah's biological parent
    {person_id: "1002_uuid", relationship: "parent"}   // Olivia's biological parent
  ]
  ```

- **Adoptive relations** have `via_chain = null` — they are direct legal relationships, not bridged through anyone.

- **Direct biological relations** also have `via_chain = null`.

The `via_chain` is **frozen at creation time**. If an intermediate person is later merged, deleted, or otherwise changes state, the KINSHIP's `via_chain` is not updated. The chain is a historical fact about why this KINSHIP exists, not a live pointer.
| `source` | enum | `inferred` \| `declared` |
| `created_by` | string | |
| `created_at` | timestamp | |
| `invalidation_reason` | enum \| null | |
| `invalidated_by` | string \| null | |

**In-law creation rule:** An in-law KINSHIP (`relation='in_law'`) is created when a person has a currently-valid `ROMANTIC` facet of any **committed tier** type — `married`, `engaged`, `domestic_partner`, or `separated`. It is **not restricted to marriage.** A long-term domestic partner's parent is also an in-law under this schema, reflecting real family structures where legal marriage is not the only committed relationship type.

**In-law cascade on divorce (divorce-only, not death):** When a `ROMANTIC(type='married')` facet is invalidated with `invalidation_reason='divorce'`, the cron auto-invalidates:

1. All `KINSHIP(relation IN ['parent_in_law', 'child_in_law', 'sibling_in_law'])` facets whose `via_chain` includes the divorcing pair. Invalidation reason: `'divorce_severed_in_law'`.
2. All `KINSHIP(relation='step_parent')` and `KINSHIP(relation='step_child')` facets whose `via_chain` includes the divorcing pair. Invalidation reason: `'divorce_severed_step'`.
3. All `KINSHIP(relation='step_sibling')` facets whose multi-anchor `via_chain` includes both persons of the divorcing pair as anchors. Invalidation reason: `'divorce_severed_step'`.

The underlying `grandparent` / `grandchild` / `parent` / `child` kinships (which do not run via the now-severed marriage) are untouched.

**Adoptive KINSHIPs are NEVER cascade-invalidated on divorce.** Adoption creates a legal parent-child bond that survives the dissolution of the adopting parent's marriage. `adoptive_parent` / `adoptive_child` / `adoptive_sibling` edges remain valid regardless of what happens to the marriage that originally triggered them.

**The cascade is divorce-specific. It does NOT fire on death or on breakup of a non-marriage ROMANTIC.** Rationale: divorce represents an active decision to end the relationship and the family structure that came with it. Death and uncoupling of a domestic partnership do not "sever" in-law or step kinships in the same way — culturally and practically, a widow remains her late spouse's mother's daughter-in-law-by-memory, and Orbiter preserves that by leaving the KINSHIPs valid. If the relationship truly ended, a user or LLM can declare-invalidate the facets manually.

| `invalidation_reason` | enum \| null | `divorce`, `breakup`, `death`, `data_correction`, `superseded` |
| `invalidated_by` | string \| null | |

**Type upgrades create a new row, not a mutation.** When a relationship progresses from `dating` → `engaged` → `married` (or `dating` → `domestic_partner`, etc.), the existing ROMANTIC facet is **invalidated** with `invalidation_reason='superseded'` and a **new ROMANTIC facet is created** with the upgraded type. This matches the "facets never mutate on state change" principle and produces a clean historical chain for queries like "when did Dana and Eli start dating" vs "when did they get engaged."

Example:
```
// Dating from 2023-05-01:
ROMANTIC row 1: { type: 'dating', valid_from: 2023-05-01, valid_to: 2025-09-15,
                  invalidation_reason: 'superseded' }

// Upgraded to domestic_partner on 2025-09-15 (moved in together):
ROMANTIC row 2: { type: 'domestic_partner', valid_from: 2025-09-15, valid_to: null,
                  is_current: true }
```

Both rows coexist. The history is queryable by walking the chain of superseded facets.

**Effect on death.** When one party in a valid `ROMANTIC` has `deceased_at` set on the `Person` node, the `deceased_freezer` cron automatically invalidates this facet with `valid_to = deceased_at` and `invalidation_reason = 'death'`. The `is_current` flag flips to `false` atomically with the invalidation.

**Effect on other facets on the same pair:**
- `BONDED` on the same pair does **not** auto-invalidate on death. It remains valid with a frozen `strength_score`. Semantically, "Sarah was bonded with her late husband" stays true.
- `RESIDED_AT` for the deceased person has `move_out = deceased_at` set automatically. The surviving partner's `RESIDED_AT` is unchanged; the `Household` node stays active if any residents remain.
- `KINSHIP(relation='in_law')` edges whose `via_chain` includes the deceased person do **not** cascade-invalidate on death (unlike divorce). See [KINSHIP — cascade rules](#kinship-family-overlay).

## `ESTRANGED` — negative-valence overlay

| | |
|---|---|
| **Pattern** | `(Person)-[:ESTRANGED]->(Person)` (canonicalized) |
| **Directionality** | Undirected |
| **Cardinality** | At most one valid per pair at a time |
| **Created when** | Inferred: ALL THREE of (interaction dropoff ≥90% MoM) + (shared context terminus within ±60d) + (negative external signal). Declared: user/LLM action. |
| **Invalidated when** | Inferred: ≥3 interactions within rolling 90d across ≥2 modalities AND LLM sentiment neutral-or-positive. Declared: only by user/LLM action. |
| **Weight** | **No own weight field.** When valid, adds a penalty of `severity × 5` (range +5 to +25) to the `KNOWS` edge of the same pair. Severity 1 = mild drift; severity 5 = active hostility. |

| Property | Type | Description |
|---|---|---|
| `uuid` | string | Edge identifier |
| `valid_from` | timestamp | |
| `valid_to` | timestamp \| null | |
| `source` | enum | `inferred` \| `declared` |
| `reason` | enum | `divorce`, `cofounder_split`, `legal`, `falling_out`, `drift`, `unknown` |
| `severity` | integer (1–5) | 1 = drifted apart; 5 = active hostility/litigation |
| `confidence` | float | Derived for inferred; 1.0 for declared |
| `evidence` | array | The specific signals that triggered creation |
| `created_by` | string | |
| `created_at` | timestamp | |
| `invalidation_reason` | enum \| null | `reconciliation`, `data_correction`, `deceased` |
| `invalidated_by` | string \| null | |

**Effect on `BONDED`:** When an inferred ESTRANGED is created, any valid inferred `BONDED` facet on the same pair must be invalidated at the same timestamp, with `invalidation_reason='superseded'`. Declared BONDED is untouched (requires declared action to invalidate).

**Effect on `strength_score`:** None. The score continues to decay normally. The `peak_strength_score` field captures historical high. This is distinct from the deceased case, which uses hard freeze.

## `MENTOR` — directed coaching relationship

**Unified with the existing production `MENTOR` edge.** Orbiter's YC pipeline (`mvp/yc/process-yc-people` #12700) already creates `MENTOR` edges in production when ingesting YC batch data (mentor = YC partner; mentee = batch founder). Known extends this edge with lifecycle, source, weight, and formality fields to turn it into a general-purpose facet. See [Migration — v1 → v2](#migration-v1-v2) below for the backward-compatible field additions and the backfill cypher.

| | |
|---|---|
| **Pattern** | `(Mentor:Person)-[:MENTOR]->(Mentee:Person)` |
| **Directionality** | **Directed** (not canonicalized). The only directed facet in Known. |
| **Cardinality** | Multiple rows allowed across time (a mentorship that ends and later resumes creates a second row). At most one *currently valid* row per (mentor, mentee, company) tuple. |
| **Weight range** | Tiered by `formality` AND modulated by `source`. Formal declared: **8–16**. Formal inferred: **10–18**. Informal declared: **13–22**. Informal inferred: **15–25**. Declared always ranks above inferred at the same formality. |
| **Existing production usage** | YC partner mentorships created by `mvp/yc/process-yc-people` (#12700), scoped to a single company via `company_uuid` / `company_name` properties. |

### Properties (v2 — the unified schema)

| Property | Type | Description | Present in v1? |
|---|---|---|---|
| `uuid` | string | Edge identifier | ✅ |
| `description` | string | Human-readable sentence, e.g. "Patricia mentored Alice as the Y Combinator partner for FooCorp." | ✅ |
| `company_uuid` | uuid \| null | FK to the `Company` node this mentorship was scoped to, if applicable. Optional — general mentorships (a senior engineer mentoring a junior outside a company context) leave this null. | ✅ |
| `company_name` | string \| null | Denormalized company name for display. | ✅ |
| `valid_from` | timestamp | When the mentorship began (domain time). For YC-sourced edges, this is the YC batch start date. | ❌ new in v2 |
| `valid_to` | timestamp \| null | Null = currently valid. Set when the mentorship ends. | ❌ new in v2 |
| `source` | enum | `inferred` \| `declared`. YC-ingested edges are `inferred`. User-declared mentorships are `declared`. | ❌ new in v2 |
| `formality` | enum | `formal` \| `informal`. Formal = structured, scheduled, explicit mentor/mentee roles (YC partnerships, formal advisor arrangements). Informal = organic knowledge-sharing relationships. | ❌ new in v2 |
| `domain` | string \| null | What the mentorship is about. Examples: `'accelerator'` (YC), `'software engineering career'`, `'startup fundraising'`, `'product leadership'`. Null if general. | ❌ new in v2 |
| `weight` | float | Computed from `strength_score` within the `(formality, source)` range. See [Weight model](/guides/known/schema-design#weight-model). | ❌ new in v2 |
| `created_by` | string | Cron name, user id, or LLM identifier. YC-sourced edges use `'mvp/yc/process-yc-people'`. | ❌ new in v2 |

| `invalidation_reason` | enum \| null | When `valid_to` is set: `ended`, `data_correction`, `superseded`, `reformalized` | ❌ new in v2 |
| `invalidated_by` | string \| null | | ❌ new in v2 |

### Creation rules

**YC partner mentorships (inferred, formal).** The existing `mvp/yc/process-yc-people` pipeline creates one `MENTOR` edge per (YC partner, founder, company) triple during company enrichment. When upgraded to v2, each creation sets:
- `source='inferred'`
- `formality='formal'`
- `domain='accelerator'`
- `valid_from` = the company's founded year (or YC batch start date, if available)
- `company_uuid`, `company_name` as before

**User-declared mentorships.** A user marks "I'm mentoring Nate" via the frontend → new `MENTOR` edge with `source='declared'`, `formality='formal'` (default, user can override to `informal`), `domain` optional.

**LLM-inferred informal mentorships.** The LLM relationship classifier can propose a `MENTOR` edge with `source='inferred'`, `formality='informal'` when it detects asymmetric knowledge-transfer interaction patterns (e.g. mostly one-directional coaching emails, scheduled 1:1s, code review patterns where one person consistently reviews the other). Conservative inference bar.

### Invalidation rules

**Declared `MENTOR` edges** are only invalidated by explicit user action (`data_correction` or `ended`). The cron never touches declared mentorships.

**Inferred `MENTOR` edges** may be invalidated by the [`mentor_invalidator`](/guides/known/scope/lifecycle#known-lifecycle-mentor-invalidator-cron) cron after 180 days of no direct interaction between the mentor and mentee pair. Reason: `ended`.

**Reformalization:** If an existing inferred informal `MENTOR` is promoted by the user to `formal` + `declared`, the existing row is invalidated with `invalidation_reason='reformalized'` and a new row is created. Same pattern as [ROMANTIC type upgrades](#romantic-romantic-relationship-overlay).

### v1 cypher (current production, YC-only)

The current YC ingest produces this cypher fragment (from `mvp/yc/process-yc-people` #12700). Documented here as the baseline for the migration:

```cypher
-- v1 MENTOR cypher (as generated by mvp/yc/process-yc-people #12700)
-- Current production behavior — idempotent MERGE per (mentor, founder, company) tuple.
MATCH (mentor:Person {uuid: "<ycPartnerUuid>"})
UNWIND [<founderList>] AS founder
MATCH (founderPerson:Person {uuid: founder.node_uuid})
MERGE (mentor)-[m:MENTOR]->(founderPerson)
SET m.uuid         = coalesce(m.uuid, randomUUID()),
    m.description  = mentor.name + ' mentored ' + founderPerson.name +
                     ' as the Y Combinator partner for ' + "<companyName>" + '.',
    m.company_uuid = "<companyNodeUuid>",
    m.company_name = "<companyName>"
```

**Limitations of v1:**
- No `valid_from` / `valid_to` — no lifecycle tracking
- No `weight` — not interleavable with other Known ranked queries
- No `source` / `formality` / `domain` — can't distinguish from other mentorship types
- `MERGE` produces one row per tuple; no historical chain

### v2 cypher (unified Known-aware ingest)

The upgraded ingest produces this cypher fragment. Backward-compatible: all v1 fields are preserved; new v2 fields are added alongside. Because v2 tracks lifecycle, the MERGE uses only the uniqueness key `(mentor, mentee, company, valid_from)` so that a YC ingest re-run doesn't create duplicates but a legitimate new mentorship (same pair, different time window) can.

```cypher
-- v2 MENTOR cypher (Known-aware — to be adopted by mvp/yc/process-yc-people #12700)
MATCH (mentor:Person {uuid: "<ycPartnerUuid>"})
UNWIND [<founderList>] AS founder
MATCH (founderPerson:Person {uuid: founder.node_uuid})

-- MERGE key includes valid_from to allow historical re-mentorships
MERGE (mentor)-[m:MENTOR {
  company_uuid: "<companyNodeUuid>",
  valid_from:   <companyFoundedYear or YC batch start date>
}]->(founderPerson)
ON CREATE SET
  m.uuid                = randomUUID(),
  m.description         = mentor.name + ' mentored ' + founderPerson.name +
                          ' as the Y Combinator partner for ' + "<companyName>" + '.',
  m.company_name        = "<companyName>",
  m.valid_to            = null,
  m.source              = 'inferred',
  m.formality           = 'formal',
  m.domain              = 'accelerator',
  m.created_by          = 'mvp/yc/process-yc-people',
  m.created_at          = datetime(),
  m.invalidation_reason = null,
  m.invalidated_by      = null,
  m.weight              = 14  -- midpoint of formal + inferred range (10–18), refined by scoring cron later

ON MATCH SET
  -- Only update mutable fields on re-runs; do NOT overwrite lifecycle state
  m.description  = mentor.name + ' mentored ' + founderPerson.name +
                   ' as the Y Combinator partner for ' + "<companyName>" + '.',
  m.company_name = "<companyName>"
```

**Key differences vs v1:**
- `valid_from` added to the MERGE key so new time-windowed rows can coexist with historical ones
- `ON CREATE SET` populates all new v2 fields with YC-appropriate defaults
- `ON MATCH SET` only updates the mutable display fields (`description`, `company_name`); lifecycle state is immutable after creation
- Initial `weight` is set to 14 (midpoint of formal-inferred range) as a bootstrap; the scoring cron computes the real weight on next pass

### Migration (v1 → v2)

<Note>
**Not required for the current Known rollout.** Known is being deployed on a **fresh graph** where no v1 MENTOR edges exist. The migration cypher below is retained as a reference pattern — for any future environment where v1 edges do exist, or if production data ever gets restored from a pre-Known backup, the same backfill approach applies. The initial rollout skips this step entirely and goes straight to the v2 ingest.
</Note>

The following one-time backfill cypher upgrades existing v1 edges to v2 in place. Run ONCE before enabling the v2 ingest, inside a bounded batch (recommend 1000 edges per pass).

```cypher
-- v1 → v2 MENTOR edge backfill (run once)
-- Upgrades existing YC-created MENTOR edges with v2 fields and conservative defaults.
MATCH ()-[m:MENTOR]->()
WHERE m.valid_from IS NULL   -- only v1 edges

-- Look up the company's founded year if available, otherwise fall back to 2020-01-01
OPTIONAL MATCH (c:Company {uuid: m.company_uuid})

SET m.valid_from          = coalesce(datetime({year: c.founded}), datetime('2020-01-01')),
    m.valid_to            = null,
    m.source              = 'inferred',
    m.formality           = 'formal',
    m.domain              = 'accelerator',
    m.created_by          = 'migration:v1_to_v2_mentor_backfill',
    m.created_at          = datetime(),
    m.invalidation_reason = null,
    m.invalidated_by      = null,
    m.weight              = 14   -- bootstrap; scoring cron will refine on next pass

RETURN count(m) AS edges_upgraded
```

**Migration notes:**
- The backfill sets `valid_from` to the company's founded year when available, falling back to `2020-01-01` as a safe historical default. Tune this fallback as needed.
- All existing edges are conservatively marked `source='inferred'`, `formality='formal'`, `domain='accelerator'` — correct for all YC-originated edges.
- The initial `weight=14` is a midpoint bootstrap. The scoring cron will recompute it to the correct value per the strength_score derivation on next pass.
- The migration does not delete or duplicate any edges. `company_uuid`, `company_name`, `description`, and `uuid` are preserved exactly.
- After running, all v1 edges become valid v2 edges and both the legacy YC ingest (until it's updated) and the new Known-aware code paths can read them safely.

### Historical note

Before the unification, Known's design temporarily used `MENTORSHIP` as the edge name (because the existing `MENTOR` edge was undocumented and overlooked during initial design). On discovery that `mvp/yc/process-yc-people` already produces `MENTOR` edges in production, the design was corrected to use `MENTOR` for the unified edge type.

## Introductions are nodes, not edges

There is no `INTRODUCED` edge type. Introductions are represented as [`Introduction` nodes](/guides/known/schema-design/nodes#introduction) connected to the introducer via `MADE_INTRODUCTION` and to the participants via `CONNECTS`. This was a deliberate design choice: FalkorDB does not support edges pointing at other edges (so an `(Introducer)-[:INTRODUCED]->(knows_edge)` pattern is impossible), and introductions are naturally N-ary.

See [Node types → Introduction](/guides/known/schema-design/nodes#introduction) for the full shape.
