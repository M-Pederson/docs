title: "Parse Functions"
description: "Email activity parsing functions — parse-from, parse-to, and parse-cc"
---

Called by `message-created-parse` (#1913) to create email activity records based on the user's role in the conversation. Each function follows the same core pattern: extract participant arrays, calculate participant count, iterate through each participant list, resolve identity, create or get-add person, log activity, increment counters, and set primary email.

## parse-from — #2126

Processes outbound emails (sent FROM the user's account). Creates activity records for every recipient.

```text
mvp/activity/parse-from — #2126
```

### Step-by-step

<Steps>
  <Step title="Extract participant arrays">
    Pulls `to`, `cc`, and `bcc` arrays from the webhook payload's `object` field.
  </Step>
  <Step title="Calculate participant count">
    Calls `calculate_participant` to get total participant count across all arrays.
  </Step>
  <Step title="Process TO recipients (activity_type 2)">
    For each TO recipient: resolves identity via `check-identity-email`, skips marketing emails, then either logs activity for an existing person or creates a new person via `name-format` + `master-person` get-add before logging. Creates `email_activity` with type 2 ("Email to [name]"), increments `total_email_activity` on `my_person`, and calls `set-primary-email_v2`.
  </Step>
  <Step title="Process CC recipients (activity_type 3)">
    Same pattern as TO but with activity_type 3 ("[name] is cc'd"). Calls `set-primary-email_v2` async.
  </Step>
  <Step title="Process BCC recipients (activity_type 4)">
    Same pattern as TO/CC but with activity_type 4 ("[name] is bcc'd"). Calls `set-primary-email_v2` async.
  </Step>
</Steps>

## parse-to — #2129

Processes inbound emails (sent TO the user's account). Creates activity records for the sender and all other recipients. Has a unique `$logActivity` flag that suppresses downstream activity logging when the sender is a brand-new person.

```text
mvp/activity/parse-to — #2129
```

### Step-by-step

<Steps>
  <Step title="Get grant info with user addon">
    Fetches `nylas_grant` with a `user:user_id` addon to get the user's `master_person_id`.
  </Step>
  <Step title="Extract participant arrays + calculate count">
    Pulls `to`, `cc`, `bcc` arrays and calls `calculate_participant`.
  </Step>
  <Step title="Process FROM sender (activity_type 1)">
    Resolves the sender via `check-identity-email-to`. Three paths: (a) existing person with my_person — log activity directly, (b) brand-new person — create via get-add, log activity, parse email signature async, then set `$logActivity = false` to suppress further logging for this message, (c) marketing — skip.
  </Step>
  <Step title="Process TO recipients (activity_type 2)">
    Skips the user's own email. Four paths: (a) existing person with my_person — log if `$logActivity` is true, (b) master_person exists but no my_person — copy-to-my then log, (c) brand-new person with logging enabled — create and log, (d) brand-new person with logging suppressed — just ensure person exists without creating activity.
  </Step>
  <Step title="Process CC recipients (activity_type 3)">
    Same 4-path pattern as TO. Skips user's own email.
  </Step>
  <Step title="Process BCC recipients (activity_type 4)">
    Same 4-path pattern as TO/CC. Skips user's own email.
  </Step>
</Steps>

<Accordion title="Full XanoScript — parse-to #2129">

// parse & log email when it is sent TO user's account
function "mvp/activity/parse-to" {
  input {
    json participants?
    int user_id?
  }

  stack {
    // ─── SETUP: get grant info ───
    db.get nylas_grant {
      field_name = "grant_id"
      field_value = $input.participants.object.grant_id
      output = ["id", "user_id", "grant_id", "email_account", "ai_enabled"]
      addon = [
        {
          name  : "user:user_id"
          output: ["master_person_id"]
          input : {user_id: $input.user_id}
        }
      ]
    } as $nylasGrant

    // ─── SETUP: extract participant arrays ───
    var $grantID {
      value = $input.participants|get:"grant_id":null
    }

    var $logActvity {
      value = true
    }

    var $to {
      value = $input.participants.object|get:"to":[]
    }

    var $cc {
      value = $input.participants.object|get:"cc":[]
    }

    var $bcc {
      value = $input.participants.object|get:"bcc":[]
    }

    // ─── SETUP: calculate participant count ───
    function.run "mvp/activity/calculate_participant" {
      input = {
        to  : $input.participants.object.to
        from: $input.participants.object.from
        cc  : $input.participants.object.cc ?: []
        bcc : $input.participants.object.bcc ?: []
      }
    } as $participant_count

    // ═══════════════════════════════════════
    // PROCESS FROM — who sent the email (activity_type 1)
    // ═══════════════════════════════════════
    group {
      stack {
        var $fromEmail {
          value = $input.participants.object.from|first|get:"email":null
        }

        function.run "mvp/identity/check-identity-email-to" {
          input = {email_address: $fromEmail, user_id: $input.user_id}
        } as $identityEmail

        // skip marketing emails
        conditional {
          if ($identityEmail|get:"marketing":null) {
            continue
          }
        }

        conditional {
          // ── existing person with my_person: log activity ──
          if ($identityEmail.my_person_id != null && $identityEmail.master_person_id != null) {
            group {
              stack {
                db.get master_person {
                  field_name = "id"
                  field_value = $identityEmail.master_person_id
                  output = ["id", "name"]
                } as $masterPerson

                db.add email_activity {
                  data = {
                    created_at        : "now"
                    user_id           : $input.user_id
                    my_person_id      : $identityEmail.my_person_id
                    activity_type_id  : 1

## parse-cc — #2132

Processes emails where the user is CC'd or BCC'd. Creates activity records across all four participant lists (TO, FROM, CC, BCC) from the CC perspective.

```text
mvp/activity/parse-cc — #2132
```

### Step-by-step

<Steps>
  <Step title="Get grant info">
    Fetches `nylas_grant` to identify the user's email account.
  </Step>
  <Step title="Extract all four participant arrays + calculate count">
    Pulls `to`, `from`, `cc`, and `bcc` arrays and calls `calculate_participant`.
  </Step>
  <Step title="Process TO participants (activity_type 2)">
    Skips the user's own email. For each recipient: resolves identity, skips marketing, then either logs for existing person (and triggers async signature parsing) or creates new person via get-add before logging.
  </Step>
  <Step title="Process FROM participants (activity_type 1)">
    Skips the user's own email. Same pattern — creates "Email from [name]" activities. Triggers async signature parsing for all senders.
  </Step>
  <Step title="Process CC participants (activity_type 3)">
    Skips the user's own email. Creates "[name] is cc'd" activities with same resolve/create pattern.
  </Step>
  <Step title="Process BCC participants (activity_type 4)">
    Skips the user's own email. Creates "[name] is bcc'd" activities with same pattern.
  </Step>
</Steps>
