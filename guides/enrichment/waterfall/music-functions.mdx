## Foundation

<Card icon="gauge-high">

```
mb/rate-limit/acquire ŌĆö 13079
```

The global 1 req/s token gate, backed by the `mb_rate_limit` singleton (#736). If the last granted call was less than 1100 ms ago it sleeps 1 s, then stamps `last_call_at`. **Fail-open** ŌĆö if the gate itself errors, the call proceeds and the failure is logged, so a broken limiter never halts the pipeline.

</Card>

<Card icon="cloud-arrow-up">

```
mb/api/request ŌĆö 13080
```

Single choke-point for **every** MusicBrainz call. Calls `acquire` for pacing, sends the canonical `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`), appends `fmt=json`, and retries up to 5 attempts with backoff sleeps between attempts (none after the final one). `400`/`404` are terminal ŌĆö no retry. Returns `{ok, terminal, status, attempts, json}`; failures `log_crash` with phases `api_bad_request` / `api_exhausted`.

</Card>

## Queue upserts

<Card icon="wrench">

```
mb/queue/upsert-artist ŌĆö 13081
mb/queue/upsert-recording ŌĆö 13082
mb/queue/upsert-label ŌĆö 13083
mb/queue/upsert-event ŌĆö 13084
mb/queue/upsert-release-group ŌĆö 13085
mb/queue/upsert-work ŌĆö 13086
```

Single source of truth for queue writes, keyed on `mbid`. Duplicate discovery ŌåÆ `count++`, **lowest tier wins**, and the source-table FK is filled once the stub exists.

Inputs default via `first_notnull` (tier 4 / depth 1) so an explicitly passed `0` survives ŌĆö this is what lets bootstrap-lane rows (`priority_tier = 0`, `cascade_depth = 0`) be queued without the defaults coercing them.

</Card>

## Orchestrators

Every phase is wrapped in `try_catch` ŌåÆ `log_crash` (phases `mb_lookup` / `mb_node` / `mb_edges` / `mb_browse` / `mb_cascade`). A terminal `404`/`400` marks the source row `gone`. All edges stamp `e.uuid` (plus the full [standard music edge envelope](/guides/ontology/edges#music-standard-edge-envelope)). Cascading is **inlined** in each orchestrator's `mb_cascade` phase ŌĆö the standalone `cascade-from-*` helpers from the design were not built.

<Card icon="layer-group">

```
mb/run-base-label-enrich ŌĆö 13091
```

Lookup `label/{mbid}?inc=url-rels+label-rels+ratings+tags+genres` ŌåÆ `Company:Music_Label` node ŌåÆ `SUBSIDIARY_OF` (w15, stub-MERGEs the other label). Then browses `release?label=`, capped at **3 pages / 300 releases** (`inc=artist-credits+release-groups`), queueing release-groups T2, credited artists T2, and ownership labels T1.

</Card>

<Card icon="layer-group">

```
mb/run-base-artist-enrich ŌĆö 13094
```

Lookup `artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels+event-rels` ŌåÆ artist/group node ŌåÆ `MEMBER_OF` edges (w5 current / w12 past, with `instruments`, `begin`/`end`/`ended`, `original`). Cascade: members T1, Albums capped at 20 chronological T2, recordings capped at 50 T2, and (v1.1, 2026-06-12) **main-performer events T3 ŌĆö queued only when the artist itself is a depth-0 seed** (the events enter the queue as depth-1 children, so their own artist fanout stops at depth 2, per the depth model). Event discovery rides the same lookup: `inc=event-rels` embeds the full event objects, so no extra browse call is needed. Collaborators are still deferred to v2. **v1.3 (2026-06-14):** a depth-0 `Person:Music_Artist` is enqueued for [person-resolution](#person-resolution) via `mb/queue/upsert-person-resolution` (13122) ŌĆö the same depth-0 gate as the event cascade, so only seed artists run it. **v1.4 (2026-06-15):** those same depth-0 Person rows now also get their serper avatar **inline at ingest** via `fetch-person-avatar` (13120, non-fatal) ŌĆö previously the avatar only came from the person-resolution drain, which ships inactive, so directly-seeded artists came in avatar-less.

</Card>

Edges: `RELEASED` w10 (`joinphrase`), `RELEASE_OF` w30, `RELEASED_ON` w50 (`catalog_number`), `APPEARS_ON` w45 (`track_number`/`position`/`disc`). Cascade: tracklist recordings T3.

</Card>

<Card icon="layer-group">

```
mb/run-base-recording-enrich ŌĆö 13100
```

Edges: `PERFORMED` w10, `PLAYED_ON` w20 (`instruments`), `PRODUCED` w18, `ENGINEERED` w22 (`role`), `RECORDING_OF` w30 (`live`/`partial`/`cover`), and `RECORDED_AT` w35 ŌĆö with `Place` nodes **materialized inline** from the embedded `place-rels` + `mb_place` rows, per [decision 6](/guides/enrichment/waterfall/music-industry-waterfall#resolved-design-decisions-2026-06-11). (The artist-credit `PERFORMED` handler already split by type; the rel-credit handler didn't ŌĆö caught on Nirvana, see the [sandbox test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run).)

Cascade: headliners T1, support/guest artists T2, guarded `childDepth <= 2`.

## Unified queue drain

<Card icon="wrench">

```
mb/queue/process-next ŌĆö 13102
```

Picks the **single best row across all 6 queues** ordered by `priority_tier ASC, cascade_depth ASC, entity_rank ASC, id ASC` ŌĆö `entity_rank` is a constant per queue (artist 1, label 2, **event 3**, work 4, release_group 5, recording 6) ŌĆö skipping rows whose `next_attempt_at` is in the future. Locks the row, dispatches the matching orchestrator, deletes on success. v1.1 (2026-06-12) wired `queue_mb_event` into the scan, the candidate ranking, and dispatch (ŌåÆ 13105), filling the rank-3 slot reserved at v1.

On failure: `retry_count++` with a `next_attempt_at` backoff of 5 min ├Ś retries; the row is dropped after 5 failures.

</Card>

## LLM edge descriptions

Mirrors the IMDB `WORKED_ON` trio (#13016ŌĆō#13018), but **one generic trio covers all music edge types** ŌĆö keyed on `e.uuid` rather than one trio per edge type. Model: **`deepseek/deepseek-v4-flash`** (temp 0.3, `reasoning:{enabled:false}` + `max_tokens:200`), producing a single factual 15ŌĆō30-word sentence. All descriptions obey **[GOOD-or-null](/guides/dev-updates/2026-06-16-music-node-descriptions#good-or-null-invariant)** ŌĆö a clean sentence or `null`, never a refusal/meta echo.

<Card icon="cloud-arrow-up">

```
mb/edge/create-music-edge-description ŌĆö 13095  (v1.4)
```

Given one edge's structured properties (opaque `via_release` mbid stripped) + endpoint node names, calls **`deepseek-v4-flash`** (reasoning off + `max_tokens:200`) and returns the single-sentence description **after cleanup**: strip wrapping quotes + leading meta-preambles, normalize U+2011/U+2010 ŌåÆ `-`, and a GOOD-or-null reject guard (refusal/meta ŌåÆ empty). **v1.4 (2026-06-17)** swapped the model from `gpt-oss-120b`, which dropped spaces ~15% of the time ("Stevensis", "Themusicgroup"). Sandbox sample: *"Sufjan Stevens is a member of the music group Danielson."*

</Card>

<Card icon="wrench">

```
mb/edge/write-music-edge-description ŌĆö 13096
```

Stamps `e.description` + `e.description_generated_at` (ISO) onto the edge via Cypher, matched on `e.uuid`.

</Card>

<Card icon="clock-rotate-left">

```
mb/tool/backfill-music-edge-descriptions ŌĆö 13097  (legacy name; forward async describer)
```

Async describer (1ŌĆō100 per run, default 10). Selects newly written edges where `description_generated_at IS NULL`, filtered by edge type (allowlist ŌĆö `ON_RECORD_LABEL` added v1.3, **`OWNED_BY` added v1.5** by the [label-owner](/guides/enrichment/waterfall/music-reconciliation#build-spec--label-owner-resolution-owned_by) session) + `labels()` membership, and runs the create ŌåÆ write pair per edge. Idempotent via the timestamp guard. **v1.6 (2026-06-17) perf:** the SELECT now anchors on the relationship types (`MATCH (a)-[e:T1|T2|ŌĆ”]->(b)`) so FalkorDB scans only the music-edge matrices, not the entire people+company edge set ŌĆö cut a live batch SELECT from >280 s (timeout) to ~3 s.

</Card>

Built 2026-06-12 (second session). Both store the original URL *and* a re-hosted GCS WebP URL ŌĆö `cover_art_url` / `gcp_cover_art_url` on `mb_recording` (#738) and `music_group_image_url` / `gcp_music_group_image_url` on `mb_artist` (#737). In forward-path docs, these are single-entity fetch helpers called directly or by the finalizer; batch sweeps are excluded.

<Card icon="wrench">

## Node descriptions

Built 2026-06-12, consolidated behind `finalize-music-node` on 2026-06-16. **`Recording`**, **`Music_Group`**, and **`Music_Artist`** nodes get generated descriptions, and their forward-path vectors are generated from that description. The other music node types (`Work`, `Release_Group`, `Release`, `Event`, `Place`, and `Company:Music_Label`) do **not** get generated descriptions today; they receive dual vectors from `name + disambiguation` instead. All generated descriptions obey the **[GOOD-or-null invariant](/guides/dev-updates/2026-06-16-music-node-descriptions#good-or-null-invariant)**: a clean Ōēż127-char factual sentence beginning with the entity name, or exactly `null` ŌĆö never an LLM refusal/meta-commentary echo.

<Card icon="cloud-arrow-up">

```
mvp/music/create-node-description ŌĆö 13112  (v2.2)
```

Generate (don't write) the LLM description for one node from a **slim, type-aware field-set** (not raw JSON ŌĆö raw-JSON@8000 was what made deepseek echo *"please provide the record"*). One OpenRouter call ŌĆö **`deepseek/deepseek-v4-flash`**, temp 0.2, **`reasoning: {enabled: false}` + `max_tokens: 200`** ŌĆö producing **one factual sentence Ōēż127 chars** that **begins with the entity name**, so the description-based embedding keeps the name signal. The field-set is passed as a JSON *user* message built in a lambda (not `util.template_engine` ŌĆö avoids the markdown-autolinker URL burn from the IMDB build). **Two GOOD-or-null guards (v2.2):** an **input-empty short-circuit** returns `null` before the LLM is called on a missing/empty record, and an **output reject-guard** strips quotes, hard-truncates to 127, and maps any refusal/meta-commentary (first-person + `the user`/`provide`/`structured`/`musicbrainz` markers) to `null`. The guard lives in the lambdas, **not** the prompt.

<Warning>
The reasoning flag is load-bearing: deepseek-v4-flash is a reasoning model, and without it the completion budget goes to thinking tokens and `message.content` comes back **empty** on larger records ŌĆö found live during the Cinq smoke test (3 of 4 rows silently skipped) and fixed the same day. CLAUDE.md trap #13.
</Warning>

</Card>

<Card icon="wrench">

```
mvp/music/finalize-music-node ŌĆö 13117
```

Forward finalizer for one node. It validates or generates the node `description` for `recording`, `music_group`, and `music_artist`, rejects bad LLM output as null, writes both vectors (`name_embedding` via #4676 and `embeddings` via #13114), and runs the relevant media side effect where applicable (recording cover art / group image / person avatar path). For `other` node kinds it skips LLM description and writes vectors from `name + disambiguation`.

</Card>

## Node embeddings

Forward writers/finalizers dual-write Gemini `embeddings` and OpenAI `name_embedding` for the node types they touch. The model provenance is fixed: `name_embedding` uses **OpenAI `openai/text-embedding-3-small`** via `vectors/create-vectors-string` #4676; `embeddings` uses **Google `google/gemini-embedding-001`** via `vectors/create-gemini-vectors` #13114.

<Warning>
**Forward-only vector coverage fixed 2026-07-01:** `Event` and inline `Place` now call `finalize-music-node` #13117 with `kind=other` where they are created. `Work`, `Release_Group`, and canonical `Release` already used that same path.
</Warning>

## ON_RECORD_LABEL edges

Built 2026-06-17. A **graph-derived forward edge writer** materializes the [`ON_RECORD_LABEL`](/guides/ontology/edges#music-structural-labels) edge (`(Recording)ŌåÆ(Company:Music_Label)`, weight **55**) ŌĆö a one-hop convenience shortcut of the existing `(Recording)-[:APPEARS_ON]->(Release)-[:RELEASED_ON]->(Company:Music_Label)` path. **No MusicBrainz fetch** ŌĆö the recording lookup (#13100) never pulls label/release data, so the edge is derived purely in-graph from the canonical release's label (only canonical releases materialize as `Release` nodes, so the path *is* the canonical-release label).

**Forward-wired (2026-06-17).** Despite the legacy helper name, the release-group orchestrator `mb/run-base-release-group-enrich` (**#13099 v1.2**) calls #13131 with the new `release_mbid` scope right after writing the canonical Release + `RELEASED_ON` + `APPEARS_ON` edges ŌĆö guarded on a non-empty canonical release mbid, in its own non-fatal `try_catch` ŌĆö so every newly-enriched release-group **auto-derives** `ON_RECORD_LABEL` for its tracks.

<Card icon="wrench">

```
mvp/music/backfill-on-record-label-edges ŌĆö 13131  (legacy name; release-scoped forward writer)
```

Single idempotent Cypher via `mvp/falkor/send-cypher`: `MATCH (rec:Recording)-[:APPEARS_ON]->(rel:Release)-[ro:RELEASED_ON]->(lbl:Company:Music_Label) MERGE (rec)-[r:ON_RECORD_LABEL]->(lbl)` ŌĆö `ON CREATE SET` stamps `uuid`, `created_at`, and semantic-null `description` / `description_generated_at`; every run sets `weight = 55`, `updated_at`, `via_release = rel.mbid`, and `catalog_number = ro.catalog_number` (carried from the canonical release's `RELEASED_ON`). `MERGE` per `(recording, label)` pair ŌåÆ idempotent, no duplicates. Forward scope is **`release_mbid`**, used by #13099 immediately after canonical release materialization. The async edge describer fills `description`.

</Card>

## Person-resolution

Built 2026-06-14. The person-side mirror of the labelŌåÆCompany bridge ŌĆö resolves a `Person:Music_Artist` node to a `master_person` and unlocks the social/people enrichment that MB artists otherwise miss (they carry no `linkedin_url`). A depth-0 `Person:Music_Artist` is enqueued at enrich time; a **standalone cron** drains the queue and, per artist, runs Exa social discovery cross-checked against the MB url-rels + aliases, mints/links a `master_person` carrying **all** verified socials as `master_link` rows (which auto-fire the twitter/instagram/youtube/linkedin scrapers + `social_insights` + influence), `SAME_AS`-bridges `Music_Artist ŌåÆ master_person`, stamps `mb_artist.master_person_id`, annotates the node `social_links`, and sets a serper.dev avatar. Full writeup: [dev update](/guides/dev-updates/2026-06-14-person-resolution). It builds the person half of [Music Reconciliation](/guides/enrichment/waterfall/music-reconciliation) **as a bridge, not a merge**.

<Card icon="cloud-arrow-up">

```
mvp/music/find-person-social ŌĆö 13118
```

Exa social discovery for one person ŌĆö the mirror of `find-label-domain`. Searches Exa, then classifies the results into `linkedin[]`, `socials[]`, and `others[]` for the orchestrator's cross-check step.

</Card>

<Card icon="wrench">

```
mvp/music/save-person-avatar ŌĆö 13119
```

GCS converter (the avatar-saver pattern): converts the source image to WebP and uploads it to `music-person-avatar/<mbid>.webp`, returning the public GCS URL.

</Card>

One person, end-to-end ŌĆö the mirror of `fetch-group-image` (13109) with the gate **inverted to persons** (`artist_type = Person` only). serper.dev images search ŌåÆ saves the avatar ŌåÆ `SET n.image_url` on the `Person:Music_Artist` graph node.

<Card icon="play">

```
mb/run-person-resolution ŌĆö 13121
```

The orchestrator. Builds a verification set from `mb_artist.json` (known social URLs from `relations[]`, aliases / legal names from `aliases[]`, area / birth-year / genres) ŌåÆ calls Exa (13118) ŌåÆ **cross-checks** candidates against the MB set to defeat same-name false positives ŌåÆ mints/links a `master_person` via `mvp/get-add/master-person-new` (#13039) passing **all verified socials in `links`** (each written as a `master_link` row, auto-firing the platform scrapers) ŌåÆ stamps `mb_artist.master_person_id` ŌåÆ writes the `SAME_AS` bridge `Music_Artist -w1-> master_person Person` ŌåÆ annotates the node (`SET n.social_links`, `n.social_resolved_at`) + fetches a serper avatar (13120). `hold_downstream = false` (run the full PDLŌåÆEnrichLayer people pipeline) **only when a LinkedIn is verified**; `true` otherwise ŌĆö the social scrapers still fire from the link writes.

A LinkedIn is accepted **only by content** (its profile text shows a music-identity signal ŌĆö a music-role keyword, the artist's stage name, or a genre), which defeats same-name strangers (v2.2). **v2.3 (2026-06-14):** a `deceased` flag derived from the MB `life-span` (`ended == true` or an `end` date) **skips the LinkedIn-accept branch entirely** for dead artists ŌĆö a deceased musician cannot run a LinkedIn, so any hit is an impersonator / fan / namesake. Caught live on Kurt Cobain (died 1994) during the [sandbox node-type test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run); socials still flow, only LinkedIn is blocked. **v2.4 (2026-06-14):** if the url-rels include an `imdb.com/name/nmŌĆ”` page, a dedicated `try_catch` phase calls `mvp/imdb/add-imdb-to-master-person` (#10509) **synchronously** (inherits the request data source ŌĆö avoids the asyncŌåÆlive leak) to enrich the **existing** master_person with IMDb bio/credits/awards (no duplicate; adds a Firecrawl + LLM of latency per IMDb-carrying artist). Verified on Cobain ŌĆö `imdb_person` landed on sandbox, FK-linked, no dup, and the master_person's graph `Person` node gained the IMDb `roles` / `short_biography` / `long_biography` / `avatar` (credits queue as `imdb_title` rows). See the [handoff](/guides/dev-updates/2026-06-14-person-resolution).

## Deviations & deferrals

Differences between the [design](/guides/enrichment/waterfall/music-industry-waterfall) and what was built:

| Deviation / deferral | Detail |
|---|---|
| ~~**Events deferred**~~ **Events built 2026-06-12** | `mb/get-add/event` (13106) + `mb/run-base-event-enrich` (13105) now exist and the drain (13102 v1.1) fills the reserved `entity_rank = 3` slot. Event-specific deviations: events are discovered **only from depth-0 artists** (`inc=event-rels`, main-performer rels, T3 ŌĆö no event browse pagination); `guest performer` (a third live rel type) maps to `PERFORMED_AT` with `role`; event `genres`/`tags`/`rating` are written but empty in practice on MusicBrainz. Smoke-tested on Reading Festival 1992 (all four phases green: node + sub-label, `HEADLINED`, `HELD_AT` place inline, headliner cascade). |
| **Collaborators deferred to v2** | The artist orchestrator (13094) does not queue collaborators ŌĆö members, albums, recordings, and (v1.1) depth-0 main-performer events only. |
| **Backfill-only utilities excluded** | Batch image sweeps, node/vector catch-up sweeps, area sweeps, and temporary Task 184 are intentionally out of this forward-path reference. |
| **Description finalizer rewrites vectors for three node types** | #13117 generates descriptions and description-based OpenAI/Gemini vectors for `Recording`, `Music_Group`, and `Music_Artist`. Other music nodes keep null descriptions and vectorize `name + disambiguation` through their forward writer/finalizer path. LLM-empty rows stay unstamped for a future finalization attempt. |
| **Cascade helpers inlined** | The standalone `mb/cascade-from-*` functions were not built; each orchestrator carries its own `mb_cascade` phase. |
| **`FROM_AREA` forward gap** | Geography edges are not forward-wired in the audited ingest path while sweeps are excluded. If `FROM_AREA` remains desired, wire country/city/region mapping directly into artist/label/place writes. |
| **Genre nodes retired** | Genres remain `genres` properties on music nodes for facets, badges, filters, and search text. Do not write `Entity:Genre` nodes or `HAS_GENRE` edges. |
| **Cron 60 s ├Ś 10 batch** | Design called for a ~2 s tick; Xano cron granularity is 60 s, so Task 180 drains up to 10 rows per tick instead ŌĆö approximately equivalent throughput. |
| **Recordings browse unsorted** | The artist cascade takes the **first 50** recordings as returned, not the top 50 by length/date. |
| **Label browse cap** | `release?label=` browse is capped at 3 pages / 300 releases; hitting the cap is logged. |
