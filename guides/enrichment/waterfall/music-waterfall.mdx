Queue-driven MusicBrainz enrichment with a tiered cascade, depth cap, loop prevention, and a **global 1 req/s rate limiter** ‚Äî pulling artists, groups, labels, recordings, release-groups, releases, works, events, and places into the Orbiter graph, with identity bridges to `master_person` and `Company`. This page is the architecture spec: source contract, rate-limit design, depth model, tiers, seed flows, loop prevention, and the resolved design decisions. The live function inventory, table schemas, and graph node + edge references live on the companion pages below.

It mirrors the [IMDB Waterfall](/guides/enrichment/waterfall/imdb-waterfall) ‚Äî queue tables, depth-capped cascade, priority tiers, upsert dedup ‚Äî with two structural differences: MusicBrainz is a **structured JSON API** (no HTML scrape, no LLM extraction in the structural path), and it enforces a **hard 1 request/second global limit**, which becomes the central design constraint.

The unified drain cron (Task 180) and person-resolution cron (Task 182) ship **INACTIVE** ‚Äî pausing/leaving them off is the manual kill.

## Source contract ‚Äî the MusicBrainz API

| Property | Value |
|---|---|
| Base | `https://musicbrainz.org/ws/2/` |
| Format | append `?fmt=json` (or `Accept: application/json`) |
| Auth | none required |
| **Rate limit** | **1 request/second, global**, with a meaningful `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`). Exceeding it returns `503`. |
| Lookup | `/{entity}/{MBID}?inc={subqueries}` |
| Browse | `/{result}?{linked}={MBID}&limit=100&offset=N&inc=...` |
| Search | `/{entity}?query={lucene}&limit=N` |
| Pagination | `limit` (default 25, **max 100**) + `offset`; increment by results returned |

**`inc=` is the efficiency lever.** A single lookup can pull aliases, tags, genres, ratings, and every relationship type (`artist-rels`, `recording-rels`, `work-rels`, `release-rels`, `label-rels`, `event-rels`, `place-rels`, `area-rels`, `url-rels`) in **one** request. One artist + two browse pages ‚âà 3 requests ‚âà 3 seconds. This is why the cascade must be tightly depth- and tier-capped.

The per-orchestrator `inc=` sets (the exact request each entity type makes) are on [Music Waterfall Functions ‚Üí Orchestrators](/guides/enrichment/waterfall/music-functions#orchestrators).

## Rate-limit architecture

The defining constraint ‚Äî **the 1 req/s ceiling is global, not per-cron**. Independent queue crons each firing lookups would collectively blow the limit, so every MusicBrainz call passes through one token bucket, and **one** cron drains all queues so total throughput never exceeds 1 req/s.

- **`mb/rate-limit/acquire` (13079)** ‚Äî the fail-open token gate backed by the singleton `mb_rate_limit` row (#736).
- **`mb/api/request` (13080)** ‚Äî the single choke-point for every MusicBrainz call: pacing via `acquire`, canonical `User-Agent`, `fmt=json`, the [retry policy](#retry-policy) below. Orchestrators never call `api.request` directly.
- **`mb/queue/process-next` (13102) + Task 180** ‚Äî **one unified cron**, not six. Each `process-next` call selects the single best pending **tier-0** row across all six queues (order `cascade_depth ASC, entity_rank ASC, id ASC`; nonzero tiers wait for [duplicate-discovery promotion](#priority-tiers)), locks it, dispatches the matching orchestrator, deletes on success. Task 180 runs every **60 s and makes up to 10 such calls per tick** (Xano cron granularity is 60 s; the design's ~2 s tick ‚Äî see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals)). A unified cron makes the global 1 req/s trivially enforceable; parallel crons would not.

Behavioral detail (sleep timings, retry internals, drain backoff): [Foundation](/guides/enrichment/waterfall/music-functions#foundation) and [Unified queue drain](/guides/enrichment/waterfall/music-functions#unified-queue-drain).

<Warning>
  **Throughput is intentionally low.** At ~1 entity / 3‚Äì6 s, a depth-0 artist with its top recordings + albums + members takes minutes, and a deep cascade takes hours ‚Äî by design. The depth cap (2), tier caps, and per-entity caps (20 albums, 50 recordings, 3 browse pages) keep total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.
</Warning>

## API errors, retries & rate limits

<Warning>
  **A meaningful `User-Agent` is mandatory.** Requests without one (or with a generic library default) are throttled hard or blocked. Use `Orbiter/1.0 ( mark@orbiter.io )`. There is no API key.
</Warning>

### Status codes

| Status | Meaning | Retry? | Handling |
|---|---|:--:|---|
| `200` | OK | ‚Äî | parse JSON |
| `400` | Bad request ‚Äî malformed MBID / Lucene query | ‚ùå terminal | `log_crash` `phase: api_bad_request`; drop the queue row |
| `404` | Entity not found / merged away | ‚ùå terminal | stamp `enrich_success = false`, mark stub `gone`, drop row |
| `429` / `503` | **Rate limited / temporarily unavailable** | ‚úÖ backoff | honor `Retry-After`; exponential backoff ‚Äî the common case |
| `500` / `502` / `504` | Server / gateway error | ‚úÖ backoff | transient ‚Äî retry with backoff |
| timeout / network | no response | ‚úÖ backoff | retry with backoff |

<Note>
  MusicBrainz returns **`503` for both** genuine overload **and** rate-limit violations (it does not consistently use `429`). Treat `503` as "back off and retry," never as a hard failure.
</Note>

### Retry policy

```text
max_attempts   = 5
base_delay     = 1000 ms                       # ‚â• the 1 req/s floor
backoff        = base_delay * 2^(attempt-1)    # 1s ‚Üí 2s ‚Üí 4s ‚Üí 8s ‚Üí 16s
jitter         = ¬± 250 ms                       # avoid herd on the shared cron
respect        = Retry-After header when present (overrides computed delay)
terminal       = {400, 404}                     # never retried
on_exhaustion  = release lock ¬∑ log_crash phase="api_exhausted" ¬∑ re-queue one tier lower
```

- **Live implementation uses bounded sleeps inside the API helper.** `mb/rate-limit/acquire` sleeps when the global 1 req/s floor has not elapsed, and `mb/api/request` sleeps between its internal retry attempts. Queue-level failures use `next_attempt_at`, so the unified cron skips rows whose enrich retry window is in the future (5 min √ó retries; dropped after 5 failures).
- **Backoff is per-entity; pacing is global.** The 1 req/s token bucket governs *spacing* between any two calls; the API retry backoff governs *re-attempts* of one failed HTTP call; the queue backoff governs failed enrich rows. All three invariants hold simultaneously.
- **Retries are idempotent.** Every write is `MERGE`-based with `coalesce`d `created_at`, so re-running a partially-applied enrich never duplicates nodes or edges.

## Core entities ‚Üí graph nodes

MusicBrainz's 13 core entities map to Orbiter graph nodes as follows. **Seed entities** (minimum bootstrap set) are **artist, recording, label, event**; "cascade" entities get their own queued lookup; inline entities/values are handled from data embedded in other lookups ‚Äî no queue, no extra API call.

| MusicBrainz entity | Graph node label | Seed? | Bridges to existing Orbiter node |
|---|---|:--:|---|
| `artist` (type = Person) | `Entity:Person:Music_Artist` | ‚úÖ | Person-shaped music node; depth-0 person-resolution bridges to canonical `master_person` via `SAME_AS` |
| `artist` (Group / Orchestra / Choir) | `Entity:Music_Group` | ‚úÖ | ‚Äî (new node type ‚Äî a band is never a `Person`) |
| `recording` | `Entity:Recording` | ‚úÖ | ‚Äî |
| `label` | `Entity:Company:Music_Label` | ‚úÖ | Company-shaped; depth-0 `SAME_AS` to canonical `Company` + `OWNED_BY` owner resolution |
| `event` | `Entity:Event` (+ `Music_*` type sub-label) | ‚úÖ | Shared Event model |
| `work` | `Entity:Work` (+`:Song` etc.) | cascade | ‚Äî |
| `release-group` | `Entity:Release_Group` (+`:Album`/`:Single`/`:EP`) | cascade | ‚Äî |
| `release` | `Entity:Release` | cascade | ‚Äî |
| `place` | `Entity:Place` (+`:Studio`/`:Venue`) | inline | ‚Äî |
| `area` | reuse `Country` / `Region` / `City` (MERGE by name; stamp `mbid`) | inline | Orbiter geography |
| `genre` | property on source music node | inline | ‚Äî |
| `series` / `instrument` / `url` | properties / deferred | ‚Äî | ‚Äî |

Every node follows Orbiter conventions: base `Entity` label, `uuid` (random) **plus `mbid`** as the natural MERGE key, `display_label`, `visibility`, dual vectors, and `created_at` / `updated_at` stamped via Cypher `timestamp()`. The pipeline-scoped catalog (keys, writers, vector/description coverage) is on [Music Nodes](/guides/enrichment/waterfall/music-nodes); canonical schemas are in [Ontology ‚Üí MusicBrainz Nodes](/guides/ontology/nodes#musicbrainz-nodes).

## Depth model

Identical to IMDB ‚Äî hard cap at depth 2.

| Depth | Meaning | Behavior |
|:--:|---|---|
| **0** | Seed (user / bootstrap) | Full lookup + all edges + cascade fanout |
| **1** | Direct discovery from a seed | Stub via `get-add`. Queued. Cron enriches + expands, but **only T1‚ÄìT2 children queue at depth 2** |
| **2** | Transitive | Stub queued. Cron enriches + writes edges, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
  **Why cap at 2?** MusicBrainz is ~2.5 M artists / ~30 M recordings. One prolific artist ‚Üí ~20 albums ‚Üí ~250 recordings ‚Üí each recording ‚Üí 1 work + ~5 credited people ‚Üí thousands of entities at depth 2. Combined with the 1 req/s limit, an uncapped cascade would never drain. Depth 2 + tier caps + per-entity caps (20 albums, 50 recordings) keep a seed's full expansion in the low thousands of API calls.
</Check>

## Priority tiers

<Note>
  **Tier 0 is the reserved bootstrap lane** ([decision 5](#resolved-design-decisions-2026-06-11)) ‚Äî manual seed batches queue at `priority_tier = 0` and drain ahead of all cascade traffic. The tiers below (1‚Äì4) apply to cascade-discovered entities.

  **Drain eligibility is tier-0-only (v1.2, 2026-07-01).** The unified drain (13102) picks only `priority_tier = 0` rows. Cascade rows enter at tiers 1‚Äì4 and are **promoted one step toward 0 on each duplicate discovery** (`priority_tier = max(0, min(existing_tier, incoming_tier) - 1)`), so repeatedly-referenced entities earn their drain slot. Tier assignment below still determines how fast that happens.
</Note>

### `queue_mb_artist` (1 = highest)

| Tier | Rule |
|:--:|---|
| **1** | Band member of a seed group; main performer on a seed recording/event; composer of a seed work |
| **2** | Collaborator; producer/engineer on a seed recording; supporting musician |
| **3** | Co-performer at a seed event; artist credited on a label-discovered release _(built: #13091 queues label-credited artists at **T2** ‚Äî see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals))_ |
| **4** | Everyone else mentioned |

### `queue_mb_recording` (1 = highest)

| Tier | Rule |
|:--:|---|
| **1** | Recording where the seed artist is the **main** performer; title track of a seed release-group _(built: #13099 queues **all** tracklist recordings at **T3** ‚Äî see [deviations](/guides/enrichment/waterfall/music-functions#deviations--deferrals))_ |
| **2** | Other recordings credited to the seed artist |
| **3** | Recording on a release discovered via a seed label |
| **4** | Low-signal (alternate takes, untitled, video) |

### `queue_mb_release_group` / `queue_mb_event`

| Tier | Release-group rule | Event rule |
|:--:|---|---|
| **1** | Album by seed artist; award-winning | Event the seed artist **headlined** |
| **2** | EP / single by seed artist | Event the seed artist performed at (support) |
| **3** | Compilation / live / soundtrack | ‚Äî |
| **4** | Other appearance | ‚Äî |



## Queue & upsert semantics

- **Natural key is the `mbid`** ‚Äî a stable UUID straight from the API. No URL normalization needed (unlike IMDB's `imdb_url` keying).
- **Queue upsert is the single source of truth** (`mb/queue/upsert-*`, 13081‚Äì13086): duplicate discovery ‚Üí `count++` + tier promotion + FK fill once the stub exists. Source metadata is set on INSERT only. One row per MBID, enforced by a unique index.
- **`first_notnull` defaults are load-bearing.** Upsert inputs default via `first_notnull` (tier 4 / depth 1), so an explicitly passed `0` survives ‚Äî this is what lets bootstrap-lane rows (`priority_tier = 0`, `cascade_depth = 0`) queue without the defaults coercing them.
- **30-day dedup gate:** `mbid` already enriched within 30 days ‚Üí the gate returns the existing row, no lookup. `gone`-tombstoned rows (404) are never re-queued.
- **Sync-enrich branch:** `cascade_depth == 0 && queue == false` invokes the orchestrator directly; otherwise the row drains through the cron.
- **`entity_rank`** biases identity-rich entities ahead of bulk catalog in the unified drain (artist 1 ¬∑ label 2 ¬∑ event 3 ¬∑ work 4 ¬∑ release\_group 5 ¬∑ recording 6) ‚Äî see [Music Waterfall Tables](/guides/enrichment/waterfall/music-tables#unified-cron-ordering--entity_rank).

Full schemas, indexes, and conventions: [Music Waterfall Tables](/guides/enrichment/waterfall/music-tables).

## Seed flows

### Seed artist (depth 0)

<Steps>
  <Step title="Dedup gate">
    Lookup `mb_artist` by `mbid`, return if enriched &lt; 30 days. (No kill-switch check ‚Äî skipped by design; pause the unified cron to stop the pipeline.)
  </Step>
  <Step title="Throttled lookup">
    `acquire` token ‚Üí `GET artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels+event-rels`. Save raw JSON.
  </Step>
  <Step title="Node + base edges">
    `MERGE` the artist node on `mbid` ‚Äî `Entity:Person:Music_Artist` (person) or `Entity:Music_Group` (group) by `type` ‚Äî stamp `created_at`/`updated_at`, dual vectors, and media/description where the finalizer applies. No inline reconciliation here ‚Äî depth-0 person artists enter the person-resolution bridge.
  </Step>
  <Step title="Membership edges">
    For each `member of band` rel ‚Üí `MEMBER_OF` edge (instruments/dates/original) + queue the member artist at **T1**.
  </Step>
  <Step title="Cascade (browse, throttled, paginated)">
    Queue children per the [artist cascade filter](#cascade-filter--artist-seed) above, each via `get-add` at `depth+1`. Depth-0 person artists also enqueue [person-resolution](/guides/enrichment/waterfall/music-functions#person-resolution) and fetch a serper avatar inline.
  </Step>
  <Step title="Mark success">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

### Seed recording / label / event

Same shape: 1 throttled lookup with the entity's `inc=` set ‚Üí `MERGE` node ‚Üí write its edges ‚Üí queue discovered neighbors at the [tiers above](#priority-tiers) ‚Üí stamp success. A label additionally browses `release?label=` (capped 3 pages / 300 releases) and, at depth 0, runs the **Company bridge** (label‚Üí`Company` `SAME_AS` + [owner resolution](/guides/enrichment/waterfall/music-functions#label--company-bridge--owner-resolution) ‚Üí `OWNED_BY`); an event reads `artist-rels` + `place-rels`. Per-orchestrator phase detail: [Music Waterfall Functions ‚Üí Orchestrators](/guides/enrichment/waterfall/music-functions#orchestrators).

### Depth-1 / depth-2 queued processing

When the cron picks up a queue row it locks it (`processing: true`) and dispatches the matching orchestrator at the row's `cascade_depth`. Depth 1 runs all phases, but **only T1‚ÄìT2 discoveries queue at depth 2**. Depth 2 enriches and writes edges but **cascades nothing**. Depth guarding is inlined in each orchestrator's `mb_cascade` phase: `cascade_depth + 1 >= 3` ‚Üí no queue.

## Cascade example ‚Äî seed artist end-to-end

Here's what happens end-to-end when an artist enters as a depth-0 seed ‚Äî the seed flow, the [artist cascade filter](#cascade-filter--artist-seed), the unified drain, and the depth model above, restated as one tree:


**API-call pacing:**

- Every MusicBrainz call in this tree passes through the global 1 req/s token bucket (`mb/rate-limit/acquire` \+ the single unified cron), so total throughput never exceeds 1 req/s no matter how many queue rows are pending.
- One artist \+ two browse pages ‚âà 3 requests ‚âà 3 seconds ‚Äî effective throughput is ~1 entity / 3‚Äì6 s.
- That is why a depth-0 artist with its top recordings \+ albums \+ members takes **minutes**, and a deep cascade takes **hours** ‚Äî by design.
- The depth cap (2), tier caps, and per-entity caps (20 albums, 50 recordings, 3 browse pages) keep a seed's total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.

## Loop prevention (six layers)

1. **Dedup gate in `get-add`:** `mbid` already enriched within 30 days ‚Üí return existing, no lookup.
2. **Queue upsert:** duplicate `mbid` ‚Üí `count++`, tier promotion. One row per MBID.
3. **Depth cap:** depth 2 enriches but does not cascade; depth 3 never queued.
4. **Tier cap at depth ‚â• 1:** only T1‚ÄìT2 discoveries queue at depth 2.
5. **Manual kill (no dedicated switch ‚Äî decided 2026-06-11):** layers 1‚Äì4 make the cascade structurally non-recursive (depth is monotonic with a hard cap; cycles collapse into `count++` on existing rows), so the planned `check-kill-switch-mb` env-var gate was skipped. Pausing the single unified cron task in Xano stops the entire pipeline.
6. **Global rate limiter:** `mb/rate-limit/acquire` + the unified cron enforce 1 req/s across the whole pipeline (the layer IMDB doesn't need).

The built person-resolution path (#13118‚Äì#13123) bridges depth-0 person artists to a canonical `master_person` with `SAME_AS` and verified social links; label-owner resolution (#13132‚Äì#13134) writes `OWNED_BY`. Label‚ÜíCompany `SAME_AS` runs in the depth-0 label orchestrator. Fixed-forward 2026-07-01: both live `SAME_AS` writers use identity weight `1`.
  </Card>
  <Card title="Area mapping ‚Äî reuse Orbiter geography" icon="earth-americas">
    **Decided.** MusicBrainz areas MERGE into the existing `Country` / `Region` / `City` nodes by name, stamping `mbid` on match. No dedicated `Music_Area` node type and no mapping table. Unmappable or historical areas fall back to the nearest modern parent (e.g. a historical region maps to its present-day country). _(Forward-path gap: `FROM_AREA` is not yet wired into the audited ingest path.)_
  </Card>
  <Card title="Genre vs tag ‚Äî genres as properties" icon="tags">
    **Decided.** Curated `genres` and free-form `tags` stay as node properties for search, filtering, facets, and badges. Do not create `Genre` hub nodes or `HAS_GENRE` edges.
  </Card>
  <Card title="Canonical release ‚Äî one Official release per release-group" icon="compact-disc">
    **Decided.** Each `Release_Group` materializes exactly **one** `Release` node: normally `status = Official`, earliest release date (any country). If the release-group came from a seed label, an Official release on that seed label wins before the generic earliest-Official fallback. Tracklists and `APPEARS_ON` edges hang off that canonical edition. All other pressings remain in the raw JSON on the `mb_*` source tables, so the decision is reversible without re-fetching.
  </Card>
  <Card title="Bootstrap ‚Äî live API only, priority lane" icon="gauge-high">
    **Decided.** No offline import from the [MusicBrainz database dumps](https://musicbrainz.org/doc/MusicBrainz_Database/Download). Large bootstraps (e.g. a full label roster) ride the same 1 req/s live API, queued at **`priority_tier = 0`** ‚Äî a reserved bootstrap lane that drains ahead of all cascade traffic. One code path, no dump tooling; multi-day seed loads are accepted.
  </Card>
  <Card title="Places ‚Äî inline materialization, no queue" icon="location-dot">
    **Decided.** `Place` nodes are written directly during recording/event enrich from the place object embedded in `place-rels` (name, type, address, coordinates, area ‚Äî everything the node spec needs). No `queue_mb_place`, no place orchestrator, zero extra API calls at the 1 req/s ceiling. The `mb_place` source table stores the embedded JSON and `mbid`, so a future scoped lookup can add URL rels without re-discovery.
  </Card>
