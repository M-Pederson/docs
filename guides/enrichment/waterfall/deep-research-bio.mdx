Deep Research Bio is the optional, most expensive tier of person enrichment: an agentic LLM run with live web-search tools that expands a person's short `bio_500` into a comprehensive, citation-backed profile — biography paragraphs, a mapped relationship network, credentials, and unverified-claim flags. The result lands on `master_person.deep_bio` and feeds the derived-bio, social-insights, and graph-node sync steps.

<Note>
  **Last verified: July 17, 2026.** Deep Bio is an optional branch, not part of
  the core person-enrichment completion contract. With `deep_research=false`,
  #13040 creates no Deep Bio dispatch, job, callback, receipt, retry, or barrier
  wait; the independently complete non-bio path closes through #13194.
</Note>

It is **asynchronous end-to-end**: Xano launches a job on an external runner and returns immediately; the runner executes the long tool-use loop and posts the result back to a webhook, which does all the write-back and fan-out.

## Flow at a glance

```text
orchestrator (deep_research=true)          #13040 / #12880 phase 4b / batch tools
  └─> mvp/deep-research/deep-person-basic  #12833  — context + researcher prompt
        └─> mvp/enrich/deep-research-person-or-company  #12747 — payload + launch
              └─> POST Cloud Run /enrich  (external runner, X-API-Key)
                    ├─ inserts profile_enrichment_job (processing=true)
                    └─ … runner executes Kimi K2.6 + web_search/web_fetch …
                          └─> POST webhooks/enrich-profile  #8303  (callback)
                                ├─ preserve callback + validate JSON/identity/biography
                                ├─ invalid → same-job retry or terminal provider error
                                └─ valid → provider_payload_committed receipt
                                      └─ #13208 callback terminal runner
                                            ├─ master_person.deep_bio ← biography[]
                                            ├─ derived bio/social/expertise/node writers
                                            └─ #13196 terminal barrier → #13194 final readback
```

## Launch points

The `deep_research` boolean threads through the person pipeline: `mvp/queue/upsert-enrich-person` (12854) OR-promotes it on queue upgrades, while `mvp/enrich/enrich-master-person` (13040) normalizes it once and gates every Deep Bio-owned surface. Only true may launch the provider, create a `profile_enrichment_job`, add profile-job terminal dependencies, or enter the durable Deep Bio barrier. False routes directly to the non-bio finalizer with `require_deep_bio=false`; stale profile-job rows cannot reactivate the optional branch. Batch tools that call a Deep Bio function directly are explicit opt-in launch points.

## The research function

<Card icon="play">

```
mvp/deep-research/deep-person-basic — 12833
```

The prompt/agent definition (v4.1). Assembles the person's verified context — name, current title, company, `bio_500`, certifications, and work history via addons — into YAML, and pairs it with a senior-researcher system prompt: a **hard 12-search budget** (3 per thread), **footprint triage** (RICH / MODERATE / THIN sizes the effort and word count), an **identity-collision protocol** (2+ anchor matches required before trusting any source; same-name individuals tracked and surfaced in `identity_conflicts`), six parallel research threads (identity, career, credentials, relationship mapping, thought leadership, recognition), relationship-strength tiers (CONFIRMED / LIKELY / INFERRED), and a strict-JSON output contract (biography paragraph array, `biography_sections`, `relationships`, `citations`, `unverified_claims`, `extracted_entities`, `metadata`).

Model: `moonshotai/kimi-k2.6`, provider-pinned to `["moonshotai", "fireworks"]` with `allow_fallbacks: false` (v4 fix — unpinned routing leaked raw tool-call tokens), temperature 0.6, top_p 0.9, max_tokens 16,000, reasoning capped at 8,000, tools `openrouter:web_search` (native engine, 6 per call / 24 total) + `openrouter:web_fetch` (6,000 content tokens).

## The transport / launcher

<Card icon="cloud-arrow-up">

```
mvp/enrich/deep-research-person-or-company — 12747
```

Shared person/company transport. Builds the OpenRouter chat payload from the caller's prompts + params (Perplexity models get collapsed to a single user message; `provider_order` and `allow_fallbacks` pass through), then POSTs it to the **external runner** — Cloud Run `https://orbiter-enrich-20506320032.us-east4.run.app/enrich`, authenticated with `X-API-Key: $env.deep_research_api_key` — along with the callback: `webhook_url = https://xh2o-yths-38lt.n7c.xano.io/api:nFBFWRKy/enrich-profile?x-data-source=<current datasource>` (the datasource is appended so the callback writes to the same data source the launch ran on).

On a `job_id` response it inserts a **`profile_enrichment_job`** row and returns. The row keeps the exact Cloud Run request in `data._cloud_run_request`, the launch response in `data._cloud_run_launch_response`, and request SHA-256, character count, model, callback datasource, and send time in `data._payload_audit`. These fields are the authoritative launch-side forensic record. A missing `job_id` writes a `log_crash` row and returns `"crushed"`.

The runner itself is the long-running piece: it executes the multi-turn Kimi + web-search loop (minutes, beyond Xano's request budget) and posts results back under the original `job_id`. Bounded retries remain on that same logical job. Xano also treats a malformed person success payload as retryable: it records the complete invalid attempt and returns a retry disposition to Cloud Run until the attempt budget is exhausted. Historical API docs: `bio-enrich.fly.dev/docs` (the runner's earlier Fly.io home).

</Card>

<Card icon="database">

```
profile_enrichment_job — 686
```

The job ledger. One row per launch: `job_id`, `request_type` (`person` | company), `master_person_id` / `master_company_id`, the outbound `data` payload, `processing` flag, and — after callbacks — `llm_response`, full `response`, or `error`. The response preserves `provider_callback_payload`, `provider_attempt_payloads`, `provider_payload_validation`, `payload_audit`, and `retry_history`. `processing = true` rows older than the runner's normal turnaround are the stuck-job queue.

</Card>

## The webhook

<Card icon="globe">

```
webhooks/enrich-profile — 8303   (POST, api group 346, canonical nFBFWRKy)
```

**The active callback.** Receives the runner's result keyed on `job_id` (raw-input JSON, no auth — the obscure job ID is the correlation key). The full callback body is preserved with its digest and character count before any disposition is chosen.

- **Provider error path:** preserves the callback and same-row attempt history. A retryable outcome stays on the same job; terminal exhaustion closes `processing`, stores the error, and performs no downstream writes.
- **Person success validation:** accepts a JSON object, complete JSON string, or harmless preamble/code fence around one complete JSON object. It requires the expected subject (or a verified alias), a non-empty string-only `biography[]`, at least 120 biography characters, and no provider meta-commentary or JSON/code fragments. Truncated JSON, identity mismatch, short/empty output, and raw prose are rejected before provider success is committed.
- **Invalid payload path:** preserves the complete callback and validation audit in `provider_attempt_payloads`, increments the same-job retry history, and returns `retrying` while attempts remain. Exhaustion records an error `provider_payload_committed` receipt and closes the job without materialization.
- **Validated success path:** stores `llm_response`, validation/audit data, and a successful `provider_payload_committed` receipt containing the payload SHA-256. It then leases and dispatches `mvp/enrich/run-person-callback-terminal` (13208). That runner independently revalidates `llm_response`, requires the webhook and receipt digests to agree, and only then writes `master_person.deep_bio` and downstream surfaces.
</Card>

<Card icon="filter">

```
mvp/expertise/llm-identify-person-expertise — 12666
```

**Expertise re-run (added v1.1, 2026-07-02).** Re-extracts the person's domains of expertise now that the deep-research material exists, feeding the resolver `resolve-person-expertise-v2` (12926). Because the resolver chain is fully idempotent (exact-name guard, `HAS_EXPERTISE` MERGE aggregation, `person_has_expertise` dedupe), a person whose expertise was already resolved gets their edges **updated** — max confidence, strongest depth, distinct new evidence appended — and newly surfaced areas added, never duplicated. Runs in its own `try_catch` (logged to `log_crash` phase `webhook_post_process`, success and failure), so an expertise hiccup never blocks the rest of the fan-out.

</Card>

## Operational notes

- **Non-blocking by design:** the orchestrator treats a deep-research launch failure as a warning, not a pipeline failure — `full_enrich` reflects required-phase health only.
- **Datasource propagation:** the callback URL carries `?x-data-source=<datasource>` from launch time, so sandbox launches write back to sandbox.
- **Provider retry boundary:** Cloud Run owns external provider execution, while Xano owns acceptance at the callback boundary. Every callback attempt remains attached to the original job row and is auditable; a separate job row is not a clean retry.
- **Stuck jobs:** a job that never calls back leaves `profile_enrichment_job.processing = true` with no `response` — there is no timeout sweeper; check the ledger when `deep_bio` seems missing.
- **Output contract:** raw prose and malformed JSON never land. Terminal success additionally requires `deep_bio` to remain a non-empty string array with at least 120 joined characters, a validated provider receipt, and the final semantic table/graph readback.
- **QA evidence:** compare `data._payload_audit.request_sha256`, `response.payload_audit.callback_sha256`, `response.provider_payload_validation.input_sha256`, the callback receipt digest, and the callback runner's independent validation. Use counts/digests and a bounded tail preview in reports; retain full payloads only in a protected temporary artifact.
