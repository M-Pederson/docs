Core graph node types plus enrichment-specific graph nodes, organized by logical group.

Most product graph nodes carry the base label **`Entity`** plus one or more type-specific labels. Legacy non-`Entity` labels are called out in the [Master Label List](#master-label-list). Nodes are created during the person and company get-add pipelines, funding ingestion, IMDB credit processing, expertise identification, and MusicBrainz waterfall enrichment.

<Note>
  **Last reviewed & updated: 2026-07-09** — labels, properties, and writers audited against the live XanoScript graph writers (workspace 3, branch `v1`).
</Note>

<Info>
  Each node type below lists the Xano functions that **create** (<Icon icon="circle-plus" />) and **update** (<Icon icon="pen-to-square" />) it, in copy-paste call-out blocks (`function path — ID`). Every documented node type stamps `created_at` on create and `updated_at` on update as Cypher `timestamp()` (epoch-millis) — see [Timestamp coverage](#timestamp-coverage-audit).
</Info>

It includes both full create-label shapes and subset / match-only label combinations such as `Person`, `Company`, `Film_TV`, or `Work`. Dynamic families are shown with the controlling source field.

<Note>

## People

Nodes representing individual humans. A Person node may carry additional labels based on their role.

### Person

The core node for any individual in the graph.

<Card icon="circle-plus">
  ```text
  mvp/node/add-person-node — 2601
  ```

  **Creates** the Person node (`MERGE` on `master_person_id`, adds the `:Entity` label). Stamps `created_at` and `updated_at`. ✅
</Card>

<Card icon="pen-to-square">
  ```text
  mvp/node/update-person-node — 2758
  ```

  **Updates** the Person node (v1.11, 2026-07-09) — refreshes properties including bios, investor fields, and a scalar `linkedin_url` (v1.10); dual-writes `name_embedding` \+ Gemini `embeddings`. Stamps `updated_at`. ✅ _Note: the dynamic `:Angel` / `:C_Suite` label writing was lost in the v1.8 rebuild and is not restored — `:Angel` is stamped by #12702 only._
</Card>

<Card icon="circle-check">
  ```text
  mvp/imdb/run-base-imdb-person-enrich — 12880
  ```

  Flips `visibility` to `true` during IMDB enrichment — `SET p.visibility = true, p.updated_at = timestamp()`. ✅ Bumps `updated_at` (fixed 2026-06-01).
</Card>

| Property | Type | Description |
| --- | --- | --- |
| `uuid` | string | Primary identifier (UUID v4) |
| `master_person_id` | integer | The MERGE key — FK to Xano `master_person.id` |
| `name` | string | Full display name |
| `display_label` | string | Always `"Person"` |
| `visibility` | boolean | Whether the node is visible in the UI (starts `false`; flipped by the phase-11 finalizer) |
| `linkedin_url` | string | Canonical LinkedIn URL (stamped by #2758 v1.10) |
| `short_biography` / `long_biography` | string | One-line Cypher-safe projections of the Xano bios (control characters stripped) |
| `image_url` / `avatar` | string | Avatar URL where synced |
| `name_embedding` | vecf32 | OpenAI 1536 semantic embedding |
| `embeddings` | vecf32 | Gemini 1536 embedding (dual-write since 2026-06-15) |
| `created_at` / `updated_at` | timestamp | Epoch-millis `timestamp()` (nodes created before mid-2026 may carry ISO strings) |

<Accordion title="Real example">
  ```json
  {
    "uuid": "cf64fc2a-4b3e-4ff9-99dc-21d7cffebf01",
    "name": "Josh Diamond",
    "display_label": "Person",
    "visibility": true,
    "created_at": "2025-01-15T10:32:45.000Z"
  }
  ```
</Accordion>

### Angel

An additional label applied to Person nodes identified as angel investors.

<Card icon="tag">
  ```text
  mvp/fundable/resolve-investors-edges — 12702
  ```

  **Adds** the `Angel` label — `SET p:Angel` on the angel lead/non-lead investor branches during the company-side deal cascade. Stamps `updated_at`. ✅ _(The former #2758 `is_angel`-driven label write was lost in its 2026-07-05 rebuild; #12702 is the only live `:Angel` writer today.)_
</Card>

**Labels:** `Entity:Person:Angel`

No additional properties beyond Person — `Angel` is a label-only modifier used for filtering and traversal.

  ```json
  }
  ```

  _Node labels: `Entity`, `Person`, `Angel`_
</Accordion>

## Organizations

Nodes representing companies, schools, VC firms, and other organizations. All share the same creation function (`mvp/node/add-company-node` #2602) with a dynamic label based on type.

### Company

The default organization node type. Represents businesses, startups, and corporations.

<Card icon="circle-plus">
  ```text
  mvp/node/add-company-node — 2602
  ```

  **Creates** the Company node — `CREATE (p:{nodeLabel} {...})` with a dynamic label (`Company`, `School`, etc.). Stamps `created_at` (active v4 template). ✅
</Card>

<Card icon="pen-to-square">
  ```text
  mvp/node/update-company-node — 4659
  ```

  **Updates** the Company node — refreshes enrich properties. Stamps `updated_at`. ✅
</Card>

<Card icon="pen-to-square">
  ```text
  tool/resync-company-node-domains — 13062   ·   tool/backfill-strip-domain-errors-nodes — 13060
  ```

  Maintenance tools that rewrite `company_domain` / `logo` on the Company node. Both stamp `updated_at`. ✅
</Card>

<Card icon="circle-check">
  ```text
  mvp/dupe/merge-company-nodes — 10505
  ```

  Folds duplicate Company nodes during dedup (`SET` survivor properties). ✅ Bumps `updated_at` on the survivor (fixed 2026-06-01).
</Card>

| Property | Type | Description |
| --- | --- | --- |
| `uuid` | string | Primary identifier |
| `name` | string | Company display name |

| `visibility` | boolean | UI visibility flag |
| `logo` | string | URL to company logo |
| `name_embedding` | vecf32 | Semantic embedding |
| `created_at` | string | ISO 8601 timestamp |

### Organization

A generic organization node. Used for non-company entities like certification bodies, volunteer organizations, and honor issuers. Can be created standalone or applied as an additional label to an existing Company node.

<Card icon="circle-plus">
  ```text
  mvp/resolve/resolve-edges-volunteering — 4575
  ```

  **Creates** a standalone Organization node — `MERGE (org:Organization:Company {name})`. ✅ Stamps `created_at` \+ `updated_at` (renamed from camelCase `createdAt`, `ON MATCH SET` added; fixed 2026-06-01).
</Card>

<Card icon="tag">
  ```text
  mvp/resolve/resolve-edges-certifications — 12578
  ```

  **Updates** an existing Company node — `MATCH (organization:Company) SET organization:Organization, organization.updated_at = timestamp()`. ✅ Bumps `updated_at` (fixed 2026-06-01).
</Card>

**Labels:** `Entity:Organization:Company` (when added to existing Company) or `Entity:Organization:Company` (when created new by volunteering)

| Property | Type | Description |
| --- | --- | --- |
| `uuid` | string | Primary identifier |
| `name` | string | Organization name (the MERGE key on the volunteering path) |
| `created_at` / `updated_at` | integer | Epoch-millis `timestamp()` (renamed from camelCase `createdAt` on 2026-06-01; a few legacy nodes still carry the old key) |

## MusicBrainz Nodes

MusicBrainz nodes are produced by the [Music Industry Waterfall](/guides/enrichment/waterfall/music-waterfall). This section is the canonical graph contract for music node labels, properties, media fields, vector fields, and reconciliation behavior. The waterfall pages should link here instead of redefining node schemas.

<Note>
  Every enriched music node converges on the standard envelope: `created_at`, `updated_at`, `uuid`, `mbid`, `display_label`, `visibility`, semantic-null or GOOD `description`, and both vector fields. `Recording`, `Music_Group`, and `Person:Music_Artist` can additionally carry `image_url`.
</Note>

<Warning>
  MusicBrainz `artist` records split into two graph shapes:

  - `type == "Person"` -\> `Entity:Person:Music_Artist`
  - `type in {Group, Orchestra, Choir, Other}` -\> `Entity:Music_Group`

  A band is never a `Person`. `MEMBER_OF` is therefore always `(Person:Music_Artist)->(Music_Group)`.
</Warning>

<a id="music-standard-node-envelope" />

### Standard Music Node Envelope

```cypher
MERGE (n:<labels> {mbid: $mbid})
SET n.uuid           = coalesce(n.uuid, randomUUID()),
    n.display_label  = "<Label>",
    n.visibility     = true,
    n.description    = <good <=127 summary | null>,
    n.name_embedding = vecf32($embedding),
    n.embeddings     = vecf32($geminiEmbedding),
    n.created_at     = coalesce(n.created_at, timestamp()),
    n.updated_at     = timestamp()
    // node-specific properties, plus image_url where supported
```

| Field | Type | Required | Notes |
| --- | --- | :-: | --- |
| `uuid` | string | Yes | `randomUUID()`, stable via `coalesce` |
| `mbid` | string | Yes | MusicBrainz ID and MERGE key |
| `display_label` | string | Yes | Product-facing label |
| `visibility` | boolean | Yes | UI flag |
| `description` | string/null | Yes | GOOD-or-null invariant: one clean factual sentence \<=127 characters beginning with the entity name, or exactly `null` |
| `name_embedding` | vecf32 | Yes | OpenAI `text-embedding-3-small`, 1536 dimensions; indexed on `:Entity` today |
| `embeddings` | vecf32 | Yes | Google `gemini-embedding-001`, 1536 dimensions, L2-normalized (3072→1536 migrated 2026-07-06); indexed @1536 on `Entity`; read cutover from `name_embedding` pending |
| `image_url` | string/null | Selective | Only `Recording`, `Music_Group`, and `Person:Music_Artist` |
| `created_at` | timestamp | Yes | `coalesce(n.created_at, timestamp())` |
| `updated_at` | timestamp | Yes | `timestamp()` on every write |

`Recording`, `Music_Group`, and `Person:Music_Artist` receive generated descriptions through `mb/node/finalize-music-node` (#13117). Other music node types intentionally carry `description = null` unless a future description pass supports them. Null is preferred over low-quality generated text because descriptions are used as embedding source text.

<a id="music-genre-policy" />

### Music Genre Policy

Genres are metadata properties on music nodes, not graph nodes.

Do not create `Entity:Genre` nodes or `HAS_GENRE` edges. Keep genre values on the source music node as normalized `genres` metadata so the product can use them as facets, filters, badges, search text, or side-panel context without turning every genre into a first-class entity.

<a id="music-media-image-url" />

### Music Media (`image_url`)

Three node types can carry a re-hosted image on `image_url`, matching the `film_tv.image_url` convention. The value is the GCS-hosted WebP URL, not the raw external URL. When no image is found, the property is explicitly set to `null`.

| Node | Source field | GCS path | Raw source |
| --- | --- | --- | --- |
| `Recording` | `mb_recording.gcp_cover_art_url` (#738) | `music-cover-art/<mbid>.webp` | Cover Art Archive |
| `Music_Group` | `mb_artist.gcp_music_group_image_url` (#737) | `music-group-image/<mbid>.webp` | serper.dev image search |
| `Person:Music_Artist` | `mb_artist.gcp_person_avatar_url` (#737) | `music-person-avatar/<mbid>.webp` | serper.dev image search |

Image writers are `mb/media/fetch-recording-cover-art` (#13108), `mb/media/fetch-group-image` (#13109), and `mb/media/fetch-person-avatar` (#13120). After the GCS URL is stamped on the source row, Cypher updates the node's `image_url`. The already-done path self-heals existing nodes.

## Timestamp coverage audit

The convention is `created_at` set in the create/`ON CREATE SET` clause and `updated_at` set in the update/`ON MATCH SET` clause — both as Cypher `timestamp()` (epoch-millis). **Every documented node type on this page stamps both** — the gaps found by the 2026-06-01 sweep were all patched (creators that omitted `created_at`: Country, SubDomainExpertise, Event:Live\_\*, Certification, Honor, Project, Publication, Organization; updaters that omitted `updated_at`: the IMDB visibility flip #12880, the company dedup fold #10505, the VC\_Firm / Organization label-adds, and the certification `SET`).

<Note>
  **Known remaining gaps** (outside the enrichment pipelines' documented types): `contextFounded` (`mvp/yc/process-yc-people` #12700) sets no `created_at` on the hub node, and the `Document`/`DocumentChunk` (`#12850`, `#4596`) and user/organization-graph types `MasterPerson` / `MasterOrganization` (`#12869`, `#12884`) remain unstamped. The `qa/resolve-edges-*` functions (#12715/#12718/#12719) are QA mirrors of the production resolvers — patch them alongside if QA parity matters.
</Note>
