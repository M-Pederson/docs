This page captures the **two-tier memory model** Mark sketched on the June 18, 2026 sync and the **Timeline / Memories tab** that surfaces it on the person sheet. It extends [Migration Plan](/guides/open-work/mem0/migration-plan) — that page documents how each chat surface reads and writes Mem0; this page answers a different question: **whose memory is it, and how do we show all of it in one place.**

Sections 1–2 are **verified live against Xano on 2026-06-22** (endpoints `#8581` / `#8583` read directly). Sections 3–6 are the **proposed design** — align with Mark before building, per his "make the blueprint first."

<Info>
  **One-line summary.** Today only the **contact tier** exists (per-user, subjective, keyed `contacts_id`), and only How I Know writes to it. The **shared master-person tier** (objective facts true for every user) does not exist yet. Completing it is additive — one extra search branch and a metadata field, no new chat endpoints. The Timeline tab is the only genuinely new Xano surface, and it is a thin read.
</Info>

## 1. The two-tier model

Mark's framing (June 18, 12:08–16:20 and 32:21–33:45): Mem0 holds **facts about people**, in two tiers that differ by *who the fact belongs to*.

| Tier | Belongs to | Mem0 metadata key | Source | Example |
| --- | --- | --- | --- | --- |
| **Contact memory** | *You* — per-user, subjective | `contacts_id` | How I Know interview; your emails about them; outcomes you run | "I met Ray at State Change; his superpower is distilling chaos" |
| **Master-person facts** | *Everyone* — shared, objective | `master_person_id` *(proposed)* | Enrichment pipeline; objective email extraction | "Scott Belsky invested in Stripe; born Minneapolis 1987" |

The same physical person is **one** `master_person` row (`#139`) shared by every user, but **each user has their own `contacts` row** for him (`#128`, formerly `my_person`). Mark's words: *"Ray Deck is master person number one, but Mark's contact could be 43 and yours could be 22."*

**Why split them.** Objective facts ("invested in Stripe") are true for everyone and should be written once at the master-person level. Subjective facts ("how *I* know him") are mine alone and must never leak across users. Keying the two tiers on different metadata fields is what enforces that boundary.

**The unification goal (33:45).** Every write path — email→Mem0, outcome→Mem0, How I Know→Mem0 — tags a consistent metadata schema so a single person view can **"call all of the memories"** about them at once, fanned across sources. That person view is the Timeline tab (§4).

## 2. Current state (verified live 2026-06-22)

Only the **contact tier** is built, and only **How I Know** writes to it. Both endpoints derive the contact id **server-side** (Option B — Mark, June 18) so callers never pass an id and a client cannot spoof another user's contact.

### Write — `#8583` `POST :1261/relationships`

On a completed interview (`interview_complete && relationship_summary != ""`), pushes the **prose `relationship_summary`** to `POST api.mem0.ai/v3/memories/add/` with **`infer: true`** — Mem0 splits the prose into discrete facts. Add metadata (built in a lambda):

```
source_id: "how_i_know:{master_person_id}:{hash(summary)}"
memory_hash, schema_version: "2026-06-10"
category: "relationship", confidence: "high"
flow_type: "how_i_know", source_type: "chat"
contacts_id: {backend-derived contact id, falls back to master_person_id}
```

User key = `workos_id` (fallback `user-{auth.id}`). Gated on `$env.MEM0_API_KEY`. The structured `relationship_nuggets` are persisted to `#729` but are **not** sent to Mem0 directly — Mem0 re-derives facts from the prose (see §5).

### Read — `#8581` `POST :1276/openui05/how-i-know-interview`

Two searches, deduped, injected into the interviewer system prompt as `[WHAT YOU ALREADY KNOW — durable facts from past chats; do not re-ask these]`:

1. **User-scope:** `filters:{user_id}` — everything you have ever stored.
2. **Contact-scope** (gated on `contact_id > 0`): `filters:{AND:[{user_id}, {metadata:{contacts_id}}]}`.

### The gap

Neither side tags `master_person_id`. There is **no way to retrieve objective facts shared across users** — the master-person tier does not exist yet. The `AND`-filter fan-out in the read is the exact seam to extend: add a third search on `{metadata:{master_person_id}}` with **no** `user_id` filter (§3).

## 3. Proposed: complete the shared tier

Minimal and additive — designed to port cleanly to GCP (§6). No new chat endpoints.

<Steps>
  <Step title="Master-person write path (not How I Know)">
    Whatever generates *objective* facts — the enrichment pipeline, and the objective-extraction half of the email→Mem0 function ([`fn 13077`](/guides/open-work/mem0/memories-from-email-threads)) — writes with metadata `master_person_id` set and **no per-user key** (or a shared system key), `flow_type` of `enrichment` / `email_fact`, and an appropriate `category`. How I Know writes stay contact-scoped and unchanged.
  </Step>
  <Step title="Three-tier read fan-out">
    Extend the existing search fan-out with a third call: `filters:{metadata:{master_person_id}}` (shared, no user filter). Merge order: contact-scope (most specific, mine) → master-person (shared) → user-scope (broad), then dedupe — the read lambda already dedupes by lowercased text.
  </Step>
  <Step title="Consistent metadata schema across all writers">
    ```
    { user_id, contacts_id?, master_person_id, flow_type, category,
      source_type, confidence, schema_version, source_id, occurred_at? }
    ```
    `occurred_at` is new and load-bearing for the Timeline (§4) — the chronology Mark wants.
  </Step>
</Steps>

## 4. The Timeline / Memories tab

A new side-panel tab on the person sheet (Mark, June 18, 14:42–15:49), **gated on contact existence** exactly like How I Know — render only when the `contacts` (`my_person`) id is present. See the How I Know gate in `profile-tab/how-i-know-section.tsx` and the feature-folder conventions in the frontend repo's `docs/agents/feature-architecture.md`.

<CardGroup cols={2}>
  <Card title="What it shows" icon="timeline">
    A chronological "timeline of facts" for this person, merging both tiers: master-person facts (shared) + my contact memories (mine), sorted by `occurred_at`.
  </Card>
  <Card title="v1 scope" icon="eye">
    Read-only — surface and sort. Manage / edit / delete is deferred to a later pass.
  </Card>
</CardGroup>

- **Icon / label:** vertical timeline glyph; label "Timeline" or "Memories" (TBD with Mark).
- **Backend:** one new read endpoint, e.g. `GET person-memories?master_person_id=…`, that runs the three-tier search (§3 step 2) and returns a merged, time-sorted list. This is the **only** genuinely new Xano surface the tab requires — keep it a thin search-and-merge so it ports to GCP in an afternoon.

## 6. Constraint: build the least possible in Xano

July 6: the GCP team migrates the enrichment engine **and all chat business** off Xano. Mark @ 1:07:43 — *"Every endpoint we make in Xano, we're going to remake in GCP."* So this blueprint is deliberately additive:

- §3 steps 1–2 add **zero endpoints** — they are metadata fields plus one extra search branch on existing requests.
- §4's Timeline read is the **only** new endpoint, kept a thin search-and-merge on purpose.
- Everything reuses the existing Mem0 add/search calls, so it ports to GCP 1:1. The metadata schema and the How I Know endpoints (`#8581` / `#8583`) are the documented template — model, prompt, and I/O are all explicit, per Mark @ 1:08:50: *"where literally is that exact system prompt."*
