## The gap this closes

The [music dual-embedding migration](/guides/enrichment/waterfall/music-nodes#embeddings-per-node-type) (2026-06-14) was scoped to the **10 music node types only**. Business-side nodes carried **`name_embedding`** (OpenAI `text-embedding-3-small`, 1536) but **not** the Gemini **`embeddings`** (`google/gemini-embedding-001`, 3072). This session wires the Gemini vector into the four business node writers, using the **same embed text** each already used for `name_embedding`, so the two vectors stay comparable.

## Design — why one helper, not four inline lambdas

The four writers split across two Cypher-building styles: one builds the query in an `api.lambda` (`add-person-node`), three use `util.template_engine`. Rather than duplicate a 3072-float normalizer four times (and risk the `\x00` / `\n` heredoc-escape traps on each), the work funnels through **one helper whose response is byte-for-byte the same shape as the proven `name_embedding` path**:

- **Template callers** splice `vecf32({{$var.geminiVecString}})` exactly as they already splice `name_embedding` — the helper returns a real bracket-string, which `util.template_engine` renders **with** brackets.
- **The lambda caller** reads the helper's response as `$var.geminiVecString`; Xano auto-parses the bracket-string back into a JS array (the [#8 embedding trap](/guides/enrichment/waterfall/music-functions#node-embeddings)), so it `JSON.stringify`s it just like `name_embedding`.

Every `embeddings` write is **gated** (`{% if … is not null %}` / `if (gemArg)`), so a Gemini miss degrades to `name_embedding`-only and the node still writes — it can never regress a node creation that succeeds today.

## Gotchas resolved this session

- **Does a bracket-string render with brackets through `util.template_engine`?** Yes — proven live: `vecf32({{$var.geminiVecString}})` rendered `vecf32([-0.0007526…])` with the opening `[` and closing `]),` intact, and the `{% if … is not null %}` gate included/excluded the line cleanly. (The existing `name_embedding` path already relied on this; now confirmed directly.)
- **Do JS-escape sequences survive the MCP serializer round-trip?** Yes — a throwaway probe confirmed `\x00` regexes and `\n` joins authored with **single** backslashes store correctly (they appear double-escaped only in the JSON view). This unblocked editing the four functions in place.
- **`vecf32` props don't render in graph browsers** (carried from the prior handoff) — verified via `RETURN n.embeddings` through `tool/run-cypher`, not the node inspector.

## Verification — sandbox end-to-end

Throwaway `master_company` / `master_person` rows were created on the **sandbox** data source, run through the CREATE then SYNC writer, and the resulting graph node's vectors measured via `mvp/falkor/send-cypher`. All four writers, both phases:

| Writer | Cypher style | `name_embedding` | `embeddings` (Gemini) |
| --- | --- | --- | --- |
| `add-company-node` #2602 | template CREATE (v4) | 1536 | **3072** |
| `update-company-node` #4659 | template SET (v2) | 1536 | **3072** |
| `add-person-node` #2601 | lambda `MERGE` | 1536 | **3072** |
| `update-person-node` #2758 | template SET | 1536 | **3072** |

`name_embedding` stays 1536 (OpenAI, unchanged); `embeddings` lands at 3072 (Gemini) on every path. Temp test endpoint + probe function were deleted after verifying. (Two throwaway sandbox rows — `master_company` "Gemini Embed Test Co", `master_person` "Gemini Embed Test Person" — were left in the sandbox data source; their graph nodes were `DETACH DELETE`d.)

## What's not done

- **Backfill (deferred, by directive).** Existing business nodes still have `embeddings IS NULL`. When a backfill is wanted, the music `mvp/music/backfill-node-embeddings` (#13115) is the template — widen its label set to `Person` + `Company` (minding its embed-text branch), or write a people-side backfill; gate on `embeddings IS NULL`. Scale check first: ~6,291 `master_person` rows + companies.
- **The `:Entity(embeddings)` 3072 vector index stays DEFERRED** (write unindexed until the OpenAI `name_embedding` is deprecated) — unchanged from the music migration. The property is stored, just not yet KNN-searchable.
