## TL;DR

`#13039 mvp/get-add/master-person-new` is the **canonical entry point** for getting (or creating) a `master_person`. It hands off to three sub-functions and a downstream enrichment orchestrator:

| Stage | Function | Purpose |
| --- | --- | --- |
| 1. Enrichment cascade | `#13037 mvp/enrich/cascade-person-data` | PDL → EnrichLayer → PDL retry → Exa. Pure data fetch. |
| 2. Resolution | `#13038 mvp/resolve/match-master-person` | 4-tier dedup against existing tables. |
| 3. Downstream enrichment | `#13040 mvp/enrich/enrich-master-person` | Phase-based orchestrator (PDL backfill, EL processing, Exa, Fundable, bio, work_experience, avatar, temp-name resolution). **Replaces the retired `#12554 run-base-person-enrich`.** |

Three additional flourishes layered on top:

- **Pre-API local dedup** (v1.6) — checks `master_link` / `master_email` / `master_person.linkedin_url` against inputs **before** any external API is called. On hit: attach inputs to existing person, return. Saves PDL + EL calls on repeat-person lookups.
- **PDL retry via EL LinkedIn URL** (v1.1 cascade) — if PDL email lookup 404s but EL surfaces a LinkedIn URL, the cascade retries PDL with that URL.
- **FullEnrich async fallback** (v1.2 / #13036) — when EL via email path returns no LinkedIn URL, dispatches `#13036` async. Webhook `#8594` writes the result back into `person_enrich_data.full_enrich`.

## Inside #13039 — execution flow

The orchestrator runs in a strict order. Earlier stops are cheaper.

### 1. Boilerplate + name parsing

- Insert `log_enrichment_person` audit row (`$errorLog`).
- If `full_name` is provided → `#2649 mvp/format/name-format` (LLM split).
- Pull a candidate LinkedIn URL out of `input.links` (`array.find icontains:"linkedin.com"`).

### 2. Kill switch

- Call `#4617 mvp/stop/check-kill-switch-v2`.
- **If ON**: do a local-only lookup across `master_link`, `master_email`, `master_phone`, `master_person.linkedin_url`. If a match exists, return it. Otherwise log a `kill_switch_blocked_people` row and return null. **No external APIs called.**

### 3. Pre-API local dedup (v1.6)

The fastest cache lookup. Skips the cascade entirely when:

- Any `input.links` matches `master_link.link_url` (after URL normalization)
- Any `input.email` matches `master_email.email_address`
- The extracted `$linkedinUrl` matches `master_person.linkedin_url` directly

On hit: attach the new `input.email` / `input.links` / `input.phone_numbers` via `#2784 / #2541 / #2783`, write a `log_crash` row with `match_via`, and return. Phones are **not** checked here — that would need `#2540 mvp/phones/llm-phone-format`, which is too expensive for a pre-flight.

### 4. Cascade (#13037)

Runs only when pre-dedup misses AND `cascade_depth == 0`. Pure data fetch — no DB writes except a `log_crash` on a caught exception.

```
PDL /v5/person/enrich (email + first/last + linkedin if any)
  │
  │  if pdl.status == 404  ($pdlEmail404 = true)
  ▼
EnrichLayer (Tier 2)
  ├── email present       → #12734 api/enrich_layer/person-email  ($usedElEmail = true)
  └── elseif linkedin     → #12612 api/enrich_layer/person-linkedin
  │
  │  if EL returned a public_identifier OR linkedin_profile_url → $elLinkedinUrl
  ▼
PDL retry  /v5/person/enrich  (profile = $elLinkedinUrl)
  │  promote retry → $pdl iff status == 200 AND likelihood > 5
  │
  │  if $pdlEmail404 AND linkedin available AND $pdl.data still empty
  ▼
#12936 mvp/enrich/get-exa-profile  (Exa Tier 3)
  │
  │  if $usedElEmail AND $elLinkedinUrl is empty
  ▼
needs_full_enrich = true   (caller dispatches FullEnrich post person_enrich_data create)
```

The cascade returns one object with: `pdl`, `pdl_status`, `pdl_email_404`, `el`, `el_linkedin_url`, `el_links` (linkedin + twitter + facebook URLs deduped), `el_profile_pic`, `exa`, `linkedin_url`, `input_email_source`, `needs_full_enrich`.

### 5. Resolution (#13038)

After cascade collects all known links/emails/phones, `#13038 mvp/resolve/match-master-person` walks four lookups in order and returns the first match:

1. `master_link.link_url` (after URL normalization)
2. `master_email.email_address`
3. `master_phone.phone_number` — with **name-aware** match (first OR last name must agree) and phone-conflict detection
4. `master_person.linkedin_url` — direct fallback (catches persons that have `linkedin_url` set but no `master_link` row)

Returns `{master_person_id, match_via, status, message, formatted_phones}`. Formats every phone via `#2540 mvp/phones/llm-phone-format` internally so the orchestrator doesn't double-LLM.

### 6. Existing-person path (match_via != null)

Attach new inputs + enrichment-discovered data to the matched person, then return.

- `#2784 add-person-email` for input emails (with `orbiter_verified`) + PDL-discovered emails
- `#2541 create-master-link` for EL links (`source: "Enrich Layer"`), input links (per-URL source override), PDL links (`source: "People Data Labs"`)
- `#2783 add-person-phone` for input phones + PDL phones
- `db.edit person_enrich_data` to merge in `exa` + `enrich_layer_data`
- `#237 replace-avatar` if EL surfaced a `profile_pic_url`
- `log_crash` with `qa_passed: true, match_via: <key>` for observability
- `upsert-enrich-person` queue dispatch if `queue: true` or `cascade_depth > 0`
- **return** — function exits here

### 7. New-person path (no match)

- `db.add master_person` (placeholder name if EL/PDL produced none)
- `#2601 mvp/node/add-person-node` — creates the FalkorDB graph node
- `#12820 mvp/format/set-person-names` — single writer for first/middle/last/suffix/nickname
- `db.edit person_enrich_data` to seed `people_data_labs`, `exa`, `enrich_layer_data`
- `#237 replace-avatar` if EL surfaced a `profile_pic_url`
- `db.add enrich_history_person` rows — one per source that returned data (`"People Data Labs"` + `"Enrich Layer"`)
- **FullEnrich dispatch** — see next section
- Attach all input + PDL emails/links/phones via the helper functions
- `#12793 mvp/fundable/link-fundable-person` if a `fundable_people` row matched

### 8. Downstream enrichment dispatch (post-create)

After the new person is committed:

| Condition | Action |
| --- | --- |
| `input.hold_downstream == true` | Log skip, do nothing (interactive QA mode). |
| `personAdded && !shouldQueue` | **`debug.stop` halts the function**, then would dispatch `#13040 mvp/enrich/enrich-master-person` async-shared. |
| `personAdded && shouldQueue` | `upsert-enrich-person` writes a queue row instead. |

`shouldQueue = $input.queue OR cascade_depth > 0`.

## FullEnrich async tier

When the cascade reports `needs_full_enrich: true` (EL via email path returned no LinkedIn URL), `#13039` dispatches `#13036` after the `person_enrich_data` row is created.

### Trigger condition

All three must be true:

- `cascadeResult.needs_full_enrich == true`
- `input.email|first` is non-empty
- `personEnrichData.id` resolved (the Section 9 `db.edit person_enrich_data as $personEnrichData` returned a row)

This fires only on the **new-person creation path**. Existing-person matches do not dispatch FullEnrich.

### Call flow

```
#13039 (after person_enrich_data created)
  │  needs_full_enrich == true
  ▼
function.run "enrich/full-enrich-reverse-email-lookup"  (async-shared)
  │   input: { email, person_enrich_data_id, data_source }
  │
  ▼
#13036  POST https://app.fullenrich.com/api/v2/contact/reverse/email/bulk
        ── webhook_url    = $env.FULL_ENRICH_WEBHOOK_URL
        ── name           = "ped__{id}__{data_source}__{email}"
        ── custom         = { person_enrich_data_id, data_source }
  │
  │   (returns immediately with FullEnrich's enrichment_id)
  ▼
… time passes (seconds to minutes) …
  │
  ▼
FullEnrich POST → api:8hTTyqsU/webhook/contact-finished  (#8594)
  │
  │   per item:
  │     1. read custom.person_enrich_data_id (fall back to name parse)
  │     2. read custom.data_source
  │     3. if data_source matches a known label ("live"/"staging"/"sandbox")
  │           → db.set_datasource { value = "<literal>" }
  │     4. db.edit person_enrich_data { full_enrich: item } by id
  │     5. fallback: db.edit email_enrich_fallback by input.email
  │
  ▼
person_enrich_data.full_enrich  (json column on table 500)
```

### Why async + threaded data_source?

FullEnrich's API is **asynchronous** — POSTing returns an `enrichment_id` immediately, the profile arrives via webhook (typically seconds to minutes). Blocking the orchestrator on it would wreck p95 latency.

FullEnrich's webhook is a fresh HTTP request that defaults to the `live` data source. When `#13039` is called in sandbox / staging, the dispatch threads the `data_source` value through `custom`, and the webhook calls `db.set_datasource` with a literal `"live"` / `"staging"` / `"sandbox"` per branch (Xano won't accept a `$var` reference there). Without this, sandbox QA's writebacks would land in live.

## Source attribution rules

The `source` text written to `master_link`, `master_email`, `master_phone`, `master_avatar` records where each row originated.

| Source value         | When                                                                                       |
| -------------------- | ------------------------------------------------------------------------------------------ |
| `"People Data Labs"` | URL came from PDL `profiles[]`, OR caller-supplied URL also appears in PDL response       |
| `"Enrich Layer"`     | URL or avatar came from EL fallback (`#12734` or `#12612`)                                 |
| `$input.source`      | Caller-supplied data not confirmed by any enrichment provider                              |
| `null`               | Cascade ran but nothing resolved (`cascade_depth > 0` or both APIs returned nothing)       |

EL links are written **first** in the create-master-link loops (sections 7 + 10), so EL wins attribution when a URL appears in multiple sources (because `#2541 create-master-link` dedups on normalized URL).

`$inputEmailSource` is the email-specific override:

- PDL initial status ≠ 404 → `"People Data Labs"`
- PDL was 404 + EL returned a linkedin → `"Enrich Layer"`
- Otherwise → null

This only affects email writes; link/phone writes still use `$input.source`.

## `source` vs `data_source` (the confusingly-similar inputs)

| Input | Type | What it's for | Where it goes |
| --- | --- | --- | --- |
| `source` | `text` | **Business attribution.** Values like `"People Data Labs"`, `"Hubspot"`, `"User Input"`. | Written to `master_link.source`, `master_email.source`, `master_phone.source`. |
| `data_source` | `text` | **Xano database environment selector.** Values `"live"` / `"staging"` / `"sandbox"`. | Never written to any business table; only echoed to FullEnrich's webhook so it can `db.set_datasource` before writing the FullEnrich payload back. |

These never overlap. `source` could be `"People Data Labs"` while `data_source` is `"sandbox"`.

## `orbiter_verified` flag

When the caller passes `orbiter_verified: true`, **only the user-supplied input emails** get marked `master_email.orbiter_verified = true`. PDL-discovered emails always stay `false` (they aren't user-verified). Wired through `#2784 mvp/email/add-person-email` v1.2.

## `hold_downstream` for QA

Pass `hold_downstream: true` to skip BOTH the direct `#13040` async dispatch AND the queue path. A `qa_passed` row is logged to `log_crash` noting the skip. Use this to run `#13039` in isolation while step-debugging — note that `master_link` table-level triggers (Twitter / LinkedIn scrapers) will still fire because those are table-level.

## Observability — `match_via`

Every `log_crash` row from a successful dedup carries `data.match_via` showing exactly which key resolved the person:

| `match_via` value              | Meaning                                                                                       |
| ------------------------------ | --------------------------------------------------------------------------------------------- |
| `master_link`                  | A normalized URL in input matched an existing `master_link` row.                              |
| `master_email`                 | An email in input matched an existing `master_email` row.                                     |
| `master_phone`                 | A formatted phone matched AND first/last name agreed.                                          |
| `master_person.linkedin_url`   | Fallback — only the direct `master_person.linkedin_url` column held the URL (no master_link). |
| `null`                         | No match → new person created.                                                                |

Pre-API dedup hits log with the same `match_via` keys but `note: "pre-API dedup hit — skipped cascade"`.

## Migration from #12553

`#12553` v3.21 and `#13039` v1.0 were drop-in identical. **`#13039` v1.1+ has diverged**:

1. New PDL retry path
2. FullEnrich async fallback
3. `el_links` array writing all EL-discovered links (not just linkedin)
4. Pre-API local dedup
5. `#13040` downstream replacing `#12554`

To migrate a caller:

1. Run `#13039` with `hold_downstream: true` first — compare `master_person`, `master_link` rows, and `log_crash` audit against `#12553`. Expect divergence on email-only inputs where PDL 404s and EL has no LinkedIn (`#13039` additionally writes a `person_enrich_data.full_enrich` row when FullEnrich's webhook lands).
2. Once parity is verified, switch the caller's `function.run` from `mvp/get-add/master-person` → `mvp/get-add/master-person-new`.
3. After all callers have moved, retire `#12553`.

## Caveats

- Cascade calls happen in one sub-function, so synchronous latency is the same as inline. The v1.1 PDL retry adds one extra synchronous PDL call when EL produced a LinkedIn URL.
- `#13038 match-master-person` always formats every supplied phone via `#2540` (it doesn't skip once a match is found, because the caller still needs the formatted list for downstream writes).
- FullEnrich is asynchronous: `person_enrich_data.full_enrich` is populated **after** `#13039` returns. Anything that needs the FullEnrich payload must read from the table or be triggered by a table trigger, not by `#13039`'s synchronous response.
- Existing-person matches (Section 7) do **not** dispatch FullEnrich. Re-runs against an existing person re-attach inputs but don't refresh FullEnrich.
