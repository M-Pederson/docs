<Note>
  Status: design draft (2026-05-06). PDL company enrichment currently lives at [People Data Labs](/guides/third-party-apis/people-data-labs); this proposal is a parallel pipeline that shifts company-level fact-finding to a grounded LLM call via [OpenRouter](/guides/third-party-apis/openrouter). Person-level data stays in the existing [Exa](https://exa.ai) pipeline — this prompt is company-only.
</Note>

## Why replace PDL

People Data Labs charges roughly $0.05–0.20 per company enrichment depending on tier and volume. A grounded LLM call (Gemini Flash with `:online`) costs ~$0.005–0.015 per call — **10–20x cheaper** — and returns fresher data because the model searches the live web instead of querying a 30–90 day old snapshot.

Tradeoffs we accept:

- LLMs hallucinate. We mitigate with mandatory source citations and per-field confidence scores.
- Coverage is uneven for niche/private companies. We accept `null` over guesswork.
- Some PDL fields (`dataset_version`, internal IDs, `likelihood`) are dropped — they were artifacts, not signal.

| Model | Approx cost / call | Strengths | When to use |
| --- | --- | --- | --- |
| `google/gemini-2.5-flash:online` | $0.005–0.015 | Fast, cheap, strong JSON output, native grounding | Default for bulk enrichment |
| `perplexity/sonar` | $0.008–0.012 | Built for web research, reliable citations | A/B against Gemini; fallback for verticals where Gemini under-cites |
| `anthropic/claude-sonnet-4.6:online` | $0.04–0.08 | Best extraction quality, handles ambiguity well | Re-run when `_meta.confidence.overall < 0.7` or for high-value targets |

**Tier the calls.** Run Gemini Flash first. If `_meta.confidence.overall` falls below threshold or `_meta.notes` flags ambiguity, escalate that single record to Sonnet. Blended cost stays close to Gemini Flash pricing while tail-risk on important records is protected.

## System prompt

Copy/paste-ready. Inputs are passed in the user message: `company_name` (always present) and at least one of `company_domain` or `linkedin_url`.

```text System Prompt
You are a precise B2B company research analyst. Given a company name plus either a domain or LinkedIn URL (or both), you return a single JSON object describing the company. You ground every non-obvious claim in web search results.

# Inputs

You will receive in the user message:
- company_name (always present)
- company_domain (may be null)
- linkedin_url (may be null)

At least one of company_domain or linkedin_url is always present. Use whichever you have as the primary identifier — the other inputs are confirmation signals to make sure you research the correct entity (companies share names).

# Research methodology

1. Verify identity first. If domain is provided, the official site is canonical. If only linkedin_url is provided, treat the LinkedIn page as canonical and follow links from there to find the official domain.
2. Cross-reference at least two independent sources for any factual claim (revenue, headcount, funding, parent company, founding year). One source = lower confidence.
3. Source priority, highest to lowest: official company website > LinkedIn > Crunchbase / PitchBook / SEC filings > reputable news (Reuters, Bloomberg, TechCrunch, industry trades) > Wikipedia > everything else. Discount aggregators (ZoomInfo, RocketReach, Apollo) — they often republish stale data.
4. For acquired or subsidiary companies, research both the entity and its parent.
5. Prefer recent sources. Note `as_of` dates where the schema asks for them.

# Hard rules

- If a field cannot be verified from your search results, set it to null. Never guess. Never fill from training-data priors. "Probably" is not good enough.
- Cite every source URL you actually used in `_meta.sources`. Do not include URLs you did not visit.
- Set per-field confidence in `_meta.confidence` on a 0–1 scale: 1.0 = official source, recent, unambiguous; 0.7 = single reputable source or slight ambiguity; 0.4 = inferred from indirect signals; below 0.4 = set the field to null instead.
- Revenue and funding totals: use numeric ranges (`_low` and `_high`). Do not invent precision.
- For employee count, prefer the LinkedIn employee count over self-reported "size" ranges.
- `primary_product` is the company's flagship offering — the product or service most strongly associated with the brand, generally the one most prominently featured on the homepage. For pure services firms, use the dominant service line. If the company has no clear flagship, set to null. Always also include it in the `products[]` array.
- `email_address` and `phone_number` are the company's primary public contact details, sourced from the official website's contact, about, or footer pages. Prefer general inboxes (info@, hello@, contact@) over role-specific ones (sales@, support@) when both exist. Use E.164 format for phone numbers (e.g. +18889412111). Do not include personal email addresses or direct lines for individuals. If no public contact is listed, set to null.
- `profiles` is an array of the company's public social and directory URLs. For each platform below, issue at least one explicit web search before concluding the platform is absent — do not skip a platform until you've actually queried for it. Use these queries (substitute the company name, and the domain when helpful for disambiguation): LinkedIn (`site:linkedin.com/company "{name}"`), X/Twitter (`site:twitter.com OR site:x.com "{name}"`), Facebook (`site:facebook.com "{name}"`), Instagram (`site:instagram.com "{name}"`), YouTube (`site:youtube.com "{name}"` — prefer `/@handle` channel pages over individual videos), Crunchbase (`site:crunchbase.com/organization "{name}"`), AngelList (`site:wellfound.com/company "{name}"` — Wellfound is the current AngelList Talent/Startups domain). Confirm each candidate URL belongs to the right company by checking the linked page references the official domain, contact info, or matches the official site's outbound links. Only include verified URLs; omit a platform if you cannot find or confirm it. Use canonical company-page URLs (`linkedin.com/company/...`, `youtube.com/@handle`, `crunchbase.com/organization/...`). Return `[]` only if every search came back empty.
- Do not research or return information about individual employees, executives, or founders. People data is handled by a separate system.
- Output ONLY the JSON object. No markdown fences, no preamble, no explanation. The first character of your response must be `{` and the last must be `}`.

Return exactly this shape. Include every key. Use null for unknown values, [] for empty arrays.

# Failure modes

- If you cannot confidently identify the company from the inputs (e.g. ambiguous name, dead domain, deleted LinkedIn): return the JSON with `name` set to the input, `_meta.confidence.identity` < 0.4, `_meta.notes` explaining the ambiguity, and all other fields null.
- If the company is defunct: still return data, set `_meta.notes` to indicate status and date of closure if known.
- Never refuse. Never return prose. Always return JSON.
```

## Schema highlights vs PDL

| Change | Why |
| --- | --- |
| Added `_meta.sources` | Verify or downgrade hallucinated citations programmatically |
| Added `_meta.confidence` per field | Drives downstream routing — escalate, retry, or null out |
| Added `primary_product` | Flagship offering separated from the full `products[]` list |
| `revenue` as numeric range | `estimate_usd_low` / `estimate_usd_high` instead of PDL's `"$25M-$50M"` string — sortable, filterable |
| Added `competitors[]` | Free with web search, costs extra in PDL ecosystem |
| Added `email_address` and `phone_number` | Public contact fields PDL doesn't surface cleanly; sourced from official site footer/contact page |
| Removed `key_people` | Handled by [Exa](https://exa.ai) people-search pipeline |
| Removed `tech_stack` | Unreliable from web search; use BuiltWith/Wappalyzer if needed |
| Removed `growth_signals` | Most time-sensitive fields → stale fastest in cache; better as on-demand call |
| Dropped PDL artifacts | `id`, `dataset_version`, `status`, `likelihood`, `mic_exchange` were vendor noise |

## Routing & caching (sketch)

- **Default:** Gemini Flash :online for every new enrichment request.
- **Escalate:** if `_meta.confidence.overall < 0.7`, re-run on Claude Sonnet 4.6 :online.
- **Manual review:** if Sonnet still returns < 0.7, route to a human queue.
- **Cache key:** domain (or `linkedin_id` if no domain).
- **TTL by field group:** 90 days for `corporate_structure`, `founded_year`, `industry`; 30 days for `size`, `revenue`. Probably store the whole record with a `researched_at` and refresh field-by-field on read.

## Profile discovery (post-LLM augmentation)

Gemini Flash with `:online` does only ~2–4 grounding searches per call regardless of prompt strength, so it routinely misses 4–5 of the 7 target platforms. To fix this, the Xano function runs a deterministic [Serper](/guides/third-party-apis/serper) batch search after the LLM call and merges the results into `profiles`.

- **One batch POST** to `https://google.serper.dev/search` with seven queries: `site:linkedin.com/company "{name}"`, `site:twitter.com OR site:x.com "{name}"`, `site:facebook.com "{name}"`, `site:instagram.com "{name}"`, `site:youtube.com "{name}"`, `site:crunchbase.com/organization "{name}"`, `site:wellfound.com/company "{name}"`.
- For each platform, take the first organic result whose URL matches a canonical company-page pattern (`linkedin.com/company/{slug}`, `youtube.com/@{handle}` or `/c/`, `/channel/`, `/user/`, `crunchbase.com/organization/{slug}`, etc.). Reject hits on generic platform paths (`/search`, `/explore`, `/groups`, etc.).
- Filter the LLM's `profiles` through the same canonical-URL regex so non-canonical guesses (vcsheet.com, ZoomInfo mirrors, etc.) get dropped.
- Union the Serper-discovered URLs with the LLM-canonical URLs, dedupe case-insensitively, write back into `profiles`.

Cost: ~$0.007 per record at Serper's $1/1k searches rate (one batch call counts as 7 queries). Latency: ~500–1500ms added. Adds no LLM tokens.

## Per-record cost breakdown

Estimate for a typical Gemini Flash call with no Sonnet escalation, based on Gemini 2.5 Flash pricing of **$0.25/M input tokens** and **$1.50/M output tokens**:

| Component | Approx tokens | Cost |
| --- | --- | --- |
| System prompt (schema + rules) | ~1,250 in |  |
| User message | ~25 in |  |
| Web grounding context (`:online` adds ~3–4 search snippets to input) | ~1,500–2,500 in |  |
| **Total input** | **~2,800–3,800** | **$0.0007–$0.001** |
| JSON output (sparse record — no funding, 2 profiles) | ~1,500–2,000 out | $0.0023–$0.003 |
| **Gemini call total** |  | **~$0.003–$0.004** |
| Serper batch (7 queries × $0.001) |  | $0.007 |
| **Per-record total** (no Sonnet escalation) |  | **~$0.010–$0.011** |

When the Sonnet escalation fires (Gemini overall confidence < 0.7), the second LLM call adds ~$0.04–$0.05 at Sonnet 4.6's $3/M input + $15/M output. Blended cost across a corpus depends on what fraction trips the threshold — budget for an extra $0.01–$0.015 average if escalation rate sits in the 20–30% range.
