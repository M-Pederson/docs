**Status: open work / plan.** Nothing on this page is built. Fundable is a living database in BigQuery; the Xano `fundable_*` tables are currently a static import. The lane being built: a sync cron pulls new/changed rows from BigQuery into the mirror tables and writes a **changelog**; after the cron completes, a **worker** loops the changelog and dispatches graph/table updates. The dispatch targets are almost entirely **existing** workspace-3 functions — this page maps each change case to the function that already handles it, and calls out the small set of genuinely new pieces.

<Note>
  Sibling pages: [Fundable table updates](/guides/open-work/fundable-table-updates) owns the sync/changelog/queue infrastructure design and its AlloyDB target runtime; [Fundable investor enrich](/guides/open-work/fundable-investor-enrich) is the on-demand sparse-investor identity resolver. This page owns the piece between them — the **changelog → action dispatch map**. The map is runtime-agnostic: built in Xano first, every action below is a direct `function.run` target; moved to the AlloyDB backend later, the same map is the behavioral contract for the port (the way the [Company Go Rebuild Spec](/guides/enrichment/waterfall/company-go-rebuild-spec) treats Xano functions as reference behavior).

## The three graph-linked tables

Only three fundable tables link directly to canonical rows and node UUIDs; everything else rolls up through them. The link columns already exist in the schemas — no migration needed to start.

| Mirror table | Link columns (already in schema) | Graph identity |
| --- | --- | --- |
| `fundable_organizations` #666 | `master_company_id` → `master_company` #142, plus `master_company_node_uuid` | `Company` node (uuid mirrored on `master_company.node_uuid`) |
| `fundable_people` #667 | `master_person_id` → `master_person` #139, plus `master_person_node_uuid` | `Person` node |
| `fundable_deals` #671 | `funding_round_node_uuid` plus `processed_at` | `Funding_Round` node — MERGE-keyed `uuid = fundable_deals.id`, so the deal id **is** the node uuid |

Child tables carry no node UUIDs of their own; a change to any of them means "re-replay the parent deal":

| Child table | Rolls up to |
| --- | --- |
| `fundable_institutional_investments` #678 | parent `deal_id` |
| `fundable_institutional_investments_person` #681 | grandparent deal via `institutional_investment_id` |
| `fundable_angel_investments` #679 | parent `deal_id` |

Changelog-only in v1 (no graph action): `fundable_articles` #680, `fundable_valuations` #672, `fundable_organization_associations` #673, the industry tables #670/#674, and the `fundable_linkedin_*` family #668/#669/#675/#676/#677.

## Building the changelog

The changelog is the contract between the sync cron and the worker: the cron writes it inside the same batch as the mirror upserts, and the worker never re-diffs anything — it trusts `changed_fields`, the resolved refs, and the link snapshot below. Run-level state (`queue_stage`, watermark, counts) lives on `fundable_sync_runs`; one changelog row describes one changed source record.

### Watermark semantics

The window is **the last successful run's watermark vs the source's `updated_at`** — with four refinements:

1. **Watermark of the last successful run, not the cron's wall-clock.** Each run stores `watermark_after` on `fundable_sync_runs`; the next run fetches `updated_at` newer than `watermark_after - lookback`, up to a `window_end` captured **before** fetching starts. A failed run never advances the watermark, so nothing it missed gets skipped.
2. **Overlap on purpose; the diff makes it free.** The lookback buffer (start at 10 minutes) deliberately re-fetches a slice of already-seen rows to absorb clock drift between BigQuery and the runner, plus late-committing rows. Because every fetched row is diffed against the mirror before a changelog row is written, overlap produces zero duplicate changelog entries — an unchanged re-fetch is bookkeeping only.
3. **Trust `updated_at` only after probing it.** Every `fundable_*` table carries `updated_at` from the feed, but the build must confirm the feed actually maintains it — that it moves on every field change and is not reset by bulk rebuilds of `fundable_data_feed_production`. Probe each table via `_bq-execute` #12690 before wiring the cron; any table with unreliable timestamps falls back to full scans with row compare.
4. **`updated_at` windows never see deletes.** A row removed upstream simply stops appearing in the window — nothing announces it. `remove` detection is a separate, lower-frequency reconciliation pass: ids present in the mirror but absent from BigQuery become `remove` changelog rows (tombstone only in v1 — no graph deletes).

Each table gets its own window on its own `updated_at` — a new investor row on an old deal surfaces in the `fundable_institutional_investments` window even though the deal row itself never changed, which is exactly what rolls up to a parent-deal replay.

### Per-batch build order

1. **Fetch** the window for one table via #12690.
2. **Sync tables in FK order** — organizations → people → deals → institutional investments → investments\_person → angel investments → articles — so parents land in the mirror before children arriving in the same window, and ref resolution never misses.
3. **Classify each fetched row against the mirror**: no mirror row → `insert`; any business field differs → `update` (build `changed_fields` plus a shallow before/after `diff_json`); identical → unchanged, no changelog row.
4. **Upsert the mirror row** — and never write the `master_*` / node-uuid backlink columns from BigQuery data (the feed doesn't know them; #12691/#12692 already preserve them across re-upserts — the new upsert helpers must follow the same rule).
5. **Resolve entity refs** (table below) and snapshot the join/link columns from the resolved rows.
6. **Precompute the gate**: `graph_eligible` \+ `skip_reason` per the Stage-1 rule.
7. **Write the changelog row**: `processing_status = "logged"` when eligible, `"ignored"` otherwise.
8. **Close the run**: stamp `watermark_after` and per-table fetched/changed/ignored counts.

### Every row resolves to organizations, deals & people

No matter which physical table changed, the changelog row must name the core entities it touches, so all filtering and dispatch happens in terms of the three graph-linked tables. Changes to join tables appear in the changelog **attributed to the org / deal / person they join to**:

| Changed table | `fundable_org_id` | `fundable_deal_id` | `fundable_person_id` |
| --- | --- | --- | --- |
| `fundable_organizations` | itself | — | — |
| `fundable_people` | — | — | itself |
| `fundable_deals` | portfolio org via `organization_id` | itself | — |
| `fundable_institutional_investments` | portfolio org via `deal_id` (investor org → `secondary_refs_json`) | `deal_id` | — |
| `fundable_institutional_investments_person` | portfolio org via investment → deal (investor org → secondary) | via `institutional_investment_id` | `person_id` |
| `fundable_angel_investments` | portfolio org via `deal_id` | `deal_id` | `person_id` |
| `fundable_organization_associations`_(changelog-only v1)_ | `organization_id` | — | `person_id` |
| `fundable_valuations`_(changelog-only v1)_ | `organization_id` | — | — |
| `fundable_articles`_(changelog-only v1)_ | — | deals in the `canonical_deal_id` group → secondary | — |
| `fundable_linkedin_person` / `_positions` / `_person_education`_(changelog-only v1)_ | — | — | `person_id` |

Deal-rooted rows always resolve `fundable_org_id` to the **portfolio** org — that is the org whose master link gates eligibility. Investor orgs ride along in `secondary_refs_json`; an unlinked investor is non-gating by design (#12702 creates investor participants as local leaves).

### Link snapshot — the pre-unlimited-PDL filter

At changelog-write time the cron copies the canonical join state from the resolved rows onto the changelog row itself:

| Snapshot column | Copied from |
| --- | --- |
| `master_company_id`, `master_company_node_uuid` | the resolved `fundable_organizations` row |
| `master_person_id`, `master_person_node_uuid` | the resolved `fundable_people` row |
| `funding_round_node_uuid`, `deal_processed_at` | the resolved `fundable_deals` row |
| `graph_eligible`, `skip_reason` | computed from the above per the Stage-1 gate |

This is what makes the pre-unlimited-PDL stage cheap: the worker's candidate query is one indexed read — `processing_status = "logged" AND graph_eligible = true` — and every change without master links is already parked as `ignored` with a reason, without a single join at worker time. When unlimited PDL arrives (Stage 2), the same rows re-qualify: the resolution columns say exactly which org / deal / person each ignored row needs bootstrapped first.

<Note>
  **Snapshot for filtering, live rows for acting.** Links move between sync and dispatch — an org gets linked by an entry gate, a merge re-points a node uuid (the resync sweeps #12963/#12964 exist precisely for that). The worker filters on the snapshot but re-reads the live `fundable_*` and `master_*` rows at dispatch time before calling anything.
</Note>

**Releasing held rows.** When an org or person gains its master link after its changes were parked (case 6's link-only reconcile, #12810 Block-1 domain repair, #12793/#12794, or any entry-gate crossing), the parked rows re-open with one indexed query: `processing_status = "ignored" AND fundable_org_id = <newly linked org>`. v1: run this as a **release sweep** at the top of every worker run. Optional later: a Xano table trigger on the `fundable_organizations.master_company_id` / `fundable_people.master_person_id` null→set transition that writes a `link_established` changelog row instead.

### Proposed changelog schema

```text
identity/audit   id · sync_run_id · source_table · source_id · change_type (insert|update|remove)
                 changed_fields (text list) · diff_json · created_at · updated_at
resolved refs    fundable_org_id · fundable_deal_id · fundable_person_id · secondary_refs_json
link snapshot    master_company_id · master_company_node_uuid
                 master_person_id · master_person_node_uuid
                 funding_round_node_uuid · deal_processed_at
gate             graph_eligible (bool) · skip_reason
lifecycle        processing_status (logged|queued|complete|ignored|failed)
                 dispatched_function · result_json · processed_at
indexes          (processing_status, created_at) · fundable_org_id · fundable_deal_id
                 fundable_person_id · sync_run_id
```

Keep the field names aligned with the sibling page's AlloyDB DDL so the eventual port is mechanical; the resolved-ref and link-snapshot columns are this page's addition to that design.

## The workhorse: the existing deal cascade

The hard part of changelog processing already exists, is already idempotent, and was end-to-end validated in the 2026-07-02 Deel smoke tests (run 5: 52 PASS / 0 FAIL). The worker mostly just needs to aim it.

<Card icon="play">


  **The per-deal replay unit.** Inputs: `fundable_deal_id` (required), `cascade_depth` (pass an explicit `0`), `origin_master_company_id` / `origin_master_person_id`. One call: ensures the portfolio company exists as a `master_company` (sync get-add, `hold_downstream: true`, local-only, not queued) → IPO signal (`company_financial.is_public` escalate when `ipo_status == "public"`) → `Funding_Round` \+ `RAISED` via #12701 → investor/partner/angel edges \+ co-investor queueing via #12702 (depth-0 gated) → stamps `fundable_deals.processed_at` **only when the node actually materialized** (v1.8).
</Card>

  Org-level fan-out: walks a fundable org's rounds **raised** (chronological, `date asc`) and rounds **invested in**, dispatching each deal into #12856 inside a per-deal `try_catch`. </Card>

Properties the worker leans on:

- **Self-gating staleness check (#12856 v1.2).** A replay is skipped unless the deal, its institutional investments, or their partner links carry an `updated_at` newer than `processed_at`. The worker can afford to over-dispatch — replaying an unchanged deal is a cheap no-op.
- **Idempotent MERGE writes.** Re-running a changed deal updates the `Funding_Round` node and edges in place; no duplicates.
- **Retry-safe failure marker (v1.8).** A node-write failure leaves `processed_at` null and logs a crash, so the changelog row can simply be retried.
- **The angel blind spot.** The staleness gate does **not** read `fundable_angel_investments` — a new angel on an already-processed deal will not re-trigger on its own. The worker must null `fundable_deals.processed_at` first (the documented escape hatch), or the gate gets the small extension proposed below.
- **Bounded participant creation.** Portfolio companies are created `hold_downstream` local-only; investors/partners/angels are local leaves plus tier-1 queue rows. A deal replay never launches recursive enrichment on its own.

## Eligibility gate

Same Stage-1 principle as [Fundable table updates](/guides/open-work/fundable-table-updates#graph-eligibility-gate): a changed row earns graph work only when it is already connected to the canonical layer. The sync precomputes this gate onto every changelog row — resolved refs, master-link snapshot, `graph_eligible` — so the worker filters without joins (see [Building the changelog](#building-the-changelog)) and re-verifies against live rows at dispatch time.

| Change on | Dispatch only when | Otherwise |
| --- | --- | --- |
| a deal (or its children) | the portfolio org (`fundable_deals.organization_id` → `fundable_organizations`) has `master_company_id` \+ `master_company_node_uuid` | mark `ignored` with a skip reason |
| an organization | `master_company_id` \+ `master_company_node_uuid` present | `ignored` |
| a person | `master_person_id` \+ `master_person_node_uuid` present | `ignored` |

This gate matters because #12856 has get-add semantics — pointed at a deal whose company we've never adopted, it will **create** that company. Against a full BigQuery mirror, ungated dispatch turns the sync into unbounded company minting. Unlinked rows are not lost: they link naturally the next time the entity crosses an entry gate (`master-company-new` #12558 §8 Fundable lookup, `master-person-new` #13039 §5b, person phase-5 #12826, company phase-6 #12810 Block-1 domain repair) — and each of those linkings is itself a changelog-worthy event that can then release the held deal replays.

`mvp/query-get-all-invetsors` #12730 returns a deal's investor org \+ angel arrays **with** their master ids and node UUIDs — a ready-made helper if the worker wants per-participant eligibility introspection.

## Worker mechanics

1. **Collapse child changes to one replay per deal.** Dedupe key `deal_replay:<fundable_deal_id>` — a new deal arriving with 3 investments and 2 partner rows is six changelog rows but **one** #12856 dispatch.
2. **Chronological order within an org.** When several deals for one company are pending, dispatch oldest-first (`date asc`) — the #12703 v2.3 lesson: `FOLLOW_ON_INVESTED_IN` detection needs earlier rounds in the graph before later ones.
3. **Release sweep first, then org link events before deal replays.** At the top of each run, re-open `ignored` rows whose org / person gained master links since parking (one indexed query on the changelog's resolved refs). After that, soft ordering — the cascade self-heals — but processing a `link-fundable-org` event first converts that org's held deal rows to eligible within the same run.
4. **Per-action `try_catch`** (the #12703 pattern): one bad deal never aborts the batch. Failures stay `failed` on the changelog for retry; #12856 v1.8 guarantees a failed node write is retryable.
5. **Always pass an explicit `cascade_depth: 0`.** The depth inputs are nullable and read via `first_notnull`; an **omitted** depth fails closed to 1, and the investor-edge half (#12702) silently never runs.
6. **Cap batch size per run** and record overflow, mirroring the 25\+25 fan-out governor thinking in #12703.
7. **Expect kill-switch outcomes.** Company #12789 / person #4617 — when ON, creation paths return local-lookup-only or blocked results. Record the outcome; don't treat it as failure.
8. **Crash logging:** `log_crash` #542 has a **strict `phase` enum** — the worker must reuse an existing phase value or extend the enum first, or its own crash logger will crash.
9. **Known side effect to inherit knowingly:** #12702 unconditionally overwrites a deal partner's `master_person.master_company_id` / `current_title` with the deal's firm context on every replay.

<Warning>
  **Queue effects.** Every deal replay writes tier-1 rows into `queue_enrich_person` / `queue_enrich_company` via #12854/#12855 — and the scheduled drains (Task #126 `process-person-queue`, Task #127 `process-company-queue`, both filtering `priority_tier` ≤ 1) still ship `active = false`. Until they are enabled (or the operator drainer `process-enrichment-queue` #12816 is run), changelog processing will **accumulate** queue rows without enriching the investors it discovers. Decide drain activation as part of go-live.
</Warning>

## Worked example — Deel raises a new round

<Steps>
  <Step title="Sync cron lands the change">
    One new `fundable_deals` row for Deel's org, three `fundable_institutional_investments`, one partner link, one angel row → six changelog rows in the same sync run.
  </Step>
  <Step title="Worker dedupes and gates">
    All six collapse to one `deal_replay` for the new deal id. Deel's org row carries `master_company_id` \+ `master_company_node_uuid` → eligible.
  </Step>
  <Step title="One dispatch">
    `#12856` with the deal id, `cascade_depth: 0`, `origin_master_company_id` = Deel's id. The staleness gate passes trivially (`processed_at` is null on a new deal — the angel blind spot only matters for angels added to already-processed deals).
  </Step>
  <Step title="The cascade does the rest">
    Portfolio ensure no-ops (Deel already linked) → #12701 MERGEs the `Funding_Round` (uuid = deal id) \+ `RAISED` edge → #12702 writes the lead/follow-on/angel/partner edge set, adds `VC_Firm` / `Angel` labels, and queues the co-investors at tier 1 → `processed_at` stamped, `funding_round_node_uuid` written back.
  </Step>
  <Step title="Close out">
    Worker marks all six changelog rows `complete` and records the dispatched function \+ result. The tier-1 queue rows wait for Task #126/#127 (or #12816).
  </Step>
</Steps>
