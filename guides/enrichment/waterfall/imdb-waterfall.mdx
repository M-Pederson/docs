## Source contract — IMDb via Firecrawl

Unlike the [Music waterfall](/guides/enrichment/waterfall/music-waterfall)'s structured JSON API, IMDb has no public API — every entity is a **Firecrawl scrape of a public IMDb page to markdown, followed by an LLM extraction to structured JSON**.

| Property | Value |
| --- | --- |
| Person source | `https://www.imdb.com/name/nm…` → `imdb_person.markdown` → LLM → `imdb_person.json` |
| Title source | `https://www.imdb.com/title/tt…` → `film_tv.markdown` → LLM → `film_tv.json` |
| Awards source | `{imdb_url}/awards/` → `award_markdown` → LLM → `awards_json` \+ `film_tv_award` rows |
| Posters | IMDb image CDN (`@._V1_` size transform recomputed) → re-hosted GCS WebP `imdb-posters/<imdb_id>` |
| Natural key | **normalized `imdb_url`** — \`trim | split:"/?" | first | rtrim:"/"\` (queues \+ source tables) |
| Staleness gate | person gate: `enriched_at` within **30 days** → no re-scrape. Title gate: **any** existing `film_tv` row dedups (v1.4) — except sparse/unenriched rows, which fall through for queue repair or sync-enrich (v1.7–v1.10); `refresh: true` forces a re-scrape |
| Rate limiting | no provider hard limit — pacing comes from the single-row cron tick (600 s per queue) |
| Firecrawl wrapper | `mvp/firecrawl/scrape` #4515 calls Firecrawl v2 `/scrape` for markdown \+ links. **v1.6 (2026-07-07):** treats Xano optional-input `0`/empty-string sentinels as missing for `timeout_ms`, `request_timeout_seconds`, and `proxy`, restoring the intended defaults of 60000 ms, 60 s, and `auto`. Required person/title page scrapes stay hard-fail; optional awards-page scrapes pass `soft_fail: true` and skip cleanly on no markdown. |
| LLM extraction | models, temps, and token ceilings per function: [Film & TV model summary](/guides/enrichment/film-tv-pipeline/model-summary) · prompts: [Film & TV system prompts](/guides/enrichment/film-tv-pipeline/llm-system-prompts) — linked, not duplicated |

<Note>
  **Rebuild-spec gaps (per the [GCP migration plan](/guides/open-work/orbiter-univers-standalone/enrichment-gcp-migration#what-already-exists-vs-what-the-spec-must-add)):** the exact Firecrawl request contract, the LLM extraction JSON schemas (`credits[]`, `film_tv` shapes), and the award-scraper output schema are not yet written down — they are Step-2 spec-extraction deliverables, to be pulled from the frozen XanoScript rather than authored by hand.

## Depth model

Identical semantics to the existing waterfall, with a hard cap at depth 2.

| Depth | Meaning | Behavior |
| :-: | --- | --- |
| **0** | Seed (user-initiated) | Full scrape \+ LLM extract \+ awards \+ cascade fanout |
| **1** | Direct discovery from a seed | Stub created via `get-add` (just `imdb_url` \+ name/title, no scrape). Queued. Cron processor scrapes \+ expands, but **only T1–T2 children are queued at depth 2** |
| **2** | Transitive (from a depth-1 entity) | Stub queued. Cron processor scrapes \+ writes data, **no further cascade** |
| **3\+** | Never queued | Hard cap |

<Check>
  **Why a hard cap at 2?** A prolific director (Neveldine: 7 director credits \+ 9 acting \+ 7 producer) at depth 0 becomes ~23 title stubs at depth 1. Each of those titles has a director \+ writer \+ DP \+ lead producer \+ 3 cast = ~7 people → ~160 people at depth 2. Capping cascade at depth 2 stops the explosion before depth 3 would reach ~1,000\+.
</Check>

## Priority tiers

### `queue_imdb_person` (1 = highest)

| Tier | Rule | Example (seed = title "Gamer") |
| :-: | --- | --- |
| **1** | Director OR Writer of seed title; award-winning/nominated person on seed | Mark Neveldine, Brian Taylor |
| **2** | Cinematographer OR lead Producer of seed title; co-winner / co-nominee on seed's award (via `shared_with`) | Ekkehart Pollack (DP), lead producers |
| **3** | Top 3 cast of seed title; other `shared_with` people | Gerard Butler, Michael C. Hall, Amber Valletta |
| **4** | Everyone else mentioned | Minor cast, thanks, video-game voice actors |

### `queue_imdb_title` (1 = highest)

| Tier | Rule | Example (seed = person "Neveldine") |
| :-: | --- | --- |
| **1** | Seed person was Director OR Writer on this title; award-winning/nominated title | Crank, Gamer, Ghost Rider, Crank: High Voltage, Jonah Hex, Pathology |
| **2** | Seed person was lead Producer (not exec) AND (rating ≥ 7 OR year = `current_year`) | Kid Revolver (8.2 Short), Great Pretenders (8.9 Short), The Olympians (upcoming) |
| **3** | Seed person was executive producer / cinematographer / minor role; OR rating ≥ 7 regardless of role | Officer Downe, Urge, Pathology (producer), Vatican Tapes (director) |
| **4** | Video games / music videos / shorts without high rating | Jonah Hex (Video Game, tt38040358) |

### Credit filter for person seed (`cascade-credits-from-person`)

```text
for each credit in imdb_person.json.credits:
    if role ∈ {director, writer}:                                → T1
    elif role = producer AND (rating ≥ 7 OR year = current_year): → T2
    elif role ∈ {executive producer, cinematographer, minor}
         OR rating ≥ 7:                                          → T3
    elif type ∈ {Video Game, Music Video}
         OR (type = Short AND rating < 8):                       → T4
    else: skip
    # T3 capped at top 20 by rating to prevent low-signal blowout
```

### Key crew filter for title seed (`cascade-crew-from-title`)

```text
from film_tv.json:
    directors[]                                      → T1
    writers[]                                        → T1
    cinematographer (if present in json)             → T2
    lead producer (first producer in credits list)   → T2
    top 3 cast (from cast[] sorted by bill order)    → T3
```

### Upsert on duplicate

When the same `imdb_url` is discovered twice (e.g., an actor appears on two seed titles), the queue entry is upserted:

- `count` is incremented
- `priority_tier` is **upgraded** to the new (lower = better) value if applicable
- No duplicate scrape is scheduled

## Seed flows

### Seed person flow (depth 0)

<Steps>
  <Step title="Kill switch (P1)">
    `check-kill-switch-imdb` (#12870). If active, return existing `imdb_person` if found; otherwise write a `log_crash` row (`kill switch active - new imdb_person blocked`) and return empty.
  </Step>
  <Step title="Dedup + scrape + LLM extract (P2)">
    Lookup by normalized `imdb_url`. If `enriched_at` within 30 days → return existing. Otherwise scrape via Firecrawl, run LLM extractor (`~google/gemini-flash-latest`), save `json` / `markdown` / `name` / `image_url` onto `imdb_person`.
  </Step>
  <Step title="Create master_person + master_link + birthday_join (P3)">
    All dedup-guarded, idempotent on re-run.
  </Step>
  <Step title="Phase 4a — CRITICAL (P4a)">
    `mvp/avatar/replace-avatar` \+ `mvp/bios/add-person-biography` \+ `mvp/node/add-person-node` \+ `mvp/node/update-person-node`. Stamp `node_uuid` \+ `master_person_id` back onto `imdb_person`. Run explicit `SET p.visibility = true` Cypher.
  </Step>
  <Step title="Phase 4b — OPTIONAL (P4b)">
    `mvp/bios/create-llm-person-bios` \+ `mvp/deep-research/deep-person-basic` \+ `mvp/social/social-insights/get-social-insights` \+ `mvp/expertise/llm-identify-person-expertise` (#12666) \+ `kickoff-cross-waterfalls-from-expertise` (#13185 — `run_imdb: false` / `run_music: true`, so a `music_industry` flag can bridge into MusicBrainz). All failures non-fatal.
  </Step>
  <Step title="Phase 4c — credit edges (P4c)">
    `mvp/imdb/add-credit-edges-from-person` (#13013). Writes `(Person)-[:WORKED_ON]->(Film_TV)` edges — `uuid` \+ description stamped at write time — for every credit whose film's `node_uuid` is already stamped, and `imdb_credit_join` rows (pending `edge_created: false` for stub-only films).
  </Step>
  <Step title="Awards (P5)">
    `add-award-records-from-imdb-person` (#12887) — scrape `{imdb_url}/awards/`, LLM extract, write `film_tv_award` rows. Passes `cascade_depth` through. Bulk award **graph** writes are deferred to `tool/backfill-award-edges-imdb` #13014 batches (v1.1p).
  </Step>
  <Step title="Cascades (P6)">
    `cascade-credits-from-person` (with `sync_top_credits: 5`), then a #13013 replay so sync-created titles get their `WORKED_ON` edges immediately (v1.9), then `cascade-awards-from-person`.
  </Step>
  <Step title="Mark success (P7)">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

### Seed title flow (depth 0)

<Steps>
  <Step title="Kill switch (P1)">
    Same as person flow.
  </Step>
  <Step title="Scrape + LLM extract + tagline + save (P2)">
    Scrape \+ LLM (`google/gemini-3.1-flash-lite`), payload cleaned via the IMDb-specific `sanitize-title-json` (#13182). Save `json` / `markdown` / `title` / `year` / `imdb_rating` / etc. Generate \+ save tagline via `tagline-generator` (#10516 v4). Canonicalize the extracted poster URL, upload to GCS via `save-imbd-poster` (#13044), and persist the public URL on `film_tv.image_url` (IMDB-URL fallback on upload failure).
  </Step>
  <Step title="Film_TV node (P3)">
    `mvp/node/add-film-tv-node` (#2320 v2, MERGE-keyed on `imdb_url`) \+ re-stamp `node_uuid` \+ register the UUID in `node_registry`.
  </Step>
  <Step title="Credit edges (P3.5)">
    `mvp/imdb/add-credit-edges-from-title` (#13015). Writes `(Person)-[:WORKED_ON {role}]->(Film_TV)` edges — `uuid` \+ description stamped at write time — for every cast/crew member whose `imdb_person.node_uuid` is already stamped, plus `imdb_credit_join` rows.
  </Step>
  <Step title="Awards (P5)">
    `add-award-records-from-title` (#12879) — optional scrape of `{imdb_url}/awards/` with `soft_fail: true`; visible Winner/Nominee rows parse deterministically, the LLM (`deepseek/deepseek-v4-flash`) only cleans up (v1.15). Writes `film_tv_award` rows; resolves `primary_nominee_imdb_url` via `get-add/imdb-person` at T2/depth\+1. Bulk award **graph** writes are deferred to #13014 batches (v1.17).
  </Step>
  <Step title="Cascades (P6)">
    `cascade-crew-from-title` \+ `cascade-awards-from-title` (seed ids validated first, v1.14).
  </Step>
  <Step title="Mark success (P7)">
    `enriched_at = now`, `enrich_success = true`, `cascade_depth = 0`.
  </Step>
</Steps>

### Depth-1 queued processing

When the cron picks up a queue entry, it sets `processing: true` and calls the same orchestrator with `cascade_depth: 1`. The orchestrator runs all phases **but the cascade helpers (P6) only queue T1–T2 discoveries at depth 2**.

### Depth-2 queued processing

Cron picks up, sets `processing: true`, runs orchestrator with `cascade_depth: 2`. **Cascade helpers are skipped entirely.** Entity gets scraped, data written, awards captured — but no children are queued.

## Cascade example — seed person end-to-end

Here's what happens end-to-end when a person enters as a depth-0 seed via `get-add/imdb-person` (`cascade_depth: 0`, `queue: false`):

```text
Depth 0: seed person (imdb_url, cascade_depth: 0, queue: false)
├── get-add gate (get-add/imdb-person)
│   ├── dedup — normalized imdb_url already in imdb_person with enriched_at
│   │   within 30 days → return existing, no scrape
│   ├── stub exists → seed_imdb_person_id set to the stub's own id
│   │   (no explicit seed passed)
│   └── depth 0 AND queue == false → orchestrator invoked synchronously
│       → returns action: "enriched_direct", no queue entry written
│
└── orchestrator (cascade_depth: 0)
    ├── P1  kill switch — check-kill-switch-imdb #12870
    │       active → return existing imdb_person, else write a log_crash
    │       row + return empty
    ├── P2  scrape + LLM extract — Firecrawl → LLM extractor
    │       (~google/gemini-flash-latest) → save json / markdown / name /
    │       image_url onto imdb_person
    ├── P3  master_person + master_link + birthday_join
    │       (dedup-guarded, idempotent on re-run)
    ├── P4a CRITICAL — replace-avatar + add-person-biography + add-person-node
    │       + update-person-node → stamp node_uuid + master_person_id back onto
    │       imdb_person → SET p.visibility = true
    ├── P4b OPTIONAL — create-llm-person-bios + deep-person-basic
    │       + get-social-insights + llm-identify-person-expertise #12666
    │       + kickoff-cross-waterfalls-from-expertise #13185
    │         (run_imdb: false — a music_industry flag bridges into MusicBrainz)
    │       (all failures non-fatal)
    ├── P4c credit edges — add-credit-edges-from-person #13013
    │       → (Person)-[:WORKED_ON]->(Film_TV) per credit whose film's
    │       node_uuid is already stamped (uuid + description at write time;
    │       imdb_credit_join rows, pending edge_created:false for stubs)
    ├── P5  awards — add-award-records-from-imdb-person #12887
    │       (cascade_depth passed through)
    │       → Firecrawl {imdb_url}/awards/ → LLM extract → film_tv_award rows
    │       (bulk award graph writes deferred to backfill-award-edges-imdb #13014)
    ├── P6  cascades (pure dispatchers; each get-add call try_catch → log_crash)
    │   ├── cascade-credits-from-person (sync_top_credits: 5)
    │   │   ├── per credit → title stub via get-add (imdb_url + title only,
    │   │   │   no scrape) → queue_imdb_title at depth 1:
    │   │   │   ├── director / writer                                   → T1
    │   │   │   ├── producer AND (rating ≥ 7 OR year = current_year)    → T2
    │   │   │   ├── exec producer / cinematographer / minor role,
    │   │   │   │   OR rating ≥ 7                                       → T3
    │   │   │   │   (T3 capped at top 20 by rating)
    │   │   │   ├── Video Game / Music Video, or Short with rating < 8  → T4
    │   │   │   └── else skip · duplicate imdb_url → count++ + tier upgrade
    │   │   │       (single queue row per URL, no duplicate scrape)
    │   │   └── top 5 credits sync-enriched inline (person at depth 0) —
    │   │       the seed returns with enriched films before the title cron runs
    │   ├── add-credit-edges-from-person #13013 replay — sync-created
    │   │   Film_TV nodes receive WORKED_ON edges immediately
    │   └── cascade-awards-from-person
    │       └── shared_with people → person stubs → queue_imdb_person at depth 1:
    │           ├── co-winner / co-nominee on the seed's award → T2
    │           └── other shared_with people                   → T3
    └── P7  mark success — enriched_at = now, enrich_success = true,
            cascade_depth = 0

Depth 1: cron drains both queues
         (single-row tick, 600 s per queue; both crons ship active = false)
├── pick order: priority_tier asc, cascade_depth asc, id asc → processing: true
├── same orchestrator with cascade_depth: 1 — ALL phases run
│   (scrape + LLM extract + master rows + edges + awards)
└── P6 cascade helpers only queue T1–T2 discoveries at depth 2
    (T3–T4 discarded)

Depth 2: terminal
├── cron picks up → same orchestrator with cascade_depth: 2
├── entity scraped, data written, awards captured — cascade helpers skipped
│   entirely, no children queued
└── depth 3 never queued — helpers guard cascade_depth + 1 >= 3
    → short-return action: "depth_exceeded"
```

External calls by depth (Firecrawl scrapes \+ LLM extracts):

- **Depth 0:** two Firecrawl scrapes \+ two LLM extracts for the seed itself — the person page (P2) and `{imdb_url}/awards/` (P5) — plus up to five inline title enrichments via `sync_top_credits: 5`. Those titles still run as discovered depth-1 titles, so their own title-side fanout remains depth-bounded.
- **Depth 1:** queueing itself is free — stubs are created via `get-add` with no scrape. External calls fire only when the cron picks the row: the same full flow (page scrape \+ LLM extract, awards scrape \+ LLM extract) per entity.
- **Depth 2:** same per-entity scrape \+ extraction cost, but cascade helpers are skipped — no new queue rows, so external calls stop here (depth 3 is never queued).
- **Pacing:** no provider hard limit — the external call rate is bounded by the single-row cron tick (600 s per queue), and both queue crons ship `active = false`.

## Loop prevention (five layers)

1. **Dedup gate in `get-add`:** person side — `imdb_url` already in `imdb_person` with `enriched_at` within 30 days → return existing, no scrape. Title side — **any** existing enriched `film_tv` row dedups (sparse stubs fall through for queue repair / sync-enrich; `refresh: true` bypasses).
2. **Queue upsert:** duplicate `imdb_url` in queue → `count++`, tier upgrade. Single row per URL.
3. **Depth cap:** depth 2 scrape happens but no cascade. Depth 3 never queued.
4. **Tier cap at depth ≥ 1:** only T1–T2 discoveries are queued at depth 2 (T3–T4 discarded).
5. **Kill switch:** zero-API blocker flippable manually via `check-kill-switch-imdb`.

## Implementation notes captured during build

- **URL normalization lives in get-add** — all queue rows and `imdb_person` / `film_tv` rows are keyed on the normalized URL (the chain is specified on [IMDB Tables](/guides/enrichment/waterfall/imdb-tables#url-normalization)).
- **Xano function-call syntax:** use `function.run "path" { input = { ... } } as $result` — NOT `function.call` (not valid in Xano).
- **Queue upsert is the single source of truth:** `get-add` always routes through `upsert-imdb-person` / `upsert-imdb-title`. No raw `db.add` into queue tables anywhere.
- **Kill switch is fail-open:** missing env var → `try_catch` logs to `log_crash` and defaults OFF.
- **`cascade_depth` \+ `queue` param on get-add:** when `queue == false`, `get-add` invokes the orchestrator synchronously after the stub exists and passes through the caller's `cascade_depth`; no queue entry is written.
- **Orchestrators are per-phase `try_catch` → `log_crash`:** each phase emits one `log_crash` row with `qa_passed: true` (clean) or `qa_passed: false` \+ stack trace, using the IMDB `phase` enum (strict — listed on [IMDB Tables](/guides/enrichment/waterfall/imdb-tables#table-inventory)).
- **Phase 1 guard:** if scrape\+LLM fails or returns no `name` / `title`, stamp `enrich_success: false` and return early — no `master_person` gets created from a junk row.
- **Cascade helpers are pure dispatchers:** every helper guards `cascade_depth + 1 >= 3` and short-returns with `action: "depth_exceeded"`. Every `function.run` to `get-add` is wrapped in `try_catch` → `log_crash`.
- **Seed ID propagation:** a depth-0 seed invocation via `get-add` must set `seed_imdb_person_id` / `seed_film_tv_id` to the just-created stub's own id when no explicit seed is passed. Required for loop-prevention to work.
- **Top-N sync credits on person seeds (`sync_top_credits: 5`):** `cascade-credits-from-person` v1.3 sync-enriches the top 5 credits inline when the person is at `cascade_depth == 0`, while preserving the discovered title depth (`cascade_depth + 1`). Guarantees a seed returns with enriched films even before the title cron runs without reclassifying those titles as roots.
- **Cron single-row tick pattern:** modeled on `process-person-queue` **#126** / `process-company-queue` **#127**. Lock-on-pick, `try_catch` around orchestrator, release lock on failure. Priority pick order: `priority_tier` asc, `cascade_depth` asc, `id` asc.
- **Missing-stub guard:** if the queue row's `imdb_person_id` / `film_tv_id` is 0, write `log_crash` and `db.del` the queue row — prevents a bad row from monopolizing the lock forever.
- **Symmetric credit-edge writers:** #13013 (person side) and #13015 (title side) write graph edges only when both endpoint UUIDs exist, stamping `uuid` \+ a description at write time (shared helper #13181) and mirroring resolved edges into `imdb_credit_join`. #13013 also materializes pending `imdb_credit_join` rows for credit-derived `film_tv` stubs with `edge_created: false`, then flips them to true on a later resolved pass. No reconciler cron needed.
- **Award graph writes are deferred:** the award scrapers (#12887 / #12879) persist `film_tv_award` rows and log `award_graph_deferred`; `tool/backfill-award-edges-imdb` #13014 drives #2334 in bounded batches so high-award people can't time out on hundreds of inline graph writes.
- **Honorary awards have no `Film_TV` anchor:** `#2334 v1.5` handles `film_tv_id == null` by gating the `Film_TV` MERGE on `hasFilm` and swapping the `Film_TV_Award` MERGE key from `film_tv_imdb_url` → `person_uuid` so honoraries remain unique per-person. Since v1.7, title-level rows with no linked person also materialize their award node \+ `HAS_AWARD` edge.
- **`node_registry` registration:** #2320 and #2334 register stamped `Film_TV` / `Film_TV_Award` UUIDs in `node_registry` (2026-07-09) using linked table names — the live type column is spelled `entitiy_type`.

## Motivation

A single IMDB person profile exposes **credits** (titles), each of which exposes more people (directors, writers, cast). Award pages expose **`shared_with`** people, each of whom exposes more credits. Without a bounded cascade, a single seed scrape of a prolific person or title can fan out to thousands of LLM-extracted scrapes over hours.

The existing person \+ company waterfall solves this for LinkedIn/PDL/Enrich Layer (and now, in v4, for the post-PDL LLM stack). This waterfall applies the same pattern — **two queue tables, depth-capped cascade, priority-tiered processing, upsert dedup** — to IMDB.

**Goal:**

- From a seed **person**: enrich the person, pull ALL their awards, ALL related people on those awards, and their **primary credits** (director/writer roles and high-value producer credits). Write `WORKED_ON` edges for every credit whose film is already in the graph.
- From a seed **title**: enrich the title, pull ALL awards on it, ALL people on those awards, and its **key crew** (director, writer, cinematographer, lead producer) \+ top 3 cast. Write `WORKED_ON` edges for every cast/crew whose person is already in the graph.
