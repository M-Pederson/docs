## Foundation

<Card icon="gauge-high">
  ```text
  mb/rate-limit/acquire — 13079
  ```

  The global 1 req/s token gate, backed by the `mb_rate_limit` singleton (#736). If the last granted call was less than 1100 ms ago it sleeps 1 s, then stamps `last_call_at`. **Fail-open** — if the gate itself errors, the call proceeds and the failure is logged, so a broken limiter never halts the pipeline.
</Card>

<Card icon="cloud-arrow-up">
  ```text
  mb/api/request — 13080
  ```

  Single choke-point for **every** MusicBrainz call. Calls `acquire` for pacing, sends the canonical `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`), appends `fmt=json`, and retries up to 5 attempts with backoff sleeps between attempts (none after the final one). `400`/`404` are terminal — no retry. Returns `{ok, terminal, status, attempts, json}`; failures `log_crash` with phases `api_bad_request` / `api_exhausted`.
</Card>

## Queue upserts

<Card icon="wrench">
  ```text
  mb/queue/upsert-artist — 13081
  mb/queue/upsert-recording — 13082
  mb/queue/upsert-label — 13083
  mb/queue/upsert-event — 13084
  mb/queue/upsert-release-group — 13085
  mb/queue/upsert-work — 13086
  ```

  Single source of truth for queue writes, keyed on `mbid`. Duplicate discovery updates the existing row instead of inserting a duplicate: `count++`, **lowest tier wins**, then the row is promoted one step toward cron eligibility with `priority_tier = max(0, min(existing_tier, incoming_tier) - 1)`. The source-table FK is filled once the stub exists.

  Inputs default via `first_notnull` (tier 4 / depth 1) so an explicitly passed `0` survives — this is what lets bootstrap-lane rows (`priority_tier = 0`, `cascade_depth = 0`) be queued without the defaults coercing them.
</Card>

## Entry gates

<Card icon="play">
  ```text
  mb/get-add/label — 13087           → run-base-label-enrich (13091)
  mb/get-add/artist — 13088          → run-base-artist-enrich (13094)
  mb/get-add/release-group — 13089   → run-base-release-group-enrich (13099)
  mb/get-add/recording — 13093       → run-base-recording-enrich (13100)
  mb/get-add/work — 13098            → run-base-work-enrich (13101)
  mb/get-add/event — 13106           → run-base-event-enrich (13105)
  ```

  One gate per enrichable entity (the event gate landed 2026-06-12, completing the set). Each gate: 30-day `enriched_at` dedup → skip `gone`-tombstoned rows → ensure a stub row in the matching `mb_*` source table → seed propagation (when no caller seed is passed, the just-created stub is its own seed).

  Branching: **sync-enrich** (invoke the orchestrator directly) when `cascade_depth == 0 && queue == false`; otherwise upsert the queue row and let the drain handle it.
</Card>

## Node writers

<Card icon="database">
  ```text
  mb/node/add-music-label-node — 13090
  ```

  MERGEs the `Entity:Company:Music_Label` node on `mbid` with the full property set from the canonical [Music\_Label node spec](/guides/ontology/nodes#music-label), including **both** embedding vecf32s — `name_embedding` (OpenAI #4676) and `embeddings` (Gemini #13114) — built from `name + disambiguation` (v1.2, dual-write for the migration).
</Card>

<Card icon="database">
  ```text
  mb/node/add-music-artist-node — 13092
  ```

  Splits on MusicBrainz `type`: a `Person` artist becomes `Entity:Person:Music_Artist`; `Group` / `Orchestra` / `Choir` / etc. become `Entity:Music_Group`. Writes **both** `name_embedding` (OpenAI #4676) and `embeddings` (Gemini #13114) from `name + disambiguation` (v1.2). Stamps `artist_type` back onto the `mb_artist` row so the post-ingest reconciliation pass can select person-type artists without parsing `json`.
</Card>

## Orchestrators

Every phase is wrapped in `try_catch` → `log_crash` (phases `mb_lookup` / `mb_node` / `mb_edges` / `mb_browse` / `mb_cascade`). A terminal `404`/`400` marks the source row `gone`. All edges stamp `e.uuid` (plus the full [standard music edge envelope](/guides/ontology/edges#music-standard-edge-envelope)). Cascading is **inlined** in each orchestrator's `mb_cascade` phase — the standalone `cascade-from-*` helpers from the design were not built.

<Card icon="layer-group">
  ```text
  mb/run-base-label-enrich — 13091
  ```

  Lookup `label/{mbid}?inc=url-rels+label-rels+ratings+tags+genres` → `Company:Music_Label` node → `SUBSIDIARY_OF` (w15, stub-MERGEs the other label). Then browses `release?label=`, capped at **3 pages / 300 releases** (`inc=artist-credits+release-groups`), queueing release-groups T2, credited artists T2, and ownership labels T1.
</Card>

<Card icon="layer-group">
  ```text
  mb/run-base-artist-enrich — 13094
  ```

  Lookup `artist/{mbid}?inc=aliases+tags+genres+ratings+url-rels+artist-rels+event-rels` → artist/group node → `MEMBER_OF` edges (w5 current / w12 past, with `instruments`, `begin`/`end`/`ended`, `original`). Cascade: members T1, Albums capped at 20 chronological T2, recordings capped at 50 T2, and (v1.1, 2026-06-12) **main-performer events T3 — queued only when the artist itself is a depth-0 seed** (the events enter the queue as depth-1 children, so their own artist fanout stops at depth 2, per the depth model). Event discovery rides the same lookup: `inc=event-rels` embeds the full event objects, so no extra browse call is needed. Collaborators are still deferred to v2. **v1.3 (2026-06-14):** a depth-0 `Person:Music_Artist` is enqueued for [person-resolution](#person-resolution) via `mb/queue/upsert-person-resolution` (13122) — the same depth-0 gate as the event cascade, so only seed artists run it. **v1.4 (2026-06-15):** those same depth-0 Person rows now also get their serper avatar **inline at ingest** via `fetch-person-avatar` (13120, non-fatal) — previously the avatar only came from the person-resolution drain, which ships inactive, so directly-seeded artists came in avatar-less.
</Card>

<Card icon="layer-group">
  ```text
  mb/run-base-release-group-enrich — 13099
  ```

  Implements the [canonical-release rule](/guides/enrichment/waterfall/music-industry-waterfall#resolved-design-decisions-2026-06-11): picks `status = Official` \+ earliest date (fallback: earliest of any status), stamps `canonical_release_mbid` on `mb_release_group`, and creates the single `mb_release` row \+ `Entity:Release` node.

  Edges: `RELEASED` w10 (`joinphrase`), `RELEASE_OF` w30, `RELEASED_ON` w50 (`catalog_number`), `APPEARS_ON` w45 (`track_number`/`position`/`disc`). Cascade: tracklist recordings T3.
</Card>

<Card icon="layer-group">
  ```text
  mb/run-base-recording-enrich — 13100
  ```

  Edges: `PERFORMED` w10, `PLAYED_ON` w20 (`instruments`), `PRODUCED` w18, `ENGINEERED` w22 (`role`), `RECORDING_OF` w30 (`live`/`partial`/`cover`), and `RECORDED_AT` w35 — with `Place` nodes **materialized inline** from the embedded `place-rels` \+ `mb_place` rows, per [decision 6](/guides/enrichment/waterfall/music-industry-waterfall#resolved-design-decisions-2026-06-11). Cascade: work T1, credit artists T1, rel artists T2.

  **v1.2 (2026-06-14):** the rel-credit edges (`PLAYED_ON`/`PRODUCED`/`ENGINEERED`) are now **type-aware** — they capture the MB `artist.type` and split into Person (`Music_Artist`) vs Group (`Music_Group`) buckets, so a band credited as producer/engineer reuses its existing `Music_Group` node instead of MERGE-creating a duplicate `Person:Music_Artist`. (The artist-credit `PERFORMED` handler already split by type; the rel-credit handler didn't — caught on Nirvana, see the [sandbox test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run).)
</Card>

<Card icon="layer-group">
  ```text
  mb/run-base-work-enrich — 13101
  ```

  Creates `Entity:Work` with a sanitized work-type sub-label (`:Song` etc.). Edges: `COMPOSED` w8, `WROTE_LYRICS` w12, `WROTE` w10 (`role`). Cascade: composers T1, writers T2.
</Card>

<Card icon="layer-group">
  ```text
  mb/run-base-event-enrich — 13105
  ```

  **Built 2026-06-12 (second session)** — the formerly deferred event path. Lookup `event/{mbid}?inc=artist-rels+place-rels+url-rels+ratings+tags+genres` → `Entity:Event` node with a sanitized `Music_*` type sub-label (`:Music_Festival` / `:Music_Concert` / `:Music_Award_Ceremony` — non-alphanumerics collapse to `_`) and props `name`, `event_type`, `begin_date`/`end_date`/`ended`, `cancelled`, `time`, `setlist` (capped at 4,000 chars), rating, genres/tags.

  Edges: `HEADLINED` w35 from `main performer` rels; `PERFORMED_AT` w45 with `role` covering **both** `support act` and `guest performer` (a third rel type discovered in live API sampling); `HELD_AT` w30 with `Place` nodes **materialized inline** from `place-rels` \+ `mb_place` rows — the same pattern as the recording orchestrator, with null-safe handling for `place.type`, `coordinates`, and `address` (all observed null/empty in the wild). Cascade: headliners T1, support/guest artists T2, guarded `childDepth <= 2`.
</Card>

## Unified queue drain

<Card icon="wrench">
  ```text
  mb/queue/process-next — 13102
  ```

  Picks the **single best tier-0 row across all 6 queues** ordered by `cascade_depth ASC, entity_rank ASC, id ASC` — `entity_rank` is a constant per queue (artist 1, label 2, **event 3**, work 4, release\_group 5, recording 6) — skipping rows whose `next_attempt_at` is in the future. Nonzero tiers stay queued until duplicate discovery promotes them to `0`. Locks the row, dispatches the matching orchestrator, deletes on success. v1.1 (2026-06-12) wired `queue_mb_event` into the scan, the candidate ranking, and dispatch (→ 13105), filling the rank-3 slot reserved at v1. v1.2 (2026-07-01) made the drain tier-0-only.

  On failure: `retry_count++` with a `next_attempt_at` backoff of 5 min × retries; the row is dropped after 5 failures.
</Card>

## LLM edge descriptions

Mirrors the IMDB `WORKED_ON` trio (#13016–#13018), but **one generic trio covers all music edge types** — keyed on `e.uuid` rather than one trio per edge type. Model: **`deepseek/deepseek-v4-flash`** (temp 0.3, `reasoning:{enabled:false}` \+ `max_tokens:200`), producing a single factual 15–30-word sentence. All descriptions obey [**GOOD-or-null**](/guides/dev-updates/2026-06-16-music-node-descriptions#good-or-null-invariant) — a clean sentence or `null`, never a refusal/meta echo.

<Note>
  **Forward behavior as of 2026-07-01:** music edge descriptions now have two paths. The live enrichment functions call a bounded, mbid-scoped describer immediately after their `mb_edges` write phase, so normal new edges get descriptions in-band. Scheduled task #181 is pinned to `sandbox` and remains the sandbox async fallback drain for leftovers, retries, historical edges, and any edge whose inline LLM call failed or exceeded the bounded per-entity limit.
</Note>

<Card icon="cloud-arrow-up">
  ```text
  mb/edge/create-music-edge-description — 13095  (v1.4)
  ```

  Given one edge's structured properties (opaque `via_release` mbid stripped) \+ endpoint node names, calls **`deepseek-v4-flash`** (reasoning off \+ `max_tokens:200`) and returns the single-sentence description **after cleanup**: strip wrapping quotes \+ leading meta-preambles, normalize U\+2011/U\+2010 → `-`, and a GOOD-or-null reject guard (refusal/meta → empty). **v1.4 (2026-06-17)** swapped the model from `gpt-oss-120b`, which dropped spaces ~15% of the time ("Stevensis", "Themusicgroup"). Sandbox sample: _"Sufjan Stevens is a member of the music group Danielson."_
</Card>

<Card icon="wrench">
  ```text
  mb/edge/write-music-edge-description — 13096
  ```

  Stamps `e.description` \+ `e.description_generated_at` (ISO) onto the edge via Cypher, matched on `e.uuid`.
</Card>

<Card icon="wrench">
  ```text
  mb/edge/write-missing-music-edge-descriptions-for-mbid — 13158
  ```

  Bounded forward-path helper for live enriches. Given one MusicBrainz `mbid`, it selects music edges touching that endpoint where `e.uuid IS NOT NULL` and `e.description_generated_at IS NULL`, then calls `mb/edge/write-music-edge-description` (#13096) for each selected edge. It uses the same music edge-type allowlist as the async describer and clamps the per-call limit to 1–100.

  Current live callers are the music graph edge writers: label (#13091), artist (#13094), release-group (#13099), recording (#13100), work (#13101), event (#13105), `ON_RECORD_LABEL` derivation (#13131), and label-owner `OWNED_BY` (#13133). Each caller wraps the helper in its own non-fatal `try_catch`, so an OpenRouter or description-write failure never blocks node/edge creation.
</Card>

<Card icon="clock-rotate-left">
  ```text
  mb/tool/backfill-music-edge-descriptions — 13097  (legacy name; forward async describer)
  ```

  Async fallback describer (1–100 per run, default 10). Selects newly written edges where `description_generated_at IS NULL`, filtered by edge type (allowlist — `ON_RECORD_LABEL` added v1.3, **`OWNED_BY` added v1.5** by the [label-owner](/guides/enrichment/waterfall/music-reconciliation#build-spec--label-owner-resolution-owned_by) session) \+ `labels()` membership, and runs the create → write pair per edge. Idempotent via the timestamp guard. **v1.6 (2026-06-17) perf:** the SELECT now anchors on the relationship types (`MATCH (a)-[e:T1|T2|…]->(b)`) so FalkorDB scans only the music-edge matrices, not the entire people\+company edge set — cut a live batch SELECT from \>280 s (timeout) to ~3 s.
</Card>

## Cover art & group images

Built 2026-06-12 (second session). Two parallel image pipelines hang media off the source tables: **recording cover art** from the [Cover Art Archive](https://coverartarchive.org) and **group photos** from a serper.dev Google Images search. Both store the original URL _and_ a re-hosted GCS WebP URL — `cover_art_url` / `gcp_cover_art_url` on `mb_recording` (#738) and `music_group_image_url` / `gcp_music_group_image_url` on `mb_artist` (#737). In forward-path docs, these are single-entity fetch helpers called directly or by the finalizer; batch sweeps are excluded.

<Card icon="wrench">
  ```text
  mvp/music/save-cover-art — 13104
  mvp/music/save-music-group-image — 13107
  ```

  GCS converters (the IMDB poster-saver pattern, #13044). Each regex-extracts the **MusicBrainz MBID (UUID)** from its identifier input (`recording_identifier` / `music_group_identifier` — callers pass the bare `mbid`), converts the source image to WebP via `mvp/avatar/convert-image-url-to-webp` (12756), and uploads via `mvp/google-storage/upload-file` (12725) to `music-cover-art/<mbid>.webp` / `music-group-image/<mbid>.webp`. Returns the public GCS URL.
</Card>

<Card icon="cloud-arrow-up">
  ```text
  mvp/music/fetch-recording-cover-art — 13108
  ```

  One recording, end-to-end: skip if `gcp_cover_art_url` already set → **1 throttled MB lookup** `recording/{mbid}?inc=releases` → Cover Art Archive JSON (`coverartarchive.org/release/{id}`) for up to 5 releases, `Official` first → saves `cover_art_url` (500px thumbnail preferred, `https`-normalized) → 13104 → saves `gcp_cover_art_url`. **v1.3 (2026-06-14):** after stamping the GCS URL it writes `SET n.image_url` on the `Recording` graph node via `send-cypher` (non-fatal; the `already_done` skip path _also_ self-heals the node, so re-driving an imaged row catches it up). `no_art` → `SET n.image_url = null` on the node and the row is re-checked on later runs.
</Card>

<Card icon="cloud-arrow-up">
  ```text
  mvp/music/fetch-group-image — 13109
  ```

  One group, end-to-end (**groups only** — `artist_type = Person` rows are skipped): skip if `gcp_music_group_image_url` set → serper.dev images search (`q = "<name> band <disambiguation>"`, key from env `SERPER_DEV`, same call shape as `mvp/enrich/find-avatar` 12846) → saves `music_group_image_url` (first result) → 13107 → saves `gcp_music_group_image_url`. **v1.3 (2026-06-14):** writes `SET n.image_url` on the `Music_Group` graph node after the GCS stamp (non-fatal; self-heals on the `already_done` path); `no_image` → `SET n.image_url = null`. `serper_error` is transient and leaves the node untouched for retry.
</Card>

## Node descriptions

Built 2026-06-12, consolidated behind `finalize-music-node` on 2026-06-16. **`Recording`**, **`Music_Group`**, and **`Music_Artist`** nodes get generated descriptions, and their forward-path vectors are generated from that description. The other music node types (`Work`, `Release_Group`, `Release`, `Event`, `Place`, and `Company:Music_Label`) do **not** get generated descriptions today; they receive dual vectors from `name + disambiguation` instead. All generated descriptions obey the [**GOOD-or-null invariant**](/guides/dev-updates/2026-06-16-music-node-descriptions#good-or-null-invariant): a clean ≤127-char factual sentence beginning with the entity name, or exactly `null` — never an LLM refusal/meta-commentary echo.

<Card icon="cloud-arrow-up">
  ```text
  mvp/music/create-node-description — 13112  (v2.2)
  ```

  Generate (don't write) the LLM description for one node from a **slim, type-aware field-set** (not raw JSON — raw-JSON@8000 was what made deepseek echo _"please provide the record"_). One OpenRouter call — **`deepseek/deepseek-v4-flash`**, temp 0.2, **`reasoning: {enabled: false}` \+ `max_tokens: 200`** — producing **one factual sentence ≤127 chars** that **begins with the entity name**, so the description-based embedding keeps the name signal. The field-set is passed as a JSON _user_ message built in a lambda (not `util.template_engine` — avoids the markdown-autolinker URL burn from the IMDB build). **Two GOOD-or-null guards (v2.2):** an **input-empty short-circuit** returns `null` before the LLM is called on a missing/empty record, and an **output reject-guard** strips quotes, hard-truncates to 127, and maps any refusal/meta-commentary (first-person \+ `the user`/`provide`/`structured`/`musicbrainz` markers) to `null`. The guard lives in the lambdas, **not** the prompt.

  <Warning>
    The reasoning flag is load-bearing: deepseek-v4-flash is a reasoning model, and without it the completion budget goes to thinking tokens and `message.content` comes back **empty** on larger records — found live during the Cinq smoke test (3 of 4 rows silently skipped) and fixed the same day. CLAUDE.md trap #13.
  </Warning>
</Card>

<Card icon="wrench">
  ```text
  mvp/music/finalize-music-node — 13117
  ```

  Forward finalizer for one node. It validates or generates the node `description` for `recording`, `music_group`, and `music_artist`, rejects bad LLM output as null, writes both vectors (`name_embedding` via #4676 and `embeddings` via #13114), and runs the relevant media side effect where applicable (recording cover art / group image / person avatar path). For `other` node kinds it skips LLM description and writes vectors from `name + disambiguation`.
</Card>

## Node embeddings

Forward writers/finalizers dual-write Gemini `embeddings` and OpenAI `name_embedding` for the node types they touch. The model provenance is fixed: `name_embedding` uses **OpenAI `openai/text-embedding-3-small`** via `vectors/create-vectors-string` #4676; `embeddings` uses **Google `google/gemini-embedding-001`** via `vectors/create-gemini-vectors` #13114.

<Warning>
  **Forward-only vector coverage fixed 2026-07-01:** `Event` and inline `Place` now call `finalize-music-node` #13117 with `kind=other` where they are created. `Work`, `Release_Group`, and canonical `Release` already used that same path.
</Warning>

## ON\_RECORD\_LABEL edges

Built 2026-06-17. A **graph-derived forward edge writer** materializes the [`ON_RECORD_LABEL`](/guides/ontology/edges#music-structural-labels) edge (`(Recording)→(Company:Music_Label)`, weight **55**) — a one-hop convenience shortcut of the existing `(Recording)-[:APPEARS_ON]->(Release)-[:RELEASED_ON]->(Company:Music_Label)` path. **No MusicBrainz fetch** — the recording lookup (#13100) never pulls label/release data, so the edge is derived purely in-graph from the canonical release's label (only canonical releases materialize as `Release` nodes, so the path _is_ the canonical-release label).

**Forward-wired (2026-06-17).** Despite the legacy helper name, the release-group orchestrator `mb/run-base-release-group-enrich` (**#13099 v1.2**) calls #13131 with the new `release_mbid` scope right after writing the canonical Release \+ `RELEASED_ON` \+ `APPEARS_ON` edges — guarded on a non-empty canonical release mbid, in its own non-fatal `try_catch` — so every newly-enriched release-group **auto-derives** `ON_RECORD_LABEL` for its tracks.

<Card icon="wrench">
  ```text
  mvp/music/backfill-on-record-label-edges — 13131  (legacy name; release-scoped forward writer)
  ```

  Single idempotent Cypher via `mvp/falkor/send-cypher`: `MATCH (rec:Recording)-[:APPEARS_ON]->(rel:Release)-[ro:RELEASED_ON]->(lbl:Company:Music_Label) MERGE (rec)-[r:ON_RECORD_LABEL]->(lbl)` — `ON CREATE SET` stamps `uuid`, `created_at`, and semantic-null `description` / `description_generated_at`; every run sets `weight = 55`, `updated_at`, `via_release = rel.mbid`, and `catalog_number = ro.catalog_number` (carried from the canonical release's `RELEASED_ON`). `MERGE` per `(recording, label)` pair → idempotent, no duplicates. Forward scope is **`release_mbid`**, used by #13099 immediately after canonical release materialization. The async edge describer fills `description`.
</Card>

## Person-resolution

Built 2026-06-14. The person-side mirror of the label→Company bridge — resolves a `Person:Music_Artist` node to a `master_person` and unlocks the social/people enrichment that MB artists otherwise miss (they carry no `linkedin_url`). A depth-0 `Person:Music_Artist` is enqueued at enrich time; a **standalone cron** drains the queue and, per artist, runs Exa social discovery cross-checked against the MB url-rels \+ aliases, mints/links a `master_person` carrying **all** verified socials as `master_link` rows (which auto-fire the twitter/instagram/youtube/linkedin scrapers \+ `social_insights` \+ influence), `SAME_AS`-bridges `Music_Artist → master_person`, stamps `mb_artist.master_person_id`, annotates the node `social_links`, and sets a serper.dev avatar. Full writeup: [dev update](/guides/dev-updates/2026-06-14-person-resolution). It builds the person half of [Music Reconciliation](/guides/enrichment/waterfall/music-reconciliation) **as a bridge, not a merge**.

<Card icon="cloud-arrow-up">
  ```text
  mvp/music/find-person-social — 13118
  ```

  Exa social discovery for one person — the mirror of `find-label-domain`. Searches Exa, then classifies the results into `linkedin[]`, `socials[]`, and `others[]` for the orchestrator's cross-check step.
</Card>

<Card icon="wrench">
  ```text
  mvp/music/save-person-avatar — 13119
  ```

  GCS converter (the avatar-saver pattern): converts the source image to WebP and uploads it to `music-person-avatar/<mbid>.webp`, returning the public GCS URL.
</Card>

<Card icon="cloud-arrow-up">
  ```text
  mvp/music/fetch-person-avatar — 13120
  ```

  One person, end-to-end — the mirror of `fetch-group-image` (13109) with the gate **inverted to persons** (`artist_type = Person` only). serper.dev images search → saves the avatar → `SET n.image_url` on the `Person:Music_Artist` graph node.
</Card>

<Card icon="play">
  ```text
  mb/run-person-resolution — 13121
  ```

  The orchestrator. Builds a verification set from `mb_artist.json` (known social URLs from `relations[]`, aliases / legal names from `aliases[]`, area / birth-year / genres) → calls Exa (13118) → **cross-checks** candidates against the MB set to defeat same-name false positives → mints/links a `master_person` via `mvp/get-add/master-person-new` (#13039) passing **all verified socials in `links`** (each written as a `master_link` row, auto-firing the platform scrapers) → stamps `mb_artist.master_person_id` → writes the `SAME_AS` bridge `Music_Artist -w1-> master_person Person` → annotates the node (`SET n.social_links`, `n.social_resolved_at`) \+ fetches a serper avatar (13120). `hold_downstream = false` (run the full PDL→EnrichLayer people pipeline) **only when a LinkedIn is verified**; `true` otherwise — the social scrapers still fire from the link writes.

  A LinkedIn is accepted **only by content** (its profile text shows a music-identity signal — a music-role keyword, the artist's stage name, or a genre), which defeats same-name strangers (v2.2). **v2.3 (2026-06-14):** a `deceased` flag derived from the MB `life-span` (`ended == true` or an `end` date) **skips the LinkedIn-accept branch entirely** for dead artists — a deceased musician cannot run a LinkedIn, so any hit is an impersonator / fan / namesake. Caught live on Kurt Cobain (died 1994) during the [sandbox node-type test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run); socials still flow, only LinkedIn is blocked. **v2.4 (2026-06-14):** if the url-rels include an `imdb.com/name/nm…` page, a dedicated `try_catch` phase calls `mvp/imdb/add-imdb-to-master-person` (#10509) **synchronously** (inherits the request data source — avoids the async→live leak) to enrich the **existing** master\_person with IMDb bio/credits/awards (no duplicate; adds a Firecrawl \+ LLM of latency per IMDb-carrying artist). Verified on Cobain — `imdb_person` landed on sandbox, FK-linked, no dup, and the master\_person's graph `Person` node gained the IMDb `roles` / `short_biography` / `long_biography` / `avatar` (credits queue as `imdb_title` rows). See the [handoff](/guides/dev-updates/2026-06-14-person-resolution).
</Card>

## Scheduled tasks

<Card icon="clock-rotate-left">
  ```text
  process-queue-musicbrainz — Task 180   (INACTIVE)
  ```

  The unified cron. Runs every **60 s** and drains up to **10 rows per tick** via `mb/queue/process-next` (13102). This deviates from the design's ~2 s cadence — Xano cron granularity is 60 s, so batch-10-per-tick gives roughly the same throughput. **Pausing this task is the pipeline's manual kill.**

  <Note>
    Task 180 is intentionally `priority_tier = 0` only. Lower numbers are more urgent, but `0` is the only cron-eligible tier. The six MusicBrainz queue upserts are `mbid`-unique, so repeated discoveries do not create duplicate queue rows; they increment `count` and promote the existing row one step toward `0` using `max(0, min(existing_tier, incoming_tier) - 1)`. Because `tool/drain-queue` also calls `mb/queue/process-next` (13102), manual drains follow the same tier-0 gate.
  </Note>
</Card>

<Card icon="clock-rotate-left">
  ```text
  backfill-music-edge-descriptions — Task 181   (ACTIVE — legacy name)
  ```

  Async safety-net task for music edge descriptions. Runs every **300 s** on the **`sandbox` datasource**, processing up to **20** undescribed music edges per tick through `mb/tool/backfill-music-edge-descriptions` (#13097), which uses the same `deepseek/deepseek-v4-flash` create/write path (#13095 → #13096) as the live enrichers.

  This task is **not the primary forward path** after the 2026-07-01 update and no longer drains live leftovers. The primary path is now in-band: each music edge-writing orchestrator calls #13158 immediately after its edge phase. Task 181 catches sandbox edges still missing `description_generated_at`: historical rows, inline LLM failures, bounded-limit overflow, and manual graph writes. It is self-maintaining: once no pending music edges exist, each tick is a cheap no-op.
</Card>

<Card icon="clock-rotate-left">
  ```text
  process-queue-person-resolution — Task 182   (INACTIVE)
  ```

  The [person-resolution](#person-resolution) drain — a **standalone cron** (decoupled from the MB queue drain so Exa/serper pacing is independent of MB's 1 req/s budget). Drains **5 rows per tick** via `mb/queue/process-next-person-resolution` (13123). Shipped `active = false`; pausing it is the kill switch.
</Card>

## Deviations & deferrals

Differences between the [design](/guides/enrichment/waterfall/music-industry-waterfall) and what was built:

| Deviation / deferral | Detail |
| --- | --- |
| **~~Events deferred~~Events built 2026-06-12** | `mb/get-add/event` (13106) \+ `mb/run-base-event-enrich` (13105) now exist and the drain (13102 v1.1) fills the reserved `entity_rank = 3` slot. Event-specific deviations: events are discovered **only from depth-0 artists** (`inc=event-rels`, main-performer rels, T3 — no event browse pagination); `guest performer` (a third live rel type) maps to `PERFORMED_AT` with `role`; event `genres`/`tags`/`rating` are written but empty in practice on MusicBrainz. Smoke-tested on Reading Festival 1992 (all four phases green: node \+ sub-label, `HEADLINED`, `HELD_AT` place inline, headliner cascade). |
| **Collaborators deferred to v2** | The artist orchestrator (13094) does not queue collaborators — members, albums, recordings, and (v1.1) depth-0 main-performer events only. |
| **Backfill-only utilities excluded** | Batch image sweeps, node/vector catch-up sweeps, area sweeps, and temporary Task 184 are intentionally out of this forward-path reference. |
| **Description finalizer rewrites vectors for three node types** | #13117 generates descriptions and description-based OpenAI/Gemini vectors for `Recording`, `Music_Group`, and `Music_Artist`. Other music nodes keep null descriptions and vectorize `name + disambiguation` through their forward writer/finalizer path. LLM-empty rows stay unstamped for a future finalization attempt. |
| **Cascade helpers inlined** | The standalone `mb/cascade-from-*` functions were not built; each orchestrator carries its own `mb_cascade` phase. |
| **`FROM_AREA` forward gap** | Geography edges are not forward-wired in the audited ingest path while sweeps are excluded. If `FROM_AREA` remains desired, wire country/city/region mapping directly into artist/label/place writes. |
| **Genre nodes retired** | Genres remain `genres` properties on music nodes for facets, badges, filters, and search text. Do not write `Entity:Genre` nodes or `HAS_GENRE` edges. |
| **Cron 60 s × 10 batch** | Design called for a ~2 s tick; Xano cron granularity is 60 s, so Task 180 drains up to 10 rows per tick instead — approximately equivalent throughput. |
| **Recordings browse unsorted** | The artist cascade takes the **first 50** recordings as returned, not the top 50 by length/date. |
| **Label browse cap** | `release?label=` browse is capped at 3 pages / 300 releases; hitting the cap is logged. |
