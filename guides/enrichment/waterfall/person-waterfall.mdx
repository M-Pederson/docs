Get-add-driven person enrichment with a **tiered provider cascade** (PDL → Enrich Layer → Exa, plus an async FullEnrich tier and the local Fundable mirror), a **contained, non-recursive cascade**, a single priority-tiered queue, and upsert dedup. This page is the architecture spec: the source contract, depth model, tiers, queue semantics, the three-stage entry gate, the phase orchestrator, the containment model, seed flows, and a full cascade example. The entry-gate walkthrough lives on [Person Waterfall Entry Points](/guides/enrichment/waterfall/person-entry-points); the live function inventory, table schemas, and graph node \+ edge references live on the companion pages below.

The person waterfall is the **oldest and richest** enrichment pipeline — everything the [Company](/guides/enrichment/waterfall/company-waterfall), [IMDB](/guides/enrichment/waterfall/imdb-waterfall), and [Music](/guides/enrichment/waterfall/music-waterfall) waterfalls share was extracted from it. Like the company waterfall, everything enters through a synchronous **get-add entry gate** (`mvp/get-add/master-person-new` #13039) that dedups, sources, and creates inline — seeds dispatch the orchestrator immediately (async), while the queue drains leaves discovered at depth \> 0. It differs structurally in three ways: the provider layer is a **fallback ladder** (each tier fires only when the previous one came back thin), one provider tier is **asynchronous** (FullEnrich returns via webhook after the gate has exited), and since the 2026-06-23 containment sweep person runs are **company-write-free** — every company a person run discovers is linked lookup-/local-only, never enqueued. See [Core Concepts](/guides/enrichment/waterfall/core-concepts) for shared mechanics (cascade depth, priority tiers, queue tables).

## Source contract — PDL → Enrich Layer → Exa (\+ FullEnrich \+ Fundable)

Unlike the company waterfall's parallel provider fan-in, the person providers form a **fallback ladder**: PDL is the cheapest first call, Enrich Layer fires only when PDL comes back thin (404 **or** 200 with empty `experience` — the cascade's v1.4 treats both as thin), Exa fires only when PDL is still thin after the EL-derived retry, and FullEnrich is the asynchronous last resort for email-only seeds. The natural keys are **profile URLs** in `master_link` (normalized), **emails**, name-guarded **phones**, and the direct `master_person.linkedin_url` fallback — walked in that order by `match-master-person` #13038. There is no unique-index backstop — dedup is the app-level ladder (pre-API local dedup \+ the 4-tier match).

| Provider | Function | What it does | Gate / debounce | Retry / terminal |
| --- | --- | --- | --- | --- |
| People Data Labs | Tier 1 in `cascade-person-data` #13037 (v1.4); Phase 0 backfill #12857 (v1.0); Phase 3 apply #12824 (v3.4) | `GET /v5/person/enrich` with email \+ name \+ LinkedIn — profile, emails, phones, social profiles, work history, education | Gate cascade runs only at depth 0, after the pre-API dedup misses; Phase 0 re-fetches when the `people_data_labs` stash is empty | 404 → EL fallback; a 200 with empty `experience` is **treated as thin** (v1.4) and continues; the retry with the EL-derived LinkedIn URL is promoted only when `status == 200 && likelihood > 5` |
| Enrich Layer | Tier 2 — `person-email` #12734 / `person-linkedin` #12612; Phase 1 #12822 (v1.5); applied by `process-enrich-layer` #12615 (v1.5) | `/v2/profile/resolve/email?lookup_depth=superficial&enrich_profile=enrich` (email path) or `/v2/profile` (LinkedIn path) — name, avatar, headline, positions, education, certs, volunteering, projects, publications, honors | Gate: thin PDL. ## Depth model

The person cascade is non-recursive by construction — depth only ever increases, and fanout privileges shrink with it.

| Depth | Meaning | Behavior |
| :-: | --- | --- |
| **0** | Seed (user / smoke / direct) | Full pipeline: the synchronous PDL → EL → Exa cascade \+ FullEnrich at the gate, then all 13 phases with seed privileges — LLM bios, optional deep research, six-resolver edge resolution, expertise, investor overlay. Deal-network fanout **additionally requires `allow_investor_network_cascade: true`** |
| **1** | Direct discovery (opt-in-cascade people; Exa C-suite direct dispatches) | The gate **skips the external cascade** — the person is created from passed data \+ Fundable and queued (or, for C-suite, direct-dispatched in `current_company_only` mode). A drained depth-1 run still fully enriches the person (Phase 0 PDL backfill \+ Phase 1 EL), but every company it touches resolves lookup-/local-only and its own phase-9 fanout is blocked (depth ≠ 0) |
| **2\+** | Not produced by live paths | Person runs enqueue **no companies at any depth**, and people are enqueued only from the seed-level opt-in investor cascade — no live person path writes a depth-2 queue row |

<Check>
  **Why non-recursive?** One seed person's PDL work history alone exposes 5–10 employers and schools; the pre-sweep policy queued every one of them, and each drained company could surface founders / C-suite whose own histories exposed more companies. An angel seed adds an investment network — portfolio companies, co-angels, VC firms, partners — where every co-angel is another angel with another network behind them. Instead: companies are linked local-only and never queued from person runs, C-suite people are current-company-scoped with broad edge resolution skipped, person-to-person fanout requires an explicit seed-level opt-in, and the drain cron ships inactive.
</Check>

## Priority tiers

### queue\_enrich\_person (1 = highest)

`queue_enrich_person` (table #582) uses the shared tier vocabulary — `1` founders / current-network people · `2` past-network / partner-adjacent · `3` schools / angels · `4` everything else, **4 = default**. Live writers:

| Tier | Who queues it | Live status |
| :-: | --- | --- |
| **1** | Co-angels \+ VC partners from `resolve-investors-edges` #12702 (depth 1) — only when the seed run opted into the investor cascade | live — repaired v1.12 \+ v1.13, 2026-07-02 |
| **1** | YC founders from the company side (`process-yc-people` #12700, seed-company only) | live |
| **1** | Operator / backfill seeds at depth 0 — empty-PDL backfill (Task #156), divergent-LP seeding (Task #179) | tasks ship inactive |
| **2** | YC partner (company side); Fundable smoke seeds (sandbox Task #185, via #13039 at depth 0 / tier 2) | live / sandbox |
| — | Exa C-suite / founder people | **NOT queued** — direct async dispatch of #13040 with `current_company_only: true` (no queue row) |
| **4** | Any gate queue path without an explicit tier | default |

### Cascade filter — seed person

```text
seed person (depth 0)
  companies (employers, schools, cert issuers, volunteer orgs,
  publishers, portfolio companies)
      → resolve lookup-/local-only, hold_downstream — NEVER queued
  co-angels / VC partners          → queue_enrich_person(depth 1, tier 1)
  VC firms                         → queue_enrich_company(depth 1, tier 2)
      → all three ONLY when allow_investor_network_cascade == true AND depth == 0
  Exa C-suite people (company-seed lane)
      → direct async dispatch {depth effective+1, current_company_only: true}
        — never queued
```

### Upsert on duplicate

When the same `master_person_id` is enqueued twice, `upsert-enrich-person` #12854 (v1.1, 2026-06-03 — the **single sanctioned writer** for all `queue_enrich_person` rows; the already-queued branch lives inside it):

- `count++` on every duplicate request (a request tally, not an attempt counter)
- `priority_tier` — only ever improves (min of existing and incoming)
- `cascade_depth` — only ever improves (min); a later seed-level request promotes a leaf row
- `deep_research` — **OR-promoted**: once any request asks for deep research, the row keeps it
- One row per person enforced in application code only — there is **no unique index** on `queue_enrich_person.master_person_id`

## Queue & upsert semantics

- **Natural key is `master_person_id`** — the person row always exists _before_ it is queued (the entry gate creates it first).
- **Sync-enrich branch:** `cascade_depth == 0 && queue == false && !hold_downstream` → dispatch #13040 async-shared immediately, no queue row. (The `debug.stop` that used to halt this dispatch was removed in #13039 v1.8.)
- **Forced queue at depth \> 0:** `queue || cascade_depth > 0` always routes to the queue upsert — a depth-1 discovery can never trigger inline full enrichment, and the gate skips the paid provider cascade for it entirely.
- **Governor flags on the entry gate:** `hold_downstream` (no dispatch, no queue row, `qa_passed` skip logged — the interactive-QA mode), `orbiter_verified` (marks user-supplied input emails only), and the v1.10 `suppress_downstream` pass-through to `create-master-link` on all contained paths.
- **Drain layer:** Task #126 (cron, **`active = false` live**) picks **one** row per 600 s tick — `priority_tier asc, cascade_depth asc, id asc` — locks it, runs #13040 with the stored `master_person_id` / `deep_research` / `cascade_depth`, deletes on success, releases \+ `log_crash` on failure. Its catch block's out-of-enum `phase: "queue"` write was fixed on 2026-07-02 (same day as company Task #127) — pre-fix, a queue-item crash made the crash logger itself throw. Operator batch drain: `process-enrichment-queue` #12816 (v1.1 — routes through the gated orchestrators so depth gates apply), or Run Now on #126. Cron detail: [Person Waterfall Crons](/guides/enrichment/waterfall/person-crons).
- **Index gaps (app-level integrity):** `queue_enrich_person` has no unique key, `master_person` has no `linkedin_url` / `node_uuid` index, and `person_enrich_data` has no `master_person_id` index — dedup and idempotency live in #13038 / #12854, not the DB. Field-by-field schemas: [Person Waterfall Tables](/guides/enrichment/waterfall/person-tables).

## Pipeline architecture — the three-stage entry gate

```text
mvp/get-add/master-person-new — #13039
```

The entry gate splits the retired monolith into three composable stages, then dispatches the orchestrator:

```text
1 · cascade-person-data #13037      PDL → Enrich Layer → PDL retry → Exa
2 · match-master-person #13038      4-tier id resolution + phone format
3 · master-person-new #13039 v1.13  kill switch · pre-API local dedup · Fundable sync ·
                                    contact write loops · add-person-node · dispatch
      → enrich-master-person #13040 v3.23 (async-shared orchestrator)
```

| Stage | Function | Responsibility |
| :-: | --- | --- |
| 1 | `mvp/enrich/cascade-person-data` #13037 (v1.4) | Pure data fetch, in order: PDL → Enrich Layer (email #12734 / LinkedIn #12612) → PDL retry via the EL-derived LinkedIn URL → Exa (#12936). Returns the payloads plus deduped `el_links`, `input_email_source`, and a `needs_full_enrich` flag when sources are thin. No DB writes except `log_crash` on a caught exception. |
| 2 | `mvp/resolve/match-master-person` #13038 (v1.0) | Resolves the `master_person` id: `master_link.link_url` → `master_email.email_address` → name-guarded `master_phone.phone_number` (with phone-conflict detection) → direct `master_person.linkedin_url` fallback. Formats every phone via `llm-phone-format` #2540 and returns `match_via` \+ the formatted list for caller reuse. No graph writes. |
| 3 | `mvp/get-add/master-person-new` #13039 (v1.13) | Kill switch #4617 → **pre-API local dedup** (v1.6: links / emails / `linkedin_url` — skips the paid cascade on repeat lookups) → cascade → match → Fundable lookup from the enriched link set → contact write loops (#2784 / #2541 / #2783) → Fundable direct-field sync (v1.9) \+ display-name repair (v1.12) → `add-person-node` #2601 (graph `Person` node) → downstream dispatch. When `needs_full_enrich`, also fires FullEnrich #13036 async. |

**Downstream dispatch.** When the person is new and not queued (`cascade_depth == 0 && queue == false && !hold_downstream`), #13039 fires **`enrich-master-person` #13040** in **async-shared** mode (`master_person_id`, `deep_research`, `cascade_depth`). Otherwise it routes through `upsert-enrich-person` #12854. `hold_downstream` skips both paths and logs a `qa_passed` skip row — and since **v1.10 (2026-06-23)** it also passes `suppress_downstream: true` on every `create-master-link` call (as does any `cascade_depth > 0` request). The temporary `debug.stop` in front of this dispatch was removed in v1.8 (2026-06-01) — the old walkthrough's claim that it still halts the function is stale.

Full calling contract, section-by-section walkthrough, and the v1.0 → v1.13 version ladder: [Person Waterfall Entry Points](/guides/enrichment/waterfall/person-entry-points). The public HTTP entry is **`POST /master-persons`** — endpoint #7977 (V2 Master Persons, group 350, `api:WKgay2AU`) → #13039.

## enrich-master-person — the 13-phase orchestrator

```text
mvp/enrich/enrich-master-person — #13040
```

**Current version: v3.23 (2026-06-30)** — the only person orchestrator (its predecessor #12554 is deleted; see [Historical reference](#historical-reference)). Dispatched by #13039 (seed gate), `process-full-enrich` #13049 (`force_run` re-run), `enrich-master-company` #12992 Step 7 (the C-suite lane), and Task #126 (queue drain). Every phase runs in its own `try_catch` writing per-phase `qa_passed` / `CRASH: phase-N` rows to `log_crash` #542 via the strict `phase` enum — failures are logged but don't block later phases ([Core Concepts → Per-Phase Crash Logging](/guides/enrichment/waterfall/core-concepts#per-phase-crash-logging)).

A completion marker flips `enrich_history_person.processing = false` regardless of crash state (v3.8) — the FullEnrich webhook polls this flag.

## The containment model

**The 2026-06-23 sweep is the defining fact of the current pipeline.** Before it, a person run queued every company it discovered (current employer at tier 1, past employers at tier 2, schools at tier 3, issuers / publishers at tier 4) and an angel seed fanned out its whole deal network on depth alone. After it, person runs write **zero `queue_enrich_company` rows** — companies are resolved lookup-/local-only everywhere — and the only queue writes a person run can make come from the **opt-in** investor cascade. Four mechanisms enforce this.

### No company enqueueing from person runs (the 2026-06-23 sweep)

Every company-resolving site in the person pipeline was converted on 2026-06-23:

| Site | Function | Containment behavior |
| --- | --- | --- |
| Phase 3 — primary employer | #12824 **v3.4** | Resolves the best current company **local-only with `hold_downstream: true`**; links `master_person.master_company_id` \+ `current_title`; never enqueues |
| Phase 4b — Exa work history | #13031 **v1.5** | Work-history companies may resolve / create local `master_company` rows but are never enqueued |
| Phase 7 — education | #12560 **v1.2** | Schools resolve / create local rows only (already depth-1 leaves since v1.1) |
| Phase 7 — work | #12562 **v1.7** | No enqueueing; v1.5/v1.6 also delegate current-role choice to `choose-best-current-role` #12621 before edge creation and skip synchronous edge descriptions |
| Phase 7 — projects & publications | #12579 **v5** | Publishers are leaf / local-only (`queue: false` \+ `hold_downstream: true`); contributor resolution is seed-person-only (v4) |
| Phase 7 — honors | #4574 **v4** | Issuer queries scoped per-person; issuers are depth-1 leaves |

Additionally, `master-company-new` (2026-06-25) makes **all `resolve-edges-work` leaves lookup-only** (`allow_create: false`) — phase-7 work edges link existing companies but cannot mint new ones. The net contract: a person run can **link** companies, but only the [company waterfall](/guides/enrichment/waterfall/company-waterfall) enriches them.

### Company-seed C-suite mode (current\_company\_only)

The reverse lane — a **company** seed creating people — is equally contained. When `enrich-master-company` #12992 (v4.17) Step 7 accepts Exa C-suite / founder matches for a seed company (**depth-0-gated** since v4.2; deeper companies stash the Exa envelope but never process people, v4.3), each match goes through `master-person-from-exa` #12997 (v1.9): normalized-LinkedIn dedup, kill-switch-gated create, `is_c_suite: true`, new leaves start `visibility: false` (v1.7), employer resolution **lookup-only / `allow_create: false`** (v1.8), a guaranteed `person_enrich_data` shell with the Exa payload stashed (v1.9). Then — instead of a queue row — a direct async dispatch of #13040:

```json
{
  "cascade_depth": 1,
  "deep_research": false,
  "current_company_only": true
}
```

(depth is `effective + 1` — 1 from a seed company.) **#12997 writes no work rows and no edges** — `work_experience` rows materialize inside the dispatched run (Phase 3 from PDL, Phase 4b from the stashed Exa), and edges come from the standard `create-work-edges` #2800 `WORKS_AT` family. There is **no `HAS_WORKED_AT` edge type** — the old page's C-suite passage described a ghost writer; legacy `WORKED_AT` edges linger in the graph from a pre-#2800 writer, but the live writer emits `WORKS_AT` (plus FOUNDED / BOARD\_MEMBER\_OF / ADVISOR\_TO / INVESTED\_IN / MEMBER\_OF) only — see [Person Edges](/guides/enrichment/waterfall/person-edges).

In-mode, the run is full for the person but narrow on expansion: PDL (v3.3), Enrich Layer (#12615 v1.2), Exa (v1.4), and FullEnrich (#13049 v1.9) all filter employment to **current roles**; Phase 1 skips the paid EL lookup for card-ready Exa leaves; and **Phase 7 is skipped entirely** (#12828 v1.2, 2026-06-25 — phase 3 already resolved the primary employer). Company-born people cannot spawn a new graph wave.

### Opt-in investor cascade (allow\_investor\_network\_cascade)

Phase 9 (#12830 **v3.3**, 2026-06-23) makes deal-network fanout **explicit opt-in**: it requires `allow_investor_network_cascade: true` **AND** `cascade_depth == 0` (v3.2) — depth 0 alone no longer fans out, which is a behavior change from the old "seed-only" policy. The orchestrator invokes the phase **twice**: an early pass right after Fundable linking (`build_investment_thesis: false` — investor overlay so the bio phase has material) and a late pass after expertise (`build_investment_thesis: true` — thesis rebuild, and the fanout when opted in).

When opted in, each `fundable_angel_investments` row routes through `cascade-deal-participants` #12856 (v1.6, 2026-07-02): the portfolio company via get-add **`hold_downstream`** (linked, NOT queued), the `Funding_Round` node \+ `RAISED` edge (#12701), the IPO signal, and `resolve-investors-edges` #12702 (v1.13, 2026-07-02) for the full investor edge set — queueing **co-angels and VC partners at depth 1 / tier 1** (people) and **VC firms at depth 1 / tier 2** (companies). These opt-in cascade rows are the **only queue writes a person run makes**. Both #12856 and #12702 carry the 2026-07-02 depth-guard repair — detail in the [implementation notes](#implementation-notes-captured-during-build).

### Scraper fan-out suppression (suppress\_downstream)

`create-master-link` #2541 (**v1.3**, 2026-06-22) adds a `suppress_downstream` input guarding **all 9 async scraper dispatch sites** (LinkedIn, X, Crunchbase, YouTube, Facebook, IMDB, …), and #13039 **v1.10** (2026-06-23) passes `suppress_downstream: true` whenever `hold_downstream` is set **or** `cascade_depth > 0` — contained paths can store scraper-discovered profile links without recursive enrichment chains. The old caveat that "table-level scrapers always fire even under `hold_downstream`" is obsolete. Complementing it, `scrape-twitter-person` #2106 (v1.10, 2026-06-30) makes Twitter-expanded companies leaf / local-only.

### Queued depth-1 processing

A depth-1 discovery (a co-angel or VC partner from the opt-in cascade) routes through `upsert-enrich-person` #12854 — one row per person, count\+\+ / tier / depth promotion, `deep_research` OR-promotion. **External calls by depth:**

- **Depth 0 (gate):** PDL (up to two calls with the EL-derived retry), Enrich Layer (one path), Exa (only if still thin) — the ladder stops at the first rich tier — plus the LLM helpers and the optional async FullEnrich dispatch.
- **Depth 0 (orchestrator):** Phase 1 EL re-check (30-day dedup), LLM bios, expertise, optional deep research (whose webhook fan-out later adds social insights) — each gated and per-phase fault-isolated.
- **Depth 1 (drained):** Phase 0 PDL backfill \+ Phase 1 EL fire the paid calls the gate skipped at creation — subject to the v1.4/v1.5 cost-skips.
- **Companies, at any depth:** zero external calls from the person run — the company waterfall pays for company enrichment on its own budget.

## Loop prevention (ten layers)

1. **Pre-API local dedup \+ the 4-tier match ladder** (#13039 v1.6 / #13038) — an existing person is returned and re-attached, never re-created.
2. **Queue upsert** #12854 — one row per person (app-level), `count++`, min tier, min depth, OR-promoted `deep_research`.
3. **The 2026-06-23 sweep** — zero `queue_enrich_company` writes: all six company-resolving sites (#12824 v3.4, #13031 v1.5, #12560 v1.2, #12562 v1.7, #12579 v5, #4574 v4) resolve lookup-/local-only.
4. **Lookup-only work leaves** — `master-company-new` (2026-06-25) blocks company minting from `resolve-edges-work` entirely (`allow_create: false`).
5. **`current_company_only`** — the C-suite lane filters employment to current roles and **skips phase 7 entirely** (#12828 v1.2).
6. **Opt-in investor cascade** — deal fanout requires `allow_investor_network_cascade: true` AND depth 0 (#12830 v3.3); depth alone is no longer enough.
7. **`suppress_downstream`** — all 9 async scraper dispatch sites in `create-master-link` suppressed on contained paths (#2541 v1.3 \+ #13039 v1.10).
8. **Stale-dispatch guard \+ 60 s source-keyed debounce** on the orchestrator (`force_run` bypass is deliberate — used by the FullEnrich re-run and by #13039 v1.11's smoke-seed safety re-dispatch).
9. **Kill switch** `check-kill-switch-v2` #4617 (headerless, stable since 2026-03-19) — `master_person` row-count cap; any check failure defaults the switch **OFF** via `log_crash`; when ON, existing people return local-only and blocked creations are audited in `log_crash` #542 only. _(The old core-concepts name "check-kill-switch-person" was wrong — no function by that name exists; legacy `check-kill-switch` #2774 is live but not the gate.)_
10. **Cron single-row tick** — Task #126 processes at most one queue row per 600 s, and it ships `active = false`: the manual-drain state is itself the hardest stop.

## Implementation notes captured during build

- **`set-person-names` #12820 exists to dodge a Xano naming-collision bug** — `db.edit` with data keys named `first_name` / `last_name` silently resolves them to the parent function's empty inputs; the helper's `fn` / `ln` / `mn` / `sfx` / `nick` inputs are the single writer for name fields. **Xano artifact — do not reimplement.**
- **Phase 10 #12831 is a literal empty stub** (stack `{}`, response null since v2.0, 2026-04-19) — and #13040 v3.23 **still calls it**. The "safe to remove in a later pass" cleanup never happened.
- **The deal-cascade depth coercion was a live regression:** from v1.11 (2026-06-23) until 2026-07-02, a `first_notempty:1` read coerced an explicit seed depth `0 → 1`, making `resolve-investors-edges` #12702 self-skip on every invocation. Repaired in #12856 v1.6 / #12702 v1.12 with a **nullable `cascade_depth` read via `first_notnull`**; v1.13 (same day) floors the ms→s division so deal dates with milliseconds can't crash the Twig `date()` render (#12701 got the same hardening). Deals processed in the window carry `Funding_Round` \+ `RAISED` only — backfill declined 2026-07-02, accepted as historical (see the company family's deviations table).
- **FullEnrich data\_source threading is literal-dispatch:** webhook #8594 calls `db.set_datasource` with a literal `"live"` / `"staging"` / `"sandbox"` per branch (Xano won't accept a `$var` there), and its v1.8 guard **refuses any item with an empty `custom.data_source`** — the Carl→Robert live-corruption fix. There is no `email_enrich_fallback` table or `matched_eef` counter anymore (branch dropped in v1.6; the old walkthrough is stale).
- **#12832's `db.query` where clause is stored as a backtick template string** with `&amp;&amp;` entities — the known inline-serializer artifact; flagged on [Person Waterfall Functions](/guides/enrichment/waterfall/person-functions). **Xano artifact — do not reimplement.**
- **Do not copy the stale `*80` weight comment** in #12925's input doc: the live `HAS_EXPERTISE` matched-edge weight is `min(round(10 + score × 160), 50)`, created nodes get flat 10, and `expertise_strength` is a separate 0–100 property — see [Person Edges](/guides/enrichment/waterfall/person-edges).
- **Integrity is app-level:** no unique key on the queue, no `linkedin_url` / `node_uuid` index on `master_person`, no `master_person_id` index on `person_enrich_data` — two calls passing the dedup ladder simultaneously WILL double-create. Index inventory: [Person Waterfall Tables](/guides/enrichment/waterfall/person-tables).

## Motivation

A single person exposes **companies** (current employer, past employers, schools, cert issuers, volunteer orgs, publishers) and — when the person is an angel — an **investment network** (portfolio companies, co-angels, VC firms, VC partners), each of whom exposes more rounds and more people. Without containment, one LinkedIn URL would mint hundreds of entities and thousands of paid provider calls two hops out.

The person waterfall solves this the same way Company, IMDB, and Music do — **a get-or-create gate, a depth-capped non-recursive cascade, a priority-tiered queue, and upsert dedup** — with two structural twists: the provider layer is a **tiered fallback ladder** (pay for the next tier only when the previous one came back thin), and company discovery is fully **decoupled from company enrichment** (person runs link companies; only the company waterfall enriches them).

**Goal:**

- From a seed **person**: create the canonical `master_person` \+ graph node, resolve contacts, avatar, LLM bios, expertise, and every edge family, and build the investor profile — with all discovered companies resolved lookup-/local-only and never queued.
- From a seed **company** (the C-suite lane): enrich every Exa founder / C-suite match fully in `current_company_only` mode — current employer linked, broad edge resolution skipped — without spawning another people or company wave.
- **Opt-in only:** investment-network expansion (portfolio rounds, co-angels, VC firms, partners) runs only when a caller explicitly passes `allow_investor_network_cascade: true` at seed depth.
