## Motivation

Conferences, summits, and trade shows are a primary signal for who-knows-who and who-invests-in-what, and they're currently invisible in the graph. Adding an `Event` node lets us answer things like:

- "Which of our people spoke at or attended LSI USA in the last 3 years?"
- "Which VC firms sponsor medtech events where portfolio CEOs present?"
- "Who else presented at the same event as this founder?"

Today these signals are scattered across bios, press mentions, and enrichment notes — nothing is structured.

## Proposed schema

### Base label

Follows convention: `Entity:Event` plus one subtype label from the **Category labels** table below.

### Core properties

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Primary identifier (UUID v4) |
| `name` | string | Full event name including edition (e.g. `"LSI USA '25 Emerging Medtech Summit"`) |
| `display_label` | string | Always `"Event"` |
| `visibility` | boolean | UI visibility flag |
| `name_embedding` | vecf32 | Semantic embedding |
| `created_at` | string | ISO 8601 timestamp |

### Identity & dedup

| Property | Type | Description |
|----------|------|-------------|
| `event_url` | string | Canonical event page URL — **proposed MERGE key** |
| `event_key` | string | Fallback dedup: `"series_slug::edition_year"` (lowercased), e.g. `"lsi-usa::2025"` |
| `slug` | string | URL-friendly event slug |

<Note>
Two MERGE candidates exist because some events don't have a stable per-edition URL. `event_url` is preferred when present; `event_key` is the fallback. **Open question** below.
</Note>

### Temporal

| Property | Type | Description |
|----------|------|-------------|
| `start_date` | string | ISO date (`YYYY-MM-DD`) |
| `end_date` | string | ISO date — equals `start_date` for single-day events |
| `duration_days` | integer | Inclusive day count |
| `edition_year` | integer | The year this edition runs |

### Lifecycle

Captures "is this event still live?" without duplicating `start_date` / `end_date`. The graph's `valid_from` / `valid_to` convention stays where it belongs — on edges.

| Property | Type | Description |
|----------|------|-------------|
| `status` | string | `"scheduled"` \| `"completed"` \| `"cancelled"` \| `"postponed"` |
| `announced_at` | string | ISO timestamp when this edition was publicly announced (predates `start_date`) |
| `cancelled_at` | string | ISO timestamp if cancelled, null otherwise |

### Recurrence

| Property | Type | Description |
|----------|------|-------------|
| `annual_recurring` | boolean | `true` if the series runs every year |
| `recurrence_pattern` | string | `"annual"` \| `"biannual"` \| `"quarterly"` \| `"monthly"` \| `"one_time"` |
| `series_name` | string | Parent series display name (`"LSI USA"`) |
| `series_slug` | string | Parent series slug (`"lsi-usa"`) — used in `event_key` |
| `first_edition_year` | integer | Year of the first edition, when known |

### Location (denormalized + edges)

Location is stored **both** as scalar props on the node (for quick access and virtual-event fallback) **and** via `LOCATED_IN` edges to existing `City`/`Region`/`Country` nodes for traversal.

| Property | Type | Description |
|----------|------|-------------|
| `venue_name` | string | Venue/building name |
| `city_name` | string | Parent city name |
| `region_name` | string | Parent region/state name |
| `country_name` | string | Parent country name |
| `format` | string | `"in_person"` \| `"virtual"` \| `"hybrid"` |

### Category / industry

| Property | Type | Description |
|----------|------|-------------|
| `industry` | string | Industry focus (`"medtech"`, `"fintech"`, `"ai"`, etc.) |
| `category` | string | Event category — see table below |
| `topics` | string | Comma-separated topic tags |

**Category labels (subtype):**

| `category` value | Additional label | Examples |
|---|---|---|
| `summit` | `Summit` | LSI USA, Milken, Aspen Ideas |
| `conference` | `Conference` | JPM Healthcare, AWS re:Invent |
| `trade_show` | `Trade_Show` | CES, HIMSS, RSNA |
| `workshop` | `Workshop` | YC office hours, bootcamps |
| `hackathon` | `Hackathon` | TreeHacks, HackMIT |
| `meetup` | `Meetup` | Local chapter events |
| `award_ceremony` | `Award_Ceremony` | Webby Awards, Emmys |
| `webinar` | `Webinar` | Virtual-only sessions |

### Attendance & audience

| Property | Type | Description |
|----------|------|-------------|
| `expected_attendance` | integer | Pre-event estimate |
| `actual_attendance` | integer | Post-event reported count (null until known) |
| `audience_types` | string | Comma-separated audience segments |
| `presenter_count` | integer | Number of presenting companies/people |
| `speaker_count` | integer | Distinct speakers (can exceed `presenter_count`) |
| `session_count` | integer | Number of sessions/talks |

### Registration

| Property | Type | Description |
|----------|------|-------------|
| `registration_url` | string | URL to register/apply |
| `registration_status` | string | `"open"` \| `"sold_out"` \| `"waitlist"` \| `"closed"` \| `"invite_only"` |
| `ticket_price_usd_min` | number | Lowest advertised ticket price in USD |
| `ticket_price_usd_max` | number | Highest advertised ticket price in USD |

### Descriptive

| Property | Type | Description |
|----------|------|-------------|
| `tagline` | string | Marketing tagline |
| `short_description` | string | One-sentence summary |
| `long_description` | string | Full description / about text |
| `logo_url` | string | Event logo |

### Social / web

| Property | Type | Description |
|----------|------|-------------|
| `website` | string | Public marketing site |
| `twitter_handle` | string | Primary Twitter/X handle |
| `linkedin_url` | string | Company-page LinkedIn URL |

## Proposed edges

Full edge spec with properties and examples lives in [Event Edges (proposed)](/guides/ontology/proposals/event-edges). Summary table below — **17 edges across 6 groups**. Weights follow existing convention (lower = stronger).

| Edge | Pattern | Weight |
|---|---|---|
| `HOSTS` | `(Organization\|Company)-[HOSTS]->(Event)` | 5 |
| `SUCCEEDS` | `(Event)-[SUCCEEDS]->(Event)` | 5 |
| `PART_OF` | `(Event)-[PART_OF]->(Event)` | 5 |
| `PRODUCES` | `(Company)-[PRODUCES]->(Event)` | 8 |
| `EMCEED` | `(Person)-[EMCEED]->(Event)` | 12 |
| `SPOKE_AT` | `(Person)-[SPOKE_AT]->(Event)` | 15 |
| `MODERATED` | `(Person)-[MODERATED]->(Event)` | 15 |
| `PRESENTED_AT` | `(Person\|Company)-[PRESENTED_AT]->(Event)` | 15 |
| `JUDGED_AT` | `(Person)-[JUDGED_AT]->(Event)` | 18 |
| `MENTORED_AT` | `(Person)-[MENTORED_AT]->(Event)` | 20 |
| `SPONSORS` | `(Company\|VC_Firm)-[SPONSORS]->(Event)` | 20–35 (by tier) |
| `FOCUSES_ON` | `(Event)-[FOCUSES_ON]->(DomainExpertise)` | 25 |
| `EXHIBITED_AT` | `(Company)-[EXHIBITED_AT]->(Event)` | 30 |
| `LOCATED_IN` | `(Event)-[LOCATED_IN]->(City)` | 30 |
| `ATTENDED_EVENT` | `(Person)-[ATTENDED_EVENT]->(Event)` | 55 |
| `REGISTERED_FOR` | `(Person)-[REGISTERED_FOR]->(Event)` | 65 |

Edge-level properties follow existing shape: `uuid`, `valid_from`, `valid_to`, `weight`, plus type-specific fields (e.g. `SPOKE_AT` carries `session_title`, `topic`, `role`).

## Proposed indexes

| Node | Property | Purpose |
|---|---|---|
| `Event` | `event_url` | MERGE dedup |
| `Event` | `event_key` | Fallback dedup |
| `Event` | `series_slug` | "All editions of LSI USA" |
| `Event` | `edition_year` | Time-range queries |
| `Event` | `status` | Filter out cancelled/postponed in default views |
| `Event` | `industry` | "All medtech events 2024–2026" |
| `Event` | `name_embedding` | Vector index for semantic search |

## Open questions

<AccordionGroup>
  <Accordion title="1. One MERGE key, or two?">
    Using both `event_url` and `event_key` means two merge paths. Recommend: `event_url` when we have it, `event_key` otherwise, with a dedup sweep to reconcile if both land for the same event. Alternative: always compute both, prefer `event_url` for MERGE.
  </Accordion>
  <Accordion title="2. Do we need an `EventSeries` parent node?">
    Right now recurrence is encoded as properties (`series_name`, `annual_recurring`) plus a `SUCCEEDS` chain between editions. An `EventSeries` node would let us attach series-level properties (total attendance across editions, aggregate sponsor list) and simplify "give me all LSI USA" queries. Tradeoff: more schema, more ingest complexity. Lean toward **no, defer** until we have a concrete query that needs it.
  </Accordion>
  <Accordion title="3. Should `ATTENDED` be a graph edge at all?">
    Attendance is often inferred and low-confidence. Risk is a flood of noisy edges. Options: (a) only create `ATTENDED` when evidence is strong (registration list, public post), (b) skip entirely and rely on `SPOKE_AT` / `PRESENTED_AT` / `EXHIBITED_AT`. Lean toward **(a)** with a confidence threshold.
  </Accordion>
  <Accordion title="4. How are events ingested?">
    Not scoped here. Candidates: manual seeding for the top-tier events we care about, scraping event pages on demand, enrichment-triggered creation when a person's bio mentions a known event. Needs its own spec.
  </Accordion>
  <Accordion title="5. Virtual / hybrid location handling">
    Virtual events have no physical city. Proposal: `format: "virtual"` sets `venue_name`, `city_name`, `region_name`, `country_name` to null; skip the `LOCATED_IN` edge. Hybrid events keep the primary physical location.
  </Accordion>
</AccordionGroup>

## Not included (out of scope)

- Agenda/session-level structure (individual talks as nodes)
- Ticket tiers as structured objects
- Real-time event state (live/ended/upcoming) — derive from `start_date`/`end_date` at query time
- Event photos / video URLs beyond a single `logo_url`
