the in-chat confirmation card is gone, and every dev-only tuning lever is tucked

## Shipped — Functional

### #35 — In-chat confirmation card removed

The in-chat `InterviewSummaryConfirm` card is deleted, along with its
`awaitingConfirm` state machine and the confirm-prefix intercept lane. When the
interview is ready, the canvas now emits a static "Ready when you are." signpost —
no card, no auto-dispatch. The **right-rail Summary panel plus the Submit button is
the sole confirm surface**. Old persisted threads that replay a legacy confirm card
route their Dispatch click through the same path for back-compat.

_Commit `044f4d12`._

### #39 — Modify a completed suggestion mints a NEW row

On Submit of a completed `status:"suggestion"` row, the app now mints a **brand-new
`suggestion_request` row** rather than mutating the original. Per the Mark/Robert 6/3
decision, new rows preserve modification history for future reinforcement learning —
a boolean flag would lose that history.

<Steps>
  <Step title="Mint a fresh id">
    `useStartOutcome` (ep 8502) creates a new `suggestion_request_id`.
  </Step>
  <Step title="Promote the new row">
    `useSubmitRequestMeta` (ep 8617) flips **that new row** from draft to processing
    and persists the result count (12) and delivery format. The original row is only
    deselected — never written.
  </Step>
  <Step title="Dispatch on the new id">
    `useOpenui05Dispatch` (ep 8497) fires on the new id with the Modify-field diff
    carried as an `update_pitch_profile` tool call.
  </Step>
</Steps>

**Fired live 6/4.** Submit on suggestion 245 minted new row **1718** (`start-outcome`
200 → `submit-request-meta` 200), and the dispatch reached the server (history shows
`/openui05/dispatch` 200 + `/openui05/find-investors` 200). The original 245 is
byte-identical afterward and fully re-openable.

## Known open gap (backend)

<Warning>
  **#39's new row never renders cards — this is a backend lifecycle gap, not a
  frontend regression.** Live 6/4, new row 1718 was minted and dispatched (server
  returned 200 on both `/openui05/dispatch` and `/openui05/find-investors`), but the
  row stays `processing` with `outcome_suggestion_count: 0` well past the ~74s the
  pipeline normally takes. The dispatch returns 200 but its results are never written
  back onto the row as `outcome_suggestions` with a status flip — the **M10 backend
  lifecycle automation** gap, which affects every row. F1's actual subject (a new
  ROW on resubmit, history preserved) is done.
</Warning>

There is also a non-blocking edge for **non-investor** classes: the
`/suggestion-requests` response carries no `class` field, so the rail can only infer
`find_investors` from pitch-profile presence. The fix is backend — either add `class`
to that response, or have the dispatch path re-stamp the class from its
re-classification of the edited summary. This edge does not affect the find_investors
path exercised in the 6/4 dogfood.
