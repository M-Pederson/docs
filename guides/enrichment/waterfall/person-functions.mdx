## Orchestrator

<Card icon="layer-group">
  ```text
  mvp/enrich/enrich-master-person ‚Äî #13040
  ```

  **v3.51 (2026-07-14) ‚Äî the person orchestrator** (phases 0‚Äì12, with phase 9 run twice). Dispatched async-shared by `master-person-new` #13039 (new, non-queued depth-0 people), `process-full-enrich` #13049 (`force_run: true` re-run after a FullEnrich writeback), the company orchestrator's C-suite mode (#12992 Step 7 via #12997), and the queue drains (Task #126 / #12816). Inputs: `master_person_id`, `deep_research`, `cascade_depth`, `force_run` (v3.10), `current_company_only` (v3.13), `allow_investor_network_cascade` (v3.14). Pre-flight: a **stale-dispatch guard** returns early with a `log_crash` entry when the `master_person` row was deleted since the async fire; then the **60 s source-keyed debounce** ‚Äî any `enrich_history_person` row with the same `master_person_id` \+ `source == "Master Person Enrich"` \+ `updated_at` inside the window logs `"DUPLICATE TRIGGER, debounced (60s window)"` and returns null (debounce rows log `qa_passed: true`); `force_run: true` bypasses it. The setup row is upsert-keyed on `(master_person_id, source)` ‚Äî source **text**, not the legacy numeric `data_source_id` ‚Äî so the debounce key is `updated_at` (touched every re-run), not `created_at`.

  **Live chain:** 0 ‚Üí 1 ‚Üí 2 ‚Üí 3 ‚Üí 4 ‚Üí 4b (Exa) ‚Üí 5 ‚Üí **EARLY phase 9** (`build_investment_thesis: false`, immediately after Fundable linking) ‚Üí 6 ‚Üí **best-current-role sync** via `mvp/work/choose-best-current-role` #12621 ‚Üí safety close ‚Üí 7 ‚Üí safety close ‚Üí 8 ‚Üí **phase 8b** (the reverse music bridge `detect-and-enter-artist` #13171, gated on `$env.reverse_music_bridge == "on"`) ‚Üí **phase 9 AGAIN** (`build_investment_thesis: true`) ‚Üí 10 (no-op stub, still called) ‚Üí 11 ‚Üí 12 ‚Üí final `enrich_history_person.processing = false` close ‚Üí closeout aggregate ‚Üí **post-thesis best-effort retry** (re-runs `create-llm-person-bios` #2537 \+ `update-person-node` #2758 so a thesis built late in the run still reaches the bio and the graph node) ‚Üí **exit repairs \+ current-company dispatch** (below). Every phase runs in its own `try_catch` writing per-phase `qa_passed` / `CRASH: phase-N` rows to `log_crash` #542 (strict `phase` enum ‚Äî see [Core Concepts ‚Üí Per-Phase Crash Logging](/guides/enrichment/waterfall/core-concepts#per-phase-crash-logging)); failures are logged but never block later phases.

  **Controlled-waterfall exit:** the closeout repairs `master_person.master_company_id` from the selected current work row and, for root runs, direct-dispatches a sparse selected current company to `enrich-master-company` #12992 at depth 0. A depth-1 queue record does not satisfy that dependency. Schools follow a different bounded contract: #12560 verifies their table/graph education state, then directly dispatches company enrichment in detached `async-shared` mode at depth 1 with no captured result and no sleep.

  **Optional Deep Bio boundary (v3.58, 2026-07-17):** `deep_research` is normalized once. Provider launch, profile-job creation, company-dependency rows for that profile job, and the durable Deep Bio barrier execute only when it is true. False goes directly to `finalize-person-semantic-state` #13194 with `require_deep_bio=false`; stale profile-job rows cannot reactivate the optional barrier. Non-bio success still requires work/date/tense/UUID parity, education, leaf surfaces, current role, expertise, classifications, owner traceability, and graph readback.

  **The v3.51 terminal contract (2026-07-14):** when root closeout selects a sparse current company, #13040 records `person_terminal_company_dependency` #760 before dispatching #12992 at depth 0. Company closeout calls #13199, which succeeds the dependency only after fresh company table/history evidence newer than `required_after`; a depth-1 queue or resolver row cannot satisfy it. #13196 holds person success until provider, callback, orchestrator, and current-company writers agree. Phase 8 recovery re-reads committed expertise and calls #13192 before accepting a previous write, so count-only or stale graph state cannot suppress a needed retry.
</Card>
