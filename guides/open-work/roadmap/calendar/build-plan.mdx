<Note>
**Status: design / spec — part of the Calendar plan; nothing built yet.** This page is the phased **build plan** that turns the [Calendar plan](/guides/open-work/roadmap/calendar) into a dependency-ordered delivery sequence. It is a design explainer, not a shipped artifact — every phase, table delta, and index here is a proposal to ratify. Read it alongside its siblings: the [Data model](/guides/open-work/roadmap/calendar/data-model) (the reused/new tables, columns, and btree indexes each phase requires), [Settings & privacy](/guides/open-work/roadmap/calendar/settings-privacy) (the per-account privacy ladder and the server-side `redact_to_rung` enforcement Phase 3 depends on), and the [NL date picker](/guides/open-work/roadmap/calendar/nl-date-picker) (the shared natural-language date input wired in Phase 2).
</Note>

Four phases, **dependency-ordered**, each shipping **standalone value** on its own. The throughline: stand up the render layer on data we already have → connect and mirror real calendars → make it writable and graph-linked → unlock the premium team/AI surface. Every phase lists its concrete table/endpoint/plugin deltas, the btree index primitives it requires (detailed on the [Data model](/guides/open-work/roadmap/calendar/data-model)), and where it sits in the free→premium gate.

The value filter is strict: each phase must make the operator's daily loop faster or safer by itself. The smallest useful slice is not "a complete calendar clone"; it is a reliable place to see dated Orbiter work and the next commitment without leaving the app. The next slice is real synced events with privacy settings and indexed reads. Only after the mirror is trustworthy does write-back, drag-and-drop, recurrence, graph binding, and AI prep become worth the risk. Team overlays, Scheduler pages, and AI cross-teammate booking wait because they are valuable only when access control, org scope, and provider reconciliation are already boring.

**Sequencing rationale.** The order is forced by hard dependencies, not preference: the **render layer + app wiring come first** (Phase 0) so every later phase paints into a proven surface — and they ride read-only `df_*` data that needs **zero Nylas**, so Phase 0 is demo-able before any OAuth exists. **Reads precede writes** (Phase 1 before Phase 2) because there is nothing to drag-reschedule until real events are mirrored, and the **time-range btrees must land before real data does** (Phase 1, before the backfill) or the canvas degrades with row count. **Graph-linking and write-back** (Phase 2) depend on the Phase-1 webhook receiver to reconcile every optimistic write. The **org-scope decision, the per-account privacy enforcement, and the purchased premium license** are deliberately deferred to **Phase 3** — team rows are not safe until org scope on the legacy tables is settled and `redact_to_rung` is enforced server-side (see [Settings & privacy](/guides/open-work/roadmap/calendar/settings-privacy)), and the `schedulerLicenseKey` is a real spend that should not gate the free-tier launch. The **AI-first prep surface** (the ranked `df_focus_entry` budget, the Copilot deep-link, meeting briefings) is read-only and rides the free tier; only *AI booking across teammates* (Availability over the resource timeline) waits on the premium Phase 3.

## Phase dependency map

| Phase | Depends on | Ships (standalone value) | Plugin tier | Free→premium gate |
| ----- | ---------- | ------------------------ | ----------- | ----------------- |
| **P0** Read-only overlays | — | Calendar active-view + month/week/day + agenda panel over `df_*` data; palette command; double-tap `0`; theme map; week-start + timezone. **No Nylas.** | Free MIT only | 100% free |
| **P1** Connect / sync / mirror + core views | P0 (render layer + wiring) | Real synced calendars rendering; Calendar Settings (privacy-ladder toggles); webhook receiver; the verified time-range btrees | Free MIT (+ tz adapter) | 100% free |
| **P2** Write-back / recurrence / graph-linking | P1 (webhook + mirror) | Drag-everywhere, single `schedule_meeting` write path, recurrence render, categories, `cal_event_link` graph edges, reminders, NL date entry, AI prep deep-link | Free MIT (+ `interaction` full, `rrule`) | 100% free |
| **P3** Team overlays / Free-Busy / AI scheduling | P2 + org-scope decision + `redact_to_rung` enforcement | Team resource timeline, Free/Busy + Availability, Nylas Scheduler pages, AI proposes-times scheduling across teammates | **Premium (`schedulerLicenseKey`)** | **Deliberate spend; free-tier P0–P2 is the permanent fallback** |

## Phase exit evidence

Each phase exits only when the next dependency is boring enough to stand on:

| Phase | Exit evidence | Explicit non-goal |
| ----- | ------------- | ----------------- |
| P0 | Two dated `df_*` layers and `important_date` render across month/week/day/agenda; single `0` and `0 0` behave; theme, week-start, timezone, keyboard, and responsive checks pass. | No Nylas OAuth, no provider event mirror, no event writes. |
| P1 | A connected grant backfills within bounds, webhook replay is idempotent, stale delivery is discarded, delta poll repairs a missed webhook, and indexed reads serve the visible window. | No drag-to-write, no teammate event exposure, no Scheduler page. |
| P2 | Button, drag, resize, chip-drop, RSVP, and confirmed AI proposal all share one pending/reconcile envelope; `cal_event_link` feeds Meeting Prep; undo and failure leave no mirror drift. | No premium resource rows, no cross-teammate Availability booking. |
| P3 | Network responses prove privacy redaction, Availability is cached and rate-limited, `schedulerLicenseKey` is production-valid, and free-tier fallback works with the license absent. | No silent AI booking beyond the explicit autonomy gate. |

That evidence is what prevents the plan from becoming a calendar-shaped wish list. The next phase starts only when the previous phase has reduced the real technical and trust risk.

## Phase 0 — Read-only overlays (no Nylas, free plugins)

> **Standalone value:** a working, theme-aware, keyboard-driven calendar active-view that paints real Deal-Flow data — shippable and demo-able with **zero** Nylas dependency.

Render dated Deal-Flow data in FullCalendar with **zero** Nylas dependency — proves the render layer, the active-view registration, the theme mapping, the command/hotkey wiring, the timezone/week-start wiring, and the Event-Source model first, so everything later paints into a known-good surface.

- **Plugins to add:** `@fullcalendar/react`, `@fullcalendar/daygrid`, `@fullcalendar/timegrid`, `@fullcalendar/list` (free MIT).
- **App wiring (the ordered wiring steps below):** register `/calendar` as an `active-view` (add `"calendar"` to the `ActiveView` union + `viewTitles` in `src/types/active-view.ts` + top-nav + side-nav + network-canvas); add the file-based route `/calendar` under `src/routes/_authed/`; scaffold `src/features/calendar/{api,components,hooks,types,constants}`; add the Calendar command (`commands.tsx`) and the double-tap `0` hotkey (`use-command-hotkeys.ts`); map FullCalendar `--fc-*` onto the OKLch tokens in `styles.css` so `.dark` flips automatically; wire `firstDay` to `user.settings.week_start` and `timeZone` to `useTimezone()` with the tz adapter; mount the canvas + side-panel.
- **Views:** full-screen month/week/day + docked agenda side-panel.
- **Data:** read-only Event Sources over `df_task` (due/start), `df_deal` (`expected_close_date`), `df_project`/`df_project_section`, `important_date` (242). **No event writes.**
- **Tables:** none new (reads Deal-Flow tables + `important_date` 242). The calendar lens is `df_view.view_type='calendar'`.
- **Indexes (P0):** btrees on `df_task(due_date)`, `df_task(start_date)`, `df_deal(expected_close_date)` (from the Deal Flow design); composite `(organization_id, due_date)` for org-scoped range scans. (Full index table on the [Data model](/guides/open-work/roadmap/calendar/data-model).)
- **Exit criteria:** single `0` opens the panel, `0 0` opens full screen (form-tag-guarded); the `df_*` layers paint with correct colors in **both** light and dark; week-start + timezone are correct; the responsive/keyboard behavior passes its acceptance criteria.
- **Invariant acceptance:** there are no Nylas grants, no provider writes, and no `event` (221) mutations in this phase; all visible dates derive from read-only `df_*` / `important_date` sources; week-start and timezone are shared by the canvas and side-panel so the same item never appears on different days between surfaces.

### Phase 0 wiring steps

The concrete, ordered edits that stand the active-view up — every path is a real file in orbiter-frontend. This is the canonical wiring reference; Phase 0 executes it against read-only `df_*` data with zero Nylas dependency.

<Steps>
  <Step title="Register the active-view (registry 1 of 2)">
    Add `"calendar"` to the `ActiveView` union and a `calendar: "Calendar"` entry to `viewTitles` in `src/types/active-view.ts` (verified existing members: `discover` / `collections` / `teams` / `outcomes` / `leverage-loops` / `serendipity` / `meeting-prep`); wire it into the top-nav, side-nav, and network-canvas the way the existing views are.
  </Step>
  <Step title="Add the route + feature folder (registry 2 of 2)">
    Create the file-based route `/calendar` under `src/routes/_authed/` and the feature folder `src/features/calendar/{api,components,hooks,types,constants}`. The route mounts the canvas; the side-panel mounts wherever the app shell already docks panels. Reuse `src/features/events` for all event reads. (The route is a separate registry from the `ActiveView` union — both must be touched.)
  </Step>
  <Step title="Add the Calendar command (palette)">
    In `src/features/command-palette/constants/commands.tsx`, add a Navigation `CommandDefinition`: `{ id:'calendar-view', label:'Calendar', category:'Navigation', icon:<CalendarIcon className='size-4'/>, action:()=>navigate({to:'/calendar'}), shortcut:'0 0' }`. `command-palette.tsx` auto-renders the **Navigation** heading by filtering `c.category === "Navigation" && !c.hidden`, so it lands beside Activity (`/`), Network (`/network`), and Email Inbox (`/email-inbox`) with no extra wiring.
  </Step>
  <Step title="Add the strict double-tap '0' hotkey">
    In `src/features/command-palette/hooks/use-command-hotkeys.ts` add a dedicated `useHotkeys('0', …)` **outside** the existing `for (const cmd of commands)` loop (which only registers `cmd.hotkey` combos via `useHotkeys(cmd.hotkey ?? "", …, { enabled: !!cmd.hotkey && !cmd.hidden })`), with the ~300ms single-vs-double disambiguation layer below. Keep the loop untouched so the `'0 0'` `shortcut` on the command stays display-only.
  </Step>
  <Step title="Map the theme tokens">
    In `styles.css`, define the `--fc-*` → OKLch token mapping (both `:root` and `.dark` flip automatically via `@custom-variant dark (&:is(.dark *))`), and bind the today/selection/"my events" colors to the selected `THEME_COLORS` accent.
  </Step>
  <Step title="Wire week-start + timezone, then mount FullCalendar">
    Set FullCalendar `firstDay={user.settings.week_start}` and `timeZone={useTimezone()}` (from `src/lib/timezone.ts`), register the FullCalendar tz adapter plugin, then mount the `@fullcalendar/react` canvas + docked side-panel. All app-side date math uses `toTZDate`/`nowTZ` on `@date-fns/tz`.
  </Step>
</Steps>

## Phase 1 — Nylas connect, sync, mirror & core views

> **Standalone value:** users connect their real Google/Outlook/iCloud calendars and see their actual events on the canvas + side-panel, kept fresh by webhook — a complete read-only synced calendar, still 100% free MIT plugins.

Connect real calendars and render them; one-direction sync (Nylas → mirror). **Ship the verified time-range btrees before the initial backfill lands**, or the canvas reads degrade with row count.

- **Plugin additions:** `@fullcalendar/interaction` (select/click only — no write yet), the FullCalendar tz adapter (`@fullcalendar/luxon3` or `@fullcalendar/moment-timezone`) for named-zone rendering.
- **Views:** month / week / day full-screen + agenda + single-day side-panel, now showing real synced events through the reused `useCalendarEvents()` (V2 group `LR2ywW7R`).
- **Settings:** add `enable_calendar` / `share_conflicts` / `share_conflict_details` (+ default-calendar pointer) to **`nylas_grant` (183)**; render the per-account toggle rows (one per grant, strict cascade) in the account modal; add `settings.week_start` to **`user` (125)**. The cascade UI is convenience only — server-side `redact_to_rung` lands with the team read in Phase 3 (see [Settings & privacy](/guides/open-work/roadmap/calendar/settings-privacy)).
- **Auth/grant:** OAuth connect via Nylas → store `grant_id` in **`nylas_grant` (183)**; list connected calendars into **`calendar` (223)** (reuse [Get calendars](/api-reference/xano/v2-calendar-events/get-calendars) / [Set active calendars](/api-reference/xano/v2-calendar-events/post-calendars) via `useCalendars()` in `src/features/events/api/use-calendars.ts`).
- **Initial sync:** backfill events into **`event` (221)** / **`master_event` (222)** (the [Historical Calendar Parse](/guides/log-parse/historical-calendar) path), **paged** and bounded by `nylas_grant.timerange` / `oldest_sync` — **after** the btrees below exist.
- **Webhook receiver:** ingest `event.created` / `event.updated` / `event.deleted` / `grant.expired` ([event-created](/guides/log-parse/event-created), [event-updated](/guides/log-parse/event-updated), [event-deleted](/guides/log-parse/event-deleted)) into raw `webhook_calendar_log` (618) → dedupe via new **`cal_webhook_event`** (unique `delivery_id`) → upsert `event` (221) by `external_id` (last-writer-wins on the provider `updated_at`/sequence).
- **Reads:** drive the canvas from the V2 [Calendar Events](/api-reference/xano/v2-calendar-events/calendar-events) endpoint against the mirror.
- **AI prep (read-only, free):** the Copilot already ranks the next meeting and drafts a prep prompt off this mirrored data (`use-conversation-starters.tsx`); surface the "Prep with AI" deep-link on agenda rows / event cards (`postMessage` target `'Copilot'` with a `prompt`). No writes yet.
- **Tables:** new **`cal_webhook_event`** (idempotency); optionally **`cal_calendar_pref`** (NEW) for color/visibility (written by `useUpdateCalendarPreference()` in `src/features/events/api/use-update-calendar-preference.ts`).
- **Indexes (P1 — the real gap, verified):** btrees on **`event(calendar_id, start)`**, **`event(start)`**, **`event(end)`**, **`event(master_event_id)`**, **`event(deleted_at)`**, **`calendar(grant_id)`**, **`calendar(user_id)`**, **`nylas_grant(grant_id)` unique**, **`nylas_grant(user_id)`**; unique on **`cal_webhook_event(delivery_id)`**. Today every one of these legacy tables carries only `primary(id)` + `btree(created_at desc)`. (Full index table on the [Data model](/guides/open-work/roadmap/calendar/data-model).)
- **Exit criteria:** a freshly connected account backfills + renders within its sync bound; a provider-side change reconciles the mirror via webhook (with redelivery as a no-op and a stale out-of-order delivery discarded); the privacy-ladder toggles persist on `nylas_grant` (183) and read via `first_notnull`; window reads hit the new btrees, not table scans.
- **State-integrity acceptance:** manually replay the same webhook twice and confirm the second delivery is a no-op; replay an older `event.updated` after a newer one and confirm it is discarded; disconnect a grant and confirm reads stop returning its event detail before the purge sweep completes; run the bounded delta poll after dropping one webhook and confirm it repairs the mirror through the same upsert comparator.
- **AI-trust acceptance:** "Prep with AI" opens with a pre-drafted prompt and read-only meeting context only; no prep or ranking surface can create, move, RSVP, or link an event in Phase 1.

## Phase 2 — Write-back, recurrence, categories & graph-linking

> **Standalone value:** the calendar becomes writable — drag to reschedule, drop external chips to schedule, link meetings to deals/people, prep with AI — all through one reversible contract, still on free MIT plugins.

Make the calendar writable and connect it to the graph. Every write depends on the Phase-1 webhook receiver to reconcile the optimistic state.

- **Plugin additions:** `@fullcalendar/interaction` (full drag-create/move/resize + external `Draggable`), `@fullcalendar/rrule` (recurrence render).
- **Write path:** `eventDrop` / `eventResize` / `select` / external-chip drop / cross-panel drag / AI confirmation → a new `use-schedule-meeting` mutation → the single `schedule_meeting` `df_action_def` verb → Nylas Events API by `grant_id` (create/update); RSVP via `useSendRsvp()` (`src/features/events/api/use-send-rsvp.ts`) → [Send RSVP](/api-reference/xano/v2-calendar-events/calendar-events-send-rsvp); reconcile through the Phase-1 webhook receiver. Optimistic pending block keyed to the pending `external_id`. A write against an expired grant or a `read_only` calendar is blocked client-side and server-side. **No direct `event` (221) mutation.**
- **Recurrence:** prefer Nylas-expanded series off `master_event` (222) (overrides/cancellations/EXDATEs intact); render with `rrule` as the render-only fallback. Recurring tasks use `df_task.recurrence_rule` + spawn fn.
- **Categories:** one Event Source per category, theme-mapped `eventColor`/`classNames` against `--chart-1..5`, each chip labelled + icon-paired.
- **Reminders:** per-event nudges into `df_reminder` → cron → `activity` (218) + push.
- **NL date entry:** wire the shared NL date input across create / reschedule / filter / reminder (consuming the resolved `TZDate`), and as the resolution layer for AI/voice-proposed times — design on [NL date picker](/guides/open-work/roadmap/calendar/nl-date-picker).
- **Graph-linking:** new **`cal_event_link`** (NEW) binds an `event` (221) to a `df_deal` / `master_company` (142) / `master_person` (139); the **external-chip drop** and the **AI's "bind to deal" proposal** both write this edge under the same confirm; feeds Meeting Prep + `df_deal.momentum_last_activity_at`.
- **AI prep deep-link (writable side):** the ranked `df_focus_entry` (`entry_type='meeting'`) attention budget drives the prep-priority list; each entry deep-links into the Copilot with a pre-drafted `prompt`; the Copilot's reschedule/bind proposals confirm through `schedule_meeting`.
- **Indexes (P2):** btrees on `cal_event_link(event_id)` and `cal_event_link(resource_type, resource_id)`; `df_reminder(remind_at)` (firing sweep); composite `event(organization_id, start)` only if `organization_id` is denormalized onto `event` (221) (otherwise org range scans resolve via the `calendar.user_id → organization_membership` join — see Phase 3). (Full index table on the [Data model](/guides/open-work/roadmap/calendar/data-model).)
- **Exit criteria:** a drag-create / drop / resize round-trips through `schedule_meeting` → Nylas → webhook and the optimistic block resolves (or rolls back with a toast); an external chip dropped on a date (or an AI bind proposal, once confirmed) writes a `cal_event_link` edge and surfaces in Meeting Prep; recurring series render off the mirror; a "Prep with AI" click opens the Copilot pre-loaded for that meeting.
- **State-integrity acceptance:** every write surface produces the same pending-state envelope and the same reconciliation behavior; a Nylas failure, expired grant, read-only calendar, or stale webhook leaves no half-written mirror row; a recurring-instance edit uses the provider result rather than client-authored RRULE mutation.
- **AI-trust acceptance:** an AI reschedule/bind proposal is an editable object with visible rationale, confidence, proposed values, and accepted values; operator edits are captured; an expired grant, read-only calendar, Nylas failure, or no-availability result renders as a failed proposal with no `event` (221) mutation; undo routes through a second `schedule_meeting` proposal.

## Phase 3 — Team overlays, availability & AI scheduling (premium)

> **Standalone value:** the premium team tier — see teammates' availability, book across calendars, and let AI propose times across participants — gated by the privacy ladder and a purchased license, with the free-tier (P0–P2) as the permanent fallback for accounts that never buy it.

The premium tier and the full AI-first scheduling surface (cross-participant booking). **Prerequisites:** the org-scope decision on the legacy tables (team rows are unsafe until it is settled), the server-side `redact_to_rung` privacy enforcement (see [Settings & privacy](/guides/open-work/roadmap/calendar/settings-privacy)), **and** a purchased `schedulerLicenseKey`.

- **Premium plugin additions — purchased `schedulerLicenseKey`:** `@fullcalendar/resource`, `@fullcalendar/resource-timeline`, `@fullcalendar/resource-timegrid`, `@fullcalendar/multimonth`.
- **Org-scope hardening (prerequisite):** decide org scope on the legacy tables — denormalize `organization_id` onto `event`/`calendar` (with the `(organization_id, start)` composite) **or** resolve `event → calendar.user_id → organization_membership` (696) on every team read. **Team rows are not safe until this is settled.**
- **Privacy enforcement (prerequisite):** the `redact_to_rung(event, viewer, owner_grant)` projection lands here — every team-facing read and every Availability fan-out passes through it, redacting to busy-only or detail per the owner's ladder rung **before serialization**, on top of `resolve_access()`. (The ladder semantics and the per-rung redaction table live on [Settings & privacy](/guides/open-work/roadmap/calendar/settings-privacy).)
- **Team overlay:** resource Timeline, one row per `team_member` (614), `resourceGroupField` by `team` (613), `resourceAreaColumns` data grid, drag-between-people (`eventDrop` re-assigns via `schedule_meeting`). **Exposure obeys the privacy ladder** server-side. Free-tier fallback = month/week/day/agenda only, no resource rows.
- **Free/Busy + Availability:** Nylas Free/Busy fills unbookable blocks across teammate rows (busy vs. detail per `share_conflict_details`, via `redact_to_rung`); Nylas Availability finds open slots across participants; the chosen slot feeds the Events API to book. **Cache against the mirror, refresh on webhook, never on scroll** — N teammates × a week is N Free/Busy fan-outs.
- **Booking pages:** Nylas Scheduler Configuration + Scheduling Pages (one-on-one / collective / round-robin / group) via `@nylas/web-elements` (`nylas-scheduling`, `nylas-scheduler-editor`); external pointer reuses `scheduling_link` (601).
- **AI-first, human-in-the-loop (full loop):** the Copilot proposes times across participants (Availability over `redact_to_rung` Free/Busy), drafts the invite, binds to the deal (`cal_event_link`), and ranks "meetings to prep" into `df_focus_entry` (`entry_type='meeting'`, `df_suggestion.source_engine='meeting_prep'`) — the operator confirms; every write is reversible/attributable through the same `schedule_meeting` verb. The Copilot is opened/handed a scheduling or prep `prompt` via `postMessage` target `'Copilot'`, and hands-free voice (Gradium/ElevenLabs TTS, Web-Speech/Gradium STT) drives the same path.
- **Indexes (P3):** composite `event(grant_id, start)` is **not** directly buildable (`event` has no `grant_id`); for cross-teammate range scans use `event(calendar_id, start)` joined through the per-teammate `calendar` rows, plus the membership composite — or the `(organization_id, start)` composite if org is denormalized. (Full index table on the [Data model](/guides/open-work/roadmap/calendar/data-model).)
- **Exit criteria:** with a valid `schedulerLicenseKey`, the resource timeline renders teammate rows that obey the privacy ladder server-side (busy-only vs detail per rung, no leaked fields on the wire); Availability returns bookable slots across participants without amplifying Nylas calls per scroll; an AI-proposed slot, once the operator confirms, books through `schedule_meeting` → Nylas → webhook; a Scheduler page books through the Events API; the free-tier degrades cleanly when no license is present.
- **Invariant acceptance:** inspect the network response for each ladder rung and confirm no detail fields are serialized below the detail rung; confirm an org member without `resolve_access()` / `df_grant` is denied before busy-only projection; confirm expired or disabled grants are excluded from teammate rows and Availability; confirm AI scheduling receives exactly the same redacted projection as the human resource-timeline read.
- **AI-trust acceptance:** every cross-teammate proposal explains only allowed facts: "busy" where detail is hidden, named conflicts only where detail is shared, and "no accessible anchor" when privacy blocks context. The proposal card must show alternate slots, retry after reconnect, and a clear failure state for rate limits or provider-specific unavailable data.
