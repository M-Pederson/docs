## Purpose

The forward spec for the important-dates model: how dated facts about people and companies **will** be enriched, captured, user-added, and surfaced — reminders, calendar, meeting prep, and mem0. Every design element is **PROPOSED** until built; every settled call is recorded in [Decisions](#decisions) with its date. The verified current-state facts that feed the build (row counts, quirks, broken endpoints) are compressed into [Migration inputs](#migration-inputs) at the end — the sections above it describe the target, not today.

Started 2026-07-16.

## The four date classes

Every stored date belongs to exactly one class — class determines scope, mutability, and merge behavior:

| Class | Example | How it flows | Scope |
|---|---|---|---|
| **Enriched** | Birthday from IMDB; company founded / funding-round dates | Written once by pipelines through the `master_date` choke point; refreshed on re-enrich; immutable by users | `master_person` / `master_company` — shareable |
| **Captured** | vCard `BDAY`, LinkedIn *Connected On*, first email | Derived deterministically from imports and activity data; recomputable, system-owned, never user-edited | Per-user contact — private |
| **User-added** | Wedding anniversary the user types in | CRUD through the reworked V2 API; fully user-editable; never touched by enrichment | Per-user contact — private |
| **Conversationally surfaced** | "My daughter graduates in June" via mem0 extraction | Proposed by extraction, confirm-first, lands as a user-tier row with memory provenance | Per-user contact — private |

Rows never cross-write between classes; conflicts resolve at read time (user-added beats enriched). The rest of this page specifies the tables, functions, environments, and rollout that make this real.
## Which backend owns what

Two environments, per [Universe &lt;&gt; Backend Sync](/guides/open-work/universe-backend-sync):

- **Universe** — the standalone source-of-truth master graph where the enrichment engine (the waterfalls) runs. `master_*` tables live ONLY here.
- **Orbiter Backend** — the per-user app server connected to the frontend. Holds entity tables plus the two-tier detail tables: `enrichment_*` (shareable, projected down from Universe) and `contacts_*` (user-scoped, private).

The rule for dates falls straight out of the class model:

| Class | Canonical home | Reaches the frontend via |
|---|---|---|
| **Enriched** (birthday from IMDB) | **Universe** (`master_date`) | Push/projection → Orbiter Backend `enrichment_date` |
| **Captured** (connected-on, first email, import birthdays) | **Orbiter Backend** (`contacts_date`) | Direct — activity + import data only exist Backend-side |
| **User-added** (wedding anniversary) | **Orbiter Backend** (`contacts_date`) | Direct |
| **mem0-promoted** | **Orbiter Backend** (`contacts_date`) | Direct — mem0 keys on the contacts record |

Rows never cross-write between tiers. Conflicts resolve at **read time** on the Backend: `contacts_date` beats `enrichment_date` (the user correcting a wrong IMDB birthday sees their version; Universe keeps enrichment truth).

<Note>
**Interim state:** until Universe standalone ships, EVERYTHING — both tiers, all tables — is built and tested in the **Xano Demo Env** (workspace 3, the current build being refactored into Universe + Orbiter Backend). See [Xano Demo Env build scope](#xano-demo-env-build-scope) for exactly what gets built there and the tier discipline that keeps the later split a lift-and-shift, not a rewrite.
</Note>

## Target tables

### `master_date` — NEW · Universe

The canonical store for enrichment-sourced dated facts about an entity — **persons AND companies, one table** (decided direction 2026-07-16: company dates are in scope — funding rounds, founded, later product announcements). Entity-polymorphic via the `master_address` pattern: two nullable FKs, exactly one set per row, null slots deduping consistently. Supersedes `birthday_join` (406).

| Column | Type | Notes |
|---|---|---|
| `master_person_id` | int → `master_person`, nullable | exactly one of person/company set per row |
| `master_company_id` | int → `master_company` (142), nullable | company dates: founded, funding rounds, IPO, acquisition, product announcements |
| `date_descriptor_id` | int → `date_descriptor` | system descriptors only (birthday, death, founded, funding_round, …) |
| `year` / `month` / `day` | int, all nullable | partial dates first-class: "June 14, no year" = (null, 6, 14) |
| `date` | date, nullable | derived, set only when all three parts present — keeps sorting and existing consumers simple |
| `natural_key` | text | `slug\|yyyy\|mm\|dd` fixed pipe slots (empty ≠ shift, per the [address spec](/guides/robert-mark/workshop-address) convention); singleton descriptors key on slug alone |
| `metadata` | json | descriptor-specific payload: `birthplace_location` for birthday, spouse name / `is_current` for anniversary |
| `data_source_id` | int → 161 | real provenance (IMDb, MusicBrainz, PDL, …) |
| `source` | text | matches the pattern on other enrichment-written tables |

Indexes: **unique `(master_person_id, master_company_id, natural_key)`** — the `master_address` composite exactly; the missing race protection, and still the generic `(node_uuid, natural_key)` upsert contract from the [AlloyDB → Xano sync design](/guides/open-work/universe-backend-sync) once Universe keys on `node_uuid`. `master_date` becomes the **fifth facet table** in that projection (after email / phone / link / address) with zero new sync machinery.

Company-date rule: **`master_date` rows are dated pointers, never a second source of truth.** A `funding_round` row carries the date + summary metadata (round type, amount, `origin_ref` → `fundable_deals` id / `Funding_Round` node uuid); the deal itself stays canonical in `fundable_deals` and the graph. Company rows are derived-recomputable (the deterministic pattern, enrichment-side): wipe + regenerate from the domain tables is always safe.

Write semantics (enforced by the choke-point writer, not by each pipeline):
- **Singleton descriptors** (birthday, death — `singleton = true` on the descriptor): one row per person per descriptor; a strictly-more-complete date **upgrades the row in place** (id + joins preserved — the address completeness policy applied to dates); conflicting equally-complete values keep the first and log the conflict.
- **Repeatable descriptors** (anniversary): date parts join the natural key; multiple rows allowed.

### `enrichment_date` — NEW · Orbiter Backend

The projection mirror of `master_date`, same shape minus provenance internals. Read-only from the app's perspective; written only by the Universe push. This is what gives the frontend birthdays without ever touching Universe at render time.

### `contacts_date` — EVOLVE `important_date` (242) · Orbiter Backend only

One user-scoped table for all three private classes, discriminated by origin. Keeps table 242's id space and the merge fold-in it already has.

| Column | Change | Notes |
|---|---|---|
| `my_person_id` | keep | → `contacts` (128) |
| `date_descriptor_id` | keep | now also accepts system + custom descriptors |
| `year` / `month` / `day` | **add** | partial-date parts, same semantics as `master_date` |
| `date` | keep | derived when parts complete |
| `annual_recurring` | keep | default from the descriptor's `default_recurring` |
| `note`, `reminder` | keep | reminder = lead-days, already designed |
| `origin` | **add** | enum `user` / `system` / `memory` / `import` — determines mutability |
| `origin_ref` | **add** | json: `memory_id` (mem0 idempotency), import row id, activity ids (recompute provenance) |
| `natural_key` | **add** | same convention as `master_date` |

Indexes: **unique `(my_person_id, natural_key)`** — the fold-in (#12951) already assumes this natural key; the index makes it real.

Mutability by origin: `user` rows fully editable; `system` rows never user-editable (at most hidden) and always safe to wipe + regenerate; `memory` rows editable but their `origin_ref.memory_id` guards against duplicate re-promotion when mem0 wipes + re-processes on merge.

### `date_descriptor` (243) — EVOLVE · BOTH

The shared vocabulary. Built-ins are seeded identically in both environments (a static, versioned seed — not synced at runtime); user-custom rows exist **only** on Orbiter Backend.

| Column | Change | Notes |
|---|---|---|
| `descriptor` | keep | display name, search-indexed |
| `slug` | **add** | stable machine key (`birthday`, `connected_on_linkedin`), unique — natural keys reference this, never the display name |
| `kind` | **add** | enum `global` / `system` / `user_custom` — system descriptors are invisible to the custom-descriptor UI |
| `singleton` | **add** | bool — one-per-person(-per-user) descriptors (birthday, death, connected-on) |
| `default_recurring` | **add** | bool — birthdays/anniversaries yes, graduation no |
| `user_id` | keep | set only on `user_custom` rows (Backend only) |

Seed set (SIGNED OFF 2026-07-16) — person: `birthday`, `death` (global/enriched); `wedding_anniversary`, `work_anniversary`, `first_met` (global — `first_met` is the user-asserted "when we met", the landing slot for How I Know promotions, distinct from system-derived `first_meeting`); `connected_on_linkedin`, `connected_on_orbiter`, `first_email`, `first_meeting`, `contact_added` (system). Company: `founded` (singleton), `funding_round` (repeatable), `ipo`, `acquisition` (singleton), `product_announcement` (repeatable, future — lands as a descriptor row when news enrichment exists, zero schema change).

**Seeding is a function, not hand-entered rows** — `dates/seed-date-descriptors`, an idempotent upsert-by-slug that owns the built-in vocabulary in code. It runs per environment and per data source (live / staging / sandbox), re-runs safely to add new built-ins, and re-runs at Universe split time. Row IDs differ across environments — which is why natural keys and writer lookups go through `slug`, never `date_descriptor_id`. The choke-point writers hard-fail on an unknown slug (a typo'd descriptor should never silently drop a date).

### Disposition of today's tables

| Table | Fate |
|---|---|
| `birthday_join` (406) | Migrate its 20 rows into `master_date` (`slug=birthday`, birthplace → `metadata`), repoint the two IMDB writers, then retire. Trivial volume. |
| `important_date` (242) | Evolves in place into `contacts_date` (columns added, nothing dropped). |
| `contacts.birthdate` (128) | Drop the column — zero rows, redundant third home. |
| `date_descriptor` (243) | Evolves in place as above. |

- **Merge (`fold-master-persons`)** — add `master_date` as a child table: repoint to survivor, dedupe on natural key, keep the more complete row. Replaces the current `birthday_join` handling in the [Merge](/guides/open-work/merge) spec.
- **Projection** — register `master_date` as the fifth facet table in the Universe → Backend push (upsert on `(node_uuid, natural_key)`, the contract the sync design already defines).

### Spec — perfect IMDB birthday capture

The IMDB path is the reference implementation for enrichment date capture — every other pipeline inherits its rules via the choke point. Five layers, each closing a verified gap:

**1. Extraction contract (prompt v5, both functions).** The birth fields change from a single ISO string to explicit parts, so the LLM captures exactly what the page shows instead of dropping partials:

```json
"birth_date": { "year": 1967, "month": 2, "day": 20 },   // any field null if not shown
"birth_place": "string or null",
"death_date": { "year": null, "month": null, "day": null },  // NEW — IMDB shows it; feeds the death descriptor
"death_place": "string or null"                              // NEW
```

Prompt rules: extract only what is explicitly on the page — never infer a year from an age, never fill a placeholder. Each function's embedded prompt gets its `// vN` header bumped and re-synced to [LLM System Prompts](/guides/enrichment/waterfall/llm-system-prompts) per the sync convention. Closes: *partial dates dropped by the full-ISO-or-nothing contract* (year-only IMDB pages currently return null).

**2. Deterministic validation (before any write).** `parse-date-parts` revalidates the LLM output — never trust extraction for data integrity:

- month ∈ 1–12; day valid for that month (leap-aware); day-without-month rejected
- year sanity window (1800 – current year); future dates rejected
- birth < death when both present
- On failure: write **nothing**, log the raw values to `log_crash` (raw JSON is already persisted, so a fixed validator can re-land it without a re-scrape)

Closes: *unvalidated LLM output written directly to the table*.

**3. Write semantics (all in `upsert-master-date`, not in the IMDB functions).** `birthday` and `death` are singleton descriptors — natural key is the slug, one row per person:

| Incoming vs existing | Action |
|---|---|
| No existing row | Insert (parts + derived `date` + `metadata.birthplace_location` + `data_source_id`=IMDb) |
| Strictly more complete, consistent (existing `1967` → incoming `1967-02-20`) | **Upgrade in place** — id + joins preserved |
| Same or less complete, consistent | No-op (idempotent re-runs) |
| **Contradiction** (different values at same completeness) | Keep existing, **log a conflict** with both values + sources for review — never silently overwrite |
| `metadata` fields (birthplace) | Non-empty incoming wins; empty incoming never overwrites — kills the #10509 null-birthplace quirk by construction |

Backstopped by the unique `(master_person_id, natural_key)` index: a concurrent duplicate insert (the #12880 / #10509 / #13121-cross-enrich race) fails cleanly and retries as an upgrade-or-noop. Closes: *first-write-wins immutability, the race, the null overwrite, missing provenance*.

**4. Identical entry points.** #12880 Phase 3 and #10509 make the same one-line call with the same payload — no divergent behavior to reason about. #10509's 30-day re-scrape becomes useful for dates (it can now complete a partial or surface a conflict), and #13121's IMDb cross-enrich inherits everything for free.

**5. Backfill + QA.**

- Migrate the 20 `birthday_join` rows (`source`=IMDb backfill).
- One-shot re-parse: every `imdb_person.json` with a `birth_date` but no `master_date` row (Phase-3 crashes, pre-dating rows) — lands from raw JSON, zero re-scrapes.
- Coverage invariant for the QA agent: **every `imdb_person` with birth data in raw JSON has a `master_date` row** — plus a periodic review of the conflict log.

Also in scope per the 2026-07-16 decision: the **current** marriage (`is_current = true`) lands as a `wedding_anniversary` row via the same call — spouse name in `metadata`; ended marriages stay raw-JSON only.

### Dates → mem0: the contact gate

The reverse direction of promotion: structured dates become mem0 memories so AI chats (meeting prep, how-I-know) know them — but **only for entities that exist in at least one user's contacts**. Enrichment waterfalls mint people and companies at scale; none of them get mem0 memories until a user actually knows them.

Date memories land in the two tiers of the [mem0 two-tier model](/guides/open-work/mem0/two-tier-architecture) (decided 2026-07-16, superseding an earlier per-contact fan-out sketch):

| Date source | mem0 tier | Metadata key | Writes per event |
|---|---|---|---|
| `contacts_date` (user / system / import) | **Contact tier** — per-user, subjective | `contacts_id` | 1 — that user's contact scope |
| `enrichment_date` (person or company) | **Shared entity tier** — objective, everyone's | `master_person_id` / `master_company_id` | **1 — total**, regardless of user count |

An enrichment date is written **once** into the entity's shared thread — e.g. frame.io's seed-round date becomes one shared memory keyed `master_company_id`. When Mark adds frame.io as a contact, his contact-scoped reads *link* to that shared thread (the third search branch on the entity metadata key, per the two-tier doc §3) — nothing is copied per user. **There is no fan-out**; the mega-company cost question dissolves by design.

**The gate — when does the shared thread get date memories?** Only when the entity is in ≥ 1 user's orbit:

- **Person:** ≥ 1 `contacts` row references the master person.
- **Company:** ≥ 1 of the user's contacts is currently employed there, OR a user explicitly tracks the company (collections / follows) — decided 2026-07-16.

Two triggers make it ordering-proof:

1. **Date-added:** `contacts_date` rows write their contact-tier memory immediately. Enrichment dates (landing Backend-side via `enrichment_date`) write the shared-tier memory **only if** the entity already has orbit; zero orbit = skip — waterfall-minted entities generate nothing. The structured tables keep everything either way.
2. **First-orbit linkage:** when an entity *enters* someone's orbit (CREATE-CONTACTS for persons; track/first-employed-contact for companies), backfill its shared thread once from `enrichment_date`. A person enriched long before anyone knows them gets their shared date memories the moment they matter.

Rules that keep it clean:

- **Loop guard:** `origin = memory` rows are NEVER echoed back into mem0 — they *came from* a memory (promotion, direction 2). Only `user` / `system` / `import` / enrichment rows sync in.
- **Derived, deterministic, regenerable:** date memories are rendered from the row by a per-descriptor template, and carry the `natural_key` + row provenance in memory metadata — so re-syncs update instead of duplicate, and the merge rule (mem0 wipe + re-process) regenerates them identically from the structured tables. The tables are the source of truth; these memories are a projection of them.
- **Tier discipline:** all mem0 writes are backend-tier, triggered off `enrichment_date` (never `master_date`) — Universe stays mem0-ignorant.
- **Dependency:** the shared entity tier is itself proposed-not-built (two-tier doc §3, plus a `master_company_id` key it doesn't yet name) — item 15 lands with or after that work.

## Why this scales

1. **Generality without schema churn** — any new date kind is a descriptor row + optional metadata keys, never a new table or column.
2. **Idempotent writers everywhere** — natural-key upserts with unique indexes; re-running any pipeline, import, or recompute is always safe.
3. **Partial dates first-class** — the parts columns handle every real-world arrival (vCard omit-year, "June 14th", year-only IMDB) without lossy coercion.
4. **Clean environment split** — enrichment truth in Universe, private context in Orbiter Backend, one projection contract already designed for four other tables.
5. **Derived tiers stay derived** — system dates and mem0 both follow wipe-and-regenerate, so merges never strand state.
6. **One conflict rule, one place** — precedence lives in the read path, not scattered across writers.

## Decisions

- **2026-07-16 — Fundable `founded_date` coercion rules.** Jan-1 dates land year-only; day-1 dates land month-only (UMG `1934-09-01` → September 1934); zero timestamps are sentinels and write nothing. Applies to `fundable_organizations.founded_date` ONLY — `fundable_deals.date` keeps full-date trust. Rare real Jan-1/first-of-month founding dates lose a level of precision; acceptable for `founded`.
- **2026-07-16 — Captured dates: MATERIALIZE with recompute.** Deterministic dates are stored as `origin=system` rows in `contacts_date` (not computed at read) — calendar, reminders, and mem0 sync read one uniform source; recompute hooks (contact create, merge, periodic sweep) keep them honest; wipe + regenerate always safe.
- **2026-07-16 — mem0 promotion is CONFIRM-FIRST.** Conversation-extracted dates (How I Know, chats, email memories) are proposed, not auto-created — the user approves before a row or reminder exists. Deterministic paths (enrichment, system, imports) never require confirmation.
- **2026-07-16 — Descriptor seed set SIGNED OFF**, including two additions: `connected_on_orbiter` (system) and `first_met` (global, user-asserted — the How I Know landing slot). Full set on the `date_descriptor` spec.
- **2026-07-16 — IMDB spouse dates: land the CURRENT marriage only.** When `is_current = true`, the marriage date becomes a `wedding_anniversary` row in `master_date` (spouse name in `metadata`) — the relationship-relevant anniversary. Ended marriages stay in `imdb_person.json` raw only.
- **2026-07-16 — Promotion is a TEMPLATE, dates ship first.** The mem0 → structured machinery (`origin=memory`, `origin_ref.memory_id` idempotency, confirmation flow) is designed reusable; dates are deliberately the first and only implementation. Job changes / relocations ride the same rails later.
- **2026-07-16 — Read-side audit DONE: `birthday_join` has NO readers in live v1.** Verified via dev-branch full-code grep + live reads of every 2026 artifact + V2 group inventory (no GET /birthdays exists). All live references are writers or maintenance (IMDB ×2, V2 POST/PUT/DELETE, merge fold, QA clear). Legacy dev-only writers (`unpack_name_v_2`, `add_master_person_imdb`, `test/birthday`) are deleted from v1. Item 5's retire step is unblocked — greenfield.
- **2026-07-16 — Dates flow INTO mem0, orbit-gated, via the TWO-TIER model (no fan-out).** User-specific dates (`contacts_date`) write to that user's contact-tier scope (`contacts_id`). Enrichment dates write **once** into the entity's SHARED tier (`master_person_id` / `master_company_id` metadata) — e.g. frame.io's seed round is one shared memory; users who add frame.io as a contact link to it at read time (the two-tier doc's third search branch). Waterfall-minted entities get nothing until they enter ≥ 1 user's orbit (backfill on first linkage makes it ordering-proof); loop guard: `origin=memory` rows never echo back. This supersedes the earlier per-contact fan-out sketch — the mega-company fan-out question dissolves by design.
- **2026-07-16 — Company orbit rule.** A company is in a user's orbit iff ≥ 1 of the user's contacts is currently employed there, OR the user explicitly tracks it (collections / follows).
- **2026-07-16 — User-added company dates: defer, design for it.** `contacts_date` ships person-contact-scoped; its natural-key builder and unique index are written so a nullable company reference can be added later without breaking anything.
- **2026-07-16 — Valuations: analytical only.** `fundable_valuations` / `latest_valuation_date` never become important-date rows — a valuation mark is not a relationship moment, and deal-tied valuations are already represented by their `funding_round` row. Company important dates = founded, funding rounds, IPO, acquisition (+ product announcements later).
- **2026-07-16 — Company dates are IN scope.** Funding-round dates and founded dates from day one; product announcements and similar later. Mechanism: ONE polymorphic `master_date` (the `master_address` two-FK pattern), NOT a separate company-dates table — separate tables would duplicate the descriptor/natural-key/choke-point/projection machinery and re-merge anyway when Universe keys on `node_uuid`. Company rows are dated pointers into canonical domain data (`fundable_deals`, the graph), derived and recomputable.
