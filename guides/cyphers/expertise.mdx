## Current Migration Decision (2026-06-18)

This is the implementation runbook. The migration is worth doing because it fixes the resolver's core calibration bug: live matching compares incoming `name` embeddings against node vectors built from different text. The new path keeps the comparison symmetric and isolates expertise matching from graph-wide node search.

### Locked Decisions

| Area | Decision | Reason |
|---|---|---|
| Scope | Migrate expertise matching only; keep graph-wide `name_embedding` untouched. | Avoids changing 17k+ non-expertise node search behavior in the same release. |
| Match vector | Use dedicated Gemini 3072-dim `expert_embeddings`. | A dedicated property keeps KNN structurally scoped to expertise nodes. |
| Representation | Node side embeds `"{name}: {description}"`; query side embeds `"{expertise_area}: {generated_blurb}"`. | Symmetric definitional text separates synonyms from sibling concepts better than name-only. |
| Blurb generator | Use `google/gemini-3.1-flash-lite` for generated definitional blurbs and new-node descriptions in this migration. | This is the create-domain winner from the golden-set model assessment; do not mix in DeepSeek without recalibrating thresholds. |
| Thresholds | Auto-MATCH `< 0.15`; adjudicate `0.15–0.25`; auto-CREATE `> 0.25`. | Matches the validated Gemini `name + definitional text` scale. |
| Edge weight | Use `min(round(10 + score * 160), 50)` on the Gemini score scale. | Keeps `0.25` at the max weak weight of `50` while preserving stronger matches below the guard band. |
| Rollback | Keep OpenAI `expertise_embedding` writes and index live through cutover. | Reverting remains a one-line KNN target change. |

### Release Sequence

1. **Preflight.** Re-count live expertise nodes, confirm #13129 has provisioned `expert_embeddings`, verify the generator used for calibration is the same generator used in the resolver, and keep `name_embedding` untouched.
2. **Backfill.** Populate `expert_embeddings` from existing `name + description` with an LLM-free batched task. Flag nodes missing descriptions instead of silently treating name-only vectors as equivalent.
3. **Shadow compare.** Before flipping live reads, run a representative batch through both old and new matchers and log old decision, new decision, score, tier, adjudication result, and final action.
4. **Cut over writes and reads.** Port #12926 and #12746 to the new resolver flow, update #12663 for vector writes, deprecate #12668 rather than porting it unless a live caller still needs it, and wire the OpenRouter out-of-credits guard into #12666/#12926.
5. **Monitor.** Watch near-duplicate creation rate, adjudication volume, created-node descriptions, mirror parity, and any known-distinct pairs that drift into MATCH.
6. **Merge historical duplicates.** Only after new writes stop creating fresh duplicates, run the historical merge as its own dry-run-reviewed project.

## Improvement Plan

This is an exploratory refactor plan only. The locked vector-migration choices live in **Current Migration Decision** above; if this section conflicts with that runbook or the golden-set model assessment, treat this section as historical context. Do not change live Xano functions, model configs, Cypher blocks, or build new functions from this section without a separate implementation pass.

The goal is to make person expertise extraction more powerful by rebuilding the input context, making refresh behavior explicitly idempotent from the person-enrich pipeline, and using different OpenRouter models for different phases instead of forcing one model to do every job. Model candidates below were checked against the live OpenRouter model API on 2026-06-17; re-run the model/pricing audit before implementation.

### Six Review Loops

1. **Refresh semantics loop** — Treat every person-enrich expertise run as a full replacement pass. Before any new expertise signals are resolved, delete all existing `(Person)-[:HAS_EXPERTISE]->(Expertise)` edges for that person so the graph reflects the latest evidence instead of accreting stale skills.
2. **Mirror consistency loop** — Because `resolve-person-expertise-v2` dual-writes to `person_has_expertise` (table 711), the cleanup plan should also clear that person's mirror rows before re-resolution. Otherwise the graph can be clean while the relational mirror still exposes stale expertise.
3. **Context quality loop** — Replace the current broad text blob with a purpose-built one-hop context package: all non-embedding properties from the `Person` graph node, all one-hop edge descriptions only, and `master_person.deep_bio` when available.
4. **Prompt contract loop** — Keep extraction strict JSON, but expand the contract to distinguish direct evidence, graph-inferred evidence, deep-bio evidence, and weak inferred evidence. Add source fields only if they survive validation and improve QA.
5. **Model ladder loop** — Split the work by phase. Use cheaper long-context models for context normalization and routine extraction, and reserve the best reasoning models for adjudication, ambiguity, or high-value records. Kimi, DeepSeek, and Qwen should be first-class benchmark candidates alongside OpenAI, Gemini, and Claude.
6. **Evaluation loop** — Add a golden set before shipping: compare current vs proposed output for precision, duplicate rate, stale-edge removal, created-node rate, graph/mirror parity, token cost, and manual spot-check quality.

### Refactor Shape

1. **Build a separate A/B function first**

   Do not refactor #12666 in place. Plan a separate sandbox-first function, tentatively `mvp/expertise/llm-identify-person-expertise-ab`, with the same person inputs plus `run_label`, `model_profile`, and `mode`.

   Suggested modes:

   | Mode | Behavior | Purpose |
   |------|----------|---------|
   | `dry_run` | Build context, call models, validate JSON, and return proposed signals without graph/table writes. | Compare old vs new extraction on the same sandbox person safely. |
   | `shadow_write` | Write proposed signals to an A/B log table or temporary candidate edge type keyed by `run_label`; do not touch canonical `HAS_EXPERTISE`. | Inspect results across batches without corrupting canonical graph state. |
   | `commit_refresh` | Delete canonical expertise edges/mirror rows, then call #12926 to write the accepted replacement set. | Promotion path after the A/B winner is selected. |

   The first benchmark should run current #12666 and the new A/B function against the same sandbox `master_person` / `Person` node, then diff signal count, evidence quality, duplicate rate, created-node risk, model cost, and runtime.

2. **Centralize the pre-run cleanup**

   Once the A/B function graduates, ensure every person-enrich path that invokes the canonical expertise extractor runs cleanup first. The live person orchestrator is `mvp/enrich/enrich-master-person` (#13040); legacy `run-base-person-enrich` (#12554) and IMDB Phase 4b callers should either route through the same canonical entry point or be audited as compatibility callers.

   Target graph cleanup shape:

   ```cypher
   MATCH (p:Entity:Person {uuid: $personUuid})-[r:HAS_EXPERTISE]->()
   DELETE r
   RETURN count(r) AS deleted_count
   ```

   Implementation note: keep the canonical direction as `Person -> Expertise`, but run a one-time audit for accidental reverse `HAS_EXPERTISE` edges before deciding whether the cleanup should delete both directions.

3. **Clear the relational mirror in the same refresh**

   Since #12926 writes `person_has_expertise`, the refactor should delete that person's table-711 rows before the resolver loop creates replacements. Confirm the exact key field during implementation (`master_person_id`, `person_node_uuid`, or equivalent) and log both graph deleted count and mirror deleted count.

4. **Build a one-hop graph context package**

   Replace the prompt input with a deterministic context builder. The graph query should start from the person node, gather only one-hop relationship descriptions, and avoid neighbor-node properties entirely.

   Target read shape:

   ```cypher
   MATCH (p:Entity:Person {uuid: $personUuid})
   OPTIONAL MATCH (p)-[r]-(n:Entity)
   RETURN properties(p) AS person_properties,
          collect(r.description) AS one_hop_edge_descriptions
   ```

   Post-query shaping should strip `embeddings`, `name_embedding`, `expertise_embedding`, and any future `*_embedding` vector fields from `person_properties`; drop blank/null edge descriptions; dedupe descriptions; and cap the final list by a deterministic token/character budget. The prompt context should contain relationship descriptions only, not neighbor node params or raw edge metadata.

5. **Merge in `master_person.deep_bio`**

   The one-hop Cypher cannot read Xano table data, so the context builder should fetch the `master_person` row separately and append `deep_bio` when present. Treat `deep_bio` as a high-signal narrative source, but label it separately from graph edge descriptions so QA can tell whether an expertise came from the graph or the deep-bio record.

6. **Use different OpenRouter models by phase**

   Proposed candidates to benchmark:

   | Phase | Default candidate | Escalation candidate | Notes |
   |-------|-------------------|----------------------|-------|
   | Context normalization / over-budget compression | `google/gemini-3.1-flash-lite` or `qwen/qwen3.7-plus` | `openai/gpt-5.4-nano` or `deepseek/deepseek-v4-flash` | Only needed if the one-hop package exceeds the extraction budget. Use structured outputs and keep reasoning minimal/none unless compression quality fails. |
   | Main expertise extraction | `google/gemini-3.1-flash-lite`, `openai/gpt-5-mini`, or `qwen/qwen3.7-plus` | `openai/gpt-5.4-mini`, `qwen/qwen3.7-max`, or `deepseek/deepseek-v4-pro` | Benchmark against current `google/gemini-3.1-flash-lite` (the live model in #12666, confirmed 2026-06-18). It keeps 1M context at the lowest viable cost on the live OpenRouter list. |
   | Ambiguity adjudication / reasoning pass | none by default | `anthropic/claude-sonnet-4.6`, `google/gemini-3.1-pro-preview`, `moonshotai/kimi-k2.7-code`, `deepseek/deepseek-v4-pro`, or `qwen/qwen3.7-max` | Run only when extraction returns low confidence, too many inferred signals, schema repair, or unusually high created-node risk. Use the strongest reasoning model that wins the golden-set eval, not the most familiar vendor. Do not attach `openrouter:web_search` for private graph context. |
   | New expertise node description | `google/gemini-3.1-flash-lite` | `openai/gpt-5-mini` | Keep short, generic, and taxonomy-safe. This is not the same as person-specific edge copy. |
   | HAS_EXPERTISE edge description | `google/gemini-3.1-flash-lite` or `openai/gpt-5-nano` | none unless quality fails | One sentence per edge should stay on a cheap model with a strict word cap. |

   Reasoning-model guardrail: when testing Kimi, DeepSeek, Qwen thinking models, GPT reasoning models, or Gemini Pro, set an explicit completion budget and validate that `choices.0.message.content` contains parseable JSON. If a model spends the budget on hidden reasoning and returns empty `content`, retry with a lower reasoning effort, higher completion cap, or a non-reasoning fallback before marking the person as having no expertise.

7. **Tighten extraction output and resolver handoff**

   Keep `expertise_area`, `evidence`, `signal_type`, `depth`, `recency`, and `confidence`, but consider adding `evidence_source` (`person_property`, `edge_description`, `deep_bio`) and `source_span` for auditability. Validate schema before calling #12926. Reject duplicate expertise at the prompt boundary before vector search so the resolver does less merge cleanup.

8. **Define rollout gates**

   Before replacing the current path, run the old and new extractors side-by-side on a representative set of sandbox people: sparse profiles, rich C-suite profiles, creators, investors, film/TV profiles, and records with deep_bio. Promote only if the new path improves accepted expertise quality without materially increasing duplicate nodes, mirror drift, or average cost per person.

### 2. Extraction context A/B — relational vs graph + deep_bio

A dry-run A/B compared the current extraction context (relational YAML from `mvp/context/full-person-context` #12667) against a new one-hop graph context (`MATCH (p:Person)-[r]-() RETURN collect(r.description)`) + joined `master_person.deep_bio`, adding an `evidence_source` field per signal.

### 4. Auto-creation node-bloat diagnosis

The resolver CREATE branch (#12926) has spawned **2,265 of 2,435 `SubDomainExpertise` nodes as `llm_identified`** (93%), with heavy near-duplication. Three root causes:

**🔴 Embedding asymmetry (calibration bug).** The resolver matches `embed(name)` against stored vectors built from *different text* — `embed(name + ": " + description)` for llm_identified nodes, `embed(description)` for synced nodes. So even an exact-name match never reaches ~0 distance:

| pair | name-vs-name | name-vs-stored (current) |
|---|---|---|
| AI strategy ↔ AI strategy (self) | **0.0000** | 0.3196 |
| product manager ↔ Product Management | 0.2129 | 0.3414 |
| AI-native worldbuilding ↔ AI-native platforms | 0.3271 | 0.4613 |
| AI agents for accounting ↔ AI agents | 0.2407 | 0.4194 |
| Product Management ↔ venture capital (distinct) | **0.7797** | — |

Matching on **name-only embeddings** collapses self-match to 0, pulls near-dups well under threshold, and keeps distinct concepts far apart.

**🟠 Creation-order bloat.** Many dups would match *today* but were created before their parent existed (`AI agents for accounting` is 0.42 from `AI agents`). Needs a one-time merge pass.

**🟡 The 0.45 threshold is roughly right** — don't lower it (would miss paraphrases like `product manager`→`Product Management` at 0.34); on the name-vs-name scale it can tighten to ~0.40.

## Embedding architecture — migration target (2026-06-18)

This section holds the supporting evidence and implementation detail behind **Current Migration Decision**. It supersedes the OpenAI `match_name_embedding` migration in §5. **Not shipped; live state unchanged.**

#### Live reality (deployed today)

- Live match vector = **`expertise_embedding`** (OpenAI `text-embedding-3-small`, 1536), read by all three resolvers (#12926 / #12668 / #12746) at the **0.45** threshold.
- It is **asymmetric** — the §4 calibration bug. Nodes embed **name + description** (`embed("{name}: {description}")`); the query embeds **name only**. So an exact-name hit never reaches ~0 distance (the 0.32 self-distance in §4). Confirmed by harness #8697.
- **`expert_embeddings`** (Gemini `gemini-embedding-001`, 3072) is **provisioned but unpopulated** — #13129 v3 creates the index; **no live function writes or reads it.** Only the sandbox prototype (#8698 v2) populates it, with Gemini `embed(name)` on ~60 of ~1,329 nodes.

#### Decision: fix the asymmetry with *symmetry*

Embed the **same kind of representation on both sides: `name + definitional text`.** **LOCKED 2026-06-18** by the sandbox calibration below — name-only is rejected, and no separate `summary` property is part of this migration.

**Calibration results** (Gemini `gemini-embedding-001` 3072, real sandbox descriptions, cosine distance — query representation vs node `name+description`):

| query representation | SYNONYM (want low) | SIBLING | DISTINCT | synonym→sibling gap |
|---|---|---|---|---|
| name only (current live, asymmetric) | 0.260 | 0.292 | 0.474 | +0.032 ❌ |
| **name + generated definitional text** (locked) | **0.104** | 0.246 | 0.435 | **+0.142** ✅ (4.4×) |

Concrete: incoming `"artificial intelligence"` vs existing node `"AI"` = **0.313** today (spawns a duplicate) → **0.038** locked (clean match). (Fully-symmetric name+desc-vs-name+desc hit synonym **0.016**; Gemini name-only gave only +0.118 — half the separation, and on OpenAI name-only actually *inverts*: synonym 0.46 > sibling 0.41.)

**End-to-end pipeline validation (sandbox, 2026-06-18).** Ran the real flow (identify #8694 → blurb #8702 → KNN #8710) on 4 fixtures (Francesco/fusion, Allan/volcanology, Christophe/perfumery, Kyle/AI-venture) against all **1,464 vectored nodes**: **24 MATCH / 1 ADJUDICATE / 2 CREATE**. True matches landed 0.02–0.11; `"generative AI"` → `"Generative AI & LLMs"` (0.211) correctly hit **ADJUDICATE**; `"perfumery ethics"` (0.289) correctly CREATEd. One miss: `"3D animation"` → `"3D Animation & Character Animation"` scored **0.232** — a near-dup that fell just past a 0.22 cutoff, so **CREATE was widened to > 0.25** (adjudicate band 0.15–0.25), and it confirms the **dual-signal name tripwire** (near-identical names) is worth shipping. Also surfaced **pre-existing duplicate `chemistry` nodes** (3 of them) — concrete evidence for the keystone step-4 historical merge. Query blurbs must come from the *same generator* as node descriptions: an off-distribution ad-hoc blurb for "artificial intelligence" scored 0.17 vs 0.04 for the #8702-generated one.

**Locked mechanics — no new `summary` property:**
1. **Node side (backfill, LLM-free):** embed **`"{name}: {description}"`** using the node's *existing* stored `description` (Gemini 3072) → `expert_embeddings`. Pure mechanical re-embedding of data we already have — no per-node LLM call.
2. **Query side:** the incoming `expertise_area` has no description (extraction #12666 emits only name + a ≤100-char person-specific `evidence`), so **generate a definitional blurb from the name** with the resolver's chat model (`qwen/qwen3.7-plus` in live #12926 as of 2026-07-03; `google/gemini-3.1-flash-lite` in the original calibration), and embed **`"{expertise_area}: {blurb}"`**. The blurb does **double duty**: discard/cache it on MATCH; on CREATE it *becomes the new node's `description`* (zero wasted generation). Do not swap in another generator without re-running calibration or smoke-testing borderline cases, because thresholds depend on the generator distribution.
3. **Tiered + cached:** Tier 0 exact/normalized-name → instant MATCH (no LLM, no embed); Tier 1 generate-blurb + KNN. Cache `name → blurb → vector` so repeats are free and deterministic. Tier 0 is required, not a nice-to-have.

## Taxonomy completion — orphan SubDomain → DomainExpertise (2026-06-19)

Domain-level search ("show me experts in AI") rolls up as a free 2-hop traversal: `(Person)-[:HAS_EXPERTISE]->(SubDomainExpertise)<-[:HAS_SUBDOMAIN]-(DomainExpertise)`. That traversal returned almost nobody because **2,265 of 2,435 `SubDomainExpertise` nodes were orphans** (no parent `DomainExpertise`) — all `source:llm_identified`, created by the resolver CREATE branch, which (until this work) never wrote `HAS_SUBDOMAIN`. Only the sync (#12663), driven by the relational `domain_expertise_id` FK, ever created parents (170 edges).

**Decision (settled — not relitigated):** keep person→subdomain edges (specific, best for GraphRAG); do **not** add person→domain edges (bloats the person ~2× and pollutes RAG with coarse domain descriptions). Instead **complete the SUBDOMAIN→DOMAIN taxonomy** so the domain rollup is a free 2-hop and person edges stay specific.

**Curated-set guard:** the resolver attaches a subdomain to an **existing** `DomainExpertise` only — it **never auto-creates a domain** (the 119-domain set is curated). No-fit subdomains are flagged `parent_unresolved=true` for human review, not forced into a wrong parent.

### Parent-domain resolver — `mvp/expertise/resolve-parent-domain` (#13141)

```text
mvp/expertise/resolve-parent-domain — 13141
```

Given a subdomain (`name` + `description`), picks the best parent from the curated `DomainExpertise` set via a single `google/gemini-3.1-flash-lite` call (`reasoning:{enabled:false}`, the full domain list fits one prompt). The 119 domains are fetched live and presented as a numbered list; the model returns a **JSON index** (`{"index": N}`, 0 = NONE), mapped deterministically back to a uuid (a digit-scan parse + index→uuid map — robust to formatting, regex-free per the escaping traps). KNN over `expert_embeddings` was rejected as the primary mechanism: with 2,435 subdomains vs 119 domains, KNN top-K is crowded out by other subdomains, whereas the full curated list fits one cheap prompt.

**Calibration (two adversarial judge-panel audits, 73 + 71 sampled assignments each judged by an independent strict judge + a second adjudicator on disputes):**

| prompt | precision | NONE rate | failure mode |
|---|---|---|---|
| v1.0 (assign-by-default) | 79.5% | 16% | **"AI Strategy & Transformation" magnet** — generic leadership / M&A / startup terms forced in |
| v1.1 (strong NONE-preference) | — | **49%** ❌ | over-corrected — abstained on `Python`, `communications`, `education` |
| **v1.2 (assign-default + scope-guard vetoes + bounded NONE)** | **83.1%** | ~39% | residual = fine-grained sibling-domain calls (judges themselves split) |

v1.2 keeps the magnet/qualifier fixes as **targeted vetoes** (AI Strategy is AI-only; AI-native systems → `AI Infrastructure` not `Software Engineering`; DB perf → `Data Science`, mobile ops → `Mobile Development` not `DevOps`; finance qualifiers `institutions`/`advisory`/`investment` are load-bearing) while defaulting to *assign* and reserving NONE for genuinely uncovered industries (agriculture, retail, hospitality, generic business, academic humanities, grassroots advocacy). The 14/15 known-wrong v1.0 cases flipped to the judge-preferred outcome; the dangerous **systematic magnet is eliminated** (the remaining errors are scattered sibling-domain disagreements at the noise floor of taxonomy ambiguity). Erring toward NONE is the safe error: a wrong parent permanently pollutes rollup, while a NONE is flagged and recoverable.

### Forward-fix helper — `mvp/expertise/attach-subdomain-parent` (#13142)

```text
mvp/expertise/attach-subdomain-parent — 13142
```

Called from the CREATE branch of #12926 (person) and #12746 (company), right after a new `SubDomainExpertise` row is written. Resolves the parent via #13141 and writes `HAS_SUBDOMAIN {weight:100}`, or flags `parent_unresolved` on NONE. Same edge shape as the sync (#12663). Keeps every newly-created subdomain parented going forward, so the orphan pool does not regrow. (Each resolver gets only a single added `function.run` line — minimal, DRY with the drainer.)

```cypher
// Version v1 — attach to an EXISTING domain only (never auto-creates)
MATCH (d:Entity:DomainExpertise {uuid: "{{$var.du}}"})
MATCH (s:Entity:SubDomainExpertise {uuid: "{{$input.sub_node_uuid}}"})
MERGE (d)-[r:HAS_SUBDOMAIN]->(s)
ON CREATE SET r.created_at = timestamp(), r.updated_at = timestamp(), r.weight = 100
RETURN d.uuid AS domain_uuid
```

```cypher
// NONE → flag for review (skipped by the drainer on later runs; converges to 0)
MATCH (s:Entity:SubDomainExpertise {uuid: "{{$input.sub_node_uuid}}"})
SET s.parent_unresolved = true, s.parent_unresolved_at = timestamp()
RETURN s.uuid AS uuid
```

### Orphan backfill drainer — `tool/expertise-backfill-parents` (#8718, API group 42)

```text
tool/expertise-backfill-parents — 8718
```

Idempotent drainer (clone of the #8709/#8716 pattern). `action=count` reports total / with_parent / orphans / flagged / processable; `action=backfill` processes the next `limit` *processable* orphans (orphan AND not yet flagged), resolving + writing `HAS_SUBDOMAIN` or flagging NONE; `action=reset_flags` clears `parent_unresolved` to re-attempt NONE cases after resolver/taxonomy improvements. NONE-flagging is what lets the drainer **converge to 0** instead of looping forever on the unresolvable tail. Scoped by `X-Data-Source`.

```cypher
// fetch next batch of processable orphans
MATCH (s:Entity:SubDomainExpertise)
WHERE (NOT (:DomainExpertise)-[:HAS_SUBDOMAIN]->(s)) AND s.parent_unresolved IS NULL
RETURN s.uuid AS uuid, s.name AS name, s.description AS description
LIMIT {{$var.lim}}
```

### MERGE HAS_EXPERTISE Edge (Matched)

When an existing expertise node matches above the similarity threshold:

```cypher
// Version v5 — v2 adds r.weight
MATCH (p:Entity:Person {uuid: "{{$input.person_node_uuid}}"})
MATCH (e:Entity {uuid: "{{$var.bestUuid}}"})
MERGE (p)-[r:HAS_EXPERTISE]->(e)
ON CREATE SET
  r.confidence = {{$var.item.confidence}},
  r.evidence = "{{$var.cleanEvidence}}",
  r.identified_as = "{{$var.cleanArea}}",
  r.signal_type = "{{$var.item.signal_type}}",
  r.depth = "{{$var.item.depth}}",
  r.recency = "{{$var.item.recency}}",
  r.match_score = {{$var.bestScore}},
  r.weight = {{$var.edgeWeight}},
  r.match_type = "matched"
ON MATCH SET
  r.evidence = r.evidence + " | " + "{{$var.cleanEvidence}}",
  r.identified_as = r.identified_as + " | " + "{{$var.cleanArea}}",
  r.signal_type = r.signal_type + " | " + "{{$var.item.signal_type}}",
  r.confidence = CASE WHEN {{$var.item.confidence}} > r.confidence THEN {{$var.item.confidence}} ELSE r.confidence END,
  r.depth = CASE
    WHEN r.depth = "creator" OR "{{$var.item.depth}}" = "creator" THEN "creator"
    WHEN r.depth = "expert" OR "{{$var.item.depth}}" = "expert" THEN "expert"
    WHEN r.depth = "practitioner" OR "{{$var.item.depth}}" = "practitioner" THEN "practitioner"
    ELSE "familiar"
  END,
  r.recency = CASE
    WHEN r.recency = "current" OR "{{$var.item.recency}}" = "current" THEN "current"
    WHEN r.recency = "recent" OR "{{$var.item.recency}}" = "recent" THEN "recent"
    ELSE "historical"
  END,
  r.match_score = CASE WHEN {{$var.bestScore}} < r.match_score THEN {{$var.bestScore}} ELSE r.match_score END,
  r.weight = CASE WHEN {{$var.edgeWeight}} < r.weight THEN {{$var.edgeWeight}} ELSE r.weight END
RETURN p.uuid AS person_uuid, e.uuid AS expertise_uuid,
       e.name AS expertise_name
```

ON MATCH merge logic: `depth` keeps strongest (creator > expert > practitioner > familiar), `recency` keeps most current (current > recent > historical), `confidence` keeps max numeric value, `weight` keeps minimum (lower = stronger match).

`$var.edgeWeight` is pre-computed before the Cypher call: `Math.min(Math.round(10 + (bestScore * 80)), 50)`. Range: 10 (perfect match) → ~46 (threshold edge).

## mvp/expertise/backfill-has-expertise-weight — #13043

One-shot backfill that stamps `r.weight` on all existing `HAS_EXPERTISE` graph edges written before `resolve-person-expertise-v2` (#12926) added the property. Also repairs any `person_has_expertise` (table 711) rows where `weight = 0`. Safe to re-run — the `WHERE r.weight IS NULL` guard means already-patched edges are skipped.

```text
mvp/expertise/backfill-has-expertise-weight — #13043
```

### Backfill Graph Edge Weights

Single bulk Cypher — all unweighted edges in one pass, no looping:

```cypher
// Version v1
MATCH (p:Entity:Person)-[r:HAS_EXPERTISE]->(e:Entity)
WHERE r.weight IS NULL
WITH r,
  CASE
    WHEN r.match_score IS NULL THEN 50
    WHEN toInteger(round(10 + r.match_score * 80)) > 50 THEN 50
    ELSE toInteger(round(10 + r.match_score * 80))
  END AS computed_weight
SET r.weight = computed_weight
RETURN count(r) AS updated
```

Edges with `match_score IS NULL` (created branch, no vector hit) get `weight = 50`. All others derive weight from stored `match_score` using the standard formula. After the graph pass, the function loops through `person_has_expertise` rows where `weight = 0` and applies the same formula via `match_score`.
