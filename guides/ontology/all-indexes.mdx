## `tool/create-indexes` — #4669

**Location:** Xano workspace 3, function `tool/create-indexes`, id `#4669`.

**Shape:** Top-level groups, each with a `util.template_engine` block holding one `CREATE INDEX` / `CREATE VECTOR INDEX` statement, piped through `mvp/falkor/send-cypher`. The final group calls `CALL db.indexes()` and returns the full index list for confirmation.

**Idempotency:** FalkorDB's `CREATE INDEX` is idempotent — re-running is safe. Run #4669 after provisioning a new FalkorDB instance or after a graph reset.

```text
Xano workspace 3 → functions → tool/create-indexes (#4669) → Run
```

## The label-scoping rule (read before proposing indexes)

FalkorDB indexes are **per-label**: a query uses an index only when the indexed label appears in the MATCH/MERGE pattern. `MATCH (p:Person {uuid: $u})` does **not** use `Entity(uuid)` even though the node carries both labels — this is why `C_Suite(uuid)` exists, and why the 2026-07-06 additions below add per-label coverage for `Person`, `Company`, and the legacy non-`Entity` labels. When writing new Cypher, prefer including `Entity` in the pattern (`MATCH (p:Entity:Person {uuid})`) so the base-label indexes apply; when a pattern can't, make sure the label it does use is indexed on the lookup key.

## 2026-07-06 audit additions — why each one exists

All justified by the label-scoping rule \+ a census of every MATCH/MERGE pattern in the documented cyphers \+ live counts (2026-07-06: Person 8,276 · Company 13,306 · Funding\_Round 2,377 · Film\_TV\_Award 2,716 · mbid-bearing 2,934).

| Index | The query it fixes | Before | Writers/readers on this path |
| --- | --- | --- | --- |
| `Funding_Round(uuid)` | `MERGE (fr:Funding_Round {uuid: $deal_id})` | full scan of 2,377 rounds per deal write — Funding\_Round nodes carry **no**`Entity` label, so nothing covered them | #12701 deal node writer, #12702 investor edges, #12856 cascade — the path the Fundable BigQuery sync will drive at ~146k-deal scale |
| `Funding_Round(announced_date)` | `MATCH (c)-[:RAISED]->(e:Funding_Round) WHERE e.announced_date < $date` | `Entity(announced_date)` was created for exactly this filter but never applied — Funding\_Round lacks `Entity` | #12702 earlier-round / `FOLLOW_ON_INVESTED_IN` detection |
| `Person(uuid)` | `MATCH (p:Person {uuid: $u})` | scans all 8,276 Person nodes; ~33 documented label-only sites | Known/KNOWS flows, merge (`absorbed:Person`), mentor \+ introduction cyphers |
| `Person(master_person_id)` | `MERGE (p:Person {master_person_id: $id})` | scan per person create/update | #2601 add-person-node (the Person MERGE key), SAME\_AS bridge writers |
| `Company(uuid)` | `MATCH (vc:Company {uuid}) SET vc:VC_Firm` | scans 13,306 Company nodes per edge write | #12702 investor edges, #2800 work edges, funding cyphers |
| `Entity(mbid)` | `MERGE (n:Entity:Recording {mbid: $mbid})` | scan per music write; catalog cascades re-MERGE constantly | all 9 music envelope writers (#13090–#13105); one base-label index covers every music type, mirroring `Entity(imdb_url)` |
| `Film_TV(imdb_url)` | `MERGE (f:Film_TV {imdb_url: $url})` | title writer's pattern omits `Entity`, so `Entity(imdb_url)` never fires | #2320 title writer, #2334 award flow |
| `Film_TV_Award(uuid)` | `MATCH (a:Film_TV_Award {uuid})` | 2,716 legacy non-`Entity` nodes, zero coverage | #2334 award/edge writers |
| `Company(name)` | `MERGE (org:Organization:Company {name})` | volunteering MERGE pattern omits `Entity` | #4575 volunteering; also name-based label↔company matching |
| `Organization(uuid)` | `MATCH (o:Organization {uuid})` | label-only pattern in certification/honor edges | #12578 certifications family |

Cost: btree indexes only contain nodes that carry the label \+ property, so every addition is small (thousands of entries) with negligible write amplification — while each converts an O(label-size) scan into an index seek on a per-write hot path.

## FalkorDB index gotchas

- **Label scoping (the big one).** An index binds to one label; the pattern must name that label. `MATCH (p:Person {uuid})` cannot use `Entity(uuid)`. Prefer `(p:Entity:Person {…})` patterns in new Cypher.
- **Not-equal filters (`<>`) are not index-accelerated.** Prefer positive predicates (`WHERE f.source = 'declared'`).
- **`IS NULL` / `IS NOT NULL` ARE accelerated** by btree indexes — this is why indexing `valid_to` directly speeds up "currently valid" filters.
- **Composite edge indexes split** into independent single-property indexes on our FalkorDB version.
- **Vector indexes require populated embeddings** — creating against an empty label is fine; queries return nothing until vectors land.
- **Btree indexes only contain nodes that carry the property** — per-label indexes on sparse properties are cheap; the real cost of over-indexing is write amplification and planner noise, not storage.
- **Edge direction in index syntax:** always `()-[r:EDGE]-()` undirected.
