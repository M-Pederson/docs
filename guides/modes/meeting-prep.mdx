## Calendar pipeline

API `GET /calendar-events` (`#4329`) is the main data source. One call drives both the sidebar and the canvas.

<AccordionGroup>
  <Accordion title="10-step processing chain" icon="list-ol">
    1. Get user timezone from `user.settings.working_timezone`
    2. Query active calendars from `calendar` table
    3. Compute UTC range (14 days ahead)
    4. Fetch Nylas events with pagination (200/page, handles `next_cursor`)
    5. Extract unique participant emails into a Set
    6. Bulk identity resolution via `master_email` + `master_person` join
    7. Network badge lookup from `my_person` table
    8. Grant email mapping via `nylas_grant`
    9. Classify participants per event by calling function `mvp/meeting-prep/classify-participants` (`#12742`)
    10. Return enriched response with `meeting_prep` embedded
  </Accordion>
  <Accordion title="Two calendar fetch paths" icon="code-branch">
    **V2 API `GET /calendar-events` (`#4329`)** — uses function `mvp/nylas/fetch-calendar-events` (`#12705`) with full enrichment. Drives the UI.

    **Robert API `GET /calendar/events` (`#8125`)** — uses older function `nylas/nylas-calendar-get-filtered-events` (`#1978`) with no enrichment. Called internally by API `POST /meeting-prep` (`#8098`) for LLM context.

    These are independent calls.
  </Accordion>
</AccordionGroup>

<Warning>
  Function `mvp/meeting-prep/classify-participants` (`#12742`) runs **per event**. A busy calendar (~50 events × 14 days) means 50 function calls, each with multiple DB queries. Watch this for performance.
</Warning>

## Classification

<Frame caption="classify-participants funnel: emails sort into My Team (green), Participating Companies (purple), Personal Participants (amber).">
  <img src="/images/meeting-prep/diagram-classify.png" alt="Classification funnel" />
</Frame>

<Tabs>
  <Tab title="Pass 1 — Domain classification">
    - Seed `my_team` from `user_email` first, so one-on-one external meetings still include the user's own team context
    - Extract the domain from each participant email
    - If a non-user participant domain matches the user's domain → add that person to `my_team`
    - Otherwise check `email_provider`:
      - Personal domain → `personal_participants`
      - Corporate domain → add to `companyDomains` for Pass 2
    - Resolve every email to `master_person` for names/avatars
  </Tab>
  <Tab title="Pass 2 — Company grouping">
    - For each unique corporate domain, look up `master_company`
    - Collect all participant emails on that domain
    - Resolve each to `master_person`
    - Fall back to `logo.dev` if company logos missing
  </Tab>
  <Tab title="Output structure">
    ```json
    {
      "my_team": {
        "master_company_id": 10,
        "name": "Orbiter",
        "logo": "https://...",
        "members": [
          { "master_person_id": 12, "name": "Mark Pederson", "title": "CEO" }
        ]
      },
      "participating_companies": [
        {
          "master_company_id": 20,
          "name": "JP Morgan",
          "logo": "...",
          "members": [
            { "master_person_id": 530, "name": "Susan", "title": "VP" }
          ]
        }
      ],
      "personal_participants": [
        { "master_person_id": 540, "name": "Bob", "title": "Consultant" }
      ]
    }
    ```
  </Tab>
</Tabs>

<Note>
  **Design insight — plaintiffs vs. defendants.** `my_team` represents "your side"; `participating_companies` is "the other side(s)". A meeting can have multiple external companies, each with its own section.
</Note>

<Info>
  `participant_emails` may intentionally exclude the signed-in user's email to avoid duplicate attendee processing. Function `mvp/meeting-prep/classify-participants-get-context` (`#13070`) still builds `my_team` because it uses `user_email` as the canonical first internal member, then adds any other same-domain attendees.
</Info>

## LLM generation

<Frame caption="Meeting prep flow — Phase 1 asks for a goal, Phase 2 gathers context and calls Claude Sonnet 4.">
  <img src="/images/meeting-prep/diagram-meeting-prep.png" alt="Meeting prep diagram" />
</Frame>

API `POST /meeting-prep` (`#8098`) runs in two phases.

<Tabs>
  <Tab title="Phase 1 — Ask first">
    No `user_goal` in the request → returns suggested goals immediately:

    - Close a deal or partnership
    - Get an introduction to someone in their network
    - Pitch my product or idea
    - Build the relationship
    - Follow up on a previous conversation
    - Ask for advice or mentorship

    If the user types their own goal up front, Phase 1 is skipped entirely.
  </Tab>
  <Tab title="Phase 2 — Context gathering">
    With `user_goal` provided, collects from 4 sources:

    1. **Person context** via function `get-person-context` (`#12570`) — YAML with work history, education, skills, roles, location
    2. **User memories** via function `build-memory-context` (`#12665`) — wrapped in `[USER MEMORY]...[/USER MEMORY]` tags
    3. **Authenticated user info** from `user` table + `master_person` addon
    4. **Calendar events** via API `GET /calendar/events` (`#8125`) — 14 days ahead, max 10 events
  </Tab>
  <Tab title="Phase 3 — LLM call">
    | Setting | Value |
    |---|---|
    | Provider | OpenRouter (`openrouter.ai/api/v1/chat/completions`) |
    | Model | `anthropic/claude-sonnet-4` |
    | Temperature | `0.4` (factual output) |
    | Max tokens | `2048` |
    | Timeout | `60s` |
    | System prompt | "You are a meeting prep AI. Output ONLY valid JSON." |

    Response parsing uses a regex fallback: strip markdown fences → `JSON.parse` → regex-extract a JSON object. On total failure, return a static error object with empty arrays.
  </Tab>
  <Tab title="Response shape">
    ```json
    {
      "success": true,
      "master_person_id": 520,
      "person": { "name": "Ray Deck", "title": "CEO", "avatar": "..." },
      "prep": {
        "person_summary": "Ray is the CEO of StateChange...",
        "talking_points": [
          {
            "topic": "Recent funding round",
            "opener": "I saw StateChange just closed their Series A...",
            "why_they_care": "They're scaling the team and looking for partnerships"
          }
        ],
        "suggested_openers": ["Congrats on the funding...", "How's the team scaling?"],
        "listen_for": ["Signs of budget constraints", "Hiring challenges"],
        "landmines": ["Don't mention competitor X", "Avoid pricing discussions"]
      },
      "model": "anthropic/claude-sonnet-4"
    }
    ```
  </Tab>
</Tabs>

## Meeting chat

API `POST /chat` (`#8064`) is the general copilot chat, with special behavior when `mode=meeting`:

- System prompt loaded from `copilot_config` table (id `4`, name `meeting`) — instructs the AI on `TALKING POINTS`, `SUGGESTED OPENERS`, `WATCH OUT FOR` sections
- Memory injection via `build-memory-context` appended to system prompt
- Person context wrapped in `[ACTIVE PERSON CONTEXT]...[/ACTIVE PERSON CONTEXT]` tags
- Temperature `0.7` (more conversational than prep's `0.4`)
- History: last 10 messages
- No dispatch confirmation — informational only

### Enriched prompt

When CrayonChat opens, the canvas auto-generates a prompt composed of:

1. `Help me prepare for my meeting "[Title]"`
2. Time, duration, relative time
3. Attendees list with network status + IDs
4. Team members on call
5. `Pull full context on [network attendees] from my network`
6. Meeting description (first 500 chars)
7. Location or video call URL
8. Desired outcome (if typed)
9. `Give me talking points, key context, and anything I should be prepared for`

### Network badge logic

<Warning>
  **Use `my_person.created_at`, not `master_person.created_at`.**
  Badges are computed from when *this user* added the contact, not the global record creation time.

  - `< 30 days` → **New to Network**
  - `≥ 30 days` → **In Network**
</Warning>

### Race condition fix (prepInitiatedRef)

<Accordion title="Why ref guards matter in React useEffect chains">
  The `useEffect([selectedId])` was resetting `prepStarted` to `false` **after** `onPrepMeeting` set it to `true` in the same render cycle. Fixed with a ref guard:

  ```javascript
  const prepInitiatedRef = useRef(false);

  // onPrepMeeting handler:
  prepInitiatedRef.current = true;   // Guard before state change
  setSelectedId(id);                  // Triggers useEffect
  setPrepStarted(true);               // Would be reset without guard

  // useEffect:
  if (prepInitiatedRef.current) {
    prepInitiatedRef.current = false; // Consume the guard
    return;                           // Skip reset
  }
  setPrepStarted(false);              // Normal reset on navigation
  ```
</Accordion>

    - Update job with `llm_response`, `processing=false`
    - Write `deep_bio` to `master_person`
    - Run function `mvp/bios/create-llm-person-bios` (`#2537`) to generate `bio_100`, `bio_300`, `bio_500`
    - Run function `mvp/node/update-person-node` (`#2758`) to sync knowledge graph
  </Step>
</Steps>

    <Frame>

## XanoScript gotchas

<Warning>
  **Safe field access is mandatory for Nylas responses.** Direct property access throws `Unable to locate var` when fields are missing. Always use:

  ```
  |get:"field":default
  ```
</Warning>

<Warning>
  **`add_days_to_timestamp` does not exist.** Use `add_secs_to_timestamp` with `days * 86400`.
</Warning>

<Warning>
  **`in` operator ≠ value check.** `"key" in obj` returns true if the key exists, even with a falsy value. Check the actual value when you need the boolean.
</Warning>
