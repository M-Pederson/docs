## Source: the MusicBrainz API

| Property | Value |
|---|---|
| Base | `https://musicbrainz.org/ws/2/` |
| Format | append `?fmt=json` (or `Accept: application/json`) |
| Auth | none required |
| **Rate limit** | **1 request/second, global**, with a meaningful `User-Agent` (e.g. `Orbiter/1.0 ( mark@orbiter.io )`). Exceeding it returns `503`. |
| Lookup | `/{entity}/{MBID}?inc={subqueries}` |
| Browse | `/{result}?{linked}={MBID}&limit=100&offset=N&inc=...` |
| Search | `/{entity}?query={lucene}&limit=N` |
| Pagination | `limit` (default 25, **max 100**) + `offset`; increment by results returned |

**`inc=` is the efficiency lever.** A single lookup can pull aliases, tags, genres, ratings, and every relationship type (`artist-rels`, `recording-rels`, `work-rels`, `release-rels`, `label-rels`, `event-rels`, `place-rels`, `area-rels`, `url-rels`) in **one** request. One artist + two browse pages ‚âà 3 requests ‚âà 3 seconds. This is why the cascade must be tightly depth- and tier-capped.

<Warning>
**The 1 req/s ceiling is global, not per-cron.** Four independent queue crons each firing lookups would collectively blow the limit. The design routes **every** MusicBrainz call through one token-bucket throttle and a **single unified queue processor** ‚Äî see [Rate-limit architecture](#rate-limit-architecture). Realistic throughput is **~1 entity per 3‚Äì6 seconds**, so this pipeline is deliberately slow and bounded.
</Warning>

## Core entities ‚Üí graph nodes

MusicBrainz's 13 core entities map to Orbiter graph nodes as follows. **Seed entities** (minimum bootstrap set) are **artist, recording, label, event**; "cascade" entities get their own queued lookup; "inline" entities (place, area, genre) are materialized directly from data embedded in other entities' lookups ‚Äî no queue, no extra API call.

| MusicBrainz entity | Graph node label | Seed? | Bridges to existing Orbiter node |
|---|---|:--:|---|
| `artist` (type = Person) | `Entity:Person:Music_Artist` | ‚úÖ | **is** an Orbiter `Person` (dedup-reconciled, not a separate node) |
| `artist` (Group / Orchestra / Choir) | `Entity:Music_Group` | ‚úÖ | ‚Äî (new node type ‚Äî a band is never a `Person`) |
| `recording` | `Entity:Recording` | ‚úÖ | ‚Äî |
| `label` | `Entity:Music_Label` | ‚úÖ | `Company` via `SAME_AS` |
| `event` | `Entity:Music_Event` (+ type sub-label) | ‚úÖ | `Live_Event` (future reconcile) |
| `work` | `Entity:Work` (+`:Song` etc.) | cascade | ‚Äî |
| `release-group` | `Entity:Release_Group` (+`:Album`/`:Single`/`:EP`) | cascade | ‚Äî |
| `release` | `Entity:Release` | cascade | ‚Äî |
| `place` | `Entity:Place` (+`:Studio`/`:Venue`) | inline | ‚Äî |
| `area` | reuse `Country` / `Region` / `City` (MERGE by name; stamp `mbid`) | inline | Orbiter geography |
| `genre` | `Entity:Genre` | inline | ‚Äî |
| `series` / `instrument` / `url` | properties / deferred | ‚Äî | ‚Äî |

Every node follows Orbiter conventions: base `Entity` label, `uuid` (random) **plus `mbid`** as the natural MERGE key, `display_label`, `visibility`, `name_embedding` (vecf32), and `created_at` / `updated_at` stamped via Cypher `timestamp()` on create and update (see [Node Types](/guides/ontology/nodes) and the [timestamp convention](/guides/ontology/nodes#timestamp-coverage-audit)).

## Edge designs

Weights follow the canonical [Edge Weights](/guides/ontology/edge-weights) scale (**lower = stronger**; hub-node edges sit Weak or below; never sub-1). Every edge stamps `uuid`, `created_at`, `updated_at` via `timestamp()`. The bands: Defining 1‚Äì5, Very strong 6‚Äì15, Strong 16‚Äì25, Moderate 26‚Äì40, Weak 41‚Äì65, Negligible 100.

<Note>
The tables below summarize each edge. For the **full parameter list per edge** ‚Äî including the standard envelope (`uuid`, `weight`, `created_at`, `updated_at`, `description`, `description_generated_at`) carried by every edge ‚Äî see the dedicated **[Music Edges](/guides/enrichment/waterfall/music-edges)** reference. **`Artist`** in a pattern means `Person:Music_Artist` or `Music_Group`; `MEMBER_OF` is always `(Person:Music_Artist)‚Üí(Music_Group)`.
</Note>

## Pipeline architecture

Mirrors IMDB: **gates** ‚Üí **orchestrators** ‚Üí **inlined cascades** ‚Üí **queue infra** ‚Üí **unified cron**, all under a **namespace `mb/`** in Xano workspace 3. No LLM/scrape phases in the structural path ‚Äî orchestrators map structured JSON straight to Cypher (an LLM pass fills edge `description`s afterwards).

<Note>
**Built 2026-06-12.** The complete function inventory ‚Äî including the release-group (13099) and work (13101) orchestrators, node writers, the LLM edge-description trio, the genre/area hub-edge backfill, operator endpoints, and the deviations from this design ‚Äî lives on the dedicated **[Music Waterfall Functions](/guides/enrichment/waterfall/music-functions)** page.
</Note>

### Entry gates

<Card icon="play">

```

mb/get-add/artist ‚Äî 13088
mb/get-add/release-group ‚Äî 13089
mb/get-add/recording ‚Äî 13093
mb/get-add/work ‚Äî 13098
```

One gate per enrichable entity (**no event gate ‚Äî events deferred**). Each: normalize MBID ‚Üí dedup on a 30-day `enriched_at` window ‚Üí skip `gone`-tombstoned rows ‚Üí ensure stub row (`mb_artist` / `mb_recording` / ‚Ä¶) ‚Üí branch: **sync-enrich** when `cascade_depth == 0 && queue == false` (invoke the orchestrator directly), else **upsert the queue row**. Seed-id propagation: when no caller seed is passed, the just-created stub is the seed. The MBID is the natural key ‚Äî no URL normalization needed (unlike IMDB).

</Card>

### Cascade helpers (depth-capped fanout)

<Card icon="wrench">

```
cascading ‚Äî inlined in each orchestrator's mb_cascade phase (no standalone functions)
```

**The standalone `mb/cascade-from-*` helpers were not built.** Each orchestrator carries its own `try_catch`-wrapped `mb_cascade` phase instead. Built fanout: **artist** (13094) ‚Üí members T1 ¬∑ Albums T2 (cap 20) ¬∑ recordings T2 (cap 50) ¬∑ main-performer events T3 (depth-0 seeds only, v1.1); **label** (13091) ‚Üí ownership labels T1 ¬∑ release-groups T2 ¬∑ credited artists T2; **release-group** (13099) ‚Üí tracklist recordings T3; **recording** (13100) ‚Üí work T1 ¬∑ credit artists T1 ¬∑ rel artists T2; **work** (13101) ‚Üí composers T1 ¬∑ writers T2; **event** (13105) ‚Üí headliners T1 ¬∑ support/guest artists T2.

Depth guarding (`cascade_depth + 1 >= 3` ‚Üí no queue) and within-run MBID dedup (first tier wins) work as designed; collaborator fanout is deferred to v2.

## Rate-limit architecture

The defining constraint. Every MusicBrainz call passes through a token bucket, and **one** cron drains all queues so total throughput never exceeds 1 req/s.

<Card icon="gauge-high">

```
mb/rate-limit/acquire ‚Äî 13079
```

A token gate backed by the singleton `mb_rate_limit` row (#736: `last_call_at`, `tokens`). Every MusicBrainz call (via `mb/api/request`) calls `acquire` first: if `now - last_call_at < 1100ms` it **sleeps 1 s** (a simplification of the designed defer-and-requeue ‚Äî requeue backoff still exists at the queue layer via `next_attempt_at`), then stamps `last_call_at = now`. **Fail-open**: if the gate errors, the call proceeds and the failure is logged. Centralizes the limit across every function that touches the API.

</Card>

<Card icon="clock-rotate-left">

```
process-queue-musicbrainz ‚Äî Task 180   (INACTIVE)  ¬∑  mb/queue/process-next ‚Äî 13102
```

**One unified cron**, not four. Each invocation of `mb/queue/process-next` (13102, v1.1) selects the single highest-priority pending row **across all six queues** ‚Äî order `priority_tier ASC, cascade_depth ASC, entity_rank ASC, id ASC` (entity_rank: artist 1 ¬∑ label 2 ¬∑ event 3 ¬∑ work 4 ¬∑ release_group 5 ¬∑ recording 6), skipping rows with a future `next_attempt_at` ‚Äî locks it, dispatches the matching `run-base-*` orchestrator, deletes on success / `retry_count++` with a 5 min √ó retries backoff on failure (dropped after 5). **Schedule deviation:** the design called for a ~2 s tick, but Xano cron granularity is 60 s ‚Äî so Task 180 runs every **60 s and drains up to 10 rows per tick**, roughly the same throughput. The task **shipped INACTIVE**; pausing it is the pipeline's manual kill. A unified cron makes the global 1 req/s trivially enforceable; four parallel crons would not.

</Card>

<Warning>
**Throughput is intentionally low.** At ~1 entity / 3‚Äì6 s, a depth-0 artist with its top recordings + albums + members takes minutes, and a deep cascade takes hours ‚Äî by design. The depth cap (2), tier caps, and per-entity caps below keep total work bounded so the queue drains in finite time rather than fanning out to the full ~30 M-entity database.
</Warning>

## API errors, retries & rate limits

Reference for handling the MusicBrainz `ws/2` API. Every call to musicbrainz.org goes through `mb/rate-limit/acquire` (pacing) **and** a shared retry wrapper (`mb/api/request`, below) that applies this policy ‚Äî orchestrators never call `api.request` directly.

<Warning>
**A meaningful `User-Agent` is mandatory.** Requests without one (or with a generic library default) are throttled hard or blocked. Use `Orbiter/1.0 ( mark@orbiter.io )`. There is no API key.
</Warning>

### Status codes

| Status | Meaning | Retry? | Handling |
|---|---|:--:|---|
| `200` | OK | ‚Äî | parse JSON |
| `400` | Bad request ‚Äî malformed MBID / Lucene query | ‚ùå terminal | `log_crash` `phase: api_bad_request`; drop the queue row |
| `404` | Entity not found / merged away | ‚ùå terminal | stamp `enrich_success = false`, mark stub `gone`, drop row |
| `429` / `503` | **Rate limited / temporarily unavailable** | ‚úÖ backoff | honor `Retry-After`; exponential backoff ‚Äî the common case |
| `500` / `502` / `504` | Server / gateway error | ‚úÖ backoff | transient ‚Äî retry with backoff |
| timeout / network | no response | ‚úÖ backoff | retry with backoff |

<Note>
MusicBrainz returns **`503` for both** genuine overload **and** rate-limit violations (it does not consistently use `429`). Treat `503` as "back off and retry," never as a hard failure.
</Note>

### Retry policy

```text
max_attempts   = 5
base_delay     = 1000 ms                       # ‚â• the 1 req/s floor
backoff        = base_delay * 2^(attempt-1)    # 1s ‚Üí 2s ‚Üí 4s ‚Üí 8s ‚Üí 16s
jitter         = ¬± 250 ms                       # avoid herd on the shared cron
respect        = Retry-After header when present (overrides computed delay)
terminal       = {400, 404}                     # never retried
on_exhaustion  = release lock ¬∑ log_crash phase="api_exhausted" ¬∑ re-queue one tier lower
```

- **Defer, don't sleep.** Xano functions shouldn't block, so `acquire` and the retry wrapper **re-queue** the entity with a `next_attempt_at` backoff timestamp instead of sleeping in-process. The unified cron skips rows whose `next_attempt_at` is in the future.
- **Backoff is per-entity; pacing is global.** The 1 req/s token bucket governs *spacing* between any two calls; the retry backoff governs *re-attempts* of one failed call. Both invariants hold simultaneously.
- **Retries are idempotent.** Every write is `MERGE`-based with `coalesce`d `created_at`, so re-running a partially-applied enrich never duplicates nodes or edges.

<Card icon="cloud-arrow-up">

```
mb/api/request ‚Äî 13080
```

Single choke-point for every MusicBrainz call: calls `acquire` (13079) for pacing ‚Üí issues the request with the canonical `User-Agent` (`Orbiter/1.0 ( mark@orbiter.io )`) and `fmt=json` appended ‚Üí classifies the response per the table above ‚Üí retries up to **5 attempts with backoff sleeps** between attempts (none after the final), `400`/`404` terminal. Returns `{ok, terminal, status, attempts, json}`; failures `log_crash` with phases `api_bad_request` / `api_exhausted`. Centralizing here means rate-limit + retry behavior lives in exactly one place, and the orchestrators stay simple.

</Card>

## Priority tiers

<Note>
**Tier 0 ‚Äî bootstrap lane (decided).** Large manual seed batches (e.g. a full label roster) are queued at `priority_tier = 0`, a reserved lane that drains ahead of all cascade traffic. There is no offline dump import ‚Äî everything rides the live 1 req/s API, and multi-day backfills are accepted (see [Resolved](#resolved-design-decisions-2026-06-11)). The tiers below (1‚Äì4) apply to cascade-discovered entities.
</Note>

### `queue_mb_artist` (1 = highest)

| Tier | Rule |
|:--:|---|
| **1** | Band member of a seed group; main performer on a seed recording/event; composer of a seed work |
| **2** | Collaborator; producer/engineer on a seed recording; supporting musician |
| **3** | Co-performer at a seed event; artist credited on a label-discovered release |
| **4** | Everyone else mentioned |

### `queue_mb_recording` (1 = highest)

| Tier | Rule |
|:--:|---|
| **1** | Recording where the seed artist is the **main** performer; title track of a seed release-group |

| **3** | Recording on a release discovered via a seed label |

| Tier | Release-group rule | Event rule |
|:--:|---|---|

## Depth model

Identical to IMDB ‚Äî hard cap at depth 2.

| Depth | Meaning | Behavior |
|:--:|---|---|
| **0** | Seed (user / bootstrap) | Full lookup + all edges + cascade fanout |
| **1** | Direct discovery from a seed | Stub via `get-add`. Queued. Cron enriches + expands, but **only T1‚ÄìT2 children queue at depth 2** |
| **2** | Transitive | Stub queued. Cron enriches + writes edges, **no further cascade** |
| **3+** | Never queued | Hard cap |

<Check>
**Why cap at 2?** MusicBrainz is ~2.5 M artists / ~30 M recordings. One prolific artist ‚Üí ~20 albums ‚Üí ~250 recordings ‚Üí each recording ‚Üí 1 work + ~5 credited people ‚Üí thousands of entities at depth 2. Combined with the 1 req/s limit, an uncapped cascade would never drain. Depth 2 + tier caps + per-entity caps (20 albums, 50 recordings) keep a seed's full expansion in the low thousands of API calls.
</Check>

## Loop prevention (six layers)

1. **Dedup gate in `get-add`:** `mbid` already enriched within 30 days ‚Üí return existing, no lookup.
2. **Queue upsert:** duplicate `mbid` ‚Üí `count++`, tier upgrade. One row per MBID.
3. **Depth cap:** depth 2 enriches but does not cascade; depth 3 never queued.
4. **Tier cap at depth ‚â• 1:** only T1‚ÄìT2 discoveries queue at depth 2.
5. **Manual kill (no dedicated switch ‚Äî decided 2026-06-11):** layers 1‚Äì4 make the cascade structurally non-recursive (depth is monotonic with a hard cap; cycles collapse into `count++` on existing rows), so the planned `check-kill-switch-mb` env-var gate was skipped. Pausing the single unified cron task in Xano stops the entire pipeline.
6. **Global rate limiter:** `mb/rate-limit/acquire` + unified cron enforce 1 req/s across the whole pipeline (the layer IMDB doesn't need).

## Resolved design decisions (2026-06-11)

The open questions from the original design, each resolved before build (the sixth ‚Äî places ‚Äî surfaced while writing the [table spec](/guides/enrichment/waterfall/music-tables)):

<CardGroup cols={1}>
  <Card title="Artist ‚Üî Person reconciliation ‚Äî separate post-ingest pass" icon="link">
    **Decided.** Ingest creates standalone `Person:Music_Artist` / `Music_Label` nodes keyed on `mbid` ‚Äî no reconciliation inline. A separate post-ingest pass links them to the existing Orbiter graph, keyed on name + shared URLs (Wikidata/IMDb/social) + ISNI: person artists **merge** with their existing `Person` node (node dedup, not an edge); `Music_Label` gets a `SAME_AS` edge to `Company`. Never at ingest ‚Äî avoids false merges and keeps the 1 req/s pipeline simple.
  </Card>
  <Card title="Area mapping ‚Äî reuse Orbiter geography" icon="earth-americas">
    **Decided.** MusicBrainz areas MERGE into the existing `Country` / `Region` / `City` nodes by name, stamping `mbid` on match. No dedicated `Music_Area` node type and no mapping table. Unmappable or historical areas fall back to the nearest modern parent (e.g. a historical region maps to its present-day country).
  </Card>
  <Card title="Genre vs tag ‚Äî genres as edges, tags as property" icon="tags">
    **Decided.** `HAS_GENRE` ‚Üí `Genre` hub nodes from the curated `genres` array only (weight 100). Free-form `tags` are stored as a csv node property for search/filtering ‚Äî never edges, so folksonomy noise stays out of the Genre hub.
  </Card>
  <Card title="Canonical release ‚Äî one Official release per release-group" icon="compact-disc">
    **Decided.** Each `Release_Group` materializes exactly **one** `Release` node: `status = Official`, earliest release date (any country). Tracklists and `APPEARS_ON` edges hang off that canonical edition. All other pressings remain in the raw JSON on the `mb_*` source tables, so the decision is reversible without re-fetching.
  </Card>
  <Card title="Backfill ‚Äî live API only, bootstrap priority lane" icon="gauge-high">
    **Decided.** No offline import from the [MusicBrainz database dumps](https://musicbrainz.org/doc/MusicBrainz_Database/Download). Large bootstraps (e.g. a full label roster) ride the same 1 req/s live API, queued at **`priority_tier = 0`** ‚Äî a reserved bootstrap lane that drains ahead of all cascade traffic. One code path, no dump tooling; multi-day backfills are accepted.
  </Card>
  <Card title="Places ‚Äî inline materialization, no queue" icon="location-dot">
    **Decided.** `Place` nodes are written directly during recording/event enrich from the place object embedded in `place-rels` (name, type, address, coordinates, area ‚Äî everything the node spec needs). No `queue_mb_place`, no place orchestrator, zero extra API calls at the 1 req/s ceiling. The `mb_place` source table stores the embedded JSON and `mbid`, so a URL-rels backfill pass can be added later without re-discovery.
  </Card>
</CardGroup>
