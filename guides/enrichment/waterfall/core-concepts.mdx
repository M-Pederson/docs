The enrichment waterfall is the sequence of functions that fires when a new entity enters the system. It starts with a single input (a domain, a LinkedIn URL, a name), creates records, enriches from external APIs, discovers directly related entities, and queues those related entities for later enrichment.

As of **2026-06-04**, the core rule is **bounded and non-recursive**: seed entities may create one direct wave of related people / companies, but queued children do not create another seed-equivalent wave. Deeper enrichment can save provider responses, build the current entity, write edges, and queue leaf companies; it must not keep expanding founders, C-suite, YC people, or funding-network participants indefinitely.

This page documents the **shared mechanics** that apply to both the company and person pipelines. See [Company Enrichment Waterfall](/guides/enrichment/waterfall/company-enrichment-phases) and [Person Enrichment Waterfall](/guides/enrichment/waterfall/person-enrichment-phases) for the pipeline-specific flow.

## Cascade Depth

Every entity in the system has a **cascade depth** — the number of hops from the original seed entity.

| Depth | Meaning | Behavior |
|:-----:|---------|----------|
| **0** | Seed entity (the one the user asked for) | Enriched immediately; seed-only fanout is allowed |
| **1** | Direct discovery | Queued for later enrichment; child fanout is gated |
| **2+** | Leaf / secondary discovery | Queued or deferred by worker policy; no recursive fanout |

When `cascade_depth > 0`, the `get-add` functions skip person external API calls (PDL, Enrich Layer) and force queue writes instead of immediate downstream dispatch. The entity is created from whatever data is already available (Fundable, Exa, YC, input params) and queued for later enrichment.

<Check>
**Default seed behavior (cascade_depth: 0):** calling `get-add/master-person-new` (#13039) with `cascade_depth: 0` gives you a fully enriched person. _(The previous entry point `get-add/master-person` #12553 was deprecated 2026-06-01; all callers migrated to #13039.)_

- The person runs through `enrich-master-person` (all 12 phases; legacy `run-base-person-enrich` remains compatible).
- Phase 3 (Process PDL) identifies the best current role — ranked **founder → C-suite → VP → latest start year** — and calls `get-add/master-company-new` with `queue: true`, `cascade_depth: 1`, and `priority_tier: 1` (v3.2, 2026-06-03). This links the primary current employer but does not fully enrich that company inline.
- `priority_tier` orders the queue only; lower values run sooner.

Past employers, schools, investors, and other discovered edges honor the bounded policy and get queued as leaf candidates. A person seed no longer triggers an immediate company enrichment wave.
</Check>

<Info>
**Seed-company exception:** calling `get-add/master-company-new` (#12558) with `cascade_depth: 0` and `queue: false` still fully enriches that company. During `enrich-master-company` (#12992), seed-only phases can create direct people: Exa founders / C-suite, YC founders / partner, and Fundable investors / partners. Those people are queued at depth 1 for later person enrichment. When their enrichment discovers companies from work history, those companies are queued as leaf companies and must not mint another people wave.

Seed-company people must be scoped to the seed company before creation. Exa founder / C-suite work-history rows are accepted only when they match the seed by canonical domain, root domain, LinkedIn company profile, exact normalized name, or a conservative prefix match that includes seed aliases and alt domains. Broad substring matches are not enough; this prevents unrelated entities with overlapping names, such as a different company containing `Yoshi`, from being created as seed C-suite.
</Info>

## Priority Tiers

Queue entries are ranked by business importance:

| Tier | Description | Examples |
|:----:|-------------|----------|
| **1** | Highest-value direct people / companies | Current employer, seed-company Exa founders / C-suite, YC founders, Fundable investor partners, angels / co-investor people |
| **2** | Investor / portfolio companies and YC partner | Portfolio companies, VC firms, YC primary partner, past employers |
| **3** | Schools and lower-business-signal orgs | Educational institutions |
| **4** | Everything else | Certification issuers, volunteer orgs, publishers, Exa work-history leaf companies |

When a queue entry already exists and a new reference arrives with a better (lower) tier, the tier is upgraded. The `count` field tracks how many times the entity was referenced.

## Queue Tables

```text
queue_enrich_company — #583
```

| Field | Type | Description |
|-------|------|-------------|
| `master_company_id` | int | FK to `master_company` |
| `processing` | bool | Lock flag for the queue worker |
| `count` | int | How many times this entity was queued (default 1) |
| `cascade_depth` | int | Hops from seed entity |
| `priority_tier` | int | 1-4, lower = more important |
| `source_function` | text | Which function queued it (e.g. `resolve-investors-edges`) |
| `source_entity_id` | int | Numeric ID of the spawning entity — the `master_person_id` or `master_company_id`. Use `0` when the source is identified by UUID instead |
| `source_entity_uuid` | text | UUID/text ID of the spawning entity (e.g. `fundable_people.id`, `fundable_orgs.id`). Used when the source is not a numeric `master_*` record |
| `source_entity_type` | text | Type of source entity — `master_person`, `master_company`, `fundable_person`, `fundable_company`, etc. Disambiguates which ID field to trust |

```text
queue_enrich_person — #582
```

Same schema with `master_person_id` instead of `master_company_id`, plus a `deep_research` boolean flag. `source_entity_uuid` + `source_entity_type` follow the same pattern so Fundable-sourced queue entries remain traceable.

## Enrichment Queue Processing

Queued entities are processed by cron jobs that pull from the queue tables in priority order.

<Warning>
**Before turning on queue task processing:** `process-person-queue` (#126) and `process-company-queue` (#127) are intentionally `active = false` until the full-enrichment depth policy is set.

Set or confirm these before flipping either task to `active = true`; the depth caps and seed-only guards should be encoded as guards in #126 / #127 or in the orchestrators they dispatch:

- **Person full-enrichment depth:** choose the maximum `cascade_depth` that should run `enrich-master-person`; the current bounded test posture is people at depth **0–1**.
- **Company full-enrichment depth:** choose the maximum `cascade_depth` that should run `enrich-master-company`; deeper company runs must keep seed-only people / deal fanout gated.
- **Seed-only expansion:** company Exa C-suite / founders, YC people, company deal fanout, and person investment-network fanout run only at `cascade_depth == 0`.
- **Funding-round investor company suppression:** VC firms discovered by `resolve-investors-edges` are created / linked and queued, but `enrich-master-company` suppresses full company enrichment for funding-round investor companies when they arrive from that source at depth > 0.
- **Depth overflow behavior:** decide what happens beyond the caps before activation — leave rows queued for manual review, mark them deferred, or delete them with a `log_crash` audit row. Do not silently full-enrich depth-2+ rows into another expansion wave.
- **Task invariants:** both queue processors must sort by `{priority_tier: "asc", cascade_depth: "asc", id: "asc"}`, pass the stored `cascade_depth` into the orchestrator, delete only their own queue table on success, and release `processing = false` on failure.
</Warning>

### Queue Upsert Pattern

When a company is queued multiple times (e.g. discovered as an employer by two different people), the system uses an **upsert** pattern:

1. Check if `queue_enrich_company` already has an entry for this `master_company_id`
2. If **yes**: increment `count`, upgrade `priority_tier` to the better (lower) value
3. If **no**: insert new queue entry with all metadata

This means a company discovered once as a tier-4 publisher and again as a tier-1 current employer will be upgraded to tier 1 — it gets enriched sooner because someone important works there.

### Deal Cascade Pattern

Investor / funding-round discovery is consolidated into one helper so the fanout is identical no matter which side of the deal surfaces it.

```text
mvp/investor/cascade-deal-participants — #12856
```

Given one `fundable_deals.id`, the helper does:

1. **Portfolio company** — `get-add/master-company-new` for the deal's target org (queued at **depth 1, tier 2**). Writes `master_company_id` back onto `fundable_organizations`.
2. **IPO / exit signal** — if `fundable_organizations.ipo_status == "public"`, sets `company_financial.is_public = true` on the portfolio company.
3. **Funding_Round node** — `add-fundable-deal-node` materializes the round + a `RAISED` edge from the portfolio company.
4. **All participants** — `resolve-investors-edges` #12702 walks the deal:
   - `fundable_angel_investments` → co-angels (people)
   - `fundable_institutional_investments` → VC firms (companies)
   - `fundable_institutional_investments_person` → VC partners at each firm on that deal (people)
   - Creates `LEAD_INVESTED_IN` / `INVESTED_IN` / `FOLLOW_ON_INVESTED_IN` / `INVESTMENT_PARTNER_IN` / `INVESTMENT_PARTNER_AT` edges in FalkorDB.

People from this helper queue at **tier 1**; investor / portfolio companies queue at **tier 2**. Direct deal investor companies are treated as leaves: they get nodes, links, and investment edges, but not a recursive company enrichment wave. A Fundable row's broad `is_investor` flag does **not** automatically make a `master_company` a VC firm; `get-add/master-company-new` classifies `is_vc` from LLM/company-type evidence so strategic corporate investors remain normal companies.

**Called from two seed-only entry points:**

- `process-person-phase-9` v3.1 — for each row in a **seed person's** `fundable_angel_investments` (angel-origin)
- `process-company-phase-7` v1.2 → `add-all-fundable-deals` — for a **seed company**, both rounds the org **raised** and rounds the org **invested in** (VC-origin)

Because both seed callers route through the same helper, a direct deal is materialized consistently from either side. Queue upsert semantics (above) keep the fanout idempotent — if two seed paths surface the same VC firm, the firm's queue entry just increments `count` and holds its best tier.

### Queue Entry Lifecycle

```mermaid
stateDiagram-v2
    [*] --> Queued: get-add creates entry
    Queued --> Queued: count++, tier upgrade
    Queued --> Processing: Cron picks up entry
    Processing --> Enriched: enrich orchestrator completes
    Processing --> Failed: Error during enrichment
    Failed --> Queued: Retry on next cron cycle
```

## Kill Switch

```text
mvp/stop/check-kill-switch-company
```

A safety valve that can be toggled to halt all new entity creation. When active:

- **Existing companies**: Still returned via local-only lookup (domain + URLs)
- **New companies**: Blocked entirely — no API calls, no records created
- **Logged**: Every blocked entity is recorded in `log_crash` with the input data for later processing

The kill switch runs in **Section 3b**, before any external API calls. This ensures zero API spend when the switch is on.

The person pipeline has the equivalent `mvp/stop/check-kill-switch-person` — blocked persons are saved to `kill_switch_blocked_people` for later reprocessing.

## Per-Phase Crash Logging

Both orchestrators write per-phase observability into `log_crash` using a dedicated `phase` enum field (added 2026-04-14, `run-base-person-enrich` v3.2). Each phase wraps its work in a `try_catch` so a failure in one phase is logged but does not block later phases.

Rows written per phase:
- `qa_passed: true` when the phase completes cleanly
- `qa_passed: false` + stack trace when the phase throws, with `phase` set to the phase name (e.g. `resolve-edges-work`, `llm-bios`)

This lets the QA dashboards filter crashes by exact phase rather than scanning unstructured messages.
