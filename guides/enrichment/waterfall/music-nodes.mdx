The complete node catalog for the [Music Industry Waterfall](/guides/enrichment/waterfall/music-industry-waterfall). Each node is keyed on its MusicBrainz `mbid` (`MERGE (n:Label {mbid})`) and is idempotent on re-enrich. Sample data is from live MusicBrainz responses.

<Note>
**Every node carries the standard envelope.** At a minimum, **all** nodes below are created with `created_at`, `updated_at`, `uuid`, `mbid`, `display_label`, `visibility`, and the two embedding vectors `name_embedding` + `embeddings`. **`Recording` and `Music_Group` additionally carry `image_url`** (the re-hosted GCS image — see [Media](#media-image_url)).

**Description coverage today:** only `Recording` and `Music_Group` receive a generated `description` (via `finalize-music-node` / `backfill-node-descriptions`); the other **eight** types currently carry `description = null` — description generation for them is not yet implemented (verified by the [sandbox node-type test](/guides/enrichment/waterfall/music-sandbox-test#findings-from-the-first-run), 2026-06-14). The natural-language `description` is generated after the structural write (like [edge descriptions](/guides/enrichment/waterfall/music-edges#edge-descriptions)) and is `null` between create and generation.
</Note>

<Note>
**Dual embeddings (OpenAI → Gemini migration, 2026-06-14).** Every node now carries **two** vectors built from the *same text*:
- **`name_embedding`** — OpenAI `text-embedding-3-small`, **1536-dim** (`vectors/create-vectors-string` #4676). Indexed today: a single FalkorDB vector index on the `:Entity` base label covers all node types.
- **`embeddings`** — Google `gemini-embedding-001`, **3072-dim** (`vectors/create-gemini-vectors` #13114). Written but **not yet indexed** — the `:Entity(embeddings)` 3072 vector index is **deferred until OpenAI `name_embedding` is deprecated and removed** (then it gets created and reads cut over).

The embed text is the node `description` for `Recording`/`Music_Group` and `name + disambiguation` for the other eight types. Forward-path writes happen in the node writers (`add-music-label-node`, `add-music-artist-node`) and the description backfill (`backfill-node-descriptions`); existing nodes and the six types with no ingest-time embedding are filled by [`mvp/music/backfill-node-embeddings`](/guides/enrichment/waterfall/music-functions#node-embeddings) (gated on `embeddings IS NULL`).

⚠️ Before this migration, `name_embedding` was in practice only written for **4** node types (`Music_Label`, `Music_Artist`, `Music_Group`, `Recording`) — the six others never received it at ingest despite the envelope claiming otherwise. The embeddings backfill now backfills **both** vectors for all ten.
</Note>

<Warning>
**Artist split — Person vs Group.** A MusicBrainz `artist` is **not** one node type:
- `type == "Person"` → **`Entity:Person:Music_Artist`** — a real human is stored as an Orbiter **`Person`** node with the `Music_Artist` label added, so it participates in **both** the people graph (`WORKS_AT`, `KNOWS`, …) **and** the music graph.
- `type ∈ {Group, Orchestra, Choir, Other}` → **`Entity:Music_Group`** — a **new, dedicated** node type. A band (Nirvana) is never a `Person`.

`MEMBER_OF` is therefore always `(Person:Music_Artist)→(Music_Group)`.
</Warning>

## Standard node envelope

```cypher
MERGE (n:<labels> {mbid: $mbid})
SET n.uuid           = coalesce(n.uuid, randomUUID()),
    n.display_label  = "<Label>",
    n.visibility     = true,
    n.description    = <generated summary | null>,
    n.name_embedding = vecf32($embedding),       // OpenAI 1536
    n.embeddings     = vecf32($geminiEmbedding),  // Gemini 3072
    n.created_at     = coalesce(n.created_at, timestamp()),
    n.updated_at     = timestamp()
    // … node-specific properties (+ image_url on Recording / Music_Group) …
```

| Field | Type | Required | Notes |
|---|---|:--:|---|
| `uuid` | string | ✅ | `randomUUID()`, stable via `coalesce` |
| `mbid` | string | ✅ | MusicBrainz ID — the MERGE key |
| `display_label` | string | ✅ | UI label |
| `visibility` | boolean | ✅ | UI flag |
| `description` | string | ✅ | one-paragraph NL summary; `null` until generated |
| `name_embedding` | vecf32 | ✅ | **OpenAI** embedding, 1536-dim (`#4676`); indexed on `:Entity` |
| `embeddings` | vecf32 | ✅ | **Gemini** embedding, 3072-dim (`#13114`); same text as `name_embedding`; index deferred |
| `image_url` | string | Recording / Music_Group | re-hosted GCS image URL; `null` when no image — see [Media](#media-image_url) |
| `created_at` | timestamp | ✅ | `coalesce(n.created_at, timestamp())` |
| `updated_at` | timestamp | ✅ | `timestamp()` on every write |

### Embeddings per node type

Which function writes each node's vectors, and the text they embed. All ten end up with **both** `name_embedding` (OpenAI 1536) and `embeddings` (Gemini 3072):

| Node type | Written by | Embed text |
|---|---|---|
| `Music_Label` | `add-music-label-node` 13090 (ingest) | name + disambiguation |
| `Music_Artist` | `add-music-artist-node` 13092 (ingest) | name + disambiguation |
| `Music_Group` | 13092 (ingest) → 13113 overwrite | description |
| `Recording` | `backfill-node-descriptions` 13113 | description |
| `Work` | `backfill-node-embeddings` 13115 | name + disambiguation |
| `Release_Group` | `backfill-node-embeddings` 13115 | name + disambiguation |
| `Release` | `backfill-node-embeddings` 13115 | name + disambiguation |
| `Music_Event` | `backfill-node-embeddings` 13115 | name + disambiguation |
| `Place` | `backfill-node-embeddings` 13115 | name + disambiguation |
| `Genre` | `backfill-node-embeddings` 13115 | name (no disambiguation) |

The four with an ingest-time writer (rows 1–4) are also caught up by the backfill; the six greenfield types (rows 5–10) get **both** vectors exclusively from `backfill-node-embeddings` 13115.

<Tip>
**What is `disambiguation`?** A MusicBrainz free-text qualifier — its "disambiguation comment" — that distinguishes entities sharing the same or similar name. Examples from real data: **Nirvana** → `1980s–1990s US grunge band`, **Kurt Cobain** → `lead singer and guitarist for Nirvana`, **Dave Foster** → `Nirvana's 3rd drummer`, **"Elvana"** → `"Elvis fronted Nirvana" tribute band`. It's a human-readable phrase, **not** an ID, and is often empty (`""`). For the eight name-based node types above it's appended to the embed text (`name + disambiguation`) so the vector carries genre / era / region signal instead of a bare, ambiguous name.
</Tip>

## People & groups

### Person:Music_Artist

A human musician — its **own** `Entity:Person:Music_Artist` graph node (a Person node, but **distinct from** the business-side `master_person` it links to). **Bridged** to a `master_person` `Person` by the [person-resolution](/guides/enrichment/waterfall/music-functions#person-resolution) pass (2026-06-14) — a `SAME_AS` edge, **not a merge** (MB artists carry no `linkedin_url`, so there is rarely an existing Person to dedup into; instead a `master_person` is minted/linked from the artist's socials and bridged). That pass also stamps the node with a serper avatar (`image_url`), a `social_links` list, and `social_resolved_at`.

**Labels:** `Entity:Person:Music_Artist`

| Property | Type | Description |
|---|---|---|
| `uuid` | string | Primary identifier |
| `mbid` | string | MusicBrainz artist ID (MERGE key) |
| `name` / `sort_name` | string | `Kurt Cobain` / `Cobain, Kurt` |
| `display_label` | string | `"Person"` |
| `gender` | string | `Male` |
| `country` | string | `US` |
| `birth_date` / `death_date` | string | `1967-02-20` / `1994-04-05` (from `life-span`) |
| `begin_area` | string | `Aberdeen` (hometown) |
| `instruments` | string (csv) | aggregated from `MEMBER_OF` / credits — `guitar, vocals` |
| `isnis` / `ipis` | string (csv) | `0000000110685005` — rights-society IDs |
| `music_rating` / `music_rating_votes` | number / int | `5` / `1` |
| `genres` | string (csv) | `grunge, rock` |
| `disambiguation` | string | `lead singer and guitarist for Nirvana` |
| `image_url` | string | *(person-resolution, 2026-06-14)* serper.dev avatar (`music-person-avatar/<mbid>.webp`); `null` when none found |
| `social_links` | list | *(person-resolution)* verified social / streaming / homepage URLs found via Exa, cross-checked against MB url-rels + aliases |
| `social_resolved_at` | string | *(person-resolution)* ISO timestamp of the resolution pass |
| `description` | string | _generated_ |
| `visibility` · `name_embedding` · `created_at` · `updated_at` | | standard envelope |

A resolved person also carries a **`SAME_AS`** edge (weight 100) to a `master_person` `Person` node — the bridge into the business-side people graph — and the matching `mb_artist.master_person_id` is stamped (#737). All verified socials are additionally written as `master_link` rows on that `master_person`, firing the platform scrapers. See [person-resolution](/guides/enrichment/waterfall/music-functions#person-resolution).

## Media (`image_url`)

Two node types carry a re-hosted image on an **`image_url`** property — matching the `film_tv.image_url` convention on the IMDb side. The value is always the **GCS-hosted WebP** (never the raw external URL, which is retained only on the `mb_*` source row), and it is **explicitly `null` when no image is available** so the property is always present.

| Node | Source field (`mb_*`) | GCS path | Raw source |
|---|---|---|---|
| `Recording` | `mb_recording.gcp_cover_art_url` (#738) | `music-cover-art/<mbid>.webp` | Cover Art Archive (`cover_art_url`) |
| `Music_Group` | `mb_artist.gcp_music_group_image_url` (#737) | `music-group-image/<mbid>.webp` | serper.dev image search (`music_group_image_url`) |

Example value: `https://storage.googleapis.com/<bucket>/music-cover-art/<mbid>.webp` (bucket is environment-specific, e.g. `orbiter-sandbox`).

The node `image_url` is set by the [image fetch functions](/guides/enrichment/waterfall/music-functions#cover-art--group-images) (`fetch-recording-cover-art` 13108, `fetch-group-image` 13109): right after the GCS URL is stamped on the source row, a Cypher `SET n.image_url` writes the node (non-fatal; also self-heals on the `already_done` path so re-running a row catches up existing nodes). When no art/image is found, the same path sets `n.image_url = null`.
