Seventeen distinct scheduled jobs — twenty-one live tasks once the work- and relationship-drain clone sets are counted individually — touch the person waterfall, and **no active task drains `queue_enrich_person`**. The only drain, **Task #126**, ships `active = false` on live — exactly like the company family's Task #127 — so queued people sit until an operator runs the shared batch drain [`process-enrichment-queue` #12816](/guides/enrichment/waterfall/person-functions#queue--gate-infrastructure) or clicks **Run Now** on #126. Everything else in the roster is a one-shot seed, a finished backfill drain, or hygiene; the single `active = true` task is the temporary music-image backfill Task #184. Queue drains follow the family tick contract: pick sorted `priority_tier asc, cascade_depth asc, id asc`, pass the row's stored `cascade_depth` into the orchestrator, delete only their own row on success, release `processing = false` on failure. Queue semantics: [Person Waterfall overview](/guides/enrichment/waterfall/person-waterfall#queue--upsert-semantics) · queue schema: [Person Waterfall Tables](/guides/enrichment/waterfall/person-tables) · orchestrator \+ drain internals: [Person Waterfall Functions](/guides/enrichment/waterfall/person-functions).

<Card icon="clock-rotate-left">
  ```text
  process-person-queue — Task #126   (INACTIVE)
  ```

  **Single-row tick on `queue_enrich_person`, every 600 s** (datasource `live`; no version header in the task script). Pick: `processing == false` sorted `priority_tier asc, cascade_depth asc, id asc` (lowest tier, then shallowest depth, then FIFO) → lock `processing: true` (advisory — not atomic with the pick) → **sync** dispatch of `enrich-master-person` #13040 v3.23 (2026-06-30) with the row's stored `master_person_id`, `deep_research`, and `cascade_depth` → delete its own row on success / `log_crash` \+ release on failure. There is no retry counter and no dead-letter: with stable sort keys, a deterministically-crashing row is re-picked on the very next tick, forever — head-of-line poison-pill blocking of the whole queue.

  **As-built (flagged):** the catch writes its `log_crash` rows with `phase: "queue"` — an out-of-enum value that makes the crash logger itself throw. See the Warning below.
</Card>

<Warning>
  **Task #126 is `active = false` on live** (verified 2026-07-02) — nothing drains the person queue automatically today. **And #126 carries an unpatched crash-logger bug:** its catch writes `log_crash` with `phase: "queue"`, which is not in the strict #542 `phase` enum, so a queue-item crash makes the crash logger itself fail. The identical defect was fixed on the company drain Task #127 on 2026-07-02; #126 (last updated 2026-06-02) did **not** get the fix. Before flipping the task active in the Xano UI: patch the phase value, and set the full-enrichment depth policy per the [Core Concepts activation checklist](/guides/enrichment/waterfall/core-concepts#enrichment-queue-processing) — the current bounded posture is people at depth **0–1**, with depth-overflow rows held for review (never silently full-enriched into another expansion wave). Max throughput once active: 6 people/hour at 1 row per 10-minute tick.
</Warning>
