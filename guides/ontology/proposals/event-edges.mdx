title: "Event Edges (proposed)"
description: "All proposed edge types for the Event node — hosting, participation, sponsorship, location, and series relationships with weights"
---

<Note>
**Status:** Proposed — companion to the [Event node spec](/guides/ontology/proposals/event). Weights follow existing convention: **lower = stronger**.
</Note>

**17 edge types** across 6 logical groups. Direction convention follows the existing [Edges reference](/guides/ontology/edges): subject-first where a subject "does something to" the event (`HOSTS`, `SPONSORS`, `ATTENDED`), event-first when describing the event's own properties (`LOCATED_IN`, `SUCCEEDS`, `PART_OF`, `FOCUSES_ON`).

<Note>
`ATTENDED` is already used for `(Person)-[ATTENDED]->(School)` with weight 40. Event attendance uses a **distinct edge name** — `ATTENDED_EVENT` — to keep queries clean. Event attendance is also a **weaker signal** than school (short duration, often passive), so its weight is **higher** (weaker) than school's 40.
</Note>

## Organizing & sponsorship

Edges from the organization or company *acting on* the event.

### HOSTS

The primary organizer that owns and runs the event.

| | |
|---|---|
| **Pattern** | `(Organization\|Company)-[HOSTS]->(Event)` |
| **Weight** | **5** (highest importance — defining relationship) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `role` | string | Host type — `"organizer"`, `"co-host"`, `"presenting_host"` |
| `valid_from` | string | When hosting relationship started (often = series start year) |
| `valid_to` | string | Null if still hosting |
| `weight` | integer | Always `5` |

<Accordion title="Real example — LSI hosts LSI USA '25">
```json
{
  "uuid": "a1b2c3d4-hosts-0001-abcd-ef0123456789",
  "role": "organizer",
  "valid_from": "2018",
  "valid_to": null,
  "weight": 5
}
```
`(Life Science Intelligence)-[HOSTS]->(LSI USA '25)`
</Accordion>

### PRODUCES

A production company or agency that executes the event on behalf of the host. Distinct from `HOSTS` — the producer is hired, the host owns the brand.

| | |
|---|---|
| **Pattern** | `(Company)-[PRODUCES]->(Event)` |
| **Weight** | **8** |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `valid_from` | string | Contract start |
| `valid_to` | string | Contract end |
| `weight` | integer | Always `8` |

---

### SPONSORS

A company, fund, or firm providing financial/in-kind sponsorship. Weight varies by sponsorship tier.

| | |
|---|---|
| **Pattern** | `(Company\|VC_Firm)-[SPONSORS]->(Event)` |
| **Weight** | **20–35** (varies by tier) |

| `tier` value | Weight | Examples |
|---|---|---|
| `title` | **20** | Named/title sponsor, logo in event name |
| `platinum` | **22** | Top-tier branded sponsor |
| `gold` | **25** | Mid-tier sponsor |
| `silver` | **28** | Standard tier |
| `bronze` | **32** | Entry tier |
| `in_kind` | **35** | Non-cash (venue, catering, swag) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `tier` | string | Sponsorship tier (see table above) |
| `amount_usd` | number | Cash value of sponsorship, when known |
| `valid_from` | string | When sponsorship was committed |
| `valid_to` | string | Event end date |
| `weight` | integer | Derived from `tier` |

<Accordion title="Real example — Accel sponsors LSI USA '25">
```json
{
  "uuid": "b2c3d4e5-sponsors-0001-bcde-f01234567890",
  "tier": "gold",
  "valid_from": "2024-12-01",
  "valid_to": "2025-03-21",
  "weight": 25
}
```
`(Accel)-[SPONSORS]->(LSI USA '25)`
</Accordion>

## People participation

Edges connecting individual humans to events.

### SPOKE_AT

Person gave a talk, keynote, or joined a panel as a speaker. Strongest individual participation signal.

| | |
|---|---|
| **Pattern** | `(Person)-[SPOKE_AT]->(Event)` |
| **Weight** | **15** |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `session_title` | string | Name of the session/panel |
| `role` | string | `"keynote"`, `"panelist"`, `"fireside"`, `"lightning_talk"` |
| `topic` | string | Session topic/theme |
| `session_date` | string | ISO date of the specific session |
| `valid_from` | string | Session start |
| `valid_to` | string | Session end |
| `weight` | integer | Always `15` |

<Accordion title="Real example — keynote at LSI USA '25">
```json
{
  "uuid": "c3d4e5f6-spoke-0001-cdef-012345678901",
  "session_title": "The Future of AI in Medical Devices",
  "role": "keynote",
  "topic": "AI-enabled devices",
  "session_date": "2025-03-18",
  "valid_from": "2025-03-18T09:00:00Z",
  "valid_to": "2025-03-18T10:00:00Z",
  "weight": 15
}
```
`(Speaker)-[SPOKE_AT]->(LSI USA '25)`
</Accordion>

### ATTENDED_EVENT

Person attended the event — confirmed post-event participation. **Distinct from `ATTENDED`** (which targets schools) to avoid ambiguous target-label filtering in queries. Weaker signal than school attendance: short duration, often passive.

| | |
|---|---|
| **Pattern** | `(Person)-[ATTENDED_EVENT]->(Event)` |
| **Weight** | **55** (weaker than school `ATTENDED` at 40 — shorter, more passive) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `attendance_source` | string | How we know — `"registration_list"`, `"public_post"`, `"photo"`, `"bio_mention"` |
| `confidence` | number | 0–1 confidence score |
| `checked_in` | boolean | Whether we have confirmed check-in |
| `valid_from` | string | Event start |
| `valid_to` | string | Event end |
| `weight` | integer | Always `55` |

<Accordion title="Real example — attended LSI USA '25">
```json
{
  "uuid": "d4e5f6a7-att-0001-def0-123456789012",
  "attendance_source": "registration_list",
  "confidence": 0.95,
  "checked_in": true,
  "valid_from": "2025-03-17",
  "valid_to": "2025-03-21",
  "weight": 55
}
```
`(Person)-[ATTENDED_EVENT]->(LSI USA '25)`
</Accordion>

### REGISTERED_FOR

Person registered/applied to attend but attendance is not yet confirmed. Pre-event state. Becomes `ATTENDED_EVENT` post-event (both edges can coexist).

| | |
|---|---|
| **Pattern** | `(Person)-[REGISTERED_FOR]->(Event)` |
| **Weight** | **65** (weakest people-event signal — intent only) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `registered_at` | string | ISO timestamp of registration |
| `ticket_tier` | string | `"vip"`, `"standard"`, `"early_bird"`, `"press"`, `"speaker"` |
| `registration_status` | string | `"registered"`, `"waitlist"`, `"cancelled"`, `"no_show"` |
| `valid_from` | string | Registration date |
| `valid_to` | string | Event end or cancellation date |
| `weight` | integer | Always `50` |

<Accordion title="Real example — registered for LSI USA '26">
```json
{
  "uuid": "e5f6a7b8-reg-0001-ef01-234567890123",
  "registered_at": "2025-11-02T14:22:00Z",
  "ticket_tier": "standard",
  "registration_status": "registered",
  "valid_from": "2025-11-02",
  "valid_to": null,
  "weight": 65
}
```
`(Person)-[REGISTERED_FOR]->(LSI USA '26)`
</Accordion>

## Location

### LOCATED_IN (Event)

**Reuses the existing `LOCATED_IN` edge** used for `Person\|Company → City\|Region\|Country`. For events, only chain to `City` — the city handles its own `LOCATED_IN` up to region and country.

| | |
|---|---|
| **Pattern** | `(Event)-[LOCATED_IN]->(City)` |
| **Weight** | **30** (matches existing `LOCATED_IN` for Person/Company → City) |

| Property | Type | Description |
|----------|------|-------------|
| `uuid` | string | Edge identifier |
| `name` | string | `"LOCATED_IN"` |
| `venue_name` | string | Specific venue within the city |
| `valid_from` | string | Event start |
| `valid_to` | string | Event end |
| `weight` | integer | Always `30` |

<Accordion title="Real example — LSI USA '25 in Dana Point">
```json
{
  "uuid": "a7b8c9d0-loc-0001-0123-456789012345",
  "name": "LOCATED_IN",
  "venue_name": "Waldorf Astoria, Monarch Beach",
  "valid_from": "2025-03-17",
  "valid_to": "2025-03-21",
  "weight": 30
}
```
`(LSI USA '25)-[LOCATED_IN]->(Dana Point [City])`
</Accordion>

## Weight summary

Sorted strongest → weakest, per graph convention.

| Weight | Edge | Direction |
|---|---|---|
| **5** | HOSTS | Organization → Event |
| **5** | SUCCEEDS | Event → Event (prior edition) |
| **5** | PART_OF | Event → Event (parent) |
| **8** | PRODUCES | Company → Event |
| **12** | EMCEED | Person → Event |
| **15** | SPOKE_AT | Person → Event |
| **15** | MODERATED | Person → Event |
| **15** | PRESENTED_AT | Person\|Company → Event |
| **18** | JUDGED_AT | Person → Event |
| **20** | MENTORED_AT | Person → Event |
| **20–35** | SPONSORS | Company\|VC_Firm → Event (by tier) |
| **25** | FOCUSES_ON | Event → DomainExpertise |
| **30** | EXHIBITED_AT | Company → Event |
| **30** | LOCATED_IN | Event → City |
| **55** | ATTENDED_EVENT | Person → Event |
| **65** | REGISTERED_FOR | Person → Event |

## Open questions

<AccordionGroup>
  <Accordion title="1. `ATTENDED_EVENT` vs reusing `ATTENDED` — resolved">
    **Decision:** use `ATTENDED_EVENT` (distinct edge name). Avoids ambiguous target-label filtering when querying `(p:Person)-[:ATTENDED]->(x)`. Weight set to **55** — weaker than school `ATTENDED` (40) because event attendance is shorter and often passive. Kept here for reference; no longer open.
  </Accordion>
  <Accordion title="2. Reuse `PRESENTED_AT` for both Person and Company subjects?">
    Same edge name with different subject labels (`Person` vs `Company`) is graph-valid and avoids near-duplicate edge types. But it means "who pitched at LSI?" returns a mixed list. Lean toward **reuse** with subject-label filtering.
  </Accordion>
  <Accordion title="3. Do we need `COLOCATED_WITH` between events?">
    Two events at the same venue on overlapping dates (e.g. JPM Healthcare + Biotech Showcase in SF). Could be derived from `LOCATED_IN` + date overlap at query time. Recommend **skip** — derive, don't materialize.
  </Accordion>
  <Accordion title="4. Should `MET_AT` exist as a Person-Person edge?">
    Tempting to materialize "these two people met at X event" as a direct edge, but it can be derived from shared `ATTENDED` / `SPOKE_AT`. Skip unless we need it for performance.
  </Accordion>
  <Accordion title="5. Weight tuning for sponsorship tiers">
    Proposed 20/22/25/28/32/35 spread. Reasonable? Tighter spread (22/24/25/26/28/30) would smooth ranking; wider spread (15/20/25/30/35/40) would amplify tier-based signal. Calls for a ranking test once a handful of events are ingested.
  </Accordion>
</AccordionGroup>
