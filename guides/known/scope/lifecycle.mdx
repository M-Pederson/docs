---
title: "Facet Lifecycle"
description: "Crons that create, invalidate, and maintain lifecycle facets (BONDED, ESTRANGED, PROFESSIONAL_TIE, KINSHIP, and deceased freeze)"
---

Facet lifecycle crons watch the scoring layer and upstream graph changes, then create or invalidate facets accordingly. These are the automated rules that turn signal patterns into state transitions.

All inferred facet creation/invalidation goes through these crons. **Declared facets (user or LLM action) are never touched by these jobs.**

See [Documentation — Facet lifecycle rules](/guides/known/documentation#facet-lifecycle-rules) for the underlying rules.

---

## `known/lifecycle/bonded_creator` (cron)

Finds `KNOWS` edges where:
- `strength_score > 70`, AND
- `shared_context_cache` has real overlap (shared employment / school / children / household / ventures), AND
- No currently-valid `BONDED` exists

Creates inferred `BONDED` with a derived `bond_type` array.

**`bond_type` derivation rules** (the array can contain multiple values):
- Currently-valid `ROMANTIC(type='married')` → include `'spouse'`
- Currently-valid `ROMANTIC(type='dating' | 'domestic_partner')` → include `'romantic'`
- Currently-valid `KINSHIP` (direct) → include `'familial'`
- Currently-valid `PROFESSIONAL_TIE(relationship_role='co_founder')` → include `'cofounder'`
- Both persons have `KINSHIP(relation='parent')` to the same third person → include `'co_parent'`
- Currently-valid `MENTOR` → include `'mentor_mentee'`
- Otherwise → `'friendship'`

**Cadence:** Nightly
**Depends on:** Scoring cron finished

## `known/lifecycle/bonded_invalidator` (cron)

Finds `KNOWS` edges with valid inferred `BONDED` where `strength_score < 55` (hysteresis — 15-point gap from the 70 creation threshold prevents flapping).

Invalidates with `invalidation_reason='ended'`. Snapshots `peak_strength_during_bond` at invalidation time.

**Cadence:** Nightly
**Depends on:** Scoring cron finished

## `known/lifecycle/estrangement_detector` (cron)

Scans `KNOWS` edges for the 3-signal estrangement pattern. **All three signals required:**

<Steps>
  <Step title="Interaction dropoff">
    `>=90%` month-over-month drop in direct interaction modalities (email, chat, calls, meetings).
  </Step>
  <Step title="Shared context terminus">
    Within ±60 days of the dropoff, at least one of: `PROFESSIONAL_TIE.valid_to` set, `ROMANTIC.valid_to` set, `RESIDED_AT.move_out` set (end of cohabitation).
  </Step>
  <Step title="Negative external signal">
    At least one of: press keyword classifier hit (split/ouster/lawsuit language), LLM sentiment pass returns negative on recent interactions, public records divorce/lawsuit detection.
  </Step>
</Steps>

When all three are present, creates `ESTRANGED(source='inferred', reason=<derived>, severity=<1-5>, confidence=<0-1>)`. **Cascade-invalidates** any valid inferred `BONDED` on the same pair with `invalidation_reason='superseded'` at the same timestamp.

**Cadence:** Nightly
**Depends on:** Scoring cron finished, LLM sentiment pass finished

## `known/lifecycle/reconciliation_detector` (cron)

Scans edges with valid inferred `ESTRANGED`. For each, checks:

1. ≥3 direct interactions within rolling 90-day window
2. Across ≥2 distinct modalities (not just 3 emails)
3. LLM sentiment pass on recent interactions returns neutral-or-positive

If all three, invalidates the `ESTRANGED` with `invalidation_reason='reconciliation'`.

**Important:** Only invalidates inferred `ESTRANGED`. Declared `ESTRANGED` requires declared reconciliation (user or LLM action).

**Cadence:** Nightly
**Depends on:** Scoring cron finished, LLM sentiment pass finished

## `known/lifecycle/professional_tie_maintainer` (trigger)

When any `WORKS_AT` edge's `valid_to` is updated, re-evaluate `PROFESSIONAL_TIE` edges that depend on the overlap. Invalidate those whose overlap has ended with `invalidation_reason='ended'`.

**Cadence:** Trigger on `WORKS_AT` change

## `known/lifecycle/kinship_divorce_cascader` (trigger)

**Renamed from `kinship_in_law_cascader`.** Now handles both in-law AND step-relation cascades on divorce.

**Fires on divorce only.** When a `ROMANTIC(type='married')` facet is invalidated with `invalidation_reason='divorce'`, the cron executes three cascade passes in sequence:

<Steps>
  <Step title="Invalidate in-law KINSHIPs">
    Find all `KINSHIP(relation IN ['parent_in_law', 'child_in_law', 'sibling_in_law'])` facets whose `via_chain` includes the divorcing pair. Invalidate with `invalidation_reason='divorce_severed_in_law'`.
  </Step>
  <Step title="Invalidate step-parent / step-child KINSHIPs">
    Find all `KINSHIP(relation IN ['step_parent', 'step_child'])` facets whose `via_chain` includes the divorcing pair. Invalidate with `invalidation_reason='divorce_severed_step'`.
  </Step>
  <Step title="Invalidate step-sibling KINSHIPs">
    Find all `KINSHIP(relation='step_sibling')` facets whose multi-anchor `via_chain` includes **both** persons of the divorcing pair as anchors. Invalidate with `invalidation_reason='divorce_severed_step'`.
  </Step>
</Steps>

**NEVER cascade-invalidates:**
- Direct biological relations (`parent`, `child`, `sibling`) — always survive
- Extended biological relations (`grandparent`, `grandchild`, `aunt_uncle`, `niece_nephew`, `cousin`) — always survive
- Adoptive relations (`adoptive_parent`, `adoptive_child`, `adoptive_sibling`) — adoption is legally permanent and survives the adopter's divorce

**Does NOT fire on death.** When `deceased_freezer` invalidates a ROMANTIC with `invalidation_reason='death'`, this cascader is suppressed. In-law and step KINSHIPs stay valid through a partner's death. Rationale: divorce is an active decision to sever the family structure; death is not.

**Does NOT fire on breakup of non-marriage ROMANTIC.** Only `invalidation_reason='divorce'` on `ROMANTIC(type='married')` triggers this cascade. Domestic partnership breakups, dating breakups, and data corrections leave in-laws and step-relations valid. Users can declare-invalidate manually if appropriate.

**Cadence:** Trigger on `ROMANTIC` invalidation where `type='married'` AND `invalidation_reason='divorce'`

## `known/lifecycle/mentor_invalidator` (cron)

Invalidates stale inferred `MENTOR` edges. Declared mentorships are never touched by this cron — only the user can end a declared mentorship.

**Scans** all valid `MENTOR` edges where `source='inferred'` AND `valid_to IS NULL` AND `last_direct_interaction_at` on the underlying `KNOWS` edge is older than 180 days.

**For each stale inferred MENTOR:**
1. Set `valid_to = now()`
2. Set `invalidation_reason = 'ended'`
3. Set `invalidated_by = 'mentor_invalidator_cron'`

The mentee or mentor may reconnect later, at which point a new inferred `MENTOR` row can be created by the LLM relationship classifier if the asymmetric interaction pattern re-emerges. The old invalidated row remains as history — producing a clean chain of "mentorship was active 2022–2024, resumed 2026" across multiple rows on the same pair.

**Does NOT fire on:**
- Declared mentorships (`source='declared'`) — only users can end these
- YC-partner mentorships specifically scoped to a company (`company_uuid IS NOT NULL` AND `domain='accelerator'`) — the mentorship persists as a historical fact even if the partner stops actively coaching the founder. Reason: the YC partner relationship is a category of its own and "ended" isn't the right semantics.

**Cadence:** Nightly
**Depends on:** Counts updated first (so `last_direct_interaction_at` on the KNOWS edge is current)
**Tunable parameters:**
- `MENTOR_STALE_DAYS` = 180 (default)
- Exclusion list of `domain` values (default: `['accelerator']`)

**Interaction with the LLM relationship classifier:** After invalidation, if the LLM classifier later re-runs on the same pair and detects continued asymmetric interaction patterns, it can create a new inferred `MENTOR` row. The 180-day threshold is the floor — the classifier can still propose a new mentorship immediately if the signals return.

## `known/lifecycle/dedupe_knows_edges` (cron)

Safety net for the narrow concurrent-write race window that `MERGE` in `create_knows_edge` cannot close. Collapses any canonical pair that somehow has more than one currently-valid `KNOWS` edge into a single surviving row — the oldest — absorbing the duplicates' evidence arrays into the keeper.

**Why this exists:** FalkorDB does not support `CREATE CONSTRAINT ... IS UNIQUE` on relationships, and `MERGE` on relationships is atomic-within-a-query but not serializable across concurrent queries. Two racing `create_knows_edge` calls for the same canonicalized pair can each evaluate the MATCH pattern, each see nothing, and each create. The window is small (single-digit ms), but non-zero. This cron is the nightly sweep that cleans up anything that slipped through. See [ingest → KNOWS uniqueness enforcement](/guides/known/scope/ingest#knows-uniqueness-enforcement) for the full defence-in-depth story.

**Algorithm:**

<Steps>
  <Step title="Find duplicate pairs">
    Scan all `KNOWS` edges where `valid_to IS NULL`. Group by canonical `(person_a_id, person_b_id)`. Any group with `count > 1` is a duplicate set.
  </Step>
  <Step title="Pick the survivor">
    The oldest row by `created_at` wins canonical. All other rows in the group are the duplicates to be absorbed.
  </Step>
  <Step title="Merge evidence arrays">
    For each duplicate, parse its `evidence` JSON array and call `append_evidence` against the survivor for every row that's not already present (dedupe on `(source, evidence_occurred_at, raw_metadata_hash)`). This preserves the full evidence history — no rows are lost.
  </Step>
  <Step title="Re-point Introduction.resulting_knows_uuid">
    For each duplicate's uuid, run `MATCH (i:Introduction) WHERE i.resulting_knows_uuid = $duplicate_uuid SET i.resulting_knows_uuid = $survivor_uuid`. This uses the `Introduction.resulting_knows_uuid` index from #4669.
  </Step>
  <Step title="Delete the duplicate KNOWS rows">
    Hard-delete each absorbed `KNOWS` edge. This is the one place in the Known write path where a hard delete is allowed — the rows are literal duplicates created by a race condition, not historically-meaningful state transitions, so invalidation-via-`valid_to` would be wrong.
  </Step>
  <Step title="Log the absorption">
    Write a structured log entry with `survivor_uuid`, `absorbed_uuids`, `evidence_rows_merged`, `introductions_repointed`. If any group had `count > 5`, page on-call — that's evidence of a pathological write loop, not a normal race.
  </Step>
</Steps>

**Cypher:** See [known cyphers → dedupe_knows_edges](/guides/cyphers/known#lifecycle-dedupe-knows-edges).

**Cadence:** Nightly (0 3 \* \* \* local time, during the scoring cron quiet period).

**Calls:** `append_evidence` (per merged evidence row), `mvp/falkor/send-cypher` (for the scan, re-point, and delete queries).

**Safety guarantees:**
- Survivor is always the oldest edge — deterministic, no data-dependency on concurrent writes.
- Evidence is always additive — no dedupe collapses can lose a row.
- The cron is idempotent: re-running on a clean graph is a no-op.
- Introduction re-pointing uses the indexed `resulting_knows_uuid` so even a large duplicate set processes in ms.

**Not built yet.** Spec only. Add to Phase 1 build order as a lifecycle cron — not blocking the initial deploy since the empty graph has no duplicates to collapse, but should be live before production ingest volume ramps.

## `known/lifecycle/release_stale_merge_locks` (cron)

Releases `merge_in_progress_at` locks on `Person` nodes that have been held for longer than 10 minutes. Handles the case where an `execute_merge` call crashed mid-execution and left the lock set, blocking subsequent merges against the same canonical.

**Cypher:** [release_stale_merge_locks janitor](/guides/cyphers/known#lifecycle-release-stale-merge-locks-janitor).

**Cadence:** Hourly
**Tunable:** `MERGE_LOCK_STALE_MINUTES` (default 10)
**Part of:** [Identity Merge — Concurrent merge handling](/guides/known/scope/identity-merge#concurrent-merge-handling)

## `known/lifecycle/deceased_freezer` (trigger)

Runs when `Person.deceased_at` is set. Cascades death-handling across multiple facet types.

<Steps>
  <Step title="Freeze KNOWS edges">
    Scan all `KNOWS` edges touching the deceased person. Set `score_frozen_at = deceased_at` and `score_frozen_reason = 'deceased'`. Further scoring passes skip frozen edges. Interaction counts continue to be updated if historical data is later ingested, but the score does not move.
  </Step>
  <Step title="Invalidate ROMANTIC facets">
    For every currently-valid `ROMANTIC` facet touching the deceased person, set `valid_to = deceased_at`, `invalidation_reason = 'death'`, `invalidated_by = 'deceased_freezer'`. Flip `is_current = false` atomically.
  </Step>
  <Step title="Close RESIDED_AT residencies">
    For every currently-valid `RESIDED_AT` edge owned by the deceased person (i.e. `(deceased)-[:RESIDED_AT]->(household)`), set `move_out = deceased_at`. The household's other residents are unaffected. If this was the last active resident, the `Household.end` field is set by the downstream household maintenance logic.
  </Step>
  <Step title="Leave BONDED valid">
    BONDED facets on the deceased person's pairs are **NOT** invalidated. They stay valid with the frozen score. Semantically, "the bond existed" remains true historically. Queries wanting "currently alive contacts" filter on `Person.deceased_at IS NULL`.
  </Step>
  <Step title="Leave KINSHIP in-laws valid">
    In-law KINSHIPs whose via_chain includes the deceased person are **NOT** invalidated. The `kinship_divorce_cascader` trigger specifically checks for `invalidation_reason='divorce'` and is not fired by `'death'`.
  </Step>
  <Step title="Leave PROFESSIONAL_TIE valid (but frozen)">
    PROFESSIONAL_TIE facets are not invalidated either. The frozen score already reflects the lack of future interaction.
  </Step>
</Steps>

**Cadence:** Trigger on `Person.deceased_at` set

**Ordering with `kinship_divorce_cascader`:** `deceased_freezer` MUST set `invalidation_reason='death'` on ROMANTIC facets. The `kinship_divorce_cascader` trigger must check the reason before firing — it only fires on `invalidation_reason='divorce'`, so the death case is naturally bypassed.

## Key rules

- **Declared facets are immutable to crons.** Every facet created by these crons has `source='inferred'`. Facets with `source='declared'` are only touched by user or LLM actions.
- **Hysteresis everywhere.** BONDED uses 70/55 (create/invalidate). Estrangement uses 3-signal bar. Reconciliation uses sustained-interaction bar. These asymmetric thresholds prevent flapping.
- **Invalidations preserve history.** Facets are never deleted. `valid_to` is set; the row remains for historical queries.
- **Cascade on state changes.** Some state changes require cascade logic (estrangement cascades to BONDED, divorce cascades to in-law kinships, death cascades to romantic). These are encoded in the specific crons/triggers above.
