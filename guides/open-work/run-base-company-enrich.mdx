## Refactor plan — v3.5 → v4

### Goals

1. **Front-load an LLM preflight.** Read `company_enrich_data.company_llm` for the target record. If empty (the common case for any record created before [new-company-enrichment](/guides/open-work/new-company-enrichment) shipped on 2026-05-11), run `mvp/enrich/new-company-enrichment` (#12974) inline and persist the response into `company_enrich_data.company_llm`. If populated, skip.
2. **Apply the LLM payload to `master_company`** when the LLM was freshly run. Mirrors Section 10a/10b of `mvp/get-add/master-company` — writes industries, specialties, abouts, addresses, contact email/phones, profile URLs (LLM-discovered + Serper-augmented), and LLM-derived scalars. Extracted into a shared helper so master-company's Section 10 and v4 call the same code.
3. **Fundable re-lookup when `fundable_org_id` is null** after the LLM ran. Use the LLM-discovered profile URLs + domain to re-attempt the `fundable_org` match. On hit, link both directions and set `master_company.is_vc = true` only when the matched `fundable_org` classifies as a VC firm.
4. **Signal NFX scrape (company-side)** as a new gated step. Guard on `company_enrich_data.signal_nfx_json`. If empty, call a new `mvp/investor/get-signal-nfx-data-company` function (modeled on the person-side #2769) that finds the Signal URL via Brave (#2640), Firecrawls it, stores raw markdown in `company_enrich_data.signal_nfx_scrape`, parses to JSON via LLM (port the prompt from archived #2698), and stores the JSON in `company_enrich_data.signal_nfx_json`.
5. **Investment thesis build/rebuild.** Once `fundable_org_id` is set (either at orchestrator entry or after the re-lookup step), check `investment_theses` by `master_company_id`. If missing OR last updated more than 30 days ago, call `thesis/build-investment-thesis-in-gcp` (#12978) — the same function for both cases (it uses `db.add_or_edit` internally).
6. **Get C-Suite & Founders.** Guard on `company_enrich_data.exa_c_suite` presence. If empty, call `mvp/enrich/get-exa-company-c-suite` (#12988), then loop the results and create master_person rows one at a time via `mvp/get-add/master-person`. If populated, skip the entire section (Exa call + loop). _(Note: `mvp/get-add/master-person` (#12553) is deprecated as of 2026-06-01; the current Exa C-suite person-creation path uses `mvp/get-add/master-person-from-exa` (#12997).)_
7. **Delete dead PDL / Enrich Layer / inline name-correction code paths** — already removed in v3.5 (2026-05-11) at the orchestrator level. v4 keeps them out by design.
8. **Bump `update-company-node` (#4659) to v1.6** to read the LLM-derived FalkorDB-bound properties (`business_model`, `primary_product_name`, `target_customers`, acquisition fields, etc.) from `company_enrich_data.company_llm` so backfilled records get those node properties.
9. **Extend the new-company-enrichment LLM schema** to include `funding.acquired.went_public_on` (IPO date) and `stock_link` (TradingView URL) so the apply-LLM-payload helper can populate every field the orphaned #2811 (`mvp/revenue/revenue-employees-ownership`) used to write — without ever calling it.
10. **Cut as a new function** `mvp/enrich/run-base-company-enrich-v4` so v3.5 stays available for rollback. Update Section 11 of `mvp/get-add/master-company` to point to v4 once smoke-tested.
11. **Active backfill** of every pre-2026-05-11 record with empty `company_llm` via `mvp/queue/upsert-enrich-company` — workers process the queue at a rate that respects OpenRouter and Firecrawl limits.

The orchestrator stays best-effort: every new step gets its own `try_catch` and feeds `$hasCrash` the same way every existing step does.

### Post-launch v4 changelog (2026-05-13)

Three patches landed today on `mvp/enrich/run-base-company-enrich-v4` (#12992):

- **60s debounce at top of stack.** Before any other work, the orchestrator now queries `enrich_history_company` for any row with the same `master_company_id` + `data_source_id == 78` + `created_at > now - 60_000`. If found, writes one `log_crash` entry (`note: "DUPLICATE TRIGGER, debounced (60s window)"`) and returns null without writing a new history row. Prevents back-to-back double-fires (queue + inline dispatch, webhook re-delivery, etc.) from producing duplicate `enrich_history_company` rows. Caveats: true simultaneous (`<10ms`) concurrent fires can still slip through — the SELECT-then-INSERT isn't atomic without a unique constraint. A genuine manual retry within 60s is also blocked; null the recent row's `created_at` to force re-run. Discovered after auditing Adobe's `enrich_history_company` on staging revealed two ds=78 rows 500ms apart for `master_company_id=124`.
- **Exa C-Suite loop — `deep_research: true`.** Step 7's `get-add/master-person` call now passes `deep_research: true` so Phase 6 of the downstream person orchestrator runs the deep-research path for every Exa-discovered C-suite person. Mirrors the parallel change to `mvp/yc/process-yc-people` v3.0 (YC founders + partner). _(Note: `mvp/get-add/master-person` #12553 is deprecated 2026-06-01; the Exa C-suite path now routes through `mvp/get-add/master-person-from-exa` #12997.)_
- **Exa C-Suite loop — seed-like inputs.** Same call site now passes `data_source_id: 101` (Exa), `cascade_depth: 0`, `priority_tier: 1`, replacing the prior `data_source_id: <unset>` / `cascade_depth: parent + 1` / `priority_tier: 2`. `cascade_depth: 0` unlocks PDL + Enrich Layer inside `mvp/get-add/master-person` (still used person-side; v3.5 only removed them company-side) and triggers downstream sub-cascades from each C-suite person (past employers, colleagues). Cost-per-company goes up; tradeoff accepted to give Exa-discovered C-suite the same enrichment depth as user-seeded persons.

### Where the funding/investor flow actually lands — relational tables to ignore

A sandbox audit (2026-05-13) of Frame.io's 4 funding rounds confirmed that the v4 → Phase 7 → `add-all-fundable-deals` → `cascade-deal-participants` → `add-fundable-deal-node` / `resolve-investors-edges` chain writes **only to FalkorDB**. Specifically:

- `Funding_Round` nodes (+ `fr.uuid = fundable_deal.id`) — created by `add-fundable-deal-node` (#12701).
- `RAISED` edge (Company → Funding_Round) — created by the same.
- `INVESTED_IN` / `LEAD_INVESTED_IN` / `FOLLOW_ON_INVESTED_IN` edges (VC_Firm → Funding_Round) — created by `resolve-investors-edges` (#12702).
- `INVESTMENT_PARTNER_IN` / `INVESTMENT_PARTNER_AT` edges (Person → Funding_Round / Person → VC_Firm) — same.

**Relational by-products from the chain:**

- `fundable_deals.processed_at` — stamped at end of `cascade-deal-participants`.
- `fundable_deals.funding_round_node_uuid` — back-reference to the Funding_Round node (same value as `fundable_deal.id`).
- `fundable_organizations.master_company_id` / `master_company_node_uuid` — back-references when the investor org is resolved.
- `fundable_people.master_person_id` / `master_person_node_uuid` — back-references for partners + angels.
- `master_company.is_vc = true` — set when the matched `fundable_organizations.is_investor == true`.
- `master_person.is_angel = true` — set on angels (and partner persons re-flagged here).

**Tables this flow does NOT write to** — don't expect rows in them after running v4:

**Verifying funding landed for a company:** query FalkorDB:

```cypher
MATCH (c:Company {uuid: $company_uuid})-[:RAISED]->(fr:Funding_Round)
OPTIONAL MATCH (investor)-[r:INVESTED_IN|LEAD_INVESTED_IN|FOLLOW_ON_INVESTED_IN]->(fr)
OPTIONAL MATCH (partner:Person)-[:INVESTMENT_PARTNER_IN]->(fr)
RETURN fr.funding_type, fr.announced_date, fr.size_usd,
       collect(DISTINCT investor.name) AS investors,
       collect(DISTINCT partner.name) AS partners
```

Don't infer "funding rounds didn't land" from `company_funding_round` being empty — that's expected.

### LLM schema extension (new-company-enrichment)

Two fields to add to the `mvp/enrich/new-company-enrichment` (#12974) JSON schema + system prompt before v4 ships:

- **`funding.acquired.went_public_on`** — IPO date in `YYYY-MM-DD` when the company has gone public. Null otherwise. Sourced from SEC filings, press releases, or the company's investor-relations page.
- **`stock_link`** — TradingView URL (`https://www.tradingview.com/symbols/{EXCHANGE}-{TICKER}/`) for currently-trading public companies. Null otherwise. Constructible from the top-level `ticker` field; including it explicitly removes ambiguity for the apply-payload helper.

### Low-confidence fallback chain

When the LLM preflight runs (step 2), the model chain is:

1. **Gemini 2.5 Flash `:online`** — default. Cost ~$0.005–$0.015 per call.
2. **Claude Sonnet 4.6 `:online`** — escalation when `gemini_confidence < 0.7`. Cost ~$0.04–$0.08 per call.
3. **Website scrape + 3rd LLM call (Sonnet 4.6, no `:online`)** — fires when Sonnet's `confidence` is still `< 0.7` AND `master_company.company_domain` is non-empty. Scrapes the homepage + `/about` + `/contact` via Firecrawl, injects the extracted text into a third LLM call as authoritative context, and re-extracts the JSON schema. The LLM still does the structured extraction; the scrape just gives it the source-of-truth content the web grounding missed.
4. **Accept result, no crash flag** — when the scrape-LLM tier also stays low confidence OR no domain is available, save whatever was returned and continue. Low confidence is information, not failure; downstream consumers can filter on `_meta.confidence`.

### Fundable-deals idempotent re-sync

Fundable updates its underlying tables continuously — new deals land, existing deals get amended, and new investor rows get attached to existing deals (sometimes without bumping the parent deal's `updated_at`). The v4 orchestrator must leave the graph fully in sync with Fundable's current state on every run, not just the first one.

**Mechanism: graduate `fundable_deals.processed_at` from a binary "processed" gate to a staleness comparison.**

Today `fundable_deals.processed_at` is set when `cascade-deal-participants` materializes a deal into the graph; if non-null, the cascade skips that deal forever. That's "process once" — it misses every subsequent Fundable update. The fix: compare timestamps instead of checking for null.

**A deal needs re-processing when any of these are true:**

```sql
deal.processed_at IS NULL                                                  -- new deal, never processed
OR deal.updated_at > deal.processed_at                                     -- Fundable updated the deal itself
OR EXISTS (
     SELECT 1 FROM fundable_institutional_investments inv
     WHERE inv.deal_id = deal.id
       AND inv.updated_at > deal.processed_at                              -- new or updated investor on this deal
   )
OR EXISTS (
     SELECT 1 FROM fundable_institutional_investments_person inv_p
     JOIN fundable_institutional_investments inv ON inv.id = inv_p.institutional_investment_id
     WHERE inv.deal_id = deal.id
       AND inv_p.updated_at > deal.processed_at                            -- new partner link on this investment
   )
```

**Where the gate lives:** inside `mvp/investor/cascade-deal-participants` (#12856), at the top of the function before any graph writes. Single source of truth — `process-company-phase-7` (v4 step 11), `process-person-phase-9`, and any future caller all get idempotent re-sync for free.

**Re-process strategy:** trust MERGE idempotency. Re-running the same `MERGE (fr:Funding_Round {id: $dealId}) ... MERGE (org)-[:INVESTED_IN]->(fr)` lands the node + edges in place; no duplicates. After a successful re-process, update `fundable_deals.processed_at = now`.

**Known limitation:** this strategy does not detect investor *removals* from a deal in Fundable (rare). If an `INVESTED_IN` row disappears from `fundable_institutional_investments` between syncs, the corresponding graph edge will remain orphaned. Accept this for v4; revisit if Fundable's investor-removal rate turns out to matter.

**Verification step before deploy:** confirm #12856 uses `MERGE` (not `CREATE`) for every Funding_Round node and every `INVESTED_IN` / `LEAD_INVESTED_IN` / `INVESTMENT_PARTNER_IN` edge. If any `CREATE` slipped in, re-running would create duplicates. The function description claims MERGE everywhere; verify by reading the XanoScript before patching the gate.

## Final logging

After all steps complete (regardless of how many crashed mid-run):

- `enrich_history_company` row's `enrich_success` is set based on `$hasCrash`: `true` if every step succeeded, `false` otherwise. `data_source_id: 101`.
- On clean success, a `log_crash` row is also written with `qa_passed: true` (the only `qa_passed: true` row this orchestrator writes — every other path crashed).

## Why each step is wrapped in `try_catch`

The orchestrator is designed to be **best-effort**, not all-or-nothing. A failing YC scrape shouldn't block Fundable backfill; a failing `update-company-node` shouldn't block specialties resolution. Every failure surfaces in `log_crash` with the step name, but execution continues. The `$hasCrash` flag at the bottom is the single signal that downstream consumers (or QA dashboards) can use to know "this record finished but with errors — needs review."

## Triggering it

`run-base-company-enrich-v3` is called once, automatically, by Section 11 of `mvp/get-add/master-company` (#12558) immediately after a new `master_company` row is created, unless the caller passed `queue: true` or `cascade_depth > 0` (in which case the record is queued via `mvp/queue/upsert-enrich-company` and a separate worker picks it up later).

```text Section 11 trigger
conditional {
  if ($companyAdded && $shouldQueue == false) {
    function.run "mvp/enrich/run-base-company-enrich-v3" {
      runtime_mode = "async-shared"
      input = {
        master_company_id: $masterCompanyID
        cascade_depth    : $effectiveDepth
      }
    }
  }
  elseif ($companyAdded && $shouldQueue) {
    function.run "mvp/queue/upsert-enrich-company" { ... }
  }
}
```

A `debug.stop` currently sits just above this conditional in v2.4 of master-company so we can smoke-test new-company-enrichment + Section 10 writes without the downstream orchestrator firing. Remove the `debug.stop` block once you're satisfied with the master-company QA pass. When v4 ships, this trigger gets updated to point at `run-base-company-enrich-v4`.

## Known characteristics

- **Async, fire-and-forget.** The caller doesn't wait for the orchestrator to finish. Failures only surface via `log_crash`.
- **Specialties resolution is the biggest external-cost step.** It embeds every distinct specialty (up to 15 per company) and, for unmatched specialties, generates a description via Claude Sonnet 4 before creating the FalkorDB node. Most cost concentrates here in v3.5. After v4, the LLM preflight + Signal NFX scrape add ~$0.02–$0.10 per backfill record on top.
- **Pre-2026-05-11 records will not have `company_llm` or `signal_nfx_json` populated** until v4's active backfill drains the queue. They run on v3.5 today with reduced fidelity (no LLM-derived business_model, target_customers, revenue, employee_range, contact email/phone).
