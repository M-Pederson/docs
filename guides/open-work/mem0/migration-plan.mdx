This page describes the target architecture for moving Orbiter AI chat memory onto Mem0, and the steps to get there. It builds on the deterministic Xano context functions and the dispatch path documented in [Current Implementation](/guides/open-work/mem0/current-implementation) — in particular the `#13069` user-context and `#13070` meeting-prep functions referenced throughout.

## Decision

Use Mem0 as a **retrieval layer for distilled memories**, not as the transcript store and not as the canonical profile store.

For the current OpenUI05 implementation, Mem0 should replace or augment the Zep read/write steps inside `openui05/dispatch`. Keep the frontend contract the same: the frontend sends `query`, stable `thread_id`, and YAML context blocks to Xano; Xano owns memory retrieval, LLM/tool routing, and memory writes.

For the target Mem0 architecture, Xano should remain the source of truth for:

- raw user and assistant messages, if first-party transcript persistence is added
- flow state for outcomes, leverage loops, meeting prep, and how-I-know chats
- deterministic context snapshots generated from Orbiter tables/functions

Mem0 stores:

- durable user preferences and constraints
- important decisions made during a flow
- run-specific working assumptions
- distilled memories from chat, email, calendar, meeting prep, and how-I-know context

Do not send every raw turn or full Xano context dump to Mem0.

## Supported Chat Surfaces

Use the same memory architecture for these surfaces:

- `outcome`: defining or refining an outcome.
- `leverage_loop`: defining or refining a leverage loop.
- `meeting_prep`: preparing for a meeting, including participants, context, questions, and follow-ups.
- `how_i_know`: capturing how the user knows a person and what relationship context should matter later.

Each surface has its own Xano thread/flow state, but they all use the same Mem0 read/write pattern.

## Entity Scope Rules

Use **one primary Mem0 entity per write**.

| Scope | Mem0 entity | Use for |
| --- | --- | --- |
| User memory | `user_id` | Durable facts that should follow the user across flows |
| Run memory | `run_id` | Facts that should only shape one outcome, leverage loop, meeting prep, or how-I-know chat |
| Agent memory | `agent_id` | Rare global behavior notes about an agent, not user-specific facts |

Do not write a single memory with both `user_id` and `run_id` unless you intentionally want Mem0's entity scoping behavior. For Orbiter, if a fact matters at both levels, write two memories: one user-scoped and one run-scoped.

Use metadata for everything else:

```json
{
  "flow_type": "leverage_loop",
  "thread_id": "suggestion_request:123",
  "source_type": "chat",
  "source_id": "copilot_message:456",
  "user_id": "15"
}
```

## Xano Persistence

Today, `openui05/dispatch` does not appear to store a complete transcript in Xano. It stores conversational memory in Zep and returns response metadata to the frontend.

If we add first-party transcript persistence for OpenUI05, either reuse the existing copilot tables or introduce a chat-specific set of tables. The existing copilot tables are:

```txt
copilot_conversations (#636)
- id
- created_at
- updated_at
- user_id
- mode
- title
- master_person_id
```

```txt
copilot_chat_messages (#637)
- id
- created_at
- conversation_id
- role
- content
- template_name
- template_props
```

For outcome/leverage-loop chats that are naturally keyed by `suggestion_request_id`, the existing schema would need either a deterministic `mode/title` convention or an added/linking field. Do not assume `suggestion_request_id` exists on `copilot_chat_messages`; the live table schema checked on 2026-06-08 does not include it.

Recommended table shape:

```txt
ai_chat_message
- id
- user_id
- flow_type              outcome | leverage_loop | meeting_prep | how_i_know
- thread_id              suggestion_request:123 | meeting_prep:88 | how_i_know:3:139

- content
- metadata
- created_at
```

```txt
ai_chat_context_snapshot
- id
- user_id
- flow_type
- thread_id
- context_markdown
- context_source         xano_context_function | meeting_prep_context | how_i_know_context
- context_version
- generated_at
```

```txt
ai_memory_event_log
```

## Turn Lifecycle

This is the target Mem0 lifecycle. For the current implementation, map "Mem0 search" to the place where `openui05/dispatch` now reads Zep thread context, and map "Mem0 write" to the place where it now posts messages back to Zep.

### 1. Start or resume a chat

1. Create or load the Xano thread state.
2. At draft creation, run `mvp/context/get-all-user-context` (`#13069`) with `{ user_id: auth.id }`.
3. Store the Markdown snapshot in `suggestion_request.llm_summary_context.openui_context.self_markdown`.
4. On every chat turn, feed that stored snapshot into `openui05/dispatch` as `self_yaml_context`, or have dispatch load it server-side from `suggestion_request_id`.
5. Do not write the full context snapshot to Mem0.

For `outcome` and `leverage_loop`, the thread ID must be stable for the lifetime of the `suggestion_request`. The current OpenUI05 comment references:

```txt
thread-outcome-{suggestion_request_id}
```

A cleaner future convention is:

```txt
suggestion_request:{suggestion_request_id}
```

Do not change the convention without migrating existing Zep/Mem0 thread keys.

For `meeting_prep`:

```txt
meeting_prep:{meeting_prep_context_id}
```

For `how_i_know`:

```txt
how_i_know:{user_master_person_id}:{target_master_person_id}
```

### 2. Before each LLM response

Load:

1. System prompt for the surface.
2. Latest context snapshot from Xano.
3. Structured flow state from Xano.
4. Recent transcript from Xano, usually last `8-16` turns.
5. Relevant Mem0 memories from user scope.
6. Relevant Mem0 memories from run scope.

Build the model input in this order. In the existing OpenUI05 path, `$mem_context` already plays this role for the downstream tool endpoints.

```txt
System instructions

Deterministic Orbiter context snapshot from Xano

Current flow state from Xano

Relevant durable user memories from Mem0

Relevant run memories from Mem0

Recent chat transcript from Xano

Latest user message
```

### 3. After each LLM response

1. Save user and assistant messages to Xano if first-party transcript persistence is enabled.
2. Run memory extraction asynchronously.
3. Save extractor output to `ai_memory_event_log`.
4. POST only approved high-confidence memories to Mem0.
5. Poll or receive the Mem0 event result and update `ai_memory_event_log`.

If memory extraction fails, the chat still succeeds.

## When Orbiter Calls Mem0

Use this rule across Orbiter:

```txt
Read from Mem0 right before an LLM response.
Write to Mem0 after the LLM turn, only if memory extraction finds something durable.
```

Mem0 is recall. It is not the place to store full Xano context snapshots, participant JSON, raw emails, raw meeting notes, or raw transcripts.

| Moment | Call Mem0? | Reason |
| --- | --- | --- |
| User opens calendar | No | Calendar browsing does not need memory recall. |
| Nylas webhook receives or updates a meeting | No | Webhooks should keep event data fresh; they should not trigger memory retrieval by themselves. |
| User chooses meeting prep | Usually no | Create/load `meeting_prep_context`, run `#13070` if needed, and store the meeting memo. If this same action immediately asks an LLM to generate prep, then search Mem0 right before that LLM call. |
| User sends the first meeting-prep chat message | Yes, search | Retrieve relevant user and run memories before building the LLM messages. |
| Every later chat turn | Yes, search | Each turn may need different memories depending on the user's latest question. |
| User opens a contact side panel | No | Load the stored relationship summary from Xano; memory recall is only needed when an AI turn starts. |
| User starts or continues a How I Know X chat | Yes, search | Retrieve prior relationship memories before the interviewer asks the next question or summarizes. |
| After the assistant responds | Maybe write | Run extraction and write only durable, high-confidence memories. |
| Running `#13069` user context | No | This is deterministic Xano context, not memory recall. |
| Running `#13070` meeting participant context | No | This is deterministic meeting context, not memory recall. |

### Leverage loops

For a leverage loop that helps a specific person, search Mem0 before each LLM turn using the target person's name plus the active objective. Do not retrieve every memory that mentions the person.

Example: the user is creating a leverage loop to help Ray Deck find strategic partners for AI media distribution.

First, Xano prepares deterministic context:

```txt
1. Create or load suggestion_request.
2. Run mvp/context/get-all-user-context (#13069), if the suggestion_request self context is missing.
3. Load Ray Deck target context from Xano.
4. Load relationship context for the user and Ray Deck.
5. Load Ray Deck company context, if relevant.
6. Load the current leverage loop draft state from suggestion_request.
```

The Ray Deck profile, relationship summary, and company context should come from Xano context functions and tables. Mem0 only adds relevant recall.

Before the LLM turn, search durable user memories:

```txt
Search user_id: 15
Query: "Ray Deck leverage loop helping Ray find strategic partners for AI media distribution"
```

Also search the active run:

```txt
Search run_id: suggestion_request:123
Query: "Ray Deck leverage loop helping Ray find strategic partners for AI media distribution"
```

Build the LLM input in this order:

```txt
System prompt

#13069 user context

Ray Deck deterministic target context from Xano

User/Ray relationship summary from Xano

Ray Deck company context from Xano, if relevant

Current leverage loop state from suggestion_request

Relevant Mem0 memories

Recent leverage-loop chat transcript

Latest user message
```

Useful Mem0 results might look like:

- `User knows Ray Deck through a previous AI/media conversation.`
- `User prefers helping Ray Deck through warm introductions, not generic outreach.`
- `Ray Deck previously cared about distribution partners more than investors.`
- `For suggestion_request:123, user rejected enterprise SaaS buyers and wants media operators.`

After the assistant responds, run memory extraction. Save only durable memory, such as:

- a confirmed user preference about how to help Ray
- a relationship fact about Ray that should matter later
- a rejected path for this specific leverage loop
- a run-specific decision that should shape the rest of the draft

Do not save Ray Deck's full profile, full Xano target context, generated candidate lists, or every brainstormed suggestion to Mem0.

### How I Know X

How I Know X is the contact-side-panel chat where the user explains how they know a specific person. The product goal is to turn a lightweight reflective interview into:

- a side-panel relationship summary stored in Xano
- durable relationship memories stored in Mem0
- better future context for meeting prep, leverage loops, introductions, and recommendations

The current Xano pull shows a `how_i_know_x` table:

```txt
how_i_know_x
- id
- created_at
- my_person_id
- display_test
```

The active Wave-35 relationship interview implementation also references `master_person_relationship`, which stores:

```txt
master_person_relationship
- user_id
- master_person_id
- transcript
- relationship_summary
- interview_complete
```

Recommendation: decide whether `how_i_know_x` becomes the canonical side-panel table or whether `master_person_relationship` remains canonical. Do not keep two divergent relationship stores long term.

If `how_i_know_x` becomes canonical, expand it so it can support the AI chat and display surface:

| Field | Recommendation |
| --- | --- |
| `my_person_id` | Keep if the side panel is keyed by the user's network row. |
| `user_id` | Add or derive reliably from `my_person_id`; needed for ownership and Mem0 scope. |
| `master_person_id` | Add or derive reliably; needed for target context and stable thread IDs. |
| `display_test` | Rename to `display_text` or replace with `relationship_summary`. The current name looks like a typo or placeholder. |
| `transcript` | Add if this table owns the interview history. |
| `interview_complete` | Add so the UI knows whether to keep asking or show the finished summary. |
| `summary_updated_at` | Add so stale summaries can be refreshed deliberately. |
| `mem0_memory_ids` | Add as JSON if we want traceability from side-panel summary to created Mem0 memories. |
| `metadata` | Add JSON for `last_exchange_count`, `source_context_version`, and model/debug fields. |



```txt

[YOUR COMPANY — the user's current company]

```

It sends the transcript as the user message:

```txt

{transcript}

Return ONE JSON object honoring the contract. If the territory is covered or the user signals done, set ready=true and put the closing reflection in summary.
```

#### Current system prompt

<Accordion title="Full How I Know X interviewer system prompt">

```txt
You are a warm, curious interviewer helping the user reflect on their relationship with a specific person. Your job is to draw out a textured, honest portrait of who this person is in the user's life — not a resume, but a relationship.

What you are trying to learn (cover several across the interview, don't ration mechanically):
- Origin: how, when, where they first met. The actual moment, not the timeline.
- Connective tissue: what keeps them in each other's orbit.
- Cadence: how often they connect, in what mediums, who initiates.
- Trust: would the user put them on speed-dial in a crisis? Vouch for them? Lend them $10K?
- Superpower: what is this person unusually good at?
- Kryptonite: where do they struggle, get tripped up, have a blind spot? Honest weakness, not humble-brag.
- Affective signal: how does the user actually feel when this person's name comes up? Energized, drained, warm, wary, complicated?
- Trajectory: is this relationship growing, steady, or fading?

Length budget: aim for 8-12 exchanges total. By exchange 6 you should have hit origin + 3 substantive dimensions; by exchange 10 you should be circling closure. Never exceed 14 exchanges without wrapping.

Style:
- One question at a time. Wait. Follow the thread before opening a new one.
- Use the target's first name. Conversation with a friend, not HR intake.
- Pull on specifics. If they say 'she is brilliant', ask 'brilliant how — give me a moment that showed it'.
- Don't moralize, don't validate every answer, don't over-summarize.
- Complicated, ambivalent, or unflattering answers are signal, not friction.
- Don't lead. Don't bury the answer in the question.
- If an answer is thin, probe gently once. If the user wants to move on, move on.

Open: start with something specific to the target from the context to show you've done your homework. Then ask how they met. Avoid 'tell me about them'.

Close: when you've covered the territory (or the user signals done), thank them and offer a short reflection of what you heard — the SHAPE of the relationship, not a bulleted list.

JSON CONTRACT (you MUST emit ONE JSON object, no prose, no fences):

When still interviewing: ready=false. Provide next_question (one warm question), options as an empty array (this is a freetext interview, not MCQ — force the user to type), allow_freetext=true, multi_select=false, summary (running short paragraph capturing the relationship shape so far), exchange_count (count of user turns so far).

When closing: ready=true. Provide summary = the closing reflection (2-4 sentences capturing the shape of the relationship, not a bulleted list), next_question='', options=[], allow_freetext=false, multi_select=false, exchange_count.

Fallback if JSON fails: ready=false, next_question='Say that one more way?', options=[], allow_freetext=true, multi_select=false, summary='', exchange_count=0.
```

#### Interview flow

When the user opens the contact side panel, do not call Mem0 just to display the panel. Load the existing Xano summary.

When the user starts or continues the How I Know X chat, build an AI interviewer turn:

```txt
1. Create or load the how_i_know_x row for the selected person.
2. Load target person context from Xano.
3. Load existing relationship_summary/display_text, if present.
4. Load recent transcript from Xano.
5. Search Mem0 for relevant relationship memories.
6. Ask one focused question or produce the final summary.
```

The interviewer should prompt for relationship details that matter later:

- how, when, and where they met
- what keeps them connected
- how often they interact and who initiates
- the person's superpower
- the person's weakness, blind spot, or "kryptonite"
- what the user would trust them with
- whether the relationship is growing, steady, fading, warm, or complicated
- what kinds of asks, intros, or opportunities fit this person
- any boundaries or sensitivities the assistant should remember

#### Mem0 search

Before each How I Know X AI turn, search Mem0 with the target person plus the relationship dimension being discussed.

Example:

```txt
Search user_id: 15
Query: "How the user knows Ray Deck relationship origin superpower weakness trust intro context"
```

Also search the run scope for the active interview:

```txt
Search run_id: how_i_know:15:139
Query: "Ray Deck how I know relationship interview origin superpower weakness"
```

Use the retrieved memories to avoid asking for things Orbiter already knows and to help the interviewer ask better follow-up questions. Do not let Mem0 override the user's new answer; if the user corrects a memory, treat the correction as the newest source of truth.

#### LLM input

Build the How I Know X chat turn in this order:

```txt
System prompt for warm relationship interview

#13069 user context, when available

Target person context from Xano

Existing how_i_know_x relationship summary/display text

Relevant Mem0 relationship memories

Recent How I Know X transcript

Latest user answer
```

The chat should ask one question at a time. It should explicitly cover the person's superpower and weakness before closing, unless the user refuses or the relationship context makes those questions inappropriate.

#### Summary and memory writes

When the interview reaches `ready=true`, create a concise LLM relationship summary from:

- the full interview transcript
- target person context from Xano
- existing relationship summary/display text
- relevant Mem0 memories
- the user's newest corrections

Store that final summary in Xano for side-panel display:

```txt
how_i_know_x.display_text or how_i_know_x.relationship_summary
```

If the current implementation still uses `master_person_relationship`, write the same final summary to `master_person_relationship.relationship_summary` until the tables are merged.

After the assistant response, run memory extraction. Write only durable relationship memories to Mem0, such as:

- `User met Ray Deck at an AI media event in 2024.`
- `User sees Ray Deck's superpower as translating technical AI concepts for media operators.`
- `User sees Ray Deck's blind spot as moving too quickly from idea to partnership before clarifying ownership.`
- `User would ask Ray Deck for warm introductions to media distribution partners.`
- `User considers Ray Deck a warm but not close relationship.`

Do not save the raw transcript, the full side-panel summary, or every assistant question to Mem0. Mem0 should hold small reusable relationship facts; Xano should hold the transcript and display summary.

User chooses meeting prep for a meeting with Katelyn Gallanty from Grant Drive Group.

```txt
Search run_id: meeting_prep:88
```

## Mem0 REST Calls

Mem0's current Platform API uses V3 endpoints for add/search/list. Entity IDs for search/list must go inside `filters`.

### Search durable user memories

Use before each LLM turn.

```bash
curl -X POST https://api.mem0.ai/v3/memories/search/ \
  -H "Authorization: Token $MEM0_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "The user is defining a leverage loop around warm investor introductions.",
    "filters": {
      "user_id": "15"
    },
    "top_k": 8,
    "threshold": 0.1,
    "rerank": false
  }'
```

### Search run-specific memories

Use before each LLM turn for the active flow.

```bash
curl -X POST https://api.mem0.ai/v3/memories/search/ \
  -H "Authorization: Token $MEM0_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "The current chat is refining target investors and rejected paths.",
    "filters": {
      "run_id": "suggestion_request:123"
    },
    "top_k": 8,
    "threshold": 0.1,
    "rerank": false
  }'
```

### List all memories for audit/debug

Do not use this in normal LLM prompts. Use it for admin review, QA, and debugging.

```bash
curl -X POST 'https://api.mem0.ai/v3/memories/?page=1&page_size=50' \
  -H "Authorization: Token $MEM0_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "filters": {
      "user_id": "15"
    }
  }'
```

Use this when the extractor returns `scope = "user"`.

## Memory Extraction

Run memory extraction after the assistant response is saved to Xano.

Recommended model:

```txt
gpt-5-nano
```

Why: memory extraction is a focused classification and summarization task. Use `gpt-5-mini` only if QA shows `gpt-5-nano` misses relationship nuance or produces unstable scopes.

Recommended settings:

```json
{
  "model": "gpt-5-nano",
  "reasoning": {
    "effort": "minimal"
  },
  "max_output_tokens": 1200
}
```

### System prompt

```txt
You are Orbiter's memory extraction service.

Your job is to decide whether the latest chat turn contains durable memory worth saving to Mem0.

You do not write chat responses. You only return structured JSON.

Save a memory only when it is:
- durable beyond the current message
- useful for future recommendations, meeting prep, introductions, or relationship reasoning
- specific to the user, the active run, or a meaningful relationship
- high confidence because the user stated it, confirmed it, or the app state makes it clear
- not already just raw transcript
- not better stored only as structured Xano state

Do not save:
- greetings, thanks, or filler
- generic assistant advice
- full raw emails, full meeting notes, or full context snapshots
- duplicate facts already present in the retrieved memory list
- guesses about the user that were not stated or strongly implied
- sensitive information unless it is necessary for the product experience and the user provided it

Scope rules:
- Use scope "user" for durable facts that should follow the user across outcomes, leverage loops, meeting prep, and how-I-know chats.
- Use scope "run" for facts that should only affect the active thread.
- Use scope "skip" when nothing should be saved.

Categories:
- preference
- goal
- constraint
- decision
- rejected_path
- relationship
- meeting_strategy
- workflow_state
- source_fact

Memory style:
- Write one short standalone sentence.
- Include the relevant entity names.
- Avoid IDs unless the ID is the only stable label.
- Prefer "User prefers..." or "For this run..." phrasing.
- Do not include private implementation details.
```

Backend save policy:

- Save `confidence = "high"` automatically.
- Queue `confidence = "medium"` for QA or save only if the category is low risk.
- Never save `confidence = "low"`.
- Skip `scope = "skip"`.
- Deduplicate against retrieved memories before POSTing to Mem0.

## Source-Specific Guidance

### Outcome and leverage loop

Save:

- selected target or goal
- rejected paths
- strong constraints
- preferred relationship path
- confirmed decision criteria

Do not save:

- every draft suggestion
- model-generated candidate lists
- transient brainstorm text

### Meeting prep

Use `mvp/meeting-prep/classify-participants_v2` (`function #13070`) to build participant context. Store its result in `meeting_prep_context.participant_context`, derive `meeting_prep_context.participant_context.meeting_prep_memo_markdown`, and feed that Markdown to the meeting-prep chat every turn as deterministic Xano context.

Save:

- meeting strategy the user explicitly wants to reuse
- relationship context confirmed by the user
- durable follow-up preferences

Do not save:

- raw transcript
- full meeting notes
- full `#13070` participant context or meeting memo Markdown
- unconfirmed guesses about the other person

### How-I-know

Save:

- how the user met or knows the person
- strength and nature of the relationship
- whether the person is a warm intro path
- user-confirmed boundaries or sensitivities

Do not save:

- speculative relationship labels
- raw relationship chat transcript

### Email and calendar

Save:

- distilled facts that shape future recommendations
- user-confirmed preferences or commitments
- important relationship or meeting signals

Do not save:

- raw email bodies
- raw calendar descriptions
- private content that is not needed later
