The [Person Enrichment Waterfall](/guides/enrichment/waterfall/person-waterfall) uses the canonical graph node definitions in [Ontology → Node Types](/guides/ontology/nodes). People land on the shared `Person` node (the `:Angel` modifier label is stamped by the investor edge writer; `:C_Suite` currently has no live writer); the phase-7 edge resolvers mint `Certification`, `Honor`, `Work:Project`, and `Work:Publication` nodes plus `Organization`-shaped companies; phase 2 fans out to the shared `City` / `Region` / `Country` chain; phase 8 mints `SubDomainExpertise` taxonomy nodes and bridges film people into the IMDB waterfall (phase 8b bridges music people). Company nodes belong to the company pipeline — under the controlled-waterfall policy a person run full-enriches its selected current company and schools, queues past employers as depth-1 leaves, and links everything else lookup-/local-only.

This page is the **pipeline-scoped catalog**: which node types this pipeline writes, keyed how, by which function. Node labels, properties, vector fields, and examples should be edited only in the ontology source of truth.

## People

### Person

Canonical section: [Person](/guides/ontology/nodes#person).

**Create** — `add-person-node` #2601 (v2.1, 2026-06-25), invoked from the entry gate `master-person-new` #13039 right after the `master_person` insert (also by the Exa C-suite helper #12997 and the IMDB person path). `MERGE` keyed on `master_person_id` — **idempotent**, unlike the company pipeline's raw-CREATE #2602. `ON CREATE` sets a random `uuid`, `display_label: "Person"`, and `visibility: false`; the `:Entity` base label is added on every run. 3-retry Falkor loop; v2.1 returns a safe `$nodeUuid` so a Falkor timeout can't throw; guarantees an idempotent `person_enrich_data` shell. Avatar fallback: Serper image search (`find-avatar` #12846) → `create-placeholder-avatar` #1832 (v1.3 — UI Avatars initials, dedup by `original_source_url`), with the Serper `replace-avatar` step **best-effort** since v2.0 (2026-06-22, try/catch non-blocking). Visibility flips to `true` at phase 11 via `complete-person-enrich` #12590 — fresh Exa C-suite leaves deliberately start `visibility: false` (#12997 v1.7).

**Sync** — `update-person-node` #2758 (v1.11, 2026-07-09): `MATCH (n:Entity {uuid: node_uuid}) SET …` property sync including bios, investor-profile fields, and — since v1.10 — a scalar `linkedin_url`; forces `master_person.visibility = true` and mirrors it onto the node. v1.8 repairs an empty `master_person.master_company_id` from resolved current work before syncing; v1.11 adds a non-fatal preflight that inactivates active `master_email` rows whose domains match past-only employers. Three callers: `complete-person-enrich` #12590 v1.3 (phase 11), the [deep-research webhook fan-out](/guides/enrichment/waterfall/deep-research-bio) (#8303), and the orchestrator's **post-thesis best-effort retry**. Embedding text falls back `bio_500` → `bio` → `name`, so Exa C-suite people get searchable nodes before a long bio exists. **One-line Cypher-safe bio projection:** node string properties (including `short_biography` / `long_biography`) get control characters/newlines stripped and whitespace collapsed; the Xano `bio` / `bio_500` keep their line breaks.

**Dual vectors** — both writers dual-write `name_embedding` (OpenAI, 1536) **and** `embeddings` (Gemini, 1536 — was 3072 until 2026-07-06, via helper #13125) since 2026-06-15 (#2601 gates the Gemini line in the builder lambda, #2758 in the Twig template — a Gemini miss still creates/syncs the node with `name_embedding` only).

### Angel \+ C\_Suite (modifier labels)

Canonical section: [Angel](/guides/ontology/nodes#angel). Label-only modifiers — no extra properties. **As-built today:** #2758's dynamic label lambda (which stamped `:Angel` / `:C_Suite` from the `master_person` bools) was lost in its v1.8 rebuild (2026-07-05) and has **not been restored** — an open item. The only live `:Angel` writer is `resolve-investors-edges` #12702 (`SET p:Angel` on its angel investor branches), and **nothing currently stamps `:C_Suite`** — the label survives on legacy nodes and keeps its `C_Suite(uuid)` index. The relational flags remain live: `is_angel` is set by phase 8 (#12829 v1.1) from the **legacy `expertise_join` table, domain 22** — not yet migrated to `person_has_expertise` #711; `is_c_suite` is set by `master-person-from-exa` #12997 on Exa C-suite/founder discovery.

## Organizations

Person enrichment **links** organization-shaped nodes, and under the 2026-07-04/05 controlled-waterfall policy it selectively enriches them again: a root run **full-enriches the selected current company** (phase 3 \+ the orchestrator's exit dispatch) and its **schools** (inline in #12560), **queues past employers as depth-1 leaf companies** (#12562 v1.8+, #13031 v1.8+ — unresolved LinkedIn-backed/name-only rows can mint leaf company shells, while generic placeholders like "Stealth Startup" are blocked), and keeps publishers (#12579 v5+) and honor/cert issuers (#4574 v4 / #12578 v2.5+) as local-only depth-1 leaves. Full Company node lifecycle: [Company Nodes](/guides/enrichment/waterfall/company-nodes).

### Company (stub)

Canonical section: [Company](/guides/ontology/nodes#company). Resolution goes through `get-add/master-company-new`, which delegates real node creation to the company writer #2602 — the person pipeline never writes a full Company node. The work-edge writer `create-work-edges` #2800 (v1.8, 2026-06-25) additionally stub-`MERGE`s both endpoints — `MERGE (c:Company {uuid})` with a name-backfill `CASE` and coalesce-stable timestamps (patched 2026-06-01) — so edges never dangle. The stub carries **no vectors and no profile fields**.

### School

Canonical section: [School](/guides/ontology/nodes#school). Schools from `education_experience` resolve via `resolve-edges-education` #12560 (v1.13, 2026-07-07; LinkedIn URL → domain → name, with a DeepSeek same-school tie-breaker and strong school-evidence gates since v1.10–v1.13) through `master-company-new`; the `:School` create-time label is applied by #2602 (`.edu` domain or `is_school`). Since v1.9 (2026-07-05) resolved schools **full-enrich inline** — `enrich-master-company` at `cascade_depth: 1`, C-suite/founder expansion still gated off.

### VC\_Firm

Canonical section: [VC\_Firm](/guides/ontology/nodes#vc_firm). Additive label only: `resolve-investors-edges` #12702 (v1.17, 2026-07-07) runs `MATCH (vc:Company) SET vc:VC_Firm, vc.updated_at = timestamp()` (now also repairing the node `name` from Fundable source names) during the company-side deal cascade.

### Organization

Canonical section: [Organization](/guides/ontology/nodes#organization). Two shapes: `resolve-edges-volunteering` #4575 (v4, 2026-06-04) standalone-`MERGE`s `(org:Organization:Company {name})` — keyed on name, created fresh when no `master_company` matched (`createdAt` → `created_at` rename \+ `ON MATCH SET`, 2026-06-01) — while `resolve-edges-certifications` #12578 adds the label to an **existing** Company node (`SET organization:Organization` \+ `updated_at`).

## Locations

Phase 2 (#12823, unversioned) writes the primary location only when `master_person.primary_location_id` is null, cascading Enrich Layer `location_str` → LinkedIn `location` → PDL `location_name`, then geocoding via Radar in `add-person-primary-location` #2766 (headerless since 2026-03-19; **fail-open** — failures land in `log_crash`) before handing the structured result to the person-side writer `primary-location-nodes-edges_v2` #4690 (four Cypher branches: city\+region\+country / city\+country / country-only / region\+country). The `LOCATED_IN` / `IN_REGION` / `IN_COUNTRY` edges are on [Person Edges](/guides/enrichment/waterfall/person-edges).

### Country

Canonical section: [Country](/guides/ontology/nodes#country). MERGE-keyed on `{name: country_name}` from the relational `country` row; profile fields written `ON CREATE` only. The Country branch's timestamps \+ `ON MATCH SET` were fixed 2026-06-01. Canonical complete Country creator: `add-all-country-nodes` #12583; company sibling writer: #4528.

### Region

Canonical section: [Region](/guides/ontology/nodes#region). MERGE-keyed on `region_key = "<region>::<country>"` (lowercased). The person writer always labels `:Entity:Region` with `display_label` `"State"` or `"Region"` — contrast the company writer #4528, which mints a dynamic `:State` label for US rows. Carries an LLM-generated `summary`; `name_embedding` refreshed on match.

### City

Canonical section: [City](/guides/ontology/nodes#city). MERGE-keyed on `city_key = "<normalized city>::<region>::<country>"` (lowercased); the raw input city stays the node `name`, and `ON MATCH` refreshes name, summary, and `name_embedding`.

## Expertise

### SubDomainExpertise

Canonical section: [SubDomainExpertise](/guides/ontology/nodes#subdomainexpertise). Phase 8 (#12829 v1.1) calls `llm-identify-person-expertise` #12666 (`qwen/qwen3.7-plus`, reasoning disabled), which clears the person's existing `HAS_EXPERTISE` edges **and their `person_has_expertise` #711 mirror rows** (2026-07-06) and hands the signals to `resolve-person-expertise-v2` #12926 (**v2.8, 2026-07-08**): Gemini `expert_embeddings` KNN against the expertise taxonomy with a **3-way guard band** — distance \< 0.15 auto-match, 0.15–0.25 Qwen LLM adjudicate, \> 0.25 create. Only a guard-band miss `CREATE`s a new `Entity:SubDomainExpertise` (`source: "llm_identified"`, dual vectors `name_embedding` \+ `expert_embeddings`). No MERGE key — **dedup is the matcher**, with the v2.4 exact-name guard as backstop; v2.4 also verifies the CREATE (`create_failed`, no dangling rows), stamps `created_at`, and dual-writes the relational `domain_expertise_id` FK. Every verified create runs `attach-subdomain-parent` #13142 (v1.4 — since v1.3 it also idempotently repairs matched/reused nodes) → `resolve-parent-domain` #13141 (v1.6 — LLM-picks a curated parent, **never auto-creates a domain**; a NONE verdict flags the node `parent_unresolved`) and MERGEs `HAS_SUBDOMAIN {weight: 100}` — deliberate hub damping. Matches and creates alike dual-write `person_has_expertise` #711 via #12925 v1.2; the deep-research webhook (#8303) safely re-runs the whole chain. `HAS_EXPERTISE` weights and properties are on [Person Edges](/guides/enrichment/waterfall/person-edges) — linked, not duplicated.

### DomainExpertise

Canonical section: [DomainExpertise](/guides/ontology/nodes#domainexpertise). **Never created by the person pipeline** — the curated domain roster is synced from the reference tables by `sync-domain-expertise-nodes` #12663; person enrichment touches domains only through the #13142 parent attach.

## Credentials & achievements

All four types are minted by the phase-7 edge resolvers (#12828 v1.2, 2026-06-25 — phase 7 is **skipped entirely** for `current_company_only` people); their edges (`HAS_CERTIFICATION` / `GRANTED_BY` / `CERTIFIED_BY`, `RECEIVED_HONOR` / `ISSUED_BY`, `CONTRIBUTED_TO`, `AUTHORED` / `PUBLISHED_BY`) are on [Person Edges](/guides/enrichment/waterfall/person-edges).

### Certification

Canonical section: [Certification](/guides/ontology/nodes#certification). `MERGE (certification:Entity:Certification {name})` — keyed on the certification name, `issue_date` / `description` / `issued_by` written when present, carries `name_embedding`, stamps `created_at` (coalesce) \+ `updated_at`. Written by `resolve-edges-certifications` #12578 (v2.6, 2026-07-07): LLM cert descriptions run on `openai/gpt-oss-120b` (v2.2) with OpenRouter web search for issuer domains (v2.3) and are normalized before graph writes (v2.6); v2.5 scopes every query to the input `master_person_id`, and issuers are always `cascade_depth=1` leaves (v2.4).

### Honor

Canonical section: [Honor](/guides/ontology/nodes#honor). Raw `CREATE (h:Honor {…})` — a **legacy non-`Entity` label, not MERGE-keyed** (uniqueness is caller-side), no vector. Written by `resolve-edges-honor` #4574 (v4, 2026-06-23): v4 scoped issuer-resolution queries to the input `master_person_id`; issuers are depth-1 leaves (v3). Timestamps added 2026-06-01.

### Project

Canonical section: [Project](/guides/ontology/nodes#project). **`CREATE (proj:Entity:Work:Project {…})` since #12579 v6 (2026-07-04)** — projects joined publications under the unified `Work` parent, with `display_label: "Project"`, `name`/`title`, a description, and timestamps (pre-v6 bare `:Project` nodes linger). v6 also made the resolver process **existing** project rows instead of self-skipping, so re-runs repair rather than duplicate. Contributor resolution is seed-person-only since v4. Sibling creators: `add-projects-publications-clado` #12576 / `-pdl` #4576. No vector.

### Work:Publication

Canonical section: [Work:Publication](/guides/ontology/nodes#publication). `CREATE (pub:Entity:Work:Publication {…})` — **relabeled from bare `Publication` to the unified `Work` parent on 2026-07-01**. CREATE-only with a Project-style row guard, but keyed on `edge_uuid` — #12579 processes only publication rows whose `edge_uuid` is still null; no vector. Publisher companies are leaf/local-only since v5 (`queue=false` \+ `hold_downstream=true`).

## Funding

### Funding\_Round

Canonical section: [Funding\_Round](/guides/ontology/nodes#funding_round). MERGE-keyed on `uuid = fundable_deals.id` — the Fundable deal id is the node uuid, a natural key. Written by `add-fundable-deal-node` #12701 (v1.17, 2026-07-09) per deal from the **company-side** deal cascade (`add-all-fundable-deals` #12703 → `cascade-deal-participants` #12856): since v1.10 (2026-07-06) new writes add the **`:Entity` base label** \+ `visibility: true` (pre-v1.10 rounds linger as bare `Funding_Round`), carry a natural display `name` and a non-empty `description`, upsert `company_funding_round` with the node uuid (v1.12), and register in `node_registry` (v1.17). The **person side no longer materializes rounds** — phase 9 (#12830, 2026-07-02 D-series) writes direct `(Person)-[:INVESTED_IN]->(Company)` edges instead. Carries `name_embedding` only — **no** Gemini `embeddings` on this type. The same writer serves the company waterfall ([Company Nodes](/guides/enrichment/waterfall/company-nodes)); the `RAISED` and investor edges are on [Person Edges](/guides/enrichment/waterfall/person-edges).

## Film & TV

### Film\_TV \+ Film\_TV\_Award

Canonical sections: [Film\_TV](/guides/ontology/nodes#film_tv) and [Film\_TV\_Award](/guides/ontology/nodes#film_tv_award). The person pipeline never writes either type directly. Phase 8, on `film_industry=true`, verifies the person's IMDb URL via `verify-imdb-url` #12677 (`google/gemini-3-flash-preview`, temp 0.1, Exa IMDb candidates disambiguated against the full person context) and — **only when that returns `confidence: "high"`** — stores it through `create-master-link` #2541 and links the row via `add-imdb-to-master-person` — from there the IMDB waterfall's writers own the nodes: `add-film-tv-node` #2320 (MERGE on `imdb_url`) and `add-node-edge-imdb-award` #2334 (`:Winner` / `:Nominee` subtype labels). Linked, not duplicated: [IMDB Nodes](/guides/enrichment/waterfall/imdb-nodes).

## Timestamp coverage

**Graph node timestamp coverage (audited 2026-06-01).** Every node create/update reachable from the person waterfall stamps `created_at` on create and `updated_at` on update — verified across `add-person-node` #2601, the phase-7 `resolve-edges-*` writers, phase-8 expertise \+ IMDB, phase-9 funding, the phase-2 location chain, and the `create-work-edges` #2800 stub-`MERGE` (all patched 2026-06-01). The orchestration functions — `master-person-new` #13039, `enrich-master-person` #13040, `cascade-person-data` #13037, `match-master-person` #13038, `process-person-exa-profile` #13031 — perform **no** graph-node Cypher; they delegate every node write to the leaf writers above.
