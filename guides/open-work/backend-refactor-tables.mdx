Design reference for the **Orbiter Universe to Orbiter backend** table refactor. For each of four person/company join tables, this shows the target **Universe (AlloyDB)** shape and the matching **Orbiter backend (Xano)** projection, each as its own table with one example record. Sample data is a fictional _Jane Doe @ Acme Robotics_ (`master_person_id = 84213`).

<Note>
  **Sync model: projection-with-lock.** The Universe (AlloyDB) is the source of truth; the Orbiter backend (Xano) is a downstream projection. Each enrichment run ships a per-person package. The Orbiter side upserts every child row keyed on `(master_person_id, natural_key)`, skips any row flagged `locked` (user hand-edit), and no-ops when `content_hash` is unchanged.
</Note>

### Column tags

- **Universe-only** — enrichment internals that stay in AlloyDB and are _not_ projected to Xano: `confidence`, `peak_confidence`, `evidence`, `first_seen_at`, `last_seen_at`, `geohash`, `match_embedding`, and the `universe_*_id` primary keys.
- **Added to Orbiter** — four sync-machinery columns added to every Xano table: `natural_key`, `content_hash`, `source_run_id`, `locked` (plus optional `place_id` on addresses).
- Everything else is **shared** — it exists in the Xano table today and is mirrored into the Universe.

## Identity plumbing

Each Universe row has its own `uuid` primary key **and** carries `xano_master_person_id` (and `xano_master_company_id`) so the apply endpoint can resolve the parent across systems. The child upsert key is always `(master_person_id, natural_key)`. Xano keeps its existing `int` primary keys untouched.

## How the sync works

The enrichment engine never writes to Xano directly. Each completed person is shipped as a **package** and applied by a single transactional endpoint on the Orbiter backend.

```mermaid
flowchart LR
  A[Enrichment run] -->|same txn| B[(AlloyDB Universe rows)]
  A -->|same txn| O[[outbox]]
  O --> D{{Cloud Run drainer}}
  D -->|project to Xano subset| P[/person package/]
  P -->|POST| E[apply_person_package]
  E -->|reconcile by natural_key| X[(Xano master_* tables)]
  E -->|receipt| D
```

<Steps>
  <Step title="Enrich + enqueue — one AlloyDB transaction">
    The run writes/updates the person's Universe rows and inserts an `outbox` row `(universe_person_id, run_id)` in the **same** Postgres transaction. A crash can never half-sync.
  </Step>
  <Step title="Drain + build the package">
    A Cloud Run drainer polls the outbox, reads the person's current Universe rows, and **projects** them down to the Xano subset (shared columns \+ the four sync columns) — one envelope per person.
  </Step>
  <Step title="POST /enrichment/apply_person_package">
    The drainer sends the envelope. On `2xx` it marks the outbox row done; on error it retries (at-least-once delivery).
  </Step>
  <Step title="Reconcile per collection — one Xano transaction">
    The endpoint resolves the parent via `master_person_id`, then reconciles each collection by `natural_key` (rules below), wrapped in a single DB transaction — all-or-nothing per person.
  </Step>
  <Step title="Return a receipt">
    The response reports created / updated / soft-deleted / skipped counts plus any conflicts, so the drainer can log and alert.
  </Step>
</Steps>

### Reconcile rules

Per collection, for each row keyed on `(master_person_id, natural_key)`:

| Situation | Action |
| --- | --- |
| In package, row exists, **not**`locked` | **Update** in place |
| In package, row exists, `locked = true` | **Skip** — user hand-edit wins |
| In package, no matching row | **Insert** |
| `content_hash` unchanged vs stored | **Skip** — idempotent no-op |
| Row exists, absent from package, `authoritative = true` | **Soft-delete** (`active_status = false`) |
| Row exists, absent from package, `authoritative = false` | **Leave** — additive merge |

`authoritative` is a per-collection flag on the envelope: `true` means "this is the complete set for this person, reconcile deletes"; `false` means "merge these in, touch nothing else."

## Apply: request then response

**Request** — `POST /enrichment/apply_person_package` with the envelope above.

**Response** — a receipt. Here `emails` and `phones` were new inserts, `links` was an update, and `addresses` was an unchanged no-op (`content_hash` matched):

```json
{
  "run_id": "enr_2026_07_08_abc",
  "master_person_id": 84213,
  "applied": {
    "emails":    { "created": 1, "updated": 0, "soft_deleted": 0, "skipped": 0 },
    "phones":    { "created": 1, "updated": 0, "soft_deleted": 0, "skipped": 0 },
    "links":     { "created": 0, "updated": 1, "soft_deleted": 0, "skipped": 0 },
    "addresses": { "created": 0, "updated": 0, "soft_deleted": 0, "skipped": 1 }
  },
  "conflicts": [],
  "locked_skipped": []
}
```

<Note>
  **Idempotency \+ the lock loop.** Re-delivering the same `run_id` is safe — every row whose `content_hash` matches the stored value is skipped (lands in `skipped`), so retries never duplicate. When a user edits a row in the app, set `locked = true`; the apply step then never overwrites it. The clean v2 is to feed that user edit back into AlloyDB as top-tier evidence, so the master re-resolves to the user's value and the next package reaffirms it — at which point the lock becomes optional.
</Note>
